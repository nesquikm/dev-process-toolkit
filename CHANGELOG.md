## [1.37.0] — 2026-04-27 — "Conventional"

M36 ships a single user-facing FR: the cutover from the bespoke `M<N>: vX.Y.Z "Codename" — title (STE-<N>)` commit-message form used for M1–M35 to [Conventional Commits v1.0.0](https://www.conventionalcommits.org/en/v1.0.0/), enforced by a local `commit-msg` hook installed by `/setup`. Hybrid posture: pure POSIX-shell hook by default (zero-dep, works on Flutter / Python / mode-none projects); opt-in commitlint variant via `/setup --commitlint` for projects with Node/Bun tooling. Dogfooded — this repo's M36 implementation commit and this very ship commit are the first two canonical CC commits on `main`, both validated by the same hook before they could land. M1–M35 commit history is intentionally pre-CC and not rewritten. Single-FR milestone, minor bump (new public surface: hook template + `--commitlint` flag + `## Commit Convention` section in the CLAUDE.md template). Test count: 1049 → 1071 (+22 — fixture-driven shell-hook test plus updated assertion in the existing `ship-milestone-shape.test.ts` for the new CC release-commit form).

### Added

- **STE-133 — Conventional Commits v1.0.0 enforcement: commit-msg hook template + skill prose + `/setup` integration.** New `plugins/dev-process-toolkit/templates/git-hooks/commit-msg.sh` (~40 lines POSIX shell) validates the message file passed as `$1`: subject regex `^(feat|fix|docs|style|refactor|perf|test|build|ci|chore|revert)(\([^)]+\))?!?: .+` plus ≤72-char length check; empty / comment-only messages allowed (interactive abort path); on invalid, exits 1 with stderr naming the failing rule (`type-prefix-missing` / `unknown-type` / `missing-colon-or-description` / `subject-too-long`) and the offending subject verbatim. Companion `commit-msg.commitlint.sh` delegates to `bunx commitlint --edit "$1"` for projects opting into full spec coverage; `commitlint.config.js` extends `@commitlint/config-conventional`. `/setup` step 6b installs the hook (default + `--commitlint` branches with idempotency tier — missing/byte-identical/different+diff prompt+manual install command on decline); `/setup` does NOT run `bun install` or `bun add` — the install command is surfaced for the user. `/implement` Phase 4 step 15 commit-message section instructs CC subject `<type>(<scope>): <title>` with a type heuristic (`feat` default, `fix`/`docs`/`chore`/`refactor` per FR class) + scope rule + `Refs: STE-<N>` footer, with the proposed type/scope rendered at the step 14 approval gate before commit; full type table + canonical example in `docs/implement-reference.md` § Commit message format (STE-133). `/ship-milestone` release-commit emits `chore(release): vX.Y.Z` with `Release: vX.Y.Z "<Codename>"` and `Refs: M<N>` footers; `/pr` PR title mirrors the feature commit's CC subject ≤72 chars. `templates/CLAUDE.md.template` and root `CLAUDE.md` gain `## Commit Convention` sections (CC v1.0.0 required, `Refs:` footer, `!` for breaking, three sample messages + commitlint opt-in note). Root `CLAUDE.md` notes M36-forward CC enforcement and that M1–M35 history is intentionally not rewritten — the user-prefs `## Git Commit Messages` style override (no Claude attribution, no robot emoji, short and humorous) composes with CC's format constraint. README `## Workflow` paragraph picks up a one-sentence CC-enforcement mention. Test coverage: `tests/conventional-commits-hook.test.ts` exercises the shell hook via `Bun.spawn` against 24 fixtures (6 valid: vanilla `feat:`, scoped `fix(adapters/linear):`, breaking `feat!:`, `chore(release): v1.37.0`, `docs(readme):`, plus subject+body+footer; 5 invalid: no type prefix, unknown `fixme:`, missing colon, empty description after `:`, >72 chars; 4 abort: empty / whitespace-only / comment-only / blank-prefix; 4 boundary: 72-char accept, 73-char reject, leading-whitespace trim, trailing-whitespace trim; 5 file-path / hook-presence). Live verification on this repo: `.git/hooks/commit-msg` rejected `git commit -m "broken"` with `type-prefix-missing` diagnostic, accepted `feat(setup): install commit-msg hook`, rejected a 94-char subject with `subject-too-long`. The M36 implementation commit (`feat(commits): adopt Conventional Commits v1.0.0`) and this ship commit (`chore(release): v1.37.0`) are the first two canonical CC commits on `main`. Out of scope (future FRs): auto-CHANGELOG generation from CC commit log; CI gate (GitHub Action) validating PR commits; CC-aware bump suggestion in `/ship-milestone`; M1–M35 history migration via `git filter-branch` (decisively rejected per /brainstorm 2026-04-27). (STE-133)

## [1.19.0] — 2026-04-22 — "FR Identity Stability"

Retires the scan-and-increment `FR-<N>` identifier in favor of collision-proof AC prefixes: the bound tracker ID in tracker mode (`AC-STE-50.1`), or the last 6 chars of the FR's ULID random portion in `mode: none` (`AC-VDTAF4.1`). The brainstorm question that motivated the milestone — *"is FR-N safe under concurrent FR creation across parallel branches?"* — answered no: FR-N was the one identifier derived from "scan + next-free-number," the classic race-condition shape. ULIDs (file identity) and server-allocated tracker IDs are collision-proof by construction; M16 brings AC prefixes and FR-facing references under the same rule.

The tail slice of the random portion (`slice(23, 29)`) is chosen deliberately over the head (`slice(13, 19)`): `ulid.ts` implements monotonic ULIDs — within the same millisecond, randomness is incremented at the least-significant byte, so burst mints share the leading random chars but differ in the tail (observed in this repo's own `fr_01KPTSA7W8N116XWSXXE0G1PY3` and `…PY4`, which share `slice(13, 19) == "N116XW"` but differ in `slice(23, 29)`). This spec deviation from the original M16 brainstorm (AC-STE-50.2) was caught during Phase A TDD and resolved before any code was written.
