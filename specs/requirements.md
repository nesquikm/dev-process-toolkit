# Requirements

## 1. Overview

**Project:** Dev Process Toolkit — a Claude Code plugin that bootstraps Spec-Driven Development (SDD) + TDD workflows into any project.
**Users:** Claude Code plugin users running SDD/TDD workflows.
**Latest shipped release:** **v2.47.0 ("Synopsis")**.

### Shipped milestones

Per-milestone plan files under `specs/plan/archive/`; archived FRs under `specs/frs/archive/`. The full release chronology lives in `CHANGELOG.md`; this overview only names the live context.

## 3. Non-Functional Requirements

### NFR-1: Skill Size

- No single skill file shall exceed 351 lines
- **Overflow rule:** If a skill approaches 351 lines during implementation, extract detailed reference material (long example tables, multi-stack examples) into a companion file at `docs/<skill-name>-reference.md` and reference it from the skill with a one-line pointer: `See docs/<skill-name>-reference.md for full examples.`
- **Bump rationale (STE-252, STE-305):** the cap raised from 300 → 350 in v2.19.0 to acknowledge spec-write's contract-locked prose pressure (~26 tests assert verbatim prose strings); STE-305 (v2.28.0) bumped to the canonical 351 to align with the actual `wc -l` headroom.

### NFR-2: CLAUDE.md Schema Stability

- All changes are additive — existing section headings must not be renamed or removed within a major version; deprecations require a major-version bump
- `<!-- ADAPT -->` markers in template files (`templates/**`) must remain functional; markers in skill/agent files are removed per FR-15
- A CLAUDE.md generated by `/setup` continues to work with newer skills until the plugin's next major version bump (forward-compat invariant within a major)
- New structured output (FR-3) is optional guidance, not mandatory format change
- The `## Task Tracking` Schema L block is opt-in — its absence is canonical for `mode: none` (AC-STE-8.5); skills must treat absence and `mode: none` identically

### NFR-3: Cognitive Load

- New sections in skills must have clear headings that can be skimmed
- Golden paths (FR-13) should reduce perceived complexity, not add to it
- Total new content per skill should be proportional to the value it provides

### NFR-4: Cross-Skill Schema Consistency

Wherever one skill produces output that another skill reads or references, the format must be standardized:

| Schema | Used By | Format |
|--------|---------|--------|
| AC traceability line | FR-1 (gate-check drift), FR-7 (spec-review) | `AC-X.Y → file:line, test-file:line` or `AC-X.Y → (not found)` |
| Drift label | FR-1 (gate-check), FR-7 (spec-review) | `potential drift` (literal string) |
| Deviation table | FR-5 (implement Phase 4), FR-6 (spec breakout) | Table with columns: `Deviation`, `Classification`, `Resolution`, `Needs Confirmation?` |
| Deviation types | FR-5, FR-6, implement Phase 2 step 9 | Exactly 4 types: `underspecified`, `ambiguous`, `contradicts`, `infeasible` |
| Shallow test anti-patterns | FR-8 (tdd), FR-8 (implement) | Same list, same order in both files |
| Pass/fail report format | FR-9 (visual-check both paths) | Checkmark list: `✓` / `✗` with description |
| Gate verdict strings | FR-1 (gate-check), FR-3 (structured output) | Exactly 3 values: `GATE PASSED`, `GATE PASSED WITH NOTES`, `GATE FAILED` — exact casing, used in verdict section and JSON output |
| Archive pointer line (Schema H, legacy v1) | FR-16, FR-17, FR-19 (v1 layouts only) | Exact literal: `> archived: M{N} — {title} → specs/plan/archive/M{N}.md ({YYYY-MM-DD})`. v2 replaces pointer lines with `git mv` — archived files are addressable directly by ULID / milestone ID. |
| AC sync diff format (Schema K) | FR-39 (`/implement` + `/spec-write` bidirectional sync) | Per-AC line: `AC-{N}.{M}: <classification> \| local: "<text>" \| tracker: "<text>"` where `<classification>` is one of `identical`, `local-only`, `tracker-only`, `edited-both` |
| Task Tracking section format (Schema L) | FR-29, FR-32, FR-34 (every mode-aware skill reads it); STE-64 (branch automation reads `branch_template`) | `## Task Tracking` heading followed by `key: value` lines (one per line). Keys: `mode`, `mcp_server`, `jira_ac_field`, `branch_template`. Blank value = not-applicable-for-mode. Absent `branch_template` ⇒ branch automation disabled (STE-64 AC-1). Section absence ≡ `mode: none`. `git log` is the audit trail for sync/migration/resolution events. See technical-spec.md §3 Cross-Skill Schema Definitions for the canonical grep-based read contract. |

### NFR-5: Live Spec Size Bound

- After any milestone completes and archival runs, live `plan.md` contains only in-flight + future milestones; live `requirements.md` contains only non-archived ACs
- Measured by comparing `wc -l` of `plan.md` and `requirements.md` before and after archival — the post-archival line count must be lower than the pre-archival line count (the delta is negative)
- Prompt caching stability: archival is a surgical move, not a file rewrite. The cache-hot prefix remains stable across runs within a milestone, invalidating only at the moment of archival

### NFR-6: Tracker Integration Latency

- Each MCP operation should complete within ~5s under normal network conditions
- Skills surface a `"waiting on tracker..."` indicator if any single call exceeds 2s, so users know the plugin isn't hung
- Latency expectations are documented in `docs/tracker-adapters.md`; not a runtime gate, but adapters that consistently exceed bounds get a documentation warning

### NFR-7: Adapter Code Size

- Each adapter TypeScript source file (e.g., `adapters/linear/src/normalize.ts`): ≤500 lines (looser than NFR-1's 351-line skill cap, since adapters carry tracker-specific quirks like Linear's description normalization and Jira's field discovery)
- Each adapter declarative `<tracker>.md`: ≤351 lines (per NFR-1)
- If a source file exceeds 500 lines, extract per-operation modules under `adapters/<tracker>/src/` rather than letting a single file grow. No compiled binaries are shipped, so size limits apply to source only

### NFR-8: MCP Call Budget per Skill Invocation

- `/implement` at start: ≤2 MCP calls (`pull_acs` + optional `upsert_ticket_metadata` if FR-39 conflict resolution writes back)
- `/gate-check`: ≤1 MCP call (re-pull for concurrency check per AC-33.3)
- `/pr`: ≤2 MCP calls (`transition_status` + optional `upsert_ticket_metadata` for PR link)
- `/spec-write` per FR save: ≤1 MCP call (`upsert_ticket_metadata`)
- Documented as guidance, not enforced at runtime — but PR review should flag adapters that violate this without justification

### NFR-9: Secrets and Tokens

- The plugin never persists OAuth tokens, API keys, refresh tokens, or auth headers anywhere in the user's project (`specs/`, `CLAUDE.md`, `.dpt/`, working files, logs)
- All token storage is delegated entirely to the MCP server's own credential store (Linear / Atlassian MCPs handle this themselves)
- Hard rule — no exceptions, no "convenience caching"

### NFR-10: Tracker Error Surfacing (Canonical Shape)

All tracker-mode errors and warnings that surface to the user conform to one canonical shape so implementations don't reinvent wording per AC. Format:

```
<one-line verdict>
Remedy: <actionable next step>
Context: mode=<mode>, ticket=<ticket-id-or-unbound>, skill=<skill-name>
```

Applies to:
- AC-30.5 (MCP test-call failure during `/setup`)
- AC-33.4 (network failure mid-MCP call)
- AC-34.5 (declared mode vs. actual state mismatch)
- AC-35.4 (empty AC parse — no ACs found in ticket)
- AC-36.7 (migration failure, transactional rollback)
- AC-38.6 (adapter capability missing — warning, not error)

Each cited AC must reference NFR-10 in its implementation; error text that deviates from the canonical shape is a review-blocking issue.

### NFR-11: Migration Performance

- `/setup --migrate` completes in ≤ 30 seconds for up to 100 FRs on typical developer hardware (Apple M-series / recent x86)
- Dry-run (`--migrate-dry-run`) has the same budget — preview generation is not allowed to be slower than the real migration
- Measured by a `tests/perf/migration.bench.ts` that runs against the `tests/fixtures/v2-minimal/` tree scaled to 100 FRs; CI flags regressions but does not block (documented as advisory)

### NFR-12: Skill Entry Overhead

- Resolving the `Provider` adds ≤ 100 ms to skill startup
- Resolved once at skill entry; no re-resolution mid-execution (AC-43.3)
- Budget documented; not a runtime gate. Measured via a one-shot microbenchmark in `tests/perf/`


### NFR-14: Tracker-Mode Migration Atomicity & Idempotency

- Tracker-mode transitions (`/setup --migrate`) are atomic: CLAUDE.md's `mode:` line is rewritten only on success (AC-STE-14.7/8)
- No-op transitions (target == current) refuse via NFR-10 canonical shape (AC-STE-35.5)
- Partial-failure handling prompts for rollback; the user decides whether to retain partial state or revert

### NFR-15: Identity Invariants

Identity is **mode-conditional** — each tracker mode owns its own identity scheme, and cross-mode symmetry is intentionally not enforced:

- **`mode: none`** — canonical identity is a short-ULID (the 6-char tail of the full ULID). Filename stem, AC prefix, and `id:` frontmatter value all carry that short-ULID; each equals the others byte-for-byte.
- **`mode: <tracker>`** — canonical identity is the tracker ID. Filename stem is `<tracker-id>.md`; AC prefix is `AC-<tracker-id>.N`; the `id:` frontmatter line is **absent** (ceremony dropped in STE-76; the tracker ID is the single source of truth, and `findFRByTrackerRef` handles tracker-workspace renames).

Cross-mode invariants enforced by `/gate-check` (AC-49.5) and monitored by `/spec-review`:

1. **Filenames are immutable after mint.** Renames are forbidden in either mode; archival moves preserve the stem (AC-41.4, NFR-15 filename-permanence).
2. **`id:` ≡ filename stem (mode-scoped to `mode: none`).** In `mode: none`, the short-ULID `id:` value equals the filename stem byte-for-byte (AC-41.2, AC-41.5). In tracker mode the `id:` line is absent — the filename stem equals the tracker ID instead. The bimodal invariant is enforced by the `identity_mode_conditional` `/gate-check` probe (STE-86 AC-STE-86.5).
3. **Providers never modify identity.** `sync()` touches `tracker:` refs and `status:`; it MUST NOT modify filename, title, milestone, or (in `mode: none`) the `id:` value (AC-42.3).

Violations are review-blocking, not advisory. Any skill that would need to violate an invariant is a design bug.

### NFR-16: Tracker-less Lock Check Escape Hatch

- Tracker-less `/implement` runs `git fetch --all` before the lock check by default (AC-46.2)
- `DPT_SKIP_FETCH=1` environment variable skips the fetch for large-repo contexts (AC-46.7) — documented trade-off (trades safety for speed; user opts in with full awareness)
- Docs explicitly state that `DPT_SKIP_FETCH=1` is a user-level choice: the tracker-less lock mechanism is already best-effort (AC-46.6), and this variable makes the best-effort character more explicit rather than less

### NFR-17: Resolver Performance

- `resolveFRArgument` completes in ≤ 10 ms on typical developer hardware (pure string parsing; no network I/O, no filesystem reads except the cached `## Task Tracking` config)
- `findFRByTrackerRef` scans up to 500 FRs in ≤ 50 ms (one frontmatter read per file; may be parallelized)
- Measured via `tests/perf/resolver.bench.ts`; advisory, not a runtime gate

### NFR-18: Resolver Backward Compatibility

- Existing `/spec-write <ULID>`, `/implement <ULID>`, `/spec-archive <ULID>` invocations remain byte-identical across releases
- Keyword arguments (`all`, `requirements`, `technical-spec`, `testing-spec`, `plan`, `M\d+`) fall through the resolver unchanged per AC-51.7 — every skill's existing argument contract is preserved
- Regression fixtures in `tests/fixtures/` pass unchanged (Pattern 9 invariant applies to every mode-aware skill)

### NFR-19: URL-Parsing Safety

- URL host matching is strictly allowlist-based: `linear.app`, `github.com`, and the Jira host configured in `## Task Tracking`. Any other host falls through as unresolved — never "best-guessed"
- No URL fetching. Parsing is purely string-based (host + path extraction via the standard `URL()` constructor); no HTTP requests, no DNS lookups
- Path-extraction regexes are anchored per-tracker; no arbitrary regex groups consumed from user input

### NFR-20: Ambiguity UX Consistency

- Every ambiguity case (multi-tracker prefix collision, bare-integer with GitHub + Jira both configured via overlapping patterns, unknown-tracker-host URL, etc.) produces the same NFR-10 canonical error shape, naming all candidate resolutions AND the explicit `<tracker>:<id>` disambiguation remedy
- Resolver never silently picks a winner; never guesses; non-interactive callers always get the same deterministic error for the same input

### NFR-21: Resolver Test Coverage

- 100% branch coverage on `resolveFRArgument` — every `kind` path, every tracker-configuration combination, every error case
- Mutation-testing probes: removing any ambiguity check, URL-host match, or fallthrough case must cause at least one test to fail
- `findFRByTrackerRef` covered by fixture-based tests with ≥3 active FRs + ≥1 archived FR having matching and non-matching tracker refs; `{includeArchive: true}` opt-in covered

### NFR-22: LLM Doc-Signature Grounding Invariant

LLM-generated doc content must not invent code-level symbols, routes, config keys, or state events that are absent from a deterministically-extracted ground-truth set. Two scopes apply:

- **Fragment writing (`/docs --quick`)** — the LLM prompt pins an `ImpactSet` as authoritative. The post-generation scan in `/docs --quick` checks the fragment body for symbol mentions not in `ImpactSet.symbols` and rejects the fragment (single retry with strictened prompt; on second failure, the fragment is written with a `<!-- warning: ... -->` header naming the invented symbols, never silently)
- **Reference regeneration (`/docs --full` / `/docs --commit`)** — the LLM prompt pins `SignatureGroundTruth`. The post-generation validator parses TypeScript code blocks in the output and asserts every declaration appears in the ground-truth set (name match AND whitespace-normalized signature match). Mismatch triggers one retry; second failure fails `/docs` loudly per NFR-10

Violations are review-blocking: a canonical reference doc that includes LLM-invented signatures is the top failure mode identified during the docs-generation design and must not ship silently. Mitigation is enforced by the `/docs` extractor + validator pair, not by documentation alone.

## 4. Edge Cases

- Gate-check drift detection when `specs/` doesn't exist: skip silently (AC-1.2)
- Spec breakout threshold: customizable via `<!-- ADAPT -->` marker (AC-6.3)
- Visual-check: MCP configured but not responding vs. not configured at all — both trigger fallback (AC-9.2)
- Doctor validation on a project with no package manager: report "no package manager detected" and skip tool version checks (AC-4.5 covers new projects)
- CI starter configs for stacks not in examples/: out of scope, covered by adaptation guide
- Code-reviewer agent invoked without specs: skip spec compliance section silently (AC-11.2)
- Reopened milestone after archival: create `M{N}-r2-{slug}.md` in archive; the live spec gets a fresh milestone block, not the old one resurrected (AC-17.7, AC-19.7)
- `/spec-archive` invoked without an explicit anchor: skill refuses and prompts for a specific `{#M...}` or `{#FR-...}` anchor (AC-17.2)
- Milestone archival when the traceability matrix is incomplete: only the `plan.md` block moves; a warning asks the user to archive the corresponding ACs manually via `/spec-archive` (AC-16.5)
- Retroactive compaction of already-shipped spec content: handled as dedicated dogfooding tasks filed as their own FRs, not as edge cases of the runtime mechanism
- Archive index referenced by `/spec-review` when index file does not exist: skip silently, do not error (AC-19.4)
- Tracker mode `/setup` when MCP server is configured but OAuth not completed: hard-stop per AC-30.5; instruct user to complete OAuth in MCP UI and re-run `/setup`
- Tracker mode branch with no ticket-ID match: fall through to the interactive prompt (STE-62 collapsed the old 3-tier resolver to 2 tiers by removing the legacy CLAUDE.md Tier-2 fallback key); decline exits cleanly
- Tracker becomes unreachable mid-session: skill fails cleanly per AC-33.4; user re-runs after restoring connectivity (no partial-state recovery)
- Two team members editing the same FR's ACs simultaneously (one local, one in tracker): caught by FR-39 conflict resolution on next `/implement` or `/spec-write` save
- `/setup --migrate` `<tracker A> → <tracker B>` failing partway: AC-36.7 transactional guarantee — CLAUDE.md mode unchanged; user prompted to retry or roll back created tickets in B
- `/spec-archive` in tracker mode: still operates on local `specs/` for technical-spec ADRs and similar non-AC content; archival of completed tracker tickets is the tracker's own concern (per its native conventions), not the plugin's
- Tracker MCP returns ACs in an unexpected schema (e.g., Linear changes its description format): adapter's normalizer raises a clear error; user updates adapter or waits for a fix release

**Parallel-safe layout edge cases:**

- Two branches claim `.dpt/locks/<ulid>` simultaneously without fetching first (tracker-less): both commits are valid on their branches. On merge, the `.dpt/locks/` files conflict at path level — git surfaces the conflict, human resolves. AC-46.6 documents this as a detectable-not-prevented race; tracker mode is the strict guarantee
- `.dpt/locks/<ulid>` on a branch that was merged or deleted (stale lock): `/gate-check` flags per AC-46.5; user runs `/gate-check --cleanup-stale-locks` for a one-commit cleanup
- ULID collision at mint time (statistically near-zero): `mintId()` detects via filesystem existence check and retries; after 3 retries, fails with a clear error naming the colliding file
- Two trackers configured but only one has the FR's ref: `sync()` updates only the tracker(s) for which a ref is present; never invents a ref for a tracker that doesn't have one
- User manually edits a frozen `specs/plan/<M#>.md` (bypassing the replan-branch discipline): `/gate-check` warns but does not auto-revert; the check lists the commit SHA that introduced the post-freeze edit
- FR referenced in `technical-spec.md` but missing from `requirements.md` (stale cross-reference): surfaced by `/gate-check` drift detection; not auto-fixed

**Tracker-native entry edge cases:**

- Tracker prefix collision across configured trackers (Linear project `FOO` + Jira project `FOO` both active): `/spec-write FOO-42` is ambiguous → resolver errors in NFR-10 shape naming `linear:FOO-42` and `jira:FOO-42`; user retries with explicit prefix
- Tracker URL from an unconfigured host (`https://gitlab.com/...`) passed to `/spec-write`: resolver returns `fallthrough`; skill treats it as a free-form argument per its existing contract (effectively no-op or prompts for a title, depending on skill)
- `/implement LIN-1234` when the tracker ticket has zero ACs: import populates `## Acceptance Criteria` with a TODO marker (AC-52.7); implementation proceeds with no AC checklist but FR-39 picks up any ACs added tracker-side later
- Two branches call `/spec-write LIN-1234` simultaneously without coordination: both import, both mint distinct ULIDs, both write `specs/frs/<ulid>.md` files with the same `tracker.linear: LIN-1234` ref. On merge, both FR files exist — `/gate-check` warns about the duplicate tracker ref, user manually consolidates (archives one, keeps the other). Prevention is the lock mechanism, but the lock is keyed by ULID not by tracker ref — this is a known gap, documented here rather than engineered around
- `/spec-archive LIN-1234` when the FR was already archived via `/spec-archive <ulid>` earlier: `findFRByTrackerRef` excludes archive by default → resolver reports "no local FR mapped" (archival intent already fulfilled). If user needs to verify, they pass `{includeArchive: true}`-equivalent or look up the ULID directly
- Tracker ticket renamed (e.g., Linear workspace-rename causes `LIN-1234` → `DPT-1234`): frontmatter still has the old `linear: LIN-1234`; `/spec-write DPT-1234` finds no match and imports as a new FR with a new ULID. No auto-migration; user runs a find-replace on `tracker.linear` frontmatter values or accepts the stale ref (FR-39 will flag it on next sync)
- `resolveFRArgument` called with an argument that matches both a ULID regex AND a tracker-ID regex (impossible by construction — ULID prefix `fr_` is deliberately distinct from any tracker prefix): covered by unit test AC-51.9 as a regression canary
- User passes a full Linear URL but the adapter config doesn't have that workspace: URL host matches `linear.app`, path-regex extracts a ticket ID, but `Provider.getMetadata` fails because the configured Linear workspace is a different one → surfaces per NFR-10 with the tracker's actual error message

## 5. Out of Scope

- No new slash commands (depth over breadth)
- No fast-track workflow skipping ceremony
- No hooks/bin/.lsp.json in the plugin
- No multi-agent orchestration
- No mutation testing gate
- No legacy code support (/ingest, /map)
- No actual CI pipeline implementation (only starter configs as example files)
- Atomic records + generated views architecture (the "Approach 4" from brainstorm) — deferred as possible v2
- Automatic archival of `technical-spec.md` content — explicitly rejected; ADRs stay in place with `Superseded-by` per convention
- Content-hash verification of archived blocks — nice-to-have, not required for correctness in v1.10.0
- Migration tooling to convert pre-FR-18 specs (missing anchor IDs) to the new format — out of scope; `/setup` doctor validation warns, user fixes manually

**Tracker-integration out-of-scope:**

- Persistent local cache of tracker state — explicitly rejected; live MCP pull only with `updatedAt` optimistic concurrency
- Tracker-hosted cross-milestone content (technical-spec, testing-spec, plan, NFRs, edge cases, traceability, archive) — all stay in local `specs/`; the tracker holds only ACs + ticket metadata
- External knowledge-base integration (Confluence, Notion, "docs project in Linear") — `specs/` remains the project knowledge base
- Webhook listeners or polling — sync is strictly skill-triggered (`/spec-write`, `/implement`, `/gate-check`, `/pr`)
- Auto-migration on plugin upgrade — `/setup --migrate` is explicit user invocation only; `none` mode projects stay on `none`
- Multi-ticket per branch / stacked diffs — single active ticket per branch; adapters fail with a clear "multiple ticket IDs detected" message rather than guess
- Custom-field discovery beyond the AC field — only Jira's AC custom-field GID is auto-discovered; other custom fields (priority, story points, sprint) are ignored
- Cross-repo / monorepo per-package ticket binding — one repo, one tracker project, one mode; out for this release (flagged in technical-spec future considerations)
- Self-hosted tracker support (self-hosted Jira, GitLab, etc.) — the plugin relies on official cloud MCPs; self-hosted may or may not work via the Atlassian MCP, explicitly not tested or supported
- Bundling MCP servers — the plugin documents and guides install but does not ship MCP server code
- OAuth flow implementation — the plugin guides the user (prints sign-in URLs, instructs steps) but does not perform token exchange itself
- Adapter for trackers beyond the shipped set (GitHub Issues, Notion databases, Trello, Monday, ClickUp, Height, Shortcut, Pivotal, Basecamp, etc.) — users may create custom adapters from `_template.md`; community contributions can be upstreamed in later releases

**Parallel-safe-layout out-of-scope:**

- Web UI or GUI for browsing specs — raw markdown files under `specs/frs/` are the only interface; no dashboard, no viewer, no IDE plugin
- Cross-tracker reconciliation logic — if an FR has both `linear:` and `github:` refs and they disagree on status, the provider surfaces the discrepancy; the plugin does NOT resolve automatically, human decides
- Automatic conflict resolution on same-FR edits — two branches editing the same `specs/frs/<ulid>.md` intentionally produce a normal git conflict, which is the correct signal. No custom merge driver
- Real-time tracker sync — `sync()` runs on skill invocations only (entering `/implement`, Phase 4 completion, `/spec-write` save). No daemon, no file watcher, no pull-on-every-read
- Spec-as-database querying — `grep`/`rg` over `specs/frs/**/*.md` is the only query surface; no DSL, no SQL over frontmatter, no Linear-style filter language
- Multi-repo spec federation — `specs/` is per-repo; teams with multiple repos run DPT independently per-repo
- Strict enforcement in tracker-less mode — `.dpt/locks/` + remote fetch is best-effort (AC-46.6); tracker mode is the strict guarantee. The plugin does NOT promise preventing parallel claims in tracker-less mode
- Renaming ULIDs after mint — never supported; the ULID is the canonical repo-stable identity for the lifetime of the FR

**Tracker-native-entry out-of-scope:**

- Bulk import — `/spec-write` handles one ticket per invocation; no "import all open tickets in milestone X" or "sync last 20 Linear issues."
- Creating new tracker tickets from scratch — `importFromTracker` handles existing tickets only; creating a new ticket from a local FR remains `upsert_ticket_metadata`'s responsibility
- Tracker discovery / search — `/spec-write` doesn't list open tickets, filter by assignee, or accept search queries; the user brings a specific ID or URL
- Auto-refresh without explicit invocation — tracker updates flow in via the FR-39 loop on next `/implement` / `/spec-write` save; no background poller, no file watcher, no webhook listener
- Re-import / overwrite — once a local FR exists with a tracker ref, `/spec-write <tracker-id>` opens it for editing; force-reimport is out (user deletes the FR file manually if that's needed)
- Per-project custom regex overrides for tracker ID patterns — patterns live in the adapter, not in the project's `## Task Tracking` config
- Shortcut aliases — no `/spec-write 1234` inferring "latest tracker, last project"; every invocation is explicit
- Tracker-ref rewriting on tracker renames — if a Linear workspace rename changes `LIN-1234` to `DPT-1234`, the plugin doesn't auto-migrate frontmatter; user runs manual find-replace or accepts the stale ref
- Interactive tracker picker for bare numeric arguments with multiple trackers configured — the resolver errors with the `<tracker>:<id>` disambiguation remedy rather than prompting
- Resolver parsing of arbitrary URLs beyond the registered tracker hosts — URLs with unknown hosts fall through as unresolved per NFR-19's allowlist discipline
