# STE-367: workspace-trust operator precondition (assert + refuse, no self-write) {#STE-367}

## Requirement

STE-356's `/smoke-test` Phase 1 **step 6b** seeds `hasTrustDialogAccepted: true` for the test-project path into the operator's live `$CLAUDE_CONFIG_DIR/.claude.json`. That programmatic write is **reliably denied** by the harness auto-mode self-modification classifier under `claude -p` (`[Self-Modification] … seeding workspace trust to bypass the trust-dialog safety gate; not explicitly requested by the user`), so the hands-off `/conformance-loop` loop cannot self-seed trust — finding F1 (2026-07-04), which blocked iteration 1 on both legs. The `<dpt:auto-approve>` marker governs only the skill's own Phase 0 gate, not harness-level guards, and covertly retrying the write with a different tool would violate the intent of the denial.

Replace the autonomous **write** with an operator **precondition**: assert both test-project paths are already trusted, and refuse (NFR-10 canonical shape) with the exact jq seed one-liner as the remedy if not. The trust decision moves out of the autonomous path entirely and back to the operator — which is exactly the workaround that succeeded on 2026-07-04, and is the classifier's own suggested remedy path.

## Acceptance Criteria

- AC-STE-367.1: **step 6b asserts, does not write.** `/smoke-test` Phase 1 step 6b no longer writes `hasTrustDialogAccepted` to the live `.claude.json` (the jq merge + pre-write backup + spinlock are removed). It reads `.projects["<abs test-project path>"].hasTrustDialogAccepted` and, if not `true`, refuses with the NFR-10 canonical shape whose remedy is the jq seed one-liner for that path; on the hit path (trust present) it proceeds. {#AC-STE-367.1}
- AC-STE-367.2: **conformance-loop up-front precondition.** `/conformance-loop` gains a pre-flight (numbered after the existing (a)–(g), fired before any `claude -p` spawn) that asserts **both** `../dpt-test-project-{linear,jira}` resolved paths are trusted in the live config, refusing (NFR-10 canonical shape) with the jq remedy naming the untrusted path(s) if either is absent. The operator seeds trust once, up front, instead of the loop hitting a mid-run denial. {#AC-STE-367.2}
- AC-STE-367.3: **Refusal content + hit-path token.** Each refusal names the missing path(s) and prints the exact jq one-liner (`jq '.projects["<path>"] |= (. // {} + {hasTrustDialogAccepted: true})' "$CFG" > tmp && mv tmp "$CFG"`) plus a re-run instruction. The assert-hit path logs the byte-checkable capability token `workspace_trust_present` (replacing STE-356's `workspace_trust_seeded`, which is removed) to the same approval record, one literal line, no inference. {#AC-STE-367.3}
- AC-STE-367.4: **Teardown no longer removes trust.** `/smoke-test` Phase 5 teardown no longer deletes `.claude.json` trust entries or the step-6b backup file (there is no seed the skill wrote and no backup it took). Trust for the two fixed test-project paths is operator-owned and intentionally persistent across runs, removing re-seed friction. The teardown's step-6b `del(.projects[$p])` + `.bak` `rm` + associated spinlock are removed. {#AC-STE-367.4}
- AC-STE-367.5: **Runtime validation (deferred `[~]`).** The next live `/conformance-loop` run with trust pre-seeded proceeds past Phase 1 step 6b on both legs with zero self-modification classifier denials; a run with trust **absent** shows the new up-front conformance-loop precondition refusing with the jq remedy before any spawn. Validated on the next live run. {#AC-STE-367.5}

## Technical Design

**Files touched:**
- `.claude/skills/smoke-test/SKILL.md` — rewrite Phase 1 step 6b from a read-merge-write (`cp` backup → jq merge → atomic `mv`, under the `/tmp/dpt-claude-json.lock` spinlock) into a read-only assert: probe `hasTrustDialogAccepted == true` for `$TEST_REAL`; miss ⇒ NFR-10 refusal with the jq remedy; hit ⇒ log `workspace_trust_present`. The existing "before the first Phase 2 spawn" trust-assert (currently a second, redundant check) collapses into step 6b (or remains as cheap defense-in-depth, but the write is gone). Remove the Phase 5 teardown trust-removal blocks (both tracker paths) + the `.bak` removal + their spinlocks.
- `.claude/skills/conformance-loop/SKILL.md` — add pre-flight **(h)** "both test-project paths trusted" after refusal (g), before Phase 0. Read `$CLAUDE_CONFIG_DIR/.claude.json`, assert both `/Users/.../dpt-test-project-{linear,jira}` (the resolved abs paths) carry `hasTrustDialogAccepted == true`; refuse (NFR-10) with the jq remedy if either absent. Update the "Inherited precondition — workspace trust (STE-356)" threat-model section to reflect the assert-not-seed model. Update the refusal-count prose (the `grep -c 'NFR-10 canonical refusal'` verify line) to include (h).
- `plugins/dev-process-toolkit/tests/m98-ste-367-workspace-trust-precondition.test.ts` (new) — asserts step 6b contains no `hasTrustDialogAccepted: true` **assignment** (no jq write) but does contain the assert + jq-remedy refusal + `workspace_trust_present`; asserts conformance-loop pre-flight (h) exists with the jq remedy; asserts teardown no longer contains the trust `del` + `.bak` `rm`.

**Token rename fallout.** `workspace_trust_seeded` → `workspace_trust_present`: grep the repo for the old token (SKILL prose, any capability-map / aggregator references, existing meta-tests) and update every occurrence, so the drift meta-test and any `closing_summary`-style aggregation reference the new token.

**Interaction with STE-365 / the -p-context probe.** Independent surfaces (trust vs spawn discipline); no ordering dependency. In a `/conformance-loop` run, pre-flight (h) passing up-front guarantees the child `/smoke-test` step-6b assert also passes (trust present), so the two layers are consistent.

## Notes

- **Origin.** 2026-07-04 `/conformance-loop` run F1 (worked around live by an operator-authorized manual seed — precisely the model this FR makes canonical).
- **Persistent trust is intended.** The two test-project paths are fixed (`../dpt-test-project-{linear,jira}`) and operator-owned; trusting them once and leaving the entries is low-risk (throwaway dirs the operator controls) and removes per-run re-seed friction. This reverses STE-356's teardown-removes-seed hygiene, which only existed because the skill was the writer.
- **STE-356 relationship.** This supersedes STE-356's step-6b *seeding* mechanism while preserving its *intent* (the scaffolded allow-list is inert in an untrusted workspace — F4). The enforcement precondition is unchanged; only who satisfies it (operator, up-front) changes.
- **Sibling FRs.** STE-365 (fire-and-exit guard) and STE-366 (zsh-safe glob fences) complete M98.
