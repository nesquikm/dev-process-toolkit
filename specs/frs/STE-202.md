# STE-202: Doc & workflow polish: /simplify in Feature row, prose drift, permissions trim, branch_template trunk-OK {#STE-202}

## Requirement

Five mechanical doc/template/config fixes that don't share logic but each rolls in cleanly with the others. Bundling avoids five trivial commits.

**Findings bundled:** F3.8, F3.11, F4.1, F2.1a, F3.10 from `/tmp/dpt-findings-2026-05-04.md`.

**Symptoms observed in `dpt_test_lib`:**

1. **F4.1.** STE-193 (`subtract`) shipped via the canonical Feature workflow `/brainstorm → /spec-write → /implement → /spec-review → /gate-check → /pr` (per `templates/CLAUDE.md.template:72-78`). The Technical Design proposed `lib/src/arithmetic.dart` for `subtract` while leaving `add` in `lib/src/dpt_test_lib_base.dart`. Result: two one-line files in the same domain with a meaningless template-derived name (`_base.dart`) surviving alongside the domain-meaningful one. Operator caught it on visual inspection ("VERY ugly, how did it happen?") and we planned a separate M3 milestone (`Consolidate arithmetic primitives into one source file`) to undo the split. M3 wouldn't have been needed if `/simplify` ran in the Feature workflow. `/simplify`'s own description says "Use after completing a feature to clean up" — but the Feature row omits it.
2. **F3.8.** `/implement` SKILL.md mixes singular `specs/plan.md` (lines 55, 58, 221) with the correct plural form `specs/plan/M<N>.md`. A literal-minded executor might `Read specs/plan.md` and fail.
3. **F3.11.** Resolver routing for `^M\d+$` is split between `docs/resolver-entry.md` (where it's a `fallthrough`) and `/implement` Phase 1 step 2 (where it's actually handled). Auditing what `/implement M1` does requires reading both docs.
4. **F2.1a.** `templates/permissions.json _common` includes `Bash(rm *)`, `Bash(mv *)`, `Bash(cp *)`, `Bash(mkdir *)`, `Bash(ls *)`. Triggers harness self-modification denial on every fresh `/setup` because the operator must authorize a broadly-destructive list. Audit: only `mkdir` is genuinely used (in `/setup` step 8); `rm`/`mv`/`cp` are never invoked from skill prose.
5. **F3.10.** CLAUDE.md `## Task Tracking` declares `branch_template: {type}/{ticket-id}-{slug}` with no allowlist of "trunk-OK" types. Pre-toolkit `chore`/`docs` commits live on trunk; first feature commit must be on a derived branch — inconsistency confuses operators.

## Acceptance Criteria

- AC-STE-202.1: `templates/CLAUDE.md.template` Workflows block lists `/simplify` in the Feature row:
  ```
  Feature: /brainstorm → /spec-write → /implement → /simplify → /spec-review → /gate-check → /pr
  ```
  Same change applied to any rendered copy in `docs/sdd-methodology.md`, `README.md`, or other workflow guides. {#AC-STE-202.1}
- AC-STE-202.2: All `specs/plan.md` (singular, with `.md`) references in `plugins/dev-process-toolkit/skills/implement/SKILL.md` replaced with `specs/plan/<M#>.md` (or `specs/plan/` directory reference, whichever fits the prose). Grep confirms zero remaining singular forms after the edit. Same scrub for any sibling skills that reference the old singular form. {#AC-STE-202.2}
- AC-STE-202.3: `adapters/_shared/src/resolve.ts` adds `milestone` as a first-class resolver kind matching `^M\d+$`. `docs/resolver-entry.md` decision table lists it. `/implement` Phase 1 step 0.b' references the new kind directly (no `fallthrough` punt for milestone codes). Step 2's free-form prose handles only non-milestone fallthrough cases. {#AC-STE-202.3}
- AC-STE-202.4: `templates/permissions.json _common` trimmed to `Bash(git *)` + `Bash(gh *)` only. `Bash(mkdir *)` moves to `/setup`'s per-skill allow-list (the only legitimate caller). `Bash(rm *)`, `Bash(mv *)`, `Bash(cp *)`, `Bash(ls *)` removed entirely; if a skill genuinely needs them, they get added per-skill. Result: the harness self-modification denial on `.claude/settings.json` no longer fires for the canonical first-run path. {#AC-STE-202.4}
- AC-STE-202.5: `branch_template:` Schema L semantics document an explicit allowlist of "trunk-OK" types — `chore`, `docs`, `ci` permitted on trunk; `feat`, `fix`, `refactor`, `perf` enforce derived branch. Documented in `templates/CLAUDE.md.template` `## Task Tracking` section commentary AND in `/implement` Phase 1 branch-validation prose. Backwards-compat: existing repos with `branch_template:` set continue to work; the type-allowlist is the new default. {#AC-STE-202.5}

## Technical Design

**No logic shared.** Each AC is a focused edit; the bundle exists for diff-coherence only.

- AC-STE-202.1 — single-line Workflows-block edit in `templates/CLAUDE.md.template`; identical edit to any other rendered workflow copy.
- AC-STE-202.2 — three line edits in `plugins/dev-process-toolkit/skills/implement/SKILL.md` lines 55, 58, 221; grep audit confirms no other `specs/plan.md` (singular, with `.md`) reference survives.
- AC-STE-202.3 — resolver-entry doc + `resolve.ts` + `/implement` step 0.b' coordinated edits. The kind is purely cosmetic — Phase 1 step 2 still handles the actual milestone routing — but promoting it documents what the resolver actually does.
- AC-STE-202.4 — JSON edit + `/setup` SKILL.md per-skill section update. After this change, fresh `/setup` runs auto-approve via narrower allowlist, eliminating the self-modification prompt.
- AC-STE-202.5 — prose-only addition; no enforcement-code changes (`branch_template:` enforcement remains optional per existing prose). New text documents the type-allowlist convention; future FR can wire enforcement if needed.
