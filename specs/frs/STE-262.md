# STE-262: Tighten /spec-write commit-gate + draft-gate to literal byte-checkable marker only {#STE-262}

## Requirement

`/spec-write`'s draft-acceptance gate (§ 0b step 4) and commit-acceptance gate (§ 7a) are auto-applying on `claude -p` invocations whose prompt body lacks the literal byte-string `<dpt:auto-approve>v1</dpt:auto-approve>`. Cross-tracker `/conformance-loop` (2026-05-08) reproduced the regression on both Linear (commit `bbddfcd` on `feat/STE-259-greet`, FR STE-261 + M3 plan written silently) and Jira (DST-23 created in Backlog + commit `04fb609 chore(specs): write FR DST-23`, no audit row, no refusal): marker-absent runs proceed silently, emitting `spec_write_commit_default_applied` while the contract says they should refuse via `RequiresInputRefusedError`.

The runtime gate is conflating the harness's autonomous-mode reminder ("work without stopping") with the byte-checkable STE-226 marker — the exact failure shape STE-213 + STE-220 hit before STE-226 introduced the marker. **This is the third attempt** at this gate. STE-213 (M55) and STE-220 (M56) were both prose-only and both falsified by subsequent smoke runs. STE-226 (M59) added the marker mechanism but the LLM-inference fallback path is still reachable. The contract — "marker is the single deterministic mechanism" — is not enforced at the runtime layer; it is asserted in prose only.

## Acceptance Criteria

- AC-STE-262.1: `requireOrRefuse(...)` (in `adapters/_shared/src/requires_input.ts` or wherever the gate is implemented) MUST treat any prompt body lacking the literal byte-string `<dpt:auto-approve>v1</dpt:auto-approve>` (case-sensitive, exact, no whitespace tolerance, no regex) as marker-absent — regardless of any "autonomous mode", "work without stopping", "imputed approval", or other conversational autonomy hint elsewhere in the prompt body or system reminders.
- AC-STE-262.2: When marker is absent AND `process.stdin.isTTY === false` (heredoc-on-stdin, `< /dev/null`, piped input), both gates MUST surface `RequiresInputRefusedError` (NFR-10 canonical shape) and exit non-zero. No silent draft, no silent commit, no conversational "imputed approval" path. The error message MUST name the missing marker and the gate site (draft vs commit) so operators can diagnose without re-reading SKILL.md.
- AC-STE-262.3: When marker is present, both gates auto-apply and emit the existing capability rows (`spec_write_draft_default_applied` and `spec_write_commit_default_applied` respectively, literal tokens) — no behavioral regression on the marker-present path. Existing marker-driven smoke fixtures (Phase 9-1 / 9-2a / 9-2b) MUST continue to PASS.
- AC-STE-262.4: New `/gate-check` source-level probe `spec_write_marker_alternate_trigger_scan` (severity: error) greps `plugins/dev-process-toolkit/skills/spec-write/SKILL.md` for prose phrases that could suggest alternate auto-apply triggers. Allow-list of forbidden phrases stored as a TypeScript array in the probe source: `Auto Mode`, `work without stopping`, `imputed approval`, `autonomous-mode reminder`, `inferred approval`, `non-interactive inference`. Hits refuse the gate-check.
- AC-STE-262.5: Reproduction integration test — under `claude -p`, spawn `/spec-write` with a prompt body that (a) lacks the marker AND (b) contains autonomous-mode-reminder text (literal "work without stopping" or similar). The gate MUST refuse with `RequiresInputRefusedError`. Same fixture passes today (current bug reproducible against the Linear `bbddfcd` and Jira `04fb609` commits), fails after the fix lands. Lives at `plugins/dev-process-toolkit/skills/spec-write/__tests__/marker_strict_byte_grep.test.ts` (or `adapters/_shared/__tests__/` equivalent).
- AC-STE-262.6: `docs/auto-mode-protocol.md § Socratic Loop Contract` updated with one-line clarification: "the harness's `Auto Mode` system reminder is NOT a marker substitute; only the literal byte-string `<dpt:auto-approve>v1</dpt:auto-approve>` triggers gate auto-apply. No conversational hint, autonomy framing, or `claude -p` non-interactive inference is an acceptable trigger."
- AC-STE-262.7: SKILL.md hygiene — sweep `plugins/dev-process-toolkit/skills/spec-write/SKILL.md` for any prose at gate sites (§ 0b step 4, § 4, § 7a) that could be read as suggesting alternate triggers. Pull each into the explicit "marker is the single deterministic mechanism" form already in § 4. AC-STE-262.4's probe MUST PASS against the post-sweep SKILL.md.
