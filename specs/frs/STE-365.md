# STE-365: smoke-test Phase 2 -p-context probe + run_in_background prohibition + drift meta-test {#STE-365}

## Requirement

The `/smoke-test` Phase 2 driver must not fall into the **fire-and-exit** failure (finding F3, 2026-07-04 `/conformance-loop` run): it spawned the `/setup` grandchild via the Bash tool's `run_in_background` task, then ended its turn "waiting for the completion notification." Under `claude -p` a session cannot resume on a background-task notification, so the parent exited rc=0, `/setup` was torn down mid-run (0 stream-json `result` events), and the canonical chain never advanced past Phase 2 step 1. Both legs did it verbatim, and both findings files asserted *"interactive session resumes on background-task completion"* — the driver **misidentified its own `-p` context as interactive**.

The load-bearing observation: the detached-spawn + bounded-poll pattern (SKILL.md lines 353, 401–442) and the STE-357 anti-yield prose ("Live-pidfile session rule", the "foreground-sleep hint is NOT license to background the wait" red-flag, the "Final-message self-check") were **all already present** and did **not** prevent the failure. Adding more of the same prose is not the fix. This FR attacks the observed root cause — context misidentification — with a **runtime probe** the driver executes (so it cannot believe it is interactive when it is not), a **co-located hard prohibition** at the exact spawn site, and a **byte-checkable drift meta-test** so the prohibition + detached snippet can never silently drift out of the skill.

Scope is the maintainer-only project-local skills `.claude/skills/smoke-test/SKILL.md` and `.claude/skills/conformance-loop/SKILL.md` (same surface as M95/M96). No plugin-shipped skill changes.

## Acceptance Criteria

- AC-STE-365.1: **Phase-2-entry context-detection probe.** `/smoke-test` Phase 2 opens (before the first grandchild spawn) with a deterministic context probe — `[ -t 0 ]` (or an equivalent stdin-tty test) — that prints a single byte-checkable banner line prefixed `SMOKE-CTX:`. Interactive branch prints `SMOKE-CTX: interactive tty`. Headless branch prints `SMOKE-CTX: headless (claude -p)` followed by an explicit statement that background-task notifications will **not** arrive and the bounded `kill -0` poll is the only sanctioned wait. The driver runs this probe and reads its output before any grandchild spawn. {#AC-STE-365.1}
- AC-STE-365.2: **Co-located run_in_background prohibition at the spawn site.** A hard "⛔ FORBIDDEN" callout is placed **at the exact Phase 2 grandchild-spawn site** (the `#### Phase 2 child-spawn discipline` / grandchild-spawn snippet region), not only in the separate `#### Grandchild spawn lifecycle` section. It forbids, by name, the Bash tool's `run_in_background` parameter, the `Monitor` tool, and ending the turn to await a completion notification — stating that under `claude -p` the notification never arrives (F3, 2026-07-04) and the bounded poll-until-exit loop is the only sanctioned wait. {#AC-STE-365.2}
- AC-STE-365.3: **Drift meta-test (smoke-test).** A deterministic meta-test in the DPT `bun test` suite fails (behaves as an error-severity gate) if `.claude/skills/smoke-test/SKILL.md` lacks any of: the `SMOKE-CTX:` context probe with its headless-branch warning, the co-located FORBIDDEN callout at the spawn site, or the detached `{ …; } &` + pidfile + bounded `kill -0` poll snippet (the last is a guard so the fix cannot accidentally delete the pattern it is protecting). {#AC-STE-365.3}
- AC-STE-365.4: **Parity + coverage for conformance-loop.** The same co-located FORBIDDEN-`run_in_background` prohibition is added to `/conformance-loop`'s Phase A leg-spawn site, and the drift meta-test asserts its presence in `.claude/skills/conformance-loop/SKILL.md` as well (both skills covered by one meta-test). {#AC-STE-365.4}
- AC-STE-365.5: **Runtime validation (deferred `[~]`).** The next live `/conformance-loop` run shows both `/smoke-test` legs running the full grandchild chain (setup → spec-write → implement → gate-check → spec-review → simplify) to completion — each grandchild log fresh + result-bearing — with no fire-and-exit. Validated on the next live run, not at implement time (no deterministic harness for a full multi-hour `-p` chain in CI). {#AC-STE-365.5}

## Technical Design

**Scope: prose + one meta-test.** Both target files are prose playbooks an LLM driver follows; the fix adds a runtime probe the driver *executes* plus a static drift guard.

**Files touched:**
- `.claude/skills/smoke-test/SKILL.md` — (a) insert the `SMOKE-CTX:` context probe at the top of Phase 2, before the first spawn; (b) add the co-located ⛔ FORBIDDEN callout inside `#### Phase 2 child-spawn discipline` immediately adjacent to the spawn snippet.
- `.claude/skills/conformance-loop/SKILL.md` — add the same ⛔ FORBIDDEN callout at the Phase A spawn-snippet region (the `{ claude -p … } &` brace-group block). `/conformance-loop`'s own parent orchestration already polls correctly, so no probe change there — the callout is parity + drift insurance for the leg-spawn instructions.
- `plugins/dev-process-toolkit/tests/m98-ste-365-fire-and-exit-guard.test.ts` (new) — reads both project-local SKILL.md files (repo-root-relative) and asserts the required byte-strings are present.

**Context probe (illustrative):**
```bash
if [ -t 0 ]; then
  echo "SMOKE-CTX: interactive tty"
else
  echo "SMOKE-CTX: headless (claude -p) — background-task notifications will NOT arrive; the ONLY sanctioned wait is the bounded kill-0 poll. Do NOT run_in_background, do NOT Monitor, do NOT yield the turn to await a grandchild."
fi
```

**Co-located FORBIDDEN callout (illustrative):**
> ⛔ **FORBIDDEN at this spawn site.** Do NOT await a grandchild with the Bash tool's `run_in_background` parameter, the `Monitor` tool, or by ending the turn "waiting for the completion notification" — under `claude -p` the notification never arrives (F3, 2026-07-04 conformance run: both legs fire-and-exited here). The ONLY sanctioned wait is the bounded `kill -0` poll-until-exit loop below.

**Why a probe + co-location beats more prose.** The failure was the driver *believing* it was interactive; a probe it runs turns that belief into an observed fact at the decision point. Co-locating the prohibition with the spawn snippet removes the "read the lifecycle section 80 lines away" gap. The meta-test freezes both against future drift. This is deliberately the surgical option (2026-07-04 design decision) — it still trusts the driver, but removes the specific rationalization that caused the failure. The "externalize spawn+poll into a checked-in script" option was considered and deferred as a larger refactor.
