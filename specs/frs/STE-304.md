# STE-304: Provider tolerance wrapper for non-key statuses (ask-at-runtime) {#STE-304}

## Requirement

STE-302's `statusToRole(config, status)` returns a three-way value: a mapped role, `null` (status is in the project's vocabulary but unmapped — a known-non-key state like `In QA`), or the `"unknown"` sentinel (status is not in the project's vocabulary at all — drift since the last `/setup` resync). STE-303 writes the config that powers those returns. This FR consumes them: a single tolerance wrapper at the Provider boundary that every read/write call site routes through, detecting non-key encounters and surfacing an `AskUserQuestion`-driven prompt (force / skip / cancel) rather than failing with the strict-equality assertion shape STE-87 and STE-54 ship today.

Single-wrapper-at-boundary chosen over per-skill branching because the surface is unbounded — every existing and future skill that touches tracker state would otherwise grow its own branching. The wrapper is the single source of truth; consumers (existing probes, /implement Phase 1 / Phase 4, future /pr) get tolerance for free. FR4 relaxes STE-87 + STE-54 to defer to the wrapper.

No marker-driven auto-apply for the force/skip/cancel choice — there is no safe default. The wrapper runs interactively or refuses with `RequiresInputRefusedError` (NFR-10 canonical shape) under non-tty stdin without the operator on the line. This matches STE-270's contract that marker auto-apply applies only to gates with documented safe defaults.

## Acceptance Criteria

- AC-STE-304.1: New module `plugins/dev-process-toolkit/adapters/_shared/src/tracker_tolerance.ts` exports `withTolerance(provider, specsDir)` — a thin wrapper returning a Provider-compatible object. Every status-touching method (`getTicketStatus`, `transitionTicket`, equivalent) reads `statusToRole` from STE-302's loader for both observed-current-status (reads) and target-status-after-transition (writes). Wrapper composes with `LocalProvider` and `TrackerProvider` identically.
- AC-STE-304.2: Three-way branching per `statusToRole` return: (a) mapped role matches the caller's expected role → pass-through (no prompt, no extra MCP call), (b) `null` (known-non-key) → prompt operator (force / skip / cancel) naming the observed status + the expected role + the project's known status list as context, (c) `"unknown"` (not in `statuses:`) → prompt operator (force / skip / cancel) naming the observed status + a discovery hint (`re-run /setup to resync the project's status list`).
- AC-STE-304.3: `AskUserQuestion` shape: 3 closed-form options + the always-on `Other` fallback. Option labels and descriptions: `force` ("treat as if it matched the expected role — caller proceeds with planned operation"); `skip` ("leave the task alone, continue calling skill with the rest of its work"); `cancel` ("halt calling skill with NFR-10 refusal — no state mutation"). Routing on response: `force` returns the expected role to the caller; `skip` returns a `Skipped` sentinel value the caller branches on; `cancel` throws `TrackerToleranceCancelledError` (NFR-10 canonical shape).
- AC-STE-304.4: Marker mode (`<dpt:auto-approve>v1</dpt:auto-approve>` present in prompt body) does NOT auto-apply force/skip/cancel — there is no safe default. Under non-tty stdin: wrapper throws `RequiresInputRefusedError` (NFR-10 canonical shape) naming the gate site (`tracker_tolerance_prompt`) and the observed status. Operator must re-invoke interactively to resolve.
- AC-STE-304.5: `mode: none` vacuous — `LocalProvider` callers skip the wrapper entirely (no `statusToRole` read, no prompt, no capability row). `withTolerance(LocalProvider, specsDir)` returns the wrapped provider unchanged. Probes guarded by the `local-no-tracker` sentinel (STE-54.5, STE-87.1, STE-101.5) preserve their carve-out unchanged.
- AC-STE-304.6: Adapter-config-absent fallback: when `readTrackerConfig(specsDir)` returns `null` (project hasn't run /setup tracker-config write yet), wrapper falls back to per-adapter `status_mapping` frontmatter (the STE-302.5 fallback path). Strict-equality behavior preserved on those projects — wrapper is a no-op until the project opts in by writing `specs/tracker-config.yaml`.
- AC-STE-304.7: Capability rows emitted to closing summary (literal-token contract per STE-238 AC.3): `tracker_status_forced` (operator chose force), `tracker_status_skipped` (operator chose skip), `tracker_status_cancelled` (operator chose cancel), `tracker_status_unknown_encountered` (the `"unknown"` sentinel fired — discovery hint surfaced), `tracker_tolerance_refused_non_tty` (non-tty stdin refusal). Rows propagate through the calling skill's closing summary.
- AC-STE-304.8: New tests at `plugins/dev-process-toolkit/tests/tracker-tolerance-wrapper.test.ts` covering: pass-through happy path (mapped role matches expected); known-non-key (`null`) → prompts; unknown (`"unknown"`) → prompts with discovery hint; force routing; skip routing returns `Skipped` sentinel; cancel routing throws `TrackerToleranceCancelledError`; non-tty stdin without marker → `RequiresInputRefusedError`; marker present + non-tty → still `RequiresInputRefusedError` (no auto-apply); `mode: none` vacuous (zero `statusToRole` calls observed); config-absent fallback (no wrapper effect).
- AC-STE-304.9: `/spec-write` § 7 static capability map gains the 5 new rows from AC.7 with plain-language renderings: `tracker_status_forced` ("operator chose force on non-key status — calling skill proceeded as if the observed status matched the expected role"); `tracker_status_skipped` ("operator chose skip — calling skill continued with remaining work, the affected task was left untouched"); `tracker_status_cancelled` ("operator chose cancel — calling skill halted with no state mutation"); `tracker_status_unknown_encountered` ("observed status not in project's known list — operator should re-run /setup to resync"); `tracker_tolerance_refused_non_tty` ("non-key status encountered under non-tty stdin without operator on the line — re-invoke interactively to resolve").
- AC-STE-304.10: New `/gate-check` probe `tracker_tolerance_wrapper_present` (severity: error) byte-checks that `adapters/_shared/src/tracker_tolerance.ts` exists AND exports `withTolerance`. Vacuous when `adapters/_shared/src/` is absent (non-toolkit projects). Defense-in-depth — if a future refactor accidentally drops the wrapper, FR4's relaxed probes would silently lose tolerance.

## Technical Design

### Architecture

```
existing call sites (gate-check probes, /implement claim/release, future /pr)
   │
   ▼
withTolerance(provider, specsDir)  ← single boundary wrapper
   │
   ├─► readTrackerConfig(specsDir)  (STE-302 loader)
   ├─► statusToRole(config, observed_status)  (three-way return)
   │     ├─► mapped role == expected → pass-through
   │     ├─► null (known-non-key) → AskUserQuestion (force/skip/cancel)
   │     └─► "unknown" → AskUserQuestion (force/skip/cancel + discovery hint)
   │
   └─► underlying Provider (LocalProvider | TrackerProvider)
```

### Return shape

The wrapper preserves Provider's existing return contract on the pass-through and force paths. The skip path returns a `Skipped` sentinel (`type Skipped = { kind: "skipped"; reason: string }`) that callers branch on — gate-check probes count it as `WARN`, /implement claim/release records it as `tracker_status_skipped` in the closing summary and continues.

### Prompt UX

Per-encounter prompt (NOT batched at end-of-skill). Rationale: operator needs immediate context (which task, which expected role, why the prompt fired); batching defers diagnosis and complicates the cancel path (does cancel apply to all batched encounters or just one?). If a skill encounters 5 non-key statuses, the operator sees 5 prompts. /gate-check probes that touch many FRs should expect this and design their UX accordingly (e.g., probe loop continues after individual cancels rather than halting the entire gate).

### Non-tty refusal

Under `process.stdin.isTTY === false`, the wrapper throws `RequiresInputRefusedError` (NFR-10 canonical shape) on the first non-key encounter. No marker auto-apply — STE-270's contract is that marker only relaxes gates with documented safe defaults, and force/skip/cancel has none (force can clobber QA's intent, skip drifts tracker from code state, cancel halts the run). Operator must re-invoke interactively to resolve.

### Capability boundary

Wrapper is composed at provider-resolution time (`buildProvider(...)` returns `withTolerance(provider, specsDir)` for tracker-mode resolutions; `LocalProvider` is returned bare). Callers that need the raw provider (rare — primarily test fixtures) call a new `rawProvider()` accessor.

## Notes

- **Why per-encounter prompts (not batched).** Operator needs immediate diagnostic context. Batching at end-of-run complicates the cancel path semantics and defers the operator's mental model of "which task is this about." Skills that touch many FRs accept the prompt count as a cost of doing business — and in practice, non-key encounters should be rare (a project whose tracker-config.yaml is current sees them only when QA manually moves things).
- **No marker auto-apply for force/skip/cancel.** No safe default exists — each branch has real cost. `force` can clobber QA's deliberate transition; `skip` drifts tracker from code state; `cancel` halts the run. Non-tty + no operator → refuse loudly (NFR-10), don't silently pick. Aligns with STE-270's marker-scope rule.
- **Wrapper composition timing.** Per-skill wiring lives at provider-resolution. Existing `buildProvider(...)` (or equivalent) call sites get a one-line edit: tracker-mode returns `withTolerance(provider, specsDir)`; mode-none returns bare `LocalProvider`. Localized blast radius.
- **Discovery hint on `"unknown"`.** When the wrapper sees a status not in `statuses:`, the project's tracker-config is stale (someone added a workflow state since the last /setup). Hint text: `re-run /setup to resync the project's status list`. Doesn't auto-trigger the resync — operator decides.
- **FR4 dependency.** This FR makes STE-87 + STE-54 *able* to be relaxed; FR4 actually relaxes them. Until FR4 ships, the strict probes still fire even with the wrapper installed (they don't yet route through it). Ship in order.
