# STE-340: Self-heal stale .dart_tool — retry dart pub get before dart-analyzer fallback {#STE-340}

## Requirement

`extractViaDartAnalyzer` (`adapters/_shared/src/signature_extractor.ts`) runs the bundled Dart helper (`adapters/_shared/dart/`) to extract public-API signatures for the `dart-analyzer` docs strategy (STE-103). It ran `dart pub get` **only** when the helper's `.dart_tool` package-config directory was **absent**; any non-zero `dart run extract_signatures.dart` exit fell straight through to `regex-fallback`.

A **stale** `.dart_tool` — its pinned `analyzer` version evicted from `~/.pub-cache` after an SDK/cache rotation — makes `dart run` fail to resolve `package:analyzer/...` and exit non-zero, so the extractor silently degraded every Dart project to `regex-fallback` (manual-review banner, zero structured signatures) even though `dart` is present on PATH and a one-shot `dart pub get` would repair the package config. Hit live on the maintainer's machine during a `bun test` run: the helper's `.dart_tool` pinned `analyzer-7.7.1` while `~/.pub-cache` held only `analyzer-9.x`/`10.x`, so the eight STE-103 happy-path tests failed with `strategy: regex-fallback`. `dart pub get` in the helper dir restored `analyzer-7.7.1` (per the tracked `pubspec.lock`) and the suite went green — exactly the repair this FR automates.

This FR makes the dart-analyzer path **self-heal**: on a non-zero `dart run` exit it refreshes deps once (`dart pub get`) and retries the extraction before degrading to `regex-fallback`.

## Acceptance Criteria

- AC-STE-340.1: **Single retry on stale config.** When the helper's `.dart_tool` is present but the first `dart run extract_signatures.dart <projectRoot>` exits non-zero, `extractViaDartAnalyzer` runs `dart pub get` once in the helper dir and retries the extraction **exactly once** before returning. A successful retry yields `strategy: dart-analyzer` with the extracted modules; the retry is bounded — no loop, at most one refresh per call. {#AC-STE-340.1}
- AC-STE-340.2: **Retry only on run-exit failure.** The refresh+retry fires only when the run process itself exits non-zero (the `runFailed` discriminator). A run that exits zero but produces empty / invalid-JSON / non-array stdout is NOT retried — a `dart pub get` cannot fix malformed helper output — and short-circuits to its existing fallthrough reason (`empty stdout` / `invalid JSON` / `not a JSON array`) unchanged. {#AC-STE-340.2}
- AC-STE-340.3: **Refresh-failure preserves the original reason.** If the retry's `dart pub get` itself fails (non-zero exit), the original, more-specific `dart-analyzer exit <code>: <stderr>` reason from the first run is preserved and the path degrades to `regex-fallback` — the refresh never masks the root-cause error with a generic pub-get message. {#AC-STE-340.3}
- AC-STE-340.4: **First-use + fallthrough behavior unchanged.** The absent-`.dart_tool` first-resolve path (run `dart pub get`, fail-fast on its non-zero exit with the `dart pub get failed (exit N): …` reason) is byte-unchanged, and the five pre-existing AC-STE-103.2 fallthrough reasons (`dart not found on PATH`, `dart pub get failed`, `invalid JSON from dart-analyzer`, `dart-analyzer output is not a JSON array`, `dart-analyzer exit <code>`) retain their exact warning substrings. {#AC-STE-340.4}
- AC-STE-340.5: **Tests green.** A new test under the `extractSignatures dart-analyzer — AC-STE-103.2 fallthrough branches` suite drives a stateful bash stub (`dart run` exits non-zero until `dart pub get` writes a `.resolved` marker, after which `run` emits a valid module array) and asserts the retry produces `strategy: dart-analyzer` with no `fell through` warning and the expected module present. The five pre-existing fallthrough tests stay green. `bun test` (full gate from `plugins/dev-process-toolkit`) + `/dev-process-toolkit:gate-check` GREEN on this repo. {#AC-STE-340.5}

## Technical Design

**Files touched:**
- `adapters/_shared/src/signature_extractor.ts` — refactor `extractViaDartAnalyzer`; add `runDartPubGet` + `runDartExtractOnce` helpers; add the single retry.
- `adapters/_shared/src/signature_extractor_dart.test.ts` — new stale-`.dart_tool` retry test in the `AC-STE-103.2 fallthrough branches` describe block.

**Shape:**
```ts
// Resolve helper deps; reused by the first-use path and the retry.
function runDartPubGet(dartBin, helperDir): { ok: true } | { ok: false; reason: string }

// One `dart run` attempt. `runFailed` separates a non-zero process exit
// (retryable — likely a stale .dart_tool) from a parse/empty/not-array failure
// (the run exited 0; a pub-get retry would not help).
function runDartExtractOnce(dartBin, helperDir, projectRoot):
  DartOk | (DartFail & { runFailed: boolean })
```

**Control flow in `extractViaDartAnalyzer`:**
1. Resolve dart binary + helper dir (unchanged).
2. If `.dart_tool` absent → `runDartPubGet`; fail-fast on its error (unchanged first-use behavior).
3. `outcome = runDartExtractOnce(...)`.
4. If `!outcome.ok && outcome.runFailed` → `runDartPubGet` once; on its success, `outcome = runDartExtractOnce(...)` (the single retry). On pub-get failure, keep the original `outcome`.
5. Return `outcome` (ok → modules; fail → reason → caller emits the `fell through to regex-fallback` warning).

The `runFailed` discriminator is the load-bearing design choice: it scopes the retry to genuine process-exit failures (the stale-config class) and leaves the JSON-shape failure reasons — which `pub get` cannot repair — short-circuiting exactly as before.

## Testing

- New test `stale .dart_tool: dart run fails first, auto \`pub get\` retry succeeds → dart-analyzer` (in `signature_extractor_dart.test.ts`): a bash stub whose `run` arm exits 254 until the `pub` arm `touch`es `./.resolved` (cwd = helperDir via `spawnSync cwd`), then emits `[{"modulePath":"lib/foo.dart","exports":[]}]`. Helper dir pre-creates `.dart_tool` to force the run-then-retry path (not the absent-`.dart_tool` first-resolve path). Asserts `strategy === "dart-analyzer"`, no `fell through` warning, and the `foo.dart` module present. Removing the retry makes it RED (regex-fallback) — a true guard, not a tautology.
- The five pre-existing AC-STE-103.2 fallthrough tests (dart-not-found / pub-get-failed / invalid-JSON / not-array / non-zero-exit) stay green; the non-zero-exit stub now harmlessly receives one extra `pub get` + retry-run and still terminates on `dart-analyzer exit 5`.
- Real-`dart` happy-path + empty-lib suites (`describe.skipIf(!hasDart)`) unchanged — they exercise the already-resolved helper.

## Notes

- **Relationship to STE-103.** STE-103 introduced the dart-analyzer strategy + the bundled helper + the absent-`.dart_tool` first-resolve `pub get`. STE-340 closes the stale-`.dart_tool` gap that the first-resolve guard left open (it only ran `pub get` when `.dart_tool` was missing, never when present-but-stale).
- **`.dart_tool` is gitignored.** The helper's `.dart_tool/` is in `adapters/_shared/dart/.gitignore`, so it is regenerated per-machine and naturally drifts when the local pub-cache rotates analyzer versions — precisely the condition this FR repairs.
- **Bounded, best-effort.** At most one `pub get` + one retry per call; a still-failing retry degrades to regex-fallback (the same safe terminal state as before), so the change can only improve outcomes, never block extraction.
- **Consumer impact.** The silent-degrade-on-stale-config failure mode affected any Dart consumer project running `/docs`, not just this repo's fixtures; this repo is a TS stack, so its own `/docs --full` regen never exercises the dart path.

## Implementation notes

- `runDartPubGet` is shared between the first-use resolve and the retry; the prior inline pub-get block (only on absent `.dart_tool`) is now this helper, keeping the `dart pub get failed (exit N): …` reason byte-identical for AC-STE-340.4.
- The retry deliberately does NOT re-check `existsSync(.dart_tool)` before refreshing — a present-but-stale `.dart_tool` is exactly the case the `existsSync` guard misses, so the retry keys off the run-exit signal, not directory presence.
