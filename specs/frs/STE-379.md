# STE-379: Token Stats opt-in enforcement — gate capture hook + render sites + `token_stats_disabled` disposition {#STE-379}

## Requirement

**[[STE-378]]** ships the `## Token Stats` config surface and the `readTokenStatsConfig` parser but changes no behavior — token capture and render are still unconditional. This FR (FR-B of M102) makes the feature **fully off when `enabled: false`**: the bundled capture hook stops writing the ledger, every render site stops emitting `## Token Stats` blocks, and `/spec-write`'s closing summary reports the disabled state with an honest, byte-checkable disposition token instead of falsely claiming a render happened.

"Fully off" (not merely "hide the tables") was the operator's explicit choice in the 2026-07-10 `/brainstorm`: because the capture hook is **plugin-bundled and auto-discovered** (M74) — it fires for every project with the plugin, with no per-consumer `settings.json` entry to omit — the only way to truly disable capture is for the hook body itself to read the flag and no-op. The gate therefore lives at four points: the hook body (capture) and the three render sites (`/spec-write`, `/implement` Phase 4, `/ship-milestone`), reconciled with the `token_stats_rendered` capability-key contract.

## Acceptance Criteria

- AC-STE-379.1: **Capture-hook gate (fully off).** The bundled hook entrypoint `templates/hooks/_lib/hooks/session-token-ledger.ts` calls `readTokenStatsConfig(projectRoot)` before writing; when `enabled === false` it writes **nothing** to `.dev-process/token-ledger.jsonl` and exits `0`. The gate is **fail-off AND fail-open**: a missing/unreadable `CLAUDE.md`, or a thrown `MalformedTokenStatsConfigError`, is caught and treated as OFF with exit `0` — the error never propagates and never gates session-end (preserves STE-344.4's fail-open posture). When `enabled === true`, capture behaves exactly as it does today. {#AC-STE-379.1}
- AC-STE-379.2: **Render-site gates.** All three `## Token Stats` render sites skip their render when `readTokenStatsConfig(projectRoot).enabled === false`: `/spec-write` § 0b Step 7 (FR-file block), `/implement` Phase 4 Milestone Archival "Token Stats re-render", and `/ship-milestone` § 7 milestone rollup. When enabled, each behaves exactly as today (including the existing vacuous-when-ledger-empty path). {#AC-STE-379.2}
- AC-STE-379.3: **`token_stats_disabled` disposition (2-token legal set).** `/spec-write`'s Step-7 closing summary emits **exactly one** literal, backticked token per run: `token_stats_rendered` when `enabled === true` (block rendered, or the existing vacuous-empty path), `token_stats_disabled` when `enabled === false`. The new key `token_stats_disabled` is registered in `CANONICAL_CAPABILITY_KEYS` (`adapters/_shared/src/closing_summary_capability_keys.ts`) with owner `spec-write` and a plain-language row in the `skills/spec-write/SKILL.md` § 7 static map. Narrative prose is insufficient — the literal token is the byte-checkable signal probe #44 greps. {#AC-STE-379.3}
- AC-STE-379.4: **Count-pin + probe reconciliation.** Registering `token_stats_disabled` increases the canonical capability-key count by **exactly one** (the new key, nothing else). Every count-pinning site is updated to `current + 1` and admits the 2-token set: the directive-count registry pinned in `tests/m84-ste-320-code-reviewer-scope-registry.test.ts`, the `closing_summary_capability_keys` probe #44 orphan/round-trip, and the `tests/token-stats-spec-write-contract.test.ts` meta-test (which learns the `token_stats_rendered` XOR `token_stats_disabled` contract). The FR states the **delta**, not an absolute count — the implementer reads the live count from the code, since the canonical set has grown since M92. {#AC-STE-379.4}
- AC-STE-379.5: **Tests.** Hook tests (`tests/token-ledger-hook.test.ts` / `tests/token-ledger-noninterference.test.ts`) assert **no ledger write when `enabled: false`**, a normal write when `enabled: true`, and fail-off (no write, exit 0) on a malformed `CLAUDE.md`. The M92 render + commit-boundary meta-tests (`token-stats-convention.test.ts`, `token-stats-commit-boundary-contract.test.ts`) gain skip-when-disabled clauses. A disposition test asserts `/spec-write` selects `token_stats_disabled` vs `token_stats_rendered` by the flag. {#AC-STE-379.5}

## Technical Design

**Scope: enforcement.** Reads `readTokenStatsConfig` (from **[[STE-378]]**); gates capture + render; adds one capability key. No new config surface.

**Files touched:**
- `templates/hooks/_lib/hooks/session-token-ledger.ts` — before `writeSessionRows(...)`, resolve the project root (the hook already derives it via `payload.cwd` for `ledgerPath`) and call `readTokenStatsConfig(projectRoot)` inside the existing fail-open `try`. `enabled === false` ⇒ return without writing, exit 0. A thrown `MalformedTokenStatsConfigError` (or any read error) is swallowed by the existing `catch` and treated as OFF — the hook's contract is already "any error ⇒ exit 0, no write," so this composes cleanly.
- `skills/spec-write/SKILL.md` § 0b Step 7 — wrap the `claimRowsForFR` + `upsertTokenStatsBlock` render in a `readTokenStatsConfig().enabled` guard; emit `token_stats_rendered` when enabled (and rendered/vacuous) XOR `token_stats_disabled` when disabled.
- `skills/implement/SKILL.md` § Milestone Archival "Token Stats re-render" — same `enabled` guard; skip the re-render + block staging when disabled.
- `skills/ship-milestone/SKILL.md` § 7 rollup — same `enabled` guard; skip `renderMilestoneRollup` / `upsertMilestoneRollup` when disabled.
- `adapters/_shared/src/closing_summary_capability_keys.ts` — register `token_stats_disabled` in `CANONICAL_CAPABILITY_KEYS` + owner-map (owner `spec-write`).
- `skills/spec-write/SKILL.md` § 7 static map — add the `token_stats_disabled` plain-language row; extend the existing `token_stats_rendered` row's contract to the 2-token XOR.
- Count-pin updates: `tests/m84-ste-320-code-reviewer-scope-registry.test.ts`, `tests/token-stats-spec-write-contract.test.ts`, plus whatever `closing_summary_capability_keys` round-trip test pins the count — each `current → current + 1`.

**Fail-off + fail-open is the load-bearing invariant.** The hook runs headless at session-end with a 5s timeout; it must never throw. `readTokenStatsConfig` *can* throw (`MalformedTokenStatsConfigError` on a bad value), so the gate treats "cannot determine enabled" identically to "disabled": no write, exit 0. This is stricter than the render sites, which may surface an NFR-10 refusal for a malformed `CLAUDE.md` because they run interactively — but the hook stays silent.

**Why a new token, not vacuous-`rendered`.** The existing empty-ledger path already emits `token_stats_rendered` vacuously, which is honest when the feature is *on but has no data yet*. When the feature is *disabled*, claiming "rendered" is dishonest and pollutes the audit trail — hence the distinct `token_stats_disabled`, matching the repo's deps-research disposition-set idiom (explicit legal tokens including skip states) and M100's enforcement-parity direction.

## Notes

- **Origin.** 2026-07-10 `/brainstorm` → `/spec-write`, same session as **[[STE-378]]**.
- **Gate-depth decision.** The operator chose "fully off (no capture)" over "render-only (still records silently)" and "capture-only (render goes vacuous)". Render-only was rejected because the ledger would keep accruing on every session even when the operator wants the feature off; capture-only was rejected because it would leave `/spec-write` emitting a vacuous `token_stats_rendered` — dishonest. Fully-off + the `token_stats_disabled` token is the honest, complete design.
- **Bundled-hook constraint.** The capture hook is plugin-bundled and auto-discovered (M74) — there is no per-consumer `settings.json` entry a flag could omit. The only true off-switch is the hook body reading the flag; hence AC-STE-379.1 gates inside `session-token-ledger.ts`.
- **Count stated as a delta.** M92-era recon put the canonical capability-key count at ~24; M100 moved it further (calibration-drift lesson: orchestrator-owned snapshots drifted keys 24→30). Pinning an absolute number in the spec would ship stale, so AC-STE-379.4 states `current + 1` and lets the implementer read the live count.
- **Depends on** **[[STE-378]]** (`readTokenStatsConfig`). Build order: STE-378 → STE-379.
