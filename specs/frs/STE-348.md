# STE-348: /setup check-skill generator + /implement scaffold-if-missing + /visual-check fallback {#STE-348}

## Requirement

STE-347 gives a project's check skill a place to plug in, but authoring one from scratch is friction — the "encourage" goal fails if every user must hand-roll a `glacy-drive`. This FR (FR-B of M93) makes a check skill **trivial to obtain**: `/setup` offers to scaffold a stack-aware "drive the project" stub at bootstrap, `/implement` offers the same scaffold when STE-347's discovery finds nothing, and — for a small web app — either can fall back to the toolkit's shipped generic `/visual-check` with zero authoring.

## Acceptance Criteria

- AC-STE-348.1: **`/setup` seed step.** `/setup` gains a new **opt-in** step (after CLAUDE.md generation) that offers to scaffold a project-local check skill at `.claude/skills/<slug>-drive/SKILL.md` with `disable-model-invocation: true`, stack-aware from the detected stack: Flutter → dart/Flutter-MCP drive stub (modeled on `glacy-drive`); web/JS/TS → Chrome-MCP or `/visual-check` stub; Python/API → HTTP e2e stub (modeled on `glacy-progress-e2e`); other/unknown → a documented generic TODO stub. On acceptance it writes `verify_skill: <slug>-drive` into the `## Verification` block (STE-347); the step is skippable and no-ops on decline. {#AC-STE-348.1}
- AC-STE-348.2: **Stub content is runnable-shaped.** The generated `SKILL.md` carries frontmatter (`name`, `description`, `disable-model-invocation: true`, stack-appropriate `allowed-tools`), a `## What this checks` section, a `## How to run` section with the stack-appropriate MCP/CLI steps, clearly-marked `TODO` placeholders where the user fills project specifics, and a link to the STE-349 authoring guide. It is a runnable scaffold, not an empty file. {#AC-STE-348.2}
- AC-STE-348.3: **`/implement` scaffold-if-missing.** When STE-347's discovery resolves no check skill (none declared, none adopted), `/implement` offers to scaffold one via the same generator, **or** — for a small web project — to use the generic `/dev-process-toolkit:visual-check` as the check. On decline, `/implement` proceeds to the step-14 report with a "no verification configured" note. It never writes a skill without the offer. {#AC-STE-348.3}
- AC-STE-348.4: **`/visual-check` fallback wiring.** `verify_skill: visual-check` is a valid `## Verification` value; when set, STE-347's Phase 4b″ runs `/dev-process-toolkit:visual-check` (its existing MCP-probe + manual-checklist fallback per HG95TV) instead of a project-local skill. This is documented as the zero-authoring path for web UIs. {#AC-STE-348.4}
- AC-STE-348.5: **Generator helper + idempotency.** The scaffold is produced by a shared helper used by both `/setup` and `/implement` that refuses to overwrite an existing `.claude/skills/<name>/SKILL.md` — it offers an alternate slug or aborts — so re-running never clobbers a hand-edited skill. {#AC-STE-348.5}

## Technical Design

**Files touched:**
- `adapters/_shared/src/scaffold_check_skill.ts` (new) — `scaffoldCheckSkill({ projectRoot, stack, slug }) → { path, wrote: boolean }`; maps stack → stub content from a template; collision-safe (refuses overwrite, suggests a `-2` slug); returns the written path.
- `templates/check-skill/{flutter,web,python,generic}.SKILL.md.template` (new) — the four stack stubs (`disable-model-invocation: true`, stack `allowed-tools`, `## What this checks`, `## How to run`, TODOs, guide link).
- `skills/setup/SKILL.md` — new opt-in seed step (after Step 5 CLAUDE.md gen, near Step 8 specs), writing `verify_skill` on accept.
- `skills/implement/SKILL.md` — the no-check branch of Phase 4b″ (STE-347) gains the scaffold-offer / `/visual-check`-fallback offer.
- `skills/visual-check/SKILL.md` — a one-line note that it is a valid `verify_skill` value.
- `skills/spec-write/SKILL.md` § 7 — new capability keys `verify_skill_scaffolded` / `verify_skill_scaffold_declined` (reusing STE-347's `verify_skill_none_declared`).

**Stack detection** reuses `/setup`'s existing stack detector (Step 1) — no new detection logic. The generator is pure content templating; the MCP wiring in each stub is TODO-marked (the toolkit can't know the project's launch/login specifics — that's the user's to fill, exactly as `glacy-drive` does).

## Testing

- `scaffold_check_skill.test.ts` — each stack → expected stub shape (`disable-model-invocation: true` present, stack `allowed-tools`, required sections), collision refusal + alternate slug.
- Meta-tests — `templates/check-skill/*.SKILL.md.template` exist, are non-empty, and carry `disable-model-invocation: true`; `skills/setup/SKILL.md` carries the seed step; `skills/implement/SKILL.md` carries the scaffold-offer + `/visual-check` fallback; new capability keys in the § 7 map.
- Gate: `bun test` + `/dev-process-toolkit:gate-check` GREEN.

## Notes

- **Depends on STE-347** (`## Verification` convention + Phase 4b″ no-check branch).
- **Why TODO-marked stubs, not turnkey.** A generic launch/login/drive path can't be inferred; `glacy-drive` needed a bespoke `test_driver/glacy_local.dart` entrypoint + CDP login. The stub encodes the *shape* (frontmatter, MCP tools, run steps) and marks the project-specific holes — the mechanical majority, leaving the bespoke minority to the user.
- **`/visual-check` is the zero-authoring escape hatch** for simple web UIs, so "encourage a check skill" never blocks a small project.
