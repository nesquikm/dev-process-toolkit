# STE-211: Linear AC-token round-trip preservation: backtick-wrap on emit + strip on import {#STE-211}

## Requirement

The Linear MCP server post-processes issue descriptions on save: bare `STE-NNN` tokens get auto-linked into `<issue id="...">STE-NNN</issue>` XML wrappers. The toolkit's `AC-<TKR>-NN.N` prefix shape (e.g., `AC-STE-203.1`) contains a literal `STE-203` substring; the auto-linker matches it even though the surrounding `AC-` prefix and `.N` suffix make it unambiguously not a bare issue reference. Round-trip identity breaks; downstream parsers misclassify.

**Findings bundled:** F7 from `/tmp/dpt-findings-2026-05-04-flutter-app.md` (Flutter calculator smoke, 2026-05-04).

**Symptoms observed in `dpt_test_lib2`:**

1. After `/spec-write` pushed an FR description containing `AC-STE-203.1` etc., the Linear-side stored description showed: `AC-<issue id="...">STE-203</issue>.1`.
2. In the Linear web UI, the AC bullet renders as `AC- STE-203 .1` with the issue badge in the middle — looks like garbled text.
3. Any downstream consumer that imports the description back into `specs/frs/<id>.md` via `importFromTracker` and then runs `scanGuessedTrackerIdLiterals` may misclassify the corrupted token as drift; at minimum the round-trip is no longer byte-identical.
4. The toolkit's idempotency probes for AC-list diff/resolve see `"remote modified"` on every push because the auto-injected XML tags weren't in the local copy.

## Acceptance Criteria

- AC-STE-211.1: When emitting an AC line to a Linear description, the toolkit wraps the AC prefix in backticks: ``AC-`STE-NNN`.N`` (the prefix `STE-NNN` is fenced; the surrounding `AC-` and `.N` remain plain). The local FR file (`specs/frs/<id>.md`) stays plain — backticks are a Linear-side output transformation only. {#AC-STE-211.1}
- AC-STE-211.2: The transformation is Linear-adapter-scoped. Jira and custom adapters are unaffected: their adapters do not auto-link `STE-NNN`-shaped tokens, so the backtick wrap would be cosmetic noise. The transformation lives in `adapters/linear/src/format_description.ts` (or equivalent), not in the shared push pipeline. {#AC-STE-211.2}
- AC-STE-211.3: `importFromTracker` round-trip — when reading a Linear description back into a local FR file, the helper strips backticks that match the AC-prefix shape: ``AC-`<TKR>-NN`.N`` → `AC-<TKR>-NN.N`. This way the local file converges on the canonical plain-text shape regardless of whether the Linear-side description has backticks (newly-pushed FRs) or `<issue id>` tags (legacy pre-fix descriptions, see AC.4). {#AC-STE-211.3}
- AC-STE-211.4: `importFromTracker` also strips Linear's auto-link XML when matched against the AC-prefix shape: `AC-<issue id="...">STE-NNN</issue>.N` → `AC-STE-NNN.N`. This handles legacy descriptions that were pushed pre-fix and got mangled. The strip is targeted: only strips `<issue id="...">...</issue>` wrappers when the inner content matches `STE-\d+` AND the wrapper is preceded by `AC-` and followed by `.\d+`. Bare issue references (e.g., `Related: STE-205`) keep their `<issue id>` wrapper untouched. {#AC-STE-211.4}
- AC-STE-211.5: Idempotency probe (the existing AC-list diff/resolve loop) treats backtick-wrapped Linear ACs and plain local ACs as equal: the diff normalizes both sides to the plain shape before comparing. After this fix, pushing an unchanged local FR to a freshly-corrected Linear description produces zero diff (no spurious "remote modified" rows). {#AC-STE-211.5}
- AC-STE-211.6: Test fixture / smoke probe: push a Linear FR with multiple AC lines, fetch it back, and assert byte-identical round-trip after the strip pass. Linear-leg only — Jira's MCP doesn't auto-link; the probe is documented as Linear-leg-only by design. {#AC-STE-211.6}
- AC-STE-211.7: Backwards-compat: existing Linear descriptions that have already been mangled by the auto-linker (e.g., STE-197..STE-202 in the current run, plus STE-209/STE-210/STE-211 themselves which were emitted with the manual backtick workaround) converge on next push without operator intervention — the import strips auto-link XML on read, the local file becomes plain, the next push wraps in backticks, and the auto-linker no longer matches. No `--migrate` flag, no manual cleanup. {#AC-STE-211.7}

## Technical Design

**Why backticks work.** Linear's auto-linker is a markdown-aware processor that recognizes bare alphanumeric tokens matching `[A-Z]+-\d+`. Inline-code fences (backticks) are a documented escape: text inside backticks is treated as literal code and is not subject to auto-linking. Empirically confirmed in M54 brainstorm by the F7 workaround: wrapping `AC-STE-203.1` produces a clean Linear render.

**Format on emit (Linear-only).** New helper `formatLinearDescription(body: string): string` in `adapters/linear/src/format_description.ts`:

```typescript
function formatLinearDescription(body: string): string {
  // Wrap AC prefixes in inline-code fences to dodge Linear's auto-linker.
  return body.replace(
    /\bAC-([A-Z]+-\d+)(\.\d+)\b/g,
    'AC-`$1`$2'
  );
}
```

Called by the Linear adapter's `upsertTicketMetadata` immediately before the `description` field is set on `save_issue`. Idempotent: applying twice produces ``AC-`STE-NNN`.N`` (already-fenced; the regex doesn't match `AC-` followed by a backtick).

**Strip on import.** New helper `stripLinearACFences(body: string): string` in the same file:

```typescript
function stripLinearACFences(body: string): string {
  // (1) Strip backtick-wrapped AC prefixes back to plain.
  let s = body.replace(
    /\bAC-`([A-Z]+-\d+)`(\.\d+)\b/g,
    'AC-$1$2'
  );
  // (2) Strip Linear's <issue id="...">...</issue> auto-link wrappers
  //     ONLY when preceded by 'AC-' and followed by '.<digits>'.
  s = s.replace(
    /\bAC-<issue id="[^"]*">([A-Z]+-\d+)<\/issue>(\.\d+)\b/g,
    'AC-$1$2'
  );
  return s;
}
```

Called by `importFromTracker` for the Linear adapter only. The double-strip handles both newly-pushed (backticks) and legacy (XML) descriptions.

**Idempotency probe normalization.** The AC-list diff/resolve loop normalizes both sides via `stripLinearACFences` before the equality check. Newly-pushed FRs that are byte-identical to the local file produce zero diff.

**Round-trip property.** `formatLinearDescription` and `stripLinearACFences` are inverses on the AC-prefix shape: `strip(format(x)) === x` and `format(strip(x)) === format(x)` for any local-file body `x`. Tested as a property unit test.

**Auto-link false-positives outside AC prefixes.** The toolkit also writes prose like `STE-202 AC.4 is dropped` or `Refs: STE-201` — these are legitimate bare issue references and SHOULD be auto-linked by Linear (the `<issue id>` badge is a feature there, not a bug). The format helper only wraps AC prefixes; bare references in prose are unaffected, and Linear's auto-linker continues to work as designed for those. The strip helper only undoes wrappers when preceded by `AC-` and followed by `.\d+` — bare references stay wrapped through round-trips (no regression).

## Testing

- Unit test: `formatLinearDescription("- AC-STE-203.1: foo")` returns `` "- AC-`STE-203`.1: foo" ``. Idempotent: applying twice produces the same output.
- Unit test: `stripLinearACFences("- AC-`STE-203`.1: foo")` returns `"- AC-STE-203.1: foo"`. Same for `<issue id>` wrapper variant.
- Property test: for any local-file body containing AC prefixes, `strip(format(x)) === x`.
- Integration test: push a multi-AC Linear FR via the adapter, fetch it back via `mcp__linear__get_issue`, run `stripLinearACFences` on the response description, assert byte-identical to the original local body.
- Integration test (legacy compat): create a fixture Linear FR with the auto-linked XML shape pre-pushed, run `importFromTracker` → local body is plain text. Push it back → Linear-side description is backtick-wrapped (auto-linker no longer matches).
- Smoke probe (Linear leg only): the round-trip property is exercised against a real Linear fixture project per smoke run.

## Notes

- Workaround during M54 implementation: this `/spec-write` run pre-emptively backtick-wraps the AC prefixes in the descriptions of the three new FRs (STE-209, STE-210, and this FR). Once this FR ships, the Linear adapter does it automatically and the manual workaround can be retired.
- The auto-link bug is a Linear-MCP behavior, not a Linear web-app behavior. We're not asking the Linear team to change anything; the toolkit-side wrap is the canonical fix.
- A more aggressive alternative (asking Linear MCP to suppress auto-linking when the matched token is preceded by a non-whitespace character — `-`, `.`, etc.) was considered and rejected: it requires upstream cooperation, has unpredictable timeline, and the toolkit-side wrap is sufficient.
- F7 doesn't apply to Jira: Jira's MCP doesn't auto-link `STE-NNN`-shaped tokens (Jira keys are project-scoped, e.g., `DPT-12`, and the MCP doesn't synthesize `<issue>` wrappers around them in description bodies). The probe (AC.6) is Linear-leg-only by design.
- Capability: a tracker-mode `/spec-write` run that creates this FR surfaces `filename_policy_override` per the unconditional rule.
