## Requirement

Several plugin-side docs contain prose that names shipped milestones, completed migrations, or pre-M18 historical state in a way that does not parse for a reader without that history:

- `layout-reference.md` (post-rename): "v2-layout behavior across spec-touching skills (STE-26..STE-25)", "v2 is the only supported layout", "Pre-M18 archives keep the legacy `fr_<ULID>.md` shape until STE-61's one-time rewrite lands".
- Other docs in `plugins/dev-process-toolkit/docs/` containing similar shipped-state references — surface during the pass.

Rewrite into current-state framing. Preserve AC citations that explain *current* behavior — those are load-bearing for debugging.

## Acceptance Criteria

- AC-STE-116.1: For each active doc under `plugins/dev-process-toolkit/docs/` and `plugins/dev-process-toolkit/skills/**/SKILL.md`, prose that frames shipped milestones as in-flight or names version migrations as ongoing is rewritten to current-state. Examples: "v2 is the only supported layout" → drop the v2/v1 framing entirely (state the layout); "STE-26..STE-25" → drop the milestone-range citation (describe what the doc covers in plain terms). {#AC-STE-116.1}
- AC-STE-116.2: Preserved unchanged — AC citations that explain current behavior, e.g., "(AC-STE-60.1)", "M18 STE-60 AC-STE-60.3". These are pointers a debugger uses; removing them is destruction, not cleanup. {#AC-STE-116.2}
- AC-STE-116.3: Out-of-scope — `**/archive/**`, `CHANGELOG.md`, `specs/frs/archive/`, `specs/plan/archive/`. {#AC-STE-116.3}
- AC-STE-116.4: Gate is reviewer judgment, not grep. Reviewer reads each diff and confirms the prose reads natural to a fresh installer. `/gate-check` green. `bun test` green. {#AC-STE-116.4}
- AC-STE-116.5: **Double-check pass after rewrite.** Re-grep touched files for residual shipped-state markers (`v1\b`, `v2 layout`, `pre-M\d+`, `STE-\d+\.\.STE-\d+`, `until STE-\d+'s one-time`). Any hit is reviewed: either it's a load-bearing AC citation (preserved per AC-STE-116.2) or it's a missed rewrite (fix). The double-check is a sanity net, not a hard gate — judgment still rules. {#AC-STE-116.5}

## Technical Design

Editorial pass. No mechanical recipe — each touched paragraph is a judgment call. Conducted file-by-file; commit per file or per logical doc cluster.

**Heuristic for "is this prose stale?":**

- Names a milestone in past tense as if still in flight → rewrite.
- Names a migration as ongoing when it has shipped → rewrite.
- Cites an FR/AC that explains *why current code behaves a way* → preserve.

**Double-check sweep (AC-STE-116.5):**

```bash
rg -n 'v1\b|v2 layout|pre-M[0-9]+|STE-[0-9]+\.\.STE-[0-9]+|until STE-[0-9]+' \
  plugins/dev-process-toolkit/docs/ \
  plugins/dev-process-toolkit/skills/
```

## Testing

No new tests. Gate is `/gate-check` + reviewer + AC-STE-116.5 sweep.
