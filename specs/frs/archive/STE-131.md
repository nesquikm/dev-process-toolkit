# STE-131: README workflow diagram: two-tier spine vs secondary styling + fix /implement orchestration prose {#STE-131}

## Requirement

M34 (v1.36.0 "Compass") shipped a mermaid `flowchart TD` workflow diagram in `README.md` with four phase subgraphs treating all 14 user-invoked skills as visual peers. Two issues surfaced post-ship:

1. **Diagram doesn't communicate hierarchy.** First-time visitors see 14 peer nodes and have to infer which ones they actually invoke. The five spine skills (`/setup`, `/brainstorm`, `/spec-write`, `/implement`, `/ship-milestone`) carry the recommended path; the other nine (`/tdd`, `/gate-check`, `/debug`, `/visual-check`, `/simplify`, `/spec-review`, `/spec-archive`, `/docs`, `/pr`) are auxiliary — opt-in entry points or auto-invoked under the hood. The diagram should signal this.
2. **Prose paragraph beneath the diagram is factually wrong.** It claims `/implement` orchestrates `/tdd`, `/gate-check`, `/docs --quick`, and `/pr`. Reality (from `skills/implement/SKILL.md`): only `/docs --quick` is invoked as a skill (Phase 4b). TDD is performed inline in Phase 2 step 8 (write test → RED → code → GREEN), not via the `/tdd` skill. Gate commands run inline in Phase 2 step 11, not via the `/gate-check` skill. `/pr` is never invoked — the user opens the PR after `/implement`'s commit lands.

Restructure the diagram with two-tier classDef styling (spine skills get bold/stadium; secondary stay plain rectangles) and rewrite the prose to match what `/implement` actually does. Approved /brainstorm decision (2026-04-27): Approach 2 (phases kept, two-tier styling inside each subgraph), with `/ship-milestone` elevated to spine.

## Acceptance Criteria

- AC-STE-131.1: `README.md` `## Workflow` mermaid block declares two `classDef` rules — one named `spine` (stadium shape via syntax `node([label])` plus distinct visual weight: e.g., bold `stroke-width:3px` and a fill color) and one named `secondary` (plain rectangle plus muted: e.g., lighter `stroke:#999` and a paler fill). Exact colors chosen at implementation time but must visually distinguish the two tiers in both GitHub light and dark themes.
- AC-STE-131.2: The five spine skills `/setup`, `/brainstorm`, `/spec-write`, `/implement`, `/ship-milestone` are each declared with the stadium-shape syntax (`id(["/skill"])`) and assigned `class <id> spine` (or per-line `:::spine` shorthand). The remaining nine skills `/tdd`, `/gate-check`, `/debug`, `/visual-check`, `/simplify`, `/spec-review`, `/spec-archive`, `/docs`, `/pr` are each declared with plain-rectangle syntax (`id["/skill"]`) and assigned `class <id> secondary` (or per-line `:::secondary` shorthand).
- AC-STE-131.3: The four phase subgraphs (`Setup`, `Plan`, `Build`, `Ship`) and inter-phase arrows (`Setup --> Plan --> Build --> Ship`) are preserved from M34. Per-skill placement inside the subgraphs is preserved exactly (`/setup` in Setup; `/brainstorm`+`/spec-write` in Plan; `/implement`+`/tdd`+`/gate-check`+`/debug`+`/visual-check`+`/simplify`+`/spec-review` in Build; `/spec-archive`+`/docs`+`/pr`+`/ship-milestone` in Ship). Total 14 skill nodes, each in exactly one subgraph.
- AC-STE-131.4: The prose paragraph beneath the diagram is rewritten to factually match `skills/implement/SKILL.md`. The replacement text states (a) `/implement` performs TDD inline (write test → RED → code → GREEN) rather than invoking the `/tdd` skill, (b) `/implement` runs the gate commands inline (e.g., `bun test`) rather than invoking the `/gate-check` skill (which layers probes on top of those commands), (c) `/implement` does invoke `/docs --quick` for the Phase 4b doc fragment, and (d) `/implement` stops at the user-approval gate before commit and the user opens the PR via `/pr` separately. Length ≤4 sentences (matches M34's STE-130 § Acceptance Criteria limit).
- AC-STE-131.5: A second short paragraph below the orchestration prose explains the visual encoding in one sentence: spine skills (bold, stadium-shaped) are the recommended invoke path; secondary skills (muted rectangles) are auxiliary tools and auto-invoked helpers. The tracker-integration one-liner from M34 (Linear / Jira / `mode: none` threading Plan → Build → Ship) is preserved verbatim.
- AC-STE-131.6: The diagram renders correctly on github.com — both `classDef` rules and `class` assignments parse without errors; the visual distinction between spine and secondary is perceptible. Verified by visual inspection on the eventual PR preview (same gate as M34's AC-STE-130.5).
- AC-STE-131.7: Diagram placement (between `## Install as Plugin`'s setup paragraph and `## What You Get`) under the `## Workflow` heading is preserved from M34.

## Technical Design

- Single edit to `README.md`. No new code, no new tests, no template changes.
- Mermaid syntax additions over M34's baseline: two `classDef` lines (e.g., `classDef spine fill:#e1f5e1,stroke:#2e7d32,stroke-width:3px,color:#000`; `classDef secondary fill:#f5f5f5,stroke:#999,stroke-width:1px,color:#555`); per-node shape change (`setup(["/setup"])` for spine vs `tdd["/tdd"]` for secondary); per-node class assignment (using inline `:::spine` / `:::secondary` shorthand to keep the diagram source tight rather than a trailing `class id1,id2,id3 spine` block).
- Walks back M34's "vanilla mermaid only" decision per the /brainstorm session — `classDef` has been supported in GitHub's mermaid renderer since 2022 and is widely used; the M34 conservatism was not load-bearing.
- Color choices: pick fills that work on both GitHub light and dark themes. A green-tinted spine + neutral-grey secondary reads consistently in both. Avoid pure white/black fills (theme-dependent).
- "As simple as possible" carry-over from M34: still no regression test asserting skill-name parity. The two-tier hierarchy does not need a probe — it's a one-line classDef rule + per-node shorthand. If future skill churn breaks the spine/secondary split (someone adds a new skill and forgets to classify it), the diagram drifts but renders.

## Testing

- No automated test added — pure doc edit per the M34 simplicity carry-over.
- Manual verification on PR preview: confirm the mermaid block renders with visible spine vs secondary distinction in both GitHub themes; confirm all 14 skills still resolve to `plugins/dev-process-toolkit/skills/<name>/SKILL.md`.
- Spot-check the new prose against `skills/implement/SKILL.md` to confirm each claim still matches the skill's actual behavior (the M34 prose drifted because nobody verified it against the SKILL.md — this verification is the discipline going forward, captured in the next bullet).
- Regression risk: if `skills/implement/SKILL.md` Phase 2/4 procedure changes (e.g., a future FR makes `/implement` actually invoke the `/tdd` skill), this prose paragraph drifts again. Mitigation: add a release-checklist bullet at `/ship-milestone` time ("if `skills/implement/SKILL.md` Phase 2 or Phase 4b changed, re-verify README ## Workflow prose paragraph"). Out-of-scope automated probe — same simplicity constraint as M34.

## Notes

- Severity: low. README cosmetic + factual fix. The factual prose error (3 of 4 wrong claims) is the more important half — it actively misleads new visitors about how `/implement` works.
- M34 retrospective: the prose was written by the same agent that authored the FR + diagram in one pass; nobody cross-checked the prose against `skills/implement/SKILL.md`. This FR fixes the artifact and documents the cross-check discipline as a regression mitigation.
- Single-FR milestone. Phase A only. `/ship-milestone` closes it as a patch (v1.36.1) — no new feature surface, just an accuracy + visual-hierarchy fix to a doc shipped one release ago.
