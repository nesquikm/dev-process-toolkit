## Requirement

`/spec-write`'s milestone-allocation step picks the next free `M<N>` based on whatever signal is convenient. During M30 spec-write 2026-04-27 the signal was a partial `ls` of `specs/plan/archive/` whose output was truncated by a `head` invocation, hiding M28.md and M29.md. Both were already shipped (M28 → v1.29.0 "Runbook", M29 → v1.30.0 "Loudly", M27 → v1.31.0 "Polyglot" most recent). The LLM picked M28 anyway. The user caught it after FR files and Linear tickets were created, requiring a full rename pass + Linear description rewrite.

A memory rule (`feedback_check_milestones_first.md` saved 2026-04-27) covers the current session but does not survive into the next user's session. The check has to live in skill code so that any LLM running `/spec-write` runs it unconditionally, and a `/gate-check` probe to catch any post-edit collisions (e.g., a hand-edit changing an active FR's `milestone:` to point at an archived `M<N>`).

## Acceptance Criteria

- AC-STE-119.1: **Mode-agnostic.** Both the helper and the probe apply to `mode: none` and tracker mode equally — milestone numbers are local-file-system identifiers and the collision risk exists regardless of tracker mode. {#AC-STE-119.1}
- AC-STE-119.2: **`nextFreeMilestoneNumber` shared helper.** New helper at `adapters/_shared/src/next_free_milestone_number.ts` exposes `nextFreeMilestoneNumber(specsDir, changelogPath?): { next: number; sources: { active: number[]; archived: number[]; changelog: number[] } }`. Implementation:
  1. List `<specsDir>/plan/M<N>.md` (active) — capture all `<N>` integers.
  2. List `<specsDir>/plan/archive/M<N>.md` (archived) — capture all `<N>` integers.
  3. If `changelogPath` provided and exists, parse `## [X.Y.Z] — ... — "Codename"` headings; for each release entry, scan its body paragraph for any `M<N>` reference. Capture the union of `<N>` integers found.
  4. Return `next = max(active ∪ archived ∪ changelog) + 1`, plus the source-by-source breakdown for diagnostic purposes.
  Edge cases: empty specs/ → `next: 1`; gaps in the sequence (e.g., `M12, M13, M16`) → still returns `max + 1`, never reuses gap numbers (gaps are intentional — preserved in case the absent number is in flight on a stash). {#AC-STE-119.2}
- AC-STE-119.3: **`/spec-write` invokes the helper before claiming a number.** When `/spec-write` is about to allocate a brand-new `M<N>` (i.e., the user-provided argument is a free-form description and a new milestone is being drafted), the skill MUST call `nextFreeMilestoneNumber(...)` and use its result. The skill MUST refuse to proceed (NFR-10 canonical shape) if the user explicitly typed an `M<N>` argument that is already in any of the three source sets. Diagnostic includes which source(s) the collision came from. {#AC-STE-119.3}
- AC-STE-119.4: **`/gate-check` probe `frontmatter-milestone-not-archived` (#27).** New probe at `adapters/_shared/src/frontmatter_milestone_not_archived.ts`. For each `status: active` FR file under `specs/frs/`:
  1. Read frontmatter `milestone:` value (e.g., `M30`).
  2. Check if `<specsDir>/plan/archive/<value>.md` exists.
  3. Hard-fail (severity `error`, NFR-10 canonical shape) when the active FR points at an archived milestone. Diagnostic names the FR file, the offending value, and the path to the archived plan file proving the collision.
  Active FRs whose milestone matches an active plan file pass. Archived FRs are vacuous (their milestone naturally matches their archived plan file). Mode-agnostic. {#AC-STE-119.4}
- AC-STE-119.5: **Probe registered.** `tracker-project-milestone-attached`-style row added to `gate-check` SKILL.md probe inventory. Tests at `frontmatter_milestone_not_archived.test.ts` cover: vacuous-on-archived-FR, happy-path active-FR pointing at active plan file, hard-fail active-FR pointing at archived plan file, hard-fail active-FR pointing at non-existent plan file (separate diagnostic), missing `milestone:` frontmatter (separate diagnostic — file is malformed). {#AC-STE-119.5}
- AC-STE-119.6: **CHANGELOG parser tolerance.** The CHANGELOG-scan step (AC-STE-119.2 step 3) is best-effort: a missing CHANGELOG.md returns `changelog: []` (vacuous, no error); a malformed `## [X.Y.Z]` heading is skipped silently; a release entry with no `M<N>` reference contributes nothing. The helper never throws based on CHANGELOG content alone — it's an additional safety net, not a hard gate. The hard gate is the file-system check (steps 1+2). Tests cover: missing CHANGELOG, empty CHANGELOG, CHANGELOG without M-references, CHANGELOG with multiple M-references in one entry. {#AC-STE-119.6}
- AC-STE-119.7: **Diagnostic format.** When the pre-allocation guard refuses, the NFR-10 message MUST include all three source breakdowns and the proposed next free number. Example:
  ```
  Refusing to allocate M28: collision detected.
    active   plan files: [30]
    archive  plan files: [1, 2, 3, ..., 28, 29]
    changelog references: [1, 2, ..., 28, 29]
  Next free: M31. Re-run with the correct argument.
  ```
  Format chosen so the user sees *why* and *what to use instead* in one screen. {#AC-STE-119.7}
- AC-STE-119.8: **`bun test` green.** All new tests pass; existing tests + STE-117/STE-118 tests pass. `/gate-check` green on this repo. {#AC-STE-119.8}

## Technical Design

**Module layout:**

```

├── next_free_milestone_number.ts                    # the helper
├── next_free_milestone_number.test.ts
├── frontmatter_milestone_not_archived.ts            # probe #27
└── frontmatter_milestone_not_archived.test.ts
```

**Helper sketch:**

```typescript
export interface MilestoneAvailability {
  next: number;
  sources: {
    active: number[];
    archived: number[];
    changelog: number[];
  };
}

const PLAN_FILENAME = /^M(\d+)\.md$/;
const CHANGELOG_M_REF = /\bM(\d+)\b/g;

export function nextFreeMilestoneNumber(
  specsDir: string,
  changelogPath?: string,
): MilestoneAvailability {
  const active = listMNumbers(`${specsDir}/plan`);
  const archived = listMNumbers(`${specsDir}/plan/archive`);
  const changelog = changelogPath && existsSync(changelogPath)
    ? scanChangelog(changelogPath)
    : [];
  const all = new Set([...active, ...archived, ...changelog]);
  const max = all.size === 0 ? 0 : Math.max(...all);
  return { next: max + 1, sources: { active, archived, changelog } };
}

function listMNumbers(dir: string): number[] {
  if (!existsSync(dir)) return [];
  return readdirSync(dir)
    .map((f) => f.match(PLAN_FILENAME)?.[1])
    .filter((s): s is string => s !== undefined)
    .map(Number)
    .sort((a, b) => a - b);
}

function scanChangelog(path: string): number[] {
  const md = readFileSync(path, "utf-8");
  const found = new Set<number>();
  for (const match of md.matchAll(CHANGELOG_M_REF)) {
    found.add(Number(match[1]));
  }
  return [...found].sort((a, b) => a - b);
}
```

**Probe scoping table (probe #27):**

| FR `status:` | Behavior |
|--------------|----------|
| `archived` | vacuous (archive matches archived plan file by construction) |
| `active`, milestone matches active plan | pass |
| `active`, milestone matches archived plan | hard fail — collision |
| `active`, milestone matches no plan file | hard fail — orphan (separate diagnostic) |
| missing `milestone:` frontmatter | hard fail — malformed (separate diagnostic) |

## Notes

**Why mode-agnostic:** milestone numbers are local-file-system identifiers, not tracker objects. The collision risk is identical in `mode: none` (where M-numbers identify plan files) and tracker mode (where they additionally bind to project milestones via STE-118). Splitting behavior by mode would introduce inconsistency for no benefit.

**Race scope (acknowledged out-of-scope):** two parallel LLM sessions could pass the pre-allocation guard simultaneously and both write the same `M<N>.md`. This is the same race as two parallel sessions creating the same FR file — no FR-file-level locking exists either. The probe (#27) catches the post-write collision at gate-check time. Multi-agent concurrency on a single repo is documented elsewhere (STE-86 / STE-101 wider locking work) and not in scope for M31.

**CHANGELOG as best-effort signal:** the CHANGELOG can be edited freely; a hand-rewritten CHANGELOG could omit M-references and silently shrink the guard's coverage. The file-system check (active + archive plan files) is the load-bearing signal; CHANGELOG is the third leg only because it caught the M28 collision in conversation when the partial `ls` failed. Three legs are better than two.

**Why the probe doesn't enforce sequential numbering:** gaps in the M-sequence (e.g., active M30 with archived M27/28/29 but no M22 or M14/15) are intentional in this codebase — branches in flight, abandoned milestones, renumbered work. Forcing sequential allocation would require either filling gaps (overwriting historical decisions) or refusing valid jumps. The guard's only job is "don't reuse a taken number." The pick-the-next-free behavior is `max + 1` precisely because gap-reuse is unsafe.

**Diagnostic format choice (AC-STE-119.7):** showing *all three source breakdowns* lets the operator instantly see whether a collision is from a recent ship (active list will contain low numbers near the requested), an old shipped milestone (archived list dominates), or a CHANGELOG-only artifact (a release that didn't generate a plan file — possible in pre-M16 history). The breakdown turns a "no, can't" refusal into actionable diagnostic data.
