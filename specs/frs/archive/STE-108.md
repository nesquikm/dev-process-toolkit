## Requirement

The v1.29.0 smoke test surfaced a class of failures that all stem from the same skill-prose ambiguity: when `/setup` is run by an agent (or by a human passing `--no-prompt`-style instructions), the skill collapses several distinct steps under one "do not prompt" rule and silently skips them. Specifically:

1. **F3** — Linear MCP verification (`list_teams` test call from `setup-tracker-mode.md`) was skipped because the agent treated "do not prompt" as "skip the verification step entirely". Verification ≠ Q&A: the call should run regardless.
2. **F5** — Tracker-mode opt-in confirmation (Step 7b) was bypassed without an audit trail. No log entry, no comment in CLAUDE.md indicating the canonical confirmation prompt was skipped. Future readers can't tell what was answered vs assumed.
3. **F6** — Several canonical Q&A steps (7b tracker confirm, 7c branch_template, 7d docs flags, optional rubber-duck) were collapsed under the same "don't prompt" instruction. Some have safe defaults (branch_template, docs flags); others genuinely require human input (tracker mode opt-in). The skill must distinguish.

The fix is to refactor `skills/setup/SKILL.md` so each step is annotated as either `default: <value>` (proceed without input) or `requires-input: <reason>` (refuse to proceed without input), and to separate `verification probes` (which always run) from `Q&A steps` (which respect the prompt mode).

## Acceptance Criteria

- AC-STE-108.1: `skills/setup/SKILL.md` is reorganized so every Q&A step is annotated `default: <value>` or `requires-input: <reason>`. The annotation is rendered in the prose, not just commented — the skill's own text reads like a contract.
- AC-STE-108.2: Verification probes (e.g., Linear `list_teams` test call, `git status` baseline check) are categorized separately from Q&A. The skill MUST run all verification probes regardless of prompt mode. Failed verification → loud abort per STE-106's "fail, don't silently skip" principle.
- AC-STE-108.3: When /setup auto-decides a `default:`-annotated step (autonomous mode or pre-baked answers), it appends an audit-trail entry to a new `## /setup audit` section in CLAUDE.md (or under `## Task Tracking` — design decision during /implement) recording: timestamp, step ID, value used, reason ("default applied" / "pre-baked answer"). Format:

  ```
  ## /setup audit
  - 2026-04-25 step:7c (branch_template) value:"{type}/{ticket-id}-{slug}" reason:"default applied"
  - 2026-04-25 step:7d (docs.packages_mode) value:false reason:"default applied"
  ```

- AC-STE-108.4: When /setup encounters a `requires-input:`-annotated step under non-interactive mode AND no pre-baked answer is supplied, the skill aborts with NFR-10 shape naming the step and the missing input. It MUST NOT proceed with a guessed value.
- AC-STE-108.5: Specifically, the tracker-mode opt-in (Step 7b) is `requires-input:` — even in autonomous mode, the agent must explicitly pass a tracker mode; the skill won't infer one.
- AC-STE-108.6: New `/gate-check` probe `setup-audit-section-presence`: if CLAUDE.md was generated by /setup (detectable via the `<!-- generated by /dev-process-toolkit:setup -->` marker, added by this FR), AND any step has a default value that differs from the user's answer history, the audit section MUST exist. Reading the audit section is sufficient signal that the project was set up autonomously. Probe lives at `plugins/dev-process-toolkit/adapters/_shared/src/setup_audit_section_presence.ts` (flat path matches existing convention); integration test at `plugins/dev-process-toolkit/tests/gate-check-setup-audit-section-presence.test.ts` per STE-82.
- AC-STE-108.7: New helper `plugins/dev-process-toolkit/adapters/_shared/src/setup/audit_log.ts` exposes `appendAuditEntry(claudeMdPath, entry)`. Pure file I/O; unit-tested.
- AC-STE-108.8: Three of the four scenarios are covered programmatically via `tests/setup_audit_log.test.ts` (helper) and `tests/gate-check-setup-audit-section-presence.test.ts` (probe), using in-memory `mkdtempSync` fixtures: interactive run (no marker → vacuous pass), autonomous run with all defaults (audit section populated), autonomous run with pre-baked tracker (toolkit-managed but no defaults emitted → vacuous pass). The fourth scenario — autonomous run with `requires-input:` step missing (abort with named step) — is **skill-prose-only**: the abort behavior lives in `skills/setup/SKILL.md` step 7b's `requires-input:` annotation and the "Step contract" section, enforced by the LLM-as-runtime via prose contract (no programmatic test). This split is consistent with the verification ≠ Q&A separation: Q&A enforcement is prose-level by design (NFR-1).
- AC-STE-108.9: `bun test` green. CHANGELOG `### Changed` entry cross-references this FR + smoke-test origin.

## Technical Design

**Skill prose annotations.** Each Q&A step in `skills/setup/SKILL.md` gets a single-line annotation directly above the question. Example:

```markdown
### Step 7c — Branch template

`default: {type}/{ticket-id}-{slug}` — proceed with this value if not asked.

Ask the user: "What branch-naming convention do you want?"
```

vs.

```markdown
### Step 7b — Tracker mode opt-in

`requires-input: tracker mode is a workspace-wide decision; no safe default exists.`

Ask the user: "Do you want tracker integration? (none/linear/jira/custom)"
```

**Audit log.** New helper writes to a `## /setup audit` section. If section doesn't exist, create it. Append-only.

**Verification probe categorization.** Distinct subsection in SKILL.md ("## Verification probes") listing what runs unconditionally: `git status`, `linear list_teams` (when tracker = linear), `bun --version` (when stack = bun), etc.

**`<!-- generated by /dev-process-toolkit:setup -->` marker.** Single HTML comment at the top of CLAUDE.md, written by /setup. Lets gate-check distinguish toolkit-managed CLAUDE.md from hand-written ones.
