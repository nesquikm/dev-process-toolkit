## Requirement

Two surfaces of the autonomous-mode / Pattern 26 contract surface drift specifically under `claude -p` non-interactive stdin and were caught by `/conformance-loop` iter-1 (2026-05-07/08, post-M66-merge):

- **F1 (entry side):** model emits `Write` / `Edit` / `NotebookEdit` before the first `AskUserQuestion` or `RequiresInputRefusedError`. Phase 8 caught violations on `/spec-write` (Linear leg, `Write@21`) AND `/setup` (Jira leg, `Write@17`) — Pattern 26 entry-contract drift is not skill-specific, the model picks one of the four in-scope skills to scaffold first under autonomous-mode pressure.
- **F2 (exit side):** model fires `AskUserQuestion` (so first-turn passes), prompts come back "dismissed" (no operator on `claude -p`), model self-rationalizes "picked safe defaults" and lands real commits + tracker writes (smoke v4 + iter-1 captured commit `9b75b4b` from `/spec-write` without operator consent).

Both share the same root cause: `claude -p` non-interactive stdin produces "dismissed" `AskUserQuestion` responses, and the model invents answers. Hardening fires only when stdin is non-tty; **interactive sessions stay byte-identical to v2.17.0** (operator-driven scope decision, 2026-05-08 brainstorm session).

STE-220 lesson re-applies: prose-only enforcement fails at runtime; byte-checkable / structural detection is required.

## Acceptance Criteria

- AC-STE-251.1: Each of `plugins/dev-process-toolkit/skills/{setup,brainstorm,spec-write,report-issue}/SKILL.md` gains a "FIRST ACTION (under non-interactive stdin)" lead clause forbidding `Write` / `Edit` / `NotebookEdit` tool calls before the first `AskUserQuestion` `tool_use` OR `RequiresInputRefusedError` raise. Read-only orientation (`Read` / `Grep` / `Glob` / `Bash`-read-only) explicitly allowed. Clause cites `docs/auto-mode-protocol.md § Socratic Loop Contract` by relative path. Interactive (tty) invocations are byte-identical — clause carries explicit "non-tty stdin only" qualifier.

- AC-STE-251.2: `requireOrRefuse(...)` at `adapters/_shared/src/requires_input.ts` (the STE-232 canonical helper) gains a non-tty branch: when `process.stdin.isTTY === false` AND the gate's required value is missing (the dismissed-AskUserQuestion path that previously fell through to "safe defaults"), throw `RequiresInputRefusedError` with NFR-10 canonical message naming the gate site + the missing value + remedy `re-invoke with --tracker=<mode> pre-bake or run interactively (tty)`. When `isTTY === true`, current behavior preserved unchanged. Unit-test matrix covers four cases: `(tty, answered)`, `(tty, declined)`, `(non-tty, answered)`, `(non-tty, dismissed)`.

- AC-STE-251.3: New `/gate-check` probe `socratic_first_turn_post_hoc_drift` at `adapters/_shared/src/socratic_first_turn_post_hoc_drift.ts` — scans the latest commit on the active branch; if commit subject matches the canonical `/spec-write` pattern (`chore(specs): write FR <tracker-id>` or `docs(specs): edit cross-cutting specs`) AND the commit body contains NEITHER an audit-row marker (`spec_write_draft_default_applied` / `spec_write_commit_default_applied`) NOR a refusal NFR-10 block (`Verdict: ... Refused` shape), surface `socratic_first_turn_post_hoc_drift_violation` capability row and **hard-fail the gate**. Probe runs on every `/gate-check` invocation; vacuous on commits not matching the pattern.

- AC-STE-251.4: `/smoke-test` Phase 8 cohort baseline updated — re-running `/smoke-test` on both Linear + Jira legs returns `outcome: ok-asked` or `outcome: ok-refused` for all four in-scope skills (no `violation` outcome). The committed F1 reproducer (the v4 violation transcript at `tests/fixtures/socratic-first-turn/spec-write-2026-05-07.json`) is moved under `tests/fixtures/socratic-first-turn/regression/spec-write-2026-05-07.json` to survive per-date overwrites by future Phase 8 runs; a new unit test `socratic_first_turn_replay.test.ts` re-feeds it through `parseStreamJsonTranscript` + `assertFirstTurnShape` and asserts the helper still detects it as a violation (forward-compat for the helper's contract).

- AC-STE-251.5: **Interactive-session invariance probe.** Manual interactive smoke pass — operator runs `/spec-write` (no marker, no autonomous-mode reminder, terminal tty stdin) on a fresh test project; produces byte-identical observable behavior to v2.17.0 — same draft prompt, same gate options (y/n/edit), same commit prompt, same closing summary shape. Captured as a one-paragraph operator-witnessed note in the FR's Phase 4 report. **Vacuous in `claude -p`** by definition.

## Technical Design

Brainstorm-approved scope (2026-05-08 session, gist NOT created — context lives in this conversation):

| Layer | Mechanism | What it catches |
|---|---|---|
| Source-level (B) | Skill body "FIRST ACTION (under -p)" lead clause + cite | Bare-prose drift in skill bodies; LLM authoring drift |
| Behavioral (B) | `requireOrRefuse(...)` non-tty branch throws on dismissed prompts | F2 silent-commit / silent-tracker-write fall-through |
| Detection (C) | `/gate-check` probe `socratic_first_turn_post_hoc_drift` | Post-hoc commits that bypassed both audit-row + refusal-block contracts |
| Detection (C) | Existing `/smoke-test` Phase 8 + `assertFirstTurnShape` (M66 already shipped) | F1 first-turn contract drift in any in-scope skill |
| Regression (C) | Replay test: re-feed F1 reproducer through helper, assert violation | Forward-compat for `assertFirstTurnShape` contract |

Maps to STE-237 + STE-232 + STE-238 by analogy:

| Layer | M65–M66 (entry-side, prior rounds) | M67 (this FR, exit-side + non-tty) |
|---|---|---|
| Skill-body prose | `AskUserQuestion` mandate (STE-237 AC.2) | + "no scaffold before first ask under -p" lead clause |
| Refusal helper | `requireOrRefuse(...)` per-step (STE-232 AC.2) | + non-tty branch on dismissed-AskUserQuestion |
| Source probe | `socratic_loop_uses_ask_user_question` (STE-237 AC.3) | + `socratic_first_turn_post_hoc_drift` (commit-side) |
| Behavioral fixture | `/smoke-test` Phase 8 (STE-237 AC.4, M66) | reuse — no new phase |
| Audit column | `imputed:` (STE-232) + `loop_entered:` (STE-237 AC.6) | reuse — no new column |

Out of scope:

- Runtime PreToolUse hook for F1 enforcement at the harness layer. Detection-only is sufficient given Phase 8's smoke-run coverage. Re-evaluate at M68 IF the next `/conformance-loop` iteration shows option 1 insufficient.
- Backwards-compat shim for legacy `Auto Mode Active` system-reminder detection (already removed per STE-226 / `project_no_users_yet`).
- Rewriting in-scope skill bodies beyond the lead-clause prepend (existing `AskUserQuestion` discipline from STE-237 stays as-is).
- Interactive-session prompt redesign — operator-driven scope guarantee: interactive bytes-identical.
