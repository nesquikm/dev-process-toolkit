# STE-391: /upgrade skill: version-ordered consumer-artifact migration registry + seeded script entries {#STE-391}

## Summary

Projects that installed the toolkit under older conventions accumulate leftover state that current releases no longer read: retired folders, orphaned marker files, and dead settings entries. A new upgrade command walks a version-ordered registry of migrations, detects which legacy states are present, previews every change, and applies the fixes in one approved commit. Each migration pairs a deterministic detector with either a scripted fix or an operator-guided transform, and re-running the command on an already-clean tree exits with nothing to do. Setup gains a one-line hint that points at the upgrade command when it spots legacy state.

## Requirement

The toolkit has shipped ~25 consumer-artifact convention changes since v1.8.0; roughly half heal via an idempotent `/setup` re-run, and the rest were shipped **forward-only** under the "zero installs" rationale — each one leaving live debt in already-bootstrapped trees (STE-289 said so explicitly for stale hook entries; STE-382 said so for the `.dpt` relayout). That rationale has eroded: the first external user surfaced 2026-07-15 (issue #39), and the operator's own consumer trees (stock_check_app, glacy-app-be) carry exactly this debt.

Ship the reversal as a **general, registry-driven surface** rather than another one-off: a version-ordered migration registry in the shared adapters layer, walked by a new user-invocable `/dev-process-toolkit:upgrade` skill. The `--migrate` flag name stays with tracker-mode switching (STE-98 kept it "for operator muscle memory"), so the runner is a standalone skill — approved via `/brainstorm` 2026-07-17 (surface question re-opened and re-decided after the flag-name collision surfaced; supersedes the same-session `/setup --migrate` cleanup-leg approval).

Rails are deliberately lean (brainstorm Approach 1): clean-tree refusal, diff preview, one approval commit, detection-based idempotency with a "Nothing to do." exit. No dry-run flag, no backup tag, no state-marker file — STE-23 shipped those for a content transform and M17 deleted the marker as rot; the script entries here are cheap, previewable file operations. (The assisted monolith-split entry, FR-B of this milestone, adds a filesystem backup because its input tree can be git-ignored and unprotected.)

## Technical Design

**Registry layout** — `adapters/_shared/src/migrations/`:

| File | Role |
|---|---|
| `index.ts` | `MigrationEntry` type + `MIGRATIONS` ordered list + load-time invariants (dup-id, version ordering) |
| `legacy_paths.ts` | Single source of retired literals (locks dir, ledger dir, v1 marker/index filenames, sync-log heading, stale hook-entry shapes) — the ONLY non-test file allow-listed by the STE-384 drift test |
| `entries/m104_legacy_state.ts` | AC.3 detector + apply |
| `entries/v1_orphans.ts` | AC.4 detector + apply |
| `entries/stale_hook_entries.ts` | AC.5 detector + apply |
| `entries/permission_shapes.ts` | AC.6 detector + apply (flagged `requires_explicit_approval: true`) |

`detect(projectRoot)` is synchronous, filesystem-only, and never mutates; `apply(projectRoot)` returns `{changed: string[], summary}` for the diff-preview surface. Entries with `kind: "assisted"` carry no `apply` — their flow lives in skill prose (FR-B registers the first one).

**Runner sequence** (`skills/upgrade/SKILL.md`): (0) never-bootstrapped probe → route to `/setup`; (1) preamble advisories (installed-version vs newest `introduced_in`; blanket `.dpt/` root-ignore — reuses the check-ignore approach from the STE-383 test matrix); (2) clean-tree gate (`git status --porcelain`); (3) detector walk → detected-set table; (4) script entries: apply → aggregated diff preview → one approval commit (subject `chore(upgrade): apply toolkit migrations <ids>`); AC.6's entry additionally gates on its own explicit approval regardless of marker; (5) assisted entries: hand off to the entry's documented flow; (6) audit rows + closing summary with capability tokens; "Nothing to do." when the detected set is empty.

**Precedents applied:** STE-23 (clean-tree refusal, Nothing-to-do exit), STE-303 (byte-compare idempotent writes), STE-133 (best-effort + audit-surface logging), STE-383 (`writeDptGitignore` reuse; check-ignore matrix), STE-384 (allow-list mechanics), STE-209 (settings-JSON surgery shape).

## Testing

| Surface | Approach |
|---|---|
| Registry invariants | Unit: dup-id and ordering rejection; entry-shape conformance for all seeded entries |
| Each seed entry | Fixture tree per state (`mkdtempSync`), real git for tracked-file legs (STE-383 AC.5 precedent): detect=true on legacy fixture, apply → detect=false (idempotency), re-apply is a no-op, non-legacy fixture untouched byte-for-byte |
| Hook-entry / permission surgery | Round-trip: settings fixture with user content + stale entries → apply → stale entries gone, user content byte-identical |
