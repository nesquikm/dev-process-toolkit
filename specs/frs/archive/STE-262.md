# STE-262: Tighten /spec-write commit-gate + draft-gate to literal byte-checkable marker only {#STE-262}

## Requirement

`/spec-write`'s draft-acceptance gate (§ 0b step 4) and commit-acceptance gate (§ 7a) are auto-applying on `claude -p` invocations whose prompt body lacks the literal byte-string `<dpt:auto-approve>v1</dpt:auto-approve>`. Cross-tracker `/conformance-loop` (2026-05-08) reproduced the regression on both Linear (commit `bbddfcd` on `feat/STE-259-greet`, FR STE-261 + M3 plan written silently) and Jira (DST-23 created in Backlog + commit `04fb609 chore(specs): write FR DST-23`, no audit row, no refusal): marker-absent runs proceed silently, emitting `spec_write_commit_default_applied` while the contract says they should refuse via `RequiresInputRefusedError`.

The runtime gate is conflating the harness's autonomous-mode reminder ("work without stopping") with the byte-checkable STE-226 marker — the exact failure shape STE-213 + STE-220 hit before STE-226 introduced the marker. **This is the third attempt** at this gate. STE-213 (M55) and STE-220 (M56) were both prose-only and both falsified by subsequent smoke runs. STE-226 (M59) added the marker mechanism but the LLM-inference fallback path is still reachable. The contract — "marker is the single deterministic mechanism" — is not enforced at the runtime layer; it is asserted in prose only, and the LLM never invokes a code path that performs the byte-grep.

## Technical Design

### Existing-state assessment

`plugins/dev-process-toolkit/adapters/_shared/src/requires_input.ts` (the helper introduced by STE-232) already implements the byte-strict, non-tty-aware refusal correctly:

- `markerPresent: boolean` is passed in by callers (no LLM inference inside the helper).
- `isStdinNonTty()` reads `process.stdin.isTTY === false` literally.
- `requireOrRefuse(...)` throws `RequiresInputRefusedError` on the refused path with NFR-10-shape message.

The bug is upstream of the helper: `/spec-write`'s SKILL.md prose at the two gate sites instructs the LLM to "default-apply `y` when the prompt body contains the marker (byte-grep, no inference)" — and the LLM does the byte-grep itself, in conversation, never invoking a code path. When the LLM mis-reads its own context (e.g., conflates the harness's autonomous-mode reminder with the marker), the gate auto-applies wrongly. STE-213 + STE-220 + STE-226 all addressed this at the prose layer; this FR addresses it at the runtime layer.

### Fix architecture (Option B from TD review — script-driven decision)

Three layers, no new harness hooks (preserves "triple-check" / no-cross-skill-blast-radius posture):

**Layer 1 — Runtime byte-grep helper.** New module `plugins/dev-process-toolkit/adapters/_shared/src/check_marker_runtime.ts`:

```ts
const MARKER = "<dpt:auto-approve>v1</dpt:auto-approve>";

export function checkMarkerRuntime(promptBody: string): { present: boolean } {
  return { present: promptBody.includes(MARKER) };
}

// CLI shim: `bun run check_marker_runtime.ts <file-path|->`
//   - <file-path>: reads file contents
//   - "-": reads stdin
//   - prints "PRESENT" or "ABSENT" to stdout
//   - exits 0 on success, non-zero on I/O errors
async function main() {
  /* read source per argv[2], call checkMarkerRuntime, print result */
}

if (import.meta.main) await main();
```

Pure function. No env reads, no filesystem reads beyond the optional file argument. Co-located unit test: `check_marker_runtime.test.ts`.

**Layer 2 — SKILL.md gate-site enforcement.** Update `plugins/dev-process-toolkit/skills/spec-write/SKILL.md` at the two gate sites (§ 0b step 4 — draft gate, § 7a — commit gate). Insert a NEW Bash precheck step BEFORE the auto-apply decision. The exact prose pattern (substitute `<gate>` with `draft` or `commit`):

```markdown
**Marker-detection (deterministic).** Before evaluating the <gate>-gate auto-apply branch, run:

```bash
bun run plugins/dev-process-toolkit/adapters/_shared/src/check_marker_runtime.ts /tmp/dpt-spec-write-prompt-body.txt
```

where `/tmp/dpt-spec-write-prompt-body.txt` carries the verbatim first
user message that invoked `/spec-write` (the LLM writes it via Bash
heredoc as the FIRST step of <gate>-gate evaluation). The script
prints `PRESENT` or `ABSENT` on a single line.

**Branch strictly on the script's stdout** — the script's output is the
single deterministic gate decision; no LLM inference, no autonomous-mode
reminder, no "work without stopping" framing influences the auto-apply
branch.

- `PRESENT` → call `requireOrRefuse(..., markerPresent: true, ...)` →
  outcome `default-applied` → emit `spec_write_<gate>_default_applied`
  capability row.
- `ABSENT` + `process.stdin.isTTY === false` → call
  `requireOrRefuse(..., markerPresent: false, defaultValue: "y", userSuppliedValue: undefined, ...)`.
  The existing helper logic falls through to the refusal branch
  (line 161-171 of `requires_input.ts`): `markerPresent && defaultValue !== undefined`
  is false because `markerPresent === false`, so default-apply is skipped
  and `RequiresInputRefusedError` is thrown (NFR-10 canonical shape) →
  exit non-zero with the gate-site name in the message.
- `ABSENT` + `process.stdin.isTTY === true` → fire interactive
  `[y / n / edit]` prompt; on `y` proceed, on `n` emit
  `spec_write_<gate>_declined` and abort.
```

The contract is byte-checkable by the AC-STE-262.4 probe — the literal sentence "the script's output is the single deterministic gate decision; no LLM inference, no autonomous-mode reminder, no 'work without stopping' framing influences the auto-apply branch" must appear at both gate sites.

**Layer 3 — Source-level /gate-check probe + reproduction integration test.** Two safety nets:

- **Source probe** in `plugins/dev-process-toolkit/adapters/_shared/src/spec_write_alternate_trigger_scan.ts`. Mirrors the existing `auto_approve_marker.ts` shape (same `Severity`, `*Violation`, `*Report` types). Globs only `plugins/dev-process-toolkit/skills/spec-write/SKILL.md`. Per-occurrence violation includes `file:line:column` + matched phrase. Severity: error. Probe registered in `plugins/dev-process-toolkit/skills/gate-check/SKILL.md` § probes after the existing `auto-approve-marker-in-canonical-spawns` entry. Test: `plugins/dev-process-toolkit/tests/gate-check-spec-write-marker-alternate-trigger-scan.test.ts` (mirrors `gate-check-auto-approve-marker.test.ts` fixture pattern — known-good vs known-bad SKILL.md fixtures).

- **Reproduction integration test** at `plugins/dev-process-toolkit/tests/spec-write-marker-strict-runtime.test.ts`. Simulates the gate-site flow in-process (no real `claude -p` spawn): constructs prompt-body strings with marker-present and marker-absent + autonomous-mode-reminder variants, calls `checkMarkerRuntime(...)` directly, asserts `{ present: ... }` matches expectation, then chains into `requireOrRefuse(...)` with simulated stdin state. The test file documents pre-fix vs post-fix expectation in JSDoc so a future reader understands the regression history.

### Backward compatibility

- **Marker-PRESENT path:** byte-identical behavior. `requireOrRefuse(...)` interface unchanged. Existing capability rows (`spec_write_*_default_applied`) emit on the same conditions. AC-STE-262.3 + smoke Phase 9-1/9-2a/b are the canaries.
- **Marker-ABSENT + interactive (tty) path:** byte-identical. The new precheck script outputs `ABSENT`, SKILL.md branches to interactive prompt — same UX as today.
- **Marker-ABSENT + non-tty (`claude -p`) path:** **behavior changes** from silent-commit (current bug) to loud refusal (intended contract). No downstream user is silently auto-committed today by intent — the contract is they should refuse — so this is a bug fix, not a feature removal.
  - Today's `claude -p /spec-write` callers without the marker are: (a) the project-local `/smoke-test` skill (Phase 1b fixture explicitly tests the marker-absent path; will see the refusal it was already designed to expect — net positive), (b) `/conformance-loop` (which spawns /smoke-test, transitive), and (c) any downstream user manually running `claude -p /spec-write` without the marker. Per `project_no_users_yet`: zero (c) callers exist today.
  - Net impact for (a) + (b): smoke fixtures that ALREADY assert refusal will start ACTUALLY refusing — no change in expected outcome, just the mechanism becomes deterministic.
  - Net impact for (c) once users exist: any operator running `claude -p /spec-write` MUST inject the marker if they want auto-apply (already documented as the contract). Without the marker, they get a loud refusal instead of a silent commit. This is the desired hardening; any `claude -p` caller relying on the silent-commit bug today is relying on broken behavior.
