# STE-155: /smoke-test Jira-mode branch + Tier 5 automation {#STE-155}

## Requirement

`.claude/skills/smoke-test/SKILL.md` currently hard-codes `mode: linear` (Phase 1 step 4 creates a Linear project via `mcp__linear__save_project`; step 6 pre-writes a Linear-only `.mcp.json`; Phase 5 teardown archives the Linear project). After M43 lands the Jira-adapter corrections, both modes need symmetric automated Tier 5 coverage so future regressions get caught the same way.

Atlassian asymmetry: the MCP exposes no `createJiraProject` / `deleteJiraIssue`. So Jira runs reuse one throwaway Space (e.g., `DST` from smoke-test #5) — Phase 1 step 4 becomes a *visibility* check (`getVisibleJiraProjects` resolves the configured key), Phase 5 teardown closes any work items the run created (matched by label `dpt-smoke` + creation-time window) rather than archiving the Space.

## Acceptance Criteria

- AC-STE-155.1: `/smoke-test` accepts `--tracker linear|jira` flag; default `linear` for back-compat.
- AC-STE-155.2: When `--tracker jira` is passed, `--jira-project <KEY>` is also required. Pre-flight refusal with NFR-10 canonical shape if absent, or if `getVisibleJiraProjects` doesn't return the configured key.
- AC-STE-155.3: Pre-flight refusal added: when `--tracker jira`, the Atlassian MCP must be loaded in `~/.claude-st/` config and OAuth-bound (probe via `atlassianUserInfo`); refuse with NFR-10 shape if missing.
- AC-STE-155.4: Phase 1 step 4 branches on `--tracker`: Linear → existing `save_project`; Jira → visibility check only (no creation; document that the operator pre-creates the Space once and reuses).
- AC-STE-155.5: Phase 1 step 6 branches: Linear → existing Linear MCP entry; Jira → Atlassian Rovo MCP entry (`{"type": "http", "url": "https://mcp.atlassian.com/v1/mcp/authv2"}`) and `### Jira` workspace-binding sub-section in CLAUDE.md set to `project: <flag-value>` + `default_labels: [dpt-smoke]` (free-form sub-section field per `docs/patterns.md`) plus the canonical Schema L `jira_ac_field: description` top-level key (the zero-config sentinel from STE-154 AC-STE-154.3). The `default_labels` value is forwarded by the Jira adapter into `mcp__atlassian__createJiraIssue.additional_fields.labels` on every issue created during the run, so Phase 5 teardown's `labels = "dpt-smoke"` JQL finds them.
- AC-STE-155.6: Phase 5 teardown branches: Linear → archive project (existing); Jira → JQL-search work items created in this run window (label `dpt-smoke`) and transition each to `Done` (no delete; documented limitation).
- AC-STE-155.7: Skill prose covers both modes with parallel structure; reference smoke-test run #5 (Jira pilot pass — surfaced M43's F1–F7 findings; AC-STE-154.9 was operator-deferred at v1.43.0 ship with only step 1 live-verified) and the first automated Jira-mode run as run #6.
- AC-STE-155.8: A clean run of `/smoke-test --tracker jira --jira-project DST` against the verified-good DST Space (post-M43 main) completes the canonical chain end-to-end without hard-failing any operation. Trace lands at `/tmp/dpt-smoke-jira-{date}.md`.

## Technical Design

Single-file edit to `.claude/skills/smoke-test/SKILL.md`. New flag-parser at top of skill (after pre-flights), branched Phase 1 / Phase 5 sections, parallel pre-flight rows for the two modes.

Child claude-st sessions inherit OAuth state from `~/.claude-st/`; no token-in-file leakage. The driver writes only the `mcpServers:` envelope to the test project's `.mcp.json` — no auth material.

Phase 5 cleanup query (Jira branch):

```
JQL: project = DST AND labels = "dpt-smoke" AND created >= "<run-start>"
→ for each: transitionJiraIssue(id, transition_to_done_id)
```

The `dpt-smoke` label is set on every work item the run creates: the `/setup` and `/spec-write` chain children get a label config nudge through CLAUDE.md `### Jira` block: `default_labels: ["dpt-smoke"]` — new free-form sub-section field, not a canonical key.

**Depends on M43 (STE-154)** — the Jira adapter must dispatch correctly before automation can drive it. Sequencing: M43 ships v1.43.0; M44 follows on top of v1.43.0.

**Out of scope:** OAuth bootstrapping (Atlassian MCP must be pre-authenticated; skill refuses if not), self-hosted Jira (already v1-out per STE-154), Confluence/Compass smoke (different surface), driver-side Jira Space creation (MCP doesn't expose).

## Testing

Skill is its own test. Verify by running:

1. `/smoke-test --tracker linear` — back-compat regression (must match pre-M44 behavior byte-for-byte on the Linear path).
2. `/smoke-test --tracker jira --jira-project DST` — first automated Jira-mode run (= run #6).

Both runs land trace files under `/tmp/dpt-smoke-{linear|jira}-<date>.md`.

## Notes

- Pre-req: Atlassian Rovo MCP authenticated against the target site (one-time OAuth in `~/.claude-st/`). Skill refuses if absent.
- Pre-req: a throwaway Space exists on the target Atlassian site (DST is the canonical one for stellartechlab; downstream operators create their own). Skill refuses if `getVisibleJiraProjects` doesn't resolve the flag value.
- AC-STE-154.9 was **operator-deferred at M43 ship** (v1.43.0) — only step 1 (visibility check) was live-verified during `/implement`; steps 2–6 stayed unverified in `specs/notes/jira-smoke-5.md` (see commit `28c595d` body). AC-STE-155.8 (this FR — smoke #6) provides the **retroactive validation** for both M43's spec corrections and M44's own automation in a single end-to-end run against `DST`. Future passes are then `/smoke-test --tracker jira` invocations only.
- Label-based teardown is best-effort. If a run aborts mid-way, the next run's Phase 5 picks up orphaned `dpt-smoke` items (idempotent transition to Done).
