# STE-399: socratic_first_turn: reject vacuous ok-asked + path-scope scaffolding detection {#STE-399}

## Summary

The arbiter that checks whether a skill honoured the Socratic first-turn contract has two blind spots. When a transcript never asks a question, never refuses, and never scaffolds, it falls through to a pass verdict — so a skill that did nothing at all reads as compliant. And it decides whether an early file write is a contract violation from the tool name alone, with no notion of where the write landed, so a write into a directory the run doesn't own (an operator's global side-effect log, a skill's own temp scratch) counts against the skill and fails the run.

The result is a verdict that can neither reliably pass nor reliably fail, and one whose outcome depends on the operator's machine. This FR gives the fall-through its own distinct outcome the consuming CLI treats as a non-pass, and scopes the write check to a caller-supplied project root so only in-project writes count.

## Requirement

The 2026-07-19 conformance run's Phase 8 exercised the arbiter live and both defects fired:

1. **Vacuous pass.** `assertFirstTurnShape` returns `{ outcome: "ok-asked" }` with no `askIndex` for any transcript that never asks, refuses, or scaffolds (the fall-through at the end of the walk). The consuming CLI checks only `outcome`, so this reads as a pass. Combined with the environment reality that `AskUserQuestion` is not exposed to `claude -p` children — zero `AskUserQuestion` `tool_use` occurrences across every capture in the run — no child can produce a genuine `ok-asked`, so every non-scaffolding child reported a vacuous pass.

2. **Path-blind violations.** `SCAFFOLDING_TOOLS` is matched by name only. Every flagged `Write` in the run landed **outside** the project under test — the operator's global-CLAUDE.md English-correction log, plus one temp-scratch path that is a skill's own documented working directory. `git status` in the project stayed clean throughout. This makes the verdict operator-machine-dependent: the same plugin passes or fails depending on whether the machine's global config mandates a Write.

The fix must make the vacuous case distinguishable and non-passing, and must scope scaffolding detection to the project under test, while keeping the existing by-name behaviour when no project root is supplied (backward compatibility for callers that don't pass one).

## Acceptance Criteria

- AC-STE-399.1: `assertFirstTurnShape` returns a distinct `"vacuous"` outcome (a new member of `FirstTurnOutcome`, not `"ok-asked"`) when the transcript contains no ask, no refusal, and no scaffolding tool_use. {#AC-STE-399.1}
- AC-STE-399.2: the `socratic_first_turn_assert` CLI prints a `vacuous` verdict line and exits non-zero for the vacuous outcome — distinct from both the `ok-*` pass exit (0) and the `violation` exit (1); the vacuous line is visually distinguishable in the emitted verdict. {#AC-STE-399.2}
- AC-STE-399.3: `assertFirstTurnShape` accepts an optional `projectRoot` parameter; when supplied, a scaffolding tool_use is a violation only if its target path resolves inside `projectRoot`. {#AC-STE-399.3}
- AC-STE-399.4: with `projectRoot` supplied, a scaffolding tool_use whose target path is outside `projectRoot` (an absolute path elsewhere, or a path that does not resolve under the root) does NOT raise a violation. {#AC-STE-399.4}
- AC-STE-399.5: `parseStreamJsonTranscript` populates each `tool_use` entry's target path from the tool_input (e.g. the `file_path` / `notebook_path` input field), exposed as an optional `path` on `TranscriptEntry`. {#AC-STE-399.5}
- AC-STE-399.6: `projectRoot` omitted ⇒ unchanged by-name behavior — any scaffolding tool_use before the first ask/refusal is a violation (backward compat); the CLI accepts an optional trailing `<projectRoot>` argument and forwards it. {#AC-STE-399.6}

## Technical Design

Three files under `plugins/dev-process-toolkit/adapters/_shared/src/`.

**`socratic_first_turn.ts`:**

- Extend `FirstTurnOutcome`: `"ok-asked" | "ok-refused" | "violation" | "vacuous"`.
- The fall-through at the end of `assertFirstTurnShape` (currently `return { outcome: "ok-asked" }`) returns `{ outcome: "vacuous" }` (no `askIndex`). The docstring's "callers must check askIndex" caveat is superseded — the outcome now carries the signal directly.
- Add an optional second parameter `opts?: { projectRoot?: string }`. In the scaffolding branch, when `opts?.projectRoot` is set AND the entry carries a `path`, only throw when the path resolves inside `projectRoot` (use `path.resolve` + a boundary-safe prefix check — resolve both and compare with a trailing separator so `/a/bcd` is not treated as inside `/a/bc`). When `projectRoot` is unset, or the entry has no `path`, preserve current behaviour (throw) — AC-STE-399.6 backward compat, and a scaffold with no observable path stays conservatively a violation.
- Extend `TranscriptEntry` with `path?: string`.

**`socratic_first_turn_stream.ts` (`parseStreamJsonTranscript`):** when projecting a `tool_use` event, read the tool_input path field (`file_path` for Write/Edit, `notebook_path` for NotebookEdit) and set `entry.path`. Non-scaffolding tools contribute no path (harmless).

**`socratic_first_turn_assert.ts` (CLI):** accept an optional third argv (`<projectRoot>`), forward it to `assertFirstTurnShape(transcript, { projectRoot })`. Map the `vacuous` outcome to a distinct verdict line (`<skill>: vacuous askIndex=-1`) and a non-zero exit (exit code distinct in intent from violation; document which — e.g. exit 3 for vacuous vs 1 for violation, so the smoke driver can tell them apart). `ok-asked`/`ok-refused` stay exit 0; `violation` stays exit 1.

The consuming smoke-test SKILL (project-local) will be updated to pass the test-project root + treat `vacuous`/non-zero as a non-pass — that wiring is outside this FR (project-local `.claude/skills/`), but the helper + CLI shape this FR ships is what makes the wiring correct.

## Testing

- `socratic_first_turn.test.ts`: a transcript of only read-only / text entries returns `vacuous` (AC-STE-399.1). With `projectRoot`, a Write entry whose `path` is inside the root still throws (AC-STE-399.3); a Write entry whose `path` is outside the root does not throw (AC-STE-399.4); a Write entry with no `path` still throws (AC-STE-399.6 conservative default). Boundary case: `/root-sibling/x` is NOT treated as inside `/root`.
- `socratic_first_turn_stream.test.ts`: a Write `tool_use` NDJSON event surfaces `file_path` as `entry.path`; a NotebookEdit surfaces `notebook_path`; a non-scaffolding tool has `path` undefined (AC-STE-399.5).
- CLI-level assertion (bun subprocess or a thin unit around the arg-parse + outcome→exit map): vacuous ⇒ non-zero + `vacuous` verdict line, distinct from `ok-*` (0) and `violation` (1) (AC-STE-399.2, AC-STE-399.6 arg forwarding).

## Notes

Origin: `/conformance-loop` 2026-07-19, findings F4 (Linear) + F10 (Jira), high — the two legs described the same two defects, which is what pins them. The deeper environmental fact (AskUserQuestion unavailable under `-p`) is why `ok-refused` is the only genuinely-achievable Phase-8 pass; making `vacuous` non-passing is what stops the harness from green-lighting a no-op child.
