# STE-126: /implement Phase 4 cleans up stale plan task-list verify lines {#STE-126}

## Requirement

When `/implement` Phase 4 deletes a file referenced in any active plan's task `verify:` line (e.g., `tests/.placeholder.test.ts` after a real test lands), the plan's task entry must be updated in the same commit — either rewritten to verify a new equivalent or marked `[x]` and the verify line dropped. Stale verify commands left in plan files create runtime drift (the verify command errors when re-run).

Same root-cause family as STE-125 — Phase 4 cleanup currently doesn't reach into plan-file prose.

**Smoke-test evidence.** Run #4 (2026-04-27, v1.34.0): `/implement STE-124` deleted `tests/.placeholder.test.ts` (correct — `src/greet.test.ts` provides real coverage) but `specs/plan/M1.md:28` still carried `verify: head -1 tests/.placeholder.test.ts shows ...`. The verify line errors out if re-run. Full report: `/tmp/dpt-smoke-findings-2026-04-27.md` § F3.

## Acceptance Criteria

- AC-STE-126.1: When `/implement` deletes a file `P` (whether `git rm` or unstaged `rm`), it scans active plan files (`specs/plan/M*.md`, excluding `archive/`) for `verify:` lines referencing `P` and either (a) rewrites the verify line to reference the replacement file when an obvious one exists in the diff, or (b) marks the parent task `[x]` and drops the verify line entirely when no replacement is detected. Verified by `grep -E '\bverify:.*\bP\b' specs/plan/M*.md` returning zero matches after Phase 4.
- AC-STE-126.2: Add a gate-check probe (e.g., probe #28 `plan-verify-line-validity`) that detects active plan `verify:` lines referencing nonexistent paths and surfaces them in the same NFR-10 shape as probe #23. Severity: warning (not gate-failing) — the file may have been intentionally moved without `/implement` involvement.
- AC-STE-126.3: `/implement` Phase 4 report lists "updated N plan task-list verify lines" (or "no plan verify lines to update") so the operator sees the work in the summary.

## Technical Design

- Hook point: same Phase 4 hook as STE-125 — same `rewrite_links.ts`-style helper but scoped to plan-file `verify:` lines and matched against the file diff list.
- Reuse the diff-walk from STE-125 — pass the deleted-files list to a new `cleanupPlanVerifyLines(specsDir, deletedFiles)` helper.
- Detection heuristic for "obvious replacement": if a single new test file under `src/` or `tests/` was added in the same diff and the deleted file's basename was `.placeholder.test.ts`, treat the new file as the replacement.

## Testing

- Test file: `tests/spec_archive/cleanup_plan_verify_lines.test.ts`.
- Bun test cases: (a) deleted-with-replacement → verify line rewritten; (b) deleted-no-replacement → task marked done, verify dropped; (c) verify line references file that wasn't deleted in this diff → no change; (d) gate-check probe surfaces stale verify lines.

## Notes

- Severity: low. M1 is conceptually "done" once a real test exists; the stale verify is documentation drift, not runtime breakage. But the cumulative effect across many milestones is plan-file rot.
- Land after STE-125 — both share the same Phase 4 hook point.
