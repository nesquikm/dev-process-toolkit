# STE-394: Warn-only gate probe #69: consumer-artifact staleness detection via the migration registry {#STE-394}

## Summary

The gate check gains a warn-only probe that walks the migration registry's detectors and reports any consumer artifact that has fallen behind current toolkit conventions. Projects that drifted no longer stay stale silently: the staleness surfaces on every gate run, in the notes class that never blocks a merge. A stale-plugin advisory that could never fire is deleted in the same change, because the registry ships inside the plugin and so the comparison it made was true by construction.

## Requirement

M108 shipped `/upgrade` and its detector registry, but nothing ever tells a project it is stale — discovery depends entirely on the operator remembering the command exists. Humans are not reliable at that.

A brainstorm (2026-07-19) explored storing a toolkit version stamp in the project and comparing it against the installed plugin. A four-way adversarial sweep refuted the stamp:

- A `stored == installed` fast path that skips the detector walk is unsound: reverts, cherry-picks, bad merges, and hand-edited `permissions.allow` leave legacy state permanently undetected. `skills/upgrade/SKILL.md:41` already concedes that code still running can "re-create the very state the migration just cleaned" — note that this line sits inside the advisory AC-STE-394.6 deletes, so the concession must be preserved in the Step 1 prose that survives.
- The stamp is a cache in front of a sub-millisecond operation — two file reads to skip a handful of `existsSync` calls across the registry's five entries (`migrations/index.ts:89-95`). Net slower than the walk it optimizes.
- The "your plugin is stale" direction is harness-owned: `known_marketplaces.json` carries `"autoUpdate": true` and `installed_plugins.json` already records per-project `version`, `lastUpdated`, and `gitCommitSha`.
- A tracked stamp cannot be written where the check runs — step 0 would dirty the tree, and seven skills (including `/upgrade` step 2) refuse on a dirty tree.

STE-391 had already rejected a state-marker file once ("STE-23 shipped those… M17 deleted the marker as rot"), and `skills/upgrade/SKILL.md:16,218` sell "no state-marker file" as a feature. Detection was already the stronger gate — it catches a project that hand-reintroduced legacy state at any version, which a version compare cannot. So the notification ships as a probe over the existing registry, with no new state.

## Acceptance Criteria

- AC-STE-394.1: Probe `upgrade_staleness` registered as #69 in `/gate-check`. **Step 0 — applicability short-circuit:** when the project's CLAUDE.md carries no toolkit-managed section, the probe emits zero rows and returns clean without walking any detector. Otherwise it imports `MIGRATIONS` and calls each entry's `detect(projectRoot)` in registry order. Every entry reporting `applies: true` renders exactly one row under GATE PASSED WITH NOTES. The probe never emits GATE FAILED — pre-existing drift must not block merges (AC-STE-67.6 shape). {#AC-STE-394.1}
- AC-STE-394.2: Row shape is byte-pinned: `<entry-id> (<introduced_in>): <evidence joined by "; ">` — `evidence` is `string[]` (an entry such as `m104_legacy_state` may return several), joined in the detector's returned order with `"; "`, never re-sorted and never truncated. {#AC-STE-394.2}
- AC-STE-394.3: Probe purity — pure file reads plus a registry module load. No git, no network, no LLM judgment (AC-STE-393.5 bar). **Detector-error containment, added at implementation time** (`underspecified` deviation, found by the Phase 3 Stage C error-path audit and reproduced against the **live** registry): the registry walk originally called `entry.detect(projectRoot)` unguarded, so any throwing detector propagated and **crashed the whole gate run** — strictly worse than the GATE FAILED that AC-STE-394.1 forbids. Reachable in the real world, not only synthetically: a toolkit-managed tree whose root `.gitignore` is unreadable makes `m104-legacy-state`'s detector throw `EACCES` (via `readLines`, `consumer_files.ts:90`). Each `detect` call is therefore wrapped per-entry; a throw **degrades to a row** rather than to a crash — the probe resolves, `violations` stays empty, and the erroring entry renders inside the AC-STE-394.2 byte-pinned shape with `detector error: <message>` as its evidence (whitespace-collapsed so a stack dump cannot break the one-row-per-line NOTES shape; a non-`Error` throw never renders `undefined`). One throwing entry never suppresses its neighbours — the walk continues in registry order. The catch is side-effect free, so purity holds on the error path too. {#AC-STE-394.3}
- AC-STE-394.4: Vacuous in both directions. Zero detectors firing produces zero NOTES rows. The Step 0 short-circuit of AC-STE-394.1 produces zero rows and no error even when legacy artifacts are present on disk — a tree carrying a legacy locks directory but no toolkit-managed CLAUDE.md section is a consumer-safety case, not a staleness hit (AC-STE-380.4 shape). {#AC-STE-394.4}
- AC-STE-394.5: The NOTES block closes with the byte-pinned remedy literal `Remedy: run /dev-process-toolkit:upgrade to apply these migrations (Claude can invoke it; it is not on the slash menu).` — emitted exactly once per run when at least one row rendered, never when zero rows rendered. A test asserts the literal. {#AC-STE-394.5}
- AC-STE-394.6: The structurally-inert stale-plugin advisory is deleted as a whole unit: `skills/upgrade/SKILL.md:41` (prose) **and** its fenced advisory block at `:43-46`. The surrounding Step 1 prose is re-keyed in the same edit — the heading at `:37` and the sentence at `:39` ("**Both** advisories below… **Neither** refuses") both become false with one advisory left, and the "code still running can re-create the state" concession from `:41` is preserved in the surviving prose. Rationale: the registry ships inside the plugin, so `max(introduced_in) <= installed` holds by construction for every released build (2.49.0 installed vs 2.46.0 highest `introduced_in` today) and the advisory can never fire for any consumer. A retired-literal ABSENT tripwire pins the string `Advisory: installed plugin v`, vacuous-passing when the file is absent. {#AC-STE-394.6}
- AC-STE-394.7: Probe-count pins bump 68 to 69 at **every** pinned surface, enumerated: `README.md:14` (`68 numbered`), `README.md:111` (`layers 68 probes`), `skills/gate-check/SKILL.md` (new `69. **upgrade_staleness**` entry after `68. **migration_coverage**`), `tests/gate-check-spec-write-next-line-doc.test.ts:381,385,387,397`, `tests/gate-check-public-surface-count-drift.test.ts:394,396,397`, and `tests/m108-ste-393-docs-pins.test.ts:236` and its describe block. {#AC-STE-394.7}

## Technical Design

Probe module alongside the existing gate probes, importing `MIGRATIONS` from `adapters/_shared/src/migrations/index.ts`. No new detector layer — AC-STE-391.1 already guarantees detectors are pure, synchronous, filesystem-only, and network-free, which is exactly the probe-purity bar AC-STE-393.5 sets.

Severity is NOTES-only. AC-STE-369.3's severity model is the reference: a NOTES row keeps the advisory permanently visible, which is the stated counter-pressure against a warn class decaying into background noise.

**Why Step 0 is required, not defensive.** The detectors do not read CLAUDE.md — `entries/m104_legacy_state.ts:31-52` decides purely on `existsSync`. So a non-toolkit project that happens to carry a similarly-named directory would otherwise render staleness rows for a toolkit it never installed. The applicability short-circuit is what makes AC-STE-394.4's consumer-safety guarantee reachable at all; without it AC-STE-394.1 and AC-STE-394.4 are mutually unsatisfiable.

**Deliberately excluded from scope** (each refuted or judged negative-value during the 2026-07-19 adversarial sweep):

- Any version stamp, in `.dpt/` or in a CLAUDE.md section. See Requirement.
- A `SessionStart` hook. A hook at `exit 0` reaches only operator stderr, cannot invoke a skill, and would fire mostly on dirty trees where `/upgrade` step 2 refuses. `additionalContext` / `hookSpecificOutput` have zero hits repo-wide.
- An *offer* (as opposed to a report) at `/gate-check`. The bundled `pre-commit-gate-check.ts:19-24` hook makes `/gate-check` run with a staged tree before every commit, so an offer there would fire almost exclusively into `/upgrade`'s clean-tree refusal. Notification is in scope; offering is not.

## Testing

Fixture trees per `mkdtempSync` (STE-391 Testing shape): one clean tree asserting zero rows; one tree per registry entry asserting exactly that entry's row and its byte-pinned shape (AC-STE-394.2, including a multi-`evidence` entry to pin the `"; "` join); one tree carrying a legacy locks directory **and** no toolkit-managed CLAUDE.md section, asserting zero rows via the Step 0 short-circuit (AC-STE-394.4 — this is the case that distinguishes the short-circuit from a plain no-hit); one run asserting the remedy literal fires exactly once with ≥ 1 row and not at all with zero rows (AC-STE-394.5).

Retired-literal ABSENT assertion for the string `Advisory: installed plugin v`, plus assertions that the `:37` heading and `:39` sentence no longer claim two advisories, vacuous-passing when the file is absent.

AC-STE-394.8 needs a **failing-first** test: a README fixture whose probe count disagrees with disk must produce a violation. Against today's code that test passes vacuously (the leg never fires), so it must be written to fail before the re-anchoring lands — otherwise the repair is unverified.

**Consumer-safety constraint on the AC-STE-394.8 re-anchor** (backfilled at implementation time — `underspecified` deviation, found by the STE-394 spec audit and reproduced empirically). Re-anchoring the parse by content arms a latent over-fire the dead leg had been masking: `computeObservedCounts` returns `maxProbeNumber: 0` whenever `plugins/dev-process-toolkit/skills/gate-check/SKILL.md` is absent — i.e. in **every consumer project** — and the `diskChecks` loop carries no toolkit-tree guard and no zero-guard. A loose `/(\d+)\s+numbered/` scan therefore reads an incidental consumer README line such as "Follow the 3 numbered steps" as a documented probe count and emits a **severity: error** violation whose remedy instructs the consumer to rewrite their own README to say "0 numbered probes". Reproduced: 1 violation at `README.md:5:12` on a toolkit-free fixture. The scan MUST therefore match the **full** token `(\d+) numbered \`/gate-check\` probes`, which every fixture and the real README already carry — narrow enough that incidental prose cannot match, while still content-anchored (a fixed line index is NOT an acceptable substitute) and still reporting the token's true 1-based line. A consumer-safety guard block asserts zero violations on a toolkit-free tree carrying incidental `N numbered <noun>` prose; it sits adjacent to, and does not weaken, the non-vacuity block above.

Traceability: add a row for AC-STE-394.1..9 to the matrix in `specs/requirements.md` (last data row is `:402`).

## Implementation notes

- Probe purity is convention-based *transitively*. The AC-STE-394.3 source scan covers only `upgrade_staleness.ts` itself, so a future registry detector that shelled out to git would keep all four purity tests green. Today's detectors are pure reads and every `Bun.spawnSync` in the registry sits on an `apply` path the probe never calls — but that is convention, not an assertion.

- `TOOLKIT_MARKER` (`<!-- generated by /dev-process-toolkit:setup -->`) is now byte-duplicated in three modules — `toolkit_bootstrap_committed.ts`, `setup_audit_section_presence.ts`, and this FR's `upgrade_staleness.ts`. Two predate this FR. Single-sourcing it (the `legacy_paths.ts` / `dpt_paths.ts` pattern) would mean rewiring two unrelated probes that carry their own byte-pin tests — deferred as a cross-cutting sweep.

- `public_surface_count_drift.ts:156` computes the agents-token column via `m[0].indexOf(m[2])`, a value search that would mismatch if the commands and agents digit strings ever overlapped (e.g. `18 commands, 8 agents` → `indexOf("8")` hits inside `18`). Pre-existing and byte-identical at this branch's merge base; cannot fire at today's 16/8 values; deliberately not fixed here because AC-STE-394.8 scopes the repair to the probe-count leg and says not to alter the others.
