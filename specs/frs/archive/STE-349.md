# STE-349: Verification-skills docs, patterns.md entry + stack examples {#STE-349}

## Requirement

The generator (STE-348) produces a stub, but the *practice* — why post-`/implement` verification matters and how to author a good check skill — needs to be teachable and discoverable beyond the moment of scaffolding. This FR (FR-C of M93) ships the "encourage" half as documentation: an authoring guide, a `patterns.md` entry, and worked per-stack examples modeled on the `glacy-app-fe` skills.

## Acceptance Criteria

- AC-STE-349.1: **Authoring guide.** A new `plugins/dev-process-toolkit/docs/verification-skills.md` documents: the verification philosophy (a passing gate ≠ a working app), the `## Verification` config contract (STE-347), the `.claude/skills/<name>` + `disable-model-invocation: true` convention, and a step-by-step "author your own check skill" walkthrough (from the `/setup` scaffold or from scratch). {#AC-STE-349.1}
- AC-STE-349.2: **`patterns.md` entry.** `plugins/dev-process-toolkit/docs/patterns.md` gains a new numbered Pattern — "Project-Authored Verification Skills" — cross-referenced from the STE-347 Phase 4b″ prose and the authoring guide, and cross-linking the existing Pattern 10 (Visual Verification via MCP) + Pattern 12 (Verification-Before-Completion). {#AC-STE-349.2}
- AC-STE-349.3: **Per-stack examples.** The `plugins/dev-process-toolkit/examples/` tree gains a worked, condensed check-skill example for each primary stack — TypeScript/web, Flutter, Python — each modeled on the `glacy-drive` (Flutter/dart-MCP drive) and `glacy-progress-e2e` (HTTP API e2e) shapes and copy-adaptable. {#AC-STE-349.3}
- AC-STE-349.4: **Discoverable cross-links.** The STE-348 stub and the `/implement` scaffold-offer reference the authoring guide by path; `README.md` and `plugins/dev-process-toolkit/docs/methodology.md` mention the verification-skill capability so it is discoverable from the top of the docs. {#AC-STE-349.4}

## Technical Design

**Files touched:**
- `docs/verification-skills.md` (new) — the authoring guide.
- `docs/patterns.md` — new Pattern entry (next free number) + cross-links to Pattern 10 + Pattern 12.
- `examples/<stack>/…` — one condensed check-skill example per stack (TS/web, Flutter, Python); exact filenames follow the existing `examples/` per-stack layout.
- `docs/methodology.md` + `README.md` — one-line capability mention + link to the guide.
- Cross-link back-references in `skills/implement/SKILL.md` (Phase 4b″) + `templates/check-skill/*` (the stubs' guide link, shared with STE-348).

**Docs-only FR** — no runtime helper. It documents against the contract STE-347 defines and the generator STE-348 ships.

## Testing

## Notes

- **References STE-347 + STE-348.** Documents the convention (347) and the generator (348); ships last in build order A→B→C.
- **Examples are condensed, not full apps** — they encode the shape (frontmatter, MCP tools, run/assert steps) a user copies, drawn from the real `glacy-drive` / `glacy-progress-e2e` patterns without dragging in Glacy-specific plumbing.
