# STE-133: Adopt Conventional Commits v1.0.0: commit-msg hook + skill prose + /setup integration {#STE-133}

## Requirement

Switch the toolkit's commit-message convention from the bespoke `M35: v1.36.1 "Spine" — title (STE-131)` form to [Conventional Commits v1.0.0](https://www.conventionalcommits.org/en/v1.0.0/). Dogfooded: this repo enforces CC from M36 forward; the plugin enforces CC in adopting projects via a `/setup`-installed hook.

Approved `/brainstorm` decision (2026-04-27):

- **Scope:** both this repo's own commits *and* the plugin's enforcement in adopting projects via `/setup`.
- **Enforcement:** local `commit-msg` hook template that validates CC format. No CI gate, no history rewrite — CC applies from cutover (M36) forward.
- **Vocabulary:** standard Angular set (`feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`) with **scopes encouraged** (e.g., `feat(brainstorm): ...`, `chore(skill/pr): ...`).
- **Tracker IDs in footer:** `Refs: STE-<N>`.
- **Release metadata in footer** (release commits only): `Release: vX.Y.Z "Codename"`.
- **Breaking changes:** `!` marker per spec (e.g., `feat!: ...`); `BREAKING CHANGE:` body footer is also accepted per the spec.
- **Hook posture:** pure POSIX shell hook by default (zero deps; works on every adopting project including the polyglot Flutter/Python examples). Opt-in `commitlint` path for projects with Node/Bun tooling that want full spec coverage. This repo (no `package.json`) uses the shell hook.
- **Cutover:** M36-forward, hard-block from commit #1. No grace period; no soft-warning phase. Existing M1–M35 commit history is intentionally pre-CC and not rewritten.

## Acceptance Criteria

- AC-STE-133.1: A POSIX-shell `commit-msg` hook template at `plugins/dev-process-toolkit/templates/git-hooks/commit-msg.sh` validates the message file passed as `$1`. The subject line (first non-blank, non-comment line) must match the regex `^(feat|fix|docs|style|refactor|perf|test|build|ci|chore|revert)(\([^)]+\))?!?: .+` and be ≤72 characters. Empty messages and comment-only messages (every non-blank line starts with `#`) are allowed (interactive abort path; git itself handles the abort). On valid: exit 0 silently. On invalid: exit 1 with stderr containing the failing rule name (`type-prefix-missing` / `unknown-type` / `missing-colon-or-description` / `subject-too-long`) and the offending subject line verbatim.
- AC-STE-133.2: The `/setup` skill (`plugins/dev-process-toolkit/skills/setup/SKILL.md`) installs the shell hook by default — copies `templates/git-hooks/commit-msg.sh` to `.git/hooks/commit-msg` with `chmod +x`. Installation is idempotent: re-running `/setup` against an already-installed hook (byte-identical content) is a no-op. If `.git/hooks/commit-msg` already exists with **different** content, `/setup` shows a diff and prompts before overwrite (decline leaves the existing hook untouched and prints the manual install command for later use).
- AC-STE-133.3: `/setup --commitlint` (a new flag) swaps the shell hook for a commitlint-delegating one: writes `commitlint.config.js` extending `@commitlint/config-conventional` at the repo root, copies `templates/git-hooks/commit-msg.commitlint.sh` to `.git/hooks/commit-msg` (the script delegates to `bunx commitlint --edit "$1"`). `/setup` does NOT run `bun install` or `bun add` itself — it surfaces the install command `bun add -D @commitlint/cli @commitlint/config-conventional` in its output for the user to run. The flag is a no-op (warns and falls through to the shell hook) on adopting projects without `bun` available on `$PATH`.
- AC-STE-133.4: The `/implement` skill (`plugins/dev-process-toolkit/skills/implement/SKILL.md`) Phase 4 step 15 commit-message section instructs the agent to author messages in CC v1.0.0 form: subject `<type>(<scope>): <title>` (≤72 chars), blank line, body (existing prose), blank line, footer block containing `Refs: STE-<N>` (one line per FR touched in the commit). The type defaults to `feat` for new functionality; the agent is instructed to choose `fix` for bug-fix FRs, `docs` for doc-only FRs, `chore` for non-functional housekeeping, and `refactor` for behavior-preserving restructures. Scope defaults to the primary touched skill or area (e.g., `skills/pr`, `adapters/linear`, `templates`). The chosen type/scope is shown to the user at the Phase 4 approval gate before commit. Breaking changes use `!` per spec.
- AC-STE-133.5: The `/ship-milestone` skill (`plugins/dev-process-toolkit/skills/ship-milestone/SKILL.md`) release-commit section emits subject `chore(release): vX.Y.Z` (≤72 chars). Footer block contains `Release: vX.Y.Z "Codename"` and `Refs: M<N>`. Body contains the existing release-checklist summary (CHANGELOG diff, files bumped, FRs included) — unchanged in shape from pre-M36.
- AC-STE-133.6: The `/pr` skill (`plugins/dev-process-toolkit/skills/pr/SKILL.md`) PR-title section emits a CC-shaped title (`<type>(<scope>): <title>`, ≤72 chars) that mirrors the underlying feature commit's subject. PR body shape is unchanged from pre-M36.
- AC-STE-133.7: The CLAUDE.md template at `plugins/dev-process-toolkit/templates/CLAUDE.md.template` carries a new `## Commit Convention` section that states (a) Conventional Commits v1.0.0 is the required format from project setup forward, (b) tracker IDs go in a `Refs:` footer line, (c) breaking changes use the `!` marker per spec, (d) the `commit-msg` hook installed by `/setup` enforces the convention locally. The section includes three sample messages (one `feat(scope):`, one `fix:`, one `chore(release):` with footers) and notes the `--commitlint` opt-in path.
- AC-STE-133.8: The root `CLAUDE.md` of this repo carries the same `## Commit Convention` section (under the existing `## Git Commit Messages` content or replacing it, per the implementer's call at /implement Phase 1). The section explicitly notes that M36 implementation commits and the M36 ship commit use CC, and that M1–M35 commit history is intentionally pre-CC and not rewritten.
- AC-STE-133.9: After M36 lands, this repo's local `.git/hooks/commit-msg` hard-blocks every non-CC commit with no grace period and no soft-warning phase. Verified by attempting `git commit -m "broken message"` after the hook is installed — the commit is rejected with the AC-STE-133.1 diagnostic. The M36 implementation commit and the M36 ship commit both pass the hook.
- AC-STE-133.10: A new test file `plugins/dev-process-toolkit/tests/conventional-commits-hook.test.ts` exercises `templates/git-hooks/commit-msg.sh` via `Bun.spawn` against ≥10 fixtures: valid cases include a vanilla `feat: add X`, a scoped `fix(adapters/linear): handle null`, a breaking `feat!: drop legacy API`, a `chore(release): v1.37.0`, and a `docs(readme): clarify install`; invalid cases include no type prefix, unknown type (`fixme: ...`), missing colon (`feat add`), subject >72 chars, and empty description after colon (`feat: `). Each fixture asserts the expected exit code and (for invalid cases) a stderr substring matching the rule name from AC-STE-133.1.

## Technical Design

- **New files** (POSIX-shell-only stack — no `package.json` introduced in this repo):
  - `plugins/dev-process-toolkit/templates/git-hooks/commit-msg.sh` — POSIX shell validator (~40 lines). Reads `$1` (the message file path git passes), strips comments and trailing blank lines, extracts the subject line, runs the regex check, runs the length check, exits 0 on pass or 1 with stderr diagnostic on fail.
  - `plugins/dev-process-toolkit/templates/git-hooks/commit-msg.commitlint.sh` — POSIX shell delegator (~10 lines). `exec bunx commitlint --edit "$1"`. If `bunx` is missing on `$PATH`, prints a one-line install hint and exits 1.
  - `plugins/dev-process-toolkit/templates/git-hooks/commitlint.config.js` — `module.exports = { extends: ["@commitlint/config-conventional"] };` (one line of config).
- **Skill prose updates** — string-only edits, no logic added:
  - `plugins/dev-process-toolkit/skills/setup/SKILL.md` — new step "Install commit-msg hook" between the existing CLAUDE.md generation and the spec-templates step. Branches on `--commitlint` arg. Idempotency check: read existing `.git/hooks/commit-msg`, compare hash to template; identical → skip; missing → install; different → diff + prompt.
  - `plugins/dev-process-toolkit/skills/implement/SKILL.md` — Phase 4 step 15 commit-message section rewrites the example commit and adds the type-heuristic + scope-heuristic guidance.
  - `plugins/dev-process-toolkit/skills/ship-milestone/SKILL.md` — release-commit section rewrites the example commit shape.
  - `plugins/dev-process-toolkit/skills/pr/SKILL.md` — PR-title section rewrites the example title shape.
- **Documentation:**
  - `plugins/dev-process-toolkit/templates/CLAUDE.md.template` — appends `## Commit Convention` section.
  - Repo root `CLAUDE.md` — appends `## Commit Convention` section (alongside existing `## Git Commit Messages` user-preferences override; the two are compatible — user prefs constrain *human* style, CC constrains *format*).
  - `README.md` — one sentence in the existing `## Workflow` paragraph noting CC enforcement (no new section; lightweight).
- **No new helper module in `adapters/_shared`** — the hook is a self-contained shell script, not a TypeScript library. The hook test exercises the script directly via `Bun.spawn`.
- **No new gate-check probe** — the deterministic hook *is* the gate. Drift in skill prose (someone edits `/implement` to author a non-CC message) would still produce a non-CC commit, but the hook hard-blocks it locally before push. The hook is the regression catch; the skill prose is the cooperative path.
- **POSIX shell only** — the hook avoids bash-isms (`[[ ]]`, arrays, `$( )` is fine, regex via `grep -E` not bash `=~`). Tested on `dash`, `bash`, `zsh` (macOS default).

## Testing

- **Unit:** `plugins/dev-process-toolkit/tests/conventional-commits-hook.test.ts` exercises the shell script via `Bun.spawn(["sh", scriptPath, fixturePath])` for each of the ≥10 fixtures listed in AC-STE-133.10. Asserts exit code (0 / 1) and, for invalid cases, asserts the rule name appears in stderr.
- **TDD ordering:** Write the test file first (RED — script not yet written → all subprocess calls fail with ENOENT). Then author `commit-msg.sh` (GREEN — fixtures pass). Then refactor.
- **Manual verification gates** at /implement Phase 4:
  - Re-run `/setup` in this repo (clean install path): `.git/hooks/commit-msg` is created, executable, byte-identical to the template.
  - Attempt `git commit -m "broken"` → blocked with `type-prefix-missing` diagnostic.
  - Attempt `git commit -m "feat(setup): install commit-msg hook"` → accepted.
  - Attempt `git commit -m "feat(setup): a subject this long should be rejected because it goes way past seventy two chars"` → blocked with `subject-too-long`.
- **Regression strategy:** the hook is the deterministic gate. If `/implement`/`/ship-milestone`/`/pr` skill prose later drifts (a future FR forgets to honor CC), the hook still hard-blocks the commit locally. No additional probe.
- **Cross-platform note:** the test suite runs on macOS (the dev environment); `sh` on macOS is `bash` invoked in POSIX mode. The test asserts POSIX-shell compatibility by virtue of running `sh` (not `bash`) explicitly.

## Notes

- Severity: medium. New convention surface; cutover affects every future commit. Not high because no history rewrite and the deterministic hook prevents drift.
- Single-FR M36; phase A only. `/ship-milestone M36` closes as a minor bump (`v1.37.0`) — new public surface (commit-msg hook template + `--commitlint` flag).
- M1–M35 commit history is intentionally pre-CC. CHANGELOG cross-references stay as-is; the `## [1.36.1] — 2026-04-27 — "Spine"` style of the CHANGELOG is independent of git commit message format and continues unchanged.
- The repo's existing `## Git Commit Messages` section in CLAUDE.md (user preferences: no Claude Code attribution, no robot emoji, no "You are absolutely right") is a *style* constraint that composes cleanly with CC's *format* constraint — both apply.
- **Future opportunities (out of scope, separately FR-able if needed):**
  - Auto-generated CHANGELOG entries from CC commit log (release-please / cz-cli style). Currently CHANGELOG is hand-written per release-checklist.
  - CI gate (GitHub Action) validating PR commits — second deterministic layer beyond the local hook.
  - Conventional-commit-aware bump suggestion in `/ship-milestone` (e.g., `BREAKING CHANGE:` → major; `feat:` → minor; `fix:`/`chore:`/`docs:` → patch).
  - Migration of M1–M35 commit subjects to CC via `git filter-branch` (decisively rejected per /brainstorm decision; included here only to record the rejection).
