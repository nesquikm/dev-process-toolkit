# STE-308: /spec-review forked-auditor migration {#STE-308}

## Requirement

`/spec-review` today runs entirely in main context. The skill's `allowed-tools: Read, Glob, Grep` declares main-context I/O only; the audit's full reasoning (every AC traced to file:line, every drift candidate evaluated) lands directly in main session tokens. There's no fork, no subagent, no fenced-block hand-off — it predates the STE-225 canonical pattern.

Migrate `/spec-review` to the canonical pattern (M80 FR-A doc, STE-306) as the first conformance example proving the pattern works on a real skill:

1. Introduce a new read-only subagent `spec-reviewer` (`plugins/dev-process-toolkit/agents/spec-reviewer.md`).
2. Introduce a paired forked child skill `spec-review-audit` (`plugins/dev-process-toolkit/skills/spec-review-audit/SKILL.md`) with `context: fork` + `agent: spec-reviewer` + `user-invocable: false`.
3. Child returns a fenced `spec-review-result` block; main `/spec-review` parses it via a deterministic parser and emits the existing user-facing report unchanged.
4. Add `/spec-review` to the STE-307 probe allowlist.

Closest precedent: STE-296 (M77) `tdd-spec-reviewer` migration of `/tdd`'s end-of-FR audit step — same architecture, different audit subject.

## Technical Design

Architecture mirrors `/tdd`'s `tdd-spec-review` stage (STE-296):

- Main `/spec-review` parses arguments, runs tracker-mode probe (existing logic from § 0), then dispatches `Skill('dev-process-toolkit:spec-review-audit')` with prompt body containing the AC list + scope.
- Child `spec-review-audit` runs the existing audit logic in isolation: read `specs/frs/`, scan implementation files, build traceability map, classify each AC as ✓/✗/⚠, scan cross-cutting drift candidates. Emits a single fenced `spec-review-result` block at end of turn.
- Main parses via `parseSpecReviewResultBlock`. On parse OK, renders the user-facing report unchanged (traceability list + status table + drift refresh hint via existing `formatDriftHint`). On parse failure, single bounded retry (mode-D shape from STE-296), then halt with NFR-10 refusal naming the offending field.

Parser shape:

```ts
export interface TraceabilityRow {
  ac: string;
  impl: string | null;
  test: string | null;
  status: 'done' | 'missing' | 'partial';
}

export interface SpecReviewResult {
  traceability: TraceabilityRow[];
  drift_count: number;
  drift_entries: string[];
  advisory_findings: string[];
}

export function parseSpecReviewResultBlock(text: string): SpecReviewResult;
```

`maxTurns: 8` matches `tdd-spec-reviewer`. `model: sonnet` — audit reasoning over potentially many ACs justifies sonnet over haiku; matches `tdd-spec-reviewer`.

Drift-hint threshold (`>= 2`) and literal-line shape stay in `adapters/_shared/src/spec_review_drift_hint.ts` unchanged — the migration only moves WHERE the count is computed (now in the fork), not the threshold logic.
