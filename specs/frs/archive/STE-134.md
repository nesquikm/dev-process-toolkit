# STE-134: /spec-write § 0b step 1: scope ULID-minting to mode: none {#STE-134}

## Requirement

`skills/spec-write/SKILL.md:47` currently reads:

> 1. Mint a ULID via `Provider.mintId()` — always local (AC-STE-20.5), offline-safe.

The "always local" qualifier describes *where* minting happens (offline, no tracker round-trip), not "always perform this step." In tracker mode the step must be skipped entirely:

- `tracker_provider.ts:9-14` (STE-85) declares `TrackerProvider` does NOT implement `IdentityMinter` — `mintId()` on a `Provider`-typed value is a TypeScript error by design.
- `fr_frontmatter.ts:87-91` rejects `buildFRFrontmatter(spec, trackerBinding)` when `spec.id` is also set, with `InvalidFrontmatterInputError` ("tracker mode forbids id: field").
- STE-76 / STE-110 enforce the bimodal frontmatter: tracker-mode FRs carry only `tracker:`, never `id:`.

A reader walking § 0b top-to-bottom in tracker mode reasonably interprets step 1 as universal and prepares to call `Provider.mintId()` — a structural violation. Tighten the prose so step 1 is explicitly mode-none-only and the tracker-mode flow shows skip-mint as the first instruction.

## Acceptance Criteria

- AC-STE-134.1: `skills/spec-write/SKILL.md` § 0b step 1 prose explicitly scopes ULID minting to `mode: none` (e.g., leads with "In `mode: none`:" or carries an inline "(skip in tracker mode)" guard) and references STE-85 / STE-76 as the structural reason.
- AC-STE-134.2: `skills/spec-write/SKILL.md` § 0b reads cleanly in tracker mode — a tracker-mode operator walking the steps top-to-bottom never reaches a step that prescribes `Provider.mintId()`.
- AC-STE-134.3: `/gate-check` passes after the edit (no FR/plan link drift, no probe regressions).

## Technical Design

Single-file prose edit. No code change. No template change.

`skills/spec-write/SKILL.md:47` rewritten to one of two equivalent shapes (implementer's call):

**Option A — bifurcate step 1:**

```
1. **`mode: none` only:** Mint a ULID via `Provider.mintId()` — always local (AC-STE-20.5), offline-safe. **Tracker mode skips this step**: `TrackerProvider` does not implement `IdentityMinter` (STE-85 capability boundary), and `buildFRFrontmatter` rejects `id:` alongside `trackerBinding` (STE-76 / AC-STE-110.2).
```

**Option B — keep step 1 mode-none-scoped, prepend a tracker-mode lead-in above § 0b's numbered list** that says "Tracker mode: skip step 1 entirely; the canonical identity is the tracker ID returned by step 4. Mode-none: proceed with step 1."

Either shape satisfies AC.1 and AC.2; the implementer picks the one that reads better in context.
