# STE-398: Widen AC_LINEAR_XML_RE to tolerate Linear's href attribute + add href test coverage {#STE-398}

## Summary

Linear's MCP server auto-links bare tracker tokens in issue descriptions, wrapping them in an issue-mention element. STE-211 taught the strip helper to unwrap that element back to a plain AC prefix on import. Linear has since started emitting a second attribute inside the wrapper alongside the id, and the strip pattern was written to match only the id-and-nothing-else shape, so it now silently does nothing — imported descriptions keep the raw markup and the AC prefix never comes back.

The unit suite did not catch the drift because every fixture encodes the older single-attribute shape the API no longer produces. This FR widens the pattern to tolerate any attribute set while keeping the match scoped to real AC prefixes, and adds coverage for the new shape on both the unwrap path and the leave-it-alone path.

## Requirement

The 2026-07-19 conformance run confirmed the regression end-to-end against real Linear data: fetching a freshly-pushed issue returned AC lines wrapped as `AC-<issue id="..." href="...">STE-397</issue>.1: ...`, and the shipped `stripLinearACFences` left the wrapper fully intact — the AC prefix was not recovered. The bug is exact and mechanical: `AC_LINEAR_XML_RE` requires the `id="..."` attribute to be immediately followed by `>`, so the added `href` attribute defeats the match on every issue Linear now returns. The fix must widen the attribute portion without loosening the surrounding `AC-` prefix + `.<N>` suffix scope that keeps bare issue references untouched, and must close the test-coverage gap that let an upstream HTML-shape change ship undetected.

## Acceptance Criteria

- AC-STE-398.1: `stripLinearACFences` recovers `AC-STE-203.1:` from an `<issue id="..." href="https://...">STE-203</issue>.1` wrapper (href present, id first). {#AC-STE-398.1}
- AC-STE-398.2: recovery holds regardless of attribute order (href before id) and with additional attributes present. {#AC-STE-398.2}
- AC-STE-398.3: a bare `<issue id="..." href="...">STE-205</issue>` reference NOT preceded by `AC-` / followed by `.N` keeps its wrapper untouched — the targeted scope from STE-211 is preserved under the widened pattern. {#AC-STE-398.3}
- AC-STE-398.4: the round-trip property `stripLinearACFences(formatLinearDescription(x)) === x` still holds for every existing AC-prefix body (no regression on the format+strip inverse). {#AC-STE-398.4}
- AC-STE-398.5: the test suite adds href-carrying fixtures for the strip path (id+href, href+id, extra attributes) and the bare-reference negative case — the coverage gap that let the upstream HTML-shape change ship undetected. {#AC-STE-398.5}

## Technical Design

Single-file change in `plugins/dev-process-toolkit/adapters/linear/src/format_description.ts`.

Current:

```ts
const AC_LINEAR_XML_RE = /\bAC-<issue id="[^"]*">([A-Z]+-\d+)<\/issue>(\.\d+)\b/g;
```

Widen the attribute portion to accept any attribute set while still requiring the `AC-` prefix and the `(\.\d+)` suffix (the two anchors that keep bare references — `Refs: STE-205` — untouched):

```ts
const AC_LINEAR_XML_RE = /\bAC-<issue\s[^>]*>([A-Z]+-\d+)<\/issue>(\.\d+)\b/g;
```

`<issue\s[^>]*>` requires at least one whitespace-separated attribute after `issue` and consumes everything up to the closing `>`; `[^>]*` cannot cross a `>` so it stays inside the single tag. The capture groups `([A-Z]+-\d+)` and `(\.\d+)` are unchanged, so the replacement `"AC-$1$2"` is unchanged. `formatLinearDescription` / `AC_PLAIN_RE` / `AC_BACKTICK_RE` are untouched — the regression is strip-side only.

The finding noted a possible second leg (the push-side preventive fence perhaps not applied for this FR); that is inference from a stored description, not confirmed, and is out of scope here — the strip-side break is definitive and self-contained.

## Testing

Extend `plugins/dev-process-toolkit/adapters/linear/src/format_description.test.ts`:

- strip recovers the prefix from `- AC-<issue id="752f4657-..." href="https://linear.app/...">STE-203</issue>.1: foo` (id-first + href) → `- AC-STE-203.1: foo` (AC-STE-398.1).
- strip recovers with attribute order reversed (`href="..." id="..."`) and with an extra attribute (e.g. a `data-*`), proving order-independence (AC-STE-398.2).
- a bare `See <issue id="abc" href="https://...">STE-205</issue> for details.` (no `AC-`/`.N`) is returned unchanged (AC-STE-398.3).
- keep the existing id-only XML fixture green (back-compat: Linear may still emit the old shape for older content).
- extend the round-trip `strip(format(x)) === x` input set — the property holds because `format` never emits the XML form, so the widened strip does not perturb it (AC-STE-398.4).

The href-carrying fixtures ARE the AC-STE-398.5 deliverable — the coverage that would have caught the upstream shape change.
