# STE-191: /smoke-test pre-flight #8 single-hit name-match KEY surfacing {#STE-191}

## Requirement

`/smoke-test` pre-flight #8 verifies that the configured Jira Space (project) KEY is visible to the authenticated principal:

```
mcp__atlassian__getVisibleJiraProjects(searchString=<KEY>)
→ inspect response.values[].key for an exact match against <KEY>
```

The 2026-05-01 Jira run 1 driver-side caveats document a usability foot-gun: pre-flight #8 fired twice on incorrect operator inputs before resolving:

1. First invocation passed `--jira-project DPT` — matched the project NAME via `searchString=DPT` but not `values[].key`. Refused per spec.
2. Second invocation passed `--jira-project "DPT Smoke Test"` — display name, not KEY. Refused per spec.
3. Third invocation passed `--jira-project DST` (the actual KEY). Resolved.

The repeated NAME-vs-KEY confusion suggests the visibility-refusal message could call out the matching Space's actual KEY when `values[]` returns exactly one hit — the operator's intent is unambiguous in that case, and surfacing the resolved KEY removes the round-trip cost of guessing again.

This FR adds a single-hit name-match branch to pre-flight #8's refusal text.

Out of scope:

- Auto-correcting the operator's input (silently substituting the matching KEY). The refusal is correct as a defensive rail; this FR only sharpens the message, not the gate.
- Multi-hit cases (operator searched a substring matching multiple Spaces). The refusal stays generic — guessing one of N is worse than asking for clarification.
- Linear-path equivalent. Linear teams are KEY-only resolved (`mcp__linear__list_teams` already takes both name and key); the same NAME/KEY confusion isn't a frequent driver issue on Linear.

## Acceptance Criteria

- AC-STE-191.1: `.claude/skills/smoke-test/SKILL.md` pre-flight #8 refusal text gains a conditional: when `mcp__atlassian__getVisibleJiraProjects(searchString=<X>)` returns **exactly one** `values[]` entry whose `name` matched `<X>` but whose `key` did NOT match `<X>`, the refusal message includes the matching Space's actual KEY: `"<X>" matched Space "<matched-name>" by display name, but the KEY is "<matched-key>". Pass --jira-project <matched-key> and retry.` {#AC-STE-191.1}
- AC-STE-191.2: When `values[]` is empty (Space genuinely not visible) OR returns multiple hits (ambiguous match), the refusal stays at the current generic shape: `getVisibleJiraProjects(searchString=<X>) → response.values[].key did not contain '<X>'`. No KEY surfacing in the multi-hit / no-hit cases. {#AC-STE-191.2}
- AC-STE-191.3: SKILL.md prose at pre-flight #8 documents the conditional shape and the rationale: single-hit name-match is the unambiguous case where the operator clearly meant the matched Space; surfacing the KEY is a cheap usability win that preserves the defensive refusal. {#AC-STE-191.3}
- AC-STE-191.4: Smoke chain Jira post-edit: an operator who passes the project's NAME (not KEY) by mistake gets a refusal that names the actual KEY; rerun with the named KEY succeeds. Operator-deferred — ships `[~]`; M53 retroactive validation (the natural NAME-vs-KEY-typo regression scenario). {#AC-STE-191.4}

### Refusal-text shape

Current pre-flight #8 single message:

```
Probe: mcp__atlassian__getVisibleJiraProjects(searchString=<X>) → response.values[].key did not contain '<X>'.
```

After this FR, pre-flight #8 emits one of three messages depending on the response shape:

| Response | Message |
|----------|---------|
| `values[]` empty | (current) `getVisibleJiraProjects(searchString=<X>) → response.values[].key did not contain '<X>'`. |
| `values[]` has exactly one entry whose `name` matched `<X>` (and whose `key` did NOT) | `'<X>' matched Space "<matched-name>" by display name, but the KEY is "<matched-key>". Pass --jira-project <matched-key> and retry.` |
| `values[]` has 2+ entries | (current) — generic refusal; surfacing one KEY out of N would be misleading. |

### Why the gate stays in place

The refusal is the defensive rail; the message is the operator-friendly hint. The gate must still refuse the bad input — silent auto-correction would mask cases where the operator named the wrong Space entirely. The KEY surfacing only fires when the single-hit case makes intent unambiguous.

- **Adapter-side equivalent.** The Jira adapter's `upsertTicketMetadata` and `transitionStatus` paths take a Jira issue ID, not a project KEY — they don't share this NAME-vs-KEY confusion. Pre-flight #8 is the only place in the toolkit that resolves a project KEY from operator input.
