# STE-176: /smoke-test coverage extension — pre-flight #9 ghost detector + Jira comment-path probe {#STE-176}

## Requirement

Two driver-side coverage extensions for `.claude/skills/smoke-test/SKILL.md` that close gaps smoke #6 (Jira) called out:

1. **Pre-flight #9 ghost detector** (Jira-only). Smoke #6 F5 noted that orphaned `dpt-smoke`-labeled non-`Done` work items accumulate in DST across runs (DST-2, DST-3, DST-4 are all earlier "greet helper" stubs in `Done` status; if any future run lands a `dpt-smoke` issue that doesn't get teardown-transitioned, it sticks). Phase 5 teardown is correctly scoped to the run's own start window, but the cumulative ghost cluster grows with every aborted or partial run. An optional pre-flight that JQL's the project for `labels = "dpt-smoke" AND status != "Done"` and warns when the count crosses a threshold gives the operator the signal to do a one-time sweep.
2. **AC-STE-154.9 AC 6 comment-path coverage.** Smoke #6 explicitly noted: "AC 6 (comment) NOT exercised by canonical chain. **Recommend an explicit smoke-test extension or a separate manual probe to close AC 6.**" The `mcp__atlassian__addCommentToJiraIssue` MCP call has no live smoke probe today; a regression there would slip past every future smoke run. A stand-alone Phase 2 step that issues an `addCommentToJiraIssue` against the run's freshly-created work item closes the gap.

The chosen approach for AC 6 coverage is a **stand-alone probe**, not a side effect of `/implement`'s claim/release narration (resolved during M49 spec authoring; rationale: stand-alone validates the MCP tool independent of `/implement`'s narration policy, which could change without breaking the underlying tool contract).

Out of scope:
- Linear-side coverage extension — Linear's MCP surface is well-exercised by the canonical chain; no analogous gap.
- Auto-sweeping the ghost cluster — pre-flight #9 only warns; the sweep stays operator-explicit.

## Acceptance Criteria

- AC-STE-176.1: `.claude/skills/smoke-test/SKILL.md` adds an optional Jira-only pre-flight #9 — runs JQL `project = <KEY> AND labels = "dpt-smoke" AND status != "Done"` (where `<KEY>` is the resolved `--jira-project` arg), warns (does not refuse) when the count exceeds a threshold (default 5; configurable via prose). Linear path skips the probe entirely. Output line shape: `pre-flight #9: N orphaned dpt-smoke items in <KEY> (status != Done) — consider one-time sweep before next run.`.
- AC-STE-176.2: A new Phase 2 step on the Jira branch issues a stand-alone `mcp__atlassian__addCommentToJiraIssue` probe against the run's freshly-created work item — body shape: `Smoke probe — AC-STE-154.9 AC 6 coverage. Run: <date> <tracker> v<plugin-version>.`. Comment surfaces in the per-skill log; canonical chain is unchanged on the Linear branch (Phase 2 step skipped).
- AC-STE-176.3: The smoke-test runs table at the bottom of `SKILL.md` carries a column or per-entry annotation that records "comment exercised" for runs after STE-176 ships, confirming AC-STE-154.9 AC 6 is now covered.
- AC-STE-176.4: `bun test` green; `/gate-check` clean.

## Technical Design

Two discrete edits in `.claude/skills/smoke-test/SKILL.md`:

1. **Pre-flight #9 (Jira-only).** Add to the pre-flight refusals section. Reference shape:

   ```markdown
   ### Pre-flight #9 — Orphaned dpt-smoke ghost cluster (Jira-only)

   Optional warning probe. Fires only when `--tracker jira` is active.

   ```
   GHOST_COUNT=$(mcp__atlassian__searchJiraIssuesUsingJql \
     jql='project = '$JIRA_PROJECT' AND labels = "dpt-smoke" AND status != "Done"' \
     | jq '.issues | length')
   ```

   When `GHOST_COUNT > 5`, emit a one-line warn:

   `pre-flight #9: $GHOST_COUNT orphaned dpt-smoke items in $JIRA_PROJECT (status != Done) — consider one-time sweep before next run.`

   Does not refuse — the run continues. Operator runs the sweep at their discretion.
   ```

2. **Phase 2 comment probe.** Add a new step to Phase 2 (Jira branch). The probe runs after the canonical chain's `/implement` step but before Phase 3 (gate-check), so the work item exists and is in `Done` state. Reference shape:

   ```markdown
   ### Phase 2 — Comment-path probe (Jira-only)

   Stand-alone probe to exercise AC-STE-154.9 AC 6 coverage. Fires only when `--tracker jira` is active.

   ```
   mcp__atlassian__addCommentToJiraIssue \
     issueIdOrKey="$LATEST_DST_ID" \
     comment="Smoke probe — AC-STE-154.9 AC 6 coverage. Run: $DATE $TRACKER v$PLUGIN_VERSION."
   ```

   The comment is the canonical evidence for AC 6; it surfaces in the per-skill log and the run's audit trail.
   ```

3. **Runs-table annotation.** Add a new column or extend the "Outcome" column rendering: post-STE-176 runs include `comment exercised` somewhere in the entry to confirm AC 6 coverage.

No new code, no new test files. Live-verification gate is the next smoke run.
