# STE-154: Jira adapter: Phase H live-MCP conformance corrections {#STE-154}

## Requirement

`adapters/jira.md` was authored without a live Atlassian MCP. Smoke-test run #5 against `stellartechlab.atlassian.net` on 2026-04-29 (via the Atlassian Rovo MCP just installed in `.mcp.json`) surfaced 7 discrepancies between the documented behavior and the live tool surface. The most severe (F7 — wrong MCP tool names) means the adapter as written cannot dispatch a single call. This FR closes Phase H Task 2 (Tier 5 manual conformance).

## Acceptance Criteria

- AC-STE-154.1: MCP tool table in `adapters/jira.md` uses live camelCase names: `getJiraIssue`, `editJiraIssue`, `transitionJiraIssue`, `getTransitionsForJiraIssue`, `createJiraIssue`, `addCommentToJiraIssue`.
- AC-STE-154.2: `transition_status` matches by `to.statusCategory.key` (`new` / `indeterminate` / `done`) as fallback when no exact `to.name` match exists. `in_review` collapses to `indeterminate` (documented).
- AC-STE-154.3: Existing top-level Schema L key `jira_ac_field:` extended to accept the sentinel value `description` alongside `customfield_XXXXX`. `/setup` prompts the user when `discover_field.ts` returns `{ ok: false }` and persists `jira_ac_field: description` (zero-config path) or `jira_ac_field: customfield_XXXXX` (user creates the custom field) based on the choice. No new canonical key; no implicit fallback.
- AC-STE-154.4: `pull_acs` / `push_ac_toggle` branch on the value of `jira_ac_field` — when it equals `description`, read/write the AC bullets under the `## Acceptance Criteria` heading in the issue's description body; otherwise treat the value as a custom-field GID and read/write that field.
- AC-STE-154.5: `upsert_ticket_metadata` defaults to `issuetype: "Task"`. Optional `jira_issue_type:` in `### Jira` block overrides.
- AC-STE-154.6: `pull_acs` / `push_ac_toggle` / `upsert_ticket_metadata` document `contentFormat: "markdown"` parameter usage.
- AC-STE-154.7: `/setup` for `mode: jira` documents manual Space pre-creation; validates `project_key` visibility via `getVisibleJiraProjects` and refuses with NFR-10 canonical shape if absent.
- AC-STE-154.8: All "Status: provisional (Phase H conformance)" warning blocks removed; replaced with one-line "Verified against live MCP 2026-04-29 (smoke-test #5)" provenance.
- AC-STE-154.9: Smoke-test #5 re-run captures all 7 corrections E2E against `DST` Space: create → assign → transition (via category fallback) → comment → cleanup. Trace persisted under `specs/notes/`.
- AC-STE-154.10: `/gate-check` passes; no spec-vs-code drift.

## Technical Design

No Jira-specific dispatch code in `adapters/_shared/src/tracker_provider.ts` yet (verified by grep — 2 hits, both comments/types). Surface:

1. **`adapters/jira.md`** — bulk rewrite (frontmatter + tool table + ops).
2. **`adapters/jira/src/discover_field.ts`** — no logic change; `{ ok: false }` becomes a first-class supported signal that `/setup` interprets as prompt.
3. **`/setup` skill** — extend AC-discovery branch to prompt + persist `jira_ac_field: description | customfield_XXXXX` (existing top-level Schema L key, extended-value).
4. **TrackerProvider Jira impl (when materialized)** — follows corrected spec; no retroactive refactor since no concrete impl exists yet.

Status-mapping algorithm:

```
resolveTransitionId(targetStatus, transitions, statusMapping):
  targetName = statusMapping[targetStatus]
  for t in transitions:
    if t.to.name == targetName: return t.id          # primary
  targetCategory = canonicalCategory(targetStatus)   # fallback
  for t in transitions:
    if t.to.statusCategory.key == targetCategory: return t.id
  raise NFR10(...)
```

`canonicalCategory`: `in_progress | in_review → indeterminate`, `done → done`. Workflows with a real "In Review" status get exact-name precedence; workflows without one collapse the two — documented.

**Out of scope:** self-hosted Jira (already v1-out), Confluence/Compass integration (smoke confirmed available, separate FR), `deleteJiraIssue` (MCP doesn't expose; archival uses status transition, not delete), OAuth-scope tightening (separate FR).
