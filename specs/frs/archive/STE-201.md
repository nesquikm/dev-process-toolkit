# STE-201: /ship-milestone pre-flight walks **Tasks:** checklist (not just FR rows) {#STE-201}

## Requirement

`/ship-milestone` pre-flight refusal #1 walks FR-row links only. A plan with unchecked `**Tasks:**` bullets but archived FR rows passes pre-flight cleanly — the skill happily tags a release for a milestone whose plan describes uncompleted scope.

**Findings bundled:** F3.12 from `/tmp/dpt-findings-2026-05-04.md`.

**Symptom observed in `dpt_test_lib`:** `specs/plan/M2.md` listed three tasks (`subtract`, `multiply`, `divide`) with one FR (STE-193) backing only `subtract`. After STE-193 shipped, pre-flight #1 (which walks FR rows in the plan's FR table) saw every FR-linked row as archived and proceeded. `/ship-milestone M2` would then bump `pubspec.yaml`, write a CHANGELOG entry titled "M2", and commit a release tagged "M2 complete" while two of three plan tasks were demonstrably unfinished and the corresponding code (`multiply`, `divide`) was absent.

**Root cause.** `ship-milestone/SKILL.md:23–25` walks the **FR rows** in the plan's FR table — it does NOT examine the `**Tasks:**` checkbox list. So the FR-completeness invariant and the task-completeness invariant diverge silently.

## Acceptance Criteria

- AC-STE-201.1: `/ship-milestone` pre-flight #1 (or a new pre-flight #2 if separation helps) walks the `**Tasks:**` block in addition to the FR rows. For each unchecked `[ ]` bullet whose corresponding FR row is missing or its FR is unbacked / not yet archived, accumulate into the refusal list. {#AC-STE-201.1}
- AC-STE-201.2: NFR-10 canonical refusal shape names the count + ordered list of unbacked tasks + remedy. Concrete shape:

  ```
  /ship-milestone: milestone M<N> plan has <count> unchecked task(s) with no FR backing:
  - <task-1>
  - <task-2>
  Cannot tag a release for a milestone whose plan describes uncompleted scope.
  Remedy: either /spec-write FRs for the unbacked tasks (then /implement them), or
  move the tasks to a later milestone's plan, then re-run /ship-milestone.
  Context: milestone=M<N>, unbacked-tasks=<count>, skill=ship-milestone
  ```
  {#AC-STE-201.2}
- AC-STE-201.3: A task explicitly marked `[deferred]` (e.g., `- [deferred] multiply — moved to M3`) is treated as ship-OK; the marker is the operator's signal that the task is intentionally deferred and the milestone is shippable as-is. Documented in the skill prose with the trigger token. {#AC-STE-201.3}
- AC-STE-201.4: New `/gate-check` probe `plan-task-fr-coverage` flags the divergence at ADVISORY: every plan task should resolve to an FR file or be marked `[deferred]`. The probe runs against active and in-flight milestones; it does not retroactively flag legacy archived plans. {#AC-STE-201.4}
- AC-STE-201.5: Smoke regression fixture: plan with FR row archived + one unchecked task bullet without `[deferred]` → `/ship-milestone M<N>` refuses with the AC-STE-201.2 shape. {#AC-STE-201.5}

## Technical Design

**Pre-flight #1 logic (revised):**

```
fr_rows = parse_fr_table(plan)
task_bullets = parse_tasks_block(plan)
unbacked_tasks = []

for bullet in task_bullets:
  if bullet.checked or bullet.marker == "[deferred]":
    continue
  matching_fr_row = find_fr_for_task(bullet, fr_rows)
  if not matching_fr_row or matching_fr_row.fr.status == "active":
    unbacked_tasks.append(bullet)

if unbacked_tasks:
  refuse with NFR-10 shape (AC-STE-201.2 template)

# fr_rows pre-flight (existing): walk fr_rows, refuse if any status: active
```

**Task-to-FR matching.** Heuristic: a task bullet matches an FR row if the task's leading verb-phrase appears in the FR title (case-insensitive substring). Operators can also explicitly link a task to an FR via inline notation: `- [ ] subtract — STE-193`. The explicit link wins over the heuristic.

**`[deferred]` marker.** Same parse path as `[x]` and `[ ]`. Recognized by both `/ship-milestone` pre-flight (ship-OK) and `/gate-check` probe (skip from coverage check). Documented in `templates/spec-templates/plan.md.template` Task Sizing section.

**Probe wiring.** `/gate-check` probe `plan-task-fr-coverage` iterates over `specs/plan/M*.md` (active only), parses Tasks blocks, parses FR table, computes the divergence set, surfaces ADVISORY rows for active milestones with un-backed unchecked tasks.

## Testing

- Unit test: pre-flight refusal correctly accumulates unbacked tasks; refusal shape matches AC-STE-201.2 verbatim.
- Unit test: `[deferred]` marker exempts the task from refusal; `[x]` exempts; `[ ]` (without backing FR) refuses.
- Integration test (smoke regression): plan file with `subtract` (FR archived) + `multiply` `[ ]` + `divide` `[ ]` → `/ship-milestone M2` refuses naming `multiply` and `divide`.
- Integration test: same plan with `multiply [deferred]` and `divide [deferred]` → `/ship-milestone M2` proceeds.
- Probe test: `/gate-check` ADVISORY fires on the same plan; passes when tasks are marked `[deferred]` or backed by archived FRs.

## Notes

- F3.12's smoke-tester suggested AC.1 of this FR (refuse `/spec-write` to add a plan task without an FR file). That alternative was rejected during brainstorm — too restrictive for incremental planning where tasks may legitimately predate FR allocation. The chosen design (refuse at `/ship-milestone` time + ADVISORY at `/gate-check`) preserves the incremental-planning flexibility.
- Existing legacy plans with multi-task scoping (some bundled FRs cover multiple tasks) may need `[deferred]` markers added retroactively. One-time migration is documented in CHANGELOG; not a blocker for this FR.
- Capability: a tracker-mode `/spec-write` run that creates this FR surfaces `filename_policy_override` per the unconditional rule.
