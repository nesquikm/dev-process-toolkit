# STE-408: Make the non-tty requires-input refusal machine-recognizable + fix tty-detection (F5/F3) {#STE-408}

## Summary

When a skill correctly refuses under a headless session, the harness cannot tell that refusal apart from a skill that simply did nothing — both read as vacuous — because the refusal comes out as prose narration rather than a structured signal the stream parser recognizes. So the Phase-8 contract can catch the bad case (a magpie bypass) but cannot render the good case (a correct refusal) as a pass.

This FR ships the deterministic half of the fix: a canonical byte-checkable refusal marker carried in the refusal error message, and stream-parser recognition that maps that marker in assistant text to a refusal — so a surfaced refusal renders as a pass. It also corrects a small tty-detection bug that misreads a heredoc stdin as interactive.

## Requirement

The 2026-07-20 v2.52.0 conformance re-run validated every M110 fix behaviorally (STE-404 held — spec-write refused the magpie bypass and made no write) but surfaced two non-tty-refusal-path defects:

**F5 (HIGH).** Phase 8's `assertFirstTurnShape` returned `vacuous` for all four in-scope skills. Under claude -p there is no `AskUserQuestion` surface, and `requireOrRefuse`'s refusal is a PROSE narration, not a structured stream event — the parser only maps a `system/refusal` event, an assistant `stop_reason: refusal`, or a `RequiresInputRefusedError` tool_use to a refusal entry. So a correct refusal is indistinguishable from "did nothing" (both `vacuous`, the STE-399 non-pass). The magpie VIOLATION is still caught (tracker-create / scaffold name+path checks fire), but the positive PASS cannot render — the STE-220 prose-vs-structured lesson resurfacing on the Phase 8 surface.

**F3 (low, real bug).** `requires_input`'s `isStdinNonTty()` returns `process.stdin.isTTY === false`, but under a heredoc / subshell stdin `isTTY` is `undefined`, so the strict `=== false` misses it and the NFR-10 Context line misreports `stdin=tty` under claude -p. Cosmetic in the observed path (the refusal still fired), but a latent risk for any three-branch resolver that routes an `undefined` isTTY to the interactive branch.

Honest bound: whether the runtime LLM SURFACES the structured error (vs narrating prose) under -p is a runtime-adherence question the plugin cannot fully force. This FR ships the deterministic infrastructure (marker + parser recognition) plus the SKILL contract to surface it, so a correctly-surfaced refusal is machine-recognizable.

## Acceptance Criteria

- AC-STE-408.1: `isStdinNonTty()` in `adapters/_shared/src/requires_input.ts` returns true when `process.stdin.isTTY !== true` (covering both `false` and `undefined`) — a heredoc / subshell stdin under claude -p is correctly non-tty; a genuine interactive tty (`isTTY === true`) stays interactive. {#AC-STE-408.1}
- AC-STE-408.2: the `RequiresInputRefusedError` NFR-10 message carries a canonical byte-checkable refusal marker `<dpt:requires-input-refused>v1</dpt:requires-input-refused>`, exported as a constant (`REQUIRES_INPUT_REFUSED_MARKER`). {#AC-STE-408.2}
- AC-STE-408.3: `parseStreamJsonLine` emits a `{ type: "refusal" }` entry for an assistant text block containing the refusal marker, so `assertFirstTurnShape` returns `ok-refused` (not `vacuous`) for a surfaced requires-input refusal. {#AC-STE-408.3}
- AC-STE-408.4: the marker is recognized in ASSISTANT text only — a marker appearing solely in a user / tool_result event (e.g. the SKILL body documenting it) does NOT emit a refusal entry (no false positive). {#AC-STE-408.4}
- AC-STE-408.5: the `/spec-write` SKILL first-action prose documents that a non-tty refusal MUST surface the `RequiresInputRefusedError` message (carrying the marker) so the refusal is machine-recognizable — no prose-only narration. {#AC-STE-408.5}

## Technical Design

**`requires_input.ts`.** (F3) `isStdinNonTty()`: `return (process.stdin as { isTTY?: boolean }).isTTY !== true;` + update the docstring (undefined and false both count as non-tty; only an explicit `true` is interactive). (F5) export `REQUIRES_INPUT_REFUSED_MARKER = "<dpt:requires-input-refused>v1</dpt:requires-input-refused>"`; `buildRefusalMessage` appends the marker to the returned message (a fourth joined line) so any surfacing of the error emits it.

**`socratic_first_turn_stream.ts`.** (F5) In `parseStreamJsonLine`, when projecting an assistant `text` block, if `block.text` includes `REQUIRES_INPUT_REFUSED_MARKER`, push `{ type: "refusal" }` instead of `{ type: "text" }`. Only assistant text blocks are scanned (the parser already ignores user / tool_result events), so a marker in the SKILL body (a user event) never triggers a false refusal (AC-STE-408.4). `assertFirstTurnShape` already maps a `refusal` entry to `ok-refused`, so no change there.

**`skills/spec-write/SKILL.md`.** (F5) At the § FIRST ACTION contract, append a sentence: under non-tty, a refusal MUST surface the `RequiresInputRefusedError` message (which carries the `<dpt:requires-input-refused>v1</dpt:requires-input-refused>` marker) so it is machine-recognizable — a prose-only refusal reads as `vacuous`. Append to the existing line to hold the SKILL line cap.

No new numbered gate probe — the deterministic enforcement is the unit tests (parser recognition + tty predicate), which run in the gate. The smoke Phase 8 driver (project-local) benefits automatically once the parser recognizes the marker.

## Testing

- `requires_input.test.ts` (or a new sibling): `isStdinNonTty()` returns true for `isTTY === undefined` and `isTTY === false`, false for `isTTY === true` (AC-STE-408.1); the refusal message contains `REQUIRES_INPUT_REFUSED_MARKER` (AC-STE-408.2).
- `socratic_first_turn_stream.test.ts`: an assistant text block containing the marker projects to a `refusal` entry (AC-STE-408.3); end-to-end, a transcript whose only "action" is a marker-bearing assistant refusal → `assertFirstTurnShape` returns `ok-refused` (not `vacuous`); a marker in a user / tool_result event projects to NO refusal (AC-STE-408.4).
- Full `bun test` green; no existing socratic/requires-input test regresses.
