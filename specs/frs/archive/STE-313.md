# STE-313: Refuse first-turn gates under non-tty stdin when auto-approve marker absent {#STE-313}

## Requirement

Under `claude -p` (non-tty stdin), the literal byte-string `<dpt:auto-approve>v1</dpt:auto-approve>` (the marker) MUST be the **sole** trigger for auto-applying first-turn gate decisions. Currently, the marker check is paraphrased away — when the marker is absent, both `/spec-write` (draft gate § 0b step 4, branch gate § 7a) and `/setup` (Socratic first-turn scaffold-Write ban, STE-237) auto-apply by rationalizing the harness's `<system-reminder>` ("work without stopping for clarifying questions") as license to proceed.

Verified by `/conformance-loop` iteration 1 (2026-05-19):

- Linear leg F1 — `/spec-write` Group 1b: marker absent → child auto-applied → `STE-311` committed on master (`3cca70a`). Child closing prose: *"proceeded under the harness's autonomous-mode reminder to honor the typed instruction."*
- Linear leg F2a — `/spec-write` Group 5b: marker absent → child auto-applied branch gate decision → `STE-312` committed (`e30bedb`). Same root cause as F1, different gate site.
- Jira leg F1 — `/setup` Phase 8: marker absent → scaffold `Write` fired at `tool_use index=12` BEFORE any `AskUserQuestion`. `assertFirstTurnShape` raised `SocraticFirstTurnViolationError`. Fixture: `plugins/dev-process-toolkit/tests/fixtures/socratic-first-turn/setup-2026-05-19.json` (112,572 bytes).
- Jira leg F2 — `/spec-write` fixtures 1b + 5b: same regression class, cross-tracker confirmation.

STE-294 AC.4 (archived) explicitly fences this regression shape in skill text, but skill text is not byte-checkable — the LLM reasons past it. STE-237 + STE-251 + STE-262 establish the deterministic mechanisms (`check_marker_runtime.ts`, `assertFirstTurnShape`, `requireOrRefuse`); the gap is that nothing forces the LLM to actually call them on every gate site before any side effect.

**Desired behavior:** marker absent + `process.stdin.isTTY === false` ⇒ `RequiresInputRefusedError` (NFR-10 canonical shape, naming the gate site), zero side effects (no `Provider.sync`, no FR write, no Linear/Jira ticket, no `git commit`). Marker present + non-tty ⇒ auto-apply as today. TTY path byte-identical to v2.27.0.

## Acceptance Criteria

- AC-STE-313.1: `/spec-write` draft gate (§ 0b step 4) — under non-tty stdin + marker absent, the skill MUST exit with `RequiresInputRefusedError` (NFR-10 shape, `gate_site=draft`) before any `Provider.sync(spec)` invocation. The Group 1b fixture replay (verbatim prompt body captured in conformance-loop F1) MUST refuse on a fresh `claude -p` invocation.
- AC-STE-313.2: `/spec-write` branch gate (§ 7a `requireCommittableBranch`) — under non-tty stdin + marker absent, the gate MUST exit with `RequiresInputRefusedError` (NFR-10 shape, `gate_site=branch`) before any `git checkout -b`. The Group 5b fixture replay MUST refuse.
- AC-STE-313.3: `/setup` Socratic first-turn (STE-237) — under non-tty stdin + marker absent, the FIRST tool call MUST be `AskUserQuestion` `tool_use` OR `RequiresInputRefusedError` raise. `Write` / `Edit` / `NotebookEdit` before either is a hard fail. The setup-2026-05-19.json fixture replay MUST refuse, not scaffold.
- AC-STE-313.4: Three verbatim regression fixtures (Group 1b prompt body, Group 5b prompt body, setup-2026-05-19.json) land in `plugins/dev-process-toolkit/tests/fixtures/marker-absent-non-tty/`. A new test loop in `plugins/dev-process-toolkit/tests/marker-absent-non-tty.test.ts` replays each via `claude -p` (or a stubbed `requireOrRefuse` harness call) and asserts the refusal class fires.
- AC-STE-313.5: `<system-reminder>` text containing `"work without stopping"`, `"autonomous-mode"`, `"standing instruction"`, pre-baked `<command-args>` prose, and `claude -p` non-interactive stdin inference are NOT acceptable triggers for auto-applying any of the three gates above. The byte-checkable runtime byte-grep against `check_marker_runtime.ts` is the SOLE evaluation path. A § Rules `NOT-a-trigger` anchor lands in both `skills/spec-write/SKILL.md` AND `skills/setup/SKILL.md` and is grep-fenced by a new `/gate-check` probe `not_a_trigger_anchor_present` (severity: error).
- AC-STE-313.6: `/gate-check` adds probe `marker_helper_invoked_per_gate` that scans the Bash-tool transcript of `/spec-write` and `/setup` runs (via stream-json capture path from F6 mitigation, or via a session-log hook) and refuses with NFR-10 if any marker-gated decision proceeded without a `check_marker_runtime.ts` invocation. Probe severity: error.

## Technical Design

Shipped enforcement: vectors (2) test-loop + (3) /gate-check probe; vector (1) PreToolUse hook deferred to a follow-up FR (rationale below). All three vectors target the same regression-class fence — the LLM rationalizing the harness `<system-reminder>` as a marker substitute under non-tty stdin.

1. **Pre-tool-use hook — deferred.** A `PreToolUse` hook that gates `Write` / `Edit` / `NotebookEdit` on a `check_marker_runtime.ts` invocation would be the strongest runtime backstop, but the ACs require behavior (`MUST exit with RequiresInputRefusedError`), not a specific enforcement vector. The existing two-step pattern at `/spec-write` § 0b step 4 + § 7a and `/setup` Socratic first-turn already produces byte-identical refusal via `check_marker_runtime` + `requireOrRefuse`. A follow-up FR will wire the new consolidated arbiter `evaluateGateMarkerRefusal` (see below) into both SKILL.md gate sites and add the hook as a runtime backstop.

2. **Runtime arbiter + regression-fixture test loop — shipped (this FR).** New consolidated single-call arbiter at `adapters/_shared/src/gate_marker_refusal.ts` (`evaluateGateMarkerRefusal({promptBody, isTty, gateSite})`) collapses the four-state matrix `(marker × stdin)` into `apply` / `prompt` / `RequiresInputRefusedError`, naming the gate site (`draft` / `branch` / `setup-socratic`) in the NFR-10-shape refusal message. The arbiter delegates to the same primitives the SKILL.md two-step pattern uses (`checkMarkerRuntime` byte-grep + `RequiresInputRefusedError` constructor with the canonical NFR-10 shape from `requires_input.ts`) — byte-identical refusal behavior. The AC-STE-313.4 test loop at `tests/marker-absent-non-tty.test.ts` replays three verbatim conformance-loop fixture bodies (Group 1b, Group 5b, `setup-2026-05-19.json`) through the arbiter + `assertFirstTurnShape` and asserts the refusal class fires with the correct gate site.

3. **`/gate-check` probes (AC-STE-313.5 + AC-STE-313.6) — shipped (this FR).** Two new error-severity probes:
   - `#55 not_a_trigger_anchor_present` at `adapters/_shared/src/not_a_trigger_anchor_present.ts` — globs `/spec-write` + `/setup` SKILL.md `## Rules` body for the five forbidden trigger phrases listed in AC-STE-313.5 plus the canonical helper-path literal; any missing phrase or missing helper-path reference is a violation.
   - `#56 marker_helper_invoked_per_gate` at `adapters/_shared/src/marker_helper_invoked_per_gate.ts` — NDJSON session-log walker that pairs every gate-decision token (`spec_write_draft_default_applied`, `spec_write_commit_default_applied`, `branch_gate_default_applied`, `setup_socratic_first_turn_*`) with a preceding `bun run …/check_marker_runtime.ts` Bash invocation in the same session. Refusal events bypass the pairing requirement (refusal IS evidence the helper fired). Vacuous when no session log path is supplied — the F6 stream-json capture path provides that path; until F6 lands, the probe ships as severity:error but cannot fire on prompt-bearing children (per AC-STE-313.6 acknowledgment + § Out of Scope).

Helper paths consumed by this FR:

- `plugins/dev-process-toolkit/adapters/_shared/src/check_marker_runtime.ts` (STE-262) — byte-grep primitive
- `plugins/dev-process-toolkit/adapters/_shared/src/requires_input.ts` → `requireOrRefuse(...)` + `RequiresInputRefusedError` (STE-251) — refusal-shape primitive
- `plugins/dev-process-toolkit/adapters/_shared/src/socratic_first_turn.ts` → `assertFirstTurnShape(...)` (STE-237) + `socratic_first_turn_stream.ts` → `parseStreamJsonTranscript(...)` — Socratic first-turn fixture replay primitive

Fixture path: `plugins/dev-process-toolkit/tests/fixtures/marker-absent-non-tty/` — the three captured bodies live here, NOT in the per-date sibling `socratic-first-turn/` path (same overwrite-protection convention as `socratic-first-turn/regression/`).
