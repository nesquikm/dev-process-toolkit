# STE-113: /setup Bun stack guidance — handle bun test zero-match exit 1 {#STE-113}

**Smoke-test findings ref:** F1, F4 in `/tmp/dpt-smoke-findings-2026-04-25.md` (2026-04-25 re-run against v1.29.0)

## Requirement

The 2026-04-25 smoke test of v1.29.0 against a fresh Bun project surfaced a stack-specific gap in `/setup`. The skill's "Scaffold the project" guidance (skill prose, step 2b) lists "Test runner configured to pass with no tests" as a critical requirement and gives the Vitest example (`passWithNoTests: true`). Bun's `bun test` has no equivalent flag — when invoked against a directory with no test files it exits **1** ("no tests found"), causing the very first gate-check after `/setup` to fail.

The /setup child worked around this by scaffolding `tests/smoke.test.ts` (a one-line `expect(1+1).toBe(2)` placeholder) so the gate would pass. The placeholder is then a permanent tax on every test count for the lifetime of the project, and there is no documented convention for it — a future operator reading the project would not know whether the file is load-bearing or removable.

This FR closes the gap two ways: documenting the recommended Bun pattern in the plugin's stack examples, and (optionally) gate-check probe coverage so a Bun project that lost its placeholder gets caught before merge.

## Acceptance Criteria

- AC-STE-113.1: A new file `plugins/dev-process-toolkit/examples/bun-typescript.md` exists, documenting (a) the `bun test` zero-match-exit-1 behavior; (b) the recommended workaround (one of: in-source `if (import.meta.main)` test placeholder, OR a `tests/.placeholder.test.ts` with a `// generated by /dev-process-toolkit:setup` marker); (c) the gate command `bunx tsc --noEmit && bun test`; (d) the canonical permissions allowlist for Bun-stack projects (cross-link to STE-106's `templates/permissions.json` row).
- AC-STE-113.2: `plugins/dev-process-toolkit/skills/setup/SKILL.md` step 2b is updated to reference the new Bun example explicitly. The "Stack-specific guidance" bullet for Bun is no longer collapsed into the TypeScript/Node bullet — Bun gets its own line with a pointer to `examples/bun-typescript.md`.
- AC-STE-113.3: When `/setup` detects a Bun stack (presence of `bun.lock` OR `package.json` with `"packageManager": "bun@..."` OR `bunfig.toml`), the scaffold-verification step (step 2c) anticipates the zero-match-exit-1 case and applies the documented workaround before running the gate. The placeholder file is created with the marker comment `// generated by /dev-process-toolkit:setup — Bun zero-match workaround (see examples/bun-typescript.md)`.
- AC-STE-113.4: New `/gate-check` probe `bun-zero-match-placeholder` (probe number 20): if the project root contains `bun.lock` AND no `*.test.ts` file exists outside `node_modules/` AND no source carries the marker comment `Bun zero-match workaround`, fail with NFR-10-shape error pointing to the recommended workaround. Probe lives at `plugins/dev-process-toolkit/adapters/_shared/src/bun_zero_match_placeholder.ts` (flat path matches existing convention); integration test at `plugins/dev-process-toolkit/tests/gate-check-bun-zero-match-placeholder.test.ts` (per STE-82 contract).
- AC-STE-113.5: Integration tests cover all four required scenarios via in-memory `mkdtempSync` fixtures in `plugins/dev-process-toolkit/tests/gate-check-bun-zero-match-placeholder.test.ts`: (a) Bun project with no tests + no placeholder → probe fails; (b) Bun project with placeholder marker present → probe passes; (c) Bun project with a real test file → probe passes; (d) non-Bun project (no `bun.lock`) → probe is vacuous and passes. Convention shifted post-STE-92 (AC-STE-92.5) — no on-disk `tests/fixtures/bun-zero-match-placeholder/` directory.
- AC-STE-113.6: `bun test` green. CHANGELOG entry under `### Added` cross-references this FR.

## Technical Design

**New example file.** `plugins/dev-process-toolkit/examples/bun-typescript.md` — short prose document modeled after `examples/typescript-node` (if it exists) or the inline guidance in `skills/setup/SKILL.md`. Sections: "Project layout", "Gate command", "Bun zero-match-exit-1 workaround", "Permissions allowlist".

**Skill prose tweak.** `skills/setup/SKILL.md` step 2b — split the one-line "TypeScript/Node" bullet into a 2-line block: "TypeScript/Node (general)" + "Bun (TypeScript)" with the second pointing to the new example. The skill's stack-detection logic (already written, infers from lock files) is unchanged.

**New probe.** `bun_zero_match_placeholder.ts` follows STE-82 contract. Reads `bun.lock` presence + scans for test files. If zero test files AND no marker comment in any source file, fail. Marker comment grep is `grep -lr "Bun zero-match workaround" .` excluding `node_modules`.

## Testing

`plugins/dev-process-toolkit/tests/gate-check-bun-zero-match-placeholder.test.ts` per STE-82 contract. Fixtures under `tests/fixtures/bun-zero-match-placeholder/` enumerate the four cases in AC-STE-113.5.
