# STE-388: Milestone-keyed branch naming — one canonical branch template across all surfaces {#STE-388}

## Summary

The toolkit names branches two different ways: tracker-mode branches take the first
ticket id in the milestone, while local-mode branches take the milestone number. This
FR unifies every branch-creating surface on one canonical, milestone-keyed shape so a
branch always names the unit that ships — the milestone. Ticket-keyed naming survives
only as a milestone-less fallback and as legacy acceptance (existing branches never
re-prompt); a setup migration re-seeds only the retired default. Forward-only.

## Requirement

The canonical branch template becomes `{type}/m{N}-{slug}` everywhere: both mode
defaults, `/spec-write`'s § 7a pre-commit branch gate, and `/implement`'s 0.b″
proposal. `{N}` renders from the FR's `milestone:` frontmatter. An explicit Schema L
`branch_template:` value still wins (operator override preserved); `/implement`'s
absent-key semantics (branch automation disabled) are unchanged. FR-scoped
acceptability checks accept the milestone-keyed shape; tracker-ID matching remains as
legacy acceptance only.

Before this FR, the split was deliberate but inconsistent (STE-64 AC-1): `mode: none`
seeded `{type}/m{N}-{slug}` while tracker mode seeded `{type}/{ticket-id}-{slug}`, and
`/spec-write`'s gate always rendered the ticket-keyed form — so a three-FR milestone
shipped on a branch named after its first ticket (e.g. M104 on a branch naming only
one of its three tickets), while milestone-scoped acceptability checks demanded an
`m<N>` token the branch never carried.

## Acceptance Criteria

- AC-STE-388.1: **Canonical template helper.** `canonicalBranchTemplate({ milestone })` — new pure export in `adapters/_shared/src/branch_proposal.ts` — returns `"{type}/m{N}-{slug}"` when `milestone` is a non-empty digit string and `"{type}/{ticket-id}-{slug}"` otherwise; unit tests cover present / absent / empty / non-digit inputs. {#AC-STE-388.1}
- AC-STE-388.2: **Single seeded default.** `/setup` step 7c seeds `branch_template: {type}/m{N}-{slug}` in BOTH tracker mode and `mode: none` — the mode-split default is retired; `docs/setup-tracker-mode.md` + `templates/CLAUDE.md.template` name the single default. {#AC-STE-388.2}
- AC-STE-388.3: **`/spec-write` gate renders the m-form.** The § 7a new-FR branch gate derives its template via `canonicalBranchTemplate` with `{N}` from the FR's `milestone:` frontmatter — milestone-bound FR ⇒ `{type}/m{N}-{slug}` proposal; milestone-less FR ⇒ `{type}/{ticket-id}-{slug}` fallback; `SpecWriteBranchInput` keeps its shape (its `milestone` field is already optional). {#AC-STE-388.3}
- AC-STE-388.4: **`/implement` renders the m-form.** 0.b″ FR-scoped runs resolve `{N}` from the resolved FR's `milestone:` frontmatter and propose the m-form; milestone runs unchanged; an explicit Schema L `branch_template:` value still wins; absent key still skips 0.b″ entirely; `docs/implement-reference.md` § Branch Proposal updated. {#AC-STE-388.4}
- AC-STE-388.5: **Acceptability re-key with legacy acceptance.** `isCurrentBranchAcceptable` — FR-scoped `RunScope` variants gain optional `milestoneNumber`; acceptable IFF off-trunk AND (word-bounded `m<N>` match when `milestoneNumber` is present OR tracker-ID / short-ULID substring). Both naming generations pass their scope check; word-boundary corpus (`m105` ≠ `m1051`) extended. {#AC-STE-388.5}
- AC-STE-388.6: **Migrate re-seed.** `/setup --migrate` re-seeds `branch_template:` to `{type}/m{N}-{slug}` when the existing value is byte-identical to the retired seeded default `{type}/{ticket-id}-{slug}`; any other value preserved verbatim; absent key stays absent; the re-seed is logged in `## /setup audit`. {#AC-STE-388.6}
- AC-STE-388.7: **Docs sweep.** `docs/patterns.md`, `docs/setup-tracker-mode.md`, `docs/implement-reference.md`, `templates/CLAUDE.md.template`: no shipped doc still presents the ticket-keyed form as a seeded default (historical/archived references exempt). {#AC-STE-388.7}
- AC-STE-388.8: **Full gate green.** `bun test` from `plugins/dev-process-toolkit` GREEN; calibration snapshots (SKILL line caps, adapters STE-token ceiling) bumped as needed; sanitizer + collision-suffix + 60-char-clamp corpus unchanged and green. {#AC-STE-388.8}

## Technical Design

- `canonicalBranchTemplate` lives beside `buildBranchProposal` (single source of
  truth; both `/spec-write`'s `branchNameFor` callers and `/implement` import it).
  Precedence: explicit Schema L `branch_template:` > `canonicalBranchTemplate`
  (milestone presence decides which canonical form).
- `buildBranchProposal` mechanics unchanged — `{N}` substitution already exists;
  callers pass milestone digits. Zero new prompts: the `[Y / e / n]` gate remains the
  only operator-override surface (STE-381 precedent — type-at-gate prompt was
  rejected there; same constraint honored here).
- `RunScope` extension is additive-optional — existing call sites compile unchanged.
- Linear `gitBranchName` stays ignored (STE-228 design + branch-naming feedback).

## Testing

- Unit matrices: `canonicalBranchTemplate` (4 input classes);
  `isCurrentBranchAcceptable` (m-named × fr-scope, ticket-named × fr-scope legacy,
  milestone-scope unchanged, word boundaries); `/setup --migrate` re-seed round-trip
  (retired default → new, custom → untouched, absent → untouched).
- Doc conformance: extend the branch-derivation doc-conformance test so the canonical
  default string is asserted in all four swept docs.
- Deterministic, no network; full gate from `plugins/dev-process-toolkit`.

## Notes

- Approved 2026-07-16 brainstorm — approach 1 of 3 (milestone-always); hybrid
  m{N}+ticket and scope-keyed status quo rejected. Core pain named: consistency as a
  principle (two coexisting schemes are a design smell regardless of concrete friction).
- Tradeoff accepted: Linear/Jira branch-name→ticket auto-linking is lost on new
  branches; the toolkit attaches PR links to tickets explicitly, so nothing breaks in
  this repo's flow. Consumers relying on tracker branch-magic keep it via an explicit
  `branch_template:` override (which always wins).
- Forward-only: no renames of existing branches. First external user exists (issue
  #39), hence the `--migrate` opt-in re-seed instead of silent default swapping.
- M106's second FR (removing an argument from `/pr`) will be specced separately and
  added to the M106 plan when it lands.
