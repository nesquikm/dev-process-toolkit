# STE-142: Override commitlint header-max-length to 72 + parity test {#STE-142}

## Requirement

The M36 spec (STE-133 AC-STE-133.1) says CC subjects must be ≤72 characters. The shell hook `templates/git-hooks/commit-msg.sh` enforces this exactly. But the opt-in commitlint variant (`/setup --commitlint`) writes `templates/git-hooks/commitlint.config.js` with the content:

```js
module.exports = { extends: ["@commitlint/config-conventional"] };
```

— no overrides. The `@commitlint/config-conventional` package's default `header-max-length` is **100**, not 72.

A project switching between hook-installation modes gets different boundary behavior:

| Subject length | Shell hook | commitlint variant |
|---|---|---|
| 72 | ✓ accept | ✓ accept |
| 73 | ✗ reject (`subject-too-long`) | ✓ accept |
| 100 | ✗ reject | ✓ accept |
| 101 | ✗ reject | ✗ reject (`header-max-length`) |

The spec (and the `CLAUDE.md` template that gets copied into user projects) promises 72. The commitlint variant has zero test coverage on its config.

Fix by adding the rule override + a parity test asserting both variants reject the same boundary subjects.

## Acceptance Criteria

- AC-STE-142.1: `templates/git-hooks/commitlint.config.js` carries the rule override `"header-max-length": [2, "always", 72]`.
- AC-STE-142.2: A test (new `tests/commitlint-config.test.ts` or extension of `tests/conventional-commits-hook.test.ts`) reads the shipped `commitlint.config.js`, parses it, and asserts the `header-max-length` rule value is 72.
- AC-STE-142.3: A parity test exercises the shell hook AND the commitlint variant against the same boundary subjects (72-byte, 73-byte, 100-byte) and asserts the same accept/reject verdict from both. The 100-byte case must be rejected by both variants.
- AC-STE-142.4: `bun test` is green.
- AC-STE-142.5: `/gate-check` clean across the live repo.
- AC-STE-142.6: A user re-running `/setup --commitlint` against an existing project gets the rule override (idempotent re-write — the `.template` change propagates on next render).

## Technical Design

**File 1 edit** — `templates/git-hooks/commitlint.config.js`:

```js
module.exports = {
  extends: ["@commitlint/config-conventional"],
  rules: { "header-max-length": [2, "always", 72] },
};
```

The `[2, "always", 72]` is commitlint's standard rule shape: severity 2 (error), enforcement direction `always`, value 72.

**File 2 add** — `tests/commitlint-config.test.ts`:

```ts
import { test, expect } from "bun:test";
import { readFileSync } from "node:fs";
import { join } from "node:path";

const configPath = join(import.meta.dir, "..", "templates", "git-hooks", "commitlint.config.js");

test("commitlint config: header-max-length is 72", () => {
  // eslint-disable-next-line @typescript-eslint/no-var-requires
  const config = require(configPath);
  expect(config.rules["header-max-length"]).toEqual([2, "always", 72]);
});
```

(Adjust the loader if Bun doesn't trivially `require()` the JS module; an alternative is to `readFileSync` and regex-match the rule value.)

**File 3 add (parity test)** — `tests/conventional-commits-hook.test.ts`:

Add a `describe("AC-STE-142.3 — shell hook ↔ commitlint variant parity", () => { ... })` block. For each boundary subject:

```ts
const subjects = [
  { len: 72, expect: "accept" },
  { len: 73, expect: "reject" },
  { len: 100, expect: "reject" },
  { len: 101, expect: "reject" },
];
```

Run each through the shell hook (existing harness) AND through `npx commitlint --config templates/git-hooks/commitlint.config.js` (or programmatic invocation). Assert both variants agree on the verdict.

If exercising commitlint programmatically is too heavy, a unit-level alternative is to load the config + a known commit-message fixture and confirm the rule fires at the right threshold.

## Testing

- **AC-STE-142.2** — config-shape test asserts the rule value.
- **AC-STE-142.3** — boundary-parity test asserts both variants agree.
- **`bun test`** is green.
- **Manual** — re-run `/setup --commitlint` on a fixture project; confirm the propagated `commitlint.config.js` carries the override.

## Risks

- **bunx / npx availability.** The parity test may need either `bunx commitlint` or `npx commitlint` in the test environment. The smoke-test driver already requires Bun; commitlint can be added as a dev-dependency (already declared per the M36 commitlint config). Mitigation: if the test runs only when `which commitlint` is non-null, document the skip-condition in the test (mirror `signature_extractor_dart.test.ts` skip pattern).

## Notes

**Byte-vs-character (G2 deferred).** The shell hook uses `awk '{print length}'` which counts BYTES on BSD awk; commitlint counts CHARACTERS. Multi-byte subjects diverge across variants. Cosmetic — most CC users write ASCII. YAGNI under no-users; could extend to `wc -m` later. NOT in scope here. The parity test uses ASCII subjects, so the byte-vs-char divergence does not affect its assertions.

The fix is one config line + a parity test. Total ~30 minutes of work plus the test scaffolding.
