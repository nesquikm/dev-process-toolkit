# STE-185: /smoke-test mandate heredoc-on-stdin for child prompts {#STE-185}

## Requirement

The 2026-05-01 Jira run 2 (against v2.4.0) hit a cross-tracker corruption bug. The driver wrote `/tmp/dpt-smoke-prompt-setup.txt` via the parent's Write tool with Jira-flavored content (`mode: jira`, `project: DST`, `default_labels=[dpt-smoke]`). Between Write completion and the spawned `claude -p` child reading the file, an external linter / process **replaced the file's contents** with a stale Linear-flavored stub from a prior Linear smoke run. The /setup child consumed the wrong prompt and went down the **Linear branch** — produced `CLAUDE.md` with `mode: linear`, merged a `linear` MCP entry into `.mcp.json`, and made bootstrap commit `b31e15f` referencing a Linear project on a Jira run.

The driver detected the corruption via the system reminder ("Note: /tmp/dpt-smoke-prompt-setup.txt was modified, either by the user or by a linter") in the child's stdout, wiped the polluted test project (`rm -rf ../dpt-test-project-jira`), and restarted Phase 1 from scratch. The recovery cost ~one extra `/setup` invocation and required driver-side recovery logic.

This is the fourth escalation in a series:

- **smoke #6 F1** — stale `/tmp/dpt-smoke-prompt-*.txt` retained across runs.
- **smoke #7 F2 + F4** — same retention, now causing Write-tool errors.
- **smoke #8 F1** — file recreated (not just retained), still stale.
- **smoke #9 / Jira run 2 F1** — content-swap (Jira-flavored content overwritten with Linear-flavored stub), causing **silent cross-tracker corruption** — exactly the failure mode per-tracker artifact isolation (M46) was meant to prevent.

The escalation curve is now severe enough to justify retiring the prompt-file path entirely. The fix from Jira run 2 F1 — heredoc-on-stdin in the parent's Bash command (`claude -p ... <<'PROMPT_EOF' ... PROMPT_EOF`) — was applied as an ad-hoc workaround for the remaining 5 skill invocations of that run and worked cleanly. With no file on disk, the linter has no attack surface.

This FR lifts heredoc-on-stdin from "ad-hoc workaround" to "first-class /smoke-test method" and **retires the `/tmp/dpt-smoke-prompt-*.txt` step entirely**.

Out of scope:

- The wrapped MCP config file `/tmp/dpt-smoke-mcp-config-<tracker>.json` (separate file, separate cleanup concern — covered by STE-186).
- The `< /dev/null` discipline for non-prompt-bearing child spawns (`/spec-review`, `/simplify`, `/gate-check` — children that take no extra prompt content) — covered by STE-188.
- Renaming the Phase 0.5 cleanup glob `/tmp/dpt-smoke-prompt-*.txt` — the glob becomes a no-op once no prompt files are written, but removing it would require dropping the safety rail that catches stale files from older driver versions during this transition. Keep the glob; document it as defense-in-depth.

### Heredoc reference shape

The canonical form for a prompt-bearing child spawn:

```
claude -p \
  --plugin-dir <toolkit-root> \
  --mcp-config /tmp/dpt-smoke-mcp-config-<tracker>.json \
  --permission-mode bypassPermissions \
  > /tmp/dpt-smoke-<tracker>-<skill>.log 2>&1 <<'PROMPT_EOF'
/dev-process-toolkit:<skill> <args>

<prompt body content, branching on tracker as needed>
PROMPT_EOF
```

`<<'PROMPT_EOF'` (single-quoted heredoc tag) prevents shell expansion in the body — important because prompt content may include `$variable` references (file paths, etc.) that should pass through to Claude verbatim. The exact tag string is style only; `<<'EOF'` works equally well as long as the closing tag matches.

### Why this is not just a doc change

The `bypassPermissions` mode in Phase 2 children depends on the parent's pre-creation of `.claude/settings.json` and `.mcp.json` to constrain blast radius (see SKILL.md threat-model section). The prompt-file path was the **last remaining file-based interaction surface** the parent shared with the child between Write and child-spawn time. Removing it shrinks the attack window to zero for the prompt content. The driver's existing artifacts (findings, logs, MCP config) are either read-only by the child (MCP config) or written-only by the child (logs, findings) — neither has the parent-writes-then-child-reads window the prompt file had.

### Per-skill prompt content

The pre-baked per-skill prompts that previously lived in `/tmp/dpt-smoke-prompt-<skill>.txt` move inline into the SKILL.md Phase 2 reference snippets. SKILL.md grows by ~50–100 lines (six skill prompt bodies inlined) and the `/tmp` artifact set shrinks by 6 files per run.
