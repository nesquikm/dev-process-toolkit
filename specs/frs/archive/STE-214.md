# STE-214: /gate-check probe #26 milestone-attach respects FR ## Notes capability-gap declaration {#STE-214}

## Requirement

`/gate-check` probe #26 (`tracker-project-milestone-attached`) gates `GATE FAILED` when an FR's tracker ticket has no `projectMilestone` binding. M54 (STE-198) already split the milestone-attach capability into two keys (`milestone_attach_skipped_adapter_limit` for adapter-level "no project_milestone support"; `milestone_create_required` for "we auto-created the milestone"), and `/spec-write` writes a corresponding capability declaration into the FR's `## Notes` section when adapter Schema M flags `project_milestone: false`. Probe #26 does not consult those declarations — it gates purely on the cross-system attachment, so every Linear smoke run against a fresh project (zero-milestone state by construction) emits `PASSED WITH ≥1 GATE FAILURE` even when the chain is otherwise clean.

**Findings bundled:** F1 from `/tmp/dpt-smoke-findings-2026-05-04-linear.md` (smoke #9 v2.5.0).

**Symptoms observed in the 2026-05-04 Linear smoke:**

1. The smoke driver creates a fresh `DPT Smoke Test (<date>)` Linear project per run; project milestones are not part of the canonical chain, so the project has zero milestones.
2. `/spec-write` emits the FR with `milestone_attach_unavailable` (deprecated alias of `milestone_attach_skipped_adapter_limit`) declared in the FR's `## Notes` body.
3. `/gate-check` probe #26 ignores the declaration and emits `GATE FAILED` for the missing `projectMilestone` binding.
4. Every Linear-tracker smoke run produces `PASSED WITH ≥1 GATE FAILURE` even when the chain is otherwise clean — degrades the signal of the audit trail.

## Acceptance Criteria

- AC-STE-214.1: Probe #26 reads the FR's `## Notes` section before issuing `GATE FAILED` for a missing `projectMilestone` binding. The probe scans the section's body text for any of the static plain-language map's milestone-attach capability keys: `milestone_attach_skipped_adapter_limit` (canonical), `milestone_attach_unavailable` (deprecated alias per STE-198), or `milestone_create_required`. {#AC-STE-214.1}
- AC-STE-214.2: When at least one such capability key is found in the FR's `## Notes`, probe #26 downgrades the result from `GATE FAILED` to `ADVISORY`. The advisory row emits the capability key found, the FR id, and the rendered plain-language prose drawn from the static map (single source of truth — the probe does not invent prose). {#AC-STE-214.2}
- AC-STE-214.3: When no milestone-attach capability key is found in the FR's `## Notes` AND the tracker ticket has no `projectMilestone` binding, probe #26 stays `GATE FAILED` (existing behavior preserved). The downgrade path is opt-in via the explicit declaration only. {#AC-STE-214.3}
- AC-STE-214.4: Recognition is keyed on the literal capability-key token (substring match). Freeform prose in `## Notes` that *describes* the missing milestone without citing the capability key (e.g., "this Linear project has no milestones yet") does NOT downgrade — the operator must explicitly write the capability key for the contract to hold. This avoids false-quiet on prose that incidentally mentions milestones. {#AC-STE-214.4}
- AC-STE-214.5: The deprecated `milestone_attach_unavailable` alias is honored for one minor-version cycle per STE-198's deprecation window; both keys map to identical advisory behavior. After the deprecation window, only the canonical `milestone_attach_skipped_adapter_limit` is recognized. The deprecation cutover is tracked via STE-198's existing rollover note. {#AC-STE-214.5}
- AC-STE-214.6: Test fixture (positive): a Linear smoke project with zero milestones + an FR carrying `milestone_attach_unavailable` in `## Notes` produces a probe #26 ADVISORY (not FAILED). Test fixture (control): the same project + the same FR with the declaration removed from `## Notes` produces a probe #26 GATE FAILED. {#AC-STE-214.6}

## Technical Design

**Where probe #26 lives.** `/gate-check`'s probe set is documented in `skills/gate-check/SKILL.md`. Probe #26 is the `tracker-project-milestone-attached` probe; its implementation traverses the FRs in scope, reads each ticket's `projectMilestone` field, and gates failed when missing.

**Notes-section scanner.** Add a small helper that, given the FR file path, reads the `## Notes` section body and returns a list of recognized milestone-attach capability keys found (substring match on the three tokens in AC.1).

**Probe branch.** Before emitting `GATE FAILED` for a missing `projectMilestone`, call the helper. If any key is returned, switch to ADVISORY shape:

```
ADVISORY: probe #26 (tracker-project-milestone-attached)
  FR: STE-NNN
  Capability declared: <key>
  Rendered prose: <static-map[key]>
```

**Static-map source.** The rendered prose is drawn from `skills/spec-write/SKILL.md` § 7's static plain-language map; both skills share the same source of truth. If the map adds a new milestone-attach key in the future, probe #26's recognized set also grows (the helper's recognized-set list is the single source of truth — both `/spec-write` step 7 and `/gate-check` probe #26 reference it).

**Deprecation cutover.** STE-198 documented a one-minor-version-cycle window for the `milestone_attach_unavailable` alias. Probe #26 honors both keys until the cutover, then drops the alias.

## Testing

- Unit test: probe #26 fixture against an FR with `milestone_attach_skipped_adapter_limit` in `## Notes` ⇒ ADVISORY (not FAILED).
- Unit test: same probe against an FR with no capability key in `## Notes` and a ticket missing `projectMilestone` ⇒ GATE FAILED (control case).
- Unit test: capability-key substring match is exact (no partial-match false positives — e.g., a `## Notes` line that says "milestone attach is great" does NOT match `milestone_attach_skipped_adapter_limit`).
- Smoke regression: 2026-05-04 Linear smoke #9 F1's reproduction case (fresh Linear project + `/spec-write`-declared FR) produces a clean ADVISORY-not-FAILED outcome on probe #26.
