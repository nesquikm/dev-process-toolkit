# STE-148: /implement — surface Stage B advisory notes {#STE-148}

## Requirement

When the code-reviewer Pass 2 returns CONCERNS that the bounded loop classifies as advisory (not gate-blocking), those concerns currently disappear from the operator's view in non-interactive (`claude -p`) runs. Only the one-line `Stage B Pass 2 CONCERNS classified as advisory notes` summary survives in the final assistant message. The operator loses the audit trail and the chance to override. Surface the notes inline in the final report AND persist them in the archived FR markdown so they survive past the session.

## Acceptance Criteria

- AC-STE-148.1: /implement's end-of-run report contains a structured `## Advisory notes` section listing every Stage B Pass 2 concern routed to advisory, one bullet per concern, with the reviewer's rationale.
- AC-STE-148.2: The same advisory list is appended to the archived FR's body under a `## Implementation notes` (canonical heading) section so it survives past the session.
- AC-STE-148.3: When zero advisory concerns exist, the section still emits the heading + the line "No advisory notes." — silence is never ambiguous between "no concerns" and "concerns hidden".
- AC-STE-148.4: Test fixture: a synthetic Stage B Pass 2 returning one advisory concern produces both (a) the inline section in the /implement report and (b) the archived-FR section, with byte-identical wording in the bullet body.

## Technical Design

Touchpoint: `skills/implement/SKILL.md` Phase 3 Stage B (the bounded review loop) and Phase 4 archive step.

Capture advisory notes in a structured field returned from the code-reviewer agent invocation. Schema:

```
advisoryNote: {
  pass: 2,
  concern: string,            // one-line concern statement
  rationale: string,          // why classified as advisory rather than blocking
  classification: 'advisory'
}
```

Thread the `advisoryNote[]` array into:
1. The end-of-run report template (new `## Advisory notes` section, rendered after the existing Files / Gate / Review-rounds table).
2. The archive write step (new `## Implementation notes` body section in the archived FR markdown, appended after `## Notes`).

Both renderings use the same source list — byte-identical bullet bodies guaranteed by sharing the formatter function.

## Testing

- Unit: render-report function with synthetic advisory list — both empty (heading + "No advisory notes.") and one-entry cases.
- Unit: archive-write step with same synthetic input — confirm the FR markdown contains the same bullet body.
- Integration: re-run /smoke-test post-fix; F2 should not re-appear in the next findings file (the advisory section will either be present with content or present with "No advisory notes.", but never absent).
