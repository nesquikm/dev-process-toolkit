# STE-189: /setup specs/ scaffolding boundary clarification {#STE-189}

## Requirement

The 2026-05-01 Jira run 2 surfaced an ambiguity in `/setup`'s deliverable boundary. The smoke driver's `/setup` prompt told the child:

> "Do not run /docs and do not run /spec-write — only /setup completes here"

The /setup child appears to have over-interpreted this as "don't even do /setup's normal scaffolding" — it produced only `CLAUDE.md` + the `commit-msg` hook + a bootstrap commit, and **skipped**:

- `specs/requirements.md`
- `specs/technical-spec.md`
- `specs/testing-spec.md`
- `specs/plan/M1.md`
- `.gitkeep` stubs in `specs/frs/`, `specs/frs/archive/`, `specs/plan/`, `specs/plan/archive/`
- `src/.placeholder.test.ts`

These six artifact classes are all explicitly named in `plugins/dev-process-toolkit/skills/setup/SKILL.md` step 2 and step 4 (or equivalent — see § Files touched for the canonical references). Prior smoke runs (e.g., #8) had `/setup` produce all of them.

The chain self-recovered (`/spec-write`, run next, scaffolded the entire `specs/` tree itself), so no user-observable regression — but the boundary between `/setup` and `/spec-write` deliverables is ambiguous from the child's perspective when restrictive smoke-driver phrasing is in the prompt.

This FR clarifies the boundary in the SKILL.md prose so the LLM cannot mis-read "only /setup completes here" as a license to skip /setup's own scaffolding deliverables.

Out of scope:

- The smoke-test driver's prompt phrasing. The driver's prompt is project-local prose; tightening it is a separate concern (and the driver is unlikely to be the only context where this misread happens — fix the SKILL.md instead).
- `/spec-write`'s scaffolding fall-back. `/spec-write` already self-recovers; the user-facing impact is zero. The fix is preventive, not reactive.
- Adding a gate-check probe asserting all scaffold artifacts exist post-`/setup`. Possible follow-up — out of M52 scope. Probe #22 (`toolkit-bootstrap-committed`, STE-179) catches the downstream case (untracked specs files), which is the more important invariant.

## Technical Design

### Files touched

| File | Change |
|------|--------|
| `plugins/dev-process-toolkit/skills/setup/SKILL.md` | Step 2 (the scaffolding step — search for "Create `specs/` plus" or "Scaffold spec templates"; the exact step number depends on current SKILL.md numbering) gets the six-artifact-class list with explicit labels and the "emitted regardless of restrictive prompt" rule. ~10–15 line addition. |

### Why prose, not code

The bug is LLM interpretation, not code. /setup is a prose-driven skill: the LLM reads SKILL.md and decides which steps to perform. The fix tightens the prose so the LLM cannot reasonably misinterpret "only /setup completes here" as licensing scaffold skips. No adapter helper, no test harness, no probe — just sharper SKILL.md text.

### Probe deferral

A `/gate-check` probe asserting "post-`/setup`, all six artifact classes exist on disk" is tempting but redundant: probe #22 (`toolkit-bootstrap-committed`, STE-179) already catches the downstream case where the skips cascade into uncommitted-or-missing toolkit files. The prose fix is the cheapest sufficient mitigation.

## Notes

- **Smoke-test source.** `/tmp/dpt-smoke-findings-2026-05-01-jira.md` § Run 2 § F2 documents the misinterpretation. Severity was low (chain self-recovered), but the prose is worth tightening so the misread doesn't recur.
- **No-users-yet memory.** Per `project_no_users_yet.md`, the plugin has zero installs; no back-compat shim needed for downstream projects that may have learned to expect the partial-scaffold behavior. Hard prose update is safe.
- **Why this isn't a /spec-write or /implement issue.** /setup owns scaffolding; /spec-write owns spec bodies; /implement owns code + spec archival. The boundary is clear in the docs taken individually; only the smoke driver's restrictive prompt phrasing pushed /setup into the misread. The fix lives where the boundary is set: /setup SKILL.md.
