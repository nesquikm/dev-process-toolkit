# STE-396: Fix token-stats same-session double-count: shared rows route to the milestone design bucket {#STE-396}

## Summary

When one working session produces several feature records, each of them currently reports the entire session's token cost as if it were its own. The totals are inflated wherever it matters most, and the milestone rollup quietly loses its per-record breakdown at the same time. Shared cost is now attributed to the milestone's design bucket instead of being copied onto every record, so the numbers stop overstating and the rollup regains the breakdown it was always meant to show.

## Requirement

Caught by dogfooding on 2026-07-19 while writing STE-394 and STE-395 in one session: both FR files rendered byte-identical `## Token Stats` blocks (same 122,107 output tokens), double-counting the session.

Root cause — `filterRowsForFR` (`adapters/_shared/src/token_stats_render.ts:303-307`) selects direct same-session rows via `row.claimed_by === undefined && lineage.has(row.session_id)` but never assigns `claimed_by`. The only assignment in the module is inside the bridging fallback at `:334`, and `claimRowsForFR`'s persistence trigger at `:367-372` keys off that mutation — so with no bridging, nothing is marked and nothing is persisted. The doc comment at `:302` ("A row claimed by a different FR is never selected — no double-count") is true only for bridged rows.

This violates a shipped AC. AC-STE-345.6 (`specs/frs/archive/STE-345.md:29`) states: "A claimed row is marked … so no FR double-counts it", and its testing note at `:48` names "no double-count across two FRs".

Reproduced in isolation against a synthetic ledger: two `claimRowsForFR` calls with the same `sessionLineage` and different `brainstormClaim` values both returned all rows (4000 + 4000 against an actual ledger total of 4000), and the ledger carried zero `claimed_by` marks afterward.

**Coverage hole.** Three tests appear to cover this; none exercises the direct path with a shared lineage. `tests/token-stats-claim-persistence.test.ts:80-100` — the row's `session_id` sits outside `sessionLineage:88`, so run 1 goes through bridging; this is the test that nominally discharges AC-STE-345.6 and it tests only the path that already works. `tests/token-stats-render.test.ts:297-315` pre-seeds `claimed_by` and uses a disjoint lineage. `:236-242` is the only direct-path test and asserts the selection alone.

**Blast radius.** `/spec-write` inflates by the number of FRs per session. `/implement` Phase 4 archive (`skills/implement/SKILL.md:259`) is worse — it normally builds every FR of a milestone in one session, so each archived FR carries the whole milestone's `implement` + `tdd` cost, inflating roughly N×. Separately, `/ship-milestone`'s per-FR subtotal lines promised by AC-STE-346.2 are never emitted at all, because `rollupLabel` (`:215-219`) falls through to `row.skill` when `claimed_by` is unset — under-attribution from the same root cause. The milestone `total` row is unaffected.

**Approved semantic** (brainstorm 2026-07-19): rows from a session that produced more than one FR are attributed to the milestone design bucket, not to any single FR. The cost genuinely was shared; the existing `DESIGN_BUCKET` is the honest home for it. Rejected: first-writer-wins (FR2 renders empty — a different wrong answer), and even-split (synthetic per-FR numbers that were never separately measured).

## Acceptance Criteria

- AC-STE-396.1: `filterRowsForFR` marks direct-path rows. A row selected because its `session_id` is in `sessionLineage` and it carries no prior claim is assigned `claimed_by = <brainstormClaim>`, so `claimRowsForFR` persists it. **Backfilled at implementation time** (`underspecified` deviation): this AC originally asserted the mark would persist "with no change to the persistence logic", which the implementation refuted. The original trigger was an identity-`Set` of already-claimed rows tested by membership, so it could only ever fire on an `undefined → X` transition — AC-STE-396.2's demotion rewrites an *already-claimed* row's **value** (`STE-394 → design/exploration`) and was therefore structurally invisible to it. The trigger is now an identity-keyed `Map` snapshot of claim **values**, compared after selection. The new trigger is a strict superset of the old (every `undefined → X` transition is also a value change), so no pre-existing path regressed. AC.1 and AC.2 were mutually inconsistent as written; this records the resolution. {#AC-STE-396.1}
- AC-STE-396.2: Shared-session demotion. When a claim run selects a row already claimed by a *different* FR AND that row's `session_id` is in this run's `sessionLineage` (i.e. both FRs came from one session), the row is re-marked `claimed_by = <DESIGN_BUCKET sentinel>` and is returned to **neither** FR. A row claimed by a different FR from a *different* session stays claimed and is simply not selected, exactly as today. {#AC-STE-396.2}
- AC-STE-396.3: **Demotion** is idempotent and order-independent. Claiming FR-A then FR-B, or FR-B then FR-A, yields the same terminal ledger state *for the demotion outcome*; a third FR from the same session changes nothing further. Re-running any claim after demotion is a no-op. **Scope clarified at implementation time** (`underspecified` deviation): the unqualified "same terminal ledger state" is false when a bridgeable detached brainstorm row is present. Demoting a session's own brainstorm row makes `hasBrainstorm` false for the second FR, newly routing it into STE-345's **first-come-wins** bridging fallback — so *which FR owns the bridged row* is order-dependent (verified: 6 permutations → 3 terminal ledgers, differing only in that ownership; 24 permutations over 4 FRs converge for the demotion itself). This is pre-existing STE-345 behaviour that STE-396 makes **reachable more often**, not new non-determinism, and the sum invariant is unaffected — the bridged row lands with exactly one FR under every permutation. This AC therefore pins the demotion outcome; bridged-row ownership is explicitly out of its scope. {#AC-STE-396.3}
- AC-STE-396.4: Callers re-render after demotion. `/spec-write` § 0b Step 7 and `/implement` Phase 4 re-render the `## Token Stats` block for **every** FR they touched in the run, after all claims settle — otherwise an FR written before a later demotion keeps a block the ledger no longer supports. {#AC-STE-396.4}
- AC-STE-396.5: Rollup attribution restored. `rollupLabel` renders demoted rows under the existing design bucket, and the per-FR subtotal lines promised by AC-STE-346.2 are emitted for rows that remain FR-claimed. The milestone `total` row is unchanged in every case — demotion moves rows between buckets, never in or out of the total. **The `/ship-milestone` row-scoping rule is part of this AC's surface** (found by the spec audit, fixed in the same change): that rule admitted a row only when its `claimed_by` named an in-scope FR **or** the row was unclaimed, so a demoted row — which is neither — was filtered out *before* `renderMilestoneRollup` ran, silently shrinking the milestone total. The scoping rule now carries a **third leg**: a row carrying the `DESIGN_BUCKET` sentinel belongs when its `git_branch` is on the milestone's branch lineage (the same lineage test the unclaimed leg uses), because the sentinel marks shared-session cost, not another milestone's cost. Note the unit tests pass the whole ledger straight to `renderMilestoneRollup` and so **cannot** exercise this scoping — the guard is a prose contract, since the scoping has no code helper. {#AC-STE-396.5}
- AC-STE-396.6: Sum invariant, stated as the two properties that actually hold **structurally** for any ledger and any set of FRs: (a) **disjointness** — every ledger row lands in exactly one rollup bucket, so no row is ever double-counted; and (b) **total fidelity** — the rendered milestone `total` equals the ledger total. Both follow from `aggregateByLabelModel` partitioning every row without filtering, so relabeling can only redistribute rows across buckets. A property-style test asserts them over a multi-FR, multi-session fixture (two sessions, three FRs, one detached brainstorm; all 6 claim-order permutations). **Restated at implementation time** (`underspecified` deviation): the original phrasing — "the sum of all rendered per-FR blocks plus the design bucket equals the ledger total" — is **not** universally true, and its counterexample is the common case rather than an exotic one. `rollupLabel` falls through to `row.skill` for any unclaimed non-brainstorm row, so an unattributed `(main-loop)` row on a session no FR owns forms its **own** bucket: `frSum + design` = 100 against a ledger total of 107. Nothing is double-counted and the rollup total stays correct — the original equation simply omitted the other legitimate buckets. {#AC-STE-396.6}
- AC-STE-396.7: The coverage hole is closed with a **failing-first** test: two `claimRowsForFR` calls sharing one `sessionLineage` with different claims. Written against today's code it must fail (both calls return the same rows). `tests/token-stats-claim-persistence.test.ts:80-100` gains a sibling case that does not route through bridging. {#AC-STE-396.7}
- AC-STE-396.8: Bridging is untouched. Every existing bridging test passes unmodified, and the detached-brainstorm claim path behaves identically. {#AC-STE-396.8}
- AC-STE-396.9: Full `bun test` from `plugins/dev-process-toolkit` passes with no probe regressing. {#AC-STE-396.9}

## Technical Design

Single module: `adapters/_shared/src/token_stats_render.ts`. The demotion needs both the row's `session_id` and the claiming FR's lineage, and both are already in `FilterRowsForFROptions` — no signature change.

Ordering matters: the first FR of a session cannot know a second is coming, so the demotion is necessarily *retroactive*. That is what makes AC-STE-396.4's re-render mandatory rather than cosmetic — without it the first FR keeps a stale block on disk. `/spec-write` already writes FR files sequentially, so the re-render is a second pass over the touched set, not a redesign.

The design bucket sentinel must be distinguishable from a real FR id so `rollupLabel` can route it. Reuse `DESIGN_BUCKET` (`:198`) rather than minting a parallel constant.

## Testing

Unit tests in `tests/token-stats-render.test.ts` and `tests/token-stats-claim-persistence.test.ts`. The failing-first case of AC-STE-396.7 is the regression anchor. Order-independence (AC-STE-396.3) is tested both ways round. The sum invariant (AC-STE-396.6) uses a fixture with two sessions, three FRs, and one detached brainstorm so the bridging and direct paths are exercised together.

The first render after this ships may *decrease* previously-recorded numbers in FR files — that is the fix landing, not a regression.

Traceability: add a row for AC-STE-396.1..9 to the matrix in `specs/requirements.md`.

## Implementation notes

- Demotion makes STE-345's bridging fallback *newly reachable* more often: an FR whose own brainstorm row was demoted now finds `hasBrainstorm` false and falls through to bridging, where first-come-wins decides ownership. Verified — 6 claim permutations yield 3 terminal ledgers differing only in which FR owns a bridged detached row. The sum invariant is unaffected (the row still lands with exactly one FR) and the bridging code is byte-unchanged per AC-STE-396.8; only its reachability grew. Recorded in AC-STE-396.3's scope note.

- After demotion strips every one of an FR's rows, that FR vanishes from the milestone rollup rather than rendering a zero row (`aggregateByLabelModel` emits no bucket for zero rows). Spec-consistent — AC-STE-396.5 conditions subtotals on "rows that remain FR-claimed" — but a reader of `specs/plan/M<N>.md` cannot distinguish "this FR cost nothing" from "this FR was not rendered".

- The pre-existing bridging tests reuse module-level mutable row fixtures that `filterRowsForFR` now mutates in place. Order-independent today only because every test in that block uses claim id `STE-345`; a future test in the same block using a different claim id would silently interact. A per-test factory would remove the coupling.

- The `ledger.length !== claimsBefore.size` guard in `claimRowsForFR` is unreachable in practice — `readLedgerRows` allocates one fresh object per line, so the identity-keyed Map size always equals `ledger.length`. It fails safe (forces a rewrite) if it ever does fire; harmless defensive code.
