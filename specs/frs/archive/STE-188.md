# STE-188: /smoke-test pipe </dev/null in non-prompt Phase 2 child spawns {#STE-188}

## Requirement

The 2026-05-01 Linear smoke run logged the following warning in `/spec-review` and `/simplify` child output:

```
Warning: no stdin data received in 3s, proceeding without it.
```

The warning comes from the `claude -p` child's CLI startup — the parent passes the prompt via the positional argument (or the slash-command name with no prompt body), not stdin, and the child's auto-detection waits 3 seconds before giving up on stdin. The warning is cosmetic; the children produce correct output and exit 0.

The warning has a documented remediation: pipe `< /dev/null` in every spawn. With stdin closed, the child detects "no stdin available" immediately and skips the 3-second wait.

This FR applies `< /dev/null` to **non-prompt-bearing** Phase 2 child spawns (`/spec-review`, `/simplify`, `/gate-check` — children that take only a slash-command name, no extra prompt content).

For **prompt-bearing** children (`/setup`, `/spec-write`, `/implement`), STE-185 mandates heredoc-on-stdin which already provides stdin content; no `< /dev/null` is needed there. The two FRs partition the Phase 2 child-spawn surface cleanly.

Out of scope:

- Prompt-bearing children. STE-185 covers them via heredoc-on-stdin.
- Suppressing the warning at the Bun / Claude-CLI source. The warning's remediation hint says "pipe `< /dev/null`" — this FR follows that hint at the spawn-call site, not at the CLI.
- The 3-second wait timeout itself. We don't need to change the child's behavior; we just provide stdin so it doesn't block.

## Technical Design

### Files touched

| File | Change |
|------|--------|
| `.claude/skills/smoke-test/SKILL.md` | Phase 2 child-spawn snippets for `/spec-review`, `/simplify`, `/gate-check` get `< /dev/null` added; prose paragraph near the snippets explains the partition with STE-185. ~3-line snippet edit + ~2-line prose addition. |

### Spawn-snippet shape

Before:
```
claude -p /spec-review --plugin-dir <root> --mcp-config <path> \
  --permission-mode bypassPermissions \
  > /tmp/dpt-smoke-<tracker>-spec-review.log 2>&1
```

After:
```
claude -p /spec-review --plugin-dir <root> --mcp-config <path> \
  --permission-mode bypassPermissions \
  < /dev/null > /tmp/dpt-smoke-<tracker>-spec-review.log 2>&1
```

The `< /dev/null` redirect closes stdin before the child reads it. Same shape for `/simplify` and `/gate-check`.

### Why not blanket `< /dev/null` on every spawn

Tempting but wrong: prompt-bearing children (`/setup`, `/spec-write`, `/implement`) read their prompt body from stdin via the heredoc STE-185 mandates. Adding `< /dev/null` there would close stdin before the heredoc body is read and break the prompt-delivery path. The two FRs deliberately partition the spawn surface by whether the child needs prompt-body input.
