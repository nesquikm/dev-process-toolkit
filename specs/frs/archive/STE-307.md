# STE-307: Gate-check probe for audit-fix loop pattern invariants {#STE-307}

## Requirement

The pattern doc landed by STE-306 (M80 FR-A) is prose-only; without enforcement, future skill authors will reintroduce the legacy Agent-tool pattern unintentionally (the HG95VF→STE-225 supersession lesson). Add a `/gate-check` probe `audit_fix_loop_pattern_invariants` (severity: error) that byte-checks the canonical pattern's frontmatter invariants on every skill listed in a const allowlist.

The probe mirrors:

- `/gate-check` probe #39 `tdd_orchestrator_integrity` (STE-225 AC.7): asserts paths exist, `context: fork` set, `agent:` resolves, `user-invocable: false`, subagent tools exclude `Agent`.
- `/gate-check` probe #50 `tdd_spec_reviewer_subagent_invariants` (STE-296 AC.8): asserts byte-checkable subagent frontmatter (tools, maxTurns, model).
- `/gate-check` probe #51 `deps_researcher_subagent_invariants` (STE-301 AC.15): same shape against the deps-research fork.

## Acceptance Criteria

- AC-STE-307.1: New probe `audit_fix_loop_pattern_invariants` (severity: error) added to `plugins/dev-process-toolkit/skills/gate-check/SKILL.md` probe table.
- AC-STE-307.2: Const allowlist `AUDIT_FIX_LOOP_CANONICAL_LOOPS` exported from `plugins/dev-process-toolkit/adapters/_shared/src/audit_fix_loop_pattern.ts`. Ships with the `/tdd` audit-fork pair `{ orchestrator: 'tdd', child: 'tdd-spec-review', subagent: 'tdd-spec-reviewer' }` only — the audit half of `/tdd`'s audit-fix loop. The RED/GREEN/REFACTOR forks (`tdd-write-test`, `tdd-implement`, `tdd-refactor`) are deliberately excluded: they are the **action** half of the loop and need Write/Edit/Bash; the allowlist tracks read-only **audit** subagents only, per the M80 plan's stated architecture ("per-skill audit fork + paired read-only subagent"). STE-308 adds the `/spec-review` audit-fork entry as part of its migration.
- AC-STE-307.3: For each allowlist entry, the probe asserts (a) paired child skill file exists at the declared path with `context: fork`, `user-invocable: false`, `agent:` resolves to an existing `plugins/dev-process-toolkit/agents/<name>.md`; (b) the subagent declares `tools: Read, Grep, Glob` only (Write/Edit/Bash/Agent excluded); (c) the child's `allowed-tools:` (when present) excludes `Agent`. Reuses frontmatter parsing + violation-rendering primitives from `adapters/_shared/src/tdd_probe_helpers.ts`.
- AC-STE-307.4: Probe is vacuous on repos with neither `plugins/dev-process-toolkit/skills/` nor `plugins/dev-process-toolkit/agents/` (matches the vacuous shape of probes #39, #50, #51).
- AC-STE-307.5: Test coverage in `plugins/dev-process-toolkit/tests/gate-check-audit-fix-loop-pattern-invariants.test.ts` covering: (a) clean repo (the `/tdd` audit-fork allowlist entry passes); (b) child skill fixture missing `context: fork` → violation row; (c) subagent declaring `tools: Read, Grep, Glob, Write` → violation row; (d) child declaring `allowed-tools: Read, Agent` → violation row; (e) absent-plugin-content fixture → vacuous.

## Technical Design

New helper `runAuditFixLoopPatternInvariantsProbe(projectRoot): ProbeResult` in `plugins/dev-process-toolkit/adapters/_shared/src/audit_fix_loop_pattern.ts`. Reuses `parseSkillFrontmatter` / `parseSubagentFrontmatter` / `formatViolationRow` from `tdd_probe_helpers.ts` (shared with probes #39, #50, #51).

Allowlist shape:

```ts
export interface AuditFixLoopEntry {
  orchestrator: string;
  child: string;
  subagent: string;
}

export const AUDIT_FIX_LOOP_CANONICAL_LOOPS: AuditFixLoopEntry[] = [
  { orchestrator: 'tdd', child: 'tdd-spec-review', subagent: 'tdd-spec-reviewer' },
];
```

STE-308's AC.5 appends the `/spec-review` audit-fork entry to this allowlist.

**Why audit-fork-only allowlist:** the canonical audit-fix loop's defining property (STE-306 Pattern 29) is that the **audit** subagent is read-only — it gathers findings, the orchestrator parses them, and a separate fork applies the fix. The RED/GREEN/REFACTOR forks in `/tdd` are the action half of `/tdd`'s TDD cycle (not the audit half) and need Write/Edit/Bash. The general invariants for *all* `/tdd` forks (`context: fork`, `user-invocable: false`, `agent:` resolves, `allowed-tools` excludes `Agent`) are already enforced by probe #39 `tdd_orchestrator_integrity`; the new probe #54 layers the read-only audit-subagent invariant on top, for the audit-fork pairs specifically.

Wire the probe into the gate-check runner with severity `error` and the standard capability-row failure mode.

## Testing

Unit tests in `tests/gate-check-audit-fix-loop-pattern-invariants.test.ts` covering the five AC scenarios with fixture trees under `tests/fixtures/audit-fix-loop-pattern/`.

Integration: existing `/gate-check` smoke pass already covers the dogfood repo; the new probe must pass clean on `main` at FR ship.

## Notes

Probes #39, #50, #51 share helpers in `tdd_probe_helpers.ts`; this probe extends the same shape. No new helper modules expected beyond the allowlist + probe wrapper.
