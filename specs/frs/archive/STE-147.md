# STE-147: /spec-write — scrub internal toolkit AC IDs from user-facing output {#STE-147}

## Requirement

/spec-write occasionally surfaces references to the toolkit's *own* internal AC identifiers (e.g., `AC-STE-118.4 milestone-attach was a no-op`) in its user-facing summary block. Project owners running /spec-write on their own repo see these literals as opaque jargon; the AC IDs only have meaning to toolkit maintainers reading the toolkit's own spec set. Replace with plain-language capability descriptions.

## Acceptance Criteria

- AC-STE-147.1: /spec-write's `Open questions / risks / inconsistencies` block contains zero literal `AC-STE-<N>.<M>` references whose source is the toolkit's own internal spec set, when /spec-write is run from a downstream project.
- AC-STE-147.2: Capability gaps surface as plain prose (e.g., "no Linear project milestones available — milestone-attach skipped"), with no bare AC identifier in the message body.
- AC-STE-147.3: User-authored ACs from the active project (legitimate `AC-<bound-tracker-id>.<N>` references in the FR being drafted) are NOT scrubbed — only toolkit-meta AC IDs are stripped.
- AC-STE-147.4: Test fixture covering the milestone-attach-no-op path asserts the rendered summary contains no `AC-STE-118` substring.

## Technical Design

Touchpoint: `skills/spec-write/SKILL.md` § 7 (Report) plus the in-band capability checks that produce the "Open questions" prose. Replace AC-ID-to-string templating with a static plain-language map keyed by capability name (e.g., `milestone_attach_unavailable`, `tracker_sync_failed`). No behavior change in the underlying milestone-attach / sync code paths — text-only fix.

Distinguishing toolkit-meta AC IDs from user ACs: toolkit-meta references are ones the skill code itself emits about the toolkit's own spec set (these are the ones to scrub). User-authored AC references in FR markdown bodies are emitted by the user, not the skill code, and pass through unmodified.

## Testing

- Unit: render the summary for the milestone-attach-no-op branch; assert no `/AC-STE-\d+/` matches in the output.
- Unit: render the summary on a happy path that mentions a user FR's AC (e.g., "covered by AC-STE-200.3"); assert the user reference passes through.
- Integration: re-run /smoke-test post-fix; F1 should not re-appear in the next findings file.
