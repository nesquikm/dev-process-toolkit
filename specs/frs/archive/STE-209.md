# STE-209: Setup-time canonical-shape hygiene: settings.json self-mod, .mcp.json shape, flutter doctor {#STE-209}

## Requirement

`/setup` writes three canonical shapes that the host (Claude Code harness) and per-stack tooling reject. Each blocks a documented happy-path step on first use.

**Findings bundled:** F2, F3, F3b, F6 from `/tmp/dpt-findings-2026-05-04-flutter-app.md` (Flutter calculator smoke, 2026-05-04). Supersedes STE-202 AC.4 — empirical verification during M54 brainstorm proved that `/setup` writing `.claude/settings.json` with even a narrow `Bash(git *)` + `Bash(gh *)` allowlist still triggers the harness self-modification denial; the original STE-202 AC.4 ("trim _common to git/gh only") is necessary but not sufficient.

**Symptoms observed in `dpt_test_lib2`:**

1. **F3 + F3b (CRITICAL).** Step 6 ("**Required, abort on failure**") writes `.claude/settings.json` with the canonical `_common` allowlist (`Bash(rm *)`, `Bash(mv *)`, `Bash(git *)`, `Bash(gh *)`, etc.). The harness denies any glob-shaped Bash rule (`Bash(<cmd> *)`) regardless of how narrow the list, citing `"Writing to the agent's own settings.json in the working repo to grant blanket Bash permissions is self-modification of agent permissions."` Empirically confirmed in M54 brainstorm: `Bash(git *)` + `Bash(gh *)` alone still denies. Step 6 cannot complete on a fresh user's first invocation in default permission mode.
2. **F6.** `docs/setup-tracker-mode.md` § MCP detection (lines 91–94) ships the canonical Linear MCP entry as `{"transport": "streamable-http", "url": "https://mcp.linear.app/mcp"}`. Claude Code's `/doctor` rejects with `"mcpServers.linear: Does not adhere to MCP server configuration schema"`. The schema actually expects `{"type": "http", "url": "..."}`. The `transport` field name appears nowhere in Claude Code's MCP schema docs.
3. **F2.** `examples/flutter-dart/gate-commands.md` declares `fvm flutter` as the invocation, but the doctor probe (step 1b) checks `flutter --version` (bare). On a fresh machine without `fvm`, the example's gate commands fail and the gap is invisible until `/gate-check` runs. Doctor should match the user's declared invocation.

## Acceptance Criteria

- AC-STE-209.1: `/setup` step 6 no longer triggers the harness self-modification denial when writing `.claude/settings.json` on a fresh repo. The fix is implementation-defined within two acceptable branches: (a) replace `templates/permissions.json _common` globs (`Bash(<cmd> *)` shapes) with an explicit-subcommand allowlist (e.g., `Bash(git status)`, `Bash(git diff)`, `Bash(git log)`, `Bash(git show)`, `Bash(gh pr list)`, `Bash(gh pr view)`, etc.) covering the read-only commands the toolkit actually invokes from skill prose, OR (b) skip writing `.claude/settings.json` at `/setup` time entirely and emit the proposed JSON to stdout with a copy-paste instruction (`Add the following block to .claude/settings.json via /permissions or by hand: ...`). Pick whichever clears the harness denial when verified empirically against the current Claude Code release; document the chosen branch in `docs/setup-reference.md` § Step 6. {#AC-STE-209.1}
- AC-STE-209.2: Empirical verification gate — implementer MUST reproduce the F3b denial against current main (write a `.claude/settings.json` with the proposed allowlist into a directory Claude Code recognizes as the active project) and confirm the denial does NOT fire on the chosen approach before merging. Negative-case fixture: an FR-level integration test that writes the proposed allowlist via the Write tool to a temp project root and asserts non-denial. Failing this gate means AC.1 is unproven; ship is blocked. {#AC-STE-209.2}
- AC-STE-209.3: `docs/setup-tracker-mode.md` § MCP detection (and any sibling template, doc, or skill prose emitting the canonical Linear MCP entry) updated to use `{"type": "http", "url": "https://mcp.linear.app/mcp"}`. The `transport` field is dropped entirely. Same shape-correction applied to any Jira / custom-tracker MCP entries the toolkit emits. {#AC-STE-209.3}
- AC-STE-209.4: New `/gate-check` probe `mcp-config-shape` validates emitted `.mcp.json` entries against the Claude Code MCP server configuration schema (or an inline regex/JSON-schema check that catches the `transport` typo). Probe runs against the toolkit's own templates and against the user-project `.mcp.json` if present. Severity: ADVISORY for user-project (don't break existing repos); ERROR for toolkit-shipped templates and docs. {#AC-STE-209.4}
- AC-STE-209.5: `/setup` step 1b (doctor) probes the stack invocation declared in CLAUDE.md, not the bare command. Concretely: when `examples/flutter-dart/gate-commands.md` (or rendered CLAUDE.md) declares `fvm flutter ...`, the doctor probe runs `fvm flutter --version`, not `flutter --version`. Implementation: doctor probe reads the per-stack invocation prefix from the rendered CLAUDE.md (or a per-stack manifest) rather than hard-coding `flutter`. Same pattern applies to any future stack with a wrapper (`bun` vs `node`, `pnpm` vs `npm`, etc.) — the probe matches the declared invocation. {#AC-STE-209.5}
- AC-STE-209.6: Backwards-compat — existing repos with already-written `.claude/settings.json` and `.mcp.json` are untouched by this FR. No `/setup --migrate` prompt; no rewrite. Drift surfaces only via the new ADVISORY probe (AC.4) and an analogous ADVISORY probe `setup-permissions-shape` that flags glob-shaped Bash rules in user-project `.claude/settings.json`. Migration is operator-driven (operator runs `/setup --migrate` if/when they want to converge); no automatic content change. {#AC-STE-209.6}

## Technical Design

**F3b fix path — preferred branch (a) "explicit subcommands."** Audit the toolkit's skill prose for every `Bash(...)` invocation; the result is a small set of read-only git/gh subcommands (e.g., `git status`, `git diff`, `git log`, `git show`, `git rev-parse`, `git ls-files`, `gh pr list`, `gh pr view`, `gh issue list`, `gh repo view`). Write rules also include destructive ops (`git add`, `git commit`, `git mv`, `git push`, `gh pr create`) — these may still trigger the harness denial when added in a glob form, so they're omitted from `_common` and added to per-skill blocks (where the per-skill scope justifies the grant).

If branch (a) still triggers the denial despite no globs, fall through to branch (b): step 6 prints the proposed JSON and instructs the operator to add it via `/permissions` slash-command or by hand. Step 6 no longer aborts on a missing file — it surfaces a `setup_settings_manual_install` capability row in the closing summary and proceeds. The toolkit's `setup-output-completeness` gate-check probe (#17) is updated to treat a missing `.claude/settings.json` as ADVISORY (not ERROR) when the closing summary carries `setup_settings_manual_install`.

**F6 .mcp.json shape correction.** Pure docs/template edit. The wrong shape lives in (at least) `docs/setup-tracker-mode.md` § MCP detection. Audit other emit sites:

- `docs/setup-tracker-mode.md` (canonical reference)
- `templates/CLAUDE.md.template` (no MCP block today, confirm)
- `skills/setup/SKILL.md` § step 7 (dry-run JSON preview)
- `examples/typescript-bun/`, `examples/flutter-dart/`, `examples/python-uv/` (per-stack tracker docs)

Replace every `transport: "streamable-http"` with `type: "http"`. Schema-validate via the new `mcp-config-shape` probe.

**F2 flutter doctor probe.** `skills/setup/SKILL.md` step 1b currently hard-codes `flutter --version` for the Dart stack. The fix: read the per-stack invocation prefix from `examples/<stack>/gate-commands.md` (or a future `examples/<stack>/doctor-invocation.txt` manifest). For Flutter the invocation is `fvm flutter` if `fvm` is declared in CLAUDE.md, else bare `flutter`. Generalizable: each per-stack example carries the canonical invocation; doctor reads it; probe matches.

**Non-breaking guarantee.** No existing user file is modified by this FR. Existing settings.json with globs continues to work (operators already authorized the broad rules; the harness denial only fires on `/setup`'s **write**, not on pre-existing content). Existing .mcp.json with `transport: streamable-http` keeps working until the operator re-runs `/setup --migrate`. Both shapes surface only as ADVISORY drift in `/gate-check`.

## Testing

- Integration test (AC.1 + AC.2): in a freshly-initialized scratch repo, run `/setup` end-to-end (dry-run mode acceptable) and assert step 6 completes without harness denial; the resulting `.claude/settings.json` content matches the chosen branch's shape.
- Unit test (AC.3): grep `templates/`, `docs/`, `skills/`, `examples/` for `transport.*streamable-http` — zero matches after the edit.
- Unit test (AC.4): probe `mcp-config-shape` against a fixture `.mcp.json` with the wrong shape → ADVISORY (user-project) or ERROR (template). Against the corrected shape → clean pass.
- Integration test (AC.5): on a Flutter project with `fvm flutter` declared, doctor probe invokes `fvm flutter --version`. On a Flutter project without `fvm` declared, probe falls back to bare `flutter --version`.
- Integration test (AC.6): existing `dpt_test_lib`-style repo with `transport: streamable-http` in `.mcp.json` runs `/gate-check` cleanly aside from a single ADVISORY row — no automatic rewrite, no abort.

## Notes

- This FR was added during M54 brainstorm (2026-05-04) after the Flutter-app smoke produced F3b's empirical confirmation that STE-202 AC.4's "trim to git/gh" fix is insufficient. STE-202 AC.4 is dropped; STE-202 AC.5 (`branch_template` trunk-OK), AC.6 (`docs_default_applied`), AC.1/AC.2/AC.3 are unaffected.
- Branch (b) of AC.1 (skip-writing-and-emit-copy-paste) is the safer fallback if the harness denies even explicit-subcommand allowlists. Preferred branch (a) keeps `/setup` autonomous on a fresh repo, which matches the existing "Required, abort on failure" contract.
- AC.5's per-stack invocation pattern is small and contained today (Dart only ships `fvm flutter` as a wrapper), but the design generalizes: when M55+ adds `examples/typescript-bun/` with `bun ...` invocations and a future stack uses `pnpm`, the same probe-from-manifest pattern applies.
- F1/F4 ("agent rationalized past `requires-input` step 7b") are NOT in this FR — they're folded into STE-199's revision (Auto Mode does NOT relax `requires-input`). Different root cause: F3b is a host/harness rejection, F1/F4 is the agent's own behavioral rationalization.
- Capability: a tracker-mode `/spec-write` run that creates this FR surfaces `filename_policy_override` per the unconditional rule.
