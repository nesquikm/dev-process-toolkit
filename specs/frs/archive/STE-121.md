## Requirement

`/spec-write` SKILL.md step 0b explicitly says: "tracker mode → omit `id:`, emit compact `tracker: { <mode>: <tracker-id> }`" (STE-110 AC-STE-110.1, M29 v1.30.0). The prose is correct. But the LLM-as-runtime authors the YAML by hand, and the smoke-test 2026-04-27 (run #3, against v1.31.0 — post-STE-110) reproduced `id: STE-120` on a freshly /spec-written tracker-mode FR. Probe-13 (`identity_mode_conditional`) catches it at gate time at error severity (M29 STE-110 AC-STE-110.4 flip), so `/gate-check` exits 1 on every new FR.

This is the M31-class discretionary-check pattern (STE-117/118/119): SKILL.md prose alone is not load-bearing; the rule has to live in helper code or a post-write self-check that survives across sessions and users. Probe-13 already lives in code; what's missing is a **system-enforced canonical-shape generator** (helper) plus a **post-write self-check inside `/spec-write`** so the malformed file never reaches the gate.

## Technical Design

**Helper module:**

```typescript
// adapters/_shared/src/fr_frontmatter.ts
export interface FRFrontmatterInput {
  id?: string;            // required in mode-none, forbidden in tracker mode
  title: string;
  milestone: string;
  createdAt: string;      // ISO 8601 timestamp
}

export interface TrackerBinding {
  key: "linear" | "jira" | string;  // adapter key
  id: string;                        // e.g., "STE-121"
}

export class InvalidFrontmatterInputError extends Error {}
export class InvalidTrackerShapeError extends Error {}
export class FRFrontmatterShapeError extends Error {}

export function buildFRFrontmatter(
  spec: FRFrontmatterInput,
  trackerBinding?: TrackerBinding,
): string {
  // Validate mode/binding consistency
  if (trackerBinding && spec.id) {
    throw new InvalidFrontmatterInputError(
      "tracker mode forbids id: field — pass spec.id=undefined when trackerBinding is provided",
    );
  }
  if (!trackerBinding && !spec.id) {
    throw new InvalidFrontmatterInputError(
      "mode-none requires spec.id (fr_<ULID>) — pass trackerBinding when in tracker mode",
    );
  }
  // Reject verbose `{ key, id, url }` shape
  if (trackerBinding && "url" in (trackerBinding as Record<string, unknown>)) {
    throw new InvalidTrackerShapeError(
      "verbose tracker shape forbidden per STE-110 AC-STE-110.2 — pass { key, id } only",
    );
  }
  // Render canonical YAML
  const lines: string[] = ["---"];
  lines.push(`title: ${escapeYamlScalar(spec.title)}`);
  lines.push(`milestone: ${spec.milestone}`);
  lines.push(`status: active`);
  lines.push(`archived_at: null`);
  if (spec.id) {
    lines.push(`id: ${spec.id}`);
  } else if (trackerBinding) {
    lines.push(`tracker:`);
    lines.push(`  ${trackerBinding.key}: ${trackerBinding.id}`);
  }
  lines.push(`created_at: ${spec.createdAt}`);
  lines.push("---");
  return lines.join("\n") + "\n";
}
```

**`escapeYamlScalar` rule:** plain ASCII titles use bare scalars; titles containing `"`, `\\`, `:`, `#`, or starting with whitespace use double-quoted form via `JSON.stringify`. Em-dash (U+2014) does not require escaping.

**`/spec-write` post-write self-check call site (SKILL.md step 0b):**

## Notes

**Sequencing constraint:** Independent of STE-122 and STE-123 — different files, different probes, no shared module. Phase A parallel.

**Why a helper instead of just tightening prose:** STE-110 already tightened prose in M29; the prose flip alone did not prevent the regression. The same M31-class lesson applies: when SKILL.md prose is the only enforcement, the LLM-as-runtime can drift. A helper makes the canonical shape impossible to deviate from at the call site, and the post-write self-check provides the second-line defense.

**Out-of-tree adapter compat:** `buildFRFrontmatter` accepts any string for `trackerBinding.key`, so out-of-tree adapters using arbitrary tracker keys (e.g., `github`, `youtrack`) continue to work. The helper does not validate the key against a known-list.

**Archive immutability:** the migration script is dry-run only and excludes `specs/frs/archive/`. Archive files keep whatever frontmatter shape they had at archival time per STE-22 / AC-STE-18.4. If the operator wants to clean archive too, that's a separate explicit decision via `git mv` + manual edit.

**Probe-13 stays at gate-time:** the post-write self-check is the in-band first line of defense; probe-13 stays at error severity at gate-check time as the safety net for any path that bypasses /spec-write (e.g., manual FR-file edits).
