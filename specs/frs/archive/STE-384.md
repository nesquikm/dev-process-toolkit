# STE-384: Path-drift meta-test: retired `.dpt-locks/` + `.dev-process/` literals cannot reappear {#STE-384}

## Requirement

STE-382 routes every `.dpt` path through `dpt_paths.ts`, but nothing stops a future edit from re-introducing a raw `.dpt-locks` or `.dev-process` literal in prose or code. The retired literals are especially prone to resurrection: they appear in ~170 places across specs, docs, skills, adapters, and tests today, and copy-paste from older files is exactly how stale literals seep back after a rename.

STE-49 ("Purge v1 `specs/archive/` Drift") shipped this defense for this reason: `tests/archive-path-drift.test.ts`, a grep meta-test that fails loudly if the retired literal reappears under `docs/`, `skills/`, `README.md`, and the live cross-cutting specs. Clone that shape for the `.dpt/` rename.

The rule is a deterministic test, not LLM judgment — the recurring lesson of the STE-220 → STE-226 → STE-238 chain is that prose-only rules drift and byte-checkable gates hold.

## Acceptance Criteria

- AC-STE-384.1: **Drift meta-test.** New `tests/dpt-path-drift.test.ts` (STE-49 shape) greps the live tree for the retired literals `.dpt-locks` and `.dev-process` and fails naming `file:line` for each survivor. Scope covers `plugins/dev-process-toolkit/{docs,skills,adapters,templates}` and `README.md` — **a floor, not a ceiling**: per AC.3 and STE-49's own shape, the live cross-cutting specs (`specs/requirements.md`, `specs/technical-spec.md`, `specs/testing-spec.md`) are additionally scanned as **optional** targets behind the `existsSync` guard. See § Notes, "Cross-cutting spec sweep". {#AC-STE-384.1}
- AC-STE-384.2: **History allow-list.** Legitimate historical mentions are exempt: `CHANGELOG.md`, `specs/frs/archive/**`, and `specs/plan/archive/**` record what the old layout was and must not be rewritten. **Test files carrying negative-case decoys are exempt on the same principle** (see § Notes, "Decoy carve-out"): `*.test.ts` under the scanned tree constructs the retired literals deliberately, to assert they are *not* scanned and *not* fallen back to — there the retired literal is the subject under test, and deleting it would destroy STE-382's regression guard. The allow-list is explicit and commented — an unexplained exemption is a bug, so each exemption states *why* it exists, not merely *that* it does. {#AC-STE-384.2}
- AC-STE-384.3: **Optional-target tolerance + stale-comment fix.** Scan targets pass through STE-49's `.filter(existsSync)` guard so the test tolerates absent optional targets (a downstream consumer running this plugin's `/gate-check` may not carry every cross-cutting spec file). **The justification is re-stated correctly:** STE-49's own comment claims the filter exists because "this repo's own `specs/` is gitignored (dogfood workspace)" — that claim is **false** as of 2026-07-15 (`specs/` carries 416 tracked files; no ignore rule matches it). The filter is retained for genuine optional-target tolerance, and the stale comment in `tests/archive-path-drift.test.ts` is corrected in the same edit so it stops propagating. {#AC-STE-384.3}
- AC-STE-384.4: **Tripwire proven.** Fixture coverage asserts the test actually fires — a fixture reintroducing `.dpt-locks/<ulid>/spec-research-result.txt` in a scanned path fails with the offending `file:line`, and a fixture carrying only allow-listed mentions passes. {#AC-STE-384.4}
- AC-STE-384.5: **Full gate green.** `bun test` from `plugins/dev-process-toolkit` passes against the post-STE-382/STE-383 tree with zero surviving literals outside the allow-list; no probe regresses. {#AC-STE-384.5}

## Technical Design

Filled at implementation. Reference: `tests/archive-path-drift.test.ts` (STE-49) — same grep-and-name shape, same `baseTargets` / `optionalTargets.filter(existsSync)` split.

**The `.dev-process` pattern MUST be dot-anchored** (verified 2026-07-15, `/implement` Phase 1). A bare `dev-process` grep matches **`dev-process-toolkit`** — this plugin's own name — which appears in ~80 files across every scan target, so the naive pattern fails instantly with a wall of false positives and would have to be neutered to pass. The dot-anchored form `\.dev-process` has **zero** false positives against the live tree: the plugin name never appears dot-prefixed (it is always path-separated, `plugins/dev-process-toolkit/`, or slash-prefixed in skill invocations, `/dev-process-toolkit:implement`). Measured at plan time: `\.dev-process` → 9 real hits, all genuine retired literals; `\.dpt-locks` → 41 hits. Both patterns must survive the STE-382/STE-383 sweep at **0 hits** outside the allow-list. A future maintainer widening the pattern to bare `dev-process` will drown the test — the anchor is load-bearing, not stylistic.

**On AC.3's stale comment.** The 2026-07-15 spec-research seed reported the `specs/`-is-gitignored rationale as fact because it faithfully quoted the test's comment; verification (`git check-ignore` + `git ls-files specs/ | wc -l` → 416) falsified it. The comment is the drift, not the filter. Correcting it in the same edit prevents the next research seed from re-propagating it — a small instance of the same "retired claim reappears via copy-paste" failure this whole FR guards against.

## Testing

Shipped 2026-07-15. Full gate green at **4307 pass / 11 skip / 0 fail** on FR exit.

| File | Role |
|---|---|
| `tests/_dpt-path-drift.ts` | **The primitive** (non-test module): `RETIRED_LITERAL_PATTERNS`, `findRetiredLiterals(targets)`, `assertNoRetiredLiterals(targets)`, `dptPathDriftScanTargets(...)`. |
| `tests/dpt-path-drift.test.ts` | **The deliverable** — thin STE-49-shaped wrapper over the primitive. |
| `tests/m104-ste-384-dpt-path-drift.test.ts` | 40 tests pinning the primitive's contract across AC.1–AC.5. |
| `tests/archive-path-drift.test.ts` | AC.3 stale-comment correction only; STE-49's assertions untouched and still green. |

**Two structural decisions, both forced rather than stylistic:**

1. **The primitive is factored out of the `.test.ts`.** STE-49's inline shape cannot satisfy AC.4 — a gate that only ever runs against a clean tree is indistinguishable from one that greps for a typo. Parameterizing the target list is the minimum change that makes "it fires" checkable.
2. **It is a `_`-prefixed non-test module, not exports from a `.test.ts`.** Verified on bun 1.3.14: importing a test file from another registers its tests into the importer's run. Precedent: `tests/_skill-md.ts`.

**AC.4 — the tripwire is proven non-vacuous by a 4-mutation matrix**, since a green suite cannot distinguish a real gate from one that greps for a typo. Each mutation of the primitive turns AC.4 red: (A) `assertNoRetiredLiterals` never throws → 3 fail; (B) `isExempt` always false, allow-list neutered → 2 fail; (C) patterns grep for an unmatchable typo → 4 fail; (D) dot-anchor removed → 1 fail. **Mutation C is the decisive one** — it is precisely the vacuity mode this AC exists to rule out. The primitive was restored after each (md5 unchanged). The thrown message was checked as a string, not merely for throwing: `docs/regressed.md:3: The fork writes \`.dpt-locks/01HZZZQK5T.../spec-research-result.txt\`` — a real `file:line` plus offending text, against the FR's exact named STE-230 shape.

**The dot-anchor is load-bearing and measured:** bare `dev-process` → **295 hits across 84 files** (all `dev-process-toolkit`, the plugin's own name); dot-anchored `\.dev-process` → **0**. Pinned so a future maintainer cannot widen the pattern and drown the test.

**Implementation choices worth preserving:** a single grep over the pattern alternation (so a line carrying both literals yields one survivor, not two); `-H` forces the filename prefix, which grep omits for a lone file operand and would break the `path:line:text` parse; `-I` skips binaries, whose `Binary file X matches` output carries no line number and would not parse.

**Calibration at FR exit:** no snapshot bumped or weakened; no cap/ceiling/count meta-test touched on this branch.

## Notes

- **Depends on STE-382 + STE-383** — the test asserts the *end state*, so it is red until both land. Build order A → B → C is load-bearing, not cosmetic.
- **Decoy carve-out (2026-07-15, STE-383 `/tdd` audit — PROVISIONAL, confirm at `/implement` Phase 4).** This FR's declared scope (AC.1) includes `adapters/`, which STE-49's precedent did not scan. That widening collides with STE-382's regression guards: **10 `.dpt-locks` literals survive under `adapters/` in test files** — `local_provider.test.ts:251,259,324,368,378,382,384,387,392` (9) and `dpt_paths.test.ts:94` (1). Every one is a deliberate negative-case assertion ("the legacy literal is NOT scanned"; "no fallback leg exists"). They would make this FR **red on arrival** through no fault of its own, and deleting them would delete the proof that STE-382's fix works. **Decision: allow-list `*.test.ts` under the scanned tree, with a comment naming this rationale** (AC.2 requires exemptions be explained, not merely listed). Rejected alternative — allow-listing the two files by name: brittle, since any future decoy test re-reds the gate for a legitimate reason. **Residual risk, accepted:** a genuinely stale literal in a test file goes uncaught by *this* grep. That risk is small and largely self-policing — a test that writes to a retired path fails loudly at runtime on its own (observed twice during M104's build: `token-usage.test.ts`'s fixture and `token-ledger-hook.test.ts`'s constant both surfaced as hard failures, not silent drift). The live prose+code surface this FR exists to protect stays fully covered.
- **Cross-cutting spec sweep (2026-07-15, STE-384 `/tdd` RED — PROVISIONAL, confirm at `/implement` Phase 4).** The RED pass found **10 live, tracked, retired `.dpt-locks` literals in this repo's own cross-cutting specs** — `specs/requirements.md` (3) and `specs/technical-spec.md` (7) — which STE-382's and STE-383's sweeps both missed. The FR was internally inconsistent about whether they are in scope: **AC.1's scope list omits `specs/`**, while **AC.3's rationale** ("a downstream consumer running this plugin's `/gate-check` may not carry every cross-cutting spec file") only makes sense if they *are* targets, and STE-49 — the precedent this FR clones — scans exactly those three files as `optionalTargets.filter(existsSync)`. **Decision: scan them (as optional targets) and sweep the 10 literals.** Rationale: leaving known-false prose standing is the precise drift this FR exists to catch, and the alternative drains AC.3's `existsSync` guard of its stated purpose. The clincher is `specs/technical-spec.md:305`, which carries `git branch -r --contains .dpt-locks/<ulid>` — the *exact invalid command* (`--contains` takes a commit, not a path) that STE-383 deliberately removed from `docs/layout-reference.md`. Shipping a milestone that fixes that sentence in `docs/` while leaving it standing in `technical-spec.md` would be incoherent. Same shape, and same resolution, as the "Unowned doc drift" finding folded into STE-383 AC.6 earlier in this milestone. Cost: 10 lines. AC.1's scope list is therefore read as a mandatory floor; the test pins it as a superset check.
- **Meta-test, not a `/gate-check` probe, by choice.** The drift it guards is internal to this repo's source tree, not a property of a consumer project. Consumer-side path correctness is already covered by STE-382's probe re-pointing and STE-383's `check-ignore` assertions. Adding a probe would export a rule consumers cannot violate.
- Origin: `/brainstorm` 2026-07-15; the STE-49 precedent surfaced via the 2026-07-15 spec-research seed, whose one factual error is recorded in AC.3 and fixed by it.

## Implementation notes

_Advisory findings from the `/implement` Phase 3 review, persisted for the audit trail (non-blocking — the FR shipped with these open by operator decision)._

- The decoy carve-out's real blast radius is narrower than § Notes states, and its residual-risk framing is subtly off. `tests/` is not a scan target at all — only `adapters/**/*.test.ts` (2 files today) can carry an exempt literal — so "a stale literal in a test file goes uncaught" overstates the exposure. And the "self-policing, fails loudly at runtime" argument covers *executable* test paths only; it does **not** cover stale prose in test *comments* (e.g. `local_provider.test.ts` has decoy mentions inside comments). That comment surface is the carve-out's true residual edge. Non-blocking; a per-line `// drift-allow` marker would close it precisely if it ever bites.
