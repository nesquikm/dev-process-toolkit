# STE-145: Document allowed-tools skill frontmatter convention (or remove from spec-review) {#STE-145}

## Requirement

`skills/spec-review/SKILL.md:4` is the only one of 14 skills that declares `allowed-tools:` in frontmatter. The other 13 skills don't carry the field. The inconsistency raises a question with two equally-plausible answers:

- **(a)** Every skill should opt-in (implicit allowlist enforcement convention) — and 13 skills are leaking permissions because they don't.
- **(b)** Skills inherit the session's allowlist by default, and `spec-review` is over-restricting itself (Read/Glob/Grep only — no Edit, Write, Bash) because it's read-only by design.

The latter is the likely truth. `spec-review` IS read-only by design — it audits specs against the codebase and reports findings, never edits. A narrow allowlist is correct posture. But the choice is unexplained anywhere in `README.md` or `docs/skill-anatomy.md`. Future contributors authoring a new read-only skill would have to dig into `spec-review`'s frontmatter to discover the pattern; ones authoring a write-capable skill might erroneously add `allowed-tools:` thinking it's required.

Fix by documenting the convention.

## Acceptance Criteria

- AC-STE-145.1: `docs/skill-anatomy.md` carries a section (or paragraph in an existing section) describing the `allowed-tools:` frontmatter field's purpose and the toolkit's convention: declare it for read-only-by-design skills; omit it for skills that need write or shell access. The section names `spec-review` as the canonical example.
- AC-STE-145.2: `skills/spec-review/SKILL.md` carries no behavioral change — its `allowed-tools:` frontmatter remains as-is. Its prose may optionally cross-reference `docs/skill-anatomy.md` for the convention.
- AC-STE-145.3: A new test (or extension of an existing skill-shape test) asserts the convention: every skill that carries `allowed-tools:` is documented as read-only in its prose; skills without `allowed-tools:` are unrestricted (the test is a structural-audit test, not a behavioral one). Optional under the no-users principle — the documentation alone clears the inconsistency. Implementer's call whether to add the test.
- AC-STE-145.4: `bun test` is green; `/gate-check` clean.
- AC-STE-145.5: A future contributor reading `docs/skill-anatomy.md` understands when to declare `allowed-tools:` and when to omit it without having to grep across 14 SKILL.md files.

## Technical Design

**Single-file edit** — `docs/skill-anatomy.md`. Add a section (or paragraph in an existing section about frontmatter):

```markdown
### `allowed-tools` (optional)

Declare `allowed-tools:` in the SKILL.md frontmatter when the skill is **read-only by design** — i.e., it audits, analyzes, or summarizes without ever editing files or running shell commands. The field is a comma-separated allowlist of Claude Code tools. Example: `Read, Glob, Grep` for an audit-only skill.

If the skill needs write or shell access (most skills do — `setup`, `implement`, `gate-check`, etc.), **omit the field** and inherit the session's full allowlist.

The toolkit's canonical read-only example is `skills/spec-review/SKILL.md`, which carries `allowed-tools: Read, Glob, Grep` because it audits specs against the codebase without ever editing.
```

Adjust the wording to match the surrounding prose style. If `skill-anatomy.md` already has a frontmatter-fields section, extend it; otherwise add a new `### Frontmatter` or `### Optional fields` heading.

Cross-reference (optional): a one-line note in `skills/spec-review/SKILL.md` near the frontmatter, e.g., `<!-- allowed-tools: read-only convention; see docs/skill-anatomy.md -->` (only if Tier-3 cleanup option (b) is the chosen direction in STE-143; otherwise the comment adds noise).

For AC-STE-145.3 (optional test), shape:

```ts
test("AC-STE-145.3 — allowed-tools convention", () => {
  for (const skill of skills) {
    const fm = parseFrontmatter(readFileSync(skillPath));
    if (fm["allowed-tools"]) {
      // assert skill is documented as read-only
      const body = readFileSync(skillPath, "utf8");
      expect(body).toMatch(/read-only|audit/i);
    }
  }
});
```

The test is heuristic; mark optional.

## Testing

- **Manual** — read the new section in `docs/skill-anatomy.md` end-to-end; confirm the convention reads clearly.
- **`bun test`** — green (no required test changes; AC-STE-145.3 is optional).
- **`/gate-check`** — clean.

## Notes

The audit recommended option (a) (document the convention) over option (b) (remove `allowed-tools:` from `spec-review`). (a) preserves the read-only safety net for `spec-review` AND clarifies the pattern for future contributors. (b) sacrifices the safety net for uniformity.

Per the no-users-yet principle, this is a low-priority polish FR. The inconsistency doesn't cause runtime bugs; the documentation gap is the only real cost. Including it in M39 because it's a 5-minute edit that bundles cleanly with the other audit-cleanup work.
