# STE-168: /gate-check probe portability — conditional probes #9b and #10 {#STE-168}

## Requirement

`/gate-check` ships two probes that unconditionally reference toolkit-internal paths and crash when run in a non-toolkit host project:

- **Probe #9b** (`SKILL.md:35` area) reads `.claude-plugin/plugin.json` `version` to verify version-consistency. Only plugin projects have that file; end-user projects don't.
- **Probe #10** (`SKILL.md:39` area) greps `plugins/dev-process-toolkit/templates/CLAUDE.md.template` to validate the template's shape. That path only exists inside the toolkit repo itself.

When `/gate-check` runs in an end-user project (the common case), both probes either crash or report a misleading failure for a check that simply doesn't apply.

The fix is small: wrap both probes in path-existence checks. If the path is absent, the probe emits an `n/a` row in the gate report rather than a fail. The probe still runs (and still gates) when the path is present — toolkit-self-runs continue to enforce consistency unchanged.

Out of scope:
- Other toolkit-internal references in `/gate-check` (none identified by the 2026-04-30 audit).
- Refactoring the probe runner / report shape.

## Acceptance Criteria

- AC-STE-168.1: Probe #9b (`.claude-plugin/plugin.json` version match) wraps an existence check around the file. When the file is absent, the probe emits an `n/a` row with the reason "no plugin manifest in this project — probe skipped"; when present, the probe runs unchanged.
- AC-STE-168.2: Probe #10 (`plugins/dev-process-toolkit/templates/CLAUDE.md.template` grep) wraps an existence check around the file. When absent, emits `n/a` with the reason "toolkit template not present — probe skipped"; when present, runs unchanged.
- AC-STE-168.3: Test fixture: `/gate-check` against a stub end-user project (no `.claude-plugin/`, no `plugins/`) reports both probes as `n/a` rows, **not** as failures, and the overall gate passes if other probes are clean.
- AC-STE-168.4: `/gate-check` self-run inside the toolkit repo (where both files exist) continues to fire both probes and enforce them — no regression in toolkit-side gating.
- AC-STE-168.5: `bun test` green; `/gate-check` clean (toolkit-self-run); `/spec-review M48` reports no deviations.

## Technical Design

**Per-probe pattern.** Each probe gets a one-line existence guard before its check:

```
- if !fs.existsSync(path): emit n/a row, return
- else: run existing probe logic
```

Where the probe logic lives (skill prose vs. TS helper) is a per-probe judgment call:

- If the probe is implemented in `SKILL.md` prose only (LLM-driven), the SKILL prose gains a "skip if path absent" sentence at the top of the probe.
- If the probe has a TS helper under `adapters/_shared/src/gate_check_*.ts`, the helper gains the existence guard and the SKILL prose is updated to match.

The 2026-04-30 audit didn't disambiguate which probe is prose vs. TS — the implementer resolves this at TDD-RED time.

**Report shape.** `n/a` is a third state alongside `pass` / `fail` in the existing gate report. If the report renderer doesn't already support `n/a`, this FR adds it (one-line addition; the renderer is small).

**No probe deletion.** The toolkit-self-run case still wants both probes firing as gates. The portability fix is "skip when not applicable", not "remove". A future FR could split these into a separate "toolkit maintainer" probe set and exclude them from end-user gate-check entirely — but that's structural change for later, not this FR.

## Testing

- Unit tests for the existence guard (one per probe).
- Integration test fixture: a stub end-user project directory at `tests/fixtures/end-user-project/` (no `.claude-plugin/`, no `plugins/`, just a minimal CLAUDE.md). Run `/gate-check`-equivalent against it, assert both probes emit `n/a`.
- Toolkit-self regression: `/gate-check` in the toolkit repo still fails if either path's content drifts (e.g., plugin.json version mismatches marketplace.json).
