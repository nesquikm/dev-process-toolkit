# STE-196: docs(skills/simplify): document no-op gate-skip carve-out in SKILL.md {#STE-196}

## Requirement

`/simplify` declares "Per skill rules, gate re-run is skipped (re-running an unchanged tree against the prior clean gate is wasted tokens)" when invoked on a clean tree with a prior clean `/gate-check` signal. The rationale lives only inside the skill body — the user-facing `plugins/dev-process-toolkit/skills/simplify/SKILL.md` does NOT surface the carve-out, so the "Per skill rules" phrasing reads as a reference to undocumented policy. Four consecutive smoke runs (#6, #7, #8, #9) have flagged this gap; the cumulative re-confirmation is itself signal — the documentation gap is durable, not transient.

Add a `## When this is a no-op` section to `skills/simplify/SKILL.md` naming the two preconditions and the rationale.

Out of scope:

- Changing the no-op behavior. The skip is correct: re-running an unchanged tree against the prior clean gate produces no signal change. This FR documents the existing behavior, doesn't modify it.
- Adding the same section to other skills. `/simplify` is the only skill with the "Per skill rules" gate-skip carve-out; other skills do their own gate handling.

## Acceptance Criteria

- AC-STE-196.1: `plugins/dev-process-toolkit/skills/simplify/SKILL.md` gains a `## When this is a no-op` section with the heading anchor `{#when-this-is-a-no-op}` (matches the heading-anchor convention from `/spec-write` § 3 stable anchor IDs). {#AC-STE-196.1}
- AC-STE-196.2: The section names both preconditions explicitly: (a) tree clean (no diff vs. prior `/simplify` entry — nothing to simplify), (b) prior `/gate-check` returned clean (no failing gate to re-verify). {#AC-STE-196.2}
- AC-STE-196.3: The section names the rationale: re-running an unchanged tree against the prior clean gate produces no signal change, so the gate re-run is skipped to save tokens. The skill body's existing "Per skill rules" phrasing now references this section by name. {#AC-STE-196.3}
- AC-STE-196.4: Regression guard: `grep -E '^## When this is a no-op' plugins/dev-process-toolkit/skills/simplify/SKILL.md` returns exactly one match. {#AC-STE-196.4}

## Technical Design

### Files touched

### Section shape

```markdown
## When this is a no-op {#when-this-is-a-no-op}

`/simplify` exits as a no-op without re-running gate checks when both:

1. The working tree is clean — no diff vs. the prior `/simplify` entry, so there is nothing to simplify.
2. The prior `/gate-check` returned clean — no failing gate to re-verify.

Rationale: re-running an unchanged tree against an already-clean gate produces no signal change, so the gate re-run is skipped to save tokens. The skill emits a single line acknowledging the no-op and exits.
```

(Exact wording at implementer's discretion; the three required signals are the two preconditions and the rationale.)
