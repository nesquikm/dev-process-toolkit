## Requirement

M20's STE-72 ships mechanical signature extraction for TypeScript only (`typedoc → ts-morph → regex-fallback`). Python projects fall through to `regex-fallback` with a "manual review advised" banner. Python is a target stack for this toolkit (per existing `examples/python/`); the parity gap is felt directly.

This FR adds a Python strategy to `signature_extractor.ts` that spawns `griffe dump <package>` (JSON output, the same tool mkdocstrings uses) and parses the result into `ModuleSignatures[]`. No bundled Python helper — `griffe` is a real CLI emitting structured data, installed via `pip install griffe` (one command).

Two-tier chain: `griffe` (preferred) → `regex-fallback` (banner). No middle tier (rejected: bundled stdlib `ast` helper — adds maintenance for marginal benefit since griffe install is one command).

Strategy recorded as `strategy: "griffe"` in `SignatureGroundTruth.strategy` (Schema Z). The enum widening (declared as `export type Strategy` at `adapters/_shared/src/signature_extractor.ts:28`) lands in the cross-cutting sibling FR (STE-105).

## Technical Design

**No bundled Python helper.** Unlike the Dart strategy (STE-103 — must bundle a helper because no Dart CLI emits API JSON), the Python strategy invokes the user-installed `griffe` CLI directly. This honors Approach 2 strictly: external tool preferred → regex-fallback, no middle tier.

**TypeScript wrapper:**

```typescript
function extractViaGriffe(projectRoot: string): ModuleSignatures[] {
  const pkgName = derivePackageName(projectRoot);  // from pyproject.toml etc.
  if (!pkgName) throw new GriffeError("could not derive package name");
  const result = spawnSync(
    "griffe",
    ["dump", pkgName],
    { cwd: projectRoot, encoding: "utf-8", timeout: 60_000 },
  );
  if (result.status !== 0 || !result.stdout) {
    throw new GriffeError(`griffe failed: ${result.stderr}`);
  }
  const griffeJson = JSON.parse(result.stdout);
  return griffeJsonToModuleSignatures(griffeJson);
}
```

**`griffeJsonToModuleSignatures()` translator:**

griffe emits a recursive object tree:

```json
{
  "name": "mypackage",
  "kind": "module",
  "members": {
    "submodule": {
      "name": "submodule",
      "kind": "module",
      "filepath": "/abs/path/submodule.py",
      "members": {
        "MyClass": { "kind": "class", "members": { ... }, "docstring": {...} },
        "my_function": { "kind": "function", "parameters": [...], "returns": {...}, "docstring": {...} },
        "MY_CONST": { "kind": "attribute", "annotation": {...} }
      }
    }
  }
}
```

The translator flattens this: each module-kind node becomes one `ModuleSignatures` entry; its function/class/attribute children become `ExportSignature[]`. Class members are NOT flattened to top-level — they live inside the class signature (mirroring how M20's TS strategies handle methods inside classes).

**Package-name derivation precedence:**

1. `pyproject.toml` `[project].name` (PEP 621)
2. `pyproject.toml` `[tool.poetry].name`
3. `setup.cfg` `[metadata].name`
4. `setup.py` parsed for `setup(name=...)` (regex; brittle but salvageable)
5. First top-level `__init__.py`-bearing directory under `src/` or project root → use directory name
6. None of the above → emit warning, fall through to regex-fallback

**Why griffe and not pdoc / sphinx-autodoc / stdlib-ast:** griffe is purpose-built for programmatic API surface extraction (it's what mkdocstrings consumes). Modern, actively maintained, JSON output first-class. pdoc is renderer-first (HTML/Markdown), JSON is secondary. sphinx-autodoc is heavyweight and config-coupled. stdlib `ast` is too low-level — re-implements what griffe gives for free.

## Notes

**Why no bundled Python helper:** griffe is one `pip install griffe` away. Bundling a stdlib-`ast` fallback adds maintenance for marginal benefit — anyone serious about Python docs already has griffe (or will, after the `/setup` probe surfaces the install command per STE-105).

**Translator complexity warning:** griffe's JSON shape is rich but not 1:1 with Schema Z. The translator is the bug-prone part. Tests at AC-STE-104.9 cover the kind mapping cases; new fixtures should be added when downstream Python projects surface unhandled cases (e.g., `@overload`, generic class type vars).

**Out-of-scope kind enum widening:** Schema Z's `kind` enum (`"function" | "class" | "type" | "interface" | "const" | "enum"`) doesn't have stack-specific members like `"protocol"` or `"dataclass"`. The translator squashes these to closest-fit per AC-STE-104.2. A future FR could widen the enum if downstream pain warrants — out of scope here.
