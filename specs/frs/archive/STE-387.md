# STE-387: Review contract: human approves Summary + Requirement + ACs; gates own Technical Design + Testing {#STE-387}

## Summary

The toolkit now states plainly what a human is expected to review when approving a spec: the
summary, the requirement, and the acceptance criteria. The detailed design and testing sections
are aimed at implementation and are verified by the deterministic gates and the audit that traces
every criterion to code and tests. The framing appears at the approval prompt, in the methodology
docs, and in generated project instructions.

## Requirement

A summary section alone does not reduce review burden if the reviewer still believes the detailed
sections demand their reading — the FR got longer, and nothing was won. The survey of 2026-07-16
found that **no shipped file states what a human reviews at spec approval**: the draft gate
presents "the FR draft" whole, § 4 says "show the user what was written", and all surrounding
prose is gate mechanics. This FR writes the contract down at every human-approval touchpoint:

> The human review surface is `## Summary` + `## Requirement` + `## Acceptance Criteria`.
> `## Technical Design` and `## Testing` are implementation-facing appendices, verified by the
> deterministic gates and the TDD audit that traces every AC to code and tests — not by the
> approving human.

The strongest existing precedent is the `--no-tech` flow, which already treats Requirement + ACs
vs Technical Design + Testing as separable review audiences (the flag defers the latter pair to a
technical reviewer); this FR generalizes that split from "optional deferral" to "stated default".
Wording at every site is **token-free** (cites mechanisms, never tracker-ID tokens): the skills
STE-token ceiling sits at 245/245 with zero headroom, and consumer-facing templates carry a
zero-tolerance namespace rule.

## Acceptance Criteria

- AC-STE-387.1: **Draft gate.** `/spec-write` § 0b step 4 states the review contract in one sentence directly after the approval prompt, before the marker-mechanics paragraphs. {#AC-STE-387.1}
- AC-STE-387.2: **Review-and-confirm + Rules.** § 4 and the `## Rules` draft-approval rule each carry the same contract in one sentence; § 5's "just like step 4" pointer inherits it unchanged. {#AC-STE-387.2}
- AC-STE-387.3: **Methodology.** `docs/sdd-methodology.md` gains the spec-time review scope as a sibling to the implementation-time human-review section (or a Key Principles entry) — same partition, same ownership statement. {#AC-STE-387.3}
- AC-STE-387.4: **Workflow overview.** The FR-draft gate row in `docs/workflow-overview.md`'s gate table names the review surface. {#AC-STE-387.4}
- AC-STE-387.5: **Consumer template.** `templates/CLAUDE.md.template` carries one reviewer-facing bullet stating the contract; the templates namespace test stays green (no tracker-ID / milestone-number literals). {#AC-STE-387.5}

## Notes

- **Contract is prose, not enforcement** — deliberately. The reviewer may always read more; the
  contract states what they are *expected* to cover for approval to be meaningful. No probe polices
  reviewer behavior.
- **The `--no-tech` cross-reference** is cited by mechanism (flag name), not by token, to hold the
  245 ceiling.
- **No new capability keys** — the contract changes what the prompt says, not what the skill emits;
  the closing-summary static map is untouched.
- Origin: GitHub issue #39 → `/brainstorm` 2026-07-16 — the operator's own framing: additive
  summary works only when paired with an explicit statement of the review surface.
