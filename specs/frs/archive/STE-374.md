# STE-374: /report-issue evidence gate — verify-before-alarm + incident-session capture {#STE-374}

## Requirement

`/report-issue` selects the session transcript to attach by **most-recent-mtime** JSONL in
the cwd-slug directory (STE-229 AC-STE-229.4 — a skill cannot read its own session UUID at
runtime, so it approximates "the current session" by recency). When the incident being
reported happened in an *earlier* session, that heuristic attaches a transcript that does
**not contain the incident**. Observed 2026-07-09: a high-severity report about a GF-23
`/deps-research` injection shipped a GF-24 transcript in which the reported fork was never
even invoked — the "verbatim payload" quoted in the report appears nowhere in the attached
evidence.

Compounding it, `/report-issue` publishes a **high-severity** report with no check that its
own evidence is present. So a report can loudly claim a high-severity fault while the
attached transcript neither reproduces nor even mentions it — precisely the false-alarm the
2026-07-09 gist turned out to be.

This FR makes `/report-issue` evidence-based. It (a) captures the session that actually
**contains** the incident — marker-matched selection over the K most-recent candidate
JSONLs, with the current most-recent-mtime pick as an explicit fallback; and (b) gates
severity on evidence — a `high`/`critical` report whose incident is absent from the captured
transcript is auto-capped and tagged `unverified`, with a `verification` block in
`metadata.json`. The check is **advisory-transparent, never blocking**: the maintainer still
receives even unverified reports, but the evidentiary gap is made explicit rather than
hidden behind a scary severity label. This is the STE-296 pattern — verify a claim against
artifacts, not against its own narrative.

## Acceptance Criteria

- AC-STE-374.1: **Incident-session selection.** `/report-issue` session selection gathers the K most-recent candidate JSONLs in the cwd-slug directory (K default configurable, e.g. 5) and selects the one whose content matches the incident markers — the skill name(s) named in the narrative and/or user-supplied marker string(s). The current most-recent-mtime pick becomes the explicit fallback used only when no candidate matches. {#AC-STE-374.1}
- AC-STE-374.2: **Selection-path token.** A capability token records which path fired: `report_issue_session_matched_marker` (a marker-matched session was chosen) or `report_issue_session_fallback_mtime` (no candidate matched; most-recent-mtime used). Both are registered in `CANONICAL_CAPABILITY_KEYS` and rendered in the `/report-issue` closing summary via the § 7 static plain-language map; exactly one emits per run. {#AC-STE-374.2}
- AC-STE-374.3: **Evidence verification.** Before publishing, `/report-issue` checks whether the captured transcript actually contains the reported incident — grep the transcript for the offending fork/skill output and/or the narrative's key symptom markers. The searched markers and the boolean result are recorded. {#AC-STE-374.3}
- AC-STE-374.4: **Severity cap on absent evidence.** When declared severity is `high` or `critical` AND the incident is **not** found in the captured transcript (AC-STE-374.3), severity is auto-capped and `metadata.json` carries `verified: false` plus a `verification: { searched: <bool>, found: false, markers: [...] }` block. When the incident **is** found, `metadata.json` carries `verified: true` and severity is unchanged. `low`/`medium` reports are recorded with the `verification` block but never capped. {#AC-STE-374.4}
- AC-STE-374.5: **Verification tokens.** `report_issue_severity_capped_unverified` MUST emit (literal, backticked) when the cap fires; `report_issue_evidence_verified` MUST emit when the incident is found in the captured transcript. Both registered in `CANONICAL_CAPABILITY_KEYS` + rendered in the § 7 map. {#AC-STE-374.5}
- AC-STE-374.6: **Advisory, not blocking.** The report still publishes on the unverified path — the capped severity + `unverified` tag + `verification` block make the evidentiary gap explicit in `metadata.json` (and the gist), but nothing aborts the upload. The maintainer receives even unverified reports; the change is that a report can no longer *silently* assert high severity without evidence. {#AC-STE-374.6}
- AC-STE-374.7: **Tests.** Coverage: wrong-session-newest + incident-in-older-session → marker match selects the incident session (+ `report_issue_session_matched_marker`); no candidate matches → mtime fallback (+ `report_issue_session_fallback_mtime`); high-severity + incident-absent → capped + `verified: false` + `report_issue_severity_capped_unverified`; high-severity + incident-present → uncapped + `verified: true` + `report_issue_evidence_verified`; low-severity → recorded, never capped. {#AC-STE-374.7}

## Technical Design

**Scope: `/report-issue` session-selection upgrade + a pre-publish evidence check + metadata + tokens. No change to the redaction pipeline or gist-upload mechanics beyond the added `verification` field.**

**Files touched:**
- `adapters/_shared/src/` (new, e.g. `report_issue_session_select.ts`) — `selectIncidentSession(cwdSlugDir, markers, { k })`: list the K most-recent JSONLs by mtime, grep each for the incident markers, return `{ path, matched: boolean }` (matched=false ⇒ the most-recent-mtime fallback). Pure/​testable; the K-scan is bounded and logged when it truncates.
- `adapters/_shared/src/` (new, e.g. `report_issue_verify_evidence.ts`) — `verifyIncidentEvidence(transcriptPath, markers)`: return `{ searched, found, markers }`; the caller applies the severity cap.
- `skills/report-issue/SKILL.md` — wire both helpers: select the incident session (AC-STE-374.1/.2), run the evidence check before upload (AC-STE-374.3), apply the cap + write the `verification` block to `metadata.json` (AC-STE-374.4/.6), and emit the four new tokens (AC-STE-374.2/.5). Keep the current behavior byte-identical on the happy path (marker match + evidence present).
- `skills/spec-write/SKILL.md` § 7 (and/or the shared capability map) — register the four new keys + rendered prose (`/report-issue` shares the static map for its `report_issue_*` rows).
- `adapters/_shared/src/closing_summary_capability_keys.ts` — register the four keys.

**Markers.** The incident markers are (a) skill/fork names parsed from the narrative (e.g. `deps-research`, `spec-write`) and (b) any explicit marker string(s) the user supplies at capture time. Selection prefers a candidate matching the most specific marker; ties break by mtime (recency).

**Why advisory, not blocking.** A maintainer needs even a mis-captured report to triage — blocking would lose signal. The cap + tag preserves the report while removing the false "high severity, trust me" assertion. This mirrors STE-296: the fix is to verify the claim against the artifact, not to suppress the claim.

## Testing

- `report-issue-session-select.test.ts` (new) — newest-session-lacks-incident + older-session-has-it → older selected, matched=true; no match → newest-by-mtime, matched=false; K-truncation logged.
- `report-issue-verify-evidence.test.ts` (new) — markers present → found=true; markers absent → found=false; multi-marker union.
- Meta-tests under `tests/` — `skills/report-issue/SKILL.md` carries the selection step, the evidence check, the cap rule, the `verification` metadata block, and the four `MUST emit` token directives; the § 7 map carries the four keys.

## Notes

- **Origin.** 2026-07-09 `/report-issue` gist (high severity) → `/brainstorm` (Approach #2, user-selected) → `/spec-write`. Sibling of STE-373, which fixes the upstream half of the same cascade (the deps-research fork enforcement asymmetry that produced the scary-looking output in the first place).
- **STE-229 constraint.** A skill still cannot read its own session UUID; this FR does not change that. It replaces the *single-shot* most-recent-mtime heuristic with a *marker-matched search over recent candidates*, keeping mtime as the fallback — so a correctly-captured session is byte-identical to today, and only the wrong-session case changes.
- **Verify-before-alarm generalizes.** The evidence gate is scoped to `/report-issue` here, but it encodes the STE-296 lesson (verify claims against artifacts, not narrative) that the 2026-07-09 incident violated at both ends: the fork's output was trusted-then-feared without verification, and the report was filed without checking its own evidence.

## Implementation notes

- `critical` severity forward-compat — the cap condition names `high` or `critical`, but the `/report-issue` severity prompt (§ 2) currently offers only `low` / `medium` / `high`, so the cap fires on `high` in practice; the `critical` branch is handled forward-compatibly and the cap-rule prose was clarified rather than expanding the severity enum (which would exceed the FR's "no change beyond the verification field" scope and break the pinned doc-conformance test).
