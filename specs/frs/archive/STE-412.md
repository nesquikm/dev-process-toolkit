# STE-412: /setup step 8: no specs/frs/ scaffold over a live monolithic requirements.md {#STE-412}

## Summary

Setup's spec scaffold unconditionally seeds placeholder keep-files into the per-FR folder, and on a legacy project that still keeps its feature sections inline in the requirements file those placeholders used to make the split migration believe the split had already happened. The detector no longer falls for it, but the scaffold still creates a confusing half-hybrid tree. This FR conditions the scaffold: when live feature sections are present in the existing requirements file, setup leaves the per-FR folder untouched, keeps the existing file, and says why — pointing at the upgrade chain instead.

## Requirement

STE-189's canonical scaffold inventory declares the `.gitkeep` stubs "emitted unconditionally". That rule exists so restrictive invocation phrasing ("only /setup completes here", "minimal mode") cannot shrink `/setup`'s artifact list — it was never a decision about legacy layouts. On a monolithic tree the stubs are worse than useless: pre-STE-409 they disabled the split detector outright (the gist's facet 2); post-STE-409 they still produce a half-hybrid tree — live `### FR-N:` sections next to a scaffolded empty per-FR skeleton — that confuses the operator and the later split triage.

The carve-out is keyed on the same detector the migration uses (`monolithSplit.detect`, both limbs): at step-8 time on a legacy tree, `specs/frs/` is absent or contentless, so the conjunction fires exactly when the split is genuinely pending.

## Acceptance Criteria

- AC-STE-412.1: step 8, when `monolithSplit.detect(projectRoot)` applies (live `### FR-N:` sections + no per-FR content), does NOT create or seed `specs/frs/{,archive/}`; the `specs/plan/{,archive/}` scaffolding and the technical/testing template copies are unaffected. {#AC-STE-412.1}
- AC-STE-412.2: the existing `specs/requirements.md` is never overwritten by the template copy — pinned as an explicit never-overwrite rule in the step 8 prose (pre-existing spec files are the user's; templates fill only gaps). {#AC-STE-412.2}
- AC-STE-412.3: when the guard fires, `/setup` prints a one-line explanation naming the detected monolithic layout and the follow-up (run the upgrade after bootstrap completes) and appends a `## /setup audit` row recording the skip with its reason. {#AC-STE-412.3}
- AC-STE-412.4: the STE-189 scaffold-deliverables inventory paragraph gains the matching carve-out sentence, and a meta-test pins both the step-8 guard clause and the inventory sentence. {#AC-STE-412.4}
- AC-STE-412.5: non-monolithic trees scaffold byte-identically to today — the requirements template's own `### FR-N: [Feature Name]` placeholder heading does not match the live-section pattern (`FR-<digits>` required), so a fresh scaffold never trips its own guard. {#AC-STE-412.5}

## Technical Design

**`skills/setup/SKILL.md` § 8.** Before creating `specs/frs/{,archive/}`, run the read-only detector check via a deterministic bun invocation named in the SKILL (a one-liner importing `monolithSplit` from `adapters/_shared/src/migrations/monolith_split` and printing the `detect(projectRoot)` verdict as JSON). When `applies: true`: skip the `specs/frs/` mkdir + `.gitkeep` seeding entirely, keep every existing spec file untouched, print the one-line explanation, and append the audit row (step:8, value `frs-scaffold-skipped`, reason naming the monolithic layout). The step-6c `upgrade_available` hint — which fires reliably once STE-409 lands — complements this from the registry side.

**Inventory paragraph** (the STE-189 "Scaffold deliverables" block): amend artifact class (5) with the carve-out clause so the inventory and step 8 cannot drift apart.

## Notes

The unconditional posture of STE-189 stays for every other artifact class and every non-monolithic tree. The flat `specs/plan.md`-file vs `specs/plan/`-directory coexistence on legacy trees is left to the monolith-split migration's freeze leg (out of scope here).
