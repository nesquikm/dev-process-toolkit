# STE-395: De-list /upgrade from the user-invocable surface: user-invocable false + count/taxonomy/remedy re-key {#STE-395}

## Summary

The upgrade command leaves the slash menu. It stays fully functional and Claude can still run it, but it no longer occupies a slot in the user-facing command surface — the new gate probe becomes the documented way to discover that a project needs upgrading. Every count, table, diagram, and remedy string describing the old surface is re-keyed in the same change, including a guard against a silent-disable trap where rewording a single line would quietly switch the count-drift probe off entirely.

## Requirement

M108 shipped `/upgrade` as user-invocable. The maintainer's stated problems are command-surface bloat and the unreliability of humans remembering to run maintenance commands. STE-394 solves discovery via a gate probe; this FR removes the command from the menu it no longer needs to occupy.

`user-invocable: false` hides a skill from the slash menu while leaving model invocation intact (`docs/skill-anatomy.md:44-46`; the flag is asserted at `tests/tdd-orchestrator-frontmatter.test.ts:77-80` and the surviving model-invocation path at `:82-87`), so the probe's notification still leads to the skill running. `disable-model-invocation: true` is explicitly NOT used — it would break that path and trips probe #59, whose allowlist is `["setup"]` only (`disable_model_invocation_allowlist.ts:38-40`).

The change is not docs-only. Probe #57 `public_surface_count_drift` counts skill *directories* (`public_surface_count_drift.ts:51-82`) and never reads `user-invocable:` frontmatter, so it will not catch a botched de-listing; every count token is hand-maintained and must move in lockstep.

## Technical Design

Frontmatter flip plus coordinated doc re-key. The blast radius was enumerated exhaustively at spec time; the affected assertion sites are:

- `tests/gate-check-public-surface-count-drift.test.ts:392` (16 commands), `:402` (`Eight additional skills`), `:405` (`24 (16 + 8)`), `:418` (16+8 split). Synthetic-fixture cases (`skillsCount: 23`) and the on-disk-total assertions at `:427-433`, `:442-451` do **not** change.
- `tests/m108-ste-391-docs-pins.test.ts:100-160` — rename and re-key; `:104` and `:128` invert; `:115-117` inverts once the README command-table row is deleted; `declaresDispatchOnly` renames to `declaresNonUserInvocable` (predicate unchanged).
- `tests/m108-ste-391-upgrade-skill.test.ts:39` inverts from `not.toMatch` to `toMatch`; `:139-165` must move in lockstep with the `skills/setup/SKILL.md:160` remedy re-key or they enforce a dead instruction.

`/upgrade` becomes the first `user-invocable: false` skill in the toolkit that is not a `context: fork` dispatch child — hence AC-STE-395.3's taxonomy split rather than a pure count bump.

**Ordering constraint.** AC-STE-395.6's deletions shift every README line number below them. Sequence the README work as: re-key content-located tokens first, delete the table row and mermaid node last. Do not drive README edits off the line numbers recorded in this FR after the first deletion lands.

**Known residual tension** (accepted, not resolved here): AC-STE-395.4 requires `CLAUDE.md:15` keep a `<N> user-invocable + <N> dispatch` literal so the probe's `splitMatch` keeps matching, but the second number becomes 8 while only 7 of those skills are genuinely fork-dispatch. The word "dispatch" is therefore load-bearing for a regex while being taxonomically wrong — the same inaccuracy AC-STE-395.3 repairs in the README. Widening the regex to admit a more accurate word is the clean fix and is deferred, because it must land in the probe and the pinned literal together.

## Notes

Build order: lands after STE-394, because the re-keyed remedy strings point at that FR's probe.

This FR reverses a decision from the 2026-07-17 brainstorm that deliberately made `/upgrade` user-invocable (after the `--migrate` flag collision with tracker-mode switching, STE-98, kept "for operator muscle memory"). The reversal is justified by STE-394's probe supplying the discovery path the original decision lacked — de-listing alone would have worsened discoverability, which is why the two FRs ship together and in this order.

Optional hardening deferred, not rejected: extend probe #57 to count on-disk `user-invocable: false` frontmatter and add a `USER_INVOCABLE_FALSE_ALLOWLIST` mirroring `DISABLE_MODEL_INVOCATION_ALLOWLIST`, so the flag stays governed rather than hand-maintained.
