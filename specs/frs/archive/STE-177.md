# STE-177: /setup Socratic prompt sequencing + shared docs/patterns.md rule {#STE-177}

## Requirement

`/brainstorm` and `/spec-write` enforce one-question-at-a-time prompting (Socratic discipline) — wait for the answer before asking the next, even when questions look independent. `/brainstorm` carries an explicit rationalization-prevention table; `/spec-write` repeats the rule in three places (requirements, technical-spec, testing-spec, plan sections).

`/setup` does not. Step 7d at `plugins/dev-process-toolkit/skills/setup/SKILL.md:195` literally instructs **"Three yes/no prompts in this exact order, after 7c"** — ambiguous wording the LLM reads as "ask the trio in one turn." Adjacent risk: steps 7b → 7c → 7d → 7e are four separate question sites within ~30 lines of skill prose; without an explicit Socratic rule binding them together, the LLM tends to collapse them into a single turn at the handoff between 7b's tracker-mode question and 7e's release-files emission.

Result: an operator running `/setup` interactively for the first time gets walled with three docs-mode yes/no prompts plus the tracker-mode + branch-template + release-files questions in a single turn, instead of the patient back-and-forth `/brainstorm` and `/spec-write` already deliver. The user explicitly flagged this on 2026-05-01 ("we use socratic discussion in /brainstorm — /setup asks a lot of questions at once").

The fix has two parts:

1. **Extract the Socratic-prompting rule** into `plugins/dev-process-toolkit/docs/patterns.md` as a numbered pattern, so it lives in one canonical place. `/brainstorm`'s rationalization-prevention table is the only existing source of truth for this rule today; relocating it lets future skills inherit it by section anchor (the same convention `/spec-write` already uses for "Tracker Mode Probe" and "Stable Anchor IDs").
2. **Update `/setup` SKILL.md** to honor the rule: rewrite step 7d's lede so it explicitly says "ask one at a time, wait for each answer before asking the next," and add a top-level Socratic-prompting rule (under `## Process` or `## Rules`) that names steps 7b, 7c, 7d, 7e and references Pattern 26 by anchor. `/spec-write` and `/brainstorm` get back-pointers; `/brainstorm` keeps its inline rationalization-prevention table because that is the table the new pattern is extracted from.

Out of scope:
- Editing `/implement`, `/ship-milestone`, `/gate-check`, `/docs` SKILL prose. Their prompt sites are already single-question (worktree question, single approval, rare fallback). The audit lands in this FR's `## Notes` so the absence-of-change is documented; SKILL.md edits to those four are explicitly **not** part of the canonical set.
- Changing `/brainstorm`'s rationalization-prevention table itself — that table is the canonical statement of the rule and pattern 26 is extracted **from** it. Both copies remain (table in `/brainstorm`, prose statement + table in `patterns.md`); the pattern is the authoritative cross-skill reference, the inline table preserves `/brainstorm`'s self-contained shape.
- Implementing the rule via runtime checks. The discipline is prose-level; gate-check has no probe for "did the LLM ask one question or three." This is a documentation contract, not a deterministic gate.

## Technical Design

Three-file edit pass, all prose. No code, no probes, no template change.

### File 1: `plugins/dev-process-toolkit/docs/patterns.md`

Insert a new section between `## Pattern 25: Dogfooding Discovery` (line ~515) and `## Root Spec Hygiene` (line ~542). Shape:

```markdown
## Pattern 26: Socratic Prompting {#pattern-socratic-prompting}

When a skill needs more than one piece of input from the user, ask one question per turn and wait for the answer before asking the next — even when the questions look independent, even at phase transitions, even when the user is being responsive. Batching prompts trades operator clarity for token-efficiency the LLM does not actually need.

**Where this rule fires.** Any skill that prompts at multiple sites within a single invocation: `/brainstorm` (clarifying questions), `/spec-write` (per-section author flow), `/setup` (steps 7b–7e tracker mode → branch template → docs modes → release files). Single-prompt skills (`/implement` worktree question, `/ship-milestone` approval, `/gate-check` no-CLAUDE.md fallback) are out of scope by construction — there is no second question to batch with.

**Rationalization-prevention table.** The one-at-a-time rule fails most often when the user is responsive and the LLM rationalizes batching as efficient. Watch for these excuses:

| Excuse | Reality |
|--------|---------|
| These two questions are independent | Ask the first, wait, then the second |
| Efficiency wins — batch them | Efficiency ≠ batching; the Socratic form is the gate |
| The user is responsive, I'll batch | Responsiveness is not license to batch |
| We're at the handoff, last chance | Phase transitions are where batching happens most — same rule applies |

Skills that adopt this pattern reference it by anchor (`docs/patterns.md § Pattern 26 — Socratic Prompting {#pattern-socratic-prompting}`) rather than restating the rule inline; the canonical statement lives here, the rationalization-prevention table is the operational nudge.
```

## Notes

### Origin

User feedback on 2026-05-01 (this conversation): *"we use socratic discussion in /brainstorm — I found that /setup asks a lot of questions at once. Can we fix it? Maybe there are more places that should be fixed too?"*

User explicitly approved the **broad** scope (option 2) over the narrow scope (option 1) — extracting the rule into `docs/patterns.md` rather than inlining the fix only in `/setup`. Rationale recorded mid-conversation: low risk (pure prose, no code, no probes), `docs/patterns.md` already hosts similar shared rules (Tracker Mode Probe, Schema L), the only failure mode is a stale anchor caught by `/gate-check`'s dead-link probes.

### Audit of remaining prompt sites (per AC-STE-177.5)

Audited prompt-bearing skills outside `/setup` for batching risk. **All four are single-question by construction; no SKILL.md edits required.**

- **`/implement`** — `plugins/dev-process-toolkit/skills/implement/SKILL.md:13` asks one question (worktree vs. current branch). The user has set a durable preference for "current branch" via memory; the skill nonetheless prompts once per invocation. Single-question, no change required.
- **`/ship-milestone`** — description line 3 says "prompts once for approval" before the release commit. Single-question, no change required.
- **`/gate-check`** — `plugins/dev-process-toolkit/skills/gate-check/SKILL.md:136` asks "what commands to run" only when CLAUDE.md is absent. Rare fallback path; single question when it fires. No change required.
- **`/docs`** — no user-facing prompts in any of the three sub-modes (`--quick` / `--commit` / `--full`). LLM-prompts internal to fragment generation are out of scope (those go to the model, not the operator). No change required.

### Pairing with M49

This FR pairs in spirit with **STE-173** (M49, v2.2.0): both are docs-only `/setup` SKILL.md prose fixes that document or sharpen conditional behavior the operator otherwise reverse-engineers from the prompt. STE-173 documented commit-msg hook best-effort installation; STE-177 documents prompt-sequencing discipline. Same surface area (`plugins/dev-process-toolkit/skills/setup/SKILL.md`), same risk profile (zero — prose-only).

### Why a separate milestone (M50) rather than appending to a follow-up batch

Single-FR milestone is unusual for this repo (M44/M45/M46/M47/M48/M49 each carried 1–7 FRs), but appropriate here for three reasons:

1. The rule extraction touches `docs/patterns.md`, which is a stable cross-skill reference. Bundling it with unrelated future work would obscure the single coherent change.
2. M50 ships under v2.3.0 (minor — additive prose, no schema/contract change). A clean single-FR minor release matches the change's scope.
3. `/ship-milestone` regenerates the release commit from the milestone's plan file; a single-FR plan keeps the codename ("Socratic") tightly coupled to the change.
