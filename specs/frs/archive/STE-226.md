# STE-226: Auto-mode/`-p` carve-out: explicit pre-authorization marker (STE-220 third-attempt; deterministic at runtime) {#STE-226}

## Requirement

The Auto-mode/`-p` carve-out at `/spec-write` § 0b step 4 + § 4 + § 7a — and the parallel carve-outs at `/smoke-test` Phase 0 + `/conformance-loop` Phase 0 — currently rely on the LLM detecting `Auto Mode Active` in `<system-reminder>` blocks or inferring `claude -p` non-interactive mode from context. Two production smoke runs have falsified this contract at runtime:

- 2026-05-04 v2.8.0 Linear smoke F2-1 + Jira C-F1: `/spec-write` halted at the FR-draft gate under `claude -p`; chain broke. STE-213 (M55) had shipped a prose-only fix.
- 2026-05-05 v2.10.0 `/conformance-loop` iter-1 F1 + M1: `/spec-write` halted again under `claude -p` (F1, both adapters); `/smoke-test`'s own Phase 0 gate also halted (M1, requiring the `/conformance-loop` parent to inject an embedded pre-approval workaround). STE-220 (M56) had shipped a second prose-only fix that was already known broken.

Architectural lesson: instructional SKILL.md prose for the LLM is not a reliable substitute for an explicit byte-checkable contract when the LLM is processing the same prose to decide whether to apply it. Two prose-only attempts have falsified across four end-to-end smoke runs.

This FR replaces the brittle "auto-detect" path with an explicit, byte-checkable **pre-authorization marker** that parent skills inject into the prompt body when spawning child sessions under `claude -p`. The child skill detects the marker by literal string match — no inference, no LLM judgment, no `<system-reminder>` introspection.

Verified by `/smoke-test` Phase 2.X group 1 fixture against both Linear and Jira adapters.
- AC-STE-226.2: When `/spec-write` is invoked under `claude -p` WITHOUT the marker, the gates fire interactively (no auto-apply). The legacy `Auto Mode Active` system-reminder detection path is removed; the marker is the single deterministic mechanism.

- AC-STE-226.4: `.claude/skills/smoke-test/SKILL.md` and `.claude/skills/conformance-loop/SKILL.md` (project-local) include the marker line in every documented `claude -p ...` reference snippet that targets a skill carrying a gate (Phase 1 `/setup` spawn, Phase 2 chain spawns, Phase A `/smoke-test` spawns, Phase B `/spec-write` + `/implement` spawns).

## Technical Design

**Marker shape.** `<dpt:auto-approve>v1</dpt:auto-approve>` on its own line. XML-style for visual distinctness (won't collide with markdown, code, or natural prose); `v1` suffix permits future revision without breaking byte-grep tests. Detection is literal string match — no regex, no whitespace-tolerant parsing.

**`/spec-write` detection.** SKILL.md § 0b step 4 + § 4 + § 7a — replace the existing detection contract with:

> Default-apply `y` when the prompt body contains the literal line `<dpt:auto-approve>v1</dpt:auto-approve>`. The marker is the single mechanism; legacy auto-detect paths (`Auto Mode Active` system-reminder, `claude -p` inference) are removed in this FR.

The contract is byte-repeated at all three gate sites — § 0b step 4 (FR-draft gate, gates `Provider.sync` + FR file write), § 4 (review-and-confirm), § 7a (commit gate). Same byte-repeat discipline as the STE-220 contract, but the substance is the byte-checkable marker rather than the brittle inference rule.

**Parent-skill injection.** `/smoke-test` (Phase 1 `/setup` spawn, Phase 2 chain spawns) + `/conformance-loop` (Phase A `/smoke-test` spawns, Phase B `/spec-write` + `/implement` spawns) each emit the marker line into the heredoc body before the child slash command. Reference snippets in SKILL.md updated to show the marker line on its own line, immediately before the slash-command line.

Example (Phase A `/smoke-test` spawn in `/conformance-loop`):

```bash
CLAUDE_CONFIG_DIR=~/.claude-st claude -p \
  --plugin-dir "${PLUGIN_DIR}" \
  --permission-mode bypassPermissions \
  > "${LOG_LINEAR}" 2>&1 <<${EOF_TAG}
<dpt:auto-approve>v1</dpt:auto-approve>
/smoke-test --tracker linear --linear-team "${LINEAR_TEAM}"
${EOF_TAG}
```

**`/gate-check` probe.** New probe `auto_approve_marker_in_canonical_spawns` reads each SKILL.md, finds every `claude -p` heredoc/spawn block (regex anchor: `claude -p` followed by heredoc-on-stdin or `< /dev/null`), and asserts the marker line is in the same fence/block. Severity: error. Hard fail when missing. Probe lives next to existing #35 setup-permissions-shape in `plugins/dev-process-toolkit/skills/gate-check/SKILL.md`.

**Phase 2.X fixture extension.** `/smoke-test` Phase 2.X group 1 (STE-220) extends to two sub-fixtures:
- 1a (existing, retitled): `/spec-write` spawn carries marker → assert `spec_write_draft_default_applied` + `spec_write_commit_default_applied` audit rows present in stdout.
- 1b (new): `/spec-write` spawn omits marker → assert stdout ends at the gate prompt without the audit rows (proves marker is the *only* trigger; no spurious auto-apply).

Diagnostic on 1a regression: `STE-220 runtime regression: spec-write-marker-missing-audit-row`. Diagnostic on 1b regression: `STE-220 runtime regression: spec-write-marker-absent-but-auto-applied` (catches the inverse failure mode where the LLM auto-applies without a marker).

**Backward compatibility.** None required — `project_no_users_yet` per memory; the toolkit's own `/conformance-loop` is the only consumer that auto-injects, and it's project-local. The legacy `Auto Mode Active` system-reminder detection path is **removed**, not deprecated — single deterministic mechanism, no shim.

## Notes

This is STE-220's third attempt. STE-213 (M55) shipped prose-only; STE-220 (M56) shipped prose-only with byte-repeat discipline; both failed runtime checks. This FR adopts the byte-checkable marker mechanism that the smoke-test driver workaround already proved works end-to-end during the 2026-05-05 `/conformance-loop` iter-1 run (when both `/smoke-test` children proceeded through their Phase 0 gates after `/conformance-loop` injected an explicit pre-approval line into the prompt body).

The architectural shift: gate behavior is no longer "LLM judgment from indirect signals" but "explicit handoff from parent to child via a byte-grep-able token in the prompt body". The `/gate-check` probe is the safety net that ensures the marker is present in every documented spawn snippet — drift is structurally impossible without a probe failure.

Memory `feedback_smoke_post_ship_retroactive` applies to AC-STE-226.6: real-`-p` archival evidence may ship deferred `[~]` if the next /smoke-test or /conformance-loop run falls within M59's archival window.
