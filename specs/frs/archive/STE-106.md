## Requirement

During the v1.29.0 smoke-test run on a fresh Bun project, `/setup` self-aborted on writes to `.claude/settings.json` and `.mcp.json` ("sandbox blocked writes to that path (sensitive)") even though `Write` was in the parent's allowlist. The skill silently swallowed the failure and moved on, producing a CLAUDE.md but leaving the project without the permissions allowlist or MCP wiring it advertises in its own report.

Two failure modes:

1. The skill *self-imposes* the block (some skill rule treats `.claude/` as sensitive and skips). If real, that rule shouldn't fire when the skill itself is the writer.
2. The harness blocks the write (out of /setup's control). If real, /setup should fail loudly rather than continue with a partial scaffold.

Either way, the user-visible behavior must change: a project that has just run `/setup` MUST have `.claude/settings.json` and (when tracker mode != none) `.mcp.json` on disk, OR `/setup` MUST exit non-zero with a clear error.

## Acceptance Criteria

- AC-STE-106.1: After /setup runs in a project that doesn't have `.claude/settings.json`, the file exists with the canonical `permissions.allow` array (Bash patterns covering the detected stack's tooling: `Bash(bun *)`, `Bash(bunx *)` for Bun; `Bash(flutter *)`, `Bash(dart *)` for Flutter; `Bash(uv *)`, `Bash(python *)` for Python; plus `Bash(git *)`, `Bash(gh *)`, `Bash(mkdir *)`, `Bash(ls *)`, `Bash(rm *)`, `Bash(mv *)`, `Bash(cp *)` for all stacks).
- AC-STE-106.2: After /setup runs in a project where the user accepts a tracker mode (linear/jira/custom) and the project has no `.mcp.json`, the file exists with the corresponding `mcpServers` entry (e.g., `{"linear": {"type": "http", "url": "https://mcp.linear.app/mcp"}}` for Linear).
- AC-STE-106.3: If writing either file fails (sandbox refusal, permissions, ENOSPC, anything), /setup emits an NFR-10-shape error naming the file and the underlying error and exits non-zero. The skill MUST NOT continue with a partial scaffold.
- AC-STE-106.4: If `.claude/settings.json` already exists, /setup MUST NOT overwrite it. Skill behavior: read the existing file, compute the canonical-allow set for the detected stack, append any missing entries, and write back. Do not strip user additions. If the existing file is malformed JSON, abort with NFR-10 shape (don't silently rewrite).
- AC-STE-106.5: New `/gate-check` probe `setup-output-completeness`: if CLAUDE.md `## Task Tracking` declares a tracker mode != none, the project root MUST contain `.mcp.json` with the corresponding adapter entry. Probe lives at `plugins/dev-process-toolkit/adapters/_shared/src/setup_output_completeness.ts` (flat path matches existing convention — `identity_mode_conditional.ts`, `archive_plan_status.ts`, `root_hygiene.ts`, etc.); integration test at `plugins/dev-process-toolkit/tests/gate-check-setup-output-completeness.test.ts` (per STE-82 probe authoring contract).
- AC-STE-106.6: Skill SKILL.md prose is updated to remove any "skip if sensitive path" guard around `.claude/` writes. The skill is the canonical writer for `.claude/settings.json`; treating it as off-limits is a category error.
- AC-STE-106.7: Integration tests cover all four required scenarios — clean project (writes both files), pre-existing `.claude/settings.json` with user additions (merge), pre-existing malformed JSON (abort), tracker mode = none (no `.mcp.json` required) — via in-memory `mkdtempSync` fixtures in `plugins/dev-process-toolkit/tests/setup_merge_settings.test.ts` (helper) and `plugins/dev-process-toolkit/tests/gate-check-setup-output-completeness.test.ts` (probe). Convention shifted post-STE-92: AC-STE-92.5 explicitly forbids new on-disk fixture directories, so M29 follows the same in-memory pattern uniformly across all probe tests.
- AC-STE-106.8: `bun test` green. CHANGELOG entry under `### Fixed` cross-references this FR + smoke-test origin.

## Technical Design

**Skill change.** `plugins/dev-process-toolkit/skills/setup/SKILL.md` — find any prose that treats `.claude/` writes as sensitive/optional and rewrite. The canonical /setup steps that write `.claude/settings.json` and `.mcp.json` move from "best-effort" to "required, abort on failure".

**Merge logic for existing `.claude/settings.json`.** New helper `plugins/dev-process-toolkit/adapters/_shared/src/setup/merge_settings.ts`:

```typescript
export function mergeAllowList(existing: SettingsJson, canonical: string[]): SettingsJson {
  const existingAllow = existing.permissions?.allow ?? [];
  const merged = Array.from(new Set([...existingAllow, ...canonical]));
  return {
    ...existing,
    permissions: { ...existing.permissions, allow: merged },
  };
}
```

Pure function; trivially testable.

**Canonical allow-list per stack.** Lookup table at `plugins/dev-process-toolkit/templates/permissions.json` keyed by detected stack (`bun`, `node`, `flutter`, `python`, `generic`). Skill reads this file at write time. Adding a new stack = adding a row.

**New gate-check probe `setup-output-completeness`.** Reads CLAUDE.md `## Task Tracking`, asserts `.mcp.json` presence + matching `mcpServers` entry. Skipped when mode = none.

## Testing

`plugins/dev-process-toolkit/tests/gate-check-setup-output-completeness.test.ts` per STE-82 contract. Helper unit test at `plugins/dev-process-toolkit/tests/setup_merge_settings.test.ts`. Fixtures under `tests/fixtures/setup-output-completeness/`.
