# STE-182: Tighten /spec-write filename_policy_override prose when no user override {#STE-182}

## Requirement

`/spec-write`'s Step 7 closing-summary block unconditionally emits a `filename_policy_override` row on every tracker-mode FR write — this is correct per the SKILL.md's "Filename-policy override row" rule (every tracker-mode write surfaces the row regardless of whether the user proposed an alternative). The unconditional-emit contract is load-bearing: it gives the operator visibility into the fact that `Provider.filenameFor(spec)` is the authoritative filename source.

The 2026-05-01 Jira smoke (v2.3.0) caught a wording bug. The static plain-language map at `skills/spec-write/SKILL.md` § Step 7 carries this entry:

| Capability key | Rendered prose |
|----------------|----------------|
| `filename_policy_override` | `FR filename derived from tracker policy (Provider.filenameFor) — the user-proposed name was overridden by the adapter convention` |

The phrase "the user-proposed name was overridden" reads as if the user proposed a name and `/spec-write` silently replaced it. On the common path (pre-baked stub or no filename mention), no user name was proposed; the prose is misleading. The smoke F4 finding called out the false-positive shape: "the message text reads as if the user supplied a name and it was silently replaced. Cosmetic, but worth tightening the wording when no override actually happened."

The fix is conditional rendering: when no user filename was proposed, emit "no user override" prose; when the user did propose an alternative, emit "overrode user-proposed: <name>" prose. The unconditional-emit rule is preserved (the row still fires) — only the prose changes.

Out of scope:
- Removing the unconditional-emit rule itself. The SKILL.md's "Filename-policy override row" section explicitly requires the row to fire on every tracker-mode FR write, "regardless of whether the user proposed an alternative filename." That contract is preserved here. Only the rendered prose under the row changes.
- Changing the rendering convention for other capability keys in the static plain-language map. Only the `filename_policy_override` row is in scope.

## Acceptance Criteria

- AC-STE-182.1: `skills/spec-write/SKILL.md` § Step 7 static plain-language map's `filename_policy_override` row's "Rendered prose" column changes from a single static string to a conditional shape, parametrized on whether a user filename was proposed:
  - **No user override** (common path): `FR filename derived from tracker policy (Provider.filenameFor) → <filename> (no user override)`.
  - **User override** (rare): `FR filename derived from tracker policy (Provider.filenameFor) → <filename> (overrode user-proposed: <user-name>)`.
  Both renderings substitute `<filename>` with the actual `Provider.filenameFor(spec)` output (e.g., `STE-182.md`); the override variant additionally substitutes `<user-name>` with the user-proposed filename. {#AC-STE-182.1}
- AC-STE-182.2: The unconditional-emit rule from the existing SKILL.md "Filename-policy override row" section is preserved verbatim. `/spec-write` continues to emit the row on every tracker-mode FR write or import (per the existing import-path-coverage clause). The ONLY change is which of the two prose variants renders. {#AC-STE-182.2}
- AC-STE-182.3: The static plain-language map in SKILL.md gets an explicit note documenting the conditional-rendering rule under `filename_policy_override`: a one-line annotation immediately below the row explaining when each variant fires. The annotation is part of the SKILL.md so future LLM passes don't re-collapse the conditional into a single static string. {#AC-STE-182.3}
- AC-STE-182.4: `mode: none` paths remain exempt from the row entirely (per the existing SKILL.md rule "`mode: none` is exempt — the short-ULID stem is local-mint via `Provider.mintId()`, never policy-overridden, so the row is absent on local-mint runs"). This FR does not touch the local-mint exemption. {#AC-STE-182.4}
- AC-STE-182.5: Round-trip integration test `tests/spec-write-filename-policy-override-prose.test.ts` covers four cases:
  - (a) Linear new-FR run with no user-proposed filename → "no user override" variant renders; `<filename>` matches `Provider.filenameFor(spec)`.
  - (b) Linear new-FR run with explicit user-proposed `foo.md` → "overrode user-proposed: foo.md" variant renders.
  - (c) Jira new-FR run, no user filename → "no user override" variant; `<filename>` is the Jira-shape (e.g., `DST-6.md`).
  - (d) `mode: none` new-FR run → row absent (existing exemption preserved). {#AC-STE-182.5}

## Technical Design

### How the LLM picks the variant

`/spec-write` knows the user-proposed filename (or absence thereof) from the resolver-entry context. When `resolveFRArgument` returns `kind: fallthrough` and the free-form argument did not include a filename hint, the LLM has no user-proposed name in scope — variant (a) renders. When the user explicitly typed a filename in conversation (e.g., "create the FR at `specs/frs/foo.md`"), the LLM picks up the proposed name and variant (b) renders.

The conditional is LLM-applied, not code-applied: there is no helper function to call. The SKILL.md prose is the contract; the integration test asserts the right variant on each path.

### Annotation in SKILL.md

The annotation reads (verbatim):

```
> Annotation: render variant (a) when the resolver-entry context contains no user-proposed
> filename; render variant (b) when the user explicitly proposed an alternative. The row fires
> on both variants — only the prose differs. Mode: none is exempt entirely.
```

This sits as a blockquote below the row in the static plain-language map.

## Testing

- Integration test `tests/spec-write-filename-policy-override-prose.test.ts` (new) — 4 cases per AC-182.5.
- The test's regression detection mode: variant (a) MUST not contain the literal string "the user-proposed name was overridden" (the old prose); variant (b) MUST contain the literal "overrode user-proposed". This catches regression to the old single-static-string shape.
- `bun test` baseline grows by 4; no other regressions.

## Notes

- **Smoke-test source.** `/tmp/dpt-smoke-findings-2026-05-01-jira.md` § F4 logged this. The Linear smoke on the same date emitted the same misleading prose; the Jira finding articulated the wording fix.
- **Why preserve the unconditional-emit rule.** The earlier Jira smoke (logged in the SKILL.md commentary) showed that the row was previously emitted ONLY when the user proposed a filename. That regression hid the policy-override signal on the common path; the unconditional-emit rule was added to fix it. This FR does NOT undo that fix — it preserves the unconditional emit and only sharpens the prose.
- **Why not delete the override variant entirely.** The override variant (b) still fires on legitimate paths where the user's prompt explicitly included a filename (e.g., a smoke-driver pre-bake that names a target file). Deleting variant (b) would lose information on those paths. Keeping both variants gives the operator full visibility regardless of which path runs.
- **Interaction with STE-179, STE-181.** STE-179 adds Step 7a (commit prompt) before Step 7; this FR edits Step 7's static plain-language map. STE-181 edits Step 7's `Next:` line at the tail. The three edits touch different sub-regions of Step 7 and do not conflict. Land in any order.
