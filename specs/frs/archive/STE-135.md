# STE-135: Fix findFRByTrackerRef silent data-loss in tracker mode {#STE-135}

## Requirement

`findFRByTrackerRef` at `plugins/dev-process-toolkit/adapters/_shared/src/resolve.ts:167-172` requires the FR file's frontmatter to carry a non-empty `id:` line. STE-76 (M21) removed `id:` from tracker-mode FR frontmatter and `/gate-check` probe #13 (`identity_mode_conditional`, error severity post-STE-110) actively forbids it. Result: in tracker mode the function always returns `null` even when the FR file exists.

Worst-case consumer impact is silent data-loss: `/spec-write <tracker-id>` on an existing tracker-mode FR falls through to the resolver's `miss → importFromTracker` branch, and `importFromTracker` calls `writeFileSync(path, body)` unconditionally at `adapters/_shared/src/import.ts:74`. Local edits to the FR's body, ACs, milestone, or tracker block are silently overwritten by the freshly-imported tracker payload.

The function is structurally inconsistent with the canonical post-STE-76 frontmatter contract. Fix by making the resolver tracker-mode-aware: in tracker mode the tracker ID *is* the canonical identity (file path is `specs/frs/<tracker-id>.md` per `Provider.filenameFor(spec)`), so the lookup reduces to `existsSync`. No `id:`-line scan is needed.

Reproduced live against `specs/frs/archive/STE-134.md` and a synthetic post-STE-76 fixture in the audit.

## Technical Design

Two implementation paths; pick (B) for cheapest blast radius.

**Path A — fix the function in place.** Make `findFRByTrackerRef` tolerant of the post-STE-76 shape. Change the loop body (lines 161-175) to: parse frontmatter; if `tracker.<key> === <id>` matches, return the file path (not the `id:` field, which may be absent). Update the function's return type to `string | null` representing the file path, and change consumers in skill prose accordingly.

**Path B — skip the function in tracker mode (recommended).** Rewrite the resolver entry in `skills/spec-write/SKILL.md` § 0a, `skills/spec-archive/SKILL.md`, and `skills/implement/SKILL.md` to branch on mode at the top:
- Tracker mode: `existsSync(specs/frs/<tracker-id>.md)` decides hit/miss. The tracker ID is the canonical identity; no resolver helper needed.
- Mode-none: keep the existing `findFRByTrackerRef` logic (still relevant for ULID-by-tracker-binding lookup).

Either way, the test fixtures in `tests/resolver-single-pattern.test.ts:25-31` get rewritten to the post-STE-76 shape (drop `id:` lines on tracker-mode rows).

Documentation update: `adapters/_shared/src/resolve.ts` JSDoc — note the function's residual scope (mode-none-only) and reference STE-135.

## Notes

The audit's option (c) ("rewrite skill prose to skip the resolver in tracker mode") is the cheapest and most surgical fix — it converts an inconsistency between code and probe into a deliberate non-use of the function. Path A is more thorough but rewrites a function whose mode-none semantics are still useful. Path B is the preferred default; pick A only if the implementer finds a consumer that genuinely needs the unified function in both modes.

The reason no test caught this is dual: (a) `tests/resolver-single-pattern.test.ts:25-31` writes `id:` lines into every tracker fixture (pre-STE-76 shape preserved), so the function passes tests but fails on real STE-76-shaped FRs; (b) smoke-test runs only create new FRs, never exercise the existing-FR edit path. AC-STE-135.4 closes (a); AC-STE-135.7 closes (a) more pointedly with a new test that exercises the post-STE-76 contract.
