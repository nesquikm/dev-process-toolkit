## Requirement

When `/spec-write` creates a tracker-bound FR, the FR's milestone (`M<N>`) is recorded in the markdown frontmatter and the plan-file heading. The Linear ticket itself, however, has no `projectMilestone` attached — the binding lives only in the local files. Existing FRs created via the recently-adopted manual flow (STE-103/M27) carry `projectMilestone: "M27 — Dart/Python docs parity"` because an operator set it by hand; STE-115/STE-116 (created during M30 spec-write 2026-04-27) do not, because no automation populated it.

Same root cause as STE-117's project-binding bug: the workflow leaves a discretionary check to the LLM. The fix is to plumb milestone attachment through the same canonical Provider operation that creates and updates tickets, so the local plan-file heading and the Linear milestone object stay in sync by construction.

## Acceptance Criteria

- AC-STE-118.1: **Tracker mode only.** This FR is vacuous on `mode: none` — no tracker, no Linear API calls. All ACs below are gated on tracker mode active. {#AC-STE-118.1}
- AC-STE-118.2: **Canonical milestone name format.** A milestone object's name in Linear MUST equal the FR plan file's H1 heading text after stripping the leading `# ` and any trailing anchor `{#M<N>}`. Example: plan-file heading `# M30 — Stale doc references {#M30}` → canonical milestone name `M30 — Stale doc references`. Used for both lookup and creation. Em-dashes and other UTF-8 chars passed opaque (per STE-117 Notes). {#AC-STE-118.2}
- AC-STE-118.3: **`attachProjectMilestone` shared helper.** New helper at `adapters/_shared/src/attach_project_milestone.ts` exposes `attachProjectMilestone(provider, project, milestoneName, ticketId): Promise<void>`. Implementation:
  1. Call `mcp__linear__list_milestones(project)` (cached per-invocation).
  2. If a milestone whose `name` byte-equals `milestoneName` exists, capture its name.
  3. Else call `mcp__linear__save_milestone(project, name: milestoneName)` to create. Capture the returned name.
  4. Call `mcp__linear__save_issue(id: ticketId, milestone: <name>)` to attach.
  5. Verify by `mcp__linear__get_issue(ticketId)` — assert `projectMilestone.name === milestoneName`. If mismatch, throw `MilestoneAttachmentError` (NFR-10 canonical shape) with both names rendered.
  Idempotent: calling on an already-attached ticket re-runs steps 1+4 without side effects (Linear `save_issue` with the same milestone is a no-op when the binding matches). {#AC-STE-118.3}
- AC-STE-118.4: **`/spec-write` invokes the helper.** After `Provider.sync(spec)` returns the tracker ID for a freshly-created FR, `/spec-write` reads the FR's `milestone:` frontmatter and the matching `specs/plan/M<N>.md` heading, builds the canonical name (AC-STE-118.2), and calls `attachProjectMilestone(...)`. Failures surface per NFR-10 — the FR file is not rolled back (the tracker create already succeeded; the spec is the source of truth) but the user sees the diagnostic. {#AC-STE-118.4}
- AC-STE-118.5: **`/implement` invokes the helper at Phase 1 entry.** Idempotent invocation guarantees that an FR created before M31 (e.g., STE-103) gains milestone attachment the next time `/implement` runs against it. Vacuous on archived FRs (`status: archived`) — Phase 1 entry runs on active FRs only. {#AC-STE-118.5}

```
└── tracker_project_milestone_attached.test.ts
```

**Helper signature:**

```typescript
export class MilestoneAttachmentError extends Error {
  constructor(public readonly expected: string, public readonly actual: string | null) {
    super(`milestone attachment mismatch — expected "${expected}", got ${actual ? `"${actual}"` : "null"}`);
  }
}

export async function attachProjectMilestone(
  provider: TrackerProvider,
  project: string,
  milestoneName: string,
  ticketId: string,
): Promise<void> {
  const existing = await provider.listMilestones(project);
  const found = existing.find((m) => m.name === milestoneName);
  if (!found) {
    await provider.saveMilestone(project, { name: milestoneName });
  }
  await provider.upsertTicketMetadata(ticketId, { milestone: milestoneName });
  const fresh = await provider.getIssue(ticketId);
  if (fresh.projectMilestone?.name !== milestoneName) {
    throw new MilestoneAttachmentError(milestoneName, fresh.projectMilestone?.name ?? null);
  }
}
```

(Note: `provider.listMilestones` / `saveMilestone` / `getIssue` are new TrackerProvider methods that wrap the corresponding Linear MCP tools; STE-117 already widens the surface with `team`/`project` carry-through, so this builds on the same provider extension.)

**Canonical-name builder:**

```typescript
export function planFileHeadingToMilestoneName(planFilePath: string): string {
  const md = readFileSync(planFilePath, "utf-8");
  const m = md.match(/^# (M\d+ — .+?)(?:\s*\{#M\d+\})?$/m);
  if (!m) throw new Error(`plan file ${planFilePath} has no recognizable H1 heading`);
  return m[1].trim();
}
```

## Notes

**Why verify via `get_issue` after `save_issue`:** mirrors the FR-67 silent-no-op trap pattern. `mcp__linear__save_issue` accepts unknown keys without erroring; if Linear ever renames the param from `milestone` to something else, the `save_issue` call returns success but the binding doesn't take. The post-call `get_issue` round-trip is the only way to detect it.

**Why match by name not by ID:** the local source of truth is the plan-file heading text. Linear assigns milestone IDs at creation; storing them locally introduces a reverse dependency. The name is the canonical join key. Trade-off: if two milestones share a name (operator error), the helper attaches to the first match. Acceptable — the user's plan-file heading is the deduplication boundary.

**Backfill scope:** M30 only. Older milestones (M27, M28, M29, etc.) are archived; their FRs are immutable. A separate FR could backfill the archive if ever needed, but that violates AC-STE-18.4 and is explicitly out of scope here.

**Rename-on-mismatch is operator-triggered:** the `--rename-milestone` flag is opt-in. Auto-renaming on every probe failure would mask scenarios where the operator renamed the Linear side intending to retire the old name and reuse the M-number for something else. Surface the diff, let the operator decide.
