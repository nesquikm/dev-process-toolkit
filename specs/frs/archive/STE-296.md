# STE-296: Spec-review audit step in /tdd pipeline {#STE-296}

## Requirement

`/tdd` (the multi-agent orchestrator from STE-225) ships RED → GREEN → REFACTOR for one FR via three forked subagents but has no independent verification that the resulting code actually traces back to every AC. The `tdd-result` hand-off block validates each subagent's *claims* — test-writer says "RED for AC-X.Y at file:line", implementer says "GREEN for AC-X.Y at file:line" — but a subagent that silently drops an AC, or claims coverage at a file:line that does not in fact implement the AC's requirement, exits the cycle clean. `/spec-review` already exists as a read-only audit primitive (Read/Grep/Glob), but it is a standalone, user-invocable skill — `/tdd` does not consume it.

Add a fourth child stage to the `/tdd` orchestrator: a forked, read-only `tdd-spec-reviewer` subagent that runs **once at end of FR after REFACTOR succeeds GREEN**, independently traces every AC to its implementation + test, and emits a structured `tdd-spec-review-result` fenced block. The orchestrator blocks the pipeline only on `missing_acs` (binary contract violations), attempts a single bounded auto-retry through `tdd-write-test` + `tdd-implement` scoped to the missing ACs, and halts non-zero with a new `spec-gap` failure mode if the second audit still finds missing ACs. Advisory findings (`partial_acs`, `drift_count`, `cross_cutting_drift`) ride along in the report but do not halt — refactor-incidental drift is normal and should not trip the cycle.

This closes the second-line-of-defense gap STE-225 deliberately left open: `tdd-result` validates subagent claims; this FR validates reality.

## Acceptance Criteria

- AC-STE-296.1: A new subagent `plugins/dev-process-toolkit/agents/tdd-spec-reviewer.md` exists with frontmatter `tools: Read, Grep, Glob` (no Write/Edit/Bash/Agent — read-only), `maxTurns: 8`, `model: sonnet`, and a `description:` line naming exclusive invocation by `/dev-process-toolkit:tdd` via `context: fork`. The subagent body explains: read the FR file, trace every AC to `file:line` + `test-file:line`, classify ✓ Done / ✗ Missing / ⚠ Partial, end with exactly one `tdd-spec-review-result` fenced block.
- AC-STE-296.2: A new child skill `plugins/dev-process-toolkit/skills/tdd-spec-review/SKILL.md` exists with `user-invocable: false`, `context: fork`, `agent: tdd-spec-reviewer`, and no `Agent` tool in `allowed-tools`. Mirrors the existing 3 TDD child skills' frontmatter shape (`tdd-write-test`, `tdd-implement`, `tdd-refactor`).
- AC-STE-296.3: A new deterministic parser `parseTddSpecReviewBlock(...)` in `plugins/dev-process-toolkit/adapters/_shared/src/tdd_spec_review_result.ts` extracts and validates the fenced block shape: required fields `role: spec-reviewer`, `status: ok | failed`, `missing_acs: string[]`, `partial_acs: string[]`, `drift_count: number`, `advisory_findings: string[]`, `cross_cutting_drift: string[]`, `command: string`, `output_excerpt: string`; optional `notes`. Missing/extra-fenced/wrong-role/wrong-type → format violation with a reason naming the offending field.
- AC-STE-296.4: `plugins/dev-process-toolkit/skills/tdd/SKILL.md` orchestrator state machine is extended so that after `tdd-refactor` returns `status: ok` and the refactor command shows GREEN, the orchestrator forks `tdd-spec-review` once with `{ FR file path, AC list, project test command }`, parses the returned block via AC.3's helper, and branches on `missing_acs.length`.
- AC-STE-296.5: When `missing_acs.length === 0`, the cycle exits GREEN regardless of `drift_count`, `partial_acs`, or `cross_cutting_drift` values. Those advisory fields are rendered in the final /tdd report (and/or surfaced via capability keys per AC.9) but do not halt or retry. The audit is not "all green" — it is "no missing ACs".
- AC-STE-296.6: When `missing_acs.length > 0`, the orchestrator re-forks `tdd-write-test` scoped to the missing ACs only (passing the subset list, not the full FR AC list), then `tdd-implement` once per missing AC, then re-forks `tdd-spec-review` exactly one more time. This audit-retry budget is tracked independently from the existing per-AC test-writer/implementer budgets (recorded via a new key in `tdd_retry_state.ts` — `audit-round`, max 1).
- AC-STE-296.7: When the second audit still shows `missing_acs.length > 0`, the orchestrator exits non-zero with a new failure mode `spec-gap` extending the 5-mode taxonomy from STE-225 AC.5. `formatHaltReport(...)` in `tdd_halt_report.ts` accepts `{ mode: "spec-gap", role: "spec-reviewer", missingAcs: string[], lastBlock: TddSpecReviewBlock }` and renders a report naming the unresolved missing ACs + their last classified trace state.
- AC-STE-296.8: A new `/gate-check` probe `tdd_spec_reviewer_subagent_invariants` (severity: error) asserts the AC.1 + AC.2 frontmatter invariants byte-checkably against the new files, mirroring the existing STE-225 AC.7 probe shape for the other three TDD subagents/skills.
- AC-STE-296.9: The orchestrator (or `/implement` parent, whichever consumes the result) emits one of three capability-key rows in its closing summary per audit outcome — `tdd_spec_audit_passed` (first audit clean), `tdd_spec_audit_missing_recovered` (first audit found Missing, retry round resolved), `tdd_spec_audit_halted` (second audit still Missing). Plain-language map entries added to `skills/spec-write/SKILL.md` § 7 static capability map and (where applicable) `skills/implement/SKILL.md`.
- AC-STE-296.10: A doc-conformance test `tests/tdd-spec-review-orchestrator.test.ts` exercises the orchestrator state machine end-to-end with fixture child outputs: (a) clean first audit, (b) Missing→retry→clean, (c) Missing→retry→still-Missing→halt, (d) format violation of the spec-review block → single retry → halt. Each path asserts the correct capability key emission + correct halt mode.

## Technical Design

### Architecture

New 4th stage in the /tdd orchestrator:

| Stage      | Child skill         | Subagent              | Cycle granularity                       |
|------------|---------------------|-----------------------|------------------------------------------|
| RED        | `tdd-write-test`    | `tdd-test-writer`     | once per FR (batched)                    |
| GREEN      | `tdd-implement`     | `tdd-implementer`     | once per AC                              |
| REFACTOR   | `tdd-refactor`      | `tdd-refactorer`      | exactly once at end                      |
| **AUDIT**  | `tdd-spec-review`   | `tdd-spec-reviewer`   | **exactly once at end, post-REFACTOR**   |

State machine extension:

```
RED → GREEN(per-AC) → REFACTOR → AUDIT(1st) → branch:
  missing_acs == [] → exit ok
  missing_acs != [] → RETRY(test-writer scoped, implementer per-AC) → AUDIT(2nd) → branch:
    missing_acs == [] → exit ok (capability: tdd_spec_audit_missing_recovered)
    missing_acs != [] → halt(spec-gap)
```

### Hand-off contract

New fenced block at end of subagent turn:

```tdd-spec-review-result
role: spec-reviewer
status: ok
missing_acs: []
partial_acs: []
drift_count: 0
advisory_findings: []
cross_cutting_drift: []
command: <project test command>
output_excerpt: |
  <first 40 lines of the runner output OR a 40-line trace summary>
notes: optional one-liner
```

Each entry in `missing_acs` / `partial_acs` is an AC prefix string (e.g., `AC-STE-296.3`). Each entry in `advisory_findings` is shaped `file:line — note`. Each entry in `cross_cutting_drift` names a cross-cutting spec file with a one-line drift description. The parser is a sibling of `tdd_result.ts` — same fence-extraction logic, different field set (per AC.3).

### Subagent body — audit procedure

The subagent body (drafted in AC.1) instructs the fork to:

1. Read the FR file at the passed path, extract every `AC-<PREFIX>.<N>` line under `## Acceptance Criteria`.
2. For each AC, grep the codebase for the AC prefix in test files and trace from the test to the source — `tests/<feature>.test.ts:LINE` references `src/<file>.ts:LINE`.
3. Classify each AC:
   - **✓ Done** — both test + source line found, source content matches the AC requirement.
   - **⚠ Partial** — test exists but source line is empty / asserts on the wrong invariant; or source exists but no test asserts on it.
   - **✗ Missing** — no test found OR no source line found.
4. Count "potential drift" — code in files touched by /tdd's implementer/refactorer that does not trace to any AC (excluding standard project boilerplate: imports, type re-exports, helper extractions clearly named after a refactor target).
5. Cross-check live spec files (`specs/requirements.md`, `specs/technical-spec.md`, `specs/testing-spec.md`) for stale references to the FR ID, ACs, or files modified.
6. Run the project test command once to verify GREEN state has not regressed during the audit's read scan (defensive — the audit is read-only but the command result is the most trustworthy "what does GREEN actually mean" signal).
7. Emit the single `tdd-spec-review-result` fenced block.

### Retry isolation

The retry path follows STE-225's load-bearing isolation rule: the audit subagent does not see the implementer's plan or the test-writer's tests — only the FR text and the codebase. On retry, the orchestrator passes the scoped missing-AC list to `tdd-write-test` (which the test-writer treats as the same batched-AC input as a normal first run, just with a smaller list) and per-AC to `tdd-implement`. **No orchestrator-side analysis is injected** into the scoped retry prompts; this is the same rule as STE-225's semantic retry — only raw failing-test output, or in the audit case, raw "AC-X.Y missing from spec-review trace" lines from the previous audit block.

### Failure modes (extends STE-225 § Failure modes)

- **(F) spec-gap** — second audit still shows `missing_acs.length > 0`. New mode; counts as 1 halt event after the bounded retry round. Independent budget from modes A–E (`audit-round`, max 1). Does not consume the per-AC test-writer/implementer budgets.

Format violation of the spec-review block itself (no fence, multiple fences, missing field, wrong role) is **mode D** per STE-225 — same single targeted retry path, no semantic re-prompt.

## Testing

Framework: `bun test`. Test files mirror existing patterns in `plugins/dev-process-toolkit/tests/tdd-*.test.ts`.

**Parser tests** (`tests/tdd-spec-review-result.test.ts`):

- Valid block with `status: ok`, empty `missing_acs` → parses cleanly.
- Valid block with `missing_acs: ["AC-STE-296.2"]` → parses cleanly; `missing_acs.length === 1`.
- Block missing `missing_acs` field → format violation, reason names `missing_acs`.
- Block with `role: implementer` → format violation, reason names "wrong role".
- Two fenced blocks in stdout → format violation (exactly-one rule).
- `drift_count` non-numeric → format violation.
- `partial_acs` non-array → format violation.

**Orchestrator integration tests** (`tests/tdd-spec-review-orchestrator.test.ts`):

- Path A — clean first audit (`missing_acs: []`) → orchestrator exits `ok`, emits `tdd_spec_audit_passed`.
- Path B — first audit `missing_acs: ["AC-X.1"]`, retry round resolves, second audit clean → orchestrator exits `ok`, emits `tdd_spec_audit_missing_recovered`.
- Path C — first audit Missing, retry round resolves test-writer + implementer for the scoped AC subset only (not the full AC list).
- Path D — first audit Missing, retry round runs, second audit still Missing → halt with mode `spec-gap`, exit non-zero, emit `tdd_spec_audit_halted`, halt report names the unresolved ACs.
- Path E — first audit block is a format violation → single targeted retry → second emission valid → proceed normally (mode D budget consumed, not audit-round).
- Path F — both first and retry-emit audit blocks are format violations → halt with mode D after 1 retry per STE-225 contract.

**Gate-check probe test** (`tests/gate-check-tdd-spec-reviewer-invariants.test.ts`):

- Fixture: conforming `agents/tdd-spec-reviewer.md` + `skills/tdd-spec-review/SKILL.md` → probe passes.
- Fixture: subagent frontmatter missing `tools: Read, Grep, Glob` → probe fails, names the violation.
- Fixture: child skill frontmatter missing `context: fork` → probe fails.
- Fixture: child skill includes `Agent` in allowed-tools → probe fails.

**Out of test scope (manual smoke / deferred):**

- End-to-end /tdd invocation against a real FR in a fixture repo — covered by existing `/tdd` smoke tests in M77's manual conformance pass (template: see prior milestone smoke loops).
- /implement integration — `tdd_spec_audit_*` capability propagation through /implement's closing summary. Covered by /implement's existing capability-key test fixtures with three new fixture rows.

## Notes

**Design heritage.**

- STE-225 (M52/archived) — multi-agent /tdd orchestrator fork pattern. This FR is a strict extension: the 4th child slots into the same `context: fork` + `agent:` + `user-invocable: false` shape; the `tdd-spec-review-result` block clones the `tdd-result` hand-off contract.
- STE-230 (archived) — read-only forked-subagent + fenced-result-block precedent (`spec-research`). The new audit subagent borrows the same allowed-tools shape (`Read, Grep, Glob`, no Bash for write paths).
- STE-172 (archived) — `/spec-review` drift-count >= 2 threshold for the live-spec refresh hint. The new audit reuses the *advisory* drift signal (does not block on it) — the threshold rationale is preserved.

**What this FR explicitly does NOT change:**

- The standalone `/spec-review` skill remains unchanged and user-invocable for wider audits (cross-cutting sweeps, post-milestone reconciliation, cold audits). The new audit is a /tdd-internal step; the two coexist.
- The existing `tdd-result` block parser is untouched; the new parser is a sibling, not a replacement.
- The per-AC retry budgets for test-writer/implementer stay at max 2 attempts per role per AC (STE-225 contract).

**Open issues for /implement to decide:**

- Whether the subagent body should grep for the AC prefix verbatim in test files (current draft assumption), or also tolerate `AC-<TKR>-NN.N` substring matches within longer assertion strings. Probable answer: verbatim only — partial matches invite false positives. Will be settled when tests/<feature>.test.ts conventions are exercised in implementation.
- Whether `cross_cutting_drift` should fire the existing `Live-spec refresh suggested — N drift(s) found …` literal hint (from `adapters/_shared/src/spec_review_drift_hint.ts`) inside the /tdd report, or suppress it (audit's audience is the operator running /tdd, not /spec-write). Current design suppresses; revisit if /tdd output feels too quiet.
