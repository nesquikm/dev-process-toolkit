# STE-366: zsh-safe glob fences across smoke-test + conformance-loop {#STE-366}

## Requirement

Every glob-based fence in `/smoke-test` and `/conformance-loop` must be safe under **zsh's default `nomatch`** option: an unmatched glob must expand to nothing (or be handled), never abort the whole command. The driver shell on the maintainer's machine is zsh (confirmed 2026-07-04: `(eval):N: no matches found` fired 3× during the run, including inside `/conformance-loop`'s own final-message self-check). The fences were authored assuming bash pass-through-glob semantics.

The load-bearing case is the `/smoke-test` **Phase 0.5 verified-wipe** fence (finding F2): `rm -f /tmp/dpt-smoke-prompt-*.txt …` and its verify `ls …` both assume bash pass-through. Under zsh, the first unmatched glob (legitimately absent on a clean machine, e.g. the post-STE-185 `dpt-smoke-prompt-*.txt` no-op glob) raises `no matches found` and **aborts the command before `rm` runs** — nothing is wiped — and the identical failure aborts the verify `ls`, so the survivors variable comes back empty and the check reports a **false PASS**. That is exactly the "self-reported cleared without on-disk assertion" mode STE-358 was written to prevent, here re-introduced by the shell rather than the model.

## Acceptance Criteria

- AC-STE-366.1: **smoke-test fences zsh-safe.** Every glob-based `rm` / `ls` / `for … in <glob>` fence in `.claude/skills/smoke-test/SKILL.md` (the Phase 0.5 wipe **and** its verify, the final-message pidfile self-check `for` loop, and any other glob fence) is made zsh-safe by the fix matched to its shape (see Technical Design): `for`-loop fences via a `null_glob`/`nullglob` guard; `rm`/`ls` fences that must preserve bash pass-through via `bash -c` or a `find`-based rewrite. {#AC-STE-366.1}
- AC-STE-366.2: **conformance-loop fences zsh-safe.** Same treatment applied to every glob fence in `.claude/skills/conformance-loop/SKILL.md` — the final-message pidfile self-check (`for PIDFILE in /tmp/dpt-conformance-loop-*.pid`), the orphan-adoption grandchild scan, the poll-loop pidfile globs, and the teardown/tidy globs. {#AC-STE-366.2}
- AC-STE-366.3: **Phase 0.5 verify cannot false-PASS under zsh.** The Phase 0.5 verified-wipe survivor check both executes the `rm` and produces a truthful survivor enumeration under zsh when its globs match nothing — it never aborts, and it never degrades to `ls` with zero args listing the CWD (the `nullglob`-on-`ls` trap). The pass condition (zero survivors on disk) is asserted by a shell-agnostic enumeration. {#AC-STE-366.3}
- AC-STE-366.4: **Drift meta-test.** A deterministic meta-test in the DPT `bun test` suite asserts the Phase 0.5 wipe/verify fence and the pidfile-liveness `for` fences (both skills) carry an explicit zsh-safe guard token (`null_glob` / `nullglob` / `bash -c` / `find`), and that no raw unguarded `ls /tmp/dpt-*…*` verify or `for … in /tmp/dpt-*…*` loop without a guard remains in either SKILL.md. {#AC-STE-366.4}

## Technical Design

**Per-fence fix by shape** (a single blanket `nullglob` is wrong — it breaks the verify `ls`, whose zero-arg form lists the CWD):

- **`for X in <glob>` loops** (final-message self-checks in both skills, orphan-adoption scan): prepend a portable guard so an unmatched glob yields zero iterations —
  `setopt local_options null_glob 2>/dev/null || shopt -s nullglob 2>/dev/null || true`
  The existing `[ -e "$X" ] || continue` guard already covers the bash pass-through case, so this is additive and safe in both shells.
- **`rm -f <globs>`** (Phase 0.5 wipe): wrap in `bash -c 'rm -f <globs>'` to restore bash pass-through semantics (`rm -f` already ignores non-existent paths), or rewrite as `find /tmp -maxdepth 1 \( -name 'dpt-smoke-…' -o … \) -delete`. `bash -c` is the minimal change.
- **`ls <globs>` verify** (Phase 0.5 survivor check): **do not** use `nullglob` here — zero args makes `ls` print the CWD. Wrap in **`bash -c 'ls <globs> 2>/dev/null'`**: under bash's pass-through-glob an unmatched glob passes to `ls` literally, `ls` errors to `/dev/null`, output is empty ⇒ zero survivors ⇒ pass — never aborts, never lists the CWD. **`bash -c`, not a `find` rewrite** (spec refinement discovered at implement time): the STE-358 verified-wipe meta-test (`smoke-driver-m96-verified-wipe-freshness.test.ts:177`) requires an `ls` executable line *after* the `rm` line *in the same fence*, so the `ls` shape + the `rm`-then-`ls` ordering must be preserved; a `find` rewrite would break that existing test (which `/implement` forbids editing to pass). `bash -c` keeps the `rm -f <globs>` + `ls <globs>` fence intact (STE-358 lines 132/161/177 stay green) while making both zsh-safe.
