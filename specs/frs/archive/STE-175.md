# STE-175: /smoke-test driver hardening — Phase 0.5 scratch reset + pre-flight #5 team-by-key + --reset flag {#STE-175}

## Requirement

Three driver-side fixes in `.claude/skills/smoke-test/SKILL.md` (project-local skill, not the plugin) — both smoke #6 (Jira) and smoke #7 (Linear) hit the same stale-scratch / team-resolve / no-reset friction and called out the same fixes:

1. **Phase 0.5 scratch-reset.** Stale `/tmp/dpt-smoke-prompt-*.txt` files cause Write-tool errors on repeat runs (the existing-file guard fires). When the driver works around this by `cat`-reading and falling through, stale prompt content gets reused — smoke #6 F1 root cause was a phantom "Linear bits" args contradiction that traced back to a 2026-04-27 Linear-flavored prompt stub still present at `/tmp/dpt-smoke-prompt-spec-write.txt`. Smoke #7 F2 reproduced this exact failure mode; smoke #7 F4 noted the same shape on the approval file.
2. **Pre-flight #5 team-by-key.** `mcp__linear__list_teams query=STE` returns name-prefix matches only; `STE` is the team key (`team.key == "STE"`), display name is `Stellarlab Mike's Sandbox`. Pre-flight refused at `STE` and forced operator to `--linear-team` override (an identity override since the override resolves to the same team). First-time-setup confusion + a re-run cost. Smoke #7 F1.
3. **`--reset` flag.** Pre-flight #2 refuses on existing `../dpt-test-project-<tracker>` with no escape hatch — operator must manually `rm -rf` after aborted runs and re-invoke. Recovery friction is small but consistent. Smoke #7 F5.

The three changes share the same SKILL.md file edit surface; bundling avoids merge conflicts with STE-176 (the second smoke-test FR).

Out of scope:
- The `--prune-old-findings` flag (smoke #7 F3) — explicitly deferred from M49 per the plan.
- Re-architecting the per-run scratch directory layout — keeps the existing `/tmp/dpt-smoke-*` flat layout, just adds rotation.

The name-prefix `query=$TEAM_KEY` path remains as a fallback only after the key probe misses. `STE` (team key) resolves cleanly without `--linear-team` override; bogus key fails with NFR-10 canonical refusal naming the unknown key + the supported keys.

## Technical Design

Three discrete edits in `.claude/skills/smoke-test/SKILL.md`:

1. **Phase 0.5 block.** After "Phase 0 — Operator approval" and before "Phase 1.1 — Test project dir create" (or whatever the first Phase 1 step is), insert:

   ```markdown
   ### Phase 0.5 — Clear stale per-run scratch

   Unconditionally clear stale prompt + log files from prior runs:

   ```
   rm -f /tmp/dpt-smoke-prompt-*.txt /tmp/dpt-smoke-<tracker>-*.log
   ```

   Do NOT clear `/tmp/dpt-smoke-findings-*.md` or `/tmp/dpt-smoke-<date>-<tracker>-approval.txt` — these are audit-trail artifacts and are intentionally retained across runs.
   ```

2. **Pre-flight #5 patch.** Locate the existing pre-flight #5 prose and replace the team-resolve probe. Reference shell snippet (already in smoke #7 F1's suggested fix):

   ```bash
   TEAM_KEY="${LINEAR_TEAM:-STE}"
   mcp__linear__list_teams limit=250 \
     | jq -e --arg k "$TEAM_KEY" '.teams[] | select(.key == $k)' >/dev/null \
     || mcp__linear__list_teams query="$TEAM_KEY" \
     | jq -e --arg k "$TEAM_KEY" '.teams[] | select(.name | test($k; "i"))' >/dev/null
   ```

3. **`--reset` flag.** Add to the argument-parsing section a new flag `--reset` (boolean, default false). Add the Phase 0 contract line `RESET: existing ../dpt-test-project-<tracker> will be deleted before run.` when the flag is present. Modify pre-flight #2's existing refusal: when `--reset` is true and the dir exists, `rm -rf` instead of refusing.

No new code, no new test files (the driver skill is itself the test). Manual exercise via the next smoke run is the live-verification gate.
