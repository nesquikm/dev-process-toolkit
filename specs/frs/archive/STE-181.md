# STE-181: /implement &lt;FR-id&gt; vs M&lt;N&gt; Phase-5 divergence — clarify or normalize {#STE-181}

## Requirement

`/implement` accepts two argument shapes:

- `/implement <FR-id>` — runs Phases 0–4 against a single FR; **silent-skips Phase 5** (no milestone-close prompt, no archival sweep). The skill output explicitly logs: *"Phase 5: single-FR invocation (`<FR-id>` is a tracker ID, not `M<N>`) → silent skip."*
- `/implement M<N>` — runs Phases 0–4 against every active FR in milestone N, then runs Phase 5 (milestone-close prompt, archival).

The two forms produce different end states: FR-id form leaves the FR `status: active` (canonical single-FR pattern per STE-83) and the milestone `status: active`; M<N> form archives both on green gate. STE-83 (archived) established that single-FR runs legitimately leave the FR active.

The problem is downstream messaging diverges:

1. **Smoke-test driver SKILL.md** prescribes `/dev-process-toolkit:implement <feature-id>` as the canonical smoke-chain step. Operators running the smoke driver hit the silent-skip foot-gun (probe #14 fires per STE-180) without being warned.
2. **`/spec-write`'s closing summary** writes `Next: Run /dev-process-toolkit:implement M<N>` (the milestone form) as its standard recommendation. This contradicts the smoke driver's prescription.
3. **`/implement`'s own SKILL.md** does not surface the divergence anywhere — both forms are documented as valid but the per-phase semantic differences are scattered across Phase 0, Phase 5, and the silent-skip log line.

The result: an operator reading the docs cannot easily tell which form to use. The smoke driver picks `<FR-id>`; `/spec-write` picks `M<N>`; `/implement` is silent. STE-180 is the gate-check-side fix for the foot-gun; this FR (STE-181) is the documentation-side clarification.

The fix is documentation-only: add an "Invocation forms" section to `/implement` SKILL.md with a side-by-side comparison table; reconcile `/spec-write`'s closing summary and the smoke driver's SKILL.md to agree on which form to recommend in which scenario.

Out of scope:
- Normalizing the two forms (e.g., having `/implement <FR-id>` auto-detect last-task-completed and run Phase 5). Out-of-scope because: (a) auto-Phase-5 from a single-FR invocation surprises operators mid-multi-FR-milestone (see STE-180 § Notes), (b) the documented divergence is by design per STE-83, (c) STE-180's advisory note already covers the user-facing nudge for the canonical "milestone ready to close" state.
- Changing the smoke driver's prescription. The smoke driver's choice (`<feature-id>` form) is correct for what it tests — single-FR ship-and-archive-later is the canonical user pattern. Only the messaging clarifies.

### Comparison table shape

The "Invocation forms" table has 7 rows (header + 6 phase rows):

| Phase | `/implement <FR-id>` | `/implement M<N>` |
|-------|----------------------|---------------------|
| 0 (Pre-flight) | runs against single FR's binding | runs against every active FR in M<N> |
| 1 (Tracker claim) | claims the FR's ticket | claims every active FR's ticket |
| 2 (Plan analysis) | plans tasks for the FR | plans tasks for every FR in M<N>, ordered by deps |
| 3 (TDD loop) | builds the FR | builds every FR in M<N> |
| 4 (Commit + close) | one feature commit + tracker → Done | one feature commit per FR + each tracker → Done |
| 5 (Milestone close) | **silent-skip** | runs milestone-close prompt + archival |

The table is the load-bearing addition; the surrounding prose names canonical use cases (per AC-181.2) but the table is what operators grep for.

### Probe semantics

`implement-invocation-grammar-doc` is a documentation-existence probe: it parses the SKILL.md, locates the heading, counts table rows, and verifies the Phase-5 row contains the expected divergence keywords. It does not validate semantic correctness (that's the integration test's job); it only verifies the section exists with the right structural shape. This makes it cheap (~1ms) and noise-free.

## Notes

- **STE-83 precedent.** STE-83 (archived) established that single-FR `/implement` runs legitimately leave the FR `status: active`. This FR documents that contract user-facing; STE-180 enforces it at the gate-check seam.
- **Why not normalize the two forms.** The two forms are intentionally divergent because they serve different operator goals (ship one FR vs close a milestone). Normalizing would be a semantic regression — operators running per-FR shipping mid-milestone would be surprised by milestone-close prompts. The right fix is clarity, not normalization.
- **Smoke-test source.** `/tmp/dpt-smoke-findings-2026-05-01-jira.md` § F3 logged this. The Linear smoke on the same date hit the same divergence (smoke driver used `<FR-id>` form, `/spec-write` recommended `M<N>`); the Jira finding articulates the foot-gun shape best.
- **Interaction with STE-180.** STE-181 documents the divergence; STE-180 enforces "this divergence is not drift." STE-181 should land first because STE-180's advisory note ("`M<N>` plan fully checked but not archived...") is most useful when the operator already knows from `/implement` SKILL.md that single-FR runs do not archive. Land STE-181 → STE-180 → smoke chain re-run.
- **Interaction with STE-179.** STE-179 changes `/spec-write`'s commit semantics; STE-181 changes its closing-summary `Next:` line. The two edits touch different parts of `skills/spec-write/SKILL.md` (Step 7a vs Step 7 final paragraph) and do not conflict. Either order works.
