## Requirement

M20 introduces docs generation (`/docs`, `/ship-milestone`), which requires per-project configuration that neither the tracker-mode `## Task Tracking` section nor the bare defaults can express: **which docs modes are enabled** (user-facing, packages, or both) and **whether `CHANGELOG.md` is owned by CI** (in which case `/ship-milestone` must not write it).

`/setup` is the canonical entry point for project configuration. STE-68 extends `/setup` with three new yes/no prompts asked after the existing tracker-mode block, persists the answers to a new `## Docs` section in `CLAUDE.md` (Schema L-style key:value lines), and makes downstream skills (`/docs`, `/ship-milestone`, `/implement`) read that section via grep. Absent section ≡ all three values `false` — meaning docs generation is disabled, which is the backward-compatible default for existing projects upgrading to v1.23.0.

The prompts are deliberately binary (not mode strings or free-form) so the Schema L format stays grep-friendly and the downstream gate in `/docs` (AC-A.4: refuse if both modes are `false`) is trivial to check.

## Acceptance Criteria

- AC-STE-68.1: `/setup` asks three new yes/no prompts after the tracker-mode block, in this exact order: (1) "Generate user-facing docs (narrative + mermaid state/flow diagrams)?", (2) "Generate packages-style API reference docs?", (3) "Is CHANGELOG.md generated by CI (if yes, /ship-milestone will not write it)?". Each prompt accepts `y`/`n`/`yes`/`no` case-insensitively; other inputs re-prompt with the remedy "answer y or n".
- AC-STE-68.2: Answers persist to a new `## Docs` section in `CLAUDE.md` immediately after the `## Task Tracking` section (or at end of file if `## Task Tracking` is absent). Format is Schema L-style key:value lines, one per line:

  ```
  ## Docs

  user_facing_mode: <true|false>
  packages_mode: <true|false>
  changelog_ci_owned: <true|false>
  ```

  Values are lowercase literal `true` or `false` — no quoting, no `yes`/`no`, no blank lines between keys.
- AC-STE-68.3: Absence of the `## Docs` section ≡ all three values `false` (backward compatibility for projects upgrading from v1.22.0 or earlier). No migration prompt; `/docs` and `/ship-milestone` treat the absent section as "docs disabled".
- AC-STE-68.4: `/setup` refuses to proceed if the user answers `no` to both `user_facing_mode` and `packages_mode` — at least one docs mode must be active for the `## Docs` section to be written. The refusal uses NFR-10 canonical shape:

  ```
  /setup: at least one docs mode must be enabled to write the ## Docs section.
  Remedy: answer yes to either "user-facing docs?" or "packages API refs?", or decline both to skip docs configuration entirely (the ## Docs section will not be written and /docs will be a no-op).
  Context: mode=<tracker-mode>, skill=setup
  ```

  After surfacing, `/setup` re-asks the two docs-mode prompts (not all three — `changelog_ci_owned` is preserved).
- AC-STE-68.5: Re-running `/setup` on a project that already has a `## Docs` section shows the current values inline with each prompt (e.g., "`Generate user-facing docs? [current: true]`") and accepts empty input as "keep current". New answers overwrite the section atomically (single file write, not partial line edits).
- AC-STE-68.6: `templates/CLAUDE.md.template` gains a commented-out `## Docs` stub with the three keys and a one-line pointer: `<!-- /setup fills this section; absent section ≡ all three false -->`. Existing projects using the template continue to work unchanged.
- AC-STE-68.7: `adapters/_shared/src/task_tracking_config.ts` (or a new sibling `docs_config.ts`) exports `readDocsConfig(claudeMdPath): DocsConfig` where `DocsConfig = { userFacingMode: boolean; packagesMode: boolean; changelogCiOwned: boolean }`. Missing section returns all-`false`. Malformed values (neither `true` nor `false`) surface as NFR-10 `MalformedDocsConfigError`.

## Technical Design

**Module edits:**
- `plugins/dev-process-toolkit/skills/setup/SKILL.md` — add the three prompts block after the existing tracker-mode Step 7 (step numbering TBD at implementation — pick the next integer after the current last step).
- `plugins/dev-process-toolkit/templates/CLAUDE.md.template` — add commented stub.
- `plugins/dev-process-toolkit/adapters/_shared/src/docs_config.ts` (new) — reader module mirroring the shape of `task_tracking_config.ts` (grep-based parse).
- `plugins/dev-process-toolkit/adapters/_shared/src/docs_config.test.ts` (new) — unit tests for parse / missing section / malformed values.

**`readDocsConfig` contract:**

```typescript
export interface DocsConfig {
  userFacingMode: boolean;
  packagesMode: boolean;
  changelogCiOwned: boolean;
}

export function readDocsConfig(claudeMdPath: string): DocsConfig;
```

Absent `## Docs` section → `{ userFacingMode: false, packagesMode: false, changelogCiOwned: false }`. Malformed value (e.g., `user_facing_mode: maybe`) → `MalformedDocsConfigError` (NFR-10 shape).

**`/setup` flow:**

After the existing tracker-mode prompt block:

1. Read current `## Docs` section if it exists (for re-run pre-fills).
2. Ask prompt 1 (user-facing mode). Pre-fill shown if re-run.
3. Ask prompt 2 (packages mode). Pre-fill shown if re-run.
4. If both answers are `false`, surface AC-STE-68.4 refusal and re-ask 1+2.
5. Ask prompt 3 (CHANGELOG CI ownership). Pre-fill shown if re-run.
6. Write the `## Docs` section to `CLAUDE.md` (full-section rewrite, not line-edits).

## Testing

`docs_config.test.ts` — 6 cases: valid both-modes-true, valid user-facing-only, valid packages-only, missing section (defaults), malformed `user_facing_mode` value, malformed `changelog_ci_owned` value.

`skills/setup/` integration — extend the existing `tests/fixtures/setup_*` tree with a new fixture `setup_docs_*` covering: first-run both-true, first-run single-mode, re-run keep-current, re-run both-false-refusal path.

No new fixture for the CLAUDE.md template test — the existing template-round-trip test gains an assertion that the `## Docs` stub is present and commented.

## Notes

**Why a separate section, not keys inside `## Task Tracking`.** The tracker-mode config answers "where do tickets live?"; the docs config answers "what docs do we generate?". They're orthogonal and evolve independently. Keeping them in separate sections means `/gate-check` audits can validate each section's shape without cross-dependency, and future docs-config keys (e.g., `mermaid_theme`, `api_ref_strategy`) don't bloat the tracker section.

**Release target:** v1.23.0 (M20 codename TBD — "Diátaxis" is a candidate).

**Dependency:** This FR must land first in M20 — STE-69 through STE-75 all grep for the `## Docs` section and its keys. Phase A of the M20 plan (STE-68 + STE-69 only).

**Why no "skip" answer.** `/setup` is the config ceremony for the whole project. Allowing "skip" on individual prompts creates a third state ("unanswered") that downstream skills would need to handle. The three-binary design keeps the state space small and the grep contract clean.
