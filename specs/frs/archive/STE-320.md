# STE-320: Code-reviewer scope + capability-key registry expansion {#STE-320}

## Requirement

Close two related self-contradictions in the toolkit's own canon, both surfaced by the 2026-05-20 review.

**A7 — `agents/code-reviewer.md` self-contradicts on spec-compliance scope**: L10 says "Spec compliance is **not** your job — `/spec-review` owns AC→code traceability." L63 then explicitly makes Pass-1 deliver "one line per AC" spec-compliance. Two doc surfaces (`docs/adaptation-guide.md:113`, `docs/skill-anatomy.md:152`) echo the L10 disclaimer. Spec-research confirms the canonical contract: HG95TM AC-23.6 (M10, archived) documents `code-reviewer` Pass-1 (spec-compliance) + Pass-2 (code-quality) as **parallel-not-subordinate** to `/spec-review`. `/spec-review` (STE-308, M80) is the deep-traceability audit; `code-reviewer` Pass-1 is the inline per-PR check inside `/implement` Phase 3 Stage B. Both own spec-compliance in different contexts. The L10 disclaimer is the regression: it was added during STE-308 as an over-reaching cross-reference and now mis-teaches the agent's own scope.

**A10 — Capability-key registry under-covers 8 of 20 emitted tokens**: `adapters/_shared/src/closing_summary_capability_keys.ts:43-56` exports 12 canonical keys. `grep -nE 'MUST emit \\\`[a-z_]+\\\`' plugins/dev-process-toolkit/skills/spec-write/SKILL.md` finds exactly **20 distinct tokens** carrying canonical `MUST emit \`<key>\`` directives (M84 audit second/third-pass verification, 2026-05-20). The 8 missing tokens (each cited to its exact `MUST emit \`<key>\`` line):

- `spec_write_draft_declined` (spec-write SKILL.md L103)
- `spec_write_commit_declined` (L261)
- `branch_gate_created` (L191)
- `branch_gate_edited` (L192)
- `branch_gate_declined` (L193)
- `branch_gate_remote_probe_skipped` (L196)
- `tracker_local_orphan_local` (L327)
- `tracker_local_reconciled` (L327)

**OUT OF SCOPE for AC-3 (no real `MUST emit` directive — only table-header labels at L330):** `tracker_status_forced`, `tracker_status_skipped`, `tracker_status_cancelled`, `tracker_status_unknown_encountered`, `tracker_tolerance_refused_non_tty`. These appear ONLY as backticked column-name labels inside the static-map table cell at L330 and DO NOT match the probe's regex `MUST emit\s*\`<key>\``. Adding them to the const would FAIL the bidirectional probe (B − A orphan-registration leg). If genuine emission paths exist for these tokens, the canonical fix is a separate FR that adds explicit `MUST emit \`<key>\`` directives at their emission sites in spec-write SKILL.md; only then can the const grow.

**Edge case flagged (`milestone_local_orphan`):** the compound-form directive at L327 reads `MUST emit \`tracker_local_orphan_local\` / \`milestone_local_orphan\``. The probe's single-key regex matches only the head (`tracker_local_orphan_local`); the trailing key is textually a `MUST emit` token but unreachable by the current regex. Tracked separately as a spec-prose hygiene item (not in this FR's scope — a future probe-regex enhancement OR an L327 rewrite that repeats `MUST emit` per key).

Probe #44 `closing_summary_capability_keys` only validates the 12 declared keys, leaving a false-negative gap for the 8 unregistered tokens. STE-238's stated mission "byte-checkable structural verification is required" is half-fulfilled.

Bundled because both are "the toolkit's own canon contradicts itself" — same audit rubric (pick the canonical winner, drop the conflicting surface).

## Technical Design

**A7 fix (3 edits)**: targeted prose replacements at the three drift sites. The new wording uses the architectural twin model verified by spec-research:

> `/spec-review` runs the deep-traceability audit (forked-auditor pattern per STE-308); `code-reviewer` Pass-1 is the inline per-PR check inside `/implement` Phase 3 Stage B (parallel pass pattern per HG95TM). Both own spec-compliance in different contexts.

**A10 fix (1 .ts file edit + probe expansion + test)**: `closing_summary_capability_keys.ts` const list grows from 12 to exactly 20 keys (the 8 directive-backed additions enumerated in AC-3). Probe shape stays unchanged; the registry grows to mirror Set A.

Probe-bidirectionality: probe #44 today walks the registry and asserts each key appears in a `MUST emit` directive. The expanded version adds the reverse walk — every `MUST emit \`<key>\`` directive in `/spec-write` SKILL.md must have a matching const entry. Regex used by the probe MUST be `MUST emit\s*\`<key>\`` (literal — captures the FIRST backticked key after `MUST emit`); the compound-form `\` / \`` continuation case at L327 (`milestone_local_orphan`) is acknowledged as a separate spec-prose hygiene item (see Requirement section).

Spec-research seed (STE-230 AC-STE-230.6): `spec_research_invoked` — 4 precedents returned (HG95V0 original Pass-1 framing, HG95TM canonical two-stage contract, STE-238 probe scope origin, STE-308 forked-auditor introducer). Decision validated by HG95TM AC-23.6: `code-reviewer` retains Pass-1 authority.

Deps-research vacuous (`specs/deps.yaml` absent — zero entries → fork skipped).
