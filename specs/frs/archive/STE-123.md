## Requirement

`/setup`'s `## /setup audit` section is the canonical signal that a project was set up autonomously / with pre-baked answers (STE-108 AC-STE-108.7). Probe-19 (`setup-audit-section-presence`, M29 STE-108) hard-fails at gate time when the section is missing in a toolkit-managed CLAUDE.md that has default-applied step outcomes visible (`branch_template:` populated OR `## Docs` block present).

The `appendAuditEntry` helper (`adapters/_shared/src/setup/audit_log.ts`) exists. The `/setup` SKILL.md prose at steps 7b/7c/7d says each default-applied outcome MUST call `appendAuditEntry` (lines 207, 213, 231 of the current SKILL.md). But the LLM-as-runtime can skip those calls — smoke-test 2026-04-27 evidence: the option-5 path produced CLAUDE.md with `## Task Tracking` + `## Docs` (probe-19 trigger conditions met) but NO `## /setup audit` heading.

Same M31-class discretionary-check pattern as STE-117/118/119/121/122. Fix: convert the audit section from a discretionary append into a deterministic post-condition that `/setup` enforces before its bootstrap commit. Probe-19 stays at gate time as the safety net.

## Acceptance Criteria

- AC-STE-123.1: **New `/setup` step 8a "audit-section post-condition".** SKILL.md gains step 8a between 7d (docs modes) and 8b (bootstrap commit). Annotation: `verification: runs unconditionally regardless of prompt mode`. The step reads CLAUDE.md, calls `hasDefaultApplicableOutcomes(content)` (the same function probe-19 uses at gate time, exported from `setup_audit_section_presence.ts`), and proceeds based on the result. {#AC-STE-123.1}
- AC-STE-123.2: **Synthesis path.** If `hasDefaultApplicableOutcomes` returns true AND the `## /setup audit` section is absent (string `"## /setup audit"` not found in content), step 8a calls `synthesizeAuditSection(claudeMdPath, resolvedDefaults)` from a new helper `adapters/_shared/src/setup/synthesize_audit.ts`. The helper reads the in-scope resolved-defaults table (`branch_template` value, three `## Docs` flag values) — passed in by /setup, never re-derived from CLAUDE.md to avoid feedback loops — and calls `appendAuditEntry` once per default-applied outcome. Bullet rendering reuses `audit_log.renderBullet` so output is byte-identical to per-step append output. {#AC-STE-123.2}
- AC-STE-123.3: **Idempotent.** If per-step append calls in 7b/7c/7d already wrote the audit section partially, step 8a appends only the missing entries. Deduplication key: `(step, field)` tuple — e.g., `(7c, branch_template)` is unique per step. Existing entries preserved verbatim (append-only contract per STE-108 AC-STE-108.7). Test fixtures cover (a) section absent + 3 defaults applied → 3 entries synthesized; (b) section partial with 1 entry present → 2 missing entries synthesized; (c) section complete with 3 entries present → no-op. {#AC-STE-123.3}
- AC-STE-123.4: **Bootstrap commit refusal.** If step 8a runs and the post-condition still fails after synthesis attempt (e.g., resolved-defaults table is empty but `hasDefaultApplicableOutcomes` returns true — invariant violation), step 8b refuses to start with NFR-10 canonical shape: `"setup: audit-section post-condition unsatisfiable — resolved-defaults table empty but CLAUDE.md shows default-applied outcomes; investigate"`. The bootstrap commit is the load-bearing exit point, so refusal here halts /setup before the commit lands. {#AC-STE-123.4}
- AC-STE-123.5: **Vacuous on fully-interactive runs.** If the user answered every prompt explicitly (no defaults applied — operator confirmed every value), `hasDefaultApplicableOutcomes` returns false (the toolkit-marker is present but neither `branch_template:` nor `## Docs` triggers the check) → step 8a is a no-op. Bootstrap commit proceeds normally. Probe-19 also passes vacuously at gate time. Test fixture covers this path. {#AC-STE-123.5}
- AC-STE-123.6: **Option-5 idempotent-merge fixture.** Test fixture covers the smoke-test 2026-04-27 reproducer: pre-existing CLAUDE.md (carrying `## Task Tracking` + `## Docs`, no `## /setup audit`, no toolkit marker yet) merged with new defaults during /setup; /setup adds the toolkit marker, applies defaults, runs step 8a; the audit section is synthesized from the resolved-defaults table; probe-19 passes immediately after step 8a returns. The fixture asserts the synthesized bullets match the byte-shape of per-step appends. {#AC-STE-123.6}
- AC-STE-123.7: **`bun test` green.** Test count delta ~+10: synthesize_audit unit (happy path / idempotent / no-op fully-interactive / refusal-on-empty-defaults / option-5 fixture / byte-identity round-trip / em-dash branch_template value), plus updates to `setup.test.ts` for step 8a integration. {#AC-STE-123.7}

## Technical Design

**Helper module:**

```typescript
// adapters/_shared/src/setup/synthesize_audit.ts

import { appendAuditEntry } from "./audit_log.js";
import { readFileSync } from "node:fs";

export interface ResolvedDefaults {
  date: string;                                       // ISO date (YYYY-MM-DD)
  branchTemplate?: { value: string; reason: string }; // step 7c
  docsUserFacing?: { value: boolean; reason: string }; // step 7d
  docsPackages?: { value: boolean; reason: string };
  docsChangelogCi?: { value: boolean; reason: string };
}

export class AuditPostconditionUnsatisfiable extends Error {}

export function synthesizeAuditSection(
  claudeMdPath: string,
  resolvedDefaults: ResolvedDefaults,
): { synthesized: number; skipped: number } {
  const content = readFileSync(claudeMdPath, "utf-8");
  const existing = parseExistingAuditEntries(content); // returns Set<`${step}:${field}`>

  const candidates = [
    resolvedDefaults.branchTemplate && {
      step: "7c", field: "branch_template",
      value: resolvedDefaults.branchTemplate.value,
      reason: resolvedDefaults.branchTemplate.reason,
    },
    resolvedDefaults.docsUserFacing && {
      step: "7d", field: "docs.user_facing_mode",
      value: resolvedDefaults.docsUserFacing.value,
      reason: resolvedDefaults.docsUserFacing.reason,
    },
    resolvedDefaults.docsPackages && {
      step: "7d", field: "docs.packages_mode",
      value: resolvedDefaults.docsPackages.value,
      reason: resolvedDefaults.docsPackages.reason,
    },
    resolvedDefaults.docsChangelogCi && {
      step: "7d", field: "docs.changelog_ci_owned",
      value: resolvedDefaults.docsChangelogCi.value,
      reason: resolvedDefaults.docsChangelogCi.reason,
    },
  ].filter((x): x is NonNullable<typeof x> => x !== undefined);

  if (candidates.length === 0) {
    throw new AuditPostconditionUnsatisfiable(
      "resolved-defaults table empty — cannot synthesize audit section",
    );
  }

  let synthesized = 0;
  let skipped = 0;
  for (const c of candidates) {
    const key = `${c.step}:${c.field}`;
    if (existing.has(key)) {
      skipped++;
      continue;
    }
    appendAuditEntry(claudeMdPath, {
      date: resolvedDefaults.date,
      step: c.step,
      field: c.field,
      value: c.value,
      reason: c.reason,
    });
    synthesized++;
  }
  return { synthesized, skipped };
}
```

**`hasDefaultApplicableOutcomes` exposed:** the function currently lives privately inside `setup_audit_section_presence.ts`. As part of this FR, export it so `/setup` step 8a can reuse the same logic probe-19 uses at gate time — single source of truth for the check.

**`/setup` step 8a (SKILL.md):**

```
### 8a. Audit-section post-condition (STE-123)

`verification:` runs unconditionally before step 8b's bootstrap commit. Reads CLAUDE.md, calls
`hasDefaultApplicableOutcomes(content)` from `adapters/_shared/src/setup_audit_section_presence.ts`,
and:

- If false: no-op. Continue to step 8b.
- If true AND `## /setup audit` heading present: no-op. Continue to step 8b.
- If true AND heading absent: call `synthesizeAuditSection(claudeMdPath, resolvedDefaults)` from
  `adapters/_shared/src/setup/synthesize_audit.ts`. Pass the in-scope resolved-defaults table
  populated during steps 7b/7c/7d. On `AuditPostconditionUnsatisfiable` error, refuse with NFR-10
  canonical shape and abort /setup before step 8b can commit malformed output.

This step is the deterministic byproduct of any autonomous /setup run; probe-19 satisfaction is
guaranteed at the file-shape level when /setup returns clean.
```

**Why pass `resolvedDefaults` instead of re-deriving from CLAUDE.md:** re-deriving from CLAUDE.md would require parsing the just-written `branch_template:` and `## Docs` flag values, which `hasDefaultApplicableOutcomes` already does at the boolean level — but parsing for *values* re-introduces the brittleness this FR is closing. The resolved-defaults table is in-memory state that /setup already has; passing it explicitly makes the synthesis path's inputs unambiguous.

**Why probe-19 stays at gate time:** probe-19 covers paths that bypass /setup entirely (manual CLAUDE.md edits, downstream toolkit consumers, future skills that mutate CLAUDE.md). The in-band post-condition is the first-line defense; probe-19 is the safety net.

## Testing

- `synthesize_audit.test.ts` — 7+ scenarios:
  - Section absent + 3 defaults applied → 3 entries synthesized; section header created at end of file with one blank line above.
  - Section partial (1 entry present, 2 candidates remaining) → 2 entries synthesized; existing entry untouched.
  - Section complete (all 3 entries present matching candidates) → no-op (`{ synthesized: 0, skipped: 3 }`).
  - Empty `resolvedDefaults` table → throws `AuditPostconditionUnsatisfiable`.
  - Em-dash in `branch_template` value (`feat/{ticket-id}-—-{slug}` — pathological but possible) → bullet rendered via `JSON.stringify` round-trips byte-identically.
  - Byte-identity round-trip: synthesized bullets match the shape of `appendAuditEntry` per-step output.
  - Reason string with embedded `"` → escaped via `JSON.stringify` correctly.
- `setup.test.ts` updates — step 8a integration:
  - Fully-interactive run (no defaults applied) → step 8a no-op; probe-19 passes vacuously.
  - Autonomous run with all defaults → step 8a synthesizes 3 entries; probe-19 passes.
  - Option-5 fixture (pre-existing CLAUDE.md, idempotent-merge branch, defaults applied) → step 8a synthesizes; probe-19 passes; bootstrap commit succeeds.
  - Empty resolved-defaults but `hasDefaultApplicableOutcomes` true → step 8a refuses with NFR-10; bootstrap commit aborted.
- `setup_audit_section_presence.test.ts` updates — assert `hasDefaultApplicableOutcomes` exported and exposed for cross-module reuse.

## Notes

**Sequencing constraint:** Independent of STE-121 and STE-122 — different files, different probes, no shared module. Phase A parallel.

**Why synthesis instead of just blocking:** blocking /setup when the audit section is missing would require the operator to re-run after manually inserting the section, which defeats the purpose of autonomous mode. Synthesis from the in-scope resolved-defaults table makes the section a deterministic byproduct, satisfying probe-19 without operator intervention.

**Why audit section must exist before commit:** the bootstrap commit (STE-109) is the load-bearing handoff point. If /setup commits without the audit section, the malformed CLAUDE.md is now in git history, and the next `/gate-check` reports the failure on a state the operator can't easily fix without rewriting history. Step 8a's pre-commit verification ensures the commit itself is clean.

**Mode-none impact:** /setup in `mode: none` writes `## Task Tracking` with `mode: none` (or skips it entirely on the no-tracker branch). `branch_template:` is conditionally written. `## Docs` is always written per STE-107. So step 8a applies to mode-none too — when defaults are applied to `## Docs` flags, the audit section is synthesized. Vacuous only when the operator answered every prompt explicitly.

**Probe-19 at gate-time semantics unchanged:** probe-19 still hard-fails at gate-check time when the audit section is missing. STE-123 closes the pathway that produces malformed output during /setup itself; the gate-time check remains the universal safety net.

**Why not a hook:** Claude Code hooks could in principle run a post-`/setup` check, but the `/setup` skill needs to remain self-contained for portability across consumer projects (NFR-1, the plugin's portability boundary). In-skill post-condition keeps the contract inside the skill.
