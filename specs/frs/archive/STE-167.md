# STE-167: /ship-milestone portability — release-config block + per-stack defaults {#STE-167}

## Requirement

Every skill under `plugins/dev-process-toolkit/skills/` is end-user-facing — `/smoke-test` is the only maintainer-only tool, and it lives outside the plugin tree by design. But `/ship-milestone` hard-codes the **toolkit's own** four release artifacts in its skill body:

1. `plugins/dev-process-toolkit/.claude-plugin/plugin.json` `version`
2. `.claude-plugin/marketplace.json` `version`
3. `CHANGELOG.md` Keep-a-Changelog `## [X.Y.Z] — YYYY-MM-DD — "Codename"` header
4. `README.md` `Latest: **vX.Y.Z — 'Codename'**` line

End-users running `/ship-milestone` in their own project hit those paths and the skill breaks: their project has no `.claude-plugin/`, no `marketplace.json`, no toolkit-shaped README line. The four-file shape is the **toolkit's release artifact list**, encoded as if it were universal.

The fix: drive the bump from a `## Release Files` block in the user's `CLAUDE.md`, populated by `/setup` from a per-stack default in `examples/<stack>/release.yml`. The toolkit's own CLAUDE.md gets the same block (populated from a new `examples/plugin/release.yml`), making the toolkit a normal consumer of its own contract — self-hosting for free.

Out of scope:
- Migrating *other* hard-coded toolkit-internal paths (e.g., `/gate-check` probe paths — those land in STE-168; `/implement` error message — STE-169).
- Codename allocation / changelog-entry composition — those are `/ship-milestone` concerns unchanged by this FR.
- Multi-package monorepo bump orchestration (one project, one block, one bump pass; monorepo support is a future FR).

## Acceptance Criteria

- AC-STE-167.1: A `## Release Files` block is defined in the CLAUDE.md schema (additive per NFR-2). The block carries a YAML-fenced payload of the shape:

  ```yaml
  files:
    - path: <relative path>
      kind: json | toml | yaml | changelog | regex
      # kind-specific fields:
      #   json/toml/yaml → field: <dot-path, e.g. version or package.version>
      #   regex          → pattern: <regex with named (?<version>...) group>
      #                    replace: <template using {version}>
      # optional: bool (default false) — missing path is `n/a` not error
  ```

  The schema and per-kind contract are documented in `plugins/dev-process-toolkit/docs/ship-milestone-reference.md`.

- AC-STE-167.2: `/setup` detects the project stack (reusing existing stack-detection in the skill) and writes the corresponding `examples/<stack>/release.yml` content into the user's CLAUDE.md `## Release Files` block. For unrecognized stacks, `/setup` writes a commented stub plus a one-line "fill this in before running /ship-milestone" pointer. If the block already exists, `/setup` leaves it alone — user-edited overrides win.

- AC-STE-167.3: Four per-stack defaults ship under `plugins/dev-process-toolkit/examples/`:
  - `typescript/release.yml` — `package.json` (kind: json), `CHANGELOG.md` (kind: changelog), `README.md` (kind: regex, optional)
  - `flutter/release.yml` — `pubspec.yaml` (kind: yaml), `CHANGELOG.md`, `README.md` (optional)
  - `python/release.yml` — `pyproject.toml` (kind: toml), `CHANGELOG.md`, `README.md` (optional)
  - `plugin/release.yml` — the toolkit's own four-file shape (the two `.claude-plugin/*.json` entries, CHANGELOG, README)

- AC-STE-167.4: `/ship-milestone` reads CLAUDE.md `## Release Files` and bumps each entry by its `kind`. The four hard-coded toolkit-specific paths are removed from the skill body. Refusal: a missing/empty block surfaces NFR-10 canonical refusal naming the absent block and pointing the user to `/setup` or to add the block manually.

- AC-STE-167.5: The toolkit's own `CLAUDE.md` gets a `## Release Files` block populated from `examples/plugin/release.yml`. The "Release Checklist" prose section in `CLAUDE.md` is rewritten to point at the block as the source of truth (single source). `/ship-milestone M48` releases the toolkit using the new code path — self-hosting verified end-to-end.

- AC-STE-167.6: Unit + round-trip test coverage. Unit tests per `kind` (parse + bump). Round-trip test loads each `examples/<stack>/release.yml`, asserts schema validity, and verifies a stub bump produces the expected diff. Integration test runs `/ship-milestone` against a stub end-user project (TS skeleton with `package.json` + `CHANGELOG.md`) and asserts both files bump correctly.

- AC-STE-167.7: `bun test` green; `/gate-check` clean; `/spec-review M48` reports no deviations.

## Technical Design

**`## Release Files` block — YAML schema.**

Required top-level key: `files: [<entry>]`. Each entry:

| Field | Required? | Type | Notes |
|-------|-----------|------|-------|
| `path` | yes | string | Repo-relative path |
| `kind` | yes | enum | `json` / `toml` / `yaml` / `changelog` / `regex` |
| `field` | iff `kind: json\|toml\|yaml` | string | Dot-path (e.g., `version`, `plugins[0].version`, `package.version`) |
| `pattern` | iff `kind: regex` | string | Regex with named `(?<version>...)` group |
| `replace` | iff `kind: regex` | string | Template — `{version}` placeholder substituted |
| `optional` | no | bool (default false) | If true, missing path is `n/a` not error |

**`kind: changelog`** is a fixed contract: insert a new `## [X.Y.Z] — YYYY-MM-DD — "Codename"` section at the top, below the intro. The codename + body are still composed by `/ship-milestone` from milestone metadata — the entry only declares *which file* gets the new section, not what goes in it.

**`kind: yaml` for Flutter `pubspec.yaml`** preserves any `+<build>` suffix on the version line — the bump rewrites the semver portion only and the build suffix passes through unchanged.

**`/ship-milestone` flow change:**

1. Parse `## Release Files` from CLAUDE.md (refuse if absent/empty).
2. For each entry: read current value, compute new value, stage the diff.
3. Show consolidated diff to user, get approval.
4. Apply, commit, done.

The codename + changelog body composition logic stays unchanged — only the *file targeting* moves to the block.

**`/setup` flow change:**

After stack detection, copy `examples/<stack>/release.yml` content into the user's CLAUDE.md `## Release Files` block. For projects already passing through `/setup` (regenerate or migrate), if the block already exists, leave it alone (user-edited overrides win).

**Toolkit self-hosting.** The toolkit's `CLAUDE.md` gains:

```yaml
## Release Files

files:
  - path: plugins/dev-process-toolkit/.claude-plugin/plugin.json
    kind: json
    field: version
  - path: .claude-plugin/marketplace.json
    kind: json
    field: plugins[0].version
  - path: CHANGELOG.md
    kind: changelog
  - path: README.md
    kind: regex
    pattern: 'Latest: \*\*v(?<version>\d+\.\d+\.\d+) — '
    replace: 'Latest: **v{version} — '
```

The "Release Checklist" prose at the top of `CLAUDE.md` is trimmed to a one-line pointer at this block.

**`docs/ship-milestone-reference.md`** (new file) carries the full schema reference + per-kind worked examples + the "writing your own override" guide for end-users with non-standard layouts.

**No legacy fallback.** Per the "no users yet" project memory, the legacy hardcoded path is removed in this FR — there's no installed base to migrate. If self-hosting at `/ship-milestone M48` time finds a bug, fix it before bumping.

## Testing

- `adapters/_shared/src/release_config.ts` (new) — parser + per-kind bump helpers.
  - Unit tests per kind: `json` (with and without nested `field`), `toml`, `yaml`, `changelog`, `regex`.
  - Negative tests: missing fields per kind, malformed YAML, missing `(?<version>)` group, conflicting fields.
  - Refusal-shape tests: NFR-10 canonical message on missing/empty block.
- Round-trip integration test: load each `examples/<stack>/release.yml`, run a stub bump from `0.1.0 → 0.2.0`, assert expected diff.
- End-to-end smoke (the M48 retroactive run also covers this): stub end-user TS project + `/setup` + `/ship-milestone` → diff shows correct bumps for `package.json` + `CHANGELOG.md`.
- Self-host smoke: `/ship-milestone M48` against the toolkit itself uses the new code path and produces a diff byte-equivalent to what the legacy hardcoded path would have produced (golden snapshot, captured before the legacy path is removed).

## Notes

- Origin: brainstorm session 2026-04-30, where the operator hit the toolkit-internal-path bug while testing `/ship-milestone` in a non-toolkit project.
- Hybrid design (convention defaults from `examples/` + CLAUDE.md override) chosen over pure-convention or pure-config — see brainstorm transcript.
- This FR removes the only end-user-facing portability bug in `/ship-milestone`. STE-168 / STE-169 close two smaller portability bugs in `/gate-check` and `/implement` in the same milestone.
- The `kind: regex` escape hatch is intentional — README "Latest:" lines and other free-form patterns can vary between projects; users with non-standard layouts get a clean override path without forcing a new `kind` enum entry per project.
- Future work (not this FR): monorepo support (multiple bump roots), per-package codename allocation, changelog auto-composition from FR titles.
