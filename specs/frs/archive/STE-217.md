# STE-217: /gate-check probe #28 recognizes negative-assertion verify lines {#STE-217}

## Requirement

`/gate-check` probe #28 (`stale-file-refs` / verify-line probe) walks `verify:` lines in milestone plan files and asserts each path-reference exists on disk. Negative-assertion verify lines (e.g., `ls src/.placeholder.test.ts returns "No such file or directory"`) deliberately reference deleted paths to assert their absence — the path *should not* exist, and probe #28 incorrectly flags those lines as stale refs.

**Findings bundled:** F4 from `/tmp/dpt-smoke-findings-2026-05-04-linear.md` (Run #10 v2.7.0).

**Symptoms observed in 2026-05-04 Linear smoke #10:**

1. `specs/plan/M2.md:30` carries a verify line of the shape `ls src/.placeholder.test.ts returns "No such file or directory"` (a *negative* existence assertion — the placeholder shield was deleted in commit `9aabd6c`).
2. Probe #28 doesn't distinguish negative-from-positive verify lines and flags the file-not-found as a stale ref.
3. The advisory is stable on every Linear smoke run as long as the M2 plan file lives in the active set; surfaces as a low-severity false positive that pollutes the smoke audit trail.
4. A future `/spec-archive M2` sweep removes the line alongside the rest of the milestone, but the probe's logic stays broken — any future negative-assertion verify line in any other milestone reproduces the same false positive.

## Acceptance Criteria

- AC-STE-217.1: Probe #28 recognizes negative-assertion verify lines and skips the path-existence check on those lines. The recognized shapes (matched in order, any-match short-circuits): (1) line contains the literal substring `returns "No such file or directory"` (case-insensitive on the message portion); (2) line starts with or contains the negation prefix `! test -` (POSIX shell negation, case-sensitive); (3) line contains the flag substring `--non-existent`; (4) line contains the substring `does NOT exist` (case-insensitive on the verb). {#AC-STE-217.1}
- AC-STE-217.2: The match logic is regex/grep-based and applies to the verify line's content as a whole. Partial matches inside a longer compound command also count — e.g., a line like `ls foo.ts returns "No such file or directory" && echo done` matches because the negative-assertion substring appears anywhere in the line. {#AC-STE-217.2}
- AC-STE-217.3: When a verify line matches any of the four negative-assertion patterns, probe #28 records it internally as `negative-assertion` (informational only — no advisory row, no GATE FAILED, no entry in the closing summary). The line is treated as if the path-existence check passed. {#AC-STE-217.3}
- AC-STE-217.4: Verify lines that do NOT match any negative-assertion pattern continue to surface as advisories on missing paths — the existing positive-assertion behavior is preserved exactly. The fix is purely additive: negative-assertion shapes get a new exemption; everything else behaves as before. {#AC-STE-217.4}
- AC-STE-217.5: Test fixture (negative): `specs/plan/M2.md:30`-style line with `returns "No such file or directory"` ⇒ no probe #28 advisory. Test fixture (positive): a verify line on a missing path with no negative-assertion marker ⇒ probe #28 advisory still fires. Test fixture (mixed): a plan with both shapes ⇒ only the positive-assertion line surfaces in the advisory output. {#AC-STE-217.5}
- AC-STE-217.6: The recognizer pattern set is documented in `skills/gate-check/SKILL.md` § probe #28 with each pattern shown alongside an example. Future negative-assertion shapes can be added by amending the pattern set (the recognizer is the single source of truth — extend by editing the list, not by adding ad-hoc match logic elsewhere). {#AC-STE-217.6}

## Technical Design

**Where probe #28 lives.** `/gate-check`'s probe set is documented in `skills/gate-check/SKILL.md`; probe #28 is the verify-line / stale-file-refs probe. Its current implementation walks each `verify:` line in the milestone plan, extracts path-references via a regex, and asserts each exists on disk.

**Recognizer set.** Add a small recognizer that, given a verify line, returns true if any of the four negative-assertion patterns match:

```typescript
function isNegativeAssertion(line: string): boolean {
  if (/returns\s+"No such file or directory"/i.test(line)) return true;
  if (/(^|\s|;|&&|\|\|)!\s*test\s+-[a-z]/.test(line)) return true;
  if (/--non-existent\b/.test(line)) return true;
  if (/does\s+NOT\s+exist/i.test(line)) return true;
  return false;
}
```

**Probe branch.** Before asserting path-existence, call `isNegativeAssertion(line)`. If true, skip the assertion (record internally, no advisory). Otherwise, proceed with the existing path-existence logic.

**Documentation.** `skills/gate-check/SKILL.md` § probe #28 grows a "Negative-assertion exemptions" subsection listing the four patterns with one example each:

```
- `returns "No such file or directory"` — POSIX file-not-found error message
- `! test -<flag>` — POSIX shell negation prefix
- `--non-existent` — flag indicating intentional absence
- `does NOT exist` — natural-language negative assertion
```

Future extensions amend that list, not the probe code.

## Testing

- Unit test: each of the 4 patterns recognized; combinations tested (e.g., a line with two markers returns true from a single match); non-matching lines pass through to the existence check.
- Unit test: case-sensitivity rules verified — `returns "no such file or directory"` (lowercase) matches; `! Test -f` (capitalized `Test`) does NOT match (POSIX is case-sensitive).
- Smoke regression: probe #28's stable advisory on `specs/plan/M2.md:30` (M54 carry-forward) silenced post-fix.
- Property test: `isNegativeAssertion` is total — every string input returns boolean, never throws.
