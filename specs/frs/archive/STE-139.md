# STE-139: NFR-1 enforcement triple: extract overflow + per-skill probe + missing self-checks {#STE-139}

## Requirement

NFR-1 declares a **≤300-line** cap per `SKILL.md` (`specs/requirements.md:17`). Two of the 14 shipped skills violate it:

```
setup           428 lines  [+43% over]
implement       371 lines  [+24% over]
spec-write      256 lines  [ok]
gate-check      211 lines  [ok]
... (other 10 skills under cap)
```

NFR-1 also says: "If a skill approaches 300 lines during implementation, extract detailed reference material into a companion file at `docs/<skill-name>-reference.md`." Companion docs already exist (`docs/setup-tracker-mode.md`, `docs/setup-docs-mode.md`, `docs/implement-tracker-mode.md`, `docs/implement-reference.md`), but the parent skills carry too much inline content anyway.

`tests/docs-skill-shape.test.ts:55-65` enforces ≤300 lines for `docs/SKILL.md` only (STE-70 AC-STE-70.7); the other 13 skills get no enforcement. Separately (F3a), 23 of 29 gate-check probe tests lack a repo-self-check, so live-repo drift like F0 (STE-136) stays invisible to `bun test`.

This FR lands all three: extract the overflow content (F1), generalize the line-cap test (F3), and backfill 23 missing repo-self-check rows (F3a).

## Technical Design

**Phase A — Extract overflow (AC-STE-139.1, 139.2).** Push to companion files:

- `setup/SKILL.md` 428 → ≤300: candidate sections to extract are the long Step 7 hook-install prose (~80 lines), the audit-section/--commitlint flag handling (~50 lines), and Step 2c bun placeholder shape (~30 lines). Target landing zones: `docs/setup-tracker-mode.md` for tracker-mode-flow detail, `docs/setup-docs-mode.md` for docs-mode flag handling, optionally a new `docs/setup-hook-install.md`.
- `implement/SKILL.md` 371 → ≤300: candidate sections to extract are the Phase 4 archival prose (~90 lines, already partly mirrored in `docs/implement-reference.md`), branch-proposal flow (~40 lines), and tracker-writes-rules (~30 lines). Target landing: `docs/implement-reference.md` (extend existing), or `docs/implement-tracker-mode.md`.

Each extracted block becomes a cross-reference in the parent skill: e.g., `**Hook install:** see \`docs/setup-hook-install.md\`.`

**Phase B — Per-skill length probe (AC-STE-139.3 / 139.4).** Generalize the existing test:

```ts
import { readdirSync, statSync, readFileSync } from "node:fs";
import { join } from "node:path";

const skillsDir = join(import.meta.dir, "..", "skills");
const skills = readdirSync(skillsDir).filter(d => statSync(join(skillsDir, d)).isDirectory());

for (const skill of skills) {
  test(`NFR-1: ${skill}/SKILL.md is ≤300 lines`, () => {
    const path = join(skillsDir, skill, "SKILL.md");
    const lines = readFileSync(path, "utf8").split("\n").length;
    expect(lines).toBeLessThanOrEqual(300);
  });
}
```

If elevating to a probe, follow the existing probe-implementation pattern in `adapters/_shared/src/` (e.g., `claudemd_docs_section.ts` shape).

## Testing

- **Per-skill length** (AC-STE-139.3): the new `tests/skill-nfr-1-length.test.ts` asserts each `SKILL.md ≤ 300`.
- **23 self-checks** (AC-STE-139.5): each backfilled block runs the probe against the live repo and asserts empty violations. AC-STE-139.8 requires verifying each probe passes BEFORE adding the assertion.
- **Smoke** — re-run the smoke-test skill end-to-end after Phase A to confirm setup/implement skills still drive their flows correctly with content moved to companions. The companions must be reachable via the cross-references the skill carries.
- **`bun test`** is green.

## Technical Risks

- **Extract too aggressively.** Pushing too much detail to companions can leave the parent skill incoherent. Mitigation: keep all step-numbered prose in the skill; move only reference-grade detail (tables, error catalogues, edge-case enumeration) to companions.
- **Cross-reference drift.** Companions must be reachable via the skill's references; renaming a section in the companion silently breaks the link. Mitigation: prefer stable anchors and run `/gate-check` traceability probe after the extraction.

## Notes

The triple-fix is bundled because each leg reinforces the others: extracting overflow (F1) without an enforcement probe (F3) is one-shot; the probe alone catches future drift but doesn't unblock today's violations; and the 23 self-checks (F3a) are the structural pattern that surfaced F0 in the first place.

Per the no-users-yet principle: setup/implement skills are large because they accreted milestone-specific prose; the right move is to extract aggressively, not to soft-land the cap.
