# STE-370: post-merge ceremony surfacing: /pr pre-flight + ship-debt offer + /pr chain + recipe {#STE-370}

## Requirement

The split flow — implementation PR merged before `/ship-milestone` runs — must be surfaced at every operator touchpoint and the leftover ceremony must collapse to one guided command, **without ever blocking legitimate early merges** (spec-only `--no-tech` PRs merge by design per STE-227; batched sibling-branch milestones stay parked per STE-369). Today `/pr` has zero milestone awareness, bare `/ship-milestone` refuses flat when no active plan exists, and the post-merge recipe is undocumented folklore even though the already-archived case works end-to-end via the STE-210 fallback.

## Acceptance Criteria

- AC-STE-370.1: **/pr ship-state pre-flight (soft).** `/pr` Step 1 gains a pre-flight: `git diff main...HEAD --name-status` detecting `specs/{plan,frs}/**` paths moved or added under `archive/` while `git log main..HEAD` contains no `chore(release):` commit ⇒ exact-format prompt `[m]erge later / [s]hip first / [a]bort`. It never auto-blocks; `s` exits with zero side effects and the hint `Run /ship-milestone M<N>, then re-run /pr`; `a` aborts cleanly. Branches with no archive moves (spec-only PRs included) see no prompt at all. {#AC-STE-370.1}
- AC-STE-370.2: **PR-body follow-up stamp.** Choosing `m` injects a `Follow-up: /ship-milestone M<N>` line into the PR body for each affected milestone, so the merged PR itself documents the outstanding ceremony. {#AC-STE-370.2}
- AC-STE-370.3: **Ship-debt offer.** Bare `/ship-milestone` (no argument) with zero `status: active` plans offers archived-unstamped-unparked milestones newest-first via the exact-format prompt `Unshipped archived milestone M<N> — ship it? [y/N]` instead of the flat "no active plan" refusal; decline preserves today's refusal text and exit code. {#AC-STE-370.3}
- AC-STE-370.4: **Opt-in /pr chain.** After a successful release commit landing off-trunk, `/ship-milestone` fires the STE-75-style exact-format opt-in `Open ceremony PR via /pr now? (y/n):`. `y` chains `/pr` in-process with all `/pr` gates intact (tracker probe, uncommitted-changes confirmation, push — `/ship-milestone` itself still never pushes); decline prints the hint `Run: /pr`; chain-start failure ⇒ NFR-10 canonical refusal. {#AC-STE-370.4}
- AC-STE-370.5: **Recipe docs + meta-tests.** `docs/ship-milestone-reference.md` gains a `## Post-merge ceremony` section: fresh branch off main → `/spec-archive M<N>` (only when FRs are still active on main) → `/ship-milestone M<N>` → chained `/pr`; it documents that the already-archived case is fully supported today via the STE-210 archive fallback + STE-228 off-trunk branch-gate no-op. Meta-tests pin the pre-flight prompt, the `Follow-up:` line shape, the debt-offer prompt, the chain prompt + decline hint, and the docs section heading; NFR-1 line caps stay green on every edited skill. {#AC-STE-370.5}

## Technical Design

- **`skills/pr/SKILL.md`:** new pre-flight section between `## Tracker Mode Probe` and `## Steps` (the skill already has the named-pre-PR-probe structural precedent). Detection is tree-based and topology-immune: archive-path additions in `git diff main...HEAD --name-status` (statuses `A`/`R*` under `specs/plan/archive/`, `specs/frs/archive/`) minus the release-marker check (`git log main..HEAD --grep '^chore(release'` non-empty ⇒ shipped ⇒ no prompt). Milestone names for the prompt/body line are parsed from the moved plan filenames.
- **`skills/ship-milestone/SKILL.md`:** (a) no-arg resolution extension — when the active-plan scan comes back empty, enumerate `specs/plan/archive/*.md` lacking both `shipped_in` and `ship_state: parked` (reuses STE-369's predicate), sort by `archived_at` descending, prompt per milestone; (b) post-ship chain — after the step-7 commit succeeds off-trunk, fire the opt-in and hand off to `/pr` in-process (STE-75 chain mechanics: prompt-not-silent-chain, downstream gates intact, exact-command fallback on decline).
- **Line-budget note:** ship-milestone's SKILL.md is the NFR-1 pressure point; if the two additions push it near 351 lines, the recipe/offer detail moves to `docs/ship-milestone-reference.md` with a one-line pointer (NFR-1 overflow rule).
