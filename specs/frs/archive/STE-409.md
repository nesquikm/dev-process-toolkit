# STE-409: Content-aware emptiness limb in the monolith-split detector {#STE-409}

## Summary

The monolith-split migration decides whether a project has already been split by asking whether the per-FR specs folder has any entries at all. The folder scaffold that setup writes — placeholder keep-files and an empty archive subfolder — counts as entries, so a legacy project that ran setup with a specs scaffold is treated as already split and the migration silently reports nothing to do. This FR makes the emptiness check content-aware: only real per-FR markdown files count as evidence of a completed split, so scaffold artifacts no longer disable the migration.

## Requirement

Origin: 2026-07-21 `/report-issue` gist (rh/stock_check_app, plugin 2.53.0, medium severity, reproducible) — two facets of one gap; this FR is the load-bearing detector half (facet 2, the `.gitkeep` sharp edge).

`detectMonolith`'s second limb treats `specs/frs/` as "the split has happened" whenever `readdirSync` returns ≥ 1 entry (`adapters/_shared/src/migrations/monolith_split.ts:235`). `readdirSync` includes dotfiles, and `/setup` step 8 scaffolds `.gitkeep` into BOTH `specs/frs/` and `specs/frs/archive/` — so on a scaffolded tree the listing is `[".gitkeep", "archive"]` and the detector returns `applies: false` with the monolith fully live in `specs/requirements.md`. Net: `/setup` with a specs scaffold, then `/upgrade` → "Nothing to do." with no signal to the operator.

A dotfile filter alone is insufficient — the `archive` directory entry survives it; the check must be a content check: real per-FR markdown is the only evidence of a completed split.

Constraint (STE-392 lineage): the mid-flow re-run semantics must survive — a half-migrated tree (sections still live in the monolith, real per-FR files already written) keeps the detector quiet; the freeze leg is what eventually clears the sections.

## Acceptance Criteria

- AC-STE-409.1: the emptiness limb holds when `specs/frs/` contains no real FR content — a recursive scan ignoring dot-entries (dotfiles and dot-directories) finds no regular `.md` file. A `.gitkeep`-only `specs/frs/` with live FR headings in `specs/requirements.md` ⇒ `applies: true`. {#AC-STE-409.1}
- AC-STE-409.2: the exact `/setup` scaffold shape — `specs/frs/` holding `.gitkeep` plus an `archive/` subdirectory that itself holds only `.gitkeep` — ⇒ `applies: true` (with live FR sections present). {#AC-STE-409.2}
- AC-STE-409.3: any real `.md` file at `specs/frs/` top level or under `archive/` ⇒ `applies: false` — the STE-392 re-run semantics are preserved on genuinely split or mid-flow trees. {#AC-STE-409.3}
- AC-STE-409.4: purity and throw-guard semantics are unchanged — an unreadable or non-directory `specs/frs/` still degrades to the "not split" verdict without throwing; the detector stays pure, synchronous, deterministic, network-free, and non-mutating (AC-STE-392.1 contract). {#AC-STE-409.4}
- AC-STE-409.5: the present-but-contentless evidence line names scaffold artifacts (wording: "specs/frs/ carries only scaffold artifacts — no FR has been split out into a per-FR file yet") and a test pins the wording byte-for-byte. {#AC-STE-409.5}
- AC-STE-409.6: regression fixtures in the colocated test cover all six shapes — `.gitkeep`-only, the full scaffold shape, real top-level `.md`, archive-only `.md`, plain-empty, and absent — each asserting the exact `applies` verdict. {#AC-STE-409.6}

## Technical Design

**`monolith_split.ts`.** Replace the `split.length > 0` verdict with a content predicate `hasRealFrContent(dir)`: `readdirSync(dir, { withFileTypes: true })`; skip entries whose name starts with `.`; a file entry whose name ends `.md` ⇒ `true`; a (non-dot) directory entry ⇒ recurse with the same rules; otherwise `false`. A throw at any level is caught and treated as "no content seen there", preserving the existing degrade-not-throw posture (the ENOTDIR guard subsumes into the predicate's catch).

The absent/unreadable branch keeps today's `split === null` semantics and evidence wording. The present-but-contentless branch's evidence line becomes the scaffold-aware wording (AC-STE-409.5). The third evidence line (flat `specs/plan.md` split-brain note) is unchanged.

Non-`.md` regular files (a stray notes file) do not evidence a split — only per-FR markdown counts. `liveFrSectionNumbers` and the first limb are untouched.

## Testing

The `specs/frs`-as-regular-file degrade guard stays covered.
- Evidence-wording byte-pin for the scaffold case (AC-STE-409.5).
- Full `bun test` from `plugins/dev-process-toolkit` green; no existing monolith-split or upgrade-staleness test regresses.

## Implementation notes

- hasRealFrContent doc-comment says the ENOTDIR guard is "subsumed", but detectMonolith retains a separate guarded readdirSync for the absent-vs-present evidence discriminator — the retained `split` read is spec-mandated (STE-409 Technical Design), so the wording is imprecise but the behavior is correct.
- evidence wording is imprecise on two untested shapes — a plain-empty specs/frs/ emits "carries only scaffold artifacts" though none exist, and a non-directory specs/frs regular file emits "is absent" though the path exists — both are spec-faithful and the `applies` verdict is correct in each.
