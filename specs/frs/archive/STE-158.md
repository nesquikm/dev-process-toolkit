# STE-158: Smoke #6+#7 minor polish {#STE-158}

## Requirement

`/smoke-test` runs #6 (Jira, `/tmp/dpt-smoke-findings-2026-04-29-jira.md`) and #7 (Linear regression, `/tmp/dpt-smoke-findings-2026-04-29.md`) surfaced four small skill-prose issues, each independently low-priority but each worth one-line fixes. Bundling them into a single FR keeps M45 traceable without inflating the milestone footprint:

1. **Linear smoke F1** — `/spec-write` emits no stdout in `claude -p` runs. The skill exits 0 with all side effects intact, but `/tmp/dpt-smoke-spec-write.log` is 1 byte. Operators driving `/spec-write` headlessly (CI, smoke runs, automated bootstrapping) get no completion signal to parse. The Step 7 closing-summary contract exists in `skills/spec-write/SKILL.md` but does not appear to fire in `-p` mode.
2. **Jira smoke F1** — Archived FR frontmatter records `archived_at: 2026-04-29T00:00:00Z` (ISO-8601 shape preserved, time component zeroed). The actual archive commit landed at ~17:23 local. Either intentional date-only normalization (worth documenting explicitly) or a precision drop in `/implement` Phase 4 step 14's frontmatter mutation.
3. **Jira smoke F2** — `/gate-check` reports `conformance-probes pass: 29/29 (most vacuous post-archive)` after the FR is archived. The wording suggests probes are running but mostly degenerating to no-ops because there is no active FR. Tighten the message to differentiate active probes from post-archive vacuous probes (e.g., `29/29 [4 active, 25 vacuous]`).
4. **Jira smoke F5** — `/spec-write` filename override messaging is implicit. The user prompt requested `<ulid>-greet-helper.md`; `/spec-write` correctly applied the jira-mode `Provider.filenameFor` policy (`DST-2.md`) but only mentioned the override when the user's prompt happened to carry an escape hatch. Surface a one-line "note: filename derived from tracker policy" in the skill's normal output regardless of whether the user proposed an alternative.

## Technical Design

All four findings are skill-prose changes, not adapter-driver code changes.

**F1 (Linear) / AC-STE-158.1** — `skills/spec-write/SKILL.md` Step 7 prose already mandates "MUST emit on every successful run, regardless of mode or invocation path". The Step 7 prose may already cover the contract; the gap is that the LLM in `-p` mode is short-circuiting before reaching it. Investigate whether the prose needs strengthening (e.g., "explicitly emit the table even when no questions were asked") or whether a smoke-test driver-side contract is missing (e.g., `--output-format stream-json`). Driver-side log-format is **out of scope** for this FR — focus on the skill prose contract first.

**F1 (Jira) / AC-STE-158.2** — Inspect the source of the date-only normalization. If it's deliberate (cross-timezone determinism), document in `skills/implement/SKILL.md` § Phase 4 step 14 with the rationale; if it's accidental (e.g., the archived_at was set from a `date +%Y-%m-%d` rather than `date +%Y-%m-%dT%H:%M:%SZ`), correct the prose to spec the full ISO timestamp. Either way, the fixture asserts the chosen contract.

**F2 (Jira) / AC-STE-158.3** — Update the gate-check post-archive summary to count active vs vacuous probes separately. The "vacuous" set is mostly the FR-traversal probes (1, 2, 7, 8, 14, 15, 25, 26, 27 — all of which scope to active FRs) when no active FRs remain post-archive. Compose the count from the existing probe runners.

**F5 (Jira) / AC-STE-158.4** — Add `filename_policy_override` to `skills/spec-write/SKILL.md` § Step 7 plain-language map and fire the row whenever `Provider.filenameFor(spec)` overrides the user-proposed filename in tracker mode (mode: none filenames are never policy-overridden — short-ULID is local-mint).

**Out of scope:** F3 (Linear, "untracked scaffold after smoke") — driver-side caveat, not a plugin issue. F4 (Linear, "Linear autolink wraps STE-prefix in `<issue>` tags") — Linear MCP behavior, not plugin. F3 (Jira, `## Docs` flag dump informational) — no action. F4 (Jira, DST-2 numbering carries cross-run history) — informational property of the no-delete Atlassian MCP. None of these need plugin-side work.
