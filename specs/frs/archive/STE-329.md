# STE-329: Jira project-milestone binding via create-on-write labels {#STE-329}

## Requirement

The Jira adapter declares `project_milestone: false` (STE-38/STE-154): an FR's `milestone: M<N>` is never mirrored onto the Jira ticket, and the milestone-attach chain (STE-118) routes Jira to the `milestone_attach_skipped_adapter_limit` no-capability gap (STE-198). Linear attaches a first-class project-milestone object and gate-check probe #26 verifies it.

The atlassian MCP exposes no version/milestone create-or-list tool (verified against the live tool surface), so a Linear-style milestone object is unreachable. But labels ARE create-on-write and MCP-writable (`editJiraIssue.additional_fields.labels`, STE-117/154). Mirror the milestone M-number onto the Jira issue as a hyphen label `milestone-M<N>`, verified by read-back — reaching parity with Linear's *used* milestone surface (name-grouping + verify) without raw REST.

## Acceptance Criteria

- AC-STE-329.1: **Capability flip.** `adapters/jira.md` Schema M sets `project_milestone: true` (was `false`) and adds `milestone_binding: label`. The `object` value (Linear / default when the key is absent) preserves projectMilestone-object semantics; `label` selects the create-on-write label path. {#AC-STE-329.1}
- AC-STE-329.2: **Label derivation.** The Jira milestone label is `milestone-<M-token>`, where `<M-token>` is the leading `M\d+` of the canonical milestone name built per STE-118 AC-STE-118.2 (plan-file H1, anchor stripped). Only `[A-Za-z0-9-]` characters (Jira labels forbid spaces). Example: canonical `M86 — Jira Project-Milestone Support` → label `milestone-M86`. {#AC-STE-329.2}
- AC-STE-329.3: **Read-merge-write attach via editJiraIssue.** Attaching the milestone label reads the issue's current labels (`getJiraIssue`), merges `milestone-<M-token>` in, and writes the union back via `mcp__atlassian__editJiraIssue.additional_fields.labels` — never clobbering existing labels. Idempotent (re-adding an existing label is a no-op). This is a dedicated milestone-label update path, distinct from the on-create-only `default_labels` rule (STE-117/154): on the update path `default_labels` are still NOT re-applied; only the milestone label is added. {#AC-STE-329.3}
- AC-STE-329.4: **Adapter-aware verify in `attach_project_milestone.ts`.** The helper branches on the adapter's `milestone_binding`. `object` (Linear): the existing list → create-if-missing → attach → verify `projectMilestone.name === canonical` path is byte-identical (no regression). `label` (Jira): skip the list/create steps (labels are create-on-write — no milestone object to enumerate), attach via the AC-STE-329.3 path, then verify by re-reading the issue and asserting `labels` contains `milestone-<M-token>`; success returns `{ capability: null }`. Missing or mismatched label → `MilestoneAttachmentError` (NFR-10 canonical shape), reusing the Unicode-aware byte-by-byte diagnostic. {#AC-STE-329.4}

The Linear branch is unchanged. `milestone_attach_skipped_adapter_limit` no longer fires for Jira (the capability is now `true`). {#AC-STE-329.5}

## Technical Design

**Files touched:**
- `plugins/dev-process-toolkit/adapters/jira.md` — Schema M `project_milestone: true` + `milestone_binding: label`; new Project Milestone section.
- `plugins/dev-process-toolkit/adapters/_shared/src/attach_project_milestone.ts` (+ `.test.ts`) — `milestone_binding` branch; M-token extraction; Jira attach + verify.
- `plugins/dev-process-toolkit/adapters/_shared/src/tracker_project_milestone_attached.ts` (+ `.test.ts`) — probe #26 Jira branch.
- `plugins/dev-process-toolkit/skills/spec-write/SKILL.md`, `skills/implement/SKILL.md` — un-Linear-only the milestone-attach prose.
- `plugins/dev-process-toolkit/skills/gate-check/SKILL.md` — probe #26 description note (adapter-aware).
- `plugins/dev-process-toolkit/docs/tracker-adapters.md` — Project-Milestone table Jira row.

**M-token:** `const M_TOKEN = canonicalName.match(/^(M\d+)/)?.[1]` — the unique milestone key; the Jira label is `` `milestone-${M_TOKEN}` ``. Identity is the M-token, not the full title (labels forbid spaces) — this is why verify is adapter-aware rather than reusing Linear's full-name byte-equality.

**Verify branch (no Linear regression):** the helper reads `milestone_binding` from the adapter (default `object`). `object` → unchanged. `label` → the Jira attach + verify path. The `MilestoneOps` / `getIssue` shape widens to optionally surface `labels: string[]` so the `label` branch can assert containment; the `object` branch ignores it.

**Read-merge-write:** `editJiraIssue.additional_fields.labels` REPLACES the label set, so the attach MUST read current labels first and write the union — otherwise it clobbers `default_labels` and any operator labels.

## Notes

- **Relationship to STE-284.** STE-284 added the optional `Provider.listMilestones()` interface for tracker-local reconciliation + the milestone-allocation leg-4. Its spec text (AC-STE-284.7) referenced a `JiraProvider.listMilestones()` "via Jira's milestone API" that never shipped — `adapters/jira/src/` holds only `discover_field.ts`, `AdapterDriver.listMilestones` is optional, and Jira omits it (→ `[]`). This FR does NOT implement Jira milestone LISTING (no JQL scan); it implements milestone ATTACH/binding via labels. Orthogonal surfaces. Milestone-number allocation stays local 3-way + git.
- **Why labels, not fixVersions/versions.** The atlassian MCP exposes no version create-or-list tool (verified against the live tool surface). `fixVersions` would require a missing create-version tool or raw REST, violating the Provider-semantics principle. Labels are the only MCP-writable create-on-write grouping surface.
- **Why the M-token, not the full name.** Jira labels forbid spaces; the label carries the M-number (the unique milestone key), not the descriptive title. Hence adapter-aware verify (Jira: token-in-labels; Linear: full-name equality) rather than loosening Linear's strict check.
- **Allocation unchanged.** Milestone numbers are allocated locally (plan files + git push-conflict arbitration); the label is a downstream mirror, never an allocator. The tracker offers no atomic milestone-number primitive (unlike issue-IDs, which ARE tracker-allocated), so delegating allocation would add a race + a network dependency for zero gain (brainstorm 2026-05-26).
- **This repo is mode:linear.** The Jira branch ships behind unit tests + a live-Jira smoke; no in-repo Jira FR exercises probe #26's Jira branch. The FR ticket itself (STE-329) is a Linear ticket on milestone M86.

## Implementation notes

- Probe #26's `label` branch routes a wrong-label scenario (e.g. stale `milestone-M30` present, `milestone-M86` absent) through the `"missing"` message kind rather than `"mismatch"` — correct-by-design for labels: there is no rename operation for labels (unlike the object branch's single named milestone object), so the fix for an absent `milestone-M86` is always "attach it" (re-run /implement Phase 1), independent of any stale label.
