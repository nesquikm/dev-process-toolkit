# STE-381: Deterministic branch-type derivation — `changelog_category`-keyed `branchTypeFor` replaces the LLM `{type}` pass {#STE-381}

## Requirement

The branch `type` in `buildBranchProposal` is a raw LLM-returned value: STE-64 § Design deliberately chose LLM inference for the chore-vs-feat call, and AC-STE-64.13's sanitizer merely clamps unknown values to `feat`. Nothing pins whether the type should reflect the spec commit (`chore(specs)`) or the eventual implementation (`feat`/`fix`), so kickoff branches flip between `chore/…` and `feat/…` across sessions for identically-shaped FRs — M100 got `chore/ste-373-…`, M102 got `feat/ste-378-…`, M103 got `feat/ste-380-…`.

Approved 2026-07-14 `/brainstorm`: replace the freestyle pass with a **shared deterministic rule** keyed on the FR's `changelog_category` frontmatter (STE-73, absent ⇒ default `Added`), with `--no-tech` forcing `chore` (a spec-only branch — implementation is far away). No new prompts: the existing `[Y / e / n]` branch gate stays the operator-override surface. Both `buildBranchProposal` consumers adopt the rule — `/spec-write` § 7a (via `branchNameFor`) and `/implement` 0.b″ — and `/spec-write` starts emitting `changelog_category` on every new FR so the rule's input is reliably present.

## Technical Design

**Helper (pure, adapter-layer).** `branchTypeFor` is a three-line decision function with an explicit precedence order:

1. `noTech === true` ⇒ `chore` — a `--no-tech` FR is spec-only at creation regardless of category; the technical reviewer's later `/spec-write <id>` auto-resume does not re-derive (the gate no-ops off-trunk).
2. `changelogCategory ∈ {Fixed, Security}` ⇒ `fix`.
3. everything else — `Added`, `Changed`, `Deprecated`, `Removed`, unknown strings, or absent (read-side default `Added` per STE-73) ⇒ `feat`.

The function never throws: category shape is validated at authoring time (closed-set choice in the draft preview), and defensive tolerance here keeps the branch gate non-blocking on legacy FRs with hand-edited frontmatter.

**Consumer wiring.** `/spec-write` § 7a: `branchNameFor`'s new-FR input gains the derived `type` (callers compute `branchTypeFor({ changelogCategory, noTech })` and pass it through — `SpecWriteBranchInput` shape unchanged, only the SKILL.md instruction for populating `type` changes from "LLM pass" to "the helper"). `/implement` 0.b″: SKILL.md instruction changes identically; the FR file is always resolved by 0.b′ before 0.b″ runs, so `changelog_category` is available. STE-227's 0.b′¹ refusal guarantees `needs_technical_review` FRs never reach 0.b″, so the `noTech` arm is `/spec-write`-only by construction.

**Frontmatter opt.** `BuildFRFrontmatterOpts.changelogCategory?: string` — emitted as `changelog_category: <value>` after `created_at:` (and after `needs_technical_review:` when both present, keeping key order stable). Omit-when-absent preserves byte-identity for existing tests and mode-none flows (STE-227 AC.2 emission convention applies to the opt plumbing; the /spec-write skill contract is what makes the field always-provided on new FRs).

**Unchanged surfaces.** `buildBranchProposal` signature + sanitizer; the `[Y / e / n]` gate flow and all `branch_gate_*` capability tokens; `TRUNK_OK_TYPES = ["ci"]`; the cross-cutting `docs/specs-cross-cutting` literal; Linear's `gitBranchName` (ignored per `feedback_branch_naming`).

## Notes

- Diverges deliberately from STE-227 AC.2's omit-when-default *skill-contract* convention — `changelog_category` is a content field (CHANGELOG grouping input + branch-type input) the drafting session must consciously pick; explicit emission makes the derivation auditable in the FR file. The *opt plumbing* still omits when unset, so the helper stays byte-compatible.
- Rejected at brainstorm (2026-07-14): run-shape-only rule (loses `fix/`); derive-once-persist (root LLM nondeterminism survives); asking the type at the gate (extra prompt against the zero-new-prompts constraint).
- Sibling of STE-380 in M103 "Signpost" — both are spec-write flow-determinism fixes; build order STE-380 → STE-381.

## Implementation notes

- `branch_type_for.test.ts` imports `skills/spec-write/branch_name_for` — an adapter-tree test reaching up into the skills layer; coverage is correct but reverses the usual skills→adapters dependency direction; candidate relocation of the wiring tests to `tests/` in a future sweep (TDD AUDIT advisory).
- `changelogCategory` typing widened to the literal-union house pattern (`"Added" | … | "Security" | string`, matching `version_bump.ts`) in `branch_type_for.ts` + `fr_frontmatter.ts` (Phase 3 Stage B Pass 2).
