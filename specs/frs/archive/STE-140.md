# STE-140: Delete drifted JSON schemas + stale validates-against doc claims {#STE-140}

## Requirement

`adapters/_shared/schemas/fr.schema.json` and `plan.schema.json` describe a pre-STE-76 / pre-archived-status world that no longer reflects the live frontmatter contract. Concretely:

- **`fr.schema.json`** declares `required: ["id", "title", "milestone", "status", "archived_at", "created_at"]`. Tracker-mode FRs have no `id:` line post-STE-76 (M21); `/gate-check` probe #13 (`identity_mode_conditional`, error severity post-STE-110) actively forbids it. **121 of 153** archived FRs would fail the schema.
- **`plan.schema.json`** declares `status: enum: ["draft", "active", "complete"]` — does NOT include `"archived"`. All 37 archived plans carry `status: archived` → 37 of 37 plans would fail. Schema also `required: revision`, but `revision:` was dropped post-M14; only 14 of 37 plans carry the field.

`adapters/_shared/schemas/schemas.test.ts` only tests the schemas' self-structure (regex patterns, required-field count). It does NOT validate any real FR/plan against the schema. Schema content can be arbitrarily wrong and tests stay green.

`docs/layout-reference.md:30,37` claims "Frontmatter is Schema Q (validates against `adapters/_shared/schemas/fr.schema.json`)" — neither claim is true today.

`grep -rn "ajv\|json-schema" plugins/` returns zero hits. No code loads or validates against these schemas. They are decorative.

Per the no-users-yet principle: subtract rather than fix. Delete the schemas + the test that asserts their self-structure + the doc claims that lie. If schema validation matters in the future, write fresh schemas + a real conformance test then.

## Technical Design

This is a deletion-only change. Three files removed, one doc edit.

**Step 1.** Verify zero runtime consumers:

```sh
grep -rn "fr\.schema\.json\|plan\.schema\.json" plugins/ --include="*.ts" --include="*.md"
```

The audit confirmed zero `ajv` / `json-schema` imports. Re-confirm at implementation time.

**Step 2.** Delete the three files:

```
plugins/dev-process-toolkit/adapters/_shared/schemas/fr.schema.json
plugins/dev-process-toolkit/adapters/_shared/schemas/plan.schema.json
plugins/dev-process-toolkit/adapters/_shared/schemas/schemas.test.ts
```

**Step 3.** Investigate `lock.schema.json`. If it has a consumer, leave it. If not, delete. Document the result.

**Step 4.** Edit `docs/layout-reference.md`:

- Line 30 (FR frontmatter section): remove or rewrite the "Schema Q (validates against …)" claim.
- Line 37 (plan frontmatter section): same treatment.

The cleanest replacement points to the live enforcement: probe #13 (`identity_mode_conditional`) for FR shape, probe #27 (`frontmatter_milestone_not_archived`) for plan shape, and the `buildFRFrontmatter` helper as the source of truth for emission.

**Step 5.** Verify `bun test` green and `/gate-check` clean.

## Testing

- **`bun test`** — passes after the deletions (the only test removed is `schemas.test.ts` itself; no other test imports it).
- **`/gate-check`** — clean across the live repo.
- **Manual** — `grep -rn` confirms no surviving references.

No new tests; this is a subtraction.

## Notes

The audit's alternative path was "update `fr.schema.json` to make `id` mode-conditional via `oneOf`, update `plan.schema.json` to add `archived` to enum and drop `revision` from required, then add a real `tests/schema-conformance.test.ts` that validates every active + archived FR/plan via ajv." That path costs significantly more and only matters if the schemas are intended to become an authoritative contributor surface. Today they aren't, and the live `/gate-check` probes are the source-of-truth contract — they're in regular use, get tested against the live repo (after STE-139's self-check backfill), and align with the actual frontmatter shape.

Subtraction is the right move under the no-users principle. If a schema-driven validation contract matters in M40+, the right path is greenfield (write the schemas to the THEN-current shape, with conformance tests that validate the live tree from day one).
