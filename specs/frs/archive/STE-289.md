# STE-289: Hooks bundling redo — ship plugin-bundled hooks/hooks.json; rip out /setup --hooks installer {#STE-289}

## Requirement

`/setup --hooks` (M71/STE-285) writes 4 Process-category hook entries into a user project's `.claude/settings.json`, with `args[0]` rendered as either the dev-clone absolute path (pre-STE-288) or the literal `${CLAUDE_PLUGIN_ROOT}/templates/hooks/process/<name>.sh` token (STE-288, v2.22.1). Empirically, neither shape fires: the Claude Code harness only expands `${CLAUDE_PLUGIN_ROOT}` in plugin-bundled `<plugin>/hooks/hooks.json`, not in user `.claude/settings.json`; the absolute-path shape pointed at the dev-clone path which doesn't exist on a marketplace user's machine (the harness loads the plugin from `~/.claude/plugins/cache/<plugin>/<version>/`, not the install-time path).

2026-05-14 discovery: operator ran `/dev-process-toolkit:setup --hooks=all` against v2.22.1 in `~/workspace/quack`, then a hook tried to fire; the harness emitted: `Hook command references ${CLAUDE_PLUGIN_ROOT} but the hook is not associated with a plugin. This variable is only available in hooks defined in a plugin's hooks/hooks.json file, not in [user settings.json]`. Empirical research via the `claude-code-guide` agent confirmed the contract: `${CLAUDE_PLUGIN_ROOT}` only expands in `<plugin>/hooks/hooks.json` (or `plugin.json` inline `hooks` field) — never in user `.claude/settings.json`. Marketplace plugins auto-discover their bundled `hooks/hooks.json` at session start and fire across every project where the plugin is enabled.

Forward-only: no migration logic for stale `.claude/settings.json` entries (zero shipped marketplace users per `project_no_users_yet`; the operator's downstream `~/workspace/quack` is cleaned by manual edit per release notes).

### How the harness loads it

Per the Claude Code plugins reference (`code.claude.com/docs/en/plugins-reference.md#hooks` + `#environment-variables`):

1. On `/plugin install` from a marketplace, Claude Code copies the plugin source into `~/.claude/plugins/cache/<marketplace>/<plugin>/<version>/`. For this repo, the cache flattens `plugins/dev-process-toolkit/` to the top level, so the bundled hook config lands at `~/.claude/plugins/cache/dev-process-toolkit/dev-process-toolkit/<version>/hooks/hooks.json`.
2. On session start, the harness auto-discovers each enabled plugin's `hooks/hooks.json` and registers the hooks against the matchers (event + tool/prompt pattern).
3. At hook-fire time, `${CLAUDE_PLUGIN_ROOT}` is substituted inline against the plugin's runtime path on the user's machine — the plugin cache directory, not the dev-clone path.
4. Plugin hooks fire in **every project** where the plugin is enabled (user scope). No per-project opt-in mechanism.

### Reversal of STE-262 / STE-270 / STE-276's rejection grounds

The /spec-research seed (2026-05-14) surfaced that STE-262/STE-270/STE-276 had explicitly rejected bundled `hooks/hooks.json` on three grounds: (1) spawn blast radius, (2) triple-check conflict, (3) no clean per-session state surface. STE-285 cited that chain (AC-STE-285.6) and chose the install-side layer instead. M74 reverses that direction. The reversal is empirically forced — the harness does not implement `${CLAUDE_PLUGIN_ROOT}` expansion in user `.claude/settings.json`, so the install-side approach STE-285 chose was structurally impossible from the start. The three rejection grounds remain valid concerns but are mitigated as follows:

- **Spawn blast radius:** the 4 hooks are PreToolUse:Bash + UserPromptSubmit:* — already-narrow matchers. The blast radius is the same as the install-side approach would have had if it worked; bundling doesn't widen it.
- **Triple-check conflict:** addressed by the per-script `session.sh::require_skill_tool_use` helper — each hook fail-opens when `$CLAUDE_SESSION_FILE` is unset (non-Claude commits) and reads the active session log to decide. Re-running across sessions doesn't cause false-positives.
- **No clean per-session state surface:** the 4 scripts use `$CLAUDE_SESSION_FILE` (provided by the harness per session). They were already session-aware under the install-side approach; bundling preserves that invariant.
