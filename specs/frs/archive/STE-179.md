# STE-179: /spec-write commits its specs/ outputs (close untracked-bootstrap drift) {#STE-179}

## Requirement

The 2026-05-01 Jira smoke run (v2.3.0) shipped `/setup → /spec-write → /implement → /gate-check` end-to-end against `dpt-test-project-jira`. After the chain completed, `git status --porcelain` reported the following files as `??`:

```
?? CLAUDE.md  (only in some runs — bootstrap commit caught this on Linear path)
?? specs/frs/DST-6.md
?? specs/technical-spec.md
?? specs/testing-spec.md
?? specs/plan/M1.md
```

Gate-check probe #22 (`setup-bootstrap-committed`, shipped under STE-109 AC-STE-109.7) caught the drift but its scope is too narrow: it asserts that toolkit-managed CLAUDE.md is committed, but does not enforce the same invariant for the spec files that `/spec-write` writes after the bootstrap. `/setup` produces the bootstrap commit (STE-109); `/implement` produces the feature commit (Phase 4 step 15); `/spec-write` is the silent middle skill that writes 4+ specs files and never owns a commit for them.

The gap matters because production users running the canonical chain on a fresh repo will discover their `specs/` tree is untracked after their first feature ships. The smoke driver caught this; a normal user might not until much later, by which point the spec files have evolved through several `/spec-write` iterations and reconstructing intent from the working-tree-only state is hard.

The fix follows the STE-109 precedent: each toolkit skill owns the commit for the files it writes. `/setup` commits its bootstrap. `/implement` commits its feature. `/spec-write` commits its specs.

Out of scope:
- Folding spec changes into `/implement`'s feature commit. That alternative was considered in the FR draft and rejected because it conflates spec-authoring with code-shipping (each skill is supposed to own one diff shape); STE-109 set the per-skill-commits precedent for `/setup` and we extend it here.
- Renaming probe #22's source file (`adapters/_shared/src/setup_bootstrap_committed.ts`) — only the probe's name and scan-set widen, not its filesystem location. Source-file renaming is a separate concern that we'd carry only if the probe expanded into multi-file scope.
- Changing the `chore(specs):` commit subject convention. Mirrors `chore(skills/setup): bootstrap` from STE-109.

### Commit subject semantics

The subject distinguishes new-FR runs from cross-cutting-only runs:

- New-FR run (Step 0b created an FR file): `chore(specs): write FR <tracker-id>` (e.g., `chore(specs): write FR STE-179`). Follows Conventional Commits — `chore` because no shipped behavior changes, only spec authorship.
- Cross-cutting-only (existing FR edits + technical/testing/plan edits, no new FR): `docs(specs): edit cross-cutting specs`. The `docs` type is appropriate: cross-cutting specs are documentation of system-wide concerns, not new features.
- Hybrid (new FR + cross-cutting edit in the same run): subject is the new-FR shape; cross-cutting edits land in the same commit as part of the FR's bootstrap.

### Default-apply UX (Auto / `-p`)

The `y / n / edit` prompt fires under interactive runs. Auto and `-p` modes auto-apply `y` and emit a single advisory row in the Step 7 summary so the operator sees, on every quiet-mode run, that the auto-apply happened. This mirrors the STE-109 precedent for `/setup`'s bootstrap.

### Probe rename impact

The renamed probe `toolkit-bootstrap-committed` replaces `setup-bootstrap-committed`. There is no aliasing; the old name is dropped per "no users yet" memory (project_no_users_yet.md). Any internal references to `setup-bootstrap-committed` in test fixtures, conformance-probe-list snapshots, or `docs/` get the new name. Migration is a single rename pass; the only behavioral delta is the widened scan-set.

## Notes

- **STE-109 precedent.** STE-109 (archived) shipped `/setup`'s bootstrap commit + the original probe #22. This FR extends both contracts to cover `/spec-write`'s output domain. The pattern is intentional: each skill owns its commit; `/gate-check` enforces no-untracked-toolkit-files at the seam.
- **Why not fold into `/implement`'s feature commit.** Considered and rejected. Folding would simplify git history (1 commit instead of 2) but conflates spec-authoring with code-shipping, breaking the per-skill-commit pattern STE-109 established. It would also obscure spec evolution: `/spec-write` is iterative (multiple invocations as drafts get refined), and folding into `/implement` means specs only commit once code is ready, defeating the iterative-drafting workflow.
- **Smoke-test source.** `/tmp/dpt-smoke-findings-2026-05-01-jira.md` § F1 logged this. The Linear smoke run on the same date had a clean `bdb1d98` bootstrap commit, but the spec files written by `/spec-write` were not in that commit — the Linear path also had untracked spec files; they just didn't trip the existing probe #22 because CLAUDE.md was committed. Re-confirms the gap is path-agnostic.
- **Scope boundary with STE-181.** STE-181 (invocation-grammar clarification) does not interact with this FR's commit semantics; STE-181 changes prose, STE-179 adds a commit. Both can land in M51 in any order.

## Implementation notes

- `toolkit_bootstrap_committed.ts` `listMdFiles` — TOCTOU window between `readdirSync` and per-file `statSync` — Existing inner try/catch on `statSync` prevents crashes; a file deleted between calls falls out of the scan-set silently. Bun/Node idiomatic fix is `readdirSync(dir, { withFileTypes: true })` (one syscall). Defer until the false-vacuous-pass risk surfaces in a real smoke run.
