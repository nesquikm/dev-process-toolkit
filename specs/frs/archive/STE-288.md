# STE-288: Hook install path portability — emit literal ${CLAUDE_PLUGIN_ROOT} token {#STE-288}

## Requirement

`/setup --hooks` (STE-285, M71) writes hook entries into `<project>/.claude/settings.json` whose `args[0]` should contain the literal string `${CLAUDE_PLUGIN_ROOT}/templates/hooks/process/<name>.sh`. The Claude Code harness expands `${CLAUDE_PLUGIN_ROOT}` at hook-fire time against the runtime plugin path, so the same `settings.json` works across dev clones, marketplace installs, and machine-to-machine transfers (the design intent documented in `/setup` SKILL.md line 166 + `docs/hooks-reference.md` line 9 per STE-285 AC-STE-285.3).

The current implementation at `plugins/dev-process-toolkit/skills/setup/install_hooks.ts:204` uses a JavaScript template literal — `args: [\`${pluginRoot}/templates/hooks/process/${name}.sh\`]` — interpolating the `pluginRoot` argument at install time. The result: an absolute path like `/Users/ns/workspace/dev-process-toolkit/plugins/dev-process-toolkit/templates/hooks/process/pre-commit-gate-check.sh` lands in JSON. Marketplace-install users whose plugin lives at `~/.claude/plugins/marketplaces/nesquikm/dev-process-toolkit/plugins/dev-process-toolkit/` cannot resolve that path; the hook either errors or silently fail-opens via the `[ -f $file ]` guard in `_lib/session.sh`.

Fixture group 8 (STE-286, M72) does not surface the bug because in-tree smoke runs resolve `${CLAUDE_PLUGIN_ROOT}` to the same absolute path regardless of which interpretation `install_hooks.ts` uses.

Per the brainstorm (2026-05-13), the plugin has zero shipped marketplace users so no migration path is required — the fix is forward-only.

## Acceptance Criteria

- AC-STE-288.1: `additionFor(name, pluginRoot)` in `install_hooks.ts` emits `args: ["${CLAUDE_PLUGIN_ROOT}/templates/hooks/process/<name>.sh"]` as a literal string (no JavaScript template-literal interpolation of `pluginRoot`). The `pluginRoot` parameter remains on the function signature for backward source-compat with callers but is unused on the write path.
  - verify: `bun test plugins/dev-process-toolkit/skills/setup/__tests__/hooks_merge_settings.test.ts` PASSES with a new test asserting `additionFor("pre-commit-gate-check", "/any/path").hook.args[0] === "${CLAUDE_PLUGIN_ROOT}/templates/hooks/process/pre-commit-gate-check.sh"` (literal string, no path interpolation).

- AC-STE-288.2: `readInstalledHookNames(settingsPath, pluginRoot)` in `install_hooks.ts` matches the literal-token prefix `${CLAUDE_PLUGIN_ROOT}/templates/hooks/process/` when scanning installed hook entries. No legacy absolute-path detection (the plugin has zero shipped users per `project_no_users_yet`).
  - verify: `bun test plugins/dev-process-toolkit/skills/setup/__tests__/setup_hooks_flag.test.ts` continues to PASS; the existing test fixture's `args[0]` uses the literal token form. A new test asserts that an entry whose `args[0]` is a dev-clone absolute path (legacy shape) is **not** detected as an installed plugin hook.

- AC-STE-288.3: Unit test coverage — a dedicated test in `plugins/dev-process-toolkit/skills/setup/__tests__/hooks_merge_settings.test.ts` (or `hooks_path_portability.test.ts` if separated) asserts the literal-token shape across all 4 seeded hook registrations. Test runs both `additionFor` and a full `installHooks` round-trip + JSON re-read to confirm `args[0]` survives the write→read cycle byte-identical.
  - verify: `bun test plugins/dev-process-toolkit/skills/setup/__tests__/hooks_merge_settings.test.ts` PASSES.

- AC-STE-288.4: Fixture group 8 install-verification (`.claude/skills/smoke-test/SKILL.md`) is extended to grep the test-project's `.claude/settings.json` for the literal token shape: `grep -c -F '${CLAUDE_PLUGIN_ROOT}/templates/hooks/process/' <test-project>/.claude/settings.json` ≥ 4 (one entry per seeded hook). The new assertion lands alongside the existing `"matcher": "Bash"` / `"matcher": "*"` grep tokens documented in the fixture's install-verification subsection.
  - verify: `bun test plugins/dev-process-toolkit/skills/setup/__tests__/smoke_fixture_group_8_doc_conformance.test.ts` PASSES with a new doc-conformance test asserting fixture group 8 prose contains both the literal `${CLAUDE_PLUGIN_ROOT}/templates/hooks/process/` grep needle and the `≥ 4` assertion shape (one per seeded hook).

The dev clone's own `.claude/settings.json` may carry stale absolute-path entries from prior `/setup --hooks` runs; the operator clears them by manually editing or re-running `/setup --hooks` (which writes the new shape).
  - verify: no fixture explicitly tests legacy migration; this AC documents the *absence* of migration logic. Prose check in the FR Notes.

### Implementation shape

Current (`install_hooks.ts:204`):

```ts
args: [`${pluginRoot}/templates/hooks/process/${name}.sh`],
```

Fixed:

```ts
args: [`\${CLAUDE_PLUGIN_ROOT}/templates/hooks/process/${name}.sh`],
```

The `\${CLAUDE_PLUGIN_ROOT}` escape in the template literal emits the literal string `${CLAUDE_PLUGIN_ROOT}` into the JSON — only `${name}` interpolates at JavaScript level (the hook name is per-registration metadata, not a runtime path).

Symmetric fix in `readInstalledHookNames` (`install_hooks.ts:321`):

```ts
const prefix = `\${CLAUDE_PLUGIN_ROOT}/templates/hooks/process/`;
```

### Risks (testing-side)

- The literal `${CLAUDE_PLUGIN_ROOT}` token in TypeScript template literals requires careful escaping (`\${CLAUDE_PLUGIN_ROOT}`). A test that asserts `args[0] === "${CLAUDE_PLUGIN_ROOT}/..."` must use the same escape in the expected string. The grep-token in fixture group 8 prose lives inside a Markdown code fence and is naturally literal.
