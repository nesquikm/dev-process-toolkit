# STE-183: Drop or qualify (v0.1.0) suffix in /setup bootstrap commit subject {#STE-183}

## Requirement

`/setup`'s bootstrap commit (shipped under STE-109) currently uses the subject `chore: bootstrap dev-process-toolkit (v0.1.0)`. The Linear smoke run on 2026-05-01 (v2.3.0 plugin) caught the issue: the `(v0.1.0)` suffix references the project's `package.json` version (the default that `bun init` writes), not the toolkit version that produced the scaffold.

A future operator walking the test-project's git log sees `bdb1d98 chore: bootstrap dev-process-toolkit (v0.1.0)` and has no signal that the scaffold came from toolkit v2.3.0. The information they want — "which version of the toolkit shipped this scaffold" — is missing; the information they don't want (the project's own boilerplate version) is foregrounded.

The fix is two-part:

1. **Drop the `(v0.1.0)` suffix from the subject.** The project's `package.json` version is not the right thing to advertise in the bootstrap subject; it conveys no useful signal.
2. **Add a `Toolkit:` footer** to the commit body, of the form `Toolkit: dev-process-toolkit v<plugin-version>`, where `<plugin-version>` is read from `plugins/dev-process-toolkit/.claude-plugin/plugin.json`. This is the version operators want to see; placing it in the footer keeps the subject clean while preserving the audit trail.

Out of scope:
- Changing other commit subjects emitted by toolkit skills (`/implement` feature commits, `/ship-milestone` release commits, etc.). Only `/setup`'s bootstrap commit is in scope.
- Adding the toolkit version to the subject itself (e.g., `chore: bootstrap dev-process-toolkit v2.3.0`). Considered and rejected: the subject is a one-line summary; the version belongs in the footer per Conventional Commits convention (footers carry metadata).
- Changing the plugin version that is read. The source of truth is `plugins/dev-process-toolkit/.claude-plugin/plugin.json` `version` field — the same field `/ship-milestone`'s release-files schema bumps. Reusing the existing field avoids divergence.

## Acceptance Criteria

- AC-STE-183.1: `/setup`'s bootstrap commit subject changes from `chore: bootstrap dev-process-toolkit (v0.1.0)` to `chore: bootstrap dev-process-toolkit`. The trailing version suffix is removed entirely (no replacement). The Conventional Commits subject convention is preserved (`<type>: <description>`, ≤72 chars). {#AC-STE-183.1}
- AC-STE-183.2: `/setup`'s bootstrap commit body gains a single new footer line: `Toolkit: dev-process-toolkit v<plugin-version>`. Reading the plugin version: load `plugins/dev-process-toolkit/.claude-plugin/plugin.json`, parse JSON, return the `version` field. The footer is appended after any existing body content (description, refs, etc.). {#AC-STE-183.2}
- AC-STE-183.3: The plugin-version read is fault-tolerant: if `plugin.json` is missing or unparseable (the operator deleted it, the file is corrupt, etc.), `/setup` skips the footer entirely (no malformed `Toolkit: dev-process-toolkit v<unknown>` line) and emits an advisory in the Step 7 (or whichever post-condition `/setup` uses) summary block. The skill does not refuse the commit on a missing plugin version — the bootstrap commit's load-bearing role takes priority over the audit footer. {#AC-STE-183.3}
- AC-STE-183.4: New conformance probe `setup-bootstrap-commit-subject` verifies the subject shape. Probe scans the most recent `chore: bootstrap dev-process-toolkit*` commit on the current branch (if any) and asserts: (a) subject is exactly `chore: bootstrap dev-process-toolkit` (no parenthesized suffix); (b) body contains a single line matching the regex `^Toolkit: dev-process-toolkit v\d+\.\d+\.\d+$` (or no `Toolkit:` line if the commit predates this FR — backwards-compat carve-out detected via commit author date < the FR ship date). The probe is vacuous on repos with no `chore: bootstrap dev-process-toolkit*` commit. {#AC-STE-183.4}
- AC-STE-183.5: Integration test `tests/setup-bootstrap-commit-subject.test.ts` covers:
  - (a) Fresh `/setup` run produces the new subject and footer; both shapes assert clean.
  - (b) `plugin.json` missing → footer absent, subject still clean, advisory in Step 7 summary.
  - (c) `plugin.json` corrupt JSON → same as (b).
  - (d) Subject regression to `(v0.1.0)` shape → probe fails. {#AC-STE-183.5}
- AC-STE-183.6: Smoke-test driver SKILL.md (or its post-condition assertions) updates only if it currently asserts the old subject shape. If it does, the assertion is updated to match the new shape. If it doesn't, no smoke-driver edit. {#AC-STE-183.6}

### Plugin-version read

```ts
function readPluginVersion(repoRoot: string): string | null {
  const path = join(repoRoot, "plugins/dev-process-toolkit/.claude-plugin/plugin.json");
  if (!existsSync(path)) return null;
  try {
    return JSON.parse(readFileSync(path, "utf8")).version ?? null;
  } catch {
    return null;
  }
}
```

The function returns `null` on any failure path (missing, unparseable, missing version field). The skill prose at `setup/SKILL.md` then conditionally emits the footer based on the return value.

### Footer placement

Conventional Commits places footers after the body. The bootstrap commit's body is currently something like:

```
chore: bootstrap dev-process-toolkit

Initial scaffold by `/setup` from dev-process-toolkit toolkit.

Refs: STE-109
```

After this FR, the footer block grows by one line:

```
chore: bootstrap dev-process-toolkit

Initial scaffold by `/setup` from dev-process-toolkit toolkit.

Refs: STE-109
Toolkit: dev-process-toolkit v2.3.1
```

The `Toolkit:` line sits below `Refs:` (Conventional Commits orders footers alphabetically by convention but no canonical order — the FR fixes one shape for predictability).

### Probe semantics

`setup-bootstrap-commit-subject` is a git-log scanner. It runs `git log --format=%H%x00%s%x00%b --grep="^chore: bootstrap dev-process-toolkit"` and parses the most recent match. The backwards-compat carve-out (AC-183.4 (b) clause) compares the commit author date to a hard-coded FR ship date constant; commits before the constant are considered legacy and not asserted.

## Notes

- **STE-109 precedent.** STE-109 (archived) shipped the bootstrap commit infrastructure including the original subject literal. This FR is a polish on the literal — STE-109's load-bearing contract (the commit happens, files land in git) is unchanged.
- **Why drop the suffix entirely vs. just qualify it.** Considered three options: (a) drop the suffix entirely; (b) replace `(v0.1.0)` with `(toolkit v2.3.1)`; (c) keep both: `(project v0.1.0, toolkit v2.3.1)`. Picked (a) because the subject should be a one-line summary; metadata belongs in footers. (b) and (c) clutter the subject and bump the char count. The footer captures the toolkit version cleanly.
- **Why `Toolkit:` not `Toolkit-Version:`.** Shorter, parses identically as a Conventional Commits footer (any `^[A-Z][A-Za-z-]+: ` line is a footer), and reads naturally. The token after the colon is the toolkit's package.json-style version string — `dev-process-toolkit v2.3.1`.
- **Smoke-test source.** `/tmp/dpt-smoke-findings-2026-05-01-linear.md` § F4 logged this. The Jira smoke didn't flag it because the smoke driver doesn't currently assert subject shape; this FR adds the conformance probe so future runs catch any subject regression.
- **Interaction with STE-179.** STE-179 renames probe #22 (`setup-bootstrap-committed` → `toolkit-bootstrap-committed`). This FR adds a different probe (`setup-bootstrap-commit-subject`). The two probes are complementary: one asserts files are committed, the other asserts the commit's subject shape. No conflict.
