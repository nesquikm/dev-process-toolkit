# STE-190: Jira pull_acs ADF-escape tolerance for checkbox bullets {#STE-190}

## Requirement

The 2026-05-01 Jira run 2 documented an ADF round-trip artifact. When the description body containing AC checkbox bullets like `- [ ] AC 1: ...` is sent through `mcp__atlassian__createJiraIssue` (markdown → ADF conversion) and re-read via `mcp__atlassian__getJiraIssue`, the result string contains escaped `\[x\]` / `\[ \]` rather than `[x]` / `[ ]`:

```
"* \\[x\\] AC 1: `greet('Alice')` returns the string `Hello, Alice!`."
```

Renders correctly as checkboxes in the Jira UI, but raw markdown tooling sees the escaped form. The Jira adapter's `pull_acs` parser empirically tolerates this — `/implement`'s AC-toggle round-trip flipped all four ACs end-to-end on the smoke run. So the behavior is correct **today**, but the tolerance is implicit (likely a side effect of the parser's existing regex tolerating `\\` characters), not explicitly tested.

This FR makes the tolerance explicit:

1. **Adapter doc.** `plugins/dev-process-toolkit/adapters/jira.md` § `pull_acs` adds a paragraph documenting the ADF round-trip artifact and the tolerance contract.
2. **Unit test.** `plugins/dev-process-toolkit/adapters/_shared/src/` (or wherever the existing `pull_acs` tests live) gains an explicit case feeding the escaped form `\[x\] AC N: ...` into `pull_acs` and asserting it parses to the same `{ id: "AC-...", text: "...", checked: true }` shape as the unescaped form.

The motivation is regression prevention: a future tweak to `pull_acs` (e.g., a regex tightening) that drops the escape tolerance would break the live Jira round-trip silently — `/implement` would produce ACs that read `[x]` locally but fail to round-trip back from Jira. The explicit test catches this at `bun test` time.

Out of scope:

- Suppressing the ADF escaping at the source (`createJiraIssue` markdown → ADF conversion). The escape is server-side; the toolkit cannot suppress it without escaping each AC bullet's own ticket id.
- Linear adapter `pull_acs`. Linear has its own round-trip artifact (server-side `<issue id="...">` mention markers — re-confirmed in smoke #9 F2) but that's already covered by the Provider's normalization layer; not in M52 scope.
- The push side (`push_ac_toggle`). The push direction writes plain `[x]` / `[ ]`; ADF re-renders on the server; the parser tolerance covers the read-back.

### Test shape

```ts
describe("pull_acs ADF-escape tolerance", () => {
  it("parses ADF-escaped \\[x\\] as checked", () => {
    const adfEscaped = "* \\[x\\] AC 1: `greet('Alice')` returns `Hello, Alice!`.";
    const expected = { id: "AC-DST-7.1", text: "...", checked: true };
    expect(pullAcs(adfEscaped)).toContain(expected);
  });
  it("parses ADF-escaped \\[ \\] as unchecked", () => {
    const adfEscaped = "* \\[ \\] AC 2: `greet()` returns `Hello, world!`.";
    const expected = { id: "AC-DST-7.2", text: "...", checked: false };
    expect(pullAcs(adfEscaped)).toContain(expected);
  });
});
```

(Exact test shape depends on the existing pull_acs API; the test author should match the existing test conventions in the same file.)

### Why a unit test, not an integration test

A live Jira round-trip integration test would require Atlassian Rovo MCP credentials in CI — not currently configured for the toolkit's `bun test` baseline. The unit test feeds the empirically-observed escaped form directly into the parser; this is sufficient to catch a future regression. The smoke chain remains the integration-level check (and AC-STE-190.4 implicitly guarantees it via the existing smoke discipline).

## Testing

- `bun test` baseline: pre-FR count → post-FR count + at least 2.
- Smoke chain Jira on the next `/smoke-test` run validates the live round-trip. AC-STE-190 explicitly does not depend on a smoke re-run for `[x]` shipping; the unit test is sufficient.

## Notes

- **Smoke-test source.** `/tmp/dpt-smoke-findings-2026-05-01-jira.md` § Run 2 § F3 documented the artifact and the suggested follow-up. Severity was low (no behavior regression — `/implement`'s AC-toggle round-trip worked), but the unit-test coverage is cheap insurance.
- **Parser-fix unlikely.** Smoke evidence: 4/4 ACs flipped end-to-end on the live Jira round-trip in run 2 — implies the parser already tolerates the escape. AC-STE-190.4 is the safety net for the edge case that the smoke happened to hit without the parser actually being escape-tolerant.
- **Future ADF-escape variants.** ADF re-rendering can also produce other escape artifacts (escaped backticks in inline code, escaped underscores in identifiers). Out of scope for this FR — the test covers the specific `\[x\]` / `\[ \]` round-trip the smoke run hit. A broader "ADF escape tolerance audit" could be a future M53+ FR if more variants surface.
