## Requirement

M20's STE-72 ships mechanical signature extraction for TypeScript only (`typedoc → ts-morph → regex-fallback`). Dart/Flutter projects fall through to `regex-fallback` with a "manual review advised" banner. Since Flutter is a primary target stack, this is a felt parity gap.

This FR adds a Dart strategy to `signature_extractor.ts` that uses `package:analyzer` (canonical Dart AST library) via a small bundled Dart helper spawned as a subprocess. The helper walks `lib/**/*.dart`, enumerates public exports, and emits JSON matching the `ModuleSignatures[]` shape; the TypeScript caller parses and returns it.

**Verified during /spec-write (2026-04-25, Dart SDK 3.11.4):** `dart doc --json` does NOT exist. `dart doc` emits HTML to `doc/api/` only. Available flags: `--output`, `--validate-links`, `--dry-run`. No JSON output mode. `package:analyzer` is therefore the primary mechanical strategy for Dart, not a fallback below an external CLI. Two-tier chain: `dart-analyzer` (preferred) → `regex-fallback` (banner).

Strategy recorded as `strategy: "dart-analyzer"` in `SignatureGroundTruth.strategy` (Schema Z). The enum widening (declared as `export type Strategy` at `adapters/_shared/src/signature_extractor.ts:28`) lands in the cross-cutting sibling FR (STE-105).

## Technical Design

**Dart helper layout:**

```
plugins/dev-process-toolkit/adapters/_shared/dart/
├── pubspec.yaml              # analyzer dep, Dart SDK constraint
├── pubspec.lock              # pinned, committed
├── extract_signatures.dart   # the helper program (~80-120 LOC)
└── .gitignore                # ignores .dart_tool/, build/
```

**`extract_signatures.dart` outline:**

```dart
import 'dart:convert';
import 'dart:io';
import 'package:analyzer/dart/analysis/analysis_context_collection.dart';
import 'package:analyzer/dart/analysis/results.dart';
import 'package:analyzer/dart/element/element.dart';

void main(List<String> args) async {
  final projectRoot = args[0];
  final collection = AnalysisContextCollection(
    includedPaths: ['$projectRoot/lib'],
  );
  final modules = <Map<String, dynamic>>[];
  for (final ctx in collection.contexts) {
    for (final path in ctx.contextRoot.analyzedFiles().where((p) => p.endsWith('.dart'))) {
      final result = await ctx.currentSession.getResolvedUnit(path);
      if (result is! ResolvedUnitResult) continue;
      final exports = result.libraryElement.topLevelElements
          .where((e) => !e.isPrivate)
          .map((e) => exportSigJson(e, result, path))
          .toList();
      if (exports.isNotEmpty) {
        modules.add({'modulePath': path, 'exports': exports});
      }
    }
  }
  stdout.write(jsonEncode(modules));
}
```

**TypeScript wrapper pattern in `signature_extractor.ts`:**

```typescript
function extractViaDartAnalyzer(projectRoot: string, pluginRoot: string): ModuleSignatures[] {
  const helperDir = `${pluginRoot}/adapters/_shared/dart`;
  if (!existsSync(`${helperDir}/.dart_tool`)) {
    const pg = spawnSync("dart", ["pub", "get"], { cwd: helperDir, encoding: "utf-8" });
    if (pg.status !== 0) throw new DartAnalyzerError(`pub get failed: ${pg.stderr}`);
  }
  const result = spawnSync(
    "dart",
    ["run", "extract_signatures.dart", projectRoot],
    { cwd: helperDir, encoding: "utf-8", timeout: 60_000 },
  );
  if (result.status !== 0 || !result.stdout) {
    throw new DartAnalyzerError(`dart-analyzer failed: ${result.stderr}`);
  }
  return JSON.parse(result.stdout) as ModuleSignatures[];
}
```

**Why bundled helper instead of `dart pub global activate <our-pkg>`:** publishing to pub.dev is real infra (naming, versioning, ownership). A bundled helper ships locked to the plugin's release, no separate package to maintain. First-use cost: ~5-10s for `pub get`; cached after.

**Why this is allowed under the "no bundled AST helpers" memory constraint:** the constraint targeted re-implementing AST walking from scratch (the M20 ts-morph layer was bundled because Node→TS is free). The Dart helper just delegates to `package:analyzer` — analogous to the typedoc strategy delegating to a real tool. The Dart SDK + analyzer team maintain the AST logic; we just wire it. Memory's open-verification clause anticipated this exact path.

## Notes

**Why we ship a Dart helper at all (vs. just regex-fallback for Dart):** Flutter is a primary target stack for downstream toolkit users. Mechanical signature extraction is load-bearing for M20's NFR-22 anti-hallucination invariant. Falling through to regex would silently degrade Flutter docs vs. TS docs. ~50-120 LOC of Dart + one pubspec + one `pub get` on first run is small.

**Cross-helper asymmetry honest framing:** Python's strategy (STE-104) does NOT bundle a helper — `griffe dump` is a real CLI emitting JSON. The asymmetry is intrinsic: Dart has no JSON-emitting CLI for API surface; Python does. Approach 2's "no middle tier" applies per stack to the chain *shape* (preferred → regex, no intermediate AST layer), not to the implementation mechanism of the preferred tier.

**Risk: `package:analyzer` ABI churn.** The analyzer package versions semi-frequently (every ~2-3 months). Pinning `analyzer: ^7.0.0` provides minor-version flexibility. Major bumps would surface as `pub get` failures or import errors at first use; the regex-fallback branch handles this by emitting a warning. A future FR could add CI to track analyzer ABI compatibility.
