## Requirement

The Dart and Python strategy FRs (STE-103 + STE-104) add new `signature_extractor.ts` strategies (`"dart-analyzer"` and `"griffe"`). Three cross-cutting changes complete the parity:

1. **`/setup` packages-mode probe extension.** STE-68 AC-STE-68.1 prompt 2 (M20) currently surfaces `"Generate packages-style API reference docs? (typedoc <detected|not found>, ts-morph <bundled>, stack: <ts|other>)"`. Extend to detect Dart/Python toolchains and surface install commands inline when missing — so users can fix the gap themselves rather than silently falling through to regex.
2. **`/gate-check` strategy validation probe.** Add a new probe that asserts the recorded `SignatureGroundTruth.strategy` for the project's stack is one of the canonical mechanical strategies — not `regex-fallback` — when the preferred tool was detected at `/setup` time. Catches the case where `griffe` was installed at /setup but disappeared by /gate-check time (env drift).
3. **`SignatureGroundTruth.strategy` enum widening.** M20's Schema Z declares `strategy: "typedoc" | "ts-morph" | "regex-fallback"` (technical-spec.md line ~397; mirrored as `export type Strategy` at `adapters/_shared/src/signature_extractor.ts:28`). Widen to `strategy: "typedoc" | "ts-morph" | "dart-analyzer" | "griffe" | "regex-fallback"`. Backwards-compatible widening (existing TS callers unaffected). Update Schema Z prose in `specs/technical-spec.md` and the `Strategy` type in `signature_extractor.ts`.

This FR is the integration glue. Land last in the milestone (after STE-103 and STE-104).

## Technical Design

**Why a shared `toolchain_probe.ts` module:** both /setup (at decision time) and /gate-check (at validation time) need to answer the same question — "is the preferred tool present?" Co-locating the probe logic prevents drift between what /setup checks and what /gate-check verifies.

**Detection details per tool:**

| Tool | Detection | Detected if... |
|------|-----------|----------------|
| typedoc | `which typedoc` OR `node_modules/.bin/typedoc` | exit 0 OR file exists |
| ts-morph | bundled with plugin's adapter deps | always true (assert via `require.resolve`) |
| dart SDK | `which dart` AND `dart --version` | exit 0 AND parses version line |
| griffe | `which griffe` AND `griffe --version` | exit 0 AND parses version line |

**Why /gate-check probe is conditional, not unconditional:**

- `mode: none` projects without /setup record have no preference to validate against — skip cleanly.
- Projects whose /setup ran when toolchain was missing (and user opted into regex-fallback explicitly) have `signature_extraction_preferred_strategy: "regex-fallback"` recorded; no degradation possible.
- The probe only fires on the "I had it at setup, lost it later" drift case — which is the actual silent-failure mode worth catching.

**Schema migration concern (low):** existing /setup records (post-M20, pre-M27) won't have `signature_extraction_preferred_strategy`. The probe treats missing field as "no recorded preference" → skip. No migration script needed; the field appears naturally when /setup is re-run on existing projects (or when /gate-check warns about its absence for non-mode-none projects — TBD whether to warn or stay silent; default silent to avoid noise on pre-M27 projects).

## Notes

**Sequencing within M27:** this FR depends on STE-103 + STE-104 landing first (the enum widening at AC-STE-105.1 references `"dart-analyzer"` and `"griffe"` — exercising those values requires the strategies to exist). Land STE-103 + STE-104 in parallel as Phase A; STE-105 as Phase B.

**Honest limitation:** the `/gate-check` probe can detect the "tool was here, now isn't" case but can't detect the deeper failure where the tool IS present but produces wrong output. Mitigation lives in M20's existing post-generation validator (AC-STE-72.4) which validates the LLM's output against the ground truth — not the ground truth itself. Spec validation of the ground truth would require a separate FR (e.g., "compare extracted signatures against a hand-maintained golden file") and is out of scope.

**Why not auto-install missing tools:** /setup surfaces the install command but does not run it. Auto-install crosses a security boundary (running `pip install` or modifying user's Dart SDK without explicit consent). User's explicit action stays in the loop.
