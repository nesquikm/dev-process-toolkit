## Requirement

`mcp__linear__save_issue` is called by `/spec-write` and `/implement` to create or update tracker tickets. The Linear MCP requires `team` (missing → loud error) but accepts a missing `project` silently — tickets created without `project` land in the team's default view but outside the user's expected project board. Discovered 2026-04-27 during M30 spec-write: STE-115 and STE-116 created without project; user surfaced.

Schema L (the `## Task Tracking` block in `CLAUDE.md`) has a closed top-level key set per STE-114 (`mode`, `mcp_server`, `jira_ac_field`, `branch_template`). Tracker-specific workspace metadata (Linear team + project, Jira project, default labels) was explicitly carved out into "sub-sections" — but no sub-section was ever defined, no skill reads from one, and the canonical adapter operation `upsert_ticket_metadata(id?, title, description, assignee)` has no slot for forwarding project/team to the tracker.

This FR closes the loop: define the sub-section shape, widen the adapter contract, prompt for the values during `/setup`, thread them through `/spec-write` and `/implement`, and add a `/gate-check` probe that hard-fails tracker mode without a populated workspace binding.

## Technical Design

**Sub-section file shape (post-FR, in this repo's CLAUDE.md):**

```markdown
## Task Tracking

mode: linear
mcp_server: linear
jira_ac_field:

### Linear

team: STE
project: DPT — Dev Process Toolkit
default_labels: []
```

**Module layout:**

```
plugins/dev-process-toolkit/adapters/_shared/src/
├── workspace_binding.ts                          # readWorkspaceBinding()
├── workspace_binding.test.ts
├── task_tracking_workspace_binding_present.ts    # probe #25
├── task_tracking_workspace_binding_present.test.ts
└── tracker_provider.ts                           # UpsertMetadataInput widened
```

**`readWorkspaceBinding` implementation sketch:**

```typescript
export interface WorkspaceBinding {
  team?: string;
  project?: string;
  defaultLabels?: string[];
}

export function readWorkspaceBinding(
  claudeMdPath: string,
  adapterKey: "linear" | "jira",
): WorkspaceBinding {
  const md = readFileSync(claudeMdPath, "utf-8");
  const taskTrackingSection = extractSection(md, "## Task Tracking");
  if (!taskTrackingSection) return {}; // mode: none equivalent
  const subTitle = adapterKey === "linear" ? "### Linear" : "### Jira";
  const sub = extractSubSection(taskTrackingSection, subTitle);
  if (!sub) return {};
  return parseYamlLikeKeys(sub); // tolerant key:value parser, same as Schema L top-level
}
```

**Probe registration** — same pattern as STE-114's `task-tracking-canonical-keys` (probe #21). Probe-inventory row in `gate-check/SKILL.md`:

```
| 25 | task-tracking-workspace-binding-present | error | tracker mode | Reject tracker mode without a populated `### Linear` / `### Jira` sub-section. |
```

**Why required-on-create vs. silently-default-to-empty:** mirrors the FR-67 silent-no-op trap pattern. Optional Linear params let bad calls succeed, masking the failure mode at call time and surfacing it later as "ticket landed somewhere unexpected." Hard rejection at call time forces the user to fix the cause.

## Notes

**Sequencing constraint:** STE-118 (auto-attach milestone) reads the `project` value from this FR's sub-section. STE-117 must land first. STE-119 has no dependency on this FR.

**STE-114 carve-out compatibility:** STE-114's probe `task-tracking-canonical-keys` (#21) explicitly scopes to *top-level* keys under `## Task Tracking` (per CHANGELOG entry "the closed-set rule applies only to the top level"). The new sub-sections are below `### Linear` / `### Jira` headings, which probe #21 already treats as out-of-scope. No probe conflict.

**Out-of-tree adapter migration:** the contract change is breaking for any custom adapter that implements `upsertTicketMetadata`. Mitigation: new params are optional in the TypeScript signature (`team?`, `project?`), so adapters that don't read them compile unchanged. The `Adapter authors` callout in `docs/tracker-adapters.md` instructs out-of-tree authors to forward the values to their tracker even if not strictly needed, to keep the Provider canonical.

**Em-dash hazard:** project names like `DPT — Dev Process Toolkit` carry a real em-dash (U+2014). The `readWorkspaceBinding` parser MUST treat the value as opaque UTF-8 — no normalization, no entity decoding. Adapters forwarding to MCP MUST pass the byte-identical string. STE-103's M27 milestone (`M27 — Dart/Python docs parity`) is the existing precedent that round-trips correctly through `mcp__linear__save_issue`.

**Telemetry / audit trail:** the `/setup audit` log entry is the durable record of which team/project the user picked. If the project moves later, an operator edits CLAUDE.md directly + re-runs `/gate-check` to re-validate; no re-run of `/setup` is required.
