# STE-127: /spec-write emits a final summary line on completion {#STE-127}

## Requirement

`/spec-write` must emit a final-line report to stdout when it completes successfully, naming the FR id, the FR file path, and the milestone updated. This mirrors the per-skill console-status contract that `/setup`, `/implement`, `/gate-check`, `/spec-review`, and `/simplify` all honor.

Today, `/spec-write` returns 1 byte of output (a single newline) on success — verified by smoke-test run #4. The skill performs all the work correctly (FR file created, plan updated, Linear ticket created) but the operator and any driver get zero visible signal of what happened. They must inspect the filesystem and Linear to learn the FR id.

**Smoke-test evidence.** Run #4 (2026-04-27, v1.34.0). `/spec-write` child log was 1 byte (`xxd` → single `0x0a`). Every other skill in the chain produced 476–7597 byte summaries. Full report: `/tmp/dpt-smoke-findings-2026-04-27.md` § F1.

## Acceptance Criteria

- AC-STE-127.1: When `/spec-write` finishes successfully, stdout contains a final-line report with: (a) the FR id (real tracker ID or short-ULID), (b) the FR file path relative to project root (`specs/frs/...`), (c) the milestone the FR was added to. Format: free-form Markdown but matching the visual shape of `/setup`'s and `/implement`'s closing reports.
- AC-STE-127.2: Successful runs produce stdout >= 100 bytes (the report itself, not counting any prior output). Verified by `wc -c` on the smoke-test log.
- AC-STE-127.3: The report is emitted regardless of mode (`linear`, `jira`, `none`) and adapts to the binding (e.g., `mode: none` shows the short-ULID; tracker mode shows the tracker ID).
- AC-STE-127.4: The report fires for both new-FR creation AND for the import path (`importFromTracker` after a tracker-ID resolve hits a non-local ticket).

## Technical Design

- Hook point: tail of `skills/spec-write/SKILL.md` Step 7 (Report) — currently the section says "Summarize what was completed" but the agent isn't producing the summary in the `claude -p` non-interactive case.
- Likely cause: the skill's Step 7 instructions are advisory ("summarize") and the model in `-p` mode skips the summary because there's no apparent "user" to address. Make the summary requirement explicit and unconditional.
- Reference shape (concrete): mimic `/implement`'s closing table layout, scaled down to the spec-write surface (one row per spec file touched + one row per FR created).

## Testing

- Manual probe via `/smoke-test` re-run after the fix: assert `wc -c /tmp/dpt-smoke-spec-write.log >= 100` and assert the FR id appears in the log.
- No new automated test scaffolding — the existing smoke-test driver is the regression guard.

## Notes

- Severity: medium. Doesn't block correctness; is a UX consistency regression. Cheap fix (one section in `skills/spec-write/SKILL.md`), but it's a blocker for any future autonomous `/smoke-test` driver that tries to chain `/spec-write` → `/implement` without filesystem inspection (driver caveat D1 in the smoke findings).
