# STE-151: /gate-check — relax probe #14 to exempt single-FR-clean state {#STE-151}

## Requirement

When `/dev-process-toolkit:implement <FR-id>` lands cleanly against a single FR mid-milestone, Phase 4 Close calls `Provider.releaseLock` and the Linear ticket transitions to **Done**, but `specs/frs/<FR-id>.md` intentionally stays at `status: active` per the milestone-bulk-archive design (`skills/implement/SKILL.md:216`). `/gate-check` probe #14 then asserts every active FR with a tracker binding must show `status == in_progress AND assignee == currentUser`, and the just-released ticket fails — producing a spurious **GATE FAILED** on the canonical `/implement <FR>` → `/gate-check` chain. Probe #14 must recognize the single-FR-clean state and skip its assertion when the FR's milestone plan still has unchecked tasks. The probe must continue to fire for the original M23 drift cases it was authored to catch (Phase 1 step 0.c skipped, wrong assignee) and for the "forgot bulk archive before /ship-milestone" regression case.

## Acceptance Criteria

- AC-STE-151.1: After `/dev-process-toolkit:implement <single-FR>` lands cleanly on a milestone whose `specs/plan/M<N>.md` still has at least one unchecked task line, `/dev-process-toolkit:gate-check` exits 0 with no probe-#14 fires for that FR — even though the ticket is `Done` and the FR is still `status: active`.
- AC-STE-151.2: Probe #14 still fires (and `/gate-check` still fails) when the FR is `status: active` AND its milestone plan has every task line checked AND the plan itself is still `status: active` — the "forgot bulk archive before `/ship-milestone`" regression case.
- AC-STE-151.3: Probe #14 still fires for active FR + `Backlog` ticket (Phase 1 step 0.c skipped) and active FR + `In Progress` ticket assigned to someone other than `currentUser` — the original M23 drift cases stay caught.
- AC-STE-151.4: `mode: none` semantics unchanged — probe #14 remains skipped (LocalProvider returns the `local-no-tracker` sentinel).
- AC-STE-151.5: When the FR's `milestone:` frontmatter resolves to no plan file (orphan), probe #14 falls back to the original strict assertion (status == `in_progress` AND assignee == `currentUser`). The orphan diagnostic is owned by probe #27 (`frontmatter-milestone-not-archived`); probe #14 must not silently pass on missing plans.
- AC-STE-151.6: `tests/gate-check-active-ticket-drift.test.ts` extends with three new behaviour cases: (a) active FR + Done ticket + plan with unchecked tasks ⇒ no fire; (b) active FR + Done ticket + plan with all tasks checked + plan `status: active` ⇒ fires with the canonical observed-vs-expected row; (c) active FR + Done ticket + plan missing ⇒ falls back to strict (fires).

## Technical Design

**Touchpoints:**
- `plugins/dev-process-toolkit/skills/gate-check/SKILL.md` — probe #14 prose: extend the predicate to read the FR's milestone plan and exempt the single-FR-clean state. Render the truth table inline.
- `plugins/dev-process-toolkit/tests/gate-check-active-ticket-drift.test.ts` — extend prose-shape assertions to match the relaxed wording; add three behaviour fixtures per AC-STE-151.6.
- `plugins/dev-process-toolkit/adapters/_shared/src/active_ticket_drift_predicate.ts` (**new**) — split into pure functions for clarity:
  - `inProgressMatches(summary, currentUser)` — original strict shape.
  - `singleFrCleanExempt(summary, planTaskState, statusMapping)` — true when `summary.status === statusMapping.done` AND `planTaskState.uncheckedTasks > 0` AND `planTaskState.planStatus === 'active'`.
  - `activeTicketDriftPasses(summary, planTaskState, statusMapping, currentUser)` — composed: `inProgressMatches(...) || singleFrCleanExempt(...)`.
- `plugins/dev-process-toolkit/adapters/_shared/src/plan_task_state.ts` (**new**) — `readPlanTaskState(specsDir, milestone)`. Reads `specs/plan/<milestone>.md` (active path only — files under `archive/` return `planStatus: 'archived'`). Counts lines matching `^\s*-\s*\[[ x]\]\s` and returns `{ totalTasks, uncheckedTasks, planStatus: 'active' | 'archived' | 'missing' }`. Strict regex avoids false positives on prose bullets.

**Predicate truth table:**

| FR status | Ticket status | Assignee     | Plan tasks       | Plan status | Outcome |
|-----------|---------------|--------------|------------------|-------------|---------|
| active    | in_progress   | currentUser  | any              | any         | **pass** |
| active    | done          | any          | unchecked > 0    | active      | **pass** (single-FR clean) |
| active    | done          | any          | all checked      | active      | **fail** (forgot bulk archive) |
| active    | done          | any          | any              | missing     | **fail** (strict fallback) |
| active    | done          | any          | any              | archived    | vacuous (probe #27 owns) |
| active    | backlog/etc.  | any          | any              | any         | **fail** (M23 drift) |
| active    | in_progress   | != current   | any              | any         | **fail** (M23 drift) |

**Failure-row shape unchanged** — when the probe fires, it still emits `<FR ULID> + <tracker ID> + observed=<status> + expected=in_progress + observed assignee + expected currentUser` to preserve compatibility with existing test assertions (AC-STE-87.4(d)).

## Testing

- Unit (new): `tests/plan-task-state.test.ts` exercises `readPlanTaskState` against fixtures — empty plan, all-checked, mixed, archived, missing. Each fixture asserts the `(totalTasks, uncheckedTasks, planStatus)` shape.
- Unit (new): `tests/active-ticket-drift-predicate.test.ts` — every row of the truth table rendered as a discrete test case.
- Integration (extension): `tests/gate-check-active-ticket-drift.test.ts` keeps every existing case green; adds the three new behaviour cases per AC-STE-151.6 and updates prose-shape assertions to match the new probe wording.
- Manual: re-run /smoke-test workflow — `/implement <single-FR>` on a fresh milestone with multiple unchecked tasks → `/gate-check` exits 0 with no probe-#14 row.

## Notes

Surfaced by /smoke-test run #4 against plugin v1.40.0, 2026-04-28. Evidence files: `/tmp/dpt-smoke-findings-2026-04-28.md`, `/tmp/dpt-smoke-gate-check.log`, `/tmp/dpt-smoke-implement.log`.

Severity: **high** — affects the canonical single-FR `/implement` workflow, the most common operator path short of `/ship-milestone`.

Resolution chosen: **Option 1** (relax probe #14) per the findings doc. Options 2 and 3 rejected:

- Option 2 (auto-archive in `/implement`) contradicts the milestone-bulk-archive design intent at `skills/implement/SKILL.md:216` and would scatter `git mv` into single-FR runs that today produce no archival churn.
- Option 3 (document `/spec-archive <FR-id>` as mandatory) reframes `/spec-archive` away from its escape-hatch role and pushes a procedural step onto every operator.

The relaxation does NOT weaken the M23 origin guarantee: probe #14 was authored to catch active FR + `Backlog` ticket (Phase 1 step 0.c skipped). That assertion stays — see AC-STE-151.3.

Future-proofing: `singleFrCleanExempt` is a pure function over `(summary, planTaskState, statusMapping)`. A follow-up FR could reuse it from probe #8 (archive-side) if a symmetric exemption ever becomes necessary, but no such case is planned.
