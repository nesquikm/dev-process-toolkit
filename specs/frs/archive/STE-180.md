# STE-180: Probe #14 carve-out: fully-checked single-FR runs (extend STE-151) {#STE-180}

## Requirement

`/gate-check` probe #14 (`ticket-state drift (active)`) asserts that every active FR with a tracker binding shows `status == in_progress AND assignee == currentUser`. STE-151 (archived) introduced a single-FR carve-out: when the FR is in the `done` state on the tracker but the FR file remains `status: active` AND the milestone plan has unchecked tasks remaining, probe #14 skips the assertion (the legitimate single-FR-clean state).

The 2026-05-01 Jira smoke (v2.3.0) caught an edge case STE-151 did not cover: when the single-FR run completes the milestone's last unchecked task, the plan has zero `[ ]` lines remaining but is still `status: active` (milestone-archive is a separate, milestone-scope concern that only `/implement M<N>` or `/spec-archive M<N>` runs). STE-151's carve-out condition `unchecked tasks remain` is false; probe #14 therefore fires with a misleading "drift" verdict on a legitimate canonical-chain state.

Concrete observed sequence (2026-05-01T15:13Z, dpt-test-project-jira, FR=DST-6, M=M1, single-task M1):

1. `/implement DST-6` runs; tracker round-trips Backlog → In Progress → Done; FR remains `status: active` (single-FR `/implement` does not archive — STE-83 documented behavior).
2. M1 plan task `[x]`-flips on commit; M1 now has zero `[ ]` lines.
3. `/gate-check` runs. Probe #14 sees: ticket=Done, FR=active, plan-fully-checked. STE-151 carve-out asks "any `[ ]` left?" — answer "no" — carve-out skipped — assertion fires — drift reported.

The verdict is misleading because the state is not drift; it is the legitimate "milestone ready to close" state. The remediation is for the operator to run `/spec-archive M<N>` (or `/implement M<N>` to drive Phase 5), not to investigate why the FR file is `status: active` after a clean ticket close.

The fix widens STE-151's carve-out to cover this state, with an advisory note that points the operator at the right next step.

Out of scope:
- Auto-closing the milestone on the last unchecked task. Considered and rejected. Auto-close changes `/implement <FR-id>` semantics from "ship one FR" to "ship one FR, possibly close the milestone" — surprising side effects when the operator is mid-multi-FR-milestone and just happens to complete the last task in a session.
- Changing probe #14's behavior on M23-class drift cases (Phase 1 step 0.c skipped, wrong assignee). Those still fail. The carve-out only widens for the fully-checked-Done state.

## Acceptance Criteria

- AC-STE-180.1: Probe #14 (`ticket-state drift (active)`) carve-out widens. New skip condition: ticket state is `done` AND the FR's milestone plan file has zero `[ ]` lines remaining (every task `[x]`). The original STE-151 carve-out (ticket=done AND milestone has `[ ]` remaining) continues to apply unchanged. Both carve-outs are evaluated; if either matches, the `status == in_progress` assertion is skipped. {#AC-STE-180.1}
- AC-STE-180.2: When the new carve-out skips the assertion, gate-check emits an advisory note in its rendered output: `M<N> plan fully checked but not archived — run /spec-archive M<N> or /implement M<N> to close`. The advisory does not fail the gate (severity = note, not error). One advisory per affected FR; deduplicated by milestone (if FRs A and B both skip via the new carve-out and both target M5, one advisory mentioning M5). {#AC-STE-180.2}
- AC-STE-180.3: Probe #14 still fires on the M23-class drift cases STE-151 was authored to catch (Phase 1 step 0.c skipped, wrong assignee, partial-archival regressions where some tasks `[ ]` and ticket=done). - AC-STE-180.4: Integration test `tests/gate-check-probe14-fully-checked-single-fr.test.ts` covers four cases:
  - (a) ticket=done + plan fully checked + FR status=active → advisory note, no fail.

- AC-STE-180.5: Documentation: `skills/gate-check/SKILL.md` probe #14 entry updates to describe both carve-outs (existing partial-checked + new fully-checked). The advisory note's exact text is reproduced in the SKILL.md so operators can grep for it. {#AC-STE-180.5}

## Technical Design

### Files touched

| File | Change |
|------|--------|
| `plugins/dev-process-toolkit/adapters/_shared/src/probe14_ticket_state_drift.ts` (or wherever probe #14 lives — locate via `grep -l "ticket-state drift" adapters/_shared/src/`) | Widen the skip condition. Add helper `planFullyChecked(planFilePath)` that scans for `[ ]` and returns `true` iff zero matches. The skip predicate becomes: `(ticket.state === "done" && (planHasUnchecked(plan) \|\| planFullyChecked(plan)))`. The same plan file scan supports both branches; under-the-hood it is "ticket=done and FR-status drift legitimate" but the advisory text differs by branch. |
| `plugins/dev-process-toolkit/skills/gate-check/SKILL.md` | Probe #14 entry: document both carve-outs side-by-side. Reproduce the advisory text verbatim. |
| `plugins/dev-process-toolkit/tests/gate-check-probe14-fully-checked-single-fr.test.ts` | New integration test per AC-180.4. |

### Advisory text shape

`M<N> plan fully checked but not archived — run /spec-archive M<N> or /implement M<N> to close`

The text is rendered by the gate-check report renderer, not the probe itself; the probe returns `{ ok: true, advisory: { kind: "milestone_ready_to_close", milestone: "M5", ... } }` and the renderer formats it. This separation keeps the probe pure and makes the advisory's prose easy to update without re-running the probe.

### Carve-out evaluation order

Both carve-outs are checked. The order doesn't matter (mutually exclusive states), but for log clarity the probe records which carve-out fired. The new fully-checked branch logs `carve_out: fully_checked_milestone`; the existing partial-checked branch logs `carve_out: partial_milestone`.

## Notes

- **STE-151 precedent.** STE-151 (archived) carved out the partial-checked case ("milestone plan still has unchecked tasks"). This FR extends the same carve-out shape to the complementary case ("milestone plan has zero unchecked tasks"). Together the two cover every legitimate single-FR-clean state.
- **Why not auto-close the milestone.** Auto-close on last-task-checked would conflate "ship this FR" with "close this milestone." Operators running multi-FR milestones may complete the last task mid-stream (e.g., running `/implement` against the smallest remaining FR last) and not be ready for the milestone-close ritual (release-files bump, ship-milestone, etc.). The advisory note is the right shape: signal readiness, but defer the close decision to the operator.
- **Smoke-test source.** `/tmp/dpt-smoke-findings-2026-05-01-jira.md` § F2 logged this. The Linear smoke on the same date did not surface this finding because the Linear chain ran `/implement` against an FR whose milestone had remaining tasks; the partial-checked STE-151 carve-out fired correctly.
- **Interaction with STE-181.** STE-181 (invocation-grammar clarification) documents that `/implement <FR-id>` silent-skips Phase 5 by design. This FR (STE-180) covers the gate-check side: when STE-181's documented behavior produces the fully-checked-Done state, probe #14 must not flag it as drift. STE-181 documents the user-visible contract; STE-180 enforces it at the probe seam. Land STE-181 first (clarifies semantics) then STE-180 (enforces the clarified semantics).
