# STE-380: Milestone-bound `Next:` recommendation — re-key /spec-write's closing-summary rule on milestone binding {#STE-380}

## Requirement

The `/spec-write` § 7 closing summary ends with a `Next:` line whose variant rule (authored by STE-181 AC-STE-181.3) keys on **new-FR presence**: any run that wrote a new FR recommends `/dev-process-toolkit:implement <tracker-id>`, and only cross-cutting-only runs recommend `/dev-process-toolkit:implement M<N>`. Since virtually every real spec session creates at least one new FR, the FR-id branch always fires — even when the session authored a multi-FR milestone plan whose natural next step is the milestone form. The two `/implement` forms diverge at Phase 5 by design (STE-83 via STE-181): the FR-id form silent-skips the milestone close + archival sweep that `/implement M<N>` runs, so the recommendation systematically points milestone-bound work away from the close; the only net today is gate-check probe #14's after-the-fact STE-180 advisory.

Approved 2026-07-14 `/brainstorm`: the discriminator becomes **milestone binding**. A milestone-bound FR always gets the M-form recommendation — including a single FR joining an in-flight milestone (the operator chose "always M form if milestone-bound" over FR-count or authorship keys). Milestone-less FRs keep the FR-id form. Because the prior rule ossified unnoticed as prose, the new rule ships with a doc-shape probe pinning its load-bearing literals (the STE-220/STE-238 lesson: prose-only rules drift; byte-checkable probes hold).

## Technical Design

**Canonical new rule text** (the paragraph probe #66 pins; replaces the current § 7 paragraph):

> **Next-line variant rule.** The discriminator is **milestone binding**, not new-FR presence. When the run wrote ≥ 1 new FR whose frontmatter carries `milestone: M<N>`, recommend the milestone form (`Run /dev-process-toolkit:implement M<N>`) — the milestone is the unit of shipping, and the M-form runs the Phase 5 close the FR-id form silent-skips (`skills/implement/SKILL.md` § Invocation forms). Render one `Next:` line per distinct milestone when a run's new FRs span several. When a new FR carries no `milestone:` binding, recommend the FR-id form (`Run /dev-process-toolkit:implement <tracker-id>`) for that FR. Cross-cutting-only runs (no new FR file written) keep the M<N> form. Hybrid runs (new FR + cross-cutting edit) follow the new FR's milestone binding.

**Template lines** (tail reference template, lockstep with the rule):

```
Next: Run `/dev-process-toolkit:implement M<N>` when specs are ready.   <!-- milestone-bound run (new FR with milestone: frontmatter) or cross-cutting-only run: recommend the M<N> form (milestone close). -->
Next: Run `/dev-process-toolkit:implement <tracker-id>` when specs are ready.   <!-- milestone-less new FR: recommend the FR-id form (single-FR ship; no milestone to close). -->
```

**Probe design.** Mirrors `implement_invocation_grammar_doc.ts` (STE-181 AC-STE-181.5) for the violation/remedy shape and `spec_write_first_turn_drift_scan.ts` (STE-270) for single-file scope + literal-substring matching. Read the SKILL.md once; check the four positive literals and the one negative tripwire; return `{ file, line, reason }[]` where line is the first-occurrence line for positives-missing (line 0/file-level note) and the matched line for the tripwire. No regex anywhere so future copy edits can't silently change match semantics.

**`/implement` lede reword.** Line ~13's parenthetical "(most common — smoke driver + `/spec-write`'s next step)" becomes a description tying the FR-id form to the smoke driver and milestone-less FRs; the smoke driver's own `<feature-id>` prescription is untouched (preserves STE-181's driver reconciliation). Probe #31's literals are structural (heading/table/Phase-5 row) and unaffected.

**Unchanged surfaces.** STE-180's probe #14 advisory; `/implement` runtime behavior (both invocation forms); the smoke driver; all capability tokens (this FR adds none — the `Next:` line is not a capability row).
