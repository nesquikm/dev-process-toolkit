## Requirement

Grandchildren spawned in fresh test-project cwds ignore the scaffolded `.claude/settings.json` allow-list — captured logs open with "Ignoring 10 permissions.allow entries from .claude/settings.json: this workspace has not been trusted" (conformance finding F4, high). The STE-252 policy artifact is therefore inert at the grandchild layer and the canonical chain ran on auto-mode classifier goodwill. Mechanism confirmed on-disk: workspace trust lives at `$CLAUDE_CONFIG_DIR/.claude.json` → `projects["<abs-path>"].hasTrustDialogAccepted`. Likely also the historical "Jira tracker WRITE path dead under autonomous claude -p" mechanism.

## Acceptance Criteria

- AC-STE-356.1: `/smoke-test` Phase 1 pre-seeds workspace trust for the resolved test-project's absolute path (`projects["<path>"].hasTrustDialogAccepted: true` via read-merge-write that preserves every unrelated key) before any child spawn; a meta-test pins the documented block.
- AC-STE-356.2: A spawn-time assertion verifies the trust entry exists before Phase 2 spawns; miss ⇒ NFR-10 canonical refusal; hit ⇒ byte-checkable capability token `workspace_trust_seeded` (same convention as `spawn_pattern_allow_present`).
- AC-STE-356.3: Runtime detector: any captured child/grandchild log carrying the "Ignoring … permissions.allow entries" warning is a high-severity finding (allow-list inert = policy breach); the STE-252 threat-model prose in both project-local SKILL.mds is amended to name workspace trust as an enforcement precondition; meta-tests cover both.
- AC-STE-356.4: Phase 5 teardown removes the seeded trust entries (config hygiene — no dead project entries accumulate); a meta-test pins the teardown block.

## Technical Design

**Trust seeding (`.claude/skills/smoke-test/SKILL.md` Phase 1, new step 6b).** After the settings/mcp pre-creation, seed trust via jq read-merge-write on `$CLAUDE_CONFIG_DIR/.claude.json`: back up the file to `/tmp/dpt-smoke-<tracker>-claude-json.bak`, then `jq '.projects["<abs test-project path>"] = ((.projects["<abs test-project path>"] // {}) + {hasTrustDialogAccepted: true})'` into a temp file and atomically `mv` over the original. Merge-only — every unrelated key (mcpServers, session state, other projects) passes through untouched. The absolute path comes from the pre-flight #6 `realpath` resolution.

**Spawn-time assertion + token.** Before the first Phase 2 spawn, `jq -e '.projects["<path>"].hasTrustDialogAccepted == true'` — miss ⇒ NFR-10 refusal (Verdict: scaffolded allow-list would be inert; Remedy: re-run Phase 1 step 6b / seed trust manually; Context: skill=smoke-test, pre-flight=workspace_trust_check); hit ⇒ log literal `workspace_trust_seeded` to the approval file.

**Runtime detector.** Extend `checkChildSpawnCapture` (or add `checkAllowlistInert(raw, child)`) in `adapters/_shared/src/smoke_child_capture.ts`: a capture whose raw text contains `Ignoring` + `permissions.allow entries` + `has not been trusted` yields a high-severity finding `STE-356 regression: allow-list inert — <child> (workspace untrusted)`. Wired into the same post-return assertion as STE-352's checks.

**Threat-model amendments.** Both SKILL.mds' threat-model sections state: the tracked allow-list is enforcement-effective ONLY when the spawn cwd's workspace is trusted; Phase 1 step 6b is the seeding step; the 2026-07-02 F4 capture is the counterexample.

**Teardown.** Phase 5 removes the `projects["<path>"]` entry (jq `del(...)`, same read-merge-write discipline) in both tracker paths.

## Testing

- Meta-tests: Phase 1 step 6b block present with jq merge shape + backup line (AC-STE-356.1); spawn-time assertion + `workspace_trust_seeded` token (AC-STE-356.2); threat-model amendment sentences in both SKILL.mds (AC-STE-356.3); Phase 5 removal block (AC-STE-356.4).
- Unit: the inert-allowlist detector over fixtures — capture with the "Ignoring … has not been trusted" stderr line ⇒ finding; healthy capture ⇒ none (AC-STE-356.3).
- Runtime (smoke-validated on the next run): grandchild logs carry NO "Ignoring … permissions.allow" warning; a run with seeding deliberately skipped surfaces the refusal.

## Notes

Provenance: finding F4, /conformance-loop 2026-07-02 iteration-1. **Risk (high, mitigated):** this FR writes to the operator's live `$CLAUDE_CONFIG_DIR/.claude.json` — a clobber would destroy MCP registrations and session state. Mitigations are load-bearing ACs: jq merge-only write, pre-write backup, atomic mv, teardown removal. Follow-up hypothesis to validate during the next Jira smoke: seeded trust revives the child tracker WRITE path (the historical "dead under autonomous claude -p" limitation), which would let the Jira leg's /implement complete tracker transitions headless.
