# STE-114: /setup Schema L canonical-key enforcement {#STE-114}

**Smoke-test findings ref:** F2 in `/tmp/dpt-smoke-findings-2026-04-25.md` (2026-04-25 re-run against v1.29.0)

## Requirement

The 2026-04-25 smoke test surfaced a Schema L drift: when /setup wrote the `## Task Tracking` block in CLAUDE.md, it added five non-canonical keys (`linear_team_key`, `linear_team_id`, `linear_team_name`, `linear_project_id`, `linear_project_name`). The skill's docs (`docs/setup-tracker-mode.md`, `docs/patterns.md`) document the canonical keys as `mode`, `mcp_server`, `jira_ac_field`, `branch_template` only. Schema L permits extra keys (blank values for non-applicable keys are legal), so the agent's choice was technically valid — but the keys are not consumed by any downstream skill (`/spec-write`, `/implement`, `/gate-check`, `/spec-archive` all read only the canonical four), and they create a parsing surface that could collide with a future canonical key of the same name.

The root cause: the smoke driver said "team and project references" in the prompt, and the agent took it literally. But the skill prose itself doesn't constrain the key set — it says "Append the `## Task Tracking` section to CLAUDE.md per Schema L with the resolved keys". A different agent under different prompting could emit a different set of extra keys, producing CLAUDE.md drift across projects that all claim to be "Schema L compliant".

This FR closes two gaps: codifying the canonical key list as a closed set in skill prose + docs, and adding a gate-check probe that flags non-canonical keys under `## Task Tracking`.

## Technical Design

**Closed-set enforcement.** Probe parses `## Task Tracking` line-by-line:

```typescript
const CANONICAL_KEYS = new Set(["mode", "mcp_server", "jira_ac_field", "branch_template"]);
function probe(claudeMd: string): ProbeResult {
  const section = extractSection(claudeMd, "## Task Tracking");
  if (!section) return { status: "pass", note: "vacuous: mode none" };
  const offenders: string[] = [];
  let inSubsection = false;
  for (const line of section.split("\n")) {
    if (line.startsWith("### ")) { inSubsection = true; continue; }
    if (inSubsection) continue;  // subsection contents are scoped out
    const match = line.match(/^([a-z_]+):/);
    if (match && !CANONICAL_KEYS.has(match[1])) offenders.push(match[1]);
  }
  if (offenders.length) return { status: "fail", note: `non-canonical keys: ${offenders.join(", ")}` };
  return { status: "pass" };
}
```

**Migration script.** Greenfield projects don't need it (M29 lands the constraint before adoption hits scale). Script is for the smoke-test project + any pre-M29 project that picked up the drift. Outputs a unified diff to stdout; operator pipes to `patch -p1` if they want to apply.
