# STE-322: Skill prose hygiene — step labels + phantom ship-milestone feature {#STE-322}

## Requirement

Two skill-prose hygiene items, bundled because both touch the toolkit's own skill SKILL.md surfaces and share STE-161's "skill reads coherently end-to-end" review rubric.

**A13 — `/implement` Phase 1 sub-step labels out of alphabetic order**: At `skills/implement/SKILL.md:40-49`, Phase 1 substeps are listed as `0.a → 0.b → 0.b′ → 0.b′¹ → 0.b′² → 0.b″ → 0.c → 0.e → 0.f → 0.d`. The terminal four (`0.c`, `0.e`, `0.f`, `0.d`) are in disk order but not alphabetic; `0.d` lands after `0.f`. Visual scan in the source breaks: a new contributor reads "tracker-mode probe at 0.d" thinking it precedes claim verification at `0.e`, when in fact 0.d is the LAST step in the phase. **Decision: rename to match disk order** — `0.d` becomes `0.g`. The brainstorm decision picks rename over reorder because the disk order is correct (probes after claim is the operator-visible flow); only the label is wrong.

**A15 — `/ship-milestone` README structure-count refresh (phantom feature)**: `docs/ship-milestone-reference.md:49-57` documents that `/ship-milestone` walks `plugins/dev-process-toolkit/{skills,docs,agents}/` and emits current counts into README's `## Structure` section. Verified false: `skills/ship-milestone/SKILL.md` flow steps 1-9 never walk these directories; no code implements the described mechanism. Spec-research traces the lineage to AC-STE-73.4 (M20, archived) which originally promised README structure counts but the implementation never landed; the documentation persists as a phantom feature. **Decision: correct the reference doc** (drop the phantom claim) rather than ship the feature now. Rationale: STE-315 (sibling FR in M84) already adds probe #57 `public_surface_count_drift` which is the byte-checkable enforcement layer; once #57 lands, a separate `/ship-milestone` README-refresh step would be defensive duplication.

**A23 — `/spec-archive` numbered list jumps 3 → 5** (LOW polish, rides along): `skills/spec-archive/SKILL.md` Single-FR archival list at L39-48 skips step 4. Two options: renumber 5 → 4, or insert the missing step. Decision: renumber — the existing 4 list items describe the actual flow; the "step 4 absent" is a numbering typo, not a missing step.

**A24 — `/ship-milestone` references "step 8 above" incorrectly** (LOW polish, rides along): `skills/ship-milestone/SKILL.md:89` says "must be `archived` after the pre-flight refusal in step 8 above". Step 8 DOES exist (`### 8. On refusal` at L173) but it's a flow step below L89, not above — and it's the "On refusal" handler, not the unshipped-FRs refusal the prose intends. The actual target is **Pre-flight refusal #1 "Unshipped FRs"** (numbered list at L23+, inside the `## Pre-flight refusals` section). Decision: rewrite the "step 8 above" phrase to "Pre-flight refusal #1 above" (or the equivalent unambiguous reference to the unshipped-FRs check).

**A21 — test comment mis-attributes 350→351 cap raise to STE-303** (LOW polish, rides along): `tests/skill-nfr-1-length.test.ts:6` comment says "350 → 351 in M79 (STE-303)" — CHANGELOG canon is STE-305. One-line comment fix.

**A22 — AC label missing `STE-` prefix** (LOW polish, rides along): `tests/setup-tracker-config-write.test.ts:172` reads `AC-303.2` instead of `AC-STE-303.2`. One-line test fix.

Five items bundled into one FR for diff coherence — all are 1–10 line targeted edits to specific files, all touch skill-prose-or-test surfaces, all share the same "operator sees the source and the source is wrong" failure mode.

## Technical Design

Five targeted edits across five files. Each is a 1–10 line change with no architectural implication.

For AC.1 (`/implement` step rename): the helper `replace_all` on `Edit` tool is suitable — every `0.d` token in `skills/implement/SKILL.md` becomes `0.g`, then renumbering proceeds as `0.c → 0.d`, `0.e → 0.e` (unchanged), `0.f → 0.f` (unchanged), and finally `0.g → 0.g` stays as the relocated terminal step OR (cleaner) the rename uses `0.d → 0.g`, `0.e → 0.d`, `0.f → 0.e`, `0.g → 0.f`. Either form satisfies the AC; implementer picks whichever preserves cleaner inbound references.

For AC.2 (phantom feature): the rewrite at `docs/ship-milestone-reference.md:49-57` cites STE-315 by ID. The cross-reference is unidirectional (this FR points at STE-315; STE-315 doesn't need to know about this FR). If STE-315 has shipped first, the cross-reference is byte-stable; if this FR ships first, the reference still resolves because STE-315 is allocated and visible at `https://linear.app/...issue/STE-315/...`.

For AC.3 (spec-archive renumber): single Edit on `skills/spec-archive/SKILL.md` L39-48 changing the step-5 label to step-4. No content drift.

For AC.4 (ship-milestone dangling reference): single Edit on `skills/ship-milestone/SKILL.md:89`. The phrase "step 8 above" refers to the unshipped-FRs check — that check is **Pre-flight refusal #1** (L23+, inside `## Pre-flight refusals`), NOT a numbered flow step. Replacement: "after Pre-flight refusal #1 above (Unshipped FRs)". One-line change.

For AC.5 (test comment fixes): two single-line Edits to test files. Pure comment / label hygiene.
