# STE-364: One-shot milestone-label backfill {#STE-364}

## Requirement

STE-362 (reliable landing) and STE-363 (archival-time assertion) stop *new* gaps, but neither can retroactively fix a ticket that already shipped without its milestone binding — the archival boundary only fires when an FR is *being* archived, and already-archived FRs are never re-processed. **GB-11** on `../st/glacy-app-be` is exactly this state today: archived FR, Done ticket, no `milestone-M5` label, and no automated path to fix it short of hand-editing the tracker.

This FR adds a **one-shot remediation sweep** — proposed surface `/spec-archive --backfill-milestone-labels` — that scans FR files (active + archived) with a tracker binding and `milestone:` frontmatter, fetches each bound ticket, and applies the milestone binding to any that lack it. It is idempotent (correctly-bound tickets are skipped), dry-run by default (previews before it writes), and best-effort per ticket (one failure does not abort the sweep), ending with an aggregate report. Scope is **FR-backed only** — it never touches a ticket that has no FR.

## Acceptance Criteria

- AC-STE-364.1: **Backfill sweep.** `/spec-archive --backfill-milestone-labels` scans FR files under `specs/frs/` **and** `specs/frs/archive/` that carry both a `tracker:` binding and `milestone:` frontmatter; for each, derive the canonical milestone name via `planFileHeadingToMilestoneName(specs/plan[/archive]/<milestone>.md)`, fetch the bound ticket, and — for any ticket missing the milestone binding (adapter-aware: object → `projectMilestone.name`; label → `labels ∋ milestone-<M-token>`) — apply it via `attachProjectMilestone`. {#AC-STE-364.1}
- AC-STE-364.2: **Idempotent + dry-run default.** A ticket already carrying the correct binding is skipped with no write. The sweep is **dry-run by default** (or explicit `--dry-run`): it prints the intended changes (`ticket → milestone`) and writes nothing; `--apply` performs the attaches. Re-running `--apply` after a clean sweep is a no-op (every ticket already correct). {#AC-STE-364.2}
- AC-STE-364.3: **Best-effort + aggregate report.** The sweep is best-effort per ticket: a single ticket's attach failure surfaces per NFR-10 and is recorded, but does **not** abort the remaining tickets. The run ends with an aggregate report — counts of `backfilled` / `already-correct` / `failed`, plus the list of failed ticket IDs and the plan file each maps to. {#AC-STE-364.3}
- AC-STE-364.4: **Scope + vacuity.** FR-backed only: the sweep enumerates FR files, never the tracker board, so a ticket with no FR (GB-12-class bug) is never fetched or touched. Vacuous — reports zero candidates, makes no tracker call — on `mode: none` and adapters with `project_milestone: false`. {#AC-STE-364.4}

## Technical Design

**Scope: a new flag on `/spec-archive` reusing the STE-363 fetch+assert loop.**

**Files touched:**
- `adapters/_shared/src/` (new, e.g. `backfill_milestone_labels.ts`) — `backfillMilestoneLabels(provider, project, specsDir, { apply })`: glob `specs/frs/*.md` + `specs/frs/archive/*.md`, filter to those with `tracker:` + `milestone:`, resolve canonical name (active or archived plan path), fetch each ticket, classify present/missing via the shared adapter-aware predicate (reused from STE-363), and on `apply` attach the missing ones. Returns `{ backfilled: [], alreadyCorrect: [], failed: [] }`.
- `skills/spec-archive/SKILL.md` — a `--backfill-milestone-labels` invocation form: parse the flag, default dry-run, print the preview table, apply on `--apply`, render the aggregate report. Document that it is FR-backed-only and vacuous on `mode: none` / `project_milestone: false`.

**Reuses.** The adapter-aware binding predicate and the retrying `attachProjectMilestone` (STE-362) are shared with STE-363 — this FR is the "sweep every FR" driver over the same per-ticket primitive the archival boundary uses on one FR at a time.

**Surface choice.** Lives as a flag on `/spec-archive` (which already owns the archived-side machinery and milestone binding) rather than a new standalone skill — one fewer command to discover, and the archival context is where "why isn't this ticket labelled" questions arise. (Open to relocation to a standalone command if the flag crowds `/spec-archive`'s surface — flagged at `/spec-write` draft time.)

**Non-goal.** Not a gate — the sweep is on-demand remediation. Ongoing enforcement is STE-362 (write side) + STE-363 (archival side); this FR is the one-time cleanup for gaps that predate them.

## Testing

- `backfill_milestone_labels.test.ts` — dry-run lists the missing and writes nothing; `--apply` attaches only the missing; already-bound tickets skipped; one ticket's attach failure records to `failed` and the sweep continues; aggregate counts (`backfilled`/`already-correct`/`failed`) correct; active + archived FRs both scanned; `mode: none` / `project_milestone: false` report zero candidates with no tracker call.
- Meta-tests under `tests/` — `skills/spec-archive/SKILL.md` carries the `--backfill-milestone-labels` invocation form, the dry-run-default + `--apply` contract, the FR-backed-only scope, and the aggregate-report shape.
- Gate: `bun test` (full gate from `plugins/dev-process-toolkit`) + `/dev-process-toolkit:gate-check` GREEN on this repo.

## Notes

- **Origin.** 2026-07-03 `/brainstorm` → `/spec-write`; the "(+ one-shot backfill)" half of the user-selected "Archival-time assertion (+ one-shot backfill)" placement.
- **Immediate consumer.** Running `/spec-archive --backfill-milestone-labels --apply` on `../st/glacy-app-be` fixes GB-11 (attaches `milestone-M5`) and reports GB-12 as out of scope (no FR → not enumerated), which is the correct Option-2 outcome. GB-11 can also be fixed by a single manual label add in the interim; this FR makes it a repeatable one-liner.
- **Why not extend probe #26.** Same rationale as STE-363: a continuous gate over all archived FRs re-fetches every historical ticket forever. A one-shot sweep is the right tool for a one-time cleanup; STE-362 + STE-363 prevent recurrence.
- **Sibling FRs.** STE-362 (write-side reliability) + STE-363 (archival-time assertion) are the durable enforcement; this FR is the remediation for the backlog of gaps that shipped before them.
