## Requirement

The `/implement` skill's tracker-mode entry (Phase 1 step 0.c) and Phase 4 close (Phase 4d steps b + c) reference `Provider.claimLock` / `Provider.releaseLock` / `Provider.getTicketStatus` as if directly callable. They are TypeScript classes with no shell entry — the LLM is the runtime and must make the equivalent MCP tool calls. The skill's existing rule "Do NOT call raw `mcp__<tracker>__save_issue` / equivalent write operations for in-flight FRs during /implement. Route through Provider.claimLock / Provider.releaseLock" reads (incorrectly) as "make zero tracker writes," because the routing target doesn't exist for the LLM-as-runtime.

This was concretely surfaced during M26 dogfood (2026-04-25): `/implement M26` skipped all 8 claim and release operations across the 8 FRs (STE-93..STE-100). Only the user's explicit observation triggered the manual backfill (16 MCP calls — 8 claim transitions through In Progress + 8 release transitions to Done — performed after the work was already committed and shipped).

This FR closes the *misinterpretation* failure mode by making the operations concrete (Option 1) and adding an early-catch (Option 2):

1. **Runbook rewrite** — replace abstract Provider references in `/implement` with a concrete adapter-agnostic runbook in `docs/implement-tracker-mode.md`. The runbook abstracts over each adapter's `transition_status` mapping by pointing at `adapters/<tracker>.md` § Tool surface for the per-tracker concrete pattern. Linear (`mcp__linear__save_issue` with `state` field) and Jira (`mcp__atlassian__transition_issue` after `get_transitions` lookup) both work via the same runbook because the per-tracker concrete tool calls live in the adapter docs, not in the runbook.

2. **Phase 1-exit self-check** — add a new step at end of Phase 1 (between 0.c and step 1) that calls `mcp__<tracker>__get_issue(<id>)`, asserts state == `status_mapping[in_progress]` AND assignee == currentUser, and refuses to enter Phase 2 with NFR-10 canonical shape on mismatch. Symmetric with Phase 4d step (c) which already exists for the release side (added in M23 STE-84) — this FR also rewrites that step to use the runbook.

The existing `/gate-check` probe #14 (active-side ticket-state drift, M24 STE-87) remains as the late-stage backstop. This FR adds the early-stage gate at Phase 1 exit.

## Technical Design

**Files touched (3 + 1 new test):**

1. `docs/implement-tracker-mode.md` (currently 120 lines) — append "## Claim runbook" + "## Release runbook" sections (~30-40 lines each).
2. `skills/implement/SKILL.md` (currently 354 lines, over NFR-1's 300 cap — pre-existing overflow not in scope here) — net change ≤+5 lines: replace 2-3 abstract Provider references with shorter runbook pointers; add 1 new Phase 1-exit self-check step (~6 lines).
3. `tests/implement-tracker-claim-runbook.test.ts` (new) — prose-shape test per STE-82 contract.

**No code changes.** Provider TS interfaces and impls (`adapters/_shared/src/provider.ts`, `local_provider.ts`, `tracker_provider.ts`) remain unchanged. The contract semantics they encode are still the spec — the runbook just makes them operationally readable for the LLM-as-runtime.

**Adapter compat verification:** both `adapters/linear.md` and `adapters/jira.md` already carry the abstractions the runbook depends on:

- `status_mapping: { in_progress, in_review, done }` (both adapters)
- `## Tool surface` table mapping abstract `transition_status` to concrete MCP calls (both adapters: Linear single `save_issue`, Jira two-step `get_transitions` + `transition_issue`)
- `getTicketStatus` → adapter-specific implementation that returns `{ status: string; assignee?: string | null }` (M24 STE-87 widening)

The runbook references these abstractions; adapter-specific concretizations stay in the adapter files. No changes needed in `adapters/linear.md` or `adapters/jira.md`.

## Notes

**Honest scope framing.** This FR closes the *misinterpretation* failure mode that caused M26's dogfood mistake — the skill's Provider abstraction made tracker writes look forbidden. After this FR:

- Runbook makes the operations concrete (no more "abstract API reference" interpretation gap).
- Phase 1-exit self-check catches a missed claim early (before any code is written, not at next /gate-check).

**What this FR does NOT solve:** the LLM remains the only enforcement mechanism. Skill prose still relies on LLM compliance. The same LLM that ignored "Call Provider.claimLock" could in principle also ignore "Now run the verification step." Stronger enforcement options (pre-commit hooks, CI checks, Claude Code hooks on skill invocation) each require a separate design conversation and live outside the plugin's portability boundary. Tracked in M28's "Out of scope" list.

**Origin:** M26 dogfood 2026-04-25. Backfilled 16 Linear MCP calls after the user observed the skip; current session captured the lesson as `feedback_implement_tracker_writes.md` in the user's per-machine memory, which is per-account and doesn't help downstream installs — hence this plugin-level fix.
