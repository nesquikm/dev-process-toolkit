# STE-227: feat(skills): --no-tech flag for non-technical FR drafting + needs_technical_review gate {#STE-227}

## Requirement

Stakeholders without technical context (PMs, product owners, operations leads) currently can't drop a draft FR into `specs/frs/` without first running through the full `/spec-write` flow, which interviews them on architecture, data model, APIs, test strategy, and other technical surfaces they don't own. The current alternatives (filing a Linear ticket separately, sending Slack notes, creating ad-hoc markdown files) bypass the toolkit's source-of-truth contract — the FR file in `specs/frs/`.

This FR adds a `--no-tech` flag to `/spec-write` and `/brainstorm` that lets non-technical stakeholders author the **WHAT** (requirement + acceptance criteria) and skip the **HOW** (technical design + testing). The resulting FR lands in `specs/frs/<id>.md` with `needs_technical_review: true` in frontmatter, the technical/testing sections as fixed placeholder lines, and a tracker ticket created on the board. `/implement` refuses to ship the FR until a technical reviewer runs `/spec-write <id>` (no flag) to fill the missing sections — auto-detected by the flag, no special incantation.

No new artifact type. Same FR file path, same five-section skeleton, same frontmatter shape plus one optional flag.

## Acceptance Criteria

The technical-design + testing interviews are skipped entirely; those two sections are written verbatim as `[needs technical review — run /spec-write <FR-id> to complete]` (literal `<FR-id>` substituted with the real ID at write time so the operator can copy the command directly).
- AC-STE-227.2: New optional FR-frontmatter field `needs_technical_review: true`. `buildFRFrontmatter(spec, trackerBinding?, opts?)` accepts `{ needsTechnicalReview?: boolean }` in opts; emits the field on the frontmatter line **only when explicitly true** (absent ≡ false; round-trip test coverage required).

- AC-STE-227.4: `/brainstorm --no-tech` runs Step 1 (Socratic clarification) only. Step 2 (Explore Approaches) is skipped — non-technical users can't pick architectural tradeoffs. - AC-STE-227.7: `/implement M<N>` refuses the whole milestone when **any** active FR in scope has `needs_technical_review: true`. NFR-10 refusal enumerates every flagged FR (so the reviewer can address them in one batch, no whack-a-mole). Refusal fires before any claim cycle starts.

## Technical Design

**Frontmatter field.** `needs_technical_review: bool` is an optional field in the FR frontmatter, emitted by `buildFRFrontmatter()` only when `opts.needsTechnicalReview === true`. Field ordering: after the `tracker:` block (or `id:`), before `created_at`. The omission-when-false convention keeps existing FR files byte-identical to their pre-M60 shape — zero migration churn for the 200+ archived FRs.

**Reference shape (tracker mode, flag set):**

```yaml
---
title: <title>
milestone: M<N>
status: active
archived_at: null
tracker:
  linear: STE-NNN
needs_technical_review: true
created_at: <ISO timestamp>
---
```

**`/spec-write --no-tech` flow.**

1. Argument parsing: extend `argument-hint` to `[--no-tech] [requirements | technical | testing | plan | all]`. The flag is parsed in resolver-entry preamble, before § 0a — it doesn't affect the resolver's `kind` routing.
2. § 0b interview gating: when `--no-tech` is set on a new-FR creation path, run only the requirement + AC interview. The technical-design + testing interviews are skipped. Write the placeholder line into `## Technical Design` and `## Testing` body (after the section heading, with one blank line separating).
3. § 0b step 2: `buildFRFrontmatter(spec, trackerBinding, { needsTechnicalReview: true })`.
4. § 0b step 4: `Provider.sync` runs. In tracker mode, `upsertTicketMetadata(null, { …, labels: [...defaultLabels, "needs-technical-review"] })`. Linear/Jira: label appended; adapters without label support: surface `needs_technical_review_label_unsupported` capability row.
5. Closing summary: `fr_needs_technical_review` capability row included unconditionally on every `--no-tech` run.

**`/spec-write <FR-id>` auto-resume.**

1. After § 0a resolver returns (FR resolved, file opened), read frontmatter.
2. If `needs_technical_review: true`: skip § 3's requirement + AC interview, jump straight to technical-design + testing-spec interview prose.
3. On save, mutate frontmatter — remove the `needs_technical_review` field via `buildFRFrontmatter(spec, trackerBinding, { needsTechnicalReview: false })` (false ≡ absent per AC.2). Mutate tracker ticket: drop `needs-technical-review` from labels via `upsertTicketMetadata(<id>, { labels: <existing> - "needs-technical-review" })`.
4. Closing summary: `fr_technical_review_cleared` capability row.
5. **No flag-clearing without completion.** If the user aborts the technical-design interview mid-stream, the flag stays — § 4 acceptance gate refuses to write the file unless both sections have non-placeholder content. Detection: post-write self-check extends to assert `## Technical Design` and `## Testing` body don't contain the placeholder substring when `needs_technical_review` was just cleared.

**`/brainstorm --no-tech` flow.**

1. Argument parsing: extend `argument-hint` to `[--no-tech] <feature description>`.
2. Step 1 (Clarify the Problem): unchanged — Socratic clarification fires regardless.
3. Step 2 (Explore Approaches): **skipped** when `--no-tech` is set. Skip the prose entirely; don't propose technical alternatives.
4. Step 3 (Get Design Approval): retitled to "Get Goal Approval" when `--no-tech` is set. Confirm the problem framing + ACs + success criteria.
5. Step 4 (Hand Off): the recommended next command is `/spec-write --no-tech` (the flag propagates). When the user says "start now", invoke `/spec-write --no-tech` directly.

**`/implement` Phase 0.b′ refusal.**

Site: between `resolveFRArgument` returning and Phase 0.c claim. Refusal fires immediately after `kind` routes to `ulid` / `tracker-id` / `url`, before the 0.b″ branch proposal logic.

```
On resolver hit (single FR):
  read frontmatter at frFilePath
  if frontmatter.needs_technical_review === true:
    refuse with NFR-10 canonical shape:
      Verdict: "/implement refused — FR <id> flagged needs_technical_review;
                technical sections are placeholders."
      Remedy: "Run /spec-write <id> (no flag) to complete the technical design
               + testing sections, then re-invoke /implement."
      Context: frFilePath, milestone, frontmatter excerpt
    emit `implement_refused_needs_technical_review` capability row
    exit non-zero
    NO claim, NO branch, NO side effects (refusal fires before 0.b″ + 0.c)

On resolver hit (milestone M<N>):
  enumerate every active FR in M<N>: ls specs/frs/*.md, filter by
    `milestone: M<N>` + `status: active`
  collect every FR with frontmatter.needs_technical_review === true
    into <flagged-list>
  if <flagged-list>.length > 0:
    refuse with NFR-10 canonical shape:
      Verdict: "/implement refused — milestone M<N> contains <N> FR(s) flagged
                needs_technical_review."
      Remedy: "Run /spec-write <id> (no flag) for each flagged FR, then
               re-invoke /implement M<N>:\n  - <id-1>\n  - <id-2>\n  - …"
      Context: every flagged FR's path
    emit `implement_refused_needs_technical_review` capability row
    exit non-zero
```

**`/gate-check` probe.** `needs_technical_review_consistency` lives next to existing FR-shape probes in `plugins/dev-process-toolkit/skills/gate-check/SKILL.md`. Implementation in `adapters/_shared/src/needs_technical_review_consistency.ts`. Glob `specs/frs/*.md`, parse frontmatter + body, assert the bidirectional invariant (flag-set ⇔ placeholder-present). Reports `file:line:column` for every violation.

## Notes

**Provenance.** Brainstormed 2026-05-05 over 6 design questions (transcript captured in conversation): artifact shape (option 1 — same FR file, two placeholder sections), refusal site (option 1 — Phase 0.b′ resolver entry), spec-write flow (option 1 — interview gating + tracker push now), brainstorm flow (option 1 — symmetric flag, skip approaches), reviewer flow (option 1 — auto-detect on `needs_technical_review: true`), milestone-scope refusal (option 1 — refuse whole milestone, list every flagged FR).

**Architectural shift.** Adds a per-FR "draft state" without introducing a new file type or path. The `needs_technical_review: true` field is the integration point — it gates `/implement`, gates the auto-resume in `/spec-write`, gates the consistency probe in `/gate-check`, and surfaces visually on the tracker board via the `needs-technical-review` label.

**Non-technical user persona.** PM / product owner / operations lead who can articulate the problem, who's affected, success criteria, and acceptance criteria — but does not own architecture, data model, API design, or test strategy. The `--no-tech` flag is opt-in: technical authors continue using bare `/spec-write` and `/brainstorm` with no behavior change.

**Memory.** `feedback_smoke_post_ship_retroactive` applies to AC-STE-227.11 (smoke fixture). May ship deferred `[~]` if M60 archival window doesn't carry a `/conformance-loop` run.
