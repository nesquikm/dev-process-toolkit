# STE-160: /spec-write: no-claim invariant on tracker-ticket create {#STE-160}

## Requirement

`/spec-write` (`plugins/dev-process-toolkit/skills/spec-write/SKILL.md`) step 4 was silent on what tracker-state a freshly-created ticket should land in. An LLM operator running `/spec-write` extrapolated from `feedback_implement_tracker_writes.md` ("claim at FR start") and pre-transitioned the new ticket to `In Progress` + assigned, on the assumption that "FR start" included spec-write time. The user caught this on STE-159 (M46 spec-write, 2026-04-30) and asked it be reverted to Backlog.

The right contract: `/spec-write` writes the spec, it does not start the work. The claim (Backlog → In Progress + assign me) is `/implement`'s job at Phase 1 FR start. A ticket sitting in `In Progress` before any code work begins misrepresents board state and trips the STE-87 active-side drift probe that expects `In Progress` to mean active development.

This FR lands an explicit "No claim on create" invariant in the spec-write skill prose so future operators (LLM or otherwise) honor the contract.

Out of scope: any change to `/implement` Phase 1 claim semantics (already correct per `feedback_implement_tracker_writes.md`); any change to `LocalProvider` (mode: none has no tracker state to manage); any change to the `Provider.sync` / `upsertTicketMetadata` helper signatures (the fix is at the call-site prose, not the helper).

## Acceptance Criteria

- AC-STE-160.1: `plugins/dev-process-toolkit/skills/spec-write/SKILL.md` § 0b step 4 prose carries an explicit **"No claim on create"** invariant before the STE-117 workspace-binding paragraph: freshly-created tracker tickets MUST land in the tracker's default state (Linear: `Backlog`, unassigned; Jira: project start state); do NOT pass `state` / `assignee` (Linear `save_issue`) or pre-transition (Jira `transitionJiraIssue`) on the create call.
- AC-STE-160.2: The invariant text cites the rationale (active-side drift probe STE-87 expects `In Progress` to mean active development; misrepresenting board state) and the claim-handoff site (`/implement` Phase 1 — see `feedback_implement_tracker_writes.md` framing in `skills/implement/SKILL.md`).
- AC-STE-160.3: A `/smoke-test` run confirms the contract: `/spec-write` creating a Linear ticket lands it in `Backlog` unassigned (default state); a `/spec-write` Jira run lands the work item in the project's start state (default). Tracker-side state inspected via `mcp__linear__get_issue` / `mcp__atlassian__getJiraIssue` post-/spec-write, before `/implement` runs. **Operator-deferred** is acceptable per `feedback_smoke_post_ship_retroactive.md` if M47 carries retroactive-validation framing; otherwise `[x]` at `/implement` time.

## Technical Design

Single-paragraph insertion in `plugins/dev-process-toolkit/skills/spec-write/SKILL.md` § 0b step 4, immediately after the lead sentence ("Call `Provider.sync(spec)` — no-op in `LocalProvider`, …") and before the **STE-117 workspace binding** paragraph. The new paragraph names both Linear (`state` / `assignee`) and Jira (`transitionJiraIssue`) failure modes by canonical MCP-tool name so the rule is unambiguous regardless of which adapter the operator's project binds to.

Already-applied (out of order, mid-conversation 2026-04-30 during STE-159 spec-write); `/implement M46` will verify the prose matches and check off ACs .1 + .2 without re-editing.

No helper change. `Provider.sync(spec)` and `upsertTicketMetadata(null, ...)` already accept omitted-state on the create call by construction; the bug was that the LLM operator chose to pass state, not that the helper required it. Fixing the helper to refuse a state argument on creates would over-engineer — the existing helper signature is correct; only the call-site instruction was wrong.

## Testing

Skill-prose change; the regression check is the smoke-test gate (AC-STE-160.3). No unit test added — the rule guards LLM operator behavior, not adapter-helper behavior, so a unit test would either (a) test the prose verbatim (brittle and uninformative) or (b) test the helper refusing state on create (out of scope per Technical Design).

A `/gate-check` probe for "freshly-created tracker tickets are in default state at end of /spec-write" is conceivable but not in scope here — would require a tracker round-trip on every probe run, which is too expensive for a routine gate. The smoke-test gate is the right cadence.
