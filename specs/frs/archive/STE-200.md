# STE-200: Scaffolding-milestone closure: --plan-only archival + slim Phase 1 + Phase 5 zero-FR skip {#STE-200}

## Requirement

Three sibling skills assume "milestone has FRs to archive / claim / ship." When a milestone is shipped at `/setup` time (code present, plan ACs all `[x]`, zero FR files), every Phase 1–5 step refuses or no-ops. Add a first-class scaffolding-closure path.

**Findings bundled:** F3.3, F3.6, F3.9 from `/tmp/dpt-findings-2026-05-04.md`.

**Symptoms observed in `dpt_test_lib`:**

1. **F3.3.** `/spec-archive M1` refuses with "No active FRs found for milestone M1; exit non-zero, no side effects." For a milestone whose plan is fully checked but has zero FR files (the M1 bootstrap case), the only canonical fix is manual `git mv` (bypasses drift check, atomic commit, traceability rewrite) or back-filling fake FR files just to satisfy the match-set rule.
2. **F3.6.** `/implement M1` arrived at Phase 1 with all tasks `[x]`, all ACs `[x]`, code/tests already present, and zero FR files. Every Phase 1–4 step assumed work to do: 0.c claim had nothing to claim; Phase 2 TDD had nothing to write; Phase 4 commit had no diff; Phase 4 archival had zero FRs. Phase 5 prompt would still fire, encouraging `/ship-milestone M1` for code that landed pre-toolkit.
3. **F3.9.** `/implement` Phase 5 silent-skip cases (single-FR arg, `'all'`/`'remaining'`/no arg, any FR still `status: active`, gate failed, non-TTY stdin) don't cover "milestone has zero FR files in `specs/frs/`." Encourages a release-version bump for code that landed pre-toolkit — meaningless commit.

The three together form a loop: `/spec-archive M1` refuses → `/gate-check` fires STE-180 advisory → operator runs `/implement M1` → has nothing to do → tries `/ship-milestone M1` → bumps a version for a placeholder milestone.

## Acceptance Criteria

- AC-STE-200.1: `/spec-archive M<N>` adds an auto-detect branch: when the FR match set is empty AND the plan file exists with all `## M<N>: ...` checkboxes done (or carries `kind: scaffolding` frontmatter from STE-197), archive the plan file alone. Run the drift Pass A grep against the live tree (no FR-row scan; the FR set was empty by definition). Surface a `plan_only_archival` capability row in the closing summary. {#AC-STE-200.1}
- AC-STE-200.2: `/spec-archive M<N>` with explicit `--plan-only` flag forces the plan-only branch even when the auto-detect heuristic would not fire (escape hatch for unusual cases). Documented in the skill prose with the trigger conditions. {#AC-STE-200.2}
- AC-STE-200.3: `/implement` Phase 1 detects "milestone has zero claimable FRs AND all plan ACs `[x]`" (or `kind: scaffolding`) and routes to **slim closure**: skip claim/build phases, jump to plan-only archival via `/spec-archive M<N>` (auto-branch from AC-STE-200.1) + Phase 5 prompt. Documented as the `/setup`-bootstrap-milestone path. {#AC-STE-200.3}
- AC-STE-200.4: `/implement` Phase 5 silent-skip cases add: "milestone has zero FR files in `specs/frs/` AND/OR plan frontmatter `kind: scaffolding`." Phase 5 chain prompt does not fire on scaffolding milestones — they don't get release-version bumps. {#AC-STE-200.4}
- AC-STE-200.5: New capability key `plan_only_archival` added to the static plain-language map: "milestone archived plan-only — no FRs in scope (scaffolding milestone or bootstrap-shipped code)." {#AC-STE-200.5}

## Technical Design

**`/spec-archive M<N>` decision tree (revised):**

```
fr_set = scan(specs/frs/*.md, milestone == M<N>)
plan_path = specs/plan/M<N>.md

if fr_set is empty:
  if plan_path doesn't exist:
    refuse: "No FRs and no plan file for M<N>"
  if --plan-only flag OR plan.frontmatter.kind == "scaffolding"
       OR every "^## M<N>:" checkbox is `[x]`:
    -> plan-only branch:
       - run drift Pass A grep (live tree only; no FR rows to walk)
       - git mv plan_path -> archive/
       - flip plan frontmatter to status: archived (per STE-197 AC.4)
       - emit capability row "plan_only_archival"
       - exit 0
  refuse: "M<N> has plan with unchecked ACs and no FRs; ambiguous state"

# fr_set non-empty: existing behavior (unchanged)
```

**`/implement` Phase 1 slim-closure detection (new step 1.5, after step 1 milestone resolution):**

```
if claimable_fr_count(M<N>) == 0:
  if plan.frontmatter.kind == "scaffolding"
       OR all_plan_acs_checked(M<N>):
    log("slim closure: milestone shipped pre-toolkit / scaffolding")
    invoke /spec-archive M<N>   # auto-takes plan-only branch per AC-STE-200.1
    if Phase 5 silent-skip evaluates true:
      exit 0   # no chain prompt
    else:
      surface chain prompt
    exit 0
```

**Phase 5 silent-skip extension:**

```
silent_skip = silent_skip OR
  (fr_count_in_specs_frs(M<N>) == 0) OR
  (plan.frontmatter.kind == "scaffolding")
```

**Capability map.** Single new key `plan_only_archival` rendered as: "milestone archived plan-only — no FRs in scope (scaffolding milestone or bootstrap-shipped code)."

## Testing

- Integration test: bootstrap a fresh project via `/setup` → `specs/plan/M1.md` carries `kind: scaffolding` (per STE-197 AC.3) → `/implement M1` slim-closes (no claim, no build) → `/spec-archive M1` runs plan-only → `/gate-check` no longer advises STE-180 → no `/ship-milestone M1` prompt fires.
- Integration test: `/spec-archive M5 --plan-only` against a milestone with active FRs refuses (`--plan-only` is for explicitly-empty cases; refuse if FRs are present) — clarify scope of the flag.
- Unit test: `/implement` Phase 1 detection correctly routes a zero-FR milestone through slim closure; correctly routes a single-FR milestone through normal flow.
- Unit test: Phase 5 silent-skip evaluator returns true on `kind: scaffolding`; returns false on a milestone with zero FRs but no scaffolding marker (operator may want manual inspection).

## Notes

- This FR depends on STE-197's `kind: scaffolding` frontmatter convention. Implement STE-197 before this one.
- The auto-detect heuristic (zero FRs + all-checked plan) covers the `dpt_test_lib` symptom even without `kind: scaffolding`; the explicit marker is the future-proof channel for scaffolding milestones, the heuristic catches retroactive cases.
- F3.3's "M<N> has plan with unchecked ACs and no FRs" remains a refusal — that's an ambiguous state (plan describes work, no FRs back it). Operator should `/spec-write` FRs first or move tasks to a later milestone.
- Capability: a tracker-mode `/spec-write` run that creates this FR surfaces `filename_policy_override` per the unconditional rule.
