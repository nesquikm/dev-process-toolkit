## Requirement

`specs/requirements.md:24` NFR-2 reads:

> "Existing CLAUDE.md files generated by v1.7.0 must continue to work with v1.8.0 skills"

v1.7.0 and v1.8.0 are 19 and 18 releases stale respectively. The NFR was meaningful when first written (it codified a backwards-compat constraint between two adjacent versions). With no installed user base to honor it for, the constraint is dead.

Two reasonable options:
- **Delete NFR-2 outright** — clean, but loses the *kind* of constraint it represented (CLAUDE.md schema stability across releases).
- **Rewrite as a live constraint** — e.g., "CLAUDE.md schema is forward-compat: a CLAUDE.md generated by `/setup` continues to work with newer skills until a major version bump." Captures the real invariant and stays meaningful as the plugin evolves.

## Acceptance Criteria

- AC-STE-99.1: `specs/requirements.md:24` no longer references v1.7.0 or v1.8.0. {#AC-STE-99.1}
- AC-STE-99.2: NFR-2 is either (a) deleted entirely with neighboring NFRs renumbered to maintain sequence, OR (b) rewritten as a live forward-compat constraint without specific version pins. /implement Phase 1 picks one based on whether any live cross-spec reference cites NFR-2 (if yes, prefer rewrite to avoid cross-ref churn; if no, delete). {#AC-STE-99.2}
- AC-STE-99.3: Cross-spec references to NFR-2 (in `technical-spec.md`, `testing-spec.md`, plan files, archived FRs) remain valid post-edit — either updated to reference the rewritten NFR, or removed if the NFR was deleted. {#AC-STE-99.3}

## Technical Design

Two-step decision tree:

1. `grep -rn "NFR-2" specs/ adapters/ plugins/` — count live references.
2. If references > 0 → rewrite NFR-2 in place (renumbering would cascade). If references == 0 → delete and renumber NFR-3+.

Either way, the v1.7.0/v1.8.0 strings are gone.
