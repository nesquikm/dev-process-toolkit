# STE-341: specs/design/ storage convention + optional ## Design References template section {#STE-341}

## Requirement

The toolkit has no convention for design reference images. When a toolkit user designs a UI feature, the mockups / Figma exports / screenshots they build toward live nowhere in the spec tree — they are lost the moment the chat scrolls away, and nothing in `/implement` can later check the built UI against them.

This FR establishes the storage half of that capability (Deliverable A from the 2026-06-30 `/brainstorm` session): a persistent, version-controlled `specs/design/` tree plus an optional FR-body section that references images from it. It deliberately does **not** add any app-driving or visual-comparison behavior — Deliverable B (the `/implement` visual conformance check) is a later milestone.

**Chosen approach — Approach 2 (persistent store, specs-only archive).** Picked over "travel-with-FR" and "tracker-native" in the brainstorm and confirmed by a duck council (GPT-5.2 + Gemini), both of which independently recommended a hybrid and both flagged *classification ambiguity* (a shared asset wrongly snatched into the archive, breaking active references) as the top risk. Approach 2 neutralizes that risk structurally: **images never move**, so no reference can ever break on archive and a mis-classified image is merely in the wrong folder, never a dangling link.

**Load-bearing property.** An FR's markdown moves one directory deeper on archival (`specs/frs/<id>.md` → `specs/frs/archive/<id>.md`). File-relative image links would silently break across that move. Therefore design-image references are **repo-root-relative** and resolved from the repo root by the toolkit (`/gate-check` + `/implement`) — the reference string is byte-identical and valid whether the FR sits in `specs/frs/` or `specs/frs/archive/`. This is what lets the `specs/design/` tree stay out of the archival machinery entirely.

## Acceptance Criteria

- AC-STE-341.1: **Layout documented.** `plugins/dev-process-toolkit/docs/layout-reference.md` documents the `specs/design/` tree with its two subtrees — `specs/design/system/` (durable design-system artifacts referenced by many FRs over time) and `specs/design/frs/<id>/` (per-feature mockups, one folder per FR keyed by the FR's `Provider.filenameFor(spec)` stem) — and states the **never-archived** rule: no skill performs `git mv` or `rewriteArchiveLinks` rewrites on any `specs/design/` path. {#AC-STE-341.1}
- AC-STE-341.2: **Optional `## Design References` section documented.** The FR-body section contract documents an **optional** `## Design References` section, placed after `## Acceptance Criteria`, whose entries are repo-root-relative image paths (e.g., `specs/design/frs/STE-341/login.png`) in a list, with a one-line per-entry caption and a worked example. The five required body sections (`## Requirement`, `## Acceptance Criteria`, `## Technical Design`, `## Testing`, `## Notes`) keep their order and required status; the new section is purely additive and optional. The `docs/layout-reference.md` line that today reads "exactly these top-level sections in order" is relaxed to admit the optional section. {#AC-STE-341.2}
- AC-STE-341.3: **Archival immutability stated at the archival surfaces.** `skills/implement/SKILL.md` (Phase 4 Milestone Archival) and `skills/spec-archive/SKILL.md` each carry an explicit one-line statement that `specs/design/` paths are never `git mv`'d and never rewritten by `rewriteArchiveLinks` — referenced images stay put; only the spec markdown moves. {#AC-STE-341.3}
- AC-STE-341.4: **No section-set drift.** If any `/gate-check` probe enforces the FR body section set or order, it is updated to permit the optional `## Design References` section (no false-positive GATE FAILED on an FR that carries it). If no such probe exists, this AC is satisfied by an explicit note in the FR-section contract that the optional section is allowed — there is nothing to relax. {#AC-STE-341.4}

## Technical Design

**Scope: documentation + convention only.** No runtime helper ships in this FR; it defines the contract that STE-342 (capture) and STE-343 (probe) implement against.

**Files touched:**
- `plugins/dev-process-toolkit/docs/layout-reference.md` — new `## Design asset access` subsection documenting the `specs/design/{system,frs/<id>}/` tree + never-archived rule; relax the FR-section line (currently "Each FR has exactly these top-level sections in order: …") to "these required sections, in order, optionally followed by `## Design References` after `## Acceptance Criteria`".
- `plugins/dev-process-toolkit/skills/spec-write/SKILL.md` — § 0b body-section description gains the optional `## Design References` section (additive; the five required sections unchanged).
- `plugins/dev-process-toolkit/skills/implement/SKILL.md` + `plugins/dev-process-toolkit/skills/spec-archive/SKILL.md` — one-line immutability statement at each archival surface (AC-STE-341.3).
- `plugins/dev-process-toolkit/templates/CLAUDE.md.template` — mention `specs/design/` in the layout overview so generated projects know the convention exists.

**Convention shape (illustrative):**
```
specs/
  design/
    system/                  # durable design-system artifacts (never tied to one FR)
      color-tokens.png
    frs/
      STE-341/               # per-feature mockups, keyed by FR filename stem
        login-screen.png
  frs/
    STE-341.md               # references design/frs/STE-341/login-screen.png (repo-root-relative)
    archive/
      STE-341.md             # SAME reference string after archive — image did not move
```

**Reference style (the load-bearing decision):** repo-root-relative paths inside an optional `## Design References` section. Not file-relative markdown `![]()` links — those would need a `../` → `../../` rewrite when the FR moves into `archive/`, reintroducing the archival machinery Approach 2 was chosen to avoid. The toolkit resolves the path from the repo root; GitHub inline preview is knowingly traded away (specs are read in-editor and consumed by the toolkit, not rendered as a site).

**Bootstrapping:** `specs/design/` is created lazily on first capture (STE-342); `/setup` does not scaffold an empty tree.

- No new runtime behavior to unit-test in this FR — the helper + probe arrive in STE-342 / STE-343.

## Notes

- **Origin.** 2026-06-30 `/brainstorm` + `/spec-write` session. Design fork ("archive images vs. a never-archived design dir?") resolved to *both, hybrid* → Approach 2 (persistent store, specs-only archive). Duck council (GPT-5.2 + Gemini) confirmed hybrid and flagged classification ambiguity as the top risk, which Approach 2 neutralizes by never moving images.
- **Foundation FR.** STE-342 (capture step) and STE-343 (gate-check probe) both build on this convention; neither is implementable before the `specs/design/` layout + `## Design References` section exist.
- **Deliverable B is deferred.** The visual conformance check in `/implement` (Dart MCP for Flutter incl. Flutter web; Chrome MCP for pure web; advisory, never a hard gate; Flutter ships first) is explicitly out of scope for M91 — it is the next milestone.
- **Why not GitHub-rendered links.** Inline image rendering in GitHub markdown preview was knowingly sacrificed for immutable references. The toolkit (not a static-site renderer) is the primary consumer, and immutability-across-archive was the brainstorm's load-bearing requirement.
