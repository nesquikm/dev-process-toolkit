# STE-125: /implement Phase 4 archival rewrites traceability links {#STE-125}

## Requirement

When `/implement <id>` archives an FR in Phase 4 (`git mv specs/frs/<id>.md specs/frs/archive/<id>.md`), it must also rewrite every relative link in the active plan files (`specs/plan/M*.md`) that points at the pre-archive path. The very next `/gate-check` (probe #23, `traceability-link-validity`) must pass on the commit `/implement` produces — no manual fix-up between commit and gate.

The link-rewrite logic already lives at `adapters/_shared/src/spec_archive/rewrite_links.ts` (named in the gate-check log itself). `/spec-archive` invokes it; `/implement` Phase 4 does not. This FR closes that gap.

**Smoke-test evidence.** Run #4 (2026-04-27, v1.34.0) on the throwaway `dpt-test-project`: `/implement STE-124` archived `specs/frs/STE-124.md` → `specs/frs/archive/STE-124.md` correctly, but `specs/plan/M1.md:17` still carried `[STE-124](../frs/STE-124.md)`. Probe #23 failed on commit `e671eb7` — the commit `/implement` claimed was clean. Full report: `/tmp/dpt-smoke-findings-2026-04-27.md` § F2.

## Acceptance Criteria

- AC-STE-125.1: After `/implement <id>` Phase 4 archives an FR, every active plan file (`specs/plan/M*.md`, excluding `archive/`) that linked to `specs/frs/<id>.md` now links to `specs/frs/archive/<id>.md`. Verified by `grep -r "frs/<id>.md" specs/plan/` returning zero non-archive matches after Phase 4 returns.
- AC-STE-125.2: The Phase 4c commit produced by `/implement` passes `/gate-check` probe #23 (`traceability-link-validity`) with no manual edits between commit and gate. Verified end-to-end by re-running the smoke-test chain on a one-FR scaffold and observing `/gate-check` pass.
- AC-STE-125.3: When the link-rewrite step itself fails (e.g., a plan file is read-only, or `rewrite_links.ts` throws), `/implement` does NOT commit the archive move. The Phase 4 report surfaces an NFR-10 canonical refusal naming the offending plan `file:line` and the link that would have been rewritten. The FR file remains in its pre-archive location and the tracker ticket stays in `In Progress` (not released to `Done`).
- AC-STE-125.4: `/implement` Phase 4 report explicitly lists "rewrote N traceability links in M plan files" (or "no traceability links to rewrite") so the operator sees the work in the summary, mirroring how `/spec-archive` reports.

## Technical Design

- Hook point: `/implement` Phase 4, between the `git mv` step and the Phase 4c commit. Invoke `rewriteLinks(specsDir, oldPath, newPath)` from `adapters/_shared/src/spec_archive/rewrite_links.ts`.
- Failure semantics: any error from `rewriteLinks` aborts Phase 4 cleanly — the `git mv` is undone (or never staged), the commit is not made, the tracker release does not fire. NFR-10 refusal shape with `file:line:column` + remedy.
- Idempotent: re-running `/implement` on an already-archived FR (edge case: operator re-runs after a partial failure) is a no-op for the link-rewrite step (no matches → zero work → no error).

## Testing

- Test file: `tests/spec_archive/rewrite_links.test.ts` (link-rewrite already has unit tests if existing; this FR adds an integration test for the `/implement` Phase 4 path specifically).
- Bun test: scaffold a fixture with `specs/plan/M1.md` linking to `specs/frs/X.md`, run the Phase 4 archival shim, assert `M1.md` now links to `specs/frs/archive/X.md` AND the commit hash points at the rewritten file.
- Failure-path test: scaffold a read-only plan file, run Phase 4, assert NFR-10 error + no commit + no tracker release.

## Notes

- Severity: high. Single-FR happy path is broken in v1.34.0; gate fails on the commit `/implement` says is clean.
- Dual-purpose: the same gap (Phase 4 cleanup not reaching plan files) is the root of STE-126 (plan task-list verify-line cleanup). Land STE-125 first; STE-126 piggybacks on the same Phase 4 hook point.
