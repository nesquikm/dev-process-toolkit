# STE-130: Add mermaid workflow diagram to README.md {#STE-130}

## Requirement

The toolkit's `README.md` has no visual representation of the workflow. New visitors landing on GitHub and mid-onboarding adopters have to reconstruct the lifecycle from the 14-row commands table — high cognitive load for what is essentially a four-phase pipeline.

Add a mermaid `flowchart TD` to `README.md` using four phased subgraphs (Setup → Plan → Build → Ship), showing only user-invoked skills inside each phase. Below the diagram, a short prose paragraph names internal skill-to-skill orchestration (e.g., `/implement` calls `/tdd`, `/gate-check`, `/docs --quick`, `/pr` under the hood) and a one-liner notes how tracker integration threads Plan → Build → Ship.

**Approved /brainstorm decision (2026-04-27):** Approach 2 (Phased subgraphs). Rejected alternatives: Approach 1 (linear flowchart with optional branches — iteration loop too cluttered) and Approach 3 (two-lane specs↔code — too complex for "simplest possible" constraint).

## Acceptance Criteria

- AC-STE-130.1: `README.md` contains a fenced ` ```mermaid ` block with a `flowchart TD` declaration and exactly four `subgraph` blocks named `Setup`, `Plan`, `Build`, `Ship` (in that lifecycle order).
- AC-STE-130.2: Every user-invoked skill from `plugins/dev-process-toolkit/skills/*/SKILL.md` (the 14-skill surface — `setup`, `brainstorm`, `spec-write`, `implement`, `tdd`, `gate-check`, `debug`, `docs`, `pr`, `ship-milestone`, `spec-review`, `spec-archive`, `simplify`, `visual-check`) appears as a node inside exactly one subgraph; no internal-only skill calls (e.g., the `/docs --quick` invocation from `/implement` Phase 4) appear as nodes in the diagram.
- AC-STE-130.3: Below the diagram, a prose paragraph (≤4 sentences) names the major internal skill-to-skill orchestration relationships: `/implement` invokes `/tdd`, `/gate-check`, `/docs --quick`, and `/pr`; `/ship-milestone` invokes `/docs --commit --full`.
- AC-STE-130.4: A one-line note below the diagram describes how tracker integration (Linear / Jira / `mode: none`) threads through Plan → Build → Ship.
- AC-STE-130.5: The diagram renders as a mermaid graph on github.com (no syntax errors when GitHub's mermaid renderer parses it). Verified by visual inspection on the eventual PR preview.
- AC-STE-130.6: Diagram is placed between the `## Install as Plugin` section's setup paragraph and the `## What You Get` heading, under a new `## Workflow` heading.

## Technical Design

- Single edit to `README.md`. No new code, no new tests, no template changes.
- Mermaid syntax: `flowchart TD` (top-down) with four `subgraph` blocks. Direction inside each subgraph is implicit (TD); arrows between subgraphs show phase progression.
- Use raw skill names (e.g., `/setup`, `/spec-write`) — no `dev-process-toolkit:` namespace prefix in the diagram (clutters the boxes; the README already establishes the namespace context above).
- "As simple as possible" constraint (per /brainstorm): no regression test asserting skill names match the file system. If the skill list drifts, the diagram drifts; release checklist + reviewer attention is the safety net. (Future FR may add a probe if drift becomes a recurring issue.)

## Testing

- No automated test added — pure doc edit per the simplicity constraint.
- Manual verification: open the PR on GitHub, confirm the mermaid block renders as four labeled subgraphs with arrows between them; confirm every skill name in the diagram resolves to a `plugins/dev-process-toolkit/skills/<name>/SKILL.md` file.
- Regression risk: future skill additions/removals make the diagram stale. Mitigation: add a release-checklist line item at /ship-milestone time ("if skill count changed, update README workflow diagram"). Out-of-scope item, tracked in the M34 plan file.

## Notes

- Severity: low. Cosmetic, but the README is the entry point for new users — every install starts here.
- Dogfooding signal: the toolkit defines a `/docs` skill prescribing Diátaxis but doesn't yet dogfood it for its own meta-docs. This FR is README-scoped only; the broader Diátaxis-adoption question is a separate concern (out of scope here).
- Single-FR milestone. Phase A only. `/ship-milestone` closes it as a patch- or minor-level release (decided at release time based on what else lands alongside).
