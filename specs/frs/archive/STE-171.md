# STE-171: /implement Stage D archive hardening — traceability row + git add untracked + cleanupPlanVerifyLines fires {#STE-171}

## Requirement

Smoke #6 (Jira, 2026-04-30, v2.1.0) caught three small holes in `/implement` Phase 4 Stage D archive flow that affect every milestone-archival run after M49:

1. **Traceability row not appended.** `specs/requirements.md` § 6 carries the inline comment "Rows are added by /spec-write or /implement once an AC ships." But Stage D doesn't append the row in tracker mode. After DST-5 archived, § 6 was empty. Smoke #6 F4 finding 1.
2. **`git mv` falls back to plain `mv`** when the FR file is untracked. Same end-state as `git mv`, but `git log --follow` loses rename history because the pre-archive state is never tracked. Smoke #6 F2 finding.
3. **`cleanupPlanVerifyLines` no-op on placeholder removal.** The helper at `plugins/dev-process-toolkit/adapters/_shared/src/spec_archive/cleanup_plan_verify_lines.ts` exists and is referenced in Stage D prose, but `/gate-check` probe #28 still warned on `src/.placeholder.test.ts` removal verify line in M1.md. Either the helper isn't invoked from Stage D in tracker mode, or its `deletedFiles[]` argument missed `src/.placeholder.test.ts` (which is removed earlier in the same run as part of the M1 placeholder-removal task). Smoke #6 F3 finding.

The three fixes share file-edit surface (Stage D in `skills/implement/SKILL.md` plus the helper directory) and ship together in one FR to avoid diff thrashing.

Out of scope:
- The `archived_at` precision fix (already shipped under M45 / STE-158).
- Cross-cutting drift propagation beyond the § 6 traceability row — directory-tree and API-surface drift in `technical-spec.md` are out of scope for an autonomous skill (they require natural-language regeneration). STE-172 surfaces a hint on top.
- Adding new conformance probes — only the existing probe #28 false-positive is closed.

## Acceptance Criteria

- AC-STE-171.1: Stage D appends a traceability row of shape `AC-<id>.1..N → <impl-files> → <test-files>` to `specs/requirements.md` § 6 per archived FR; idempotent on re-run (already-present row is detected and not duplicated). Empty `specs/requirements.md` § 6 (toolkit-side or end-user-side) gains exactly one row per archived FR per archive commit.
- AC-STE-171.2: Stage D detects untracked FR files (`?? specs/frs/<tracker-id>.md` in initial git status) and `git add`s them before `git mv`. Rename history is preserved in `git log --follow specs/frs/archive/<tracker-id>.md` (showing both pre-archive and archive commits).
- AC-STE-171.3: `cleanupPlanVerifyLines` reliably fires on archive — Jira-mode (and Linear-mode) archive run on a fresh M1 with the placeholder-removal task no longer trips `/gate-check` probe #28 (`plan-verify-line-validity`) for `src/.placeholder.test.ts`. Either the helper invocation is fixed (correct `deletedFiles[]` argument including run-removed files) or the helper is patched to recognize "removed" / "deleted" / "no longer exists" verify-line semantics — implementer's choice with rationale captured in the technical design.
- AC-STE-171.4: Existing pre-archived plans are unchanged by the cleanup helper — round-trip test asserts a previously-archived `M*.md` with verify lines pointing at deliberately-preserved deleted paths is byte-identical pre/post helper run.
- AC-STE-171.5: `bun test` green; `/gate-check` clean; `/spec-review M49` reports no deviations.

## Technical Design

Three changes in `plugins/dev-process-toolkit/skills/implement/SKILL.md` Phase 4 Stage D plus one possible patch in `plugins/dev-process-toolkit/adapters/_shared/src/spec_archive/cleanup_plan_verify_lines.ts`:

1. **Traceability row append.** Stage D currently says "If `specs/requirements.md` has a traceability matrix, update Implementation and Tests columns with actual file paths." The prose is opt-in and the LLM short-circuits in tracker mode. Tighten the contract: when `specs/requirements.md` § 6 exists, Stage D MUST append one row per archived FR. The row shape is `| AC-<tracker-id>.1..N | <comma-separated impl-files> | <comma-separated test-files> |`. Idempotency: probe for an existing row matching the FR's AC prefix before appending; skip on hit.

2. **`git add` untracked FR file.** Before `git mv specs/frs/<id>.md specs/frs/archive/<id>.md`, parse the output of `git status --porcelain specs/frs/<id>.md`. If the file shows `??` (untracked), `git add specs/frs/<id>.md` first. Same atomic commit captures both the add and the move. Rename detection works because the pre-archive blob is now in the index.

3. **`cleanupPlanVerifyLines` reliable invocation.** Diagnose first — read the smoke #6 trace, the helper source, and the Stage D prose to determine whether (a) Stage D doesn't call the helper in tracker mode, (b) the helper is called but `deletedFiles[]` doesn't include `src/.placeholder.test.ts`, or (c) the helper is called correctly but its match logic doesn't recognize the verify-line shape. The fix branches on the diagnosis — likely a single-line addition of the run-removed file to `deletedFiles[]`, or extending the helper's match to also rewrite verify lines whose surrounding sentence contains "removed" / "deleted" / "no longer exists".

Round-trip safety: the helper operates only on plans staged in the current archive commit (`addedTestFiles + deletedFiles` from the current run, not historical state). Pre-existing archived plans are out of its scope by construction.
