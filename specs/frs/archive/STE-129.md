# STE-129: specs/requirements.md § 2 scope — cross-cutting only or per-FR ledger {#STE-129}

## Requirement

The toolkit must explicitly decide whether `specs/requirements.md` § 2 is (a) a per-FR ledger that `/spec-write` should populate as FRs are created, or (b) truly cross-cutting (high-level requirements only; per-FR detail lives in `specs/frs/<id>.md` exclusively). Today the file's own header advertises "cross-cutting only" but `/setup` scaffolds an `FR-1: [Feature Name] {#FR-1}` template block with `<tracker-id>` literals — which tempts users (and `/spec-write`) to fill it as a per-FR list.

Recommended choice: **cross-cutting only** — matches the file's stated intent and the M18 STE-60 architectural decision that per-FR content lives in `specs/frs/`. Drop the placeholder block from the template; replace with examples of cross-cutting requirements (auth scheme, observability surface, etc.).

**Smoke-test evidence.** Run #4 (2026-04-27, v1.34.0). After `/implement STE-124` shipped: `specs/frs/archive/STE-124.md` exists with full FR detail, but `specs/requirements.md` § 2 still carries the unfilled `FR-1` placeholder. No probe catches this — gate-check probe #23 is plan-scoped, not requirements-scoped. Full report: `/tmp/dpt-smoke-findings-2026-04-27.md` § F4.

## Acceptance Criteria

- AC-STE-129.1: Explicit decision documented in `templates/spec-templates/requirements.md.template` header — the template comment names the chosen scope and gives 1–2 examples.
- AC-STE-129.2: If "cross-cutting only" is chosen: drop the `FR-1: [Feature Name] {#FR-1}` placeholder block from the template. Replace with a comment block showing 2–3 example cross-cutting topics (auth, observability, accessibility) without `<tracker-id>` literals.
- AC-STE-129.3: `/spec-write` Step 3 (requirements.md) updated to match the chosen scope. If "cross-cutting only": Step 3 only fires for genuinely cross-cutting concerns; per-FR creation path goes straight to `specs/frs/` without touching `requirements.md`.
- AC-STE-129.4: Add a gate-check probe (e.g., probe #29 `requirements-md-no-placeholder`) that detects the literal `<tracker-id>` or `[Feature Name]` strings in `specs/requirements.md` and surfaces them. Severity: warning.

## Technical Design

- Decision lives as a `<!-- scope: cross-cutting only -->` comment at the top of `templates/spec-templates/requirements.md.template`.
- `/spec-write` Step 3 prose (`skills/spec-write/SKILL.md`) is updated in the same commit — keep the skill prose and the template in sync.
- Probe #29 is a single grep — same shape as probe #23.

## Testing

- Test file: `tests/templates/requirements_md_scope.test.ts`.
- Cases: (a) `/setup`-generated `requirements.md` → probe #29 passes (no placeholders); (b) hand-edited `requirements.md` with `FR-1: [Feature Name]` → probe fails; (c) `/spec-write` run on a per-FR feature → does NOT modify `requirements.md` (the FR file is the destination).

## Notes

- Severity: low. Informational drift, not blocking. But the placeholder is actively misleading — it suggests a workflow that the architecture doesn't actually support post-M18.
- Per the "no users yet" memory, no migration flow needed — drop the placeholder cleanly.
- Pairs naturally with STE-128 (test-layout policy) — both are template-cleanup work surfaced by the same smoke run.
