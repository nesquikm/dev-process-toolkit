# STE-138: Fix CLAUDE.md Release Checklist drift (4→5 files; ## Structure → ## What's Inside) {#STE-138}

## Requirement

`CLAUDE.md:30-37` (`## Release Checklist`) is itself drifted on two counts. The release-process documentation that defines how to ship without bugs has its own bugs.

**G20 — file count is wrong.** The checklist enumerates four files but `specs/requirements.md:7` carries `**Latest shipped release:** **v1.37.1 ("Disambiguate")**.`, which also has to flip on every release. `grep -rln "v1\.37\.1"` finds the version string in CHANGELOG, README, both manifests, AND `specs/requirements.md`. Actual count is **five files**. The current sync at v1.37.1 is enforced by manual discipline, not by the documented checklist — a future release could miss `specs/requirements.md` and ship with a stale "Latest shipped release" claim in the spec.

**G21 — README anchor is wrong.** The checklist says "*update the 'Latest:' line in the `## Release Notes` section, and refresh any counts in the `## Structure` list*". `## Release Notes` exists at `README.md:113`. `## Structure` does NOT exist in README. `grep -n "^## " README.md` returns 8 sections: `Install as Plugin`, `Workflow`, `What You Get`, `What's Inside`, `Release Notes`, `Core Philosophy`, `Proven Across`, `Documentation`. The directory-tree the checklist actually targets lives under `## What's Inside` (`README.md:91`). A reader following the checklist would `Cmd-F` for "## Structure", find nothing, and silently miss the count refresh on release day.

Both bugs are in the same paragraph. Fix together.

## Acceptance Criteria

- AC-STE-138.1: `CLAUDE.md` `## Release Checklist` enumerates **five** files. The new entry covers `specs/requirements.md` — specifically the `**Latest shipped release:**` line at line 7. The entry is added in dependency order alongside the existing `CHANGELOG.md` / `README.md` entries (it depends on the version + codename being chosen).
- AC-STE-138.2: `CLAUDE.md` `## Release Checklist` references **`## What's Inside`** (not `## Structure`) when pointing at the README directory tree the checklist targets.
- AC-STE-138.3: A spot-check after the edit: bumping the file count to 5 catches a future regression where `specs/requirements.md:7` is left at the prior version after a release.
- AC-STE-138.4: A new gate-check probe (or test) asserts the consistency of the Release Checklist with reality:
  1. The version-string footprint (`grep -rln "vX\.Y\.Z"`) is enumerated at exactly five paths matching the checklist (or the test enforces the canonical five paths).
  2. Every `##`-anchor mentioned in the checklist resolves to a real heading in the referenced file.
- AC-STE-138.5: `bun test` passes; `/gate-check` clean.

## Technical Design

Two edits in `CLAUDE.md` § Release Checklist:

1. **Insert a new bullet for `specs/requirements.md`:**
   - Before:
     ```
     1. `plugins/dev-process-toolkit/.claude-plugin/plugin.json` — `"version"` field
     2. `.claude-plugin/marketplace.json` — `"version"` field in the plugin entry
     3. `CHANGELOG.md` — add a new `## [X.Y.Z]` section…
     4. `README.md` — update the "Latest:" line in the `## Release Notes` section, and refresh any counts in the `## Structure` list…
     ```
   - After:
     ```
     1. `plugins/dev-process-toolkit/.claude-plugin/plugin.json` — `"version"` field
     2. `.claude-plugin/marketplace.json` — `"version"` field in the plugin entry
     3. `CHANGELOG.md` — add a new `## [X.Y.Z]` section…
     4. `README.md` — update the "Latest:" line in the `## Release Notes` section, and refresh any counts in the `## What's Inside` list…
     5. `specs/requirements.md` — bump the `**Latest shipped release:** **vX.Y.Z ("Codename")**.` line at line 7.
     ```
   The "**these four files** MUST all be updated together" prose flips to "**these five files**".

2. **`## Structure` → `## What's Inside`** in the README pointer (item 4 above).

For AC-STE-138.4 (the durable-fix probe), implementation options:
- **(a)** Hand-coded check in a new `tests/release-checklist-consistency.test.ts`. Reads `CLAUDE.md`, parses the `## Release Checklist` numbered list, walks each entry, asserts the file path exists and the named `##` heading exists in the target file.
- **(b)** A `/gate-check` probe (#30 "release-checklist-consistency"). More involved; defer to the implementer's judgment.

Default to (a) — a single static test, easy to maintain.
