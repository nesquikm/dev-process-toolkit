# STE-143: Strip internal namespace from adapter docs + skill prose + docs (F8-Tier-2/3) {#STE-143}

## Requirement

Tier-2 + Tier-3 follow-up to M38 STE-137 (F8-Tier-1). The `STE-N` / `M<N>` references in shipped artifacts that don't directly copy into user projects are still de-facto noise the LLM has to read past on every invocation. ~715 references across two surfaces (adapters: 15; skills + docs: ~700). Per the user's stated principle (2026-04-27): "we need to ship 'clean' plugin."

This FR ships both tiers under one umbrella because they share the same mechanical-sweep approach (script-driven where it makes sense). The implementer can split the commit into two atomic changes if review pressure benefits.

**Tier 2 — adapter `.md` (15 references, hand-edited):**

- `adapters/jira.md` (3 hits) — same treatment.
- `adapters/_template.md` (1 hit).
- `adapters/_shared/dart/pubspec.yaml:description` — strip `(STE-103)` (latent surface — `publish_to: none` today, but cosmetic).

**Tier 3 — skill prose + docs (~700 references, scripted sweep):**

## Acceptance Criteria

- AC-STE-143.1: `adapters/linear.md`, `adapters/jira.md`, `adapters/_template.md` carry no literal `STE-\d+` or `M\d+` (post-edit). All references replaced with generic placeholders or stripped along with their surrounding parenthetical.
- AC-STE-143.2: `adapters/_shared/dart/pubspec.yaml` description carries no literal `STE-\d+`.
- AC-STE-143.3: `skills/*/SKILL.md` (14 files) — net STE-N count ≤ 5 (allow a small residue for genuinely load-bearing references; document each survivor in the FR's commit message). The (STE-NN AC-STE-NN.X) parentheticals are either stripped or moved to HTML comments per the implementer's chosen Option (a) or (b) — see Technical Design.
- AC-STE-143.4: `docs/*.md` (20 files) — net STE-N count ≤ 5 (same caveat: load-bearing residue is documented).
- AC-STE-143.5: A new test asserts the residual STE-N / M-N count never re-grows: `tests/shipped-prose-no-internal-namespace.test.ts` (or similar) reads each file in the listed surfaces and asserts the count of `\b(STE|M)\d+\b` matches a recorded ceiling.
- AC-STE-143.6: `bun test` is green after the sweep. Smoke-test runs through `/setup`, `/spec-write`, `/implement`, `/gate-check` end-to-end on a fixture project; no skill prose references a stripped ID by name in a load-bearing way.
- AC-STE-143.7: `/gate-check` clean across the live repo (no traceability-link probe regressions).
- AC-STE-143.8: A `git diff --stat` summary in the FR's commit message documents the per-file delta (e.g., `skills/gate-check/SKILL.md: -112 STE-N references`).

## Technical Design

**Phase A — Tier 2 (hand edits, ~30m).** Four files, ~15 references. Walk each file once, replace each STE-N with `LIN-67` / `<TICKET-ID>` / nothing depending on context. The historical `(STE-65)` parentheticals are usually safe to delete entirely — they document the FR that introduced a behavior, which is irrelevant to the user.

**Phase B — Tier 3 (scripted sweep, multi-hour).** Two implementation paths; the implementer picks at execution time:

**Option (a) — strip entirely.** A one-shot Bun walker (or `sed` invocation) removes every `(STE-NN)` and `(STE-NN AC-STE-NN.X)` parenthetical from `skills/` + `docs/`. Lose contributor traceability; gain a clean shipped surface. Probably right under the no-users principle.

**Option (b) — move to HTML comments.** Same walker rewrites `(STE-NN)` → `<!-- (STE-NN) -->`. Compromise; preserves traceability while reducing noise the LLM reads. The HTML comment renders as nothing in skill prose, so the user-visible surface is clean, but contributors looking at the markdown source still see the cross-reference.

Skeleton walker (option (a)):

```ts
import { readFileSync, writeFileSync, readdirSync } from "node:fs";
import { join } from "node:path";

const dirs = ["plugins/dev-process-toolkit/skills", "plugins/dev-process-toolkit/docs"];
const pattern = /\s?\((?:STE|AC-STE)-\d+(?:[\s,]*(?:AC-)?STE-\d+\.\d+)?\)/g;

for (const dir of dirs) {
  walk(dir, (path) => {
    const content = readFileSync(path, "utf8");
    const out = content.replace(pattern, "");
    if (out !== content) writeFileSync(path, out);
  });
}
```

(Pattern needs careful tuning — the audit lists 437 + 319 hits across diverse shapes; a single pass with the wrong regex can strip too aggressively.)

**Walk verification.** After running the walker, manually spot-check the top offenders (`gate-check/SKILL.md`, `patterns.md`, `implement-tracker-mode.md`) to confirm the prose remains coherent. The parentheticals must always be additive context, never load-bearing — if a sentence reads "as STE-NN's AC ensures …", that's load-bearing and the sweep needs to keep the reference (or rewrite the sentence).

**Phase C — guard regression.** New test `tests/shipped-prose-no-internal-namespace.test.ts`:

```ts
const surfaces = [
  "plugins/dev-process-toolkit/skills",
  "plugins/dev-process-toolkit/docs",
  "plugins/dev-process-toolkit/adapters",
];
const ceiling = { skills: 5, docs: 5, adapters: 0 };

// Walk each surface, count `\b(STE|M)\d+\b` (excluding fenced code blocks),
// assert <= ceiling.
```

The ceiling values are calibrated post-sweep (record actual residual counts as the new baseline).

## Testing

- **Phase A** — manual diff review of the four adapter files.
- **Phase B** — script the walker, run it, eyeball the top-offender files. Re-run skill prose mentally: does each skill still read coherently?
- **Phase C** — `bun test` green; the new ceiling test guards future drift.
- **Smoke-test** — full `/setup` → `/spec-write` → `/implement` → `/gate-check` cycle on a fixture project. The skills must still drive their flows correctly with internal references stripped.
- **`/gate-check` traceability probe** — clean across the live repo. The probe asserts every doc-link target exists; the sweep doesn't touch link targets, only parentheticals.

## Risks

- **Regex over-strip.** A walker pattern that's too greedy can damage prose. Mitigation: run on a single file first, eyeball, iterate. Walk reverts trivially via `git`.
- **Load-bearing references.** Some `(STE-NN AC-STE-NN.X)` parentheticals are NOT additive — they document a constraint the surrounding sentence depends on. The walker must skip these. Implementer judgment + manual top-offender review.
- **Skill behavioral regression.** Skills inline ID references for a reason — sometimes to point the LLM at a specific FR's design. Stripping them MIGHT degrade skill behavior. Mitigation: full smoke-test cycle post-sweep.

## Notes

The user's framing: Tier 2 is mechanical and small (~30m, 15 refs); Tier 3 is the bulk (~700 refs, multi-hour). Bundling them in one FR is a judgment call — they share the cleanup goal and the regression-guard test, but Tier 3 carries materially more risk than Tier 2. If the implementation review pressure benefits from splitting, two atomic commits within the same FR are fine.

Per the no-users-yet principle: shipped surface should be clean. Contributor traceability is preserved either via Option (b) (HTML comments) or via `git log` / `git blame` / archived FR files. The runtime cost (LLM reading past 700+ noise tokens per invocation) is the durable rationale.
