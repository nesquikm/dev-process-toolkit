# STE-172: /spec-review live-spec drift refresh hint when drift count crosses threshold {#STE-172}

## Requirement

Smoke #6 (Jira, 2026-04-30, v2.1.0) F4 surfaced four drifts in live cross-cutting specs after DST-5 archived (`requirements.md` § 6 traceability matrix empty; `technical-spec.md` § 1 directory-tree and § 3 API surface still describing the placeholder; `testing-spec.md` § 2 illustrative tree still referencing `.placeholder.test.ts`). `/spec-review` correctly flagged all four — verdict was `PASS WITH NOTES`. But the operator gets no nudge to act, so a future `/implement` invocation against M2 would inherit the drift, propagating across milestones.

STE-171 closes the **structured** part of the drift (the traceability row). The remaining three drifts in `technical-spec.md` and `testing-spec.md` are natural-language and out of scope for an autonomous skill — but a one-line nudge from `/spec-review` keeps the operator in the loop.

The hint must not fire on every audit. A single cosmetic drift during normal `/implement` churn is not actionable and would train operators to ignore the nudge. Threshold the hint at `drift_count >= 2` so it fires on accumulated drift, not on every minor `/implement` round.

Out of scope:
- The structured traceability-row drift — STE-171 fixes that directly.
- Auto-regenerating the natural-language sections — that's a deeper change owned by `/docs --full` (already shipped) and not a `/spec-review` concern.

## Acceptance Criteria

- AC-STE-172.1: `plugins/dev-process-toolkit/skills/spec-review/SKILL.md` carries a `drift_count >= 2 ⇒ one-line refresh hint` rule near the verdict-emit step. The threshold is documented inline (`>= 2 to suppress single-line cosmetic drifts during normal /implement churn`) so a future revisit understands the rationale.
- AC-STE-172.2: When the drift count is `>= 2`, `/spec-review` emits the literal line `Live-spec refresh suggested — N drift(s) found in cross-cutting specs; consider rerunning /spec-write before next /implement.` at the end of the report (after the verdict, before the closing summary).
- AC-STE-172.3: When the drift count is `0` or `1`, the hint line is absent. The verdict line itself is unchanged for both branches (still `PASS` / `PASS WITH NOTES` per existing logic).
- AC-STE-172.4: Integration test `plugins/dev-process-toolkit/tests/spec-review-drift-hint.test.ts` covers three branches: 0-drift fixture (no hint), 1-drift fixture (no hint), 4-drift fixture (hint with `N=4`).
- AC-STE-172.5: `bun test` green; `/gate-check` clean; `/spec-review M49` clean.

## Technical Design

Single-edit pass in `plugins/dev-process-toolkit/skills/spec-review/SKILL.md`:

- Locate the verdict-emit step (the section where `PASS` / `PASS WITH NOTES` / `FAIL` is rendered).
- Add a conditional: after the verdict line and before the closing `## /spec-review summary` block, when the audit's drift count is `>= 2`, emit the literal hint line with `N` substituted for the actual drift count.
- The drift count is the existing audit's deviation count (no new accounting needed). Verify the existing prose already tracks deviations as a numeric count; if it tracks them as a free-form list, derive the count via `length(drifts)`.

No code change required for the threshold logic — `/spec-review` is a prose skill; the rule is enforced by the LLM reading the SKILL.md.
