# STE-161: Skill-prose hygiene: scrub toolkit-meta tracker-ID refs {#STE-161}

## Requirement

Active skill prose under `plugins/dev-process-toolkit/skills/*/SKILL.md` is dotted with toolkit-meta tracker-ID cross-references — `STE-117 workspace binding`, `STE-118 milestone attachment`, `STE-87 active-side drift probe`, `AC-STE-110.4`, etc. Quick survey (2026-04-30):

- gate-check: 60 STE-N refs
- setup: 32
- spec-write: 19 unique
- brainstorm: 2
- (others not yet measured; estimate 100+ across the remaining skills)

Total: ~200+ inline tracker-ID refs in active skill prose.

These are jargon to downstream operators. The plugin will be installed in projects with their own tracker (or none); a Linear-workspace-internal ID is meaningless to anyone outside our org. The skill prose itself carries the rule that rendered output must never carry toolkit-meta `AC-STE-<N>.<M>` references — but the source prose the LLM reads at runtime is full of exactly those refs. The rule applies one level up: source prose, not just rendered output, should be free of toolkit-meta IDs. The right home for "this rule came from FR X" provenance is the FR file + commit message, not embedded inline in the skill the LLM reads at runtime.

Migration concerns are also real (tracker rename, ID renumber, workspace migration), but the downstream-operator argument dominates: the plugin's prose reads as Linear-internal jargon to anyone outside our org.

Out of scope:
- Historical references in `specs/plan/archive/`, `specs/frs/archive/`, `CHANGELOG.md`, and `specs/notes/` (frozen by design — describe past runs).
- Test-fixture names and adapter-internal capability keys in `adapters/_shared/src/*` and adapter test files (useful internal traceability; not operator-facing).
- Per-FR `## Notes` provenance lines like "Origin: STE-159 surfaced this" — those live in FR files, not skill prose, and are valid project-internal documentation.

## Acceptance Criteria

- AC-STE-161.1: All inline `STE-<N>` and `AC-STE-<N>.<M>` cross-references inside active `plugins/dev-process-toolkit/skills/*/SKILL.md` files are removed or rewritten in plain language. Provenance moves to FR files / commit messages. Verified by `grep -rE 'STE-[0-9]+|AC-STE-[0-9]+' plugins/dev-process-toolkit/skills/*/SKILL.md` returning zero matches (excluding placeholder forms `STE-<N>`, `<tracker-id>`, `AC-<tracker-id>.<N>` which are template markers).
- AC-STE-161.2: Same scrub applied to `plugins/dev-process-toolkit/templates/spec-templates/*.template` and `plugins/dev-process-toolkit/docs/*.md`. Operator-facing prose only; placeholder forms preserved.
- AC-STE-161.3: Adapter docs (`plugins/dev-process-toolkit/adapters/*.md`) scrubbed where the prose is operator-facing (rule statements, behavioral contracts); refs in adapter-internal sections (test-fixture names, internal capability keys) preserved.
- AC-STE-161.4: Spot-check: each of the top 5 affected skills (`gate-check`, `setup`, `spec-write`, `implement`, `spec-archive`) reads coherently end-to-end after the scrub — no dangling "see X" sentences with the X removed; the prose still names the rule by its plain-language description.
- AC-STE-161.5: `bun test` green; `/gate-check` clean; `/spec-review M47` reports no deviations.
- AC-STE-161.6: A `/smoke-test` run post-scrub confirms no behavioral regression in the canonical chain. **Operator-deferred** acceptable per `feedback_smoke_post_ship_retroactive.md` if M48 carries retroactive-validation framing; otherwise `[x]` at `/implement` time.

## Technical Design

Mechanical scrub, but high-volume (~200 edits). Strategy:

1. **Bulk pass with sed/perl** for the simplest cases — standalone `STE-N` references at end of sentences, parenthetical attributions like `(STE-117)`. A regex like `\bSTE-\d+\b` matched against sentence-end punctuation captures most.
2. **Per-occurrence rewrite** for contextual phrases — `STE-117 workspace binding` → `workspace-binding rule` or `the workspace-binding paragraph`. These need judgment to preserve readability.
3. **Spot-check pass** — `/implement`, `/gate-check`, `/setup`, `/spec-write`, `/spec-archive` skills read end-to-end. Any "see X" / "per X" sentences where X was removed get rewritten or deleted.
4. **`AC-STE-<N>.<M>` references** — almost always describe a behavioral rule. Replace with the rule's plain-language description. Drop the "AC-…" attribution entirely.

Affected files (estimated):
- `plugins/dev-process-toolkit/skills/*/SKILL.md` — primary target (~200 refs across ~12 skills)
- `plugins/dev-process-toolkit/docs/*.md` — secondary (operator-facing reference docs)
- `plugins/dev-process-toolkit/adapters/*.md` — partial (rule statements only)
- `plugins/dev-process-toolkit/templates/spec-templates/*.template` — operator-facing examples

NOT affected:
- `plugins/dev-process-toolkit/adapters/_shared/src/*.ts` — test names, internal capability keys
- `specs/plan/archive/`, `specs/frs/archive/`, `CHANGELOG.md`, `specs/notes/` — frozen historical prose
- FR `## Notes` sections (those are project-internal, not skill-runtime prose)

## Testing

- `/gate-check` — full project gate clean.
- `/spec-review M47` — no deviations.
- Manual: read each top-5 skill end-to-end and verify it still makes sense without the tracker-ID anchors.
- `/smoke-test` (operator-deferred per AC-STE-161.6) — canonical chain unchanged.

## Notes

- Origin: M46 spec-write (STE-160) surfaced the issue when the user questioned the use of `STE-117 workspace binding` in freshly authored prose. The survey then revealed the project-wide pattern.
- A bulk-edit script could mechanize most of the work, but care must be taken on `AC-STE-<N>.<M>` patterns where the AC describes a specific behavioral rule — those need plain-language substitution, not deletion.
- This FR does NOT change skill *behavior* — only prose. `/implement` can lean heavily on `bun test` + `/gate-check` + manual spot-check rather than full smoke-test for AC.1–.4. AC.6 (smoke regression check) is the safety net.
- The scrub creates a one-way prose convention: future skill-prose edits (in any milestone) should NOT reintroduce inline tracker-ID refs. A `/gate-check` probe enforcing this convention is a possible follow-up FR for a later milestone.
