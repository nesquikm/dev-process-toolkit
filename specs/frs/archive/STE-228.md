# STE-228: feat(skills): universal pre-commit branch gate; trunk-OK narrowed to ci {#STE-228}

## Requirement

Every commit-producing skill in the toolkit (`/setup`, `/spec-write`, `/spec-archive`, `/ship-milestone`, `/implement`) MUST refuse to commit on a protected trunk branch (`main`, `master`) unless the Conventional Commits type is in the narrow trunk-OK allowlist. When the gate fires, the skill proposes a branch name (via a per-skill `branchNameFor(...)` builder), prompts `[Y] create / [e] edit / [n] abort`, and runs `git checkout -b` on `Y`. The trunk-OK allowlist is **`["ci"]` only** — `chore` and `docs` no longer ship directly to trunk.

The motivation is concrete: protected-trunk repos (the toolkit's own repo, and the typical user repo configuration) reject pushes of commits made directly on `main`. Today `/spec-write`'s `chore(specs):` and `docs(specs):` commits land on `main` because `/implement` step 0.b″ documents `chore` and `docs` as trunk-OK; the local commit succeeds but the subsequent push fails. `/setup`, `/spec-archive`, and `/ship-milestone` have **no branch logic at all**, so they hit the same wall. This FR closes the gap by lifting `/implement`'s existing branch-proposal helper into a shared pre-commit gate that every commit-producing skill calls before staging, and narrows the trunk-OK list to `ci` only.

After STE-228 ships:

- A toolkit user (or this repo's maintainers) running any commit-producing skill on `main` is prompted to create a branch first; on `Y`, the skill creates the branch and proceeds; on `n`, the skill exits cleanly with zero side effects.
- Concurrent runs of literal-named skills (`/setup` bootstrap, `/spec-write` cross-cutting) coexist via collision-suffix probe (`-2`, `-3`, …).
- `/implement`'s existing branch-proposal flow is unchanged at the call-site level; only the trunk-OK constant narrows from `[chore, docs, ci]` to `[ci]`.
- Off-trunk runs are unaffected — the gate is a silent no-op when the current branch is not in `PROTECTED_TRUNKS`. The existing `feedback_branch_isolation` memory rule (stay on current branch for `/implement`) composes cleanly.

No new artifact type. No Schema L expansion. The gate is git-only, doesn't read CLAUDE.md, and works in fresh `git init` repos (`/setup` runs before any project context exists).

## Acceptance Criteria

- AC-STE-228.1: New shared helper `requireCommittableBranch(opts)` lives at `plugins/dev-process-toolkit/adapters/_shared/src/require_committable_branch.ts`. Signature accepts `{ commitType: string, proposedBranchName: string, currentBranch: string, isAutoMode: boolean }`; returns `{ outcome: "no-op" | "created" | "edited" | "declined", branchName?: string }`. Pure orchestration on top of the existing `isCurrentBranchAcceptable` from `branch_proposal.ts`; runs `git checkout -b` on `created`/`edited` outcomes (git probes injected for testability).
- AC-STE-228.2: Trunk-OK allowlist is the constant `TRUNK_OK_TYPES = ["ci"]` exported from the same module. `chore` and `docs` are removed from the list `/implement` step 0.b″ documents today; this is a behavior change that **WILL** break any workflow that relied on `chore(specs):` or `docs(...):` landing on `main`. Per `project_no_users_yet`, no migration shim ships.
- AC-STE-228.3: Protected trunk list is the constant `PROTECTED_TRUNKS = ["main", "master"]`. Test verifies both names trigger the gate; `develop`, `trunk`, `release/*`, and arbitrary feature-branch names do not.
- AC-STE-228.4: Each commit-producing skill ships a `branchNameFor(...)` builder, colocated with the skill source, returning the **clean** proposed branch name as a string (the gate's collision probe per AC-STE-228.11 appends suffix when needed):
  - `/setup` bootstrap → `chore/setup-bootstrap` (literal; no FR exists yet)
  - `/spec-write` new-FR run → existing `branch_template:` rendering via `buildBranchProposal` (delegates to current helper)
  - `/spec-write` cross-cutting-only run → `docs/specs-cross-cutting`
  - `/spec-archive` FR archive → `chore/archive-<tracker-id>` (e.g., `chore/archive-ste-227`)
  - `/spec-archive` milestone archive → `chore/archive-m<N>` (e.g., `chore/archive-m61`)
  - `/ship-milestone` → `release/v<X.Y.Z>`
  - `/implement` → unchanged (existing `branch_template:` flow)
- AC-STE-228.5: Gate prompt UX: when current branch is in `PROTECTED_TRUNKS` AND `commitType ∉ TRUNK_OK_TYPES`, the skill prints the proposed branch name and prompts `[Y] create / [e] edit / [n] abort`. `Y` runs `git checkout -b <branchName>`; `e` re-prompts for an edited name (validated against `^[a-z][a-z0-9._/-]*$`, rejected if it matches a protected name OR already exists locally/remotely); `n` exits cleanly with zero side effects (no commit, no staging mutation, no branch). The prompt always shows the **final** name the gate will use after collision probe — no surprise rename between `Y` and `git checkout -b`.
- AC-STE-228.6: When current branch is NOT in `PROTECTED_TRUNKS`, gate is a silent no-op — the existing user preference of staying on a feature branch is preserved (composes with the `feedback_branch_isolation` memory rule).
- AC-STE-228.7: Auto-mode / `claude -p` non-interactive default — the gate auto-creates the proposed branch (after collision probe) and emits a `branch_gate_default_applied` capability row in the closing summary, naming the created branch. Matches the existing draft-gate auto-apply pattern (STE-213/STE-220/STE-226). Detection contract: byte-checkable `<dpt:auto-approve>v1</dpt:auto-approve>` marker per M59's contract; legacy `Auto Mode Active` system-reminder detection is **not** used.
- AC-STE-228.8: Each commit-producing skill's closing summary surfaces the gate outcome via capability row (added to the static plain-language map at `/spec-write` § 7 and inherited by every skill that emits a summary):
  - `branch_gate_created` → `gate created branch <branchName> — commit will land there`
  - `branch_gate_edited` → `gate accepted operator-supplied branch name <branchName>`
  - `branch_gate_declined` → `gate declined — skill exited non-zero with zero side effects (no commit, no staging mutation)`
  - `branch_gate_default_applied` → `gate auto-created branch <branchName> (auto-mode / -p) — verify before push`
  - `branch_gate_remote_probe_skipped` → `gate could not probe remote branches (offline / no remote / fetch timeout) — local-only collision check used; verify name is unique on remote before push`
  - No row when the gate was a silent no-op (off-trunk path).
- AC-STE-228.9: New `/gate-check` probe `commit_producing_skill_branch_gate` parses each commit-producing skill's SKILL.md and refuses any `git commit` reference (literal, fenced or inline) that is not preceded (in document order) by a documented call to `requireCommittableBranch`. Severity: error. Catches future drift when a new skill is added or an existing skill grows a new commit site. Probe is colocated with the existing `auto_approve_marker_in_canonical_spawns` probe added in M59.
- AC-STE-228.10: On `n` decline at the gate, any staging the skill performed prior to the gate is rolled back via `git reset HEAD <paths>` (paths are the explicit list the skill staged — never `git reset --hard`). The skill exits non-zero so callers see the decline. Working-tree contents are untouched (no file edits reverted).
- AC-STE-228.11: **Collision-suffix probe.** Before `git checkout -b`, the gate probes for the proposed name's existence — local (`git branch --list`) AND remote (`git ls-remote --heads origin`, with a `git fetch` guarded by a 5s timeout). If the name is taken, append `-2` and increment until free (`-2`, `-3`, …). Probe results are cached per gate invocation. When the remote probe fails (offline / no remote / fetch timeout), the gate falls back to local-only check and emits a `branch_gate_remote_probe_skipped` capability row. The 60-char `MAX_BRANCH_LENGTH` cap still applies — the slug is truncated further to make room for the suffix; `EmptySlugError` raises if truncation would leave a trailing-hyphen branch name.

## Technical Design

**New module.** `plugins/dev-process-toolkit/adapters/_shared/src/require_committable_branch.ts` exports:

- `TRUNK_OK_TYPES: readonly string[] = ["ci"]`
- `PROTECTED_TRUNKS: readonly string[] = ["main", "master"]`
- `requireCommittableBranch(opts)` — main orchestration entry
- `findFreeBranchName(proposedName, deps?)` — collision-suffix helper (pure function over its inputs; git probes injected via deps)
- `CommitGateError` — NFR-10 canonical-shape exception class for invalid edited branch names

**Composition with existing helpers.** The new module imports `isCurrentBranchAcceptable` from `branch_proposal.ts` for the protected-trunk check. The existing `buildBranchProposal` is reused unchanged by `/spec-write` new-FR and `/implement` flows via their `branchNameFor(...)` builders — no rendering logic is duplicated.

**Per-skill `branchNameFor(...)` builders.** Colocated with each skill's source, returning the clean proposed branch name. Examples:

```
plugins/dev-process-toolkit/skills/setup/branch_name_for.ts
plugins/dev-process-toolkit/skills/spec-write/branch_name_for.ts
plugins/dev-process-toolkit/skills/spec-archive/branch_name_for.ts
plugins/dev-process-toolkit/skills/ship-milestone/branch_name_for.ts
```

`/implement` does NOT add a builder — its existing 0.b″ flow renders the branch via `buildBranchProposal` directly and continues to do so. The new gate is invoked at the existing call-site; only the trunk-OK constant changes.

**Call sequence in each skill** (idempotent):

1. Skill computes the would-be commit subject (e.g., `chore(specs): write FR STE-228`) and extracts the Conventional Commits `commitType` (`chore`).
2. Skill calls its `branchNameFor(...)` builder to get the clean proposed branch name.
3. Skill calls `requireCommittableBranch({ commitType, proposedBranchName, currentBranch, isAutoMode })`.
4. Gate checks `currentBranch ∈ PROTECTED_TRUNKS`. If not → returns `{ outcome: "no-op" }`. Skill proceeds with staging + commit on the current branch.
5. Gate checks `commitType ∈ TRUNK_OK_TYPES`. If yes → returns `{ outcome: "no-op" }`.
6. Gate calls `findFreeBranchName(proposedBranchName)` — local + remote probe, suffix increment, cache.
7. Gate prompts (or auto-applies on `isAutoMode`).
8. On `Y`/auto-apply/`e` → `git checkout -b <finalName>` → returns `{ outcome: "created" | "edited" | "default-applied", branchName }`. Skill proceeds with staging + commit on the new branch.
9. On `n` → returns `{ outcome: "declined" }`. Skill rolls back staging via `git reset HEAD <paths>`, exits non-zero.

**Branch-name canonical table** (single source of truth — referenced from each skill's SKILL.md):

| Skill | Run shape | Clean branch name | After collision |
|---|---|---|---|
| `/setup` | bootstrap | `chore/setup-bootstrap` | `chore/setup-bootstrap-2` |
| `/spec-write` | new-FR run, tracker mode | `<branch_template>` rendering | template + `-2` |
| `/spec-write` | new-FR run, `mode: none` | `<branch_template>` rendering | template + `-2` |
| `/spec-write` | cross-cutting-only | `docs/specs-cross-cutting` | `docs/specs-cross-cutting-2` |
| `/spec-archive` | FR archive | `chore/archive-<tracker-id>` | `chore/archive-<tracker-id>-2` |
| `/spec-archive` | milestone archive | `chore/archive-m<N>` | `chore/archive-m<N>-2` |
| `/ship-milestone` | release | `release/v<X.Y.Z>` | `release/v<X.Y.Z>-2` (rare) |
| `/implement` | FR run | `<branch_template>` rendering | template + `-2` |
| `/implement` | milestone run | `<branch_template>` rendering | template + `-2` |

**`/setup` carve-out.** The bootstrap commit at step 8b runs `requireCommittableBranch` BEFORE writing CLAUDE.md exists. The gate is git-only and doesn't read CLAUDE.md, so the ordering is fine. `/setup` runs `git init` early enough that `git ls-remote --heads origin` returns empty (no remote yet) — the gate falls back to local-only probe, which returns `chore/setup-bootstrap` clean on first run.

**Adapter neutrality.** The gate has no tracker awareness. Per-skill `branchNameFor(...)` builders pass tracker-derived names through opaquely; the gate sees only opaque strings. Linear/Jira-specific naming (e.g., `mikhaildrapoliuk/ste-228-...`) is **not** used — the toolkit's branch templates are independent of tracker `gitBranchName` outputs.

**`/implement` step 0.b″ supersession.** STE-202 AC-STE-202.5 (the original trunk-OK rule with `[chore, docs, ci]`) is superseded by AC-STE-228.2. The M61 plan file documents the supersession; STE-202's archived FR file is **not** edited (history-preserving — supersessions live forward in time, not retro-rewritten).

**Migration / no-users.** Per `project_no_users_yet`, the behavior change ships clean — no opt-out flag, no Schema L key, no deprecation period. The first place this lands is the toolkit's own `/spec-write` flows; subsequent FRs in this milestone (and beyond) will be authored on feature branches as a result.

## Testing

**Unit tests** (`plugins/dev-process-toolkit/adapters/_shared/src/require_committable_branch.test.ts`):

- Protected trunk + non-trunk-OK type → outcome `"created"`, `git checkout -b` called once with the proposed name.
- Protected trunk + `ci` type → outcome `"no-op"`, no git side effects.
- Non-protected branch (`feat/foo`) + any type → outcome `"no-op"`, no git side effects.
- Auto-mode default-apply → outcome `"created"`, `branch_gate_default_applied` capability row signaled in result.
- `n` decline (interactive simulation) → outcome `"declined"`, no git side effects.
- `e` edited name accepted (valid) → outcome `"edited"` with the operator-supplied name.
- `e` edited name rejected (invalid pattern, protected name, or existing) → re-prompt loop covered.

**Unit tests** for `findFreeBranchName`:

- Clean path (suffix=0): name unique locally and remotely → returns `{ name, suffixApplied: 0 }`.
- Local collision (suffix=2): `git branch --list` returns the proposed name → returns `{ name: <proposed>-2, suffixApplied: 2 }`.
- Local + remote collision (suffix=3): both probes return the proposed AND `-2` → returns `{ name: <proposed>-3, suffixApplied: 3 }`.
- Remote-probe failure (fetch timeout / no remote): falls back to local-only, returns `{ remoteProbeSkipped: true }`.
- Length cap respected: when suffix forces slug truncation past 60 chars, `EmptySlugError` raised before the rendered name reaches `git checkout -b`.

**Per-skill `branchNameFor(...)` unit tests** — one test file per skill, fixed inputs, snapshot output. Examples:

- `/setup` builder → `chore/setup-bootstrap` (no inputs).
- `/spec-write` cross-cutting builder → `docs/specs-cross-cutting` (no inputs).
- `/spec-write` new-FR builder → delegates to `buildBranchProposal`, mocked to verify pass-through.
- `/spec-archive` FR archive builder → `chore/archive-<tracker-id>` for input `{ trackerId: "STE-227" }` → `chore/archive-ste-227`.
- `/spec-archive` milestone archive builder → `chore/archive-m<N>` for input `{ milestone: 61 }` → `chore/archive-m61`.
- `/ship-milestone` builder → `release/v<X.Y.Z>` for input `{ version: "2.11.0" }` → `release/v2.11.0`.

**Integration smoke per skill.** New fixtures in `/smoke-test` Phase 2.X:

- Start on `main`, run skill via `claude -p` with the auto-approve marker, assert: (a) new branch was created with the documented name, (b) commit landed on the new branch (not on `main`), (c) `branch_gate_default_applied` capability row in stdout.
- One sub-fixture per commit-producing skill (5 fixtures total).

**`/gate-check` probe contract test.** New test `gate-check-commit-producing-skill-branch-gate.test.ts`:

- Positive case: a fixture SKILL.md that documents `requireCommittableBranch` before every `git commit` reference → probe passes.
- Negative case: a fixture SKILL.md with an undocumented `git commit` (no preceding `requireCommittableBranch` call) → probe fails with the file path + line number of the violation.

**Doc-conformance.** `tests/branch-gate-doc-conformance.test.ts` (new):

- Each commit-producing skill's SKILL.md MUST reference `requireCommittableBranch` at least once.
- The branch-name canonical table MUST appear in STE-228's FR file (this file) and be referenced (not duplicated) from each skill's SKILL.md.

**Existing `branch_proposal.test.ts`** stays green — we don't touch the underlying helper.
