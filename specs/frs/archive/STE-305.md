# STE-305: Relax STE-87 + STE-54 strict status probes via tolerance wrapper {#STE-305}

## Requirement

STE-87 (`/gate-check` probe #14 active-FR drift) and STE-54 (probe #8 archived-FR release verification) currently use strict equality against `status_mapping.in_progress` and `status_mapping.done` respectively. STE-302's loader, STE-303's `/setup` writer, and STE-304's tolerance wrapper change the underlying machinery — non-key statuses (e.g., QA moved a task to `In QA` between gate-check runs) now have an explicit handling path through `withTolerance(provider, specsDir)` instead of failing the gate.

This FR refactors both probes to route through the wrapper and adopt advisory degrade semantics for non-tty contexts (CI runs without an operator on the line). Two carve-outs preserve from STE-151's precedent: (a) genuine drift (mapped role mismatches expected; e.g., active FR sitting in `Backlog` when it should be `In Progress`) still fires GATE FAILED unchanged, (b) `mode: none` vacuous via the `local-no-tracker` sentinel (no behavior change).

This is the final FR in M79. After FR4 ships, the per-project tracker-config chain is complete: `tracker-config.yaml` is the source of truth, `/setup` writes it, the wrapper routes Provider operations through it, and the gate-check probes honor the wrapper's three-way return shape.

## Acceptance Criteria

- AC-STE-305.1: Probe #14 (`active_ticket_drift`, STE-87) refactor: replace the strict `status_mapping.in_progress` equality with a wrapper-aware check. Resolve `Provider` via the standard `buildProvider(...)` path; the wrapper is composed at provider-resolution time per STE-304's `withTolerance(...)` composition. Probe asserts the wrapper returns the expected role (`in_progress`) on read; non-key encounters route through the wrapper's three-way branching (interactive prompt under tty; advisory under non-tty per AC.5).
- AC-STE-305.2: Probe #8 (`ticket_state_drift`, STE-54) refactor: same pattern — replace strict `status_mapping.done` equality with wrapper-aware check. Probe asserts wrapper returns the `done` role on archived-FR reads; non-key encounters route through the wrapper.
- AC-STE-305.3: Genuine-drift preservation: when the wrapper returns a *mapped role that does not match the expected role* (e.g., active FR sitting in `Backlog` when `in_progress` was expected), both probes still emit GATE FAILED with the same row shape as today. This is the canonical drift case the original probes were built to catch; relaxation does NOT change behavior here. Truth table follows STE-151's precedent table — every prior failing case still fails.
- AC-STE-305.4: STE-151 carve-outs preserved verbatim — single-FR-clean exemption (active FR + ticket=done + plan has unchecked tasks) and fully-checked-single-FR exemption (active FR + ticket=done + plan fully checked + total > 0) continue to fire as today. The wrapper integration sits *inside* the existing carve-out branching, not replacing it. Carve-out tests in `tests/gate-check-active-ticket-drift.test.ts` continue to pass unchanged.
- AC-STE-305.5: Non-tty advisory routing: under `process.stdin.isTTY === false`, when the wrapper would prompt (non-key encounter), the probe instead emits an ADVISORY note (severity = warning, NOT GATE FAILED) naming the FR + observed status + the prompt-text-that-would-have-fired. Format: `<FR-id> sits in non-key status <observed>; expected role <role>. Re-run /gate-check interactively to resolve.` The capability row `tracker_status_advisory_non_tty` surfaces in the gate-check closing summary. Operator re-runs locally; CI does NOT block on tolerance prompts.
- AC-STE-305.6: Config-absent fallback (STE-304.6 alignment): when `readTrackerConfig(specsDir)` returns `null` (project hasn't run `/setup`'s tracker-config write yet), wrapper falls back to per-adapter `status_mapping` frontmatter. Probes retain strict-equality behavior on those projects — no advisory degrade, no wrapper prompt. The wrapper is a no-op until the project opts in by writing `specs/tracker-config.yaml`.
- AC-STE-305.7: `mode: none` vacuous — `LocalProvider.getTicketStatus` continues to return `local-no-tracker` sentinel; both probes short-circuit verbatim per existing STE-54.5 / STE-87.1 carve-outs. No behavior change for non-tracker projects.
- AC-STE-305.8: Capability rows added to `/spec-write` § 7 static capability map (and emitted by `/gate-check`'s closing summary): `tracker_status_advisory_non_tty` ("non-key status encountered under non-tty stdin — probe emitted advisory note instead of prompting; re-run /gate-check interactively to resolve"), `tracker_status_genuine_drift` ("active or archived FR's tracker status is a mapped role that does not match the expected role — drift, not tolerance — GATE FAILED"). Literal-token emission per STE-238 AC.3.
- AC-STE-305.9: Test updates: `tests/gate-check-active-ticket-drift.test.ts` and `tests/gate-check-status-probes-tolerant.test.ts` (new) cover: wrapper-mapped-match happy path; wrapper-mapped-mismatch → genuine-drift GATE FAILED; wrapper-null (known-non-key) under tty → prompt fires (mocked AskUserQuestion); wrapper-null under non-tty → ADVISORY note + `tracker_status_advisory_non_tty` row; wrapper-unknown under non-tty → ADVISORY with discovery hint; config-absent → strict fallback per AC.6; mode-none vacuous per AC.7; STE-151 carve-outs unchanged.
- AC-STE-305.10: Documentation update: `skills/gate-check/SKILL.md` probe #8 + probe #14 paragraphs gain a "Tolerance wrapper integration (STE-305)" sub-paragraph naming the FR + the wrapper module + the advisory-routing semantics. Existing strict-equality narrative is replaced (not appended) to avoid stale documentation drift.

## Technical Design

### Architecture

```
/gate-check probe #14 (active_ticket_drift, STE-87)         /gate-check probe #8 (ticket_state_drift, STE-54)
   │                                                                 │
   ▼                                                                 ▼
   buildProvider(...)  ←  composes withTolerance(provider, specsDir) at resolution time (STE-304)
   │
   ├─► wrapper.getTicketStatus(<tracker-ref>)
   │     ├─► mapped role == expected → pass-through (probe asserts match)
   │     ├─► mapped role != expected → genuine drift → GATE FAILED (existing shape)
   │     ├─► null (known-non-key) + tty → AskUserQuestion (force/skip/cancel)
   │     ├─► null (known-non-key) + non-tty → ADVISORY note + capability row
   │     ├─► "unknown" + tty → AskUserQuestion + discovery hint
   │     └─► "unknown" + non-tty → ADVISORY + discovery hint + capability row
   │
   └─► STE-151 carve-outs branch as before (single-FR-clean / fully-checked exemptions)
```

### Decision matrix (extends STE-151's table)

| FR status | Wrapper return | Stdin | Outcome |
|-----------|---------------|-------|---------|
| active | mapped role = `in_progress` | any | **pass** |
| active | mapped role != `in_progress` | any | **fail** (genuine drift; STE-151 carve-outs apply first) |
| active | `null` (known-non-key) | tty | wrapper prompt; probe outcome routed per operator choice |
| active | `null` (known-non-key) | non-tty | **advisory** (`tracker_status_advisory_non_tty`) |
| active | `"unknown"` (drift) | tty | wrapper prompt + discovery hint |
| active | `"unknown"` (drift) | non-tty | **advisory** + discovery hint |
| archived | mapped role = `done` | any | **pass** |
| archived | mapped role != `done` | any | **fail** (genuine drift) |
| archived | non-key | tty | wrapper prompt |
| archived | non-key | non-tty | **advisory** |
| any | `local-no-tracker` sentinel | any | vacuous (mode: none carve-out) |
| any | config-absent | any | strict-equality fallback (no wrapper effect) |

### Capability-row routing

The two new rows extend `/spec-write` § 7's static capability map and are emitted by `/gate-check`'s closing summary on every triggering probe run. The probe loop continues after an advisory — it does NOT halt on the first non-key encounter under non-tty (matches the user's "ADVISORY routing" choice from brainstorm).

### Order dependency

This FR requires STE-302 (loader), STE-303 (writer), STE-304 (wrapper) to be merged first. Until then, the wrapper isn't installed at provider-resolution time and the refactor would be a no-op (or, worse, break tests). Ship strictly last in M79.

## Notes

- **Why ADVISORY (not GATE FAILED) under non-tty.** Tolerance is a humans-in-the-loop concept — there's no safe default for force/skip/cancel. CI runs without an operator can't resolve a non-key encounter, so blocking would force re-runs without diagnostic value. Advisory surfaces the drift; operator resolves on the next interactive run. The genuine-drift path (mapped role mismatch) is unaffected — that still hard-fails, because it indicates real misalignment, not tolerance.
- **STE-151 precedent as template.** STE-151 already showed how to relax STE-87 selectively without breaking the M23 drift detection. This FR follows the same pattern at a layer below — the wrapper is the new selective-relaxation primitive; STE-151's carve-outs continue to live in probe-side logic.
- **Last in M79's ship order.** STE-302 → STE-303 → STE-304 → STE-305. Earlier FRs are pure additions (new module, new write path, new wrapper); this FR is the cutover that activates the tolerance machinery against the existing strict probes. Shipping it before STE-304 would leave the probes calling a non-existent wrapper; shipping it before STE-302 / STE-303 would leave the wrapper with no config to read.
- **No probe-shape change.** Both probes keep their existing identifiers (`active_ticket_drift`, `ticket_state_drift`), severity levels (error), and report shapes. The change is internal — the comparison logic switches from `Provider.getTicketStatus() == status_mapping.<role>` to `wrapper.getTicketStatus() → branch on three-way return`. External consumers (`/gate-check` JSON output, CI integrations) see no schema change.
