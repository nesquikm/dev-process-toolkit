# STE-213: /spec-write FR-draft acceptance gate default-apply under Auto mode / -p {#STE-213}

## Requirement

`/spec-write` already pre-authorizes its **commit gate** under Auto mode / `-p` non-interactive — the existing `spec_write_commit_default_applied` capability key emits a row when the auto-apply path fires. The **upstream FR-draft acceptance gate** ("Approve and proceed? Reply `y` to create the Linear/Jira issue and write the files") has no symmetric default-apply path: a `claude -p` invocation that reaches the draft gate stops there, exits 0 with stdout terminating mid-flow, and produces no FR file and no tracker issue.

**Findings bundled:** B-F1 from `/tmp/dpt-smoke-findings-2026-05-04-jira.md` (Run B v2.7.0).

**Symptoms observed in `dpt-test-project-jira`:**

1. The smoke driver's first `/spec-write` `-p` attempt completed cleanly through the draft phase but stopped at the `Approve and proceed?` interactive gate — no Jira issue created, no FR file written, exit 0 with stdout terminating mid-flow.
2. Retrying with explicit `PRE-AUTHORIZED (auto mode / -p non-interactive): … Default-apply 'y' to: the FR draft acceptance …` in the prompt body cleared the gate and the second attempt completed end-to-end.
3. Run #6 (2026-04-30 jira, v2.1.0) and Run #8 (2026-05-01 linear, v2.3.0) did not surface this — likely the prior driver prompt template carried implicit pre-authorization that was lost during the STE-185 heredoc-on-stdin migration, or the skill behavior changed when the FR-draft gate was hardened independently.
4. The asymmetry between the two `-p` paths (commit auto-applies, draft does not) is now load-bearing for any non-interactive `/spec-write` invocation.

## Acceptance Criteria

- AC-STE-213.1: Under Auto mode / `-p` non-interactive, the `/spec-write` FR-draft acceptance gate (the "Approve and proceed?" prompt that fires before tracker issue creation and FR file write) **default-applies `y`** instead of pausing for input. The default-apply path mirrors the existing `spec_write_commit_default_applied` cap-key contract for the commit gate. {#AC-STE-213.1}
- AC-STE-213.2: New capability key `spec_write_draft_default_applied` is registered in the `/spec-write` step 7 static plain-language map (`skills/spec-write/SKILL.md`). Rendered prose: `/spec-write FR-draft auto-approved (Auto mode / -p) — verify the draft via specs/frs/<id>.md before /implement`. The map row sits adjacent to the existing `spec_write_commit_default_applied` row so the two auto-apply contracts are co-located. {#AC-STE-213.2}
- AC-STE-213.3: Interactive runs (no `-p`, no Auto mode) preserve existing behavior — the `Approve and proceed?` prompt fires; `y`/`n`/`edit` choices are unchanged; no `spec_write_draft_default_applied` row emits on interactive `y` (matches the existing commit-gate behavior, where the row emits only on the auto-apply path). {#AC-STE-213.3}
- AC-STE-213.4: When Auto mode default-applies the draft gate, the row emits in the step 7 closing summary alongside any existing `spec_write_commit_default_applied` row; the operator sees both auto-applies on every quiet-mode run that creates a new FR. The existing >=100 byte summary floor is unaffected (the new row is additive, not replacing). {#AC-STE-213.4}
- AC-STE-213.5: New capability key `spec_write_draft_declined` is registered for the symmetric interactive case where the operator declined draft acceptance with `n`: rendered prose `/spec-write FR-draft acceptance declined — files not written; re-invoke to retry`. Vacuous on Auto mode / `-p` (default-apply takes the `y` branch); fires only on interactive `n` decline. Mirrors the existing `spec_write_commit_declined` shape. {#AC-STE-213.5}
- AC-STE-213.6: Test fixture: a `claude -p /spec-write …` invocation on an empty repo with all upstream questions pre-baked completes end-to-end (FR file written under `specs/frs/`, tracker issue created, commit landed) without manual prompt-body pre-authorization for the draft gate. The captured stdout summary contains both `spec_write_draft_default_applied` AND `spec_write_commit_default_applied` rows. {#AC-STE-213.6}

## Technical Design

**Where the draft gate lives.** `skills/spec-write/SKILL.md` § 0b "FR creation path" and § 4 "Review and confirm" describe the draft acceptance gate prose. The current text reads "Present the plan and ask for approval" / "Show the user what was written; Ask for approval before saving" — neither carries an Auto-mode / `-p` carve-out.

**Symmetric carve-out.** Add a one-paragraph default-apply rule at the draft gate, mirroring the commit-gate prose at § 7a:

> Under **Auto mode** / **`-p` non-interactive**, **default-apply `y`** at the FR-draft acceptance gate; the default-apply path emits a `spec_write_draft_default_applied` row in the Step 7 summary (interactive `y` does not emit the row). On `n` decline (interactive only), emit a `spec_write_draft_declined` row so the operator knows the draft was not written.

**Capability-map registration.** Append two rows to the static plain-language map in § 7:

| `spec_write_draft_default_applied` | `/spec-write FR-draft auto-approved (Auto mode / -p) — verify the draft via specs/frs/<id>.md before /implement` |
| `spec_write_draft_declined` | `/spec-write FR-draft acceptance declined — files not written; re-invoke to retry` |

**Detection.** Auto-mode detection follows the same path as the commit gate's existing default-apply (the system-reminder block carries an `Auto mode active` signal; `-p` non-interactive carries a separate runtime flag). No new detection logic; reuse the commit-gate's branch.

**Smoke driver simplification.** The `/smoke-test` driver's `/spec-write` heredoc body currently injects `Default-apply 'y' to: the FR draft acceptance` as a workaround. Once this FR ships, the workaround can be removed (separate cleanup, not part of this FR's ACs).

## Testing

- Integration test: a `claude -p /spec-write …` invocation against a throwaway test project completes end-to-end; capture stdout; assert both `spec_write_draft_default_applied` and `spec_write_commit_default_applied` rows appear.
