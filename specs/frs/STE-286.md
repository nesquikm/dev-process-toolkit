# STE-286: Smoke fixture group 8 ‚ÄĒ STE-285 hooks runtime regression {#STE-286}

## Requirement

STE-285 (M71) shipped 4 opt-in Process-category hooks (`pre-commit-gate-check`, `pre-pr-spec-review`, `pre-spec-write-brainstorm-reminder`, `pre-commit-tdd-orchestrator`) with per-hook unit tests under `templates/hooks/__tests__/`. Per-hook unit tests cover script-level behavior, but the FULL pipeline ‚ÄĒ `settings.json` write via `installHooks` ‚Üí Claude Code harness reads `.claude/settings.json` ‚Üí PreToolUse hook fires on Bash ‚Üí script reads `$CLAUDE_SESSION_FILE` ‚Üí NFR-10 refusal emitted to stderr ‚Üí harness surfaces refusal to the model ‚ÄĒ is **not** exercised at E2E level. STE-283's symmetric contract (TDD Orchestrator) already has runtime evidence via `/smoke-test` fixture group 7 (`STE-225 TDD orchestrator forks runtime`, per STE-231). STE-285 lacks the symmetric probe.

This FR adds **`/smoke-test` Phase 2.X fixture group 8** ‚ÄĒ *STE-285 hooks runtime regression* ‚ÄĒ that exercises install + runtime refusal for all 4 hooks via one shared marker-stripped `claude -p` child spawn with 4 sequenced scenarios + `$CLAUDE_SESSION_FILE` rotation. Bundles a small STE-285 scope expansion: `/setup --hooks` accepts a non-interactive preselect flag (`--hooks=all` and `--hooks=<comma-list>`) so the smoke driver can install hooks without hitting the multi-select default-off.

Operator-acknowledged residual risk: same as STE-285 itself ‚ÄĒ Claude Code's `$CLAUDE_SESSION_FILE` format is internal and a format change breaks every hook. The shared `_lib/session.sh` helper is the single insulation point.

## Technical Design

### Files touched

- `plugins/dev-process-toolkit/skills/setup/SKILL.md` ‚ÄĒ document `--hooks=all` / `--hooks=<comma-list>` preselect flag semantics.
- `plugins/dev-process-toolkit/skills/setup/install_hooks.ts` ‚ÄĒ add `parsePreselectFlag(arg): { all: boolean; names: string[] }` (or similar) exporting a deterministic name parser; reject unknown names per `HOOK_REGISTRATIONS` keys.
- `.claude/skills/smoke-test/SKILL.md` ‚ÄĒ new Phase 2.X fixture group 8 block following existing group-7 prose shape.
- New test file: `plugins/dev-process-toolkit/skills/setup/__tests__/setup_hooks_preselect.test.ts`.

### Preselect flag shape

`/setup --hooks=all` ‚áí all 4 names from `HOOK_REGISTRATIONS`. `/setup --hooks=name1,name2,...` ‚áí filtered list, rejected on first unknown name with NFR-10 stderr `Refusing: unknown hook name '<X>' ‚ÄĒ known names: <list>`. `mode: none` + tracker-mode behave identically (flag is install-side, tracker-agnostic). Idempotent re-run with the same names = no-op (per existing `installHooks` semantics).

### Smoke driver flow (fixture group 8)

1. Phase 2 step 3a (after `/setup` invocation in the canonical chain): re-invoke `/setup --hooks=all` in the same test-project child via a separate `claude -p` to materialize hooks in `.claude/settings.json`. (Cheaper than re-doing full setup; the flag-only path is fast.)
2. Read `<test-project>/.claude/settings.json`, assert 4 entries with correct shapes (AC.3).
3. Spawn marker-stripped `claude -p` child with 4 sequenced scenarios; rotate `$CLAUDE_SESSION_FILE` per scenario.
4. For each scenario: capture log, run AC.5 grep assertions, append `STE-285 runtime regression: <name>` to findings on failure.
5. Phase 2.X summary line `STE-285 runtime check: PASS|FAIL` appended to per-tracker summary.

### Anti-probe-collision

The 4-scenario child spawn is described in **prose only** inside `.claude/skills/smoke-test/SKILL.md` (not in a fenced `bash` block) to avoid the `auto_approve_marker_in_canonical_spawns` gate-check probe collision (STE-226 / STE-231 group-5b precedent). Driver constructs the actual spawn at runtime via a Bash heredoc.
