# STE-410: Probe #69: legacy-monolith advisory row on unmanaged trees {#STE-410}

## Summary

The staleness probe deliberately stays silent on projects whose main config file the toolkit does not manage, so a legacy project that predates bootstrap gets no signal that a purpose-built migration exists for its layout. This FR carves one narrow, read-only exception: when the retired monolithic layout is unmistakably present, the probe emits a single advisory note naming the remedy chain — bootstrap first, then upgrade. The probe stays warn-only, never fails the gate, and remains silent on every other unmanaged tree.

## Requirement

Facet 1 of the 2026-07-21 gist: on a tree whose `CLAUDE.md` carries no toolkit-managed section, probe #69 (`upgrade_staleness`) emits zero rows and returns clean without walking a single detector — the STE-394 Step 0 short-circuit, a deliberate consumer-safety decision (most registry detectors key on existsSync-shaped state and would render false staleness rows on foreign trees).

But the monolithic spec layout is DPT's own retired default (pre-1.16.0), and its detector's two-limb conjunction — live `### FR-N:` sections in `specs/requirements.md` AND no per-FR content under `specs/frs/` — is DPT-specific evidence. A monolith-only carve-out therefore does not reopen the false-positive class the short-circuit exists to prevent. Today an unmanaged legacy tree gets no signal from any surface, and `/gate-check` is where operators actually look.

The hint must route to `/setup` first because `/upgrade` Step 0 (rightly, STE-391) refuses unmanaged trees; a remedy naming `/upgrade` alone would bounce.

## Acceptance Criteria

- AC-STE-410.1: `adapters/_shared/src/upgrade_staleness.ts` exports the byte-pinned literal `LEGACY_MONOLITH_HINT` = "legacy monolithic specs detected — run /dev-process-toolkit:setup to bootstrap, then /dev-process-toolkit:upgrade to split specs/requirements.md into per-FR files." and a test asserts the literal verbatim. {#AC-STE-410.1}
- AC-STE-410.2: on an unmanaged tree, `runUpgradeStalenessProbe` runs ONLY the monolith-split sniff (the rest of the registry stays unwalked); when it applies ⇒ exactly one row shaped `monolith-split (1.16.0): <evidence joined by "; ">` with `notes = [row, LEGACY_MONOLITH_HINT]`; `violations` stays empty — the probe never emits GATE FAILED. {#AC-STE-410.2}
- AC-STE-410.3: unmanaged tree + sniff not applying ⇒ zero rows and empty notes, byte-identical to today — the AC-STE-394.4 vacuity-in-both-directions contract is preserved. {#AC-STE-410.3}
- AC-STE-410.4: managed-tree behavior is byte-identical to today — full registry walk, `UPGRADE_STALENESS_REMEDY` closing line, and `LEGACY_MONOLITH_HINT` is never emitted on the managed path. {#AC-STE-410.4}
- AC-STE-410.5: a sniff throw on an unmanaged tree degrades to the clean zero-row return — silence, not a row; foreign trees never see noise, and the purity contract holds on the error path. {#AC-STE-410.5}
- AC-STE-410.6: the gate-check SKILL probe #69 entry documents the carve-out (Step 0 wording amended); the AC-STE-394.7 pinned-surface sweep runs (docs-pin meta-tests + fixtures updated); the probe COUNT is unchanged — no new probe number. {#AC-STE-410.6}

## Technical Design

**`upgrade_staleness.ts`.** Export `LEGACY_MONOLITH_HINT` (AC-STE-410.1 wording). Import `{ monolithSplit }` from `./migrations/monolith_split`. In `runUpgradeStalenessProbe`, the Step 0 branch becomes:

- `!isToolkitManaged` ⇒ `try { const d = monolithSplit.detect(projectRoot); if (d.applies) { const row = `<id> (<introduced_in>): <evidence joined>`; return { rows: [row], notes: [row, LEGACY_MONOLITH_HINT], violations: [] }; } } catch { /* degrade to silence */ }` then the existing clean return. The row renderer is shared with the managed path (extract the template into a small helper) so the byte shape stays uniform.

**`skills/gate-check/SKILL.md`** probe #69 entry: amend the Step 0 sentence — zero rows on unmanaged trees, EXCEPT the legacy-monolith sniff, which renders exactly one warn-only NOTES row closed by the `LEGACY_MONOLITH_HINT` literal; severity stays notes, never GATE FAILED.

Depends on STE-409 (content-aware limb) so the sniff is scaffold-proof.

## Notes

This deliberately and narrowly amends the STE-394 Step 0 pinned decision; the carve-out is exactly one registry entry, chosen because its evidence conjunction is toolkit-specific. Cheapness is preserved — one pure detector call on the unmanaged path, no registry walk, no git, no network.
