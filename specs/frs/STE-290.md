# STE-290: Fix hook session.sh — read transcript_path from stdin JSON instead of $CLAUDE_SESSION_FILE env var {#STE-290}

## Requirement

STE-285 (M71, v2.21.0 "Honored", shipped 2026-05-13) introduced the byte-checkable toolkit-contract enforcement layer via 4 plugin-bundled hooks (`pre-commit-gate-check`, `pre-pr-spec-review`, `pre-commit-tdd-orchestrator`, `pre-spec-write-brainstorm-reminder`) sharing `templates/hooks/_lib/session.sh::require_skill_tool_use`. That helper reads `$CLAUDE_SESSION_FILE` (env var) to locate the active session log, then greps it for the required Skill tool_use marker.

**The Claude Code harness never sets `$CLAUDE_SESSION_FILE`.** Empirical probe on 2026-05-14 (via a one-off `/tmp/dpt-hook-probe` plugin run under raw `claude`, `cca`, and `claude-st` launchers across iTerm2 and cmux terminal hosts) confirmed: the harness passes session info on **stdin as JSON** carrying `{session_id, transcript_path, cwd, permission_mode, hook_event_name, tool_name, tool_input, tool_use_id, prompt}`. The `transcript_path` field is the active session log path and correctly tracks `CLAUDE_CONFIG_DIR`. STE-286's empirical-finding note (recorded 2026-05-13) had documented the parent-env clobbering issue but only worked around it for smoke fixtures; this FR ships the proper fix.

Consequence of the unfixed bug: `require_skill_tool_use` always hits its "fail-open when unset" branch and exits 0 silently. The byte-checkable enforcement layer has been **inert since v2.21.0 shipped** (~24 hours ago, no real production window). Every contract claim made by STE-285 / STE-289 has been runtime-vacuous despite per-script unit tests passing — the tests verified the env-var path while production used the never-set env var.

This FR rewrites `_lib/session.sh` as a Bun TS library + 4 per-hook TS modules invoked via 2-line bash shims, ports the 4 per-script tests to inject stdin payloads per the actual contract, and adds 5 dedicated unit test files for failure isolation. Forward-only per `project_no_users_yet` — no migration logic.

### session.ts API contract (AC.1 detail)

```ts
export type HookPayload = {
  session_id: string;
  transcript_path: string;
  cwd: string;
  permission_mode?: string;
  hook_event_name: "PreToolUse" | "UserPromptSubmit" | "Stop" | string;
  tool_name?: string;
  tool_input?: { command?: string; description?: string; [k: string]: unknown };
  tool_use_id?: string;
  prompt?: string;
};

export function parseHookPayload(stdin: string): HookPayload | null;
export function requireSkillToolUse(skill: string, hook: string, payload: HookPayload): { found: boolean };
export function emitNFR10(verdict: "Refusing" | "Reminder", why: string, how: string, skill: string, hook: string): void;
```

### Per-hook entrypoint pattern (AC.2 detail)

```ts
// pre-commit-gate-check.ts
import { parseHookPayload, requireSkillToolUse, emitNFR10 } from "../session.ts";

const stdin = await Bun.stdin.text();
const payload = parseHookPayload(stdin);
if (!payload) process.exit(0);                          // fail-open: empty / unparseable

const cmd = payload.tool_input?.command ?? "";
if (!/^git commit\b/.test(cmd)) process.exit(0);        // command-pattern guard

const { found } = requireSkillToolUse(
  "dev-process-toolkit:gate-check",
  "pre-commit-gate-check",
  payload,
);
process.exit(found ? 0 : 1);
```

The brainstorm-reminder variant is an advisory (Reminder verdict, exits 0 even on miss). The tdd-orchestrator variant also runs `git diff --cached --name-only` after command-pattern match to filter for FR-related staged files.

### Bash shim pattern (AC.3 detail)

```bash
#!/usr/bin/env bash
exec bun run "${CLAUDE_PLUGIN_ROOT}/templates/hooks/_lib/hooks/pre-commit-gate-check.ts"
```

Two lines per hook script. Path + `.sh` extension preserved so `bundled-hooks-shape.test.ts` (STE-289 AC.6) keeps passing — it asserts the command path resolves to an existing `.sh` file.

### Reversal of STE-285's env-var contract assumption

STE-285 inherited the `$CLAUDE_SESSION_FILE` read pattern from an early draft of the Claude Code hook docs (or from an internal assumption — the contract isn't documented at `code.claude.com/docs/en/plugins-reference.md`). STE-286's empirical-finding note (2026-05-13) documented the env-var was being clobbered by the harness on subprocess spawn, but the workaround only covered smoke fixtures, not production hook firings.

This FR ships the proper fix using the verified-2026-05-14 stdin-JSON contract. The new architecture is **launcher-agnostic and config-dir-agnostic** — same code path under raw `claude`, `cca`, `claude-st`, iTerm2, cmux. `transcript_path` correctly tracks the active `CLAUDE_CONFIG_DIR` because the harness pre-resolves it.

Each integration test drives the bash shim end-to-end via `Bun.spawn({ stdin: JSON.stringify(payload), cmd: ["bash", "<wrapper>.sh"] })` — verifies the bash→TS handoff works. Reduced to 2 cases per file (happy + refusal). The `$CLAUDE_SESSION_FILE` / `$CLAUDE_STAGED_FILES` / `$CLAUDE_USER_PROMPT` env-var injection paths are removed.

- Adding a temp git repo fixture for the tdd-orchestrator staged-file test costs ~50-100ms per test case. Acceptable for unit-test latency budget.
- Bun.spawn stdin contract is stable across Bun versions; no compatibility risk.
