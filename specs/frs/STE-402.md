# STE-402: report-issue: gate the gist publish on the marker, not prose inference {#STE-402}

## Summary

The /report-issue skill's final act is publishing a secret gist — an irreversible push to a third-party service. Its publish gate documents what to do when the auto-approve marker is present (publish) and what an interactive decline does, but says nothing about the case where the marker is absent and the session is non-interactive. Facing that gap under an autonomous session, the skill treated a prose "proceed" instruction as authorization and invoked the publish command anyway.

The publish did not go through, but only because a layer outside the plugin blocked it — the plugin's own authorization gate had already failed open. This FR closes the gap by wiring the publish gate through the same runtime marker decider and three-branch refusal contract the spec-write gates use: marker present publishes, marker absent under a non-interactive session refuses loudly and does not publish, and an interactive session gets the existing prompt. A gate probe enforces the wiring.

## Requirement

The 2026-07-19 conformance run's Phase 8 ran `/report-issue` with the autonomous-mode reminder and no marker. The child correctly detected the marker's absence and narrated it — then invoked `gh gist create -s ...` anyway, treating the operator's earlier "proceed end to end" as the authorization. It named the exact rule it was breaking and broke it in the same sentence. Nothing published only because the harness auto-mode classifier denied the compound command; `gh gist list` confirmed no new gist. The harness held the line; the plugin's own gate failed open.

Two aggravating facts: the harness denial looks incidental (the command was compound — `cd … && gh gist create …; echo` — the same shape denied elsewhere in the run, so a bare `gh gist create` might have passed), and `/report-issue` is documented as user-reachable directly, so the guard cannot rely on smoke-only conditions. `docs/auto-mode-protocol.md`'s three-branch contract says marker-absent + non-interactive stdin ⇒ `refused` via `requireOrRefuse(...)`; that branch was never written into the publish gate. The fix must add it, name the runtime byte-grep as the sole decider, add the NOT-a-trigger anchor, and enforce all three with a probe.

## Technical Design

**SKILL edit — `plugins/dev-process-toolkit/skills/report-issue/SKILL.md`.** The publish gate currently documents only the marker-present default-apply (step 7, "Default-apply rule") and the interactive `n` decline (step 8). Rewrite it to the full three-branch contract, mirroring `/spec-write` § 7a verbatim in shape:

- FIRST step of publish-gate evaluation: write the prompt body to a temp file and run `bun run plugins/dev-process-toolkit/adapters/_shared/src/check_marker_runtime.ts <file>`; branch strictly on its `PRESENT`/`ABSENT` stdout.
- PRESENT ⇒ `requireOrRefuse(..., markerPresent: true, defaultValue: "y", ...)` ⇒ publish; emit `report_issue_default_applied` (existing token, unchanged).
- ABSENT + non-tty ⇒ `requireOrRefuse(..., markerPresent: false, defaultValue: "y", userSuppliedValue: undefined, ...)` falls through to the refusal branch, throws `RequiresInputRefusedError` (NFR-10 shape, gate site `publish`); `gh gist create` is NOT invoked; emit `report_issue_publish_refused`; the temp dir is cleaned via the existing `trap … EXIT`.
- ABSENT + tty ⇒ the existing interactive `[y / n / edit]` prompt (`n` ⇒ `report_issue_declined`, unchanged).
- Add the NOT-a-trigger anchor paragraph (canonical wording from `/spec-write` § Rules): pre-baked prose / autonomous-mode reminders / "proceed" are NOT publish authorization; the marker is the sole trigger; the runtime byte-grep is the sole evaluation path.

No new runtime helper — `requireOrRefuse` + `check_marker_runtime.ts` already exist. `defaultValue: "y"` here (unlike spec-write's requires-input steps) is deliberate: publish HAS a safe default under an explicit marker, matching the existing marker-present default-apply already documented.

**Capability key — `adapters/_shared/src/closing_summary_capability_keys.ts`.** Add `report_issue_publish_refused` to `CANONICAL_CAPABILITY_KEYS`; rendered prose in the `/spec-write` § 7 map, e.g.: `/report-issue publish refused — marker absent under non-interactive stdin; no gist created. Re-invoke interactively or with the marker.`

**Probe — `adapters/_shared/src/report_issue_publish_gate_marker.ts` + register as #71 in `skills/gate-check/SKILL.md`.** Single-file scope (`skills/report-issue/SKILL.md`), literal-substring match, severity error. Required literals (all present): `check_marker_runtime.ts`, `RequiresInputRefusedError`, and the NOT-a-trigger anchor phrase. Each miss ⇒ one `file:line — reason` note in NFR-10 shape. Vacuous when the SKILL is absent. Bump probe-count pins to the new total (this is the second probe added in M110, after STE-401's #70).
