# STE-230: feat(skills): spec-research subagent for context retrieval in /brainstorm and /spec-write {#STE-230}

## Requirement

A new read-only Haiku-powered subagent (`spec-researcher`) and paired forked skill (`spec-research`) ship with the plugin so `/dev-process-toolkit:brainstorm` and `/dev-process-toolkit:spec-write` can pull *targeted* context from prior FRs (active + archived) without polluting the parent skill's context window. Given a topic description, the subagent reads `specs/frs/*.md` (active) and `specs/frs/archive/*.md` (history), selects relevant entries by topic match, and returns a **fixed-shape 3-section block** (Related FRs, Prior Decisions, Reusable ACs / Patterns) capped at Ōēż 25 lines, prefixed with a "may be stale" banner. The parent skill receives only the summary block ŌĆö never the raw archive contents.

The motivation is concrete: prior decision rationale lives in archived FR `## Technical Design` sections and dies on archive (the archive directory is currently excluded from `/gate-check`, `/docs`, `ac-lint`, and `/spec-review`). New brainstorm sessions re-litigate tradeoffs already settled; new FRs miss reusable AC patterns from cousin features. Industry SDD tooling (GitHub Spec Kit, OpenSpec) addresses this either via a project-level "constitution" doc or by lifting deltas into living specs at archive time. This FR takes a third path: **on-demand topic-aware retrieval via a context-isolated subagent**, no schema churn, no new artifact, archive stays a passive audit trail.

After STE-230 ships:

- A user running `/brainstorm` finishes Step 1 (clarify the problem); before Step 2 (propose approaches), the parent skill spawns the `spec-research` forked skill with the clarified topic. The subagent returns a 3-section block; Step 2's proposed approaches reference real precedents instead of hypothetical tradeoffs.
- A user running `/spec-write` on a new FR ŌĆö after ┬¦ 0b "Build canonical frontmatter", before "AC prefix" ŌĆö gets the same retrieval. The subsequent `## Technical Design` interview asks targeted questions like "FR-X chose approach A here ŌĆö follow or diverge?" instead of starting from zero.
- The `--no-tech` flag short-circuits the `/spec-write` invocation (no technical-design interview to inform; vacuous). The brainstorm path is also skipped under `--no-tech` since Step 2 itself is skipped.
- The subagent is read-only (`Read, Grep, Glob`), runs on Haiku, and returns Ōēż 25 lines / ~500 tokens. It does not write files, does not modify state, and is not wired into any skill other than `/brainstorm` and `/spec-write`.

The skill writes nothing under VCS ŌĆö it is a pure read-side query and is therefore exempt from STE-228's `commit_producing_skill_branch_gate` probe (added to the `NON_COMMIT_PRODUCING_SKILLS` allowlist alongside `report-issue`).

## Technical Design

**New subagent.** `plugins/dev-process-toolkit/agents/spec-researcher.md` ŌĆö single file. Frontmatter (canonical shape):

```yaml
---
name: spec-researcher
description: Topic-aware retrieval of related FRs (active + archived) for /brainstorm and /spec-write. Read-only research; returns a fixed-shape Ōēż25-line block. Invoked exclusively via the spec-research forked skill; not for direct delegation.
tools: Read, Grep, Glob
model: haiku
---
```

System prompt directs the subagent to:

1. Read the topic from the prompt body (`$ARGUMENTS` substituted by the parent skill).
2. Glob `specs/frs/*.md` (active) and `specs/frs/archive/*.md` (history).
3. Scan each file's `## Requirement` and `## Technical Design` sections for topic overlap (keyword + title match; LLM-quality ranking).
4. Select the top Ōēż 3 candidates.
5. Emit the banner + 3-section block per AC-STE-230.3 + AC-STE-230.4.
6. Self-truncate to Ōēż 25 lines per AC-STE-230.5; append `- (ŌĆ” <K> more truncated)` if any section overflowed.

**New skill.** `plugins/dev-process-toolkit/skills/spec-research/SKILL.md` ŌĆö single file. Frontmatter:

```yaml
---
name: spec-research
description: Internal research fork ŌĆö invoked exclusively by /brainstorm and /spec-write via context:fork pairing with the spec-researcher subagent. Returns related-FR retrieval as a Ōēż25-line block. Do not invoke directly.
context: fork
agent: spec-researcher
user-invocable: false
argument-hint: '<topic description>'
---
```

Skill body is the prompt sent to the subagent ŌĆö short, with the 3-section output contract restated for the LLM running inside the forked context.

**Composition with existing helpers.** None required. The skill stands alone ŌĆö no `requireCommittableBranch` (per AC-STE-230.10), no `Provider.sync`, no `buildFRFrontmatter`. It is a pure read-side primitive parallel to the existing `tdd-implement` / `tdd-write-test` / `tdd-refactor` forked skills (STE-225's pattern).

**Subagent context isolation.** Per `code.claude.com/docs/en/sub-agents`, subagents run in their own context window with a custom system prompt and tool restrictions. The parent skill's conversation history does not carry over; the subagent receives only the skill body (rendered with `$ARGUMENTS`) plus environment essentials (CLAUDE.md, working directory). On exit, only the summary block returns to the parent ŌĆö the subagent's intermediate file reads, greps, and reasoning are discarded. This matches the design intent: the parent skill's context window stays small.

**Output token budget.** Ōēż 25 lines ├Ś ~20 tokens/line Ōēł 500 tokens output cap. Combined with Haiku pricing (~$0.001/call at expected input ~5K tokens), per-invocation cost is ~$0.001ŌĆō0.005. Every `/brainstorm` and `/spec-write` (sans `--no-tech`) adds one such call.

**No-memoization rationale.** Each `/spec-write` and `/brainstorm` invocation is typically a new topic; cache hit rate would be low. A disk cache introduces staleness (new archive between calls invalidates entries); a session-scoped in-process cache would require state primitives Claude Code skills don't expose. The fresh-call cost is ~$0.001ŌĆō0.005 per run ŌĆö too low to motivate the cache machinery. If the agent fires twice on the *same* topic in real workflows, revisit; not now.

**Stale-rationale handling.** Archived FRs may carry decisions that contradict current `requirements.md` / `technical-spec.md` / `testing-spec.md`. The "may be stale" banner is a deterministic hint to the parent skill's LLM, not a content guarantee. The decision to apply or diverge belongs to the user + parent skill at design time, not to the subagent. No gate-probe on contradiction detection ŌĆö shape-only validation per AC-STE-230.12. This is a deliberate refinement during brainstorm: the original "## Cautions" section in the output contract was dropped on user feedback ("just instruct the LLM that these decisions can be stale ŌĆö just for information about what was in the past").

**Adapter neutrality.** The subagent is mode-agnostic ŌĆö it reads `specs/frs/<filename>.md` regardless of whether `<filename>` is a tracker ID (`STE-228.md`) or short-ULID (`VDTAF4.md`). No `Provider` resolution. This matches the existing `tdd-*` subagents' adapter-neutrality.

**Result-log persistence (probe input).** AC-STE-230.12's gate-probe needs a recorded block to inspect. Convention TBD at implementation time, but the natural site is `.dpt-locks/<ulid>/spec-research-result.txt` (parent skill writes the most recent forked-skill output there before consuming it). The probe is vacuous when no log exists ŌĆö implementations that don't persist (because the operator never invoked the spec-research path) pass cleanly.

## Notes

**Provenance.** Drafted 2026-05-06 from a `/brainstorm` session same day. Three approaches considered:

1. Recency-window injection (read 3-5 most recently archived FRs into parent context) ŌĆö rejected: recency ŌēĀ relevance, archive grows unbounded.
2. Decision lift on archive (lift rationale into ADR-style files at archive time) ŌĆö rejected: requires new artifact + manual judgment per archive.
3. Topic-aware subagent retrieval (this FR) ŌĆö selected: zero schema churn, archive stays passive, LLM-quality ranking.

The selected approach was further refined: the original `## Cautions` section in the output contract was dropped on user feedback (gate-probing contradiction detection is shape-not-content; trust the parent LLM with a "may be stale" banner instead).

**Industry references.**

- GitHub Spec Kit and Kiro: per-feature specs that don't carry forward as context across features; `constitution.md` is the only persistent guidance.
- OpenSpec: archive folder as audit trail; deltas merged into living `openspec/specs/` on archive (analogous to our `requirements.md` / `technical-spec.md` / `testing-spec.md`).
- ADR retrieval research (`intent-driven.dev/blog/2026/04/29/spec-driven-development-with-adr/`, `arxiv.org/html/2604.03826`): durable rationale lives outside change folders; small recency window (3ŌĆō5 records) gives best context-quality trade-off.

This FR adopts the **on-demand retrieval** path that none of the existing toolkits ship ŌĆö the subagent does the lookup at design time, parent skills consume the summary, archive stays passive.
