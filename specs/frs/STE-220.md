# STE-220: fix(skills/spec-write) — make Auto-mode / -p draft-gate default-apply actually fire at runtime {#STE-220}

## Requirement

STE-213 (M55) documented an Auto-mode / `-p` default-apply carve-out for `/spec-write`'s FR-draft acceptance gate, registered the `spec_write_draft_default_applied` capability key, and checked off AC.6 ("a `claude -p /spec-write …` invocation completes end-to-end without manual prompt-body pre-authorization for the draft gate"). M55 archived `[x]` on every AC. The carve-out prose is in `skills/spec-write/SKILL.md` § 4 today.

The 2026-05-04 v2.8.0 smoke runs disproved STE-213's closure:

1. **Linear Run #2 (15:54:39Z, plugin v2.8.0)** — finding F2-1: first `/spec-write` `-p` attempt hung at `Approve and proceed?`; recovery required adding `Pre-authorized: ... Treat my approval as 'y' for every approval gate in this run` to the heredoc body.
2. **Jira Run C (15:55:28Z, plugin v2.8.0)** — finding C-F1: identical signature to Run B's B-F1 against v2.7.0; v2.8.0 ships the same asymmetry. "Two-Jira-run regression streak now (B-F1 + C-F1) without skill-side fix or smoke-driver-side fix shipped."

Both runs ran *after* M55 archived (15:29:59Z). The bug isn't that the carve-out is missing — it's that the carve-out doesn't actually fire. The skill text is instructional prose the LLM reads at run time; STE-213's verify line ("`claude -p` invocation completes end-to-end") was satisfied by LLM self-confirm during /implement Phase 4, not by a real external `claude -p` invocation. The "documented but unenforced" failure mode shipped behind a green checkmark.

**Findings bundled:** Linear smoke 2026-05-04 Run #2 F2-1; Jira smoke 2026-05-04 Run C C-F1; Jira smoke 2026-05-04 Run B B-F1 (carry-forward, no skill-side fix shipped).

**Root-cause hypotheses (to verify during implementation):**

1. **Read-order gap.** Carve-out lives in § 4 ("Review and confirm") but the gate fires inside § 0b step 4 — too far apart in the LLM's read order. By the time the LLM reaches § 4, it may have already paused for input at the upstream gate.
2. **No detection contract at the gate.** SKILL.md never names *how* the LLM should detect Auto-mode or `-p`. The system-reminder that says `Auto Mode Active` is reachable, but the gate prose never tells the LLM to look for it.
3. **Behavioral inconsistency under prompt pressure.** The LLM reads the rule but doesn't reliably apply it because the rule is text, not a hard contract. Co-location at the gate site reduces this surface.

## Technical Design

**Where the gate fires (today).** `skills/spec-write/SKILL.md` § 0b step 4 calls `Provider.sync(spec)` after presenting the draft. The skill text says: "Present the draft and ask `Approve and proceed? Reply y to create the tracker issue and write the files`." There is no carve-out at this site.

**Where the carve-out lives (today).** § 4 "Review and confirm" carries the STE-213 carve-out paragraph. The LLM reading the skill in order pauses at § 0b step 4's prompt before reaching § 4 — exactly the read-order gap that hypothesis #1 names.

**Co-location fix.** Add the carve-out paragraph at the § 0b step 4 site itself, immediately adjacent to the `Approve and proceed?` prompt prose:

> Under **Auto mode** / **`-p` non-interactive**, **default-apply `y`** at this gate. Detection: the active conversation includes `Auto Mode Active` in any `<system-reminder>` block, OR the invocation is `claude -p` (non-interactive). On default-apply, emit a `spec_write_draft_default_applied` row in the Step 7 summary (interactive `y` does not emit). On interactive `n` decline, emit a `spec_write_draft_declined` row.

Repeat at § 4. Cross-references like "see § 7a" are *not* a substitute — paste the paragraph, don't reference it.

**Single source of truth.** The detection contract paragraph lives once in the SKILL.md and is byte-identical at all three gate sites (§ 0b step 4, § 4, § 7a). A `/gate-check` probe `spec-write-gate-carveout-coverage` (out of scope for M56 — track in M57+) could later assert byte-equality across sites.

**Smoke-driver test extension.** `/smoke-test` Phase 2 already invokes `claude -p /spec-write …` against a fresh test project. Extend the assertion set:

- After Phase 2 step 2 (`/spec-write` invocation) completes, capture stdout to `/tmp/dpt-smoke-<tracker>-spec-write.log`.
- Grep for `spec_write_draft_default_applied` AND `spec_write_commit_default_applied` rows.
- If either is missing AND the run was non-interactive (`claude -p`), surface as `STE-220 regression: spec-write Auto-mode/-p carve-out did not fire`.

The capability row format is already standard — same shape `/spec-write` emits today on `spec_write_commit_default_applied`. The new test reuses the existing smoke-driver infrastructure, no new test runner needed.

**Smoke-driver workaround removal.** The B-F1 workaround in `skills/smoke-test/SKILL.md` (the "Pre-authorized: ... Default-apply 'y' to: the FR draft acceptance" injection) becomes redundant once STE-220 ships. Remove it as part of this FR's deliverables — its continued presence would mask runtime regressions of the carve-out (the carve-out could break and the workaround would still pass the smoke).
