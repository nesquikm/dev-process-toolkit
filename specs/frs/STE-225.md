# STE-225: feat(skills/tdd): multi-agent TDD context isolation via skill+subagent fork pattern {#STE-225}

## Requirement

Replace the existing single-context `plugins/dev-process-toolkit/skills/tdd/SKILL.md` with a four-skill orchestrator that runs each TDD phase in an isolated subagent context, paired via the Skills `context: fork` + `agent:` mechanism.

1. **Orchestrator skill** `skills/tdd/SKILL.md` runs in main context (no `context: fork`); calls each child via the Skill tool.
2. **Three child skills** `skills/tdd-write-test/`, `skills/tdd-implement/`, `skills/tdd-refactor/`, each with `context: fork`, `agent: tdd-{test-writer|implementer|refactorer}`, `user-invocable: false`.
3. **Three subagents** `agents/tdd-test-writer.md`, `agents/tdd-implementer.md`, `agents/tdd-refactorer.md`, each with `tools: Read, Grep, Glob, Write, Edit, Bash` (no `Agent`, no `WebFetch`/`WebSearch`), `maxTurns: 8`, narrow descriptions naming the orchestrator as the sole invoker.
4. **Cycle granularity** — test-writer runs once per FR (batched across AC1..N); implementer runs per AC; refactorer runs **once at end** of the FR after all ACs are GREEN.
5. **Hand-off contract** — every child must end its turn with a single fenced `tdd-result` YAML block parsed deterministically by the orchestrator.
6. **Failure recovery** — bounded retry (max 2 per role per AC for semantic failures; single targeted retry for format violations; `maxTurns` exhaustion counts as a failed attempt). Halt + escalate after the cap.
7. **Replace** existing `/tdd` rather than ship `/tdd-strict` alongside (per `project_no_users_yet`). `/implement` Phase 3 calls the new orchestrator inline.

## Acceptance Criteria

- AC-STE-225.1: `plugins/dev-process-toolkit/skills/tdd/SKILL.md` is the orchestrator (no `context: fork` in frontmatter); the three child skills `tdd-write-test`, `tdd-implement`, `tdd-refactor` each carry `context: fork`, an `agent:` field that resolves to a real `agents/*.md` file, and `user-invocable: false`. `disable-model-invocation` is **not** set on the children (orchestrator must invoke them via the Skill tool).
- AC-STE-225.2: The three subagents `agents/tdd-test-writer.md`, `agents/tdd-implementer.md`, `agents/tdd-refactorer.md` each declare `tools: Read, Grep, Glob, Write, Edit, Bash` (allowlist; no `Agent`, no `WebFetch`, no `WebSearch`), `maxTurns: 8`, and a description that names the orchestrator as the sole invoker (e.g., "Internal TDD test-writer subagent. Invoked exclusively by /tdd orchestrator. Do not invoke directly.").
- AC-STE-225.3: Cycle granularity — orchestrator invokes `tdd-write-test` exactly **once per FR** with the full AC list batched into the prompt; invokes `tdd-implement` **per AC** (one fork per AC, passing only that AC's text + the failing-test command); invokes `tdd-refactor` **exactly once** after all ACs are GREEN.
- AC-STE-225.4: Hand-off contract — each child ends its turn with exactly one fenced block tagged ` ```tdd-result `. Required fields per role:
  - `role: test-writer` ⇒ `status` (`ok`|`failed`), `files` (test file paths), `command` (the failing-test command), `output_excerpt` (first 40 lines of test runner output, must show RED), `notes` (optional).
  - `role: implementer` ⇒ `status`, `files` (modified source paths), `command`, `output_excerpt` (must show GREEN), `notes`.
  - `role: refactorer` ⇒ `status`, `files` (modified paths, possibly empty), `command`, `output_excerpt` (must still show GREEN), `notes`.
  Orchestrator parses the block as YAML; missing block, missing role, missing required field, or wrong role for the invocation is a format violation.
- AC-STE-225.5: Failure recovery — orchestrator distinguishes five failure modes: (A) false-RED (test-writer's tests don't fail when run); (B) implementer can't reach GREEN; (C) refactorer breaks GREEN; (D) format violation (no valid `tdd-result` block, wrong role, or missing required field); (E) `maxTurns` exhaustion (subagent stopped without producing a block). For (A)/(B)/(C): max 2 attempts per role per AC, then halt. For (D): single targeted retry ("re-emit your last message with a valid `tdd-result` block"), then halt. For (E): counts as a failed attempt under whichever of (A)/(B)/(C) applies. On halt: orchestrator reports the failure mode, retry count, last `tdd-result` block (or raw output if none), and exits non-zero. **Retry prompt injects only raw failing-test output, no orchestrator-side analysis** — preserves test-writer-cannot-see-implementation isolation.
- AC-STE-225.6: `/implement` Phase 3 invokes the new `/dev-process-toolkit:tdd` orchestrator inline (no separate opt-in path; matches existing pacing memory `feedback_implement_pacing`). The previous single-context `/tdd` content is fully replaced — no shim, no `/tdd-strict` alongside.
- AC-STE-225.7: `/gate-check` carries a new structural probe `tdd_orchestrator_integrity` that asserts (and only asserts) load-bearing invariants: (a) the four skill paths exist; (b) the three child skills carry `context: fork`; (c) each child's `agent:` field resolves to a real `agents/*.md` file; (d) each child carries `user-invocable: false`; (e) each subagent's `tools` field excludes `Agent`. Probe does **not** assert specific allowed-tools beyond the `Agent` exclusion or specific prompt phrasing — content-level drift is verified by the smoke, not the probe.
- AC-STE-225.8: Bun-mocked unit tests cover (a) the `tdd-result` fenced-block parser (well-formed blocks of each role accepted; malformed blocks rejected with role-specific reason); (b) the retry state machine (semantic failure → 1 retry → 2nd failure → halt; format violation → 1 targeted retry → halt; `maxTurns` counted as failed attempt); (c) the orchestrator's batching invariant (test-writer invoked once with all ACs; implementer invoked exactly N times for N ACs; refactorer invoked exactly once after all ACs GREEN); (d) the halt path emits the failure mode, retry count, and last block. Run on every `bun test`, deterministic, no live model calls.
- AC-STE-225.9: Headless live smoke at `plugins/dev-process-toolkit/tests/fixtures/tdd-smoke/` contains a fixture FR with one trivial AC ("function `add(a, b)` returns sum of two numbers"). ## Technical Design

**Architecture.** Orchestrator skill `tdd/SKILL.md` runs in the main session. Its body parses the input (`<FR-id>` or per-AC reference), reads `specs/frs/<id>.md`, builds the test-writer prompt with the full AC list, then invokes child skills via the Skill tool with `context: fork` + the named `agent:`. Each invocation is a fork-subagent boundary: the child sees only the rendered SKILL.md content as its task prompt, plus CLAUDE.md, plus its own subagent system prompt. The child's final message is the only return channel; orchestrator parses the `tdd-result` fenced block deterministically.

**Why this shape (not three subagent files invoked directly).** The skill is the unit of *task instruction*; the subagent is the unit of *execution sandbox*. `context: fork` pairs them — task-as-prompt + locked-down tools + isolated context. Confirmed against `https://code.claude.com/docs/en/skills#run-skills-in-a-subagent`.

**Hand-off contract (load-bearing).** The `tdd-result` block is the *only* in-band channel between orchestrator and child — Claude Code's subagent-final-message is the sole return signal (no structured return slot per `https://code.claude.com/docs/en/sub-agents`). Strict YAML keeps the orchestrator's parser deterministic, matching DPT's "deterministic gates override LLM judgment" principle.

**Failure recovery rationale.** Max-2 matches DPT's existing self-review-bounded principle (memory `project_no_users_yet` and CLAUDE.md "Self-review is bounded — max 2 rounds"). Format-violation single-retry is a separate budget because re-emit-the-block is a cheaper class of fix and shouldn't burn semantic-failure attempts. `maxTurns: 8` is the upper-bound circuit breaker on the subagent side; running out of turns is treated as a failed attempt under the relevant semantic mode. **Retry prompt must inject only raw failing-test output** — orchestrator-side analysis is the soft hand-off the gap was created to eliminate.

**Plugin-subagent constraint.** Per `https://code.claude.com/docs/en/sub-agents#choose-the-subagent-scope`, plugin-bundled subagents drop `hooks`, `mcpServers`, `permissionMode`. The dossier's earlier "PreToolUse hook validates X" idea is not viable on the plugin path; behavior comes from the SKILL.md prompt + the `tools` allowlist exclusively. Probe `tdd_orchestrator_integrity` enforces this on the input side (no Agent in tools); AC-STE-225.10 enforces it on the output side (none of the three forbidden fields present).

**Refactor batching (Q1 decision).** Once-at-end after all ACs GREEN. Rationale: per-AC isolation matters most for the test-writer-cannot-see-implementation guarantee; refactor isolation matters less because by then all tests are GREEN and the refactorer's correctness gate is "tests still pass." A single global pass costs less, sees more (cross-AC duplication), and the regression concern is mitigated by the GREEN-at-exit gate.

**Smoke shape (Q5 decision).** Two-tier:
- **Bun-mocked tier** (every commit): fixture directory of canned `tdd-result` blocks (well-formed + malformed variants of each role, plus failure variants). Orchestrator runs in `--dry-run` mode that reads from fixtures instead of spawning real subagents. Tests assert parser, retry state machine, halt path, batching invariant. Pattern matches the existing `conformance-loop-dry-run.test.ts`.
- **Headless live tier** (nightly + pre-ship): real `claude -p` invocation against the trivial-AC fixture FR; behavioral assertions only. Findings file shape matches `/smoke-test`'s `/tmp/dpt-smoke-findings-<date>-<tracker>.md` convention.

## Testing

**Unit-level (deterministic; runs every `bun test`).**
- Frontmatter shape — orchestrator + three children + three subagents (AC.1, AC.2, AC.10).
- `tdd-result` fenced-block parser — well-formed/malformed variants per role (AC.4, AC.8a).
- Retry state machine — semantic-failure path, format-violation path, `maxTurns` path (AC.5, AC.8b).
- Batching invariant — orchestrator's invocation count per role per FR (AC.3, AC.8c).
- Halt path — orchestrator's failure-report shape (AC.5, AC.8d).
- `/gate-check` probe `tdd_orchestrator_integrity` — positive + adversarial cases (AC.7).

**Integration-level.**
- `--dry-run` mode reads canned fixtures; covers parser + retry + batching end-to-end without live model calls.

**Headless live smoke (env-var-gated; nightly + pre-ship).**
- `DPT_TDD_LIVE_SMOKE=1 bun test plugins/dev-process-toolkit/tests/tdd-live-smoke.test.ts` (AC.9). Runs real `claude -p` against the fixture FR; behavioral assertions only. Findings written to `/tmp/dpt-tdd-smoke-findings-<date>.md`.

## Notes

**Risk — model nondeterminism in live smoke.** Different runs may pick different test filenames, different implementation styles, different test phrasing. Mitigation per AC.9: assertions are behavioral (file exists, function exports a callable, exit code 0), never exact-string. Threshold-tuning may be needed after first nightly runs.

**Risk — retry context-leak.** On retry, the orchestrator must inject only raw failing-test output, not analysis. Otherwise the orchestrator's "tests fail because X — fix Y" leaks information that defeats the test-writer-cannot-see-implementation isolation. AC.5 captures this constraint; the retry prompt template enforces it deterministically.

**Risk — `/implement` Phase 3 pacing.** Memory `feedback_implement_pacing` says don't pause between phases. The new orchestrator's halt path **does** pause (escalates to user). This is the intended behavior — halt only fires after the bounded retry cap, so it's a real failure surfacing, not a routine gate. Document this in `/implement` Phase 3 prose.

**Open question deferred to a later milestone.** Should `/implement`'s self-review-bounded loop (max 2) also apply to TDD cycles, or do TDD cycles get their own retry budget independent of `/implement`'s? Current AC.5 makes TDD-internal retries independent (per role per AC); that may compound with `/implement`'s top-level retries. If so, surface in a follow-up FR — not load-bearing for M58.
