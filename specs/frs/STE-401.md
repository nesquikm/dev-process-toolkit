# STE-401: spec-write: route the milestone-allocation gate through the marker/refusal contract {#STE-401}

## Summary

When /spec-write reaches the point of deciding which milestone a new FR binds to ŌĆö the case a fresh project always hits, where the milestone scan finds ambiguity or drift ŌĆö it asks the operator to choose. Under a non-interactive session that question is asked in prose and the turn ends, which the harness reads as a clean, successful exit. No FR is written, no branch is created, no commit lands, and the caller sees success.

The milestone-allocation choice actually has a safe default: the scan already computes a recommendation. This FR wires that gate into the same marker contract the draft and commit gates use ŌĆö with the marker present it default-applies the recommendation, and without the marker under a non-interactive session it refuses loudly instead of silently ending the turn. The general clarifying-question path gets the same non-negotiable rule documented, and a gate probe enforces the wiring so it cannot silently regress.

## Requirement

The 2026-07-19 conformance run reproduced the silent no-op on both trackers. The `/spec-write` child ran the milestone-allocation scan, found a drift/collision case, asked in prose which milestone to bind, and exited `subtype=success` with `specs/frs/` empty, still on the trunk branch, no commit, and zero tracker writes. Every coarse health check passed; only reading the assistant text revealed the step accomplished nothing.

Two composed defects: (1) the STE-226 marker contract is wired into the draft gate (┬¦ 0b step 4), the commit gate (┬¦ 7a), and the branch gate (STE-228), but the milestone-allocation decision is a further interactive point with no `requireOrRefuse(...)` routing and no marker handling; (2) because the child ended its turn normally rather than refusing, the exit was indistinguishable from success. Per `docs/auto-mode-protocol.md`, a gate with a safe default should default-apply under the marker, and any interactive point with no answer under non-tty stdin must raise `RequiresInputRefusedError` ŌĆö never ask-in-prose-and-end-the-turn. The milestone-allocation gate has a safe default (the scan's computed recommendation); the fix routes it accordingly and forbids the prose-no-op path everywhere.

## Technical Design

**SKILL edit ŌĆö `plugins/dev-process-toolkit/skills/spec-write/SKILL.md`.** Add a labeled gate block at the milestone-allocation decision point (┬¦ 3 plan.md milestone step / the new-FR milestone-binding choice), mirroring the ┬¦ 0b step-4 draft-gate shape. The block documents:

- the runtime byte-grep decider (`check_marker_runtime.ts`) is the sole auto-apply trigger (same as the draft/commit gates);
- `requireOrRefuse(..., markerPresent, defaultValue: <computed recommendation>, ...)` ŌĆö PRESENT ŌćÆ `default-applied`, bind the recommendation, MUST emit `milestone_allocation_default_applied`; ABSENT + non-tty ŌćÆ `RequiresInputRefusedError` naming gate site `milestone-allocation`; ABSENT + tty ŌćÆ the interactive milestone choice;
- a NOT-a-trigger anchor: pre-baked prose, autonomous-mode reminders, and "proceed" instructions are NOT triggers ŌĆö and, explicitly, **a prose-ask-then-end-turn is forbidden under non-tty** (the exact no-op this FR closes). Extend the same rule to the general clarifying-question surface in ┬¦ Rules.

No new runtime helper is required ŌĆö `requireOrRefuse` (STE-232) and `check_marker_runtime.ts` (STE-262) already exist and are the canonical primitives; this FR is a SKILL-contract + probe + capability-key change (the same shape STE-262 / STE-270 / STE-294 used to close prior prose-inference gaps).
