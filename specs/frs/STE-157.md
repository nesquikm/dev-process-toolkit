# STE-157: /spec-write idempotency hardening on Gateway-Timeout retry {#STE-157}

## Requirement

`/smoke-test --tracker jira` run #6 (M44 / smoke #6 trace at `/tmp/dpt-smoke-findings-2026-04-29-jira.md`, finding F6) caught a real defect: `/spec-write`'s Jira-side push retry left an orphan duplicate work item (DST-3) when the create call returned a Gateway-Timeout response while the server-side write succeeded as DST-2. The advertised JQL idempotency probe ("verified via JQL search before re-issuing the create to avoid a duplicate") failed to find DST-2 in the narrow eventual-consistency window between the timeout and the retry, so the retry created DST-3 with the still-placeholder description body (`## Acceptance Criteria\n\n(populated after tracker ID allocation)`). The real FR landed on DST-2 with full ACs; DST-3 is permanently stub-content because no later step references it. Phase 5 cleanup transitioned both to Done, but the orphan persists in the project (Atlassian MCP exposes no `deleteJiraIssue`).

The defect contradicts M43 STE-154's idempotency claim ("no duplicates") and will accumulate stub issues in DST over repeated smoke runs that hit Gateway-Timeouts. M43 cannot be re-opened (already archived per the M44 risk-table policy); this FR carries the fix forward.

## Acceptance Criteria

- AC-STE-157.1: `/spec-write`'s Jira-mode `upsertTicketMetadata(null, ...)` retry path widens its idempotency window. Before falling through to a fresh create after a network-error response, retry the JQL idempotency probe with backoff (suggested: 1s + 2s + 4s, three attempts) so eventual-consistency lag has time to settle. Existing single-shot probe stays as the fast path.
- AC-STE-157.2: Add a regression test under `tests/spec-write-jira-retry-idempotency.test.ts` (or equivalent placement) that simulates a Gateway-Timeout response on the first `createJiraIssue` call, asserts the retry path runs the JQL probe with backoff, and asserts the second create is **not** issued when the original write landed server-side.
- AC-STE-157.3: When the backoff probe still misses (genuine duplicate-create scenario, e.g., the original write actually failed on the server), surface a one-line warning in `/spec-write`'s Step 7 summary table (capability key `tracker_idempotency_uncertain` per `docs/spec-write` § Capability-gap rendering) so the operator knows to manually verify before downstream skills bind to the new ID. Add the new key to the static plain-language map in `skills/spec-write/SKILL.md` § Step 7.
- AC-STE-157.4: Document the retry-with-backoff contract in `adapters/jira.md` § `upsert_ticket_metadata` row — which timeouts trigger the backoff path, the canonical 1+2+4s schedule, and the "still ambiguous" fallback. Adapter authors writing custom Jira-flavored adapters need this.
- AC-STE-157.5: A re-run of `/smoke-test --tracker jira --jira-project DST` against the post-fix tree shipping with no new orphan-duplicate work items (run window check: only the FR's own canonical issue plus any pre-existing orphans from earlier runs).

## Technical Design

`/spec-write`'s tracker-mode push (`Provider.sync(spec)` → `upsertTicketMetadata(null, ...)`) is an in-skill prose contract today (no TS driver implementation in-tree). The fix is two-layered: prose change + adapter-side test scaffolding.

**Prose change** in `skills/spec-write/SKILL.md` step 4: add the backoff-probe contract for tracker-mode `upsertTicketMetadata(null, ...)` calls. Spec the canonical schedule (1+2+4s, three attempts) and the fallback (warning row in Step 7 summary, key `tracker_idempotency_uncertain`).

**Adapter docs** in `adapters/jira.md` (and symmetric note in `adapters/linear.md` since Linear's `save_issue` has the same Gateway-Timeout class of failure mode): document the retry-with-backoff path on `upsert_ticket_metadata` row.

**Test fixture** in `tests/spec-write-jira-retry-idempotency.test.ts` simulates the Gateway-Timeout on first call via stub. The test exercises the contract that the retry path uses backoff, not a single-shot probe.

**Smoke #8 regression** — re-run `/smoke-test --tracker jira --jira-project DST` against the post-fix tree to confirm no orphan duplicate landed.

**Out of scope:** any Atlassian MCP-side fix (we don't own that surface), Linear-side equivalents beyond the documentation symmetry note (no Linear smoke run reported the same defect — the issue surfaced specifically on the Atlassian Rovo MCP's Gateway-Timeout response shape), idempotency-key headers (Atlassian Rovo MCP does not appear to support them today; revisit if a future MCP version exposes them).

## Testing

`tests/spec-write-jira-retry-idempotency.test.ts` covers the AC-STE-157.2 contract (single-shot probe miss → backoff probe hit → no duplicate create) and the AC-STE-157.3 contract (single-shot + backoff both miss → fresh create → warning surfaces in Step 7 summary).

Smoke #8 (post-fix re-run of `/smoke-test --tracker jira`) is the live-verification gate before /ship-milestone fires for whichever milestone closes this FR.
