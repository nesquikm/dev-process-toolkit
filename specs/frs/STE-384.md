# STE-384: Path-drift meta-test: retired `.dpt-locks/` + `.dev-process/` literals cannot reappear {#STE-384}

## Requirement

STE-382 routes every `.dpt` path through `dpt_paths.ts`, but nothing stops a future edit from re-introducing a raw `.dpt-locks` or `.dev-process` literal in prose or code. The retired literals are especially prone to resurrection: they appear in ~170 places across specs, docs, skills, adapters, and tests today, and copy-paste from older files is exactly how stale literals seep back after a rename.

STE-49 ("Purge v1 `specs/archive/` Drift") shipped this defense for this reason: `tests/archive-path-drift.test.ts`, a grep meta-test that fails loudly if the retired literal reappears under `docs/`, `skills/`, `README.md`, and the live cross-cutting specs. Clone that shape for the `.dpt/` rename.

The rule is a deterministic test, not LLM judgment — the recurring lesson of the STE-220 → STE-226 → STE-238 chain is that prose-only rules drift and byte-checkable gates hold.

## Acceptance Criteria

- AC-STE-384.1: **Drift meta-test.** New `tests/dpt-path-drift.test.ts` (STE-49 shape) greps the live tree for the retired literals `.dpt-locks` and `.dev-process` and fails naming `file:line` for each survivor. Scope covers `plugins/dev-process-toolkit/{docs,skills,adapters,templates}` and `README.md`. {#AC-STE-384.1}
- AC-STE-384.2: **History allow-list.** Legitimate historical mentions are exempt: `CHANGELOG.md`, `specs/frs/archive/**`, and `specs/plan/archive/**` record what the old layout was and must not be rewritten. The allow-list is explicit and commented — an unexplained exemption is a bug. {#AC-STE-384.2}
- AC-STE-384.3: **Optional-target tolerance + stale-comment fix.** Scan targets pass through STE-49's `.filter(existsSync)` guard so the test tolerates absent optional targets (a downstream consumer running this plugin's `/gate-check` may not carry every cross-cutting spec file). **The justification is re-stated correctly:** STE-49's own comment claims the filter exists because "this repo's own `specs/` is gitignored (dogfood workspace)" — that claim is **false** as of 2026-07-15 (`specs/` carries 416 tracked files; no ignore rule matches it). The filter is retained for genuine optional-target tolerance, and the stale comment in `tests/archive-path-drift.test.ts` is corrected in the same edit so it stops propagating. {#AC-STE-384.3}
- AC-STE-384.4: **Tripwire proven.** Fixture coverage asserts the test actually fires — a fixture reintroducing `.dpt-locks/<ulid>/spec-research-result.txt` in a scanned path fails with the offending `file:line`, and a fixture carrying only allow-listed mentions passes. {#AC-STE-384.4}
- AC-STE-384.5: **Full gate green.** `bun test` from `plugins/dev-process-toolkit` passes against the post-STE-382/STE-383 tree with zero surviving literals outside the allow-list; no probe regresses. {#AC-STE-384.5}

## Technical Design

Filled at implementation. Reference: `tests/archive-path-drift.test.ts` (STE-49) — same grep-and-name shape, same `baseTargets` / `optionalTargets.filter(existsSync)` split.

**On AC.3's stale comment.** The 2026-07-15 spec-research seed reported the `specs/`-is-gitignored rationale as fact because it faithfully quoted the test's comment; verification (`git check-ignore` + `git ls-files specs/ | wc -l` → 416) falsified it. The comment is the drift, not the filter. Correcting it in the same edit prevents the next research seed from re-propagating it — a small instance of the same "retired claim reappears via copy-paste" failure this whole FR guards against.

## Notes

- **Depends on STE-382 + STE-383** — the test asserts the *end state*, so it is red until both land. Build order A → B → C is load-bearing, not cosmetic.
- **Meta-test, not a `/gate-check` probe, by choice.** The drift it guards is internal to this repo's source tree, not a property of a consumer project. Consumer-side path correctness is already covered by STE-382's probe re-pointing and STE-383's `check-ignore` assertions. Adding a probe would export a rule consumers cannot violate.
