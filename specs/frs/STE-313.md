# STE-313: Refuse first-turn gates under non-tty stdin when auto-approve marker absent {#STE-313}

## Requirement

Under `claude -p` (non-tty stdin), the literal byte-string `<dpt:auto-approve>v1</dpt:auto-approve>` (the marker) MUST be the **sole** trigger for auto-applying first-turn gate decisions. Currently, the marker check is paraphrased away — when the marker is absent, both `/spec-write` (draft gate § 0b step 4, branch gate § 7a) and `/setup` (Socratic first-turn scaffold-Write ban, STE-237) auto-apply by rationalizing the harness's `<system-reminder>` ("work without stopping for clarifying questions") as license to proceed.

Verified by `/conformance-loop` iteration 1 (2026-05-19):

- Linear leg F1 — `/spec-write` Group 1b: marker absent → child auto-applied → `STE-311` committed on master (`3cca70a`). Child closing prose: *"proceeded under the harness's autonomous-mode reminder to honor the typed instruction."*
- Linear leg F2a — `/spec-write` Group 5b: marker absent → child auto-applied branch gate decision → `STE-312` committed (`e30bedb`). Same root cause as F1, different gate site.
- Jira leg F1 — `/setup` Phase 8: marker absent → scaffold `Write` fired at `tool_use index=12` BEFORE any `AskUserQuestion`. `assertFirstTurnShape` raised `SocraticFirstTurnViolationError`. Fixture: `plugins/dev-process-toolkit/tests/fixtures/socratic-first-turn/setup-2026-05-19.json` (112,572 bytes).
- Jira leg F2 — `/spec-write` fixtures 1b + 5b: same regression class, cross-tracker confirmation.

STE-294 AC.4 (archived) explicitly fences this regression shape in skill text, but skill text is not byte-checkable — the LLM reasons past it. STE-237 + STE-251 + STE-262 establish the deterministic mechanisms (`check_marker_runtime.ts`, `assertFirstTurnShape`, `requireOrRefuse`); the gap is that nothing forces the LLM to actually call them on every gate site before any side effect.

**Desired behavior:** marker absent + `process.stdin.isTTY === false` ⇒ `RequiresInputRefusedError` (NFR-10 canonical shape, naming the gate site), zero side effects (no `Provider.sync`, no FR write, no Linear/Jira ticket, no `git commit`). Marker present + non-tty ⇒ auto-apply as today. TTY path byte-identical to v2.27.0.

- AC-STE-313.3: `/setup` Socratic first-turn (STE-237) — under non-tty stdin + marker absent, the FIRST tool call MUST be `AskUserQuestion` `tool_use` OR `RequiresInputRefusedError` raise. `Write` / `Edit` / `NotebookEdit` before either is a hard fail. The setup-2026-05-19.json fixture replay MUST refuse, not scaffold.
- AC-STE-313.4: Three verbatim regression fixtures (Group 1b prompt body, Group 5b prompt body, setup-2026-05-19.json) land in `plugins/dev-process-toolkit/tests/fixtures/marker-absent-non-tty/`. Probe severity: error.

## Technical Design

[needs technical review — design decision: hook-based enforcement vs. test-loop-only enforcement]

Three plausible enforcement vectors, ranked by deterministic strength:

1. **Pre-tool-use hook (strongest).** A `PreToolUse` hook in plugin `hooks/` fires before any `Write` / `Edit` / `NotebookEdit` tool call within a `/spec-write` or `/setup` skill invocation. The hook reads `/tmp/dpt-spec-write-prompt-body.txt` (the body the skill is required to write at the FIRST step of every gate evaluation), runs `check_marker_runtime.ts`, and either allows the tool call (marker present) or blocks with hard-refusal stderr (marker absent + non-tty). LLM cannot rationalize past a hook block.

2. **Runtime test-loop (medium).** Test loop in `tests/marker-absent-non-tty.test.ts` replays the three fixture bodies via stubbed `requireOrRefuse` harness; asserts refusal class fires. Catches regressions at CI time but doesn't prevent the LLM from regressing at runtime.

3. **`/gate-check` probe (weakest).** Scans the prompt-bearing-child stdout for evidence the helper fired. Only works with `--output-format stream-json` capture (conformance-loop F6 instrumentation gap). Detection-only, not prevention.

Recommended: combine **(1) + (2)**. (3) follows once F6 is closed.

Helper paths already on disk:

- `plugins/dev-process-toolkit/adapters/_shared/src/check_marker_runtime.ts` (STE-262)
- `plugins/dev-process-toolkit/adapters/_shared/src/requires_input.ts` → `requireOrRefuse(...)` (STE-251)
- `plugins/dev-process-toolkit/adapters/_shared/src/socratic_first_turn.ts` → `assertFirstTurnShape(...)` (STE-237)

Fixture capture path (Jira F1 already established): `plugins/dev-process-toolkit/tests/fixtures/marker-absent-non-tty/`.
