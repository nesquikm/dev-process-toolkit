# STE-323: parseBun test-count parser — anchor on summary line, not first match {#STE-323}

## Requirement

Fix a HIGH-severity bug in `adapters/_shared/src/test_count_parser.ts:26` where `parseBun` captures the FIRST `\d+\s+pass\b` occurrence in Bun's stdout, not the trailing summary line. When Bun emits per-file pass counts before the summary, the parser returns the per-file count, not the total. The CHANGELOG `Total test count at release: <N> tests` line — written by `/ship-milestone` per AC-STE-73.12 — is therefore unreliable; the release audit-trail numbers swing wildly between adjacent releases when Bun's per-file output ordering varies.

Current code (verified on `main` @ `d3516db`):

```typescript
// adapters/_shared/src/test_count_parser.ts:26
const passMatch = /(\d+)\s+pass\b/.exec(output);
```

`.exec()` returns the first match. Bun's actual output (verified locally on this repo):

```
plugins/dev-process-toolkit/tests/some-test.test.ts:
  ✓ test 1 passed
  ✓ test 2 passed

... more files ...

 2891 pass
 11 skip
 0 fail
 6340 expect() calls
Ran 2902 tests across 257 files. [28.45s]
```

The parser sees per-file `\d+ pass` patterns first and captures them. Fix: anchor on the LAST `\d+ pass` match OR (cleaner) anchor on Bun's explicit `Ran (\d+) tests` summary line which is unambiguously the total.

Critical because `parseBun`'s output feeds `/ship-milestone`'s release-gate (AC-STE-73.12: "failed-test count blocks release BEFORE the approval prompt — parser accuracy is load-bearing for release-gate correctness"). A first-match capture that says "8 pass" when the total is "2891 pass" would mis-report release readiness.

(Historical CHANGELOG numbers are out-of-scope per `project_no_users_yet` — only the forward-going parser correctness matters.)

## Acceptance Criteria

- AC-STE-323.1: `parseBun` in `adapters/_shared/src/test_count_parser.ts` is rewritten to anchor on Bun's explicit summary line. Two-step approach: (a) prefer `Ran (\d+) tests across \d+ files` (the canonical summary line); (b) when summary is absent (older Bun versions OR truncated output), fall back to the LAST `\d+ pass` match using `/(\d+)\s+pass\b/g` global regex + selecting the last result. Fall-through path: when neither anchor matches, return NFR-10 canonical fallback "could not determine test count — Bun output lacks both `Ran N tests` summary and `N pass` summary line" (preserves STE-73 Technical Design's fallback voice).
- AC-STE-323.2: `parseBun` output shape stays `TestCountParseResult = { ok: true; count: TestCount }` where `TestCount = { total: number; failures: number; errors: number }` (STE-73-canonical, also consumed by STE-105 — verify against the actual type definition in `test_count_parser.ts` at edit time). `/ship-milestone` integration is byte-stable on shape; the semantic shift is that `count.total` now reflects Bun's `Ran N tests` (includes `skip`), whereas the legacy first-match path returned `pass` only (excludes `skip`). For zero-skip runs the two are identical; for runs with skipped tests, the new total is strictly ≥ the old. This is the intended fix — the audit-trail number should reflect what Bun ran, not just what passed.

## Technical Design

Single source-file change + test fixture additions.

Updated `parseBun` (sketch — implementer adapts to the existing `TestCount` shape in `test_count_parser.ts`):

```typescript
function parseBun(output: string): TestCountParseResult {
  // For failures: take the LAST `N fail` match (Bun emits per-file `N fail` lines
  // before the summary in multi-file runs — same first-match bug class).
  const failMatches = Array.from(output.matchAll(/(\d+)\s+fail\b/g));
  const failures = failMatches.length > 0
    ? parseInt(failMatches[failMatches.length - 1][1], 10)
    : 0;
  const errors = 0; // Bun does not emit a distinct "error" line; folded into fail.

  // Primary: Bun's "Ran N tests across M files" summary
  const summaryMatch = /Ran (\d+) tests across \d+ files/.exec(output);
  if (summaryMatch) {
    return { ok: true, count: { total: parseInt(summaryMatch[1], 10), failures, errors } };
  }
  // Fallback: last "N pass" line (older Bun or truncated output)
  const passMatches = Array.from(output.matchAll(/(\d+)\s+pass\b/g));
  if (passMatches.length > 0) {
    const lastMatch = passMatches[passMatches.length - 1];
    return { ok: true, count: { total: parseInt(lastMatch[1], 10), failures, errors } };
  }
  return { ok: false, reason: "could not determine test count — Bun output lacks both `Ran N tests` summary and `N pass` summary line" };
}
```

There is no pre-existing `parseFailuresAndErrors` helper in `test_count_parser.ts`; the failures/errors extraction inlines into `parseBun` as shown. **Important — sketch covers the same first-match bug class for `N fail` lines too**: per-file `N fail` summaries can appear before the trailing block, so the fail extractor uses `.matchAll(...).pop()` (last-match), mirroring the AC.1 pass-line fallback. Implementer verifies the precise field names of `TestCount` and the matching sibling-parser conventions at edit time.
