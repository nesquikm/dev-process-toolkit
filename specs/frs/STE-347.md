# STE-347: Verification convention + /implement verify-and-analyze step {#STE-347}

## Requirement

`/implement` verifies a change with the deterministic gate check (compiler + linter + tests) and a bounded self-review, then stops at the Phase 4 commit-approval pause. Nothing in that flow answers the question a human asks last: *does the built thing actually run / look right in the real app?* Projects solve this today with hand-authored, product-specific "drive the running thing" skills — e.g. the sibling `glacy-app-fe` ships `glacy-drive` (drives the running Flutter web app via the dart/Flutter MCP: tap, type, read the widget tree, check runtime errors) and `glacy-progress-e2e` (asserts the backend journey API end-to-end over HTTP). The toolkit can't ship those generically, but it can give them a **place to plug in**.

This FR adds the **hook + convention** half (FR-A of M93): a `## Verification` block in the project's CLAUDE.md that declares the project's check skill, and a new `/implement` **Phase 4b″** step that discovers + runs it after gate-check GREEN, surfaces the result advisorily, and — on failure — analyzes and *proposes* a remediation route back into the SDD loop. The generator/scaffold half (STE-348) and the docs/methodology half (STE-349) build on this convention.

**Chosen approach — hybrid "Approach 1 UX + Approach 2 guts"** (2026-07-01 `/brainstorm`). The seamless "just run it" UX of a purpose-built `/implement` step, but the failure-routing is **native** — it proposes the shipped `/brainstorm` / `/spec-write` / inline-fix paths rather than delegating to `/conformance-loop`, which is maintainer-only (it lives in the toolkit's own `.claude/skills/` QA harness) and is therefore absent from a toolkit *user's* project. STE-224's `/conformance-loop` is the design template for the verify→fix pattern, not a runtime dependency.

## Acceptance Criteria

- AC-STE-347.1: **`## Verification` convention documented.** CLAUDE.md gains an optional `## Verification` section with a closed top-level key set `{verify_skill, verify_mode}`. `verify_skill` names a project-local skill (a `.claude/skills/<name>` slug) or the literal `visual-check`; `verify_mode ∈ {advisory, blocking, manual}` with **default `advisory`** when the section or the key is absent. The convention is documented in `plugins/dev-process-toolkit/docs/layout-reference.md` and shown in `plugins/dev-process-toolkit/templates/CLAUDE.md.template`. {#AC-STE-347.1}
- AC-STE-347.2: **Discovery precedence.** A shared resolver resolves the check skill in order: (1) if `verify_skill` is set, use it; (2) else scan `.claude/skills/*/SKILL.md` for candidates whose slug matches `*drive*`/`*check*`/`*verify*` or whose frontmatter carries a `verify: true` marker, and if **exactly one** candidate is found, `/implement` offers to adopt it (writing `verify_skill` into `## Verification`) — never silently running an undeclared skill; (3) if multiple candidates are found, `/implement` lists them and asks — never guesses; (4) if none, the path is "no check declared" (handed to STE-348). {#AC-STE-347.2}
- AC-STE-347.3: **Run placement + advisory report.** When a check skill is resolved and `verify_mode ≠ manual`, `/implement` runs it in a new **Phase 4b″** hook — after Phase 4a (gate-check GREEN) and the 4b/4b′ hooks, before the Phase 4c step-14 report — and renders the outcome (pass/fail + a short captured-output summary) as a row/section in the step-14 report. With `verify_mode: manual`, `/implement` does **not** auto-run the skill; it prints a one-line reminder naming the skill and how to run it. {#AC-STE-347.3}
- AC-STE-347.4: **Failure analysis + propose (never auto-invoke).** On a failing check, `/implement` classifies the failure as `spec-gap` (built behavior diverges from the intended design/spec) or `impl-bug` (a code defect) and surfaces a recommendation naming the exact next command — `/dev-process-toolkit:brainstorm`, `/dev-process-toolkit:spec-write <FR-id>`, or an inline fix — rendered in the step-14 report. `/implement` does **not** auto-invoke any of them; the human runs the recommended command. {#AC-STE-347.4}
- AC-STE-347.5: **`blocking` gates the commit; `advisory` does not.** With `verify_mode: blocking`, a failing check blocks the Phase 4c step-15 commit approval — the commit is not offered until the check passes or the operator explicitly overrides — identical in spirit to a failed gate check. With `verify_mode: advisory` (the default), a failing check is reported but the step-15 approval still proceeds (the human decides). `manual` never blocks (no auto-run). {#AC-STE-347.5}
- AC-STE-347.6: **Capability tokens.** The Phase 4 step-14 closing summary emits exactly one literal capability token for the verification outcome — one of `verify_skill_passed`, `verify_skill_failed_advisory`, `verify_skill_failed_blocking`, `verify_skill_manual_reminder`, `verify_skill_none_declared` — and each token is added to the `skills/spec-write/SKILL.md` § 7 static capability map (with a `MUST emit` directive in `/implement`) so `/gate-check`'s `closing_summary_capability_keys` probe enforces it. {#AC-STE-347.6}

## Technical Design

**Scope: convention + `/implement` runtime hook.** New shared helpers; a new `/implement` Phase 4b″ section; doc + template + capability-map edits.

**Files touched:**
- `adapters/_shared/src/verification_config.ts` (new) — `readVerificationConfig(claudeMdPath) → { verifySkill: string | null, verifyMode: 'advisory' | 'blocking' | 'manual' }`. Parses the `## Verification` block line-by-line (mirrors `readDocsConfig` / task-tracking parsing); default `{ verifySkill: null, verifyMode: 'advisory' }` when the section or a key is absent; rejects out-of-closed-set keys/values with an NFR-10 canonical error.
- `adapters/_shared/src/scan_candidate_check_skills.ts` (new) — `scanCandidateCheckSkills(projectRoot) → { slug, path }[]` scanning `.claude/skills/*/SKILL.md` by slug pattern + `verify:` frontmatter marker; deterministic, read-only.
- `skills/implement/SKILL.md` — new **Phase 4b″: Project Verification** section between Phase 4b′ and Phase 4c; discovery precedence; advisory report rows into step-14; failure classify+propose block; `blocking` gate wiring at step-15; the six `MUST emit` capability-token directives.
- `templates/CLAUDE.md.template` — documented optional `## Verification` block (commented example, default advisory).
- `docs/layout-reference.md` — `## Verification` convention + closed key set.
- `skills/spec-write/SKILL.md` § 7 — six new capability keys + rendered prose in the static map.

**Config shape (illustrative):**
```
## Verification

verify_skill: glacy-drive        # a .claude/skills/<name> slug, or `visual-check`
verify_mode: advisory            # advisory (default) | blocking | manual
```

**Phase 4b″ placement (the load-bearing decision):** the check runs *after* the gate is GREEN (so a red gate short-circuits first — never drive a broken build) and *before* the step-14 report + step-15 approval (so the result informs the human's commit decision and `blocking` can gate it). It reuses `/implement`'s existing human-gated, advisory-note machinery rather than introducing a second loop.

**Non-goal:** no auto-fix loop. Failure routing is advisory (AC-STE-347.4); the shipped `/brainstorm` / `/spec-write` / inline-fix skills are proposed, never auto-invoked. STE-224's `/conformance-loop` is the pattern reference, not a dependency (it is maintainer-only).

## Notes

- **Origin.** 2026-07-01 `/brainstorm` (topic: "encourage users to create custom check skills, incl. visual checks after `/implement`; see `../st/glacy-app-fe` drive* skills") → `/spec-write`. Approved hybrid "Approach 1 UX + Approach 2 guts."
- **Key finding.** `/conformance-loop` + `/smoke-test` live in the toolkit's own `.claude/skills/` (maintainer-only QA harness), **not** in the shipped `plugins/dev-process-toolkit/skills/`. So "reuse Approach 2's guts" became *native failure-routing* — a toolkit user's project has no `/conformance-loop` to delegate to. STE-224 is the pattern template only.
- **Lineage.** Realizes the "Deliverable B — /implement visual conformance check" that M91 (STE-341) explicitly deferred to "the next milestone": a project's check skill (or `/visual-check`) is exactly what compares built UI against `specs/design/` references.
- **Foundation FR.** STE-348 (generator + scaffold + fallback) and STE-349 (docs + examples) both build on the `## Verification` convention defined here.
- **`verify_mode` default is advisory** because a project check may need a live environment (backend up, Chrome, login — like `glacy-drive`); a hard commit-block by default would wedge `/implement` on a flaky env. `blocking` is opt-in per project.
