# STE-378: Token Stats opt-in — `## Token Stats` config section + `readTokenStatsConfig` + `/setup` interview {#STE-378}

## Requirement

The M92 per-skill token-usage stats feature (STE-344 capture hook → STE-345/346 renders) is **unconditionally on** for every project that has the plugin installed: the bundled `SessionEnd`/`Stop` hook writes `.dev-process/token-ledger.jsonl` on every session, and three skills render `## Token Stats` blocks at commit boundaries. There is no per-feature switch — the only "off" is disabling the whole plugin. Operators who don't want token accounting (privacy, no background writes, less noise in their FR files) have no opt-out, and the feature turns itself on the moment a project installs the toolkit.

This FR adds the **config-surface half** (FR-A of M102): a dedicated, always-emitted `## Token Stats` section in the project's `CLAUDE.md` with a closed key set `{enabled}` defaulting to `false`; a shared `readTokenStatsConfig` parser that any skill (and the capture hook) reads; and a `/setup` interview question that asks the operator, always writes the section, and records an audit row. **[[STE-379]]** (FR-B) consumes the parser to actually gate capture + render — this FR ships the surface, FR-B ships the enforcement.

**Chosen approach — dedicated `## Token Stats` section** (2026-07-10 `/brainstorm`). Three homes were weighed: (1) a dedicated section mirroring `## Verification` (STE-347); (2) a bare top-level line like `milestone_scan_fetch`; (3) a fourth key inside the existing `## Docs` block. The operator chose the dedicated section for clean semantics (token stats are not documentation), discoverability, and extensibility (room for future keys such as retention or model filters). The "existing projects default OFF until they re-run `/setup`" guarantee comes free from the `absent ⇒ default` parser convention proven by STE-68's `readDocsConfig` and STE-347's `readVerificationConfig`.

## Acceptance Criteria

- AC-STE-378.1: **`## Token Stats` convention.** `CLAUDE.md` gains an **always-emitted** `## Token Stats` section with a closed top-level key set `{enabled}`; `enabled ∈ {true, false}` with **default `false`** when the section or the key is absent. The section is always written (like `## Docs`, not a commented example like `## Verification`) so `/setup` can toggle it and downstream readers have something to parse. Documented in `plugins/dev-process-toolkit/docs/layout-reference.md` and shown in `plugins/dev-process-toolkit/templates/CLAUDE.md.template` with `enabled: false`. {#AC-STE-378.1}
- AC-STE-378.2: **`readTokenStatsConfig` parser.** A new `adapters/_shared/src/token_stats_config.ts` exports `readTokenStatsConfig(claudeMdPath): { enabled: boolean }`, parsing the `## Token Stats` block line-by-line and terminating at the next `^#{1,4} ` heading (mirrors `readVerificationConfig` / `readDocsConfig`). A missing file ⇒ `{ enabled: false }`; a present file with no `## Token Stats` section ⇒ `{ enabled: false }` (the backward-compat guarantee — projects predating this flag stay OFF until they re-run `/setup`); `enabled` accepts only lowercase literal `true`/`false`, and any other value throws a named `MalformedTokenStatsConfigError` in the NFR-10 canonical shape. Keys outside the closed set are rejected in the same NFR-10 shape. {#AC-STE-378.2}
- AC-STE-378.3: **`/setup` interview.** `/setup` asks a single one-at-a-time `AskUserQuestion` (per the Socratic Loop Contract — never bare prose) whether to enable token stats, with a safe **default of `false`**. It **always** writes the `## Token Stats` section (`enabled: false` when declined or unanswered). On a re-run over a project that already has the section, the prompt shows `[current: <bool>]` and empty input keeps the current value; the section is spliced atomically (full-block rewrite, not per-line edits) exactly as STE-68's `## Docs` re-run does. {#AC-STE-378.3}
- AC-STE-378.4: **Audit row.** Each resolved value appends a `## /setup audit` row via the existing `appendAuditRow` / `synthesize_audit.ts` machinery, of the form `step:<n> (token_stats.enabled) value:<bool> reason:"<user-supplied|default applied>"` — same provenance shape the three `docs.<flag>` rows use. {#AC-STE-378.4}
- AC-STE-378.5: **Dogfood — this repo opts in.** The toolkit's own `CLAUDE.md` carries a `## Token Stats` section with `enabled: true`, so the toolkit keeps recording and rendering its **own** token stats after default-off ships. The template default stays `false`; only this repo (which dogfoods the feature) opts in. {#AC-STE-378.5}
- AC-STE-378.6: **Tests.** `adapters/_shared/src/token_stats_config.test.ts` covers the parser matrix: absent file ⇒ `enabled:false`; present file without the section ⇒ `enabled:false`; `enabled: true` and `enabled: false` parsed; malformed value throws `MalformedTokenStatsConfigError`; a key outside `{enabled}` rejected; a `token_stats`-shaped key inside a trailing comment/next section is not misread (boundary termination). A template-shape meta-test under `tests/` asserts `templates/CLAUDE.md.template` carries the always-emitted `## Token Stats` heading with `enabled: false`. A `/setup` meta-test asserts the interview step exists and the audit row is synthesized. {#AC-STE-378.6}

## Technical Design

**Scope: config surface + parser + `/setup` interview.** No capture or render behavior changes here — that is **[[STE-379]]**. This FR produces the parser and the section that FR-B reads.

**Files touched:**
- `adapters/_shared/src/token_stats_config.ts` — **new.** `readTokenStatsConfig(claudeMdPath) → { enabled: boolean }`. Near-copy of `verification_config.ts`: find `## Token Stats`, read flat `key: value` pairs until the next heading, `switch` on the single known key `enabled`, `parseBool` accepts only lowercase `true`/`false`. `const ALL_FALSE = { enabled: false }` returned for missing file / missing section. `MalformedTokenStatsConfigError extends Error` in the NFR-10 canonical shape (message names the offending value + remedy); out-of-closed-set keys throw the same.
- `templates/CLAUDE.md.template` — add an always-emitted `## Token Stats` block (documented HTML-comment preamble mirroring the `## Docs` preamble + the live section):
  ```
  ## Token Stats

  enabled: false
  ```
- `skills/setup/SKILL.md` — add a new one-at-a-time interview step (a Step-7 sibling, e.g. **7f**), `default: false`, `requires-input: false` (safe default). Prompt: *"Enable per-skill token-usage stats? (records token usage per skill into a git-ignored ledger and renders `## Token Stats` tables at commit boundaries)"* `[y/n]`. Always emit the section; re-run shows `[current: <bool>]`, empty = keep; atomic full-section splice. Under non-tty stdin with no pre-baked answer, the safe default (`false`) applies — this is not a `requires-input` gate.
- `adapters/_shared/src/setup/synthesize_audit.ts` — synthesize the `token_stats.enabled` audit row alongside the three `docs.<flag>` rows.
- `CLAUDE.md` (this repo) — add `## Token Stats` / `enabled: true` (dogfood; AC-STE-378.5).
- `docs/layout-reference.md` — document the `## Token Stats` convention + closed key set.

**Config shape (illustrative):**
```
## Token Stats

enabled: false            # true | false — record + render per-skill token stats. Default false.
```

**Absent ⇒ OFF is load-bearing.** The parser's `ALL_FALSE`-on-missing-section behavior is what gives existing projects the "off until they re-run `/setup`" guarantee for free (STE-68 AC.3 pattern). No migration step, no probe, no backfill — a project that predates the flag simply has no `## Token Stats` section, and every reader (including FR-B's hook + render gates) sees `enabled:false`.

**No presence probe.** Unlike `## Docs` (probe #18 requires the section in a toolkit-managed `CLAUDE.md`), `## Token Stats` gets **no** hard gate-check presence probe: `absent ⇒ OFF` is a legal, supported state for any project that has not re-run `/setup`. The template carries the section (enforced by the template-shape meta-test) so fresh setups get it; older projects are legally without it.

## Testing

- `adapters/_shared/src/token_stats_config.test.ts` — the full parser matrix in AC-STE-378.6 (absent file, absent section, `true`/`false`, malformed-throws, out-of-set key, boundary termination), byte-for-byte the shape of `verification_config.test.ts`.
- Meta-tests under `tests/` — `templates/CLAUDE.md.template` carries the always-emitted `## Token Stats` + `enabled: false`; `skills/setup/SKILL.md` carries the new interview step (one-at-a-time, default false, re-run `[current]` handling); `synthesize_audit.ts` emits the `token_stats.enabled` row; `docs/layout-reference.md` documents the convention.
- Gate: `bun test` (full gate from `plugins/dev-process-toolkit`) + `/dev-process-toolkit:gate-check` GREEN on this repo.

## Notes

- **Origin.** 2026-07-10 `/brainstorm` (topic: "token tracker behind a config flag — `/setup` asks, disabled by default, disabled for existing projects until `/setup` turns it on") → `/spec-write`. Approved design: dedicated `## Token Stats` section, fully-off semantics (FR-B), new `token_stats_disabled` disposition (FR-B).
- **Rejected alternatives.** (a) A fourth key inside `## Docs` — cheapest, but semantically muddy (token stats are not docs) and entangled with the `## Docs` both-false NFR-10 refusal logic. (b) A bare top-level line like `milestone_scan_fetch` — leanest parse, mode-agnostic, but least discoverable and not extensible. The operator chose the dedicated section for clarity + room to grow.
- **Precedents.** STE-347 `## Verification` (dedicated optional section + `readVerificationConfig` safe-default parser) is the structural twin; STE-68 `## Docs` (`readDocsConfig` absent⇒false + atomic re-run splice) is the interview + backward-compat pattern.
- **Dogfood consequence.** Default-off applies to this repo too. Without AC-STE-378.5 the toolkit would stop rendering its own `## Token Stats` blocks the moment FR-B's gate lands; setting `enabled: true` in this repo's `CLAUDE.md` keeps the dogfood alive.
- **Foundation FR.** **[[STE-379]]** (capture-hook + render gating + disposition token) reads `readTokenStatsConfig` and is not implementable before this FR exists. Build order: STE-378 → STE-379.
