# STE-392: Assisted migration entry: monolithic specs → per-FR layout (hybrid, status-aware split) {#STE-392}

## Summary

Long-lived projects that predate the per-feature spec layout keep everything in one large requirements file and one large plan file. A new operator-guided migration splits that content: open work becomes individual feature files under the current conventions, while shipped history is frozen as a read-only legacy archive with a pointer left behind. The operator triages which acceptance criteria are genuinely still open before anything moves, a mandatory filesystem backup protects trees whose specs are not under version control, and the migration is re-runnable — a tree that already conforms reports nothing to do.

## Requirement

The monolithic layout (single `specs/requirements.md` carrying every FR section + single `specs/plan.md` carrying every milestone) was retired in v1.16.0, and the one-way migrator that covered it was deleted in v1.20.0 — since then a v1-vintage tree has had **no supported path** to the current conventions. The operator's own stock_check_app is exactly such a tree: 33 FR-heading sections (FR-8…FR-41) in a 1221-line monolith, a 1098-line plan with all checkbox state, hand-rolled archives, and the whole `specs/` tree git-ignored (so no git-side safety net exists for any transform).

This FR registers the first `kind: "assisted"` entry in STE-391's migration registry: an operator-guided monolith→per-FR split with the **hybrid, status-aware** shape approved at brainstorm (2026-07-17) — open work is split into active per-FR files; shipped history is frozen, not ticketed. Deterministic mechanics (detection, parsing, classification, prefix rewriting) are pure tested helpers; judgment steps (AC triage, split approval) stay with the operator via the Socratic contract.

## Acceptance Criteria

- AC-STE-392.1: **Registry registration.** The registry gains its first `kind: "assisted"` entry (`id: monolith-split`, `introduced_in: v1.16.0` — the convention-pivot release). Detector: FR-heading sections present in `specs/requirements.md` AND `specs/frs/` absent or empty ⇒ `{applies: true, evidence}` naming the section count and plan shape; a conforming tree ⇒ `applies: false`. Pure, deterministic, network-free, non-mutating. {#AC-STE-392.1}
- AC-STE-392.2: **Backup rail.** Before any mutation, the flow copies the full `specs/` tree to a timestamped sibling backup directory at the project root; copy failure aborts pre-mutation with NFR-10 shape. Mandatory regardless of VCS state — a git-ignored specs tree has no other safety net. The backup path is surfaced in the closing summary. {#AC-STE-392.2}
- AC-STE-392.3: **Status triage.** The flow derives a per-FR open/shipped classification from plan checkbox state + monolith cross-references (pure helpers: FR-section parser, checkbox-state extractor, classifier) and presents the derived table for operator confirmation with per-FR override capability before anything moves — no silent classification, `AskUserQuestion` per the Socratic contract. {#AC-STE-392.3}
- AC-STE-392.4: **Split step (open work).** Operator-confirmed open FRs become active `specs/frs/` files via the canonical helpers (`buildFRFrontmatter`, `acPrefix`, `Provider.filenameFor`), identity-mode-aware per the STE-134 rules: `mode: none` mints ULIDs locally (short-ULID filenames + AC prefixes); tracker mode routes each open FR through the `/spec-write` § 0b creation path (ticket create, no claim on create). Legacy dotted AC IDs are rewritten to the derived prefix; each split file carries a one-line provenance note naming its legacy FR number. Milestone bindings retain the legacy M-numbers. {#AC-STE-392.4}
- AC-STE-392.5: **Freeze step (shipped history).** The monolithic requirements + plan files relocate to the specs archive as read-only legacy documents (`git mv` when tracked, filesystem move otherwise) with content preserved byte-for-byte; a fresh cross-cutting-only `requirements.md` is scaffolded from the current template (STE-129 scope) carrying a pointer line to the legacy archive; milestones with ≥ 1 surviving open FR get an active `specs/plan/M<N>.md` stub carrying the remaining work — fully-shipped milestones stay frozen in the legacy archive. An empty open set (everything shipped) is a legal outcome: freeze-everything, zero active files. {#AC-STE-392.5}
- AC-STE-392.6: **Ignored-specs advisory + commit leg.** When `specs/` is git-ignored, the flow surfaces a loud advisory (current conventions treat committed specs as the source of truth; recommend removing the ignore rule) and skips the commit leg; when tracked, the standard one-approval-commit rail from AC-STE-391.2 applies. The advisory never edits the consumer's `.gitignore`. {#AC-STE-392.6}
- AC-STE-392.7: **Docs.** `docs/upgrade-reference.md` gains the assisted-entry walkthrough (detect → backup → triage → split → freeze → commit/advisory), including re-run semantics (post-split detector goes quiet) and the restore-from-backup recovery path. {#AC-STE-392.7}
- AC-STE-392.8: **Full gate green.** `bun test` from `plugins/dev-process-toolkit` passes; detector + parser + classifier + rewriter fixtures prove non-vacuity (monolith fixture applies; post-split fixture does not); no probe regresses. {#AC-STE-392.8}

## Technical Design

**Module:** `adapters/_shared/src/migrations/monolith_split.ts` — the entry's `detect` plus the pure mechanics:

| Helper | Contract |
|---|---|
| `parseMonolithFRSections(md)` | `### FR-<N>: <title> {#FR-N}` sections → `{frNumber, title, acLines, background}` ; skips `> archived:` pointer blockquotes |
| `extractPlanCheckboxState(md)` | plan checkbox rows → map of AC-ref → checked/unchecked/partial (`[~]`) |
| `classifyFRs(sections, state)` | per-FR `{disposition: open | shipped, evidence}` — split-brain tolerant (checkbox state lives only in the plan) |
| `rewriteAcPrefix(lines, prefix)` | legacy dotted `AC-<N>.<M>` → `AC-<PREFIX>.<M>`, order preserved, provenance line appended |

**Flow prose** lives in `skills/upgrade/SKILL.md` § Assisted entries (registered by id). Sequence: detect → backup (`specs-backup-<ISO-ts>/` sibling dir, collision-suffixed) → triage (derived table + operator overrides) → split (per identity mode) → freeze (relocate + fresh cross-cutting scaffold from `templates/spec-templates/requirements.md.template` + plan stubs) → commit or ignored-specs advisory → summary.

**Recovery:** any partial failure instructs restore-from-backup (documented in AC.7's walkthrough); the detector re-fires on a restored tree, so re-running is the recovery path — no bespoke rollback machinery.

## Notes

- **Depends on STE-391** — registry, runner rails, and the assisted-entry routing must exist. Build order: A → B.
- **Pilot tree:** stock_check_app — mode `none` by omission, specs git-ignored, 33 FR sections; operator confirmed 2026-07-17 that all 45 unchecked plan ACs are actually complete ⇒ the pilot takes the freeze-everything path (empty split set). Triage exists precisely because checkbox state lies.
- **Identity-mode rules** per STE-134: TrackerProvider has no `mintId()`; tracker-mode split routes through ticket creation instead. Mode-none is the pilot's path.
- **Out of scope:** restructuring `technical-spec.md` / `testing-spec.md` (they remain cross-cutting files under current conventions); tracker backfill of shipped history (frozen, never ticketed); editing the consumer's root `.gitignore` (advisory only).
- Origin: `/brainstorm` 2026-07-17 (hybrid depth chosen over full-fidelity split and forward-only adoption; "the --migrate should be smart" — status-aware with operator triage).
