# STE-153: /setup always synthesizes ## /setup audit for toolkit-managed CLAUDE.md {#STE-153}

## Requirement

When `/dev-process-toolkit:setup` runs in autonomous mode with all five Schema L answers explicitly supplied (the canonical "pre-baked answers" path), the resulting `CLAUDE.md` is missing the `## /setup audit` section. `/gate-check` probe #19 (`setup-audit-section-presence`) then hard-fails on the next pass: the toolkit marker is present, `branch_template:` is populated, the `## Docs` block is present, but the audit heading is absent. Verdict: **GATE FAILED** on the most common autonomous code path.

Root cause: `/setup` step 8a's verification is gated on `hasDefaultApplicableOutcomes(content)` (true for this run — `branch_template:` populated AND `## Docs` present), but the in-scope `resolvedDefaults` table only contains entries that were *actually* default-resolved. With all five answers user-supplied, the table is empty, so `synthesizeAuditSection` either no-ops (no candidates) or throws `AuditPostconditionUnsatisfiable`. Either way, the file lands without the audit section while probe #19 demands it.

The fix flips the contract: every Schema L resolution at steps 7b/7c/7d records into the in-scope resolved-values table with a per-entry `reason:` field — `"user-supplied"` or `"default applied"` — not just default-applied ones. The audit section becomes a complete provenance log of every resolved value, present on every well-formed `/setup` run that produces a toolkit-managed file. Probe #19 stays as-is; `AuditPostconditionUnsatisfiable` becomes truly unreachable on a well-formed run.

## Acceptance Criteria

- AC-STE-153.1: After `/setup` runs with **all** five Schema L answers explicitly supplied (no defaults applied), `CLAUDE.md` contains a `## /setup audit` section with one entry per resolved value (`step:7c branch_template`, `step:7d docs.user_facing_mode`, `step:7d docs.packages_mode`, `step:7d docs.changelog_ci_owned`), each carrying `reason: "user-supplied"`.
- AC-STE-153.2: After `/setup` runs in fully autonomous mode with no answers supplied (all defaults applied), `CLAUDE.md` contains the same set of audit entries with `reason: "default applied"` — existing behaviour preserved byte-for-byte.
- AC-STE-153.3: A mixed run (some user-supplied, some default-applied) produces audit entries with the appropriate per-entry `reason:` field — provenance is preserved per resolution, not per run.
- AC-STE-153.4: `/gate-check` probe #19 (`setup-audit-section-presence`) exits clean after every `/setup` invocation that produces a toolkit-managed `CLAUDE.md` with `branch_template:` populated **or** `## Docs` present, regardless of provenance mix. The smoke-test "all-explicit-answer" scenario from run #4 is now green.
- AC-STE-153.5: `synthesizeAuditSection` no longer throws `AuditPostconditionUnsatisfiable` on a well-formed `/setup` run. The throw stays as a defensive invariant for genuinely inconsistent in-memory state, with updated JSDoc clarifying it is now unreachable from the canonical 7b/7c/7d → 8a path.
- AC-STE-153.6: `skills/setup/SKILL.md` lines 17, 176, 189, and the step 8a block (lines 222–229) update to drop the "default-applied only" framing — wording reads "every resolved outcome (user-supplied or default-applied)". - AC-STE-153.7: `docs/setup-reference.md` § Step 8a updates to match the new framing — the M31-class "discretionary-check gap" prose is rewritten to describe the unconditional-synthesis model.
- AC-STE-153.8: All existing `tests/setup_audit_log.test.ts` cases stay green. All existing `adapters/_shared/src/setup/synthesize_audit.test.ts` cases stay green — including AC-STE-123.4 (the throw-on-empty defensive test) which exercises a synthetic invariant violation, not a real `/setup` path.

## Technical Design

**Touchpoints:**

- `plugins/dev-process-toolkit/skills/setup/SKILL.md`
  - Line 17 (top-of-file overview of `default:` semantics): replace "Each default-applied outcome appends" with "Every resolved outcome (user-supplied or default-applied) appends" — provenance recorded in the entry's `reason:` field.
  - Line 176 (step 7c): "Every resolved branch_template value appends `## /setup audit` entry recording `step:7c value:"<resolved>" reason:"<user-supplied|default applied>"`."
  - Line 189 (step 7d): "Every resolved `## Docs` flag appends `## /setup audit` entry: `step:7d (docs.<flag>) value:<bool> reason:"<user-supplied|default applied>"`."
  - Lines 222–229 (step 8a): rewrite from "branch on `hasDefaultApplicableOutcomes`" to the unconditional flow:
    1. Read `CLAUDE.md` once.
    2. If toolkit marker absent ⇒ no-op.
    3. If at least one Schema L surface is populated (`branch_template:` populated **or** `## Docs` present) ⇒ call `synthesizeAuditSection(claudeMdPath, resolvedSchemaLValues)`. Helper is idempotent (`(step, field)` dedup).
    4. The trigger predicate stays in `setup_audit_section_presence.ts`; both `/setup` step 8a and probe #19 reuse it.

- `plugins/dev-process-toolkit/adapters/_shared/src/setup/synthesize_audit.ts`
  - Rename `ResolvedDefaults` interface → `ResolvedSchemaLValues` (alias keeps the old name for one minor version if any external consumer references it; otherwise drop).
  - Rename `ResolvedDefaultEntry<T>` → `ResolvedSchemaLEntry<T>` — `reason: string` field unchanged, accepts both `"user-supplied"` and `"default applied"`.
  - JSDoc on `AuditPostconditionUnsatisfiable`: clarify the throw is a defensive invariant for inconsistent in-memory state — unreachable on the canonical `/setup` 7b/7c/7d → 8a path because every Schema L resolution always populates the table.
  - `collectCandidates` body unchanged — still returns one Candidate per populated entry. The behaviour change is in *what populates the table*, not in `synthesizeAuditSection` itself.

- `plugins/dev-process-toolkit/adapters/_shared/src/setup/audit_log.ts`
  - **No code change.** `appendAuditEntry`'s `entry.reason: string` already accepts arbitrary provenance strings.

- `plugins/dev-process-toolkit/adapters/_shared/src/setup_audit_section_presence.ts`
  - **No probe code change.** Trigger predicate `hasDefaultApplicableOutcomes` retains its current shape (rename optional; the function is reused as-is by both /setup step 8a and probe #19).

- `plugins/dev-process-toolkit/docs/setup-reference.md`
  - Lines 134–143 (§ Step 8a) update — drop the "M31-class discretionary-check gap" framing in favor of the unconditional-synthesis model. The audit section is now a deterministic byproduct of every well-formed `/setup` run, not a recovery mechanism for missed appends.

**Behaviour matrix:**

| Schema L answers     | `branchTemplate` resolved-values entry                              | `docs.*` resolved-values entries                                    | Audit section after step 8a                                       | Probe #19 |
|----------------------|---------------------------------------------------------------------|---------------------------------------------------------------------|-------------------------------------------------------------------|-----------|
| All user-supplied    | `value:"<custom>", reason:"user-supplied"`                          | three entries, each `reason:"user-supplied"`                        | 4 entries, all `reason:"user-supplied"`                           | pass      |
| All defaults applied | `value:"<default-for-mode>", reason:"default applied"`              | three entries, each `value:false reason:"default applied"`          | 4 entries, all `reason:"default applied"`                         | pass      |
| Mixed                | (per-question)                                                      | (per-question)                                                      | 4 entries with mixed `reason:` values                             | pass      |
| `mode: none` + 7c skip | absent (branch_template not written)                              | three entries (always emitted)                                      | 3 entries (no `step:7c` row)                                      | pass      |
| Hand-written CLAUDE.md (no toolkit marker) | (call site never reached)                         | (call site never reached)                                           | unchanged                                                         | vacuous   |

`mode: none` row uses the existing 7c skip rule — `branch_template:` is omitted from `CLAUDE.md`, the resolved-values table omits the `branchTemplate` entry, and `synthesizeAuditSection` simply produces 3 audit entries instead of 4. Probe #19 still passes because `## Docs` is present.

## Notes

Surfaced by `/smoke-test` run #4 against plugin v1.41.0, 2026-04-29. Evidence files: `/tmp/dpt-smoke-findings-2026-04-29.md`, `/tmp/dpt-smoke-setup.log`, `/tmp/dpt-smoke-gate-check.log`.

Severity: **high** — affects the canonical autonomous-mode `/setup` path (all-explicit answers), the most common scripted/CI scenario.

Resolution chosen: **option A** from the findings doc — fix on `/setup`'s side by recording every resolved value with provenance, so the audit section becomes a deterministic byproduct of every well-formed run. **Option B rejected** (probe #19 differentiates provenance): would require the probe to re-read the audit section to learn provenance, which is circular and adds parser complexity to gate-check for no operator-visible benefit.

The change strictly extends the audit log — every entry that previously landed (default-applied) still lands with the same `reason:` value byte-for-byte. New entries are user-supplied resolutions that previously left no trace; their addition cannot regress any existing /gate-check probe (probe #19 only checks heading presence; per-entry inspection is not in scope today).

Future-proofing: the `reason:` field is already a free-form string, so further provenance values (e.g., `"pre-existing"` for the idempotent-merge branch from run #3) can be added without schema change.
