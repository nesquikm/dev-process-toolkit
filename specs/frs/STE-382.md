# STE-382: Consolidate `.dpt-locks/` + `.dev-process/` into a single `.dpt/` tree with a path single-source-of-truth {#STE-382}

## Requirement

The toolkit creates two root-level folders in every consumer project with **opposite VCS semantics**: `.dpt-locks/` holds tracked, committed lock files (`LocalProvider.claimLock` does `git add` + `git commit` per claim; cross-branch detection reads them via `git ls-tree` against remote branch tips), while `.dev-process/` holds the git-ignored `token-ledger.jsonl`. Consolidating them into one `.dpt/` tree removes the root clutter, but the split that actually matters is **tracked vs ignored**, and it cuts across both current folders ŌĆö research scratch (`*-research-result.txt`) currently lives in the **tracked** `.dpt-locks/`, where STE-230 sited it for adjacency, not durability (STE-230:117 called the site "TBD at implementation timeŌĆ” the natural site is `.dpt-locks/<ulid>/spec-research-result.txt`").

That co-tenancy is also a live namespace collision: `.dpt-locks/<id>` is a **file** (locks, `writeFileSync` at `local_provider.ts:142`) while `.dpt-locks/<ulid>/spec-research-result.txt` makes `<ulid>` a **directory**. `findStaleLocks` (`local_provider.ts:216-217`) readdirs the folder and `readFileSync`s every entry with no `isDirectory` guard and no try/catch ŌĆö a research-scratch dir makes it throw `EISDIR`. Mechanism reproduced 2026-07-15; it is latent rather than live only because `findStaleLocks` has callers exclusively in `mode: none` while this repo is `mode: linear`. Separating tracked locks from ignored scratch dissolves the collision structurally; no defensive guard is needed once the namespaces never overlap.

The rename also exposes a **latent bug**: `findRemoteBranchWithLock` (`local_provider.ts:160`) hard-codes the literal `.dpt-locks/${id}` in its `ls-tree` pathspec, while the write (`:107`) and release (`:167`) paths derive from `this.locksDir`. Today the default coincides with the literal, so it is harmless ŌĆö and any test injecting a custom `locksDir` is testing a path production never takes. After the move, `:160` would scan a folder that no longer exists, `ls-tree` would return empty for every branch, and lock detection would silently answer "unclaimed" ŌĆö the exact double-claim failure this milestone exists to prevent, hiding one line from the fix.

Approved via `/brainstorm` 2026-07-15. Zero installs ŌćÆ forward-only; no migration rails (STE-23's backup-tag / dry-run / migration-commit ceremony is dead weight with no consumer trees to migrate). The existing `.dev-process/token-ledger.jsonl` is abandoned in place (operator choice); the ledger re-accrues under `.dpt/ledger/`.

## Acceptance Criteria

- AC-STE-382.1: **Path single-source-of-truth.** New `adapters/_shared/src/dpt_paths.ts` is the sole composer of `.dpt` path literals, exporting at minimum `dptRoot(projectRoot)`, `locksDir(projectRoot)` ŌåÆ `<root>/.dpt/locks`, `ledgerPath(projectRoot)` ŌåÆ `<root>/.dpt/ledger/token-ledger.jsonl`, and `scratchDir(projectRoot, ulid)` ŌåÆ `<root>/.dpt/scratch/<ulid>`. No other module composes a `.dpt` string literal. {#AC-STE-382.1}
- AC-STE-382.2: **`:160` hard-code fixed.** `LocalProvider.findRemoteBranchWithLock` derives its `ls-tree` pathspec from `this.locksDir` (repo-root-relative via `relative(this.repoRoot, this.locksDir)`), not a hard-coded string. A unit test injects a non-default `locksDir` and asserts the cross-branch scan looks under the injected path ŌĆö the assertion fails against today's code. {#AC-STE-382.2}
- AC-STE-382.3: **Lock paths re-pointed.** `LocalProvider`'s `locksDir` default derives from `dpt_paths` ŌåÆ `.dpt/locks`; `claimLock` / `releaseLock` / `findStaleLocks` / `cleanupStaleLocks` all operate under it, and locks remain **tracked and committed** (STE-28 semantics preserved: `git add` + `git commit` per claim, `ls-tree` cross-branch detection intact). {#AC-STE-382.3}
- AC-STE-382.4: **Ledger path re-pointed.** `token_usage.ts` `ledgerPath()` derives from `dpt_paths` ŌåÆ `.dpt/ledger/token-ledger.jsonl`. The capture hook's fail-open contract and atomic temp-file+rename write are unchanged; no tracked file is ever modified by the capture path (the AC-STE-344.6 invariant is preserved under the new path). {#AC-STE-382.4}
- AC-STE-382.5: **Scratch relocated out of the tracked namespace.** Research-result persistence moves from `.dpt-locks/<ulid>/` to `.dpt/scratch/<ulid>/`; probe #41 (`spec_research_result_shape`) and probe #64 (`deps_research_result_shape`) walk `.dpt/scratch/**` via `dpt_paths`, including the flat-fallback path. **Vacuity is preserved**: an absent scratch dir ŌćÆ pass with no note (the STE-230 AC-STE-230.12 contract), so a run that invoked no research fork stays green. {#AC-STE-382.5}
- AC-STE-382.6: **Remaining literals swept.** `bun_zero_match_placeholder.ts` `SKIP_DIRS` carries `.dpt` (not `.dpt-locks`); `/spec-write` ┬¦ 0b step 2.5b and `/brainstorm` ┬¦ 1.5b persist-path prose name `.dpt/scratch/<ulid>/<name>-research-result.txt`; `token_stats_render.ts` comments and `skills/setup/SKILL.md` prose no longer name `.dev-process/`. {#AC-STE-382.6}
- AC-STE-382.7: **EISDIR collision dissolved.** With locks under `.dpt/locks/` and scratch under `.dpt/scratch/`, `findStaleLocks` reads only lock files; a regression test places a directory-shaped entry in the scratch tree and asserts the stale-lock scan is unaffected. No defensive `isDirectory` guard is added ŌĆö the layout is the fix. {#AC-STE-382.7}
- AC-STE-382.8: **Full gate green.** `bun test` from `plugins/dev-process-toolkit` passes with meta-test calibration snapshots (SKILL.md line caps, adapters STE-token ceiling, README count pins) bumped as needed; no probe regresses. {#AC-STE-382.8}

| Site | Shape | Note |
|---|---|---|

`ledgerPath()` (`token_usage.ts:44`) is a genuine single source, but the `.dev-process` string also lives in `skills/setup/SKILL.md:158` prose and `token_stats_render.ts:347` comments.

## Notes

- **Out of scope ŌĆö symlink-cycle guard.** STE-373's known gap in the shared `findResultFiles` walk (`statSync` follows symlinks with no cycle guard, inherited byte-for-byte from #41). Its own Notes recommend a dedicated cross-cutting FR across the ~15-probe walk family; folding it in would smuggle a probe-hardening milestone into a relayout. Risk accepted there because `.dpt-locks/` is toolkit-created, not attacker-controlled ŌĆö the same reasoning holds for `.dpt/scratch/`.
- **Out of scope ŌĆö ledger migration.** Forward-only per operator choice: the legacy `.dev-process/` is abandoned in place (still git-ignored) and the ledger re-accrues under `.dpt/ledger/`. Rendered `## Token Stats` blocks already in FR files are durable, so the loss is limited to unrendered rows and their `claimed_by` marks.
- The `EISDIR` finding is latent, not live: `findStaleLocks` is reachable only in `mode: none`, and this repo is `mode: linear`. It would fire in a consumer project that runs `mode: none` and has ever invoked a research fork.
