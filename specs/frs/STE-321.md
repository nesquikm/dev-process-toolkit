# STE-321: Adapter shape — drop tracker_config allowlist + align mode-none frontmatter {#STE-321}

## Requirement

Two adapter-shape design decisions resolved in the M84 brainstorm session (2026-05-20), shipped as one FR.

**A11 — `tracker:` field in mode-none FR frontmatter (Code wins)**: Three sources disagreed on whether `tracker:` should appear in mode-none FR frontmatter. The brainstorm decision: **code wins — `tracker:` absent in mode-none.** `buildFRFrontmatter` at `adapters/_shared/src/fr_frontmatter.ts:120-125` is the canonical runtime; the helper emits only `id:` in mode-none and no `tracker:` field. The two drift surfaces are `specs/technical-spec.md:275-276` Schema Q example (shows `tracker: {}`) and `skills/gate-check/SKILL.md:26` (claims `tracker:` is mode-invariant required). Both need to align with the helper.

**A12 — `tracker_config.yaml` linear|jira allowlist (Drop allowlist)**: The brainstorm decision: **drop the hardcoded allowlist; cross-check against active adapter's `name:` field.** `adapters/_shared/src/tracker_config.ts:32` declares `type TrackerKey = "linear" | "jira"` and L44 declares `SUPPORTED_TRACKER_KEYS: readonly string[] = ["linear", "jira"]`. L109-114 validator hard-rejects anything else with `TrackerConfigShapeError`. But `adapters/_template.md` + `docs/tracker-adapters.md § Authoring a custom adapter` advertise custom-adapter extensibility. Net effect: a `gitlab` / `azure-devops` adapter author CAN write the adapter `.md` but CANNOT use `specs/tracker-config.yaml` and therefore loses STE-303 (MCP best-match) + STE-304 (tolerance wrapper).

The fix: drop `SUPPORTED_TRACKER_KEYS` and rely on the cross-check (partially present at L194-199) that `tracker_key` matches the active adapter's `name:` from `adapters/<name>.md`. STE-10 AC.2 confirms `name` is the canonical adapter identifier.

Bundled because both touch the adapter-shape contract and the design decisions were resolved together.

## Technical Design

**A11 fix**: two prose edits + probe-test expansion.

- `specs/technical-spec.md:275-276` — rewrite Schema Q mode-none example to drop `tracker: {}`. The Schema Q definition itself (the "what fields are required when" rule) stays mode-conditional per AC.4.
- `skills/gate-check/SKILL.md:26` — rewrite to make `id:` and `tracker:` parallel mode-conditional fields per probe #13. Same phrasing pattern STE-316 already establishes for `id:`.

Existing mode-none FR files: scan via `find specs/frs -name '*.md' -not -path '*/archive/*' -exec grep -l '^id: fr_' {} \;` returns the mode-none subset. Each one already lacks `tracker:` per `buildFRFrontmatter`'s current behavior — no migration needed. If the scan finds any with `tracker: {}` (unexpected drift), the FR adds a migration helper that strips the empty block.

**A12 fix**: code change + helper + test.

- Drop `type TrackerKey` and `SUPPORTED_TRACKER_KEYS` const from `tracker_config.ts`. Type signatures across the file generalize to `string` (matching `name` field from `adapters/<name>.md`).
- New helper `readAdapterName(claudeMdPath, adaptersDir, mode)` at `adapters/_shared/src/read_adapter_name.ts`. Reads `adapters/<mode>.md`, extracts the `name:` frontmatter field. Throws `MalformedAdapterMetadataError` (NFR-10 canonical) when the adapter file is absent or missing `name:`.
- Validator at L109-114 calls the helper and asserts `tracker_key === adapterName`. Mismatch is the regression `TrackerConfigShapeError` it always was, just sourced from runtime data instead of a hardcoded allowlist.

Sibling-package retrieval (STE-302 + STE-316 precedents): both follow the same "code-wins drift resolution" + "mode-conditional gate-check probe shape" patterns. The architectural-twin asymmetry between the two FRs (STE-302 introduced the allowlist intentionally; this FR drops it intentionally) is documented in the Notes section.

Spec-research seed: `spec_research_invoked` — 3 precedents returned (STE-302 tracker-config introducer, STE-303 mode-none vacuous pattern + adapter capability gate, STE-316 active sibling mode-conditional pattern).

Deps-research vacuous (`specs/deps.yaml` absent — zero entries → fork skipped).
