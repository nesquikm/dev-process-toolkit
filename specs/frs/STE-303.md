# STE-303: /setup MCP best-match + idempotent tracker-config write {#STE-303}

## Requirement

STE-302 introduces `specs/tracker-config.yaml` as the per-project source of truth for the project's status vocabulary + role-to-status mapping, with a loader API that adapters consume. The file is empty by construction until something writes it. This FR is the writer: `/setup` (in tracker mode, on first-run AND on re-entry against an already-configured project) queries the project's tracker MCP for the canonical status list, proposes a role mapping by LLM-judgment at runtime, surfaces a unified-diff approval gate borrowed from STE-9's canonical dry-run pattern, and writes the file via `writeTrackerConfig`.

LLM-judgment chosen over a curated alias table because (a) workspace vocabularies are unbounded — `In Code Review`, `Ready for QA`, `Awaiting Sign-off` aren't worth chasing in a static alias list, and (b) the proposal is gated by an explicit operator-approval diff, so non-determinism is bounded to a single human-reviewed proposal per `/setup` invocation. The judgment is composed inline in the running `/setup` skill — no new subagent, no separate fork — and the operator sees one-line reasoning per role.

Idempotent re-entry is the canonical resync path. On already-configured projects, `/setup` reads the existing `tracker-config.yaml` as the baseline, computes a fresh proposal from the current MCP-fetched status list (which may have drifted since the file was first written), and shows the operator a diff between the two. Accept, edit, or cancel; cancel keeps the existing file untouched. Mirrors STE-199's deferred-binding pattern: re-run is always safe.

- AC-STE-303.2: Linear adapter status-fetch: uses `mcp__linear__list_issue_statuses` (or the active adapter's documented canonical call — `adapters/linear.md` § `list_project_statuses` capability) to fetch the full per-team status list, scoped to the team bound in `## Task Tracking` § `### Linear`. Output: ordered array of status names verbatim from Linear (preserves casing, whitespace, special chars).
- AC-STE-303.3: Jira adapter status-fetch: uses the Jira workflow-introspection MCP call documented in `adapters/jira.md` § `list_project_statuses` capability (canonical method TBD at implementation time — likely `getJiraIssueTypeMetaWithFields` + extracting workflow states, or a dedicated workflow endpoint). Output shape matches AC.2 (ordered string array).
- AC-STE-303.4: LLM-judgment mapping format: for each of the 4 canonical roles (`initial`, `in_progress`, `in_review`, `done`), the proposal includes (a) the chosen status name from the fetched list, (b) a one-line reasoning string visible to the operator. If no plausible match exists for a role (e.g., project has no `In Review`-equivalent), the proposal explicitly emits `null` for that role + reasoning naming the gap. Operator can manually pick from the full status list during the `edit` branch of the approval gate.
- AC-STE-303.5: Unified-diff approval gate: rendered as standard text diff between baseline (current file or empty) and proposal (YAML serialization). Operator response routes via `AskUserQuestion`: `approve` → `writeTrackerConfig` lands the file; `edit` → opens an interactive per-role pick from the full status list (one `AskUserQuestion` per role with the status list as options); `cancel` → no write, file unchanged, capability row `tracker_config_write_cancelled` surfaced. Marker-driven default-apply (`<dpt:auto-approve>v1</dpt:auto-approve>`) auto-applies `approve` per STE-262.
- AC-STE-303.6: Idempotent re-entry: when `specs/tracker-config.yaml` exists, the file is loaded as baseline and compared against the fresh MCP-driven proposal. The diff shows only the delta. If the diff is empty (baseline == proposal), no prompt is shown and the run reports `tracker_config_unchanged` in the closing summary — operator wasn't asked for input on a no-op.
- AC-STE-303.7: `mode: none` vacuous — no MCP call, no proposal, no diff, no write, no closing-summary row. Branches via the same `LocalProvider` sentinel STE-302.6 honors.
- AC-STE-303.8: MCP-unavailable failure modes (STE-199 alignment): (a) MCP server unreachable, (b) `list_project_statuses` capability missing on adapter, (c) credentials invalid → surface NFR-10 canonical refusal naming the specific failure mode + remedy (`re-authenticate and re-run /setup`). No partial write. Re-run after fix is the resync path (STE-199 deferred-binding precedent).
- AC-STE-303.9: Adapter Schema W (capability declaration) gains `list_project_statuses: true` for `linear.md` and `jira.md`. Adapters without this capability surface `tracker_config_write_skipped_adapter_limit` capability row and skip the write path (graceful degrade).
- AC-STE-303.10: New tests at `plugins/dev-process-toolkit/tests/setup-tracker-config-write.test.ts` covering: first-run happy path (empty baseline → proposal → write); idempotent re-entry no-op (baseline == proposal → no prompt, no write); idempotent re-entry with delta (operator approves → file rewritten); `cancel` keeps baseline; `edit` path (per-role manual pick); `mode: none` vacuous (no MCP call observed); MCP unavailable → NFR-10 refusal; adapter without capability → graceful skip.
- AC-STE-303.11: Closing-summary capability keys added to `/setup` SKILL.md § closing summary static map (and mirrored to `/spec-write` § 7 cross-skill cross-reference): `tracker_config_write_succeeded`, `tracker_config_write_cancelled`, `tracker_config_unchanged`, `tracker_config_write_skipped_adapter_limit`, `tracker_config_write_mcp_unavailable`. ## Technical Design

### Architecture

```
/setup skill (tracker mode, idempotent re-entry)
   │
   ▼
1. detect mode + load CLAUDE.md workspace binding
2. MCP fetch project status list (Linear: list_issue_statuses / Jira: workflow-introspection)
3. readTrackerConfig(specsDir)  ← STE-302 loader; null on first-run
4. LLM-judgment role mapping (inline conversational step)
5. unified diff baseline ↔ proposal
6. AskUserQuestion: approve / edit / cancel
7. writeTrackerConfig(specsDir, config)  ← STE-302 loader; atomic
8. closing-summary capability row
```

### LLM-judgment placement

The role-mapping proposal is generated inline by the running `/setup` skill — the model composes it as part of its conversational flow, not via a forked subagent. Rationale:

- /setup is already an interactive, model-driven skill; the proposal is one more conversation step
- No fixed-shape block contract to maintain (unlike `spec-research-result` / `deps-research-result`)
- No extra agent file / subagent registration cost
- Reasoning lines are conversational text the operator reads in-flow

The skill body prompt for the mapping step is a structured request: given (status_list, role_enum), propose a mapping + 1-line reasoning per role. The result is YAML-serialized and shown in the diff.

### Diff rendering

Baseline ↔ proposal diff uses standard unified-diff format on the YAML serialization. Empty baseline (first-run) renders as `+` lines only. Identical baseline+proposal renders as a no-op short-circuit (AC.6).

### Capability boundary

`list_project_statuses` is a new adapter capability per Schema W. Adapters that opt out (e.g., a custom non-MCP adapter that can't introspect workflow) surface the `tracker_config_write_skipped_adapter_limit` capability row at /setup time; the loader's adapter-default fallback (STE-302.5) handles their case at read time.

## Notes

- **Why LLM-judgment over a curated alias table.** Workspace vocabularies are unbounded — `In Code Review`, `Awaiting Sign-off`, `Blocked on Customer` aren't worth chasing in a static list. Operator-approval gate bounds the non-determinism to a reviewable single proposal per `/setup` invocation. The operator can always pick manually via the `edit` branch.
- **Why inline (not a subagent).** Mapping doesn't have a fixed-shape block contract like `spec-research-result`. /setup is already a model-driven interactive skill. Adding a fork adds boilerplate without a clear win.
- **Re-entry as the resync path.** The same flow handles first-run and re-entry — only the baseline differs (empty vs. existing file). No separate "migrate" command needed.
- **Open question (deferred to implementation):** Linear's `mcp__linear__list_issue_statuses` returns statuses per team. The bound team in `## Task Tracking` § `### Linear` is the scope. Verify at implementation time that the MCP call signature accepts a team filter (or filter client-side).
- **Open question (deferred to implementation):** Jira's workflow-introspection canonical MCP call name not yet locked. Three candidates: (a) `getJiraIssueTypeMetaWithFields` + extracting statuses, (b) `searchJiraIssuesUsingJql` returning all unique statuses observed in the project, (c) a dedicated workflow-states endpoint. Pick during FR2 implementation; document the choice in `adapters/jira.md` § `list_project_statuses`.
