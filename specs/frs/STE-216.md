# STE-216: /simplify emits loud advisory when leaving the tree dirty {#STE-216}

## Requirement

`/simplify` modifies tracked files (e.g., collapsing `src/greet.ts` from 4 lines → 3 lines, removing intermediate `const`s without changing observable behavior) and verifies gates re-pass, but it leaves the modifications **uncommitted**. The operator silently inherits a dirty working tree; the actionable next step (`/pr` or `git commit`) is undocumented in the skill output, so the dirty-tree state can persist into the next skill invocation, the smoke driver's Phase 4 capture, or the operator's next session.

**Findings bundled:** smoke #6 F6, smoke #7 F7, smoke #8 F2, F1 from `/tmp/dpt-smoke-findings-2026-05-04-linear.md` (Run #10 v2.7.0). 4× recurring; consistent across four consecutive smoke runs.

**Symptoms observed across multiple smoke runs:**

1. After `/simplify` exits cleanly, `git status` shows `M src/<file>` — the simplification is staged-but-uncommitted.
2. The `/simplify` closing summary does not warn about the dirty tree; the operator must run `git status` independently to discover it.
3. Auto-commit was considered as the symmetric fix (matching `/implement` Phase 4) but the brainstorm-stage decision rejected it: CLAUDE.md core principle "human approval before commit" makes auto-commit inappropriate for `/simplify`.
4. The fix is a **loud advisory** that closes the silent-dirty-tree footgun while preserving the operator's explicit-commit posture.

## Acceptance Criteria

- AC-STE-216.1: After `/simplify` writes any change, the closing summary block runs `git status --porcelain` and inspects the result. If any file is modified, added, deleted, or otherwise dirty, the summary emits a one-line advisory: `⚠ tree dirty: M <files> — run /pr or git commit to land the simplification`. The `<files>` list contains the relative paths from `git status --porcelain`. {#AC-STE-216.1}
- AC-STE-216.2: The advisory line is prefixed with the warning glyph `⚠` and uses the literal token `tree dirty:` (machine-greppable; the smoke driver and operator-side automation can detect the dirty case deterministically by grepping for that token in `/simplify`'s stdout). {#AC-STE-216.2}
- AC-STE-216.3: New capability key `simplify_tree_dirty` is registered in the static plain-language map (the same map shared by `/spec-write` step 7 — see `skills/spec-write/SKILL.md` § 7). Rendered prose: `tree dirty after /simplify — modified files: <list>; run /pr or git commit to land the simplification`. The capability row emits in the closing summary alongside the inline advisory line. {#AC-STE-216.3}
- AC-STE-216.4: When the tree is clean post-`/simplify` (the no-op case where `/simplify` recognized the diff was already minimal and made no changes, or where the operator committed concurrently), no advisory fires; no capability row emits. The two-line dirty-case output is opt-in by tree state. {#AC-STE-216.4}
- AC-STE-216.5: The advisory fires regardless of whether `/simplify` itself made the change (the canonical case) or `/simplify` ran on a tree that was already dirty before `/simplify` started (the "operator did not stage prior changes" case). Distinguishing source is non-essential; the actionable signal is the dirty tree at exit time, and the operator's next-step (`/pr` or `git commit`) is the same in both cases. {#AC-STE-216.5}
- AC-STE-216.6: Test fixture: a `/simplify` run that collapses an existing file produces the advisory line and the capability row in the closing summary; a `/simplify` run on a tree where the prior commit already shipped the simpler form (no-op case, gate-skip path) produces neither. {#AC-STE-216.6}

## Technical Design

**Where the closing summary lives.** `skills/simplify/SKILL.md` documents the closing summary contract; the simplify skill currently emits a "Per skill rules, gate re-run is skipped" line on no-op runs and a per-file change summary on dirty-make runs. Neither path emits a tree-dirty advisory.

**Closing-summary addition.** At the end of the closing summary block, before exit:

```
git status --porcelain  # captured to a variable
if non-empty:
  emit "⚠ tree dirty: M <files> — run /pr or git commit to land the simplification"
  emit "Capability: simplify_tree_dirty"
  emit "  Rendered: tree dirty after /simplify — modified files: <list>; run /pr or git commit to land the simplification"
```

The `<files>` list is the per-file paths from `git status --porcelain` output, joined with `, `.

**Static-map registration.** Append to the static plain-language map in `skills/spec-write/SKILL.md` § 7:

| `simplify_tree_dirty` | `tree dirty after /simplify — modified files: <list>; run /pr or git commit to land the simplification` |

(The map is the single source of truth shared by `/spec-write`, `/gate-check` probe #26, and other skills that surface capability rows. `/simplify` reads the same map.)

**No exit-code change.** The advisory is informational; `/simplify` continues to exit 0 in the dirty case. Subsequent `/pr` / `/gate-check` / `git commit` are the operator's responsibility; the fix is purely a visibility increase.

## Testing

- Unit test: `simplify` fixture on a tree with `M src/foo.ts` produces the advisory line; assert the literal `tree dirty:` token in stdout.
- Unit test: `simplify` fixture on a clean tree produces no advisory line; assert no `tree dirty:` token in stdout.
- Integration test: full `/simplify` invocation on a project where simplification fires; capture stdout; assert both the advisory line and the `simplify_tree_dirty` capability row.
- Smoke regression: 2026-05-04 Linear smoke #10 F1's "M src/greet.ts" reproduction case now emits the advisory; subsequent smoke runs confirm the operator workflow no longer silently leaves the tree dirty.
