## Requirement

Both smoke drivers executed the M95 detached spawn correctly on the 2026-07-02 iter-2 run and then fire-and-exited at the WAIT step (finding F1, high): the Linear leg via harness `run_in_background` + end-turn, the Jira leg via a background poll-watcher + one-notification-then-die (a `-p` session terminates after responding). Contributing factor (verified in-session): the harness blocks Bash calls that LEAD with `sleep` and its error hint recommends `run_in_background`/Monitor — steering drivers into the forbidden pattern; the documented poll shape (`kill -0` first, `sleep` in branch) IS executable. Chain completion was 0/6 (linear) and 1/6 (jira) despite RC 0/0.

## Acceptance Criteria

- AC-STE-357.1: Both driver SKILL.mds' poll fences become bounded multi-iteration loops (`for i in $(seq 1 18); do kill -0 "$(cat <pidfile>)" 2>/dev/null || break; sleep 30; done` ≈ ≤540 s per call — one call per ~9 min instead of ~80 single-check calls per 40-min grandchild); no single-check-then-end-turn shape remains sanctioned; a meta-test pins the multi-iteration shape in both files.
- AC-STE-357.2: Red-flag prose in both SKILL.mds names the harness foreground-sleep block's error hint (recommending `run_in_background`/Monitor) as NOT-license — background-wait + end-turn IS the F3 fire-and-exit failure; a meta-test pins the literal warning in both files.
- AC-STE-357.3: Driver self-check: before emitting any final message, run the pidfile-liveness fence; any live pidfile ⇒ resume polling, never end the turn; a meta-test pins the fence + prose. Runtime validation ships `[~]` (next conformance run: both legs poll to completion).

## Technical Design

**Poll-fence upgrade (both `.claude/skills/smoke-test/SKILL.md` § Grandchild spawn lifecycle and `.claude/skills/conformance-loop/SKILL.md` § Phase A).** Replace the single-check poll fence (`if kill -0 …; then sleep 30 …`) with the bounded multi-iteration form:

```bash
# One bounded poll call — up to 18 checks ≈ 9 min, well under the 600s ceiling.
for i in $(seq 1 18); do
  kill -0 "$(cat <pidfile>)" 2>/dev/null || break
  sleep 30
done
```

The loop must not LEAD with `sleep` (the harness blocks leading-sleep calls); `kill -0` first keeps the call executable. The conformance-loop variant iterates over both legs' pidfiles inside the same bounded loop. A 40-minute grandchild needs ~5 calls instead of ~80, shrinking the LLM-compliance surface that killed both iter-2 legs.

**NOT-license red-flag prose.** Both SKILL.mds gain a red-flag paragraph adjacent to the poll fence: the harness's foreground-sleep block error message ("To wait for a condition, use Monitor… use run_in_background") is NOT license to background the wait — a `-p` session cannot resume on task notifications, so `run_in_background`/Monitor/watcher-task waits ARE the F3 fire-and-exit failure. The only sanctioned wait is repeating the bounded poll call in-session.

**Final-message self-check.** Both SKILL.mds gain a hard rule + fence: before emitting any final message (success or failure), run the pidfile-liveness check over the run's pidfile globs (`/tmp/dpt-smoke-<tracker>-*.pid` / `/tmp/dpt-conformance-loop-<date>-iter-<N>-*.pid`); any PID still answering `kill -0` ⇒ resume the poll loop. Composes with M95's live-pidfile session rule (this FR adds the *mechanical check*; M95 stated the prohibition).
