# STE-385: FR `## Summary` section: /spec-write authors a plain-language summary as the first FR body section {#STE-385}

## Summary

Every new FR now opens with a short plain-language summary — what the change is, why it matters,
and what it touches — written for human reviewers rather than for the gates. Because it is part of
the FR body, it also becomes the opening of the tracker issue description automatically, on
creation and on every later sync. The detailed sections keep their current precision; nothing
about how the gates verify specs changes.

## Requirement

GitHub issue #39 — the toolkit's first external-user feedback — reports that generated technical
specs are too detailed to review comfortably: method-level detail that an IDE shows better at
implementation time buries the reviewable substance. `/brainstorm` 2026-07-16 confirmed the core
problem is **human review burden** (not drift risk or authoring cost) and approved the additive
answer: keep every existing section at its current precision and add a plain-language `## Summary`
as the **first body section** of every new FR (immediately after the H1 title line, before
`## Requirement`), authored by `/spec-write` at draft time under an **altitude rule** — 3–6
non-empty lines; zero backtick characters; no AC-ID tokens; no path-like tokens (slash +
dot-extension in one whitespace-delimited token); no code fences (subsumed by zero backticks).

The tracker leg is **proven, not built**. Both adapters' `ticket_description_template` render
`{fr_body}` — the full FR body — on create and on every push (`adapters/linear.md:18-23`,
`adapters/jira.md:19-24`), and the deterministic composition site `TrackerProvider.sync`
(`tracker_provider.ts:236-257`) passes `spec.body` verbatim; a first-in-body section therefore
opens the ticket description on every sync with zero adapter code. The bidirectional AC sync is
structurally blind to it: `normalize()` slices only the `## Acceptance Criteria` section
(`adapters/linear/src/normalize.ts:23-38`) and `classify_diff.ts:43` compares parsed AC lists,
never whole descriptions. Verified 2026-07-16: **no deterministic gate rejects a sixth section** —
the five-section shape lives only in prose (`docs/layout-reference.md:31`, `/spec-write` § 0b
step 2's section inventory, and the probe #40 remedy string at
`needs_technical_review_consistency.ts:126`), so the contract amendment is textual, guarded by two
phrase-pinning convention tests.

## Technical Design

**Authoring is skill-prose only** — no new helper, no template file (there is no per-FR template;
the body shape is defined in `/spec-write` § 0b). The altitude rule is stated once in
`skills/spec-write/SKILL.md` and becomes the single rule set STE-386's scanner enforces — the
prose and the scanner must agree on the four checks.

## Notes

- **Out of scope — import path.** `importFromTracker`'s body composer intentionally keeps the
  five-section scaffold; tracker-imported FRs carry no `## Summary`, which is legal because
  presence is optional and STE-386's probe is vacuous on absence.
- **Out of scope — retroactive backfill.** Existing active/archived FRs are not edited.
- **Brainstorm seeding.** `/brainstorm`'s Step 4 hand-off already produces a 2–3 sentence approved-
  decision recap — the natural seed for this section; prose note only, no new contract.
- **No new capability keys.** The closing-summary static map is untouched (spec-write § 7 has zero
  line headroom; the summary is part of the draft the human approves, so no disposition token is
  needed).
- Origin: GitHub issue #39 → `/brainstorm` 2026-07-16 (Approach 2: summary + review contract,
  additive-only; tracker-lead threading confirmed as single-source ride-along, not a second copy).
