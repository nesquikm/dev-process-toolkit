# STE-339: Jira listMilestones() via milestone-label enumeration {#STE-339}

## Requirement

STE-329 added the Jira `milestone-M<N>` create-on-write label binding but explicitly did **not** implement milestone LISTING — `AdapterDriver.listMilestones` is optional and the Jira driver omits it (→ `[]`), so `TrackerProvider.listMilestones()` returns `[]` in Jira mode. Consequently the tracker leg of `nextFreeMilestoneNumber` (STE-284) is a no-op under Jira: a milestone allocated on one clone (mirrored only as a label) is invisible to another clone's allocation scan.

STE-284 AC-STE-284.7 referenced a `JiraProvider.listMilestones()` "via Jira's milestone API" that never shipped. This FR delivers it via label enumeration — the only MCP-writable milestone surface, per STE-329 — so the tracker leg fires in Jira mode and pairs with STE-338's git-branch leg to defend Jira-mode allocation from both directions. Approved via `/brainstorm` 2026-06-25 (the Jira-label half of the hybrid).

## Acceptance Criteria

- AC-STE-339.1: **Label-scan listing.** The Jira `AdapterDriver` implements `listMilestones()`: JQL-search the bound project for issues carrying labels, client-side filter labels matching `^milestone-(M\d+)$`, extract the `M<N>` token, dedupe, and return `{ name: "M<N>" }[]` — the M-token, **not** the `milestone-` label — so the existing `scanTracker` `^M(\d+)` extractor in `next_free_milestone_number.ts` consumes it unchanged. {#AC-STE-339.1}
- AC-STE-339.2: **Paginate, no silent cap.** The scan paginates `searchJiraIssuesUsingJql` to a documented cap; anything dropped beyond the cap is surfaced in a one-line log (no silent truncation). {#AC-STE-339.2}
- AC-STE-339.3: **Fail-soft.** Any JQL / network failure degrades `listMilestones()` to `[]`; it never throws into allocation — non-load-bearing exactly like Linear's leg and STE-338's git leg. {#AC-STE-339.3}
- AC-STE-339.4: **Label scope.** Only exact `^milestone-(M\d+)$` labels count; operator labels and `milestone-` substrings inside other labels are ignored — matching the STE-329 derivation (`milestone-<M-token>`, `[A-Za-z0-9-]`, spaces forbidden). {#AC-STE-339.4}
- AC-STE-339.5: **Tests green.** Unit coverage with a mocked JQL response: a mix of `milestone-M30` / `milestone-M86` / non-milestone labels / label-less issues yields `[{name:"M30"},{name:"M86"}]` (deduped, sorted); a failure response yields `[]`; pagination spans two pages. Existing Jira adapter tests + `tracker_provider` `listMilestones` delegation stay green. `bun test` + `/gate-check` GREEN. {#AC-STE-339.5}
- AC-STE-339.6: **Docs + supersede note.** `adapters/jira.md` documents the `listMilestones` label-scan; STE-284 AC-STE-284.7's deferred reference is marked fulfilled, and STE-329's "does NOT implement listing" caveat is repointed here. {#AC-STE-339.6}

## Technical Design

**Files touched:**
- `plugins/dev-process-toolkit/adapters/jira/src/` — implement `listMilestones()` on the Jira `AdapterDriver` (currently omitted → `[]`).
- `plugins/dev-process-toolkit/adapters/jira.md` — document the label-scan listing + supersede note.
- adapter unit tests — Jira `listMilestones` coverage. `TrackerProvider.listMilestones()` already delegates `driver.listMilestones?.() ?? []` (`tracker_provider.ts:226-229`), so no provider change is needed.

**Implementation:**
- JQL: search the bound project for labelled issues — e.g. `project = <projectKey> AND labels IS NOT EMPTY ORDER BY created DESC` — paginating via `mcp__atlassian__searchJiraIssuesUsingJql`.
- Client-side: for each issue's `labels`, match `/^milestone-(M\d+)$/`; collect the captured `M<N>` token.
- dedupe → `{ name: "M<N>" }[]`. Best-effort: any JQL/network throw ⇒ `[]`.
- Pagination cap documented; dropped pages logged (AC-STE-339.2).

**Why label-scan, not a milestone API:** STE-329 verified the atlassian MCP exposes no version/milestone list tool; labels are the only enumerable milestone surface. JQL has no label-wildcard operator, so enumeration scans labelled issues and filters client-side.

**Return-shape contract:** matches `MilestoneListingProvider` (`{ name: string }[]`); returning the bare M-token (not `milestone-M<N>`) lets `scanTracker`'s `^M(\d+)` extractor consume it without change.

## Testing

- Jira adapter unit (mocked `searchJiraIssuesUsingJql`): mixed labels → deduped + sorted M-tokens; failure → `[]`; two-page pagination accumulates across pages.
- `tracker_provider` delegation: `listMilestones()` returns `driver.listMilestones?.() ?? []` — now non-empty for Jira.
