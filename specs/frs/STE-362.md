# STE-362: Reliable milestone-label landing at attach time {#STE-362}

## Requirement

The toolkit mirrors an FR's milestone onto its tracker ticket via `attachProjectMilestone` — Linear binds the native project milestone; Jira read-merge-writes a `milestone-<M-token>` label. Today that attach is **best-effort and one-shot**: at `/spec-write` a failure surfaces as a non-blocking capability row and the flow continues (the FR file is deliberately not rolled back — the spec is the source of truth); at `/implement` § 0.e the same helper runs an idempotent re-attach, but it is documented "vacuous on archived FRs" and its coverage on the milestone-scope (`/implement M<N>`) fan-out is unspecified. The net effect surfaced 2026-07-03 on `../st/glacy-app-be`: **GB-11** (Story, M5) shipped with no `milestone-M5` label while every sibling FR ticket (GB-1…GB-10) got theirs.

This FR makes the binding **reliably land** on a freshly-created or in-progress FR-backed ticket: transient/network failures are retried with the canonical `1s+2s+4s` backoff; the re-attach + read-back verify is guaranteed on **every** `/implement` path (single-FR *and* milestone-scope); and a permanent failure is surfaced **loudly** (severity ≥ warning), never as a silent info line. The spec-is-source-of-truth invariant is preserved — a permanent failure still writes the FR file, it just makes the miss unmissable rather than swallowing it.

## Acceptance Criteria

- AC-STE-362.1: **Transient-only retry wrapper.** The `attachProjectMilestone` attach + read-back-verify round-trip retries on a transient/network failure (Gateway-Timeout / 504 / connection reset / equivalent) using the canonical `1s + 2s + 4s` schedule (3 attempts), reusing the `upsertTicketMetadata` backoff pattern. A non-transient `MilestoneAttachmentError` (binding mismatch — the helper wrote but the read-back disagrees) does **not** retry; it surfaces immediately. The success path adds no latency (retry only fires on a caught transient error). {#AC-STE-362.1}
- AC-STE-362.2: **All-`/implement`-paths attach guarantee.** Every FR-backed ticket built through **any** `/implement` path has the milestone attach + verify applied before its Phase 4 close: the single-FR path (`/implement <FR-id>`, § 0.e) and the milestone-scope path (`/implement M<N>` → one `/tdd` orchestrator per FR) both run the attach per FR. A meta-test asserts the milestone-scope fan-out runs the per-FR attach (closing the ambiguity that let a milestone-scope-built ticket skip § 0.e). {#AC-STE-362.2}
- AC-STE-362.3: **Loud permanent-failure surface.** When the attach fails permanently (retries exhausted, or a non-transient mismatch), `/spec-write` and `/implement` render the outcome as a **loud** capability row — the literal token `milestone_attach_failed` (severity ≥ warning) — in the closing summary, added to the `skills/spec-write/SKILL.md` § 7 static capability map (with `MUST emit` directives in `/spec-write` + `/implement`) so `/gate-check`'s `closing_summary_capability_keys` probe enforces it. The row is **not** a plain informational line, and the FR file is still written (spec is source of truth — not rolled back). {#AC-STE-362.3}
- AC-STE-362.4: **Vacuity.** Byte-identical to today on `mode: none` and on adapters with `project_milestone: false`: no retry wrapper fires, no `milestone_attach_failed` row is emitted, no extra tracker call is made. {#AC-STE-362.4}

## Technical Design

**Scope: harden the existing attach helper + its two call sites; no new gate.**

**Files touched:**
- `adapters/_shared/src/attach_project_milestone.ts` — wrap the attach + `getIssue` verify round-trip in a bounded retry helper. Classify the caught error: transient (network/Gateway-Timeout class) → retry on `1s/2s/4s`; binding-mismatch `MilestoneAttachmentError` → rethrow immediately (no retry). Factor the transient-classifier + backoff to share with the `upsertTicketMetadata` idempotency-retry shape rather than duplicating the schedule.
- `skills/implement/SKILL.md` — § 0.e: state that the attach+verify runs **per FR on the milestone-scope path**, not only single-FR resolve (hoist the call into the per-FR `/tdd` fan-out, or run it in the milestone loop before each FR's Phase 4 close). Add the `MUST emit milestone_attach_failed` directive on permanent failure.
- `skills/spec-write/SKILL.md` — § milestone-attach: on permanent failure, the capability row is loud (≥ warning), not a plain info line; add `milestone_attach_failed` to the § 7 static map (rendered prose) + `MUST emit` directive.

**Non-goal.** No change to the spec-is-source-of-truth contract: a permanent attach failure still writes the FR file and does not roll back. This FR changes *how loudly* the miss is reported and *how many times / on which paths* the attach is attempted — not whether the spec write survives.

**Transient vs. permanent (the load-bearing distinction).** A transient failure (timeout, reset) means "the write may not have landed — try again." A binding-mismatch `MilestoneAttachmentError` means "the write landed but disagrees" — retrying is pointless and would mask a real config bug (e.g., the adapter forwarding a milestone ID instead of a name). Only the former retries.

## Testing

- `attach_project_milestone.test.ts` — transient error retries on the `1s/2s/4s` schedule and succeeds on a later attempt; binding-mismatch does **not** retry (single attempt then throw); clean success makes exactly one attach + one verify (no added latency); `project_milestone: false` / `mode: none` short-circuit with no retry.
- Meta-tests under `tests/` — `skills/implement/SKILL.md` § 0.e carries the milestone-scope per-FR attach prose; `skills/spec-write/SKILL.md` § 7 map carries `milestone_attach_failed`; both skills carry the `MUST emit` directive (guards `closing_summary_capability_keys`).
- Gate: `bun test` (full gate from `plugins/dev-process-toolkit`) + `/dev-process-toolkit:gate-check` GREEN on this repo.

## Notes

- **Origin.** 2026-07-03 `/brainstorm` (topic: "two Jira tickets on `../st/glacy-app-be` have no milestone label — how, and do we have a gate?") → `/spec-write`. Approved **Option 2** (enforce on FR-backed tickets; bugs exempt) + archival-time durable check.
- **Root-cause evidence.** GB-11's FR (`specs/frs/archive/GB-11.md`) correctly declared `milestone: M5`, but the Jira ticket carried no `milestone-M5` label; GB-1…GB-10 all carried theirs. The attach helper already does a read-back verify that *can* block (it throws `MilestoneAttachmentError`) — but it is non-blocking at `/spec-write` and vacuous-on-archive at `/implement` § 0.e, so the miss was seen-then-lost. This FR closes the "seen-then-lost" path on the write side; STE-363 closes it on the archival side.
- **Reuses existing machinery.** The `1s+2s+4s` backoff and transient-error classifier already exist for the `upsertTicketMetadata` Gateway-Timeout idempotency path (see `adapters/jira.md` / `adapters/linear.md` § `upsert_ticket_metadata`); this FR shares that shape rather than inventing a second schedule.
- **Sibling FRs.** STE-363 (archival-time assertion) is the durable net; STE-364 (one-shot backfill) remediates already-shipped gaps like GB-11 today.
