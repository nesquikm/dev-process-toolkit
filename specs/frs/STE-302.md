# STE-302: Per-project tracker-config.yaml + status role vocabulary {#STE-302}

## Requirement

The toolkit's tracker integration anchors on a small set of canonical role names. Two are in production today (`in_progress` — used by STE-87 active-FR probe; `done` — used by STE-54 release verification), declared in each adapter's frontmatter `status_mapping` block (per STE-10, STE-16). The mapping is per-adapter: the Linear adapter ships one `status_mapping`, the Jira adapter ships one, and any workspace whose workflow uses different status names (or extra workflow states like `In QA`, `Ready for Code Review`) doesn't match. A live Jira project surfaced this — the hardcoded mapping doesn't survive a non-default workflow.

This FR introduces the foundation: a per-project `specs/tracker-config.yaml` file that carries (a) the project's actual status vocabulary verbatim from the tracker, and (b) a role-to-status map drawn from a fixed canonical role enum. Adapter-side consumers (gate probes, `/implement` claim/release) switch from reading per-adapter `status_mapping` frontmatter to calling a new loader API that reads the per-project file. The fixed role enum for M79: `initial`, `in_progress`, `in_review`, `done` — covers /spec-write create + /implement claim + /pr (future) + /implement release. `mode: none` remains vacuous (LocalProvider has no statuses to map). FR2–FR4 build on this foundation: FR2 writes the file via /setup, FR3 wraps Provider operations to tolerate non-key statuses, FR4 relaxes the strict gate probes.

## Acceptance Criteria

- AC-STE-302.1: New module `plugins/dev-process-toolkit/adapters/_shared/src/tracker_config.ts` exposes helpers `readTrackerConfig(specsDir)`, `writeTrackerConfig(specsDir, config)`, `validateTrackerConfig(config)`, `roleToStatus(config, role)`, `statusToRole(config, status)`. Throws `TrackerConfigShapeError` (NFR-10 canonical refusal shape) on malformed input. `readTrackerConfig` returns `null` when the file is absent — caller falls back to per-adapter default.
- AC-STE-302.2: Canonical config path: `specs/tracker-config.yaml` (git-tracked, lives beside `requirements.md`). Schema: top-level `statuses:` (array of strings, verbatim from tracker, MUST contain ≥ 1 entry) + `roles:` (map from canonical role name to status name; each value MUST appear in `statuses:`). File-level `tracker_key:` (`linear` | `jira`) for cross-reference with the active adapter — mismatch surfaces validation error.
- AC-STE-302.3: Canonical role enum locked at exactly four values: `initial`, `in_progress`, `in_review`, `done`. Extending the enum requires a plugin change (intentional friction — downstream probes assume the closed set). `roles:` MUST declare all four; missing role surfaces validation error naming the gap.
- AC-STE-302.4: `roleToStatus(config, role)` returns the mapped status string; unknown role (not in the four-value enum) throws `TrackerConfigShapeError`. `statusToRole(config, status)` returns the role name when the status maps to one, OR `null` when the status is in `statuses:` but not in any role mapping (known-non-key), OR the sentinel `"unknown"` when the status is not in `statuses:` at all (genuinely unrecognized). The three-way return shape powers the FR3 tolerance wrapper.
- AC-STE-302.5: Adapter-side consumer refactor: every read of `status_mapping` in adapter frontmatter switches to `readTrackerConfig(specsDir)` with adapter-default frontmatter fallback when the file is absent. Affected call sites enumerated in technical design. Per-adapter `status_mapping` frontmatter stays in place as the fallback for projects that haven't migrated (FR2 writes the file on /setup re-entry).
- AC-STE-302.6: `mode: none` vacuous — `LocalProvider` callers skip the loader entirely (no `tracker-config.yaml` is read, none is required, none is written). Probes already guarded by the `local-no-tracker` sentinel (STE-54.5, STE-87.1, STE-101.5) preserve their carve-out unchanged.
- AC-STE-302.7: New tests at `plugins/dev-process-toolkit/tests/tracker-config.test.ts` covering: parser happy path; missing file → `null`; malformed YAML → `TrackerConfigShapeError`; missing required role → validation error; role value not in `statuses:` → validation error; unknown role passed to `roleToStatus` → throw; three-way return shape of `statusToRole` (mapped role / `null` / `"unknown"` sentinel); `tracker_key` mismatch against active adapter → validation error. Round-trip test: write then read returns equivalent config.
- AC-STE-302.8: New `/gate-check` probe `tracker_config_shape` (severity: error) byte-checks `specs/tracker-config.yaml` shape when the file exists: schema invariants per AC.2 + AC.3 + AC.4. Vacuous when file absent (FR2's responsibility to write it). `mode: none` short-circuits (no read attempted).

## Technical Design

### Architecture

```
specs/tracker-config.yaml
   │
   ▼
adapters/_shared/src/tracker_config.ts   ← parser + validators + role/status mappers
   │
   ├─► adapters/linear (existing call sites that read status_mapping)
   │     readTrackerConfig(specsDir) ?? adapterFrontmatter.status_mapping
   │
   ├─► adapters/jira (same pattern)
   │
   ├─► (FR2) skills/setup writes the file
   ├─► (FR3) wrapper reads + branches on three-way statusToRole return
   └─► (FR4) probes call statusToRole instead of strict equality
```

### File shape (illustrative)

```yaml
# specs/tracker-config.yaml
tracker_key: linear
statuses:
  - Backlog
  - In Progress
  - In Review
  - In QA
  - Ready for QA
  - Done
  - Cancelled
roles:
  initial: Backlog
  in_progress: In Progress
  in_review: In Review
  done: Done
```

`In QA`, `Ready for QA`, `Cancelled` are in `statuses` but unmapped — they're known-non-key states. The tolerance wrapper (FR3) reads `statusToRole` returning `null` for these and prompts the operator. A status not in `statuses:` at all (e.g., a freshly-added workflow state the project hasn't synced yet) returns `"unknown"` and surfaces a discovery hint.

### Loader API contract

| Function | Return | Failure mode |
|----------|--------|--------------|
| `readTrackerConfig(specsDir)` | `TrackerConfig \| null` (null when file absent) | `TrackerConfigShapeError` on malformed YAML |
| `writeTrackerConfig(specsDir, config)` | `void` (atomic write) | `TrackerConfigShapeError` on invalid shape |
| `validateTrackerConfig(config)` | `void` (throws on invalid) | `TrackerConfigShapeError` naming the gap |
| `roleToStatus(config, role)` | `string` | throws on unknown role |
| `statusToRole(config, status)` | `Role \| null \| "unknown"` | never throws |

### Migration / call-site changes

Every read of `status_mapping.<role>` in the toolkit's adapter glue switches to `readTrackerConfig(specsDir)?.roles?.<role> ?? adapterDefault.<role>`. The fallback preserves backward compatibility for projects that haven't run M79's `/setup` yet. Once M79 has shipped to the dogfood repo, the dogfood's own `tracker-config.yaml` lands via /setup re-entry and the adapter-default fallback becomes dead code (target removal: M80 or later, once we confirm zero external consumers).

## Notes

- **Closed role enum justification.** Adding roles later requires plugin-side coordination (probe consumers, runbook updates). Per `project_no_users_yet`, the closed set is the conservative default. Future extensions (e.g., `blocked`, `cancelled`) get their own FR.
- **Why `statuses:` is verbatim from the tracker.** Powers FR3's "is this state known to the project at all?" check without re-querying MCP on every operation. Drift between repo file and tracker reality is acceptable — FR2's /setup re-entry is the resync path.
- **Adapter default fallback removal.** Tracked separately — not in M79 to avoid coupling new-feature ship with deletion of working code.
