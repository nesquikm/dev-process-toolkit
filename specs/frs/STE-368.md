# STE-368: /ship-milestone shipped_in stamp + legacy archived-plan backfill {#STE-368}

## Requirement

"Shipped" must become a deterministic file property. Today every version signal — `plugin.json`, `marketplace.json`, CHANGELOG's topmost `## [X.Y.Z]`, and requirements.md's `Latest shipped release` line — is bumped only by `/ship-milestone`, so when a milestone's implementation merges to main before the ceremony runs, all signals stay mutually **coherent-but-stale**: gate-check probe #9b passes, no probe cross-references archived plans against CHANGELOG, and the debt is invisible (M98 sat merged-but-unshipped for two days, 2026-07-04 → 2026-07-06). This FR writes the missing signal — `shipped_in: v<X.Y.Z>` in the plan's frontmatter — at the only moment it is authoritatively known: inside `/ship-milestone`'s atomic release commit. The stamp is the source of truth STE-369's `plan_ship_coherence` probe reads.

## Acceptance Criteria

- AC-STE-368.1: **Release-commit stamp.** A successful `/ship-milestone M<N>` run writes `shipped_in: v<X.Y.Z>` (the final version chosen at step 7, after any `--version` override) into the resolved plan file's frontmatter as part of the single atomic release commit — on both the live-plan path (`specs/plan/M<N>.md`) and the STE-210 archive-fallback path (`specs/plan/archive/M<N>.md`). {#AC-STE-368.1}
- AC-STE-368.2: **Approval + expected-set coverage.** Pre-flight #2's expected-modified set includes the resolved plan path, and the step-6 unified diff preview renders the frontmatter stamp alongside the release-file bumps — the stamp rides the existing single `Apply?` approval, no extra prompt. {#AC-STE-368.2}
- AC-STE-368.3: **Stamp helper.** `adapters/_shared/src/plan_ship_stamp.ts` exports the stamp writer: it preserves all other frontmatter keys and the body byte-for-byte, is an idempotent no-op when the existing `shipped_in` equals the new version, and refuses with the NFR-10 canonical shape when the plan already carries a *different* `shipped_in` (double-ship guard). All three behaviors unit-tested. {#AC-STE-368.3}
- AC-STE-368.4: **One-shot backfill.** A run-once local script (never shipped; deleted after the backfill commit per STE-92 precedent) stamps every existing `specs/plan/archive/M<N>.md`, resolving each plan's version in deterministic order: (a) a release commit whose `Refs:`/body names `M<N>` alongside a `Release: v<X.Y.Z>` footer; (b) the earliest release commit that is a descendant of the plan's archival commit (archival commit located via `git log -1 --diff-filter=A -- <archive path>`, STE-92 precedent); (c) unresolvable → left unstamped and listed in the script's report for manual assignment before the commit. The full mapping lands as one reviewable diff. {#AC-STE-368.4}
- AC-STE-368.5: **Same-commit compliance.** After the backfill commit, every plan under `specs/plan/archive/` carries `shipped_in` (or a deliberately-chosen `ship_state: parked`), so STE-369's probe is green from its first run — new ERROR probes ship with same-commit repo compliance (AC-STE-92.7 precedent). {#AC-STE-368.5}

## Technical Design

- **Helper** `adapters/_shared/src/plan_ship_stamp.ts`: `stampShippedIn(planPath, version)` — parse frontmatter, insert/verify `shipped_in`, rewrite preserving key order and body bytes. Same frontmatter-edit discipline as `archiveFRWithFlip` (STE-210): edit at the resolved path, no `git add -A`.
- **`skills/ship-milestone/SKILL.md` touch points:** step 1 (plan resolution) notes the resolved path is later stamped; pre-flight #2's expected-modified enumeration adds the resolved plan path; step 6's diff preview includes the stamp hunk; step 7's commit step calls the helper before `git commit`. The single-atomic-release-commit invariant is unchanged — the stamp is one more hunk inside it.
- **Backfill mechanics:** local script iterates `specs/plan/archive/M*.md`, skips plans already stamped or parked, resolves versions per AC-STE-368.4's order against `git log` + CHANGELOG, emits a mapping table for review, applies stamps. Deleted in the same commit that lands the stamps.
- **Stamp semantics (documented in the SKILL.md):** `shipped_in` is written only by `/ship-milestone` (or the one-shot backfill); absence on an *archived* plan means unshipped debt; absence on a *live* plan is normal.

**Files touched:**
- `adapters/_shared/src/plan_ship_stamp.ts` (new) + unit test
- `plugins/dev-process-toolkit/skills/ship-milestone/SKILL.md` — pre-flight #2, step 1, step 6, step 7
