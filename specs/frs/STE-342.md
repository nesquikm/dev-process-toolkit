# STE-342: Image-capture step in /spec-write (persister) + /brainstorm (capture + hand-off) {#STE-342}

## Requirement

Given the `specs/design/` convention from STE-341, the toolkit needs a way to actually get a user's design images *into* that tree and referenced from the right FR. This FR adds the capture step to the two skills where a user supplies design context: `/spec-write` (the canonical persister — it knows the FR id and writes files anyway) and `/brainstorm` (capture + classify, then hand off — it must not write spec content itself).

Without this FR, STE-341's convention is documentation with no on-ramp: a user pasting a mockup into a session has no skill-driven path to persist it.

## Acceptance Criteria

- AC-STE-342.1: **`/spec-write` persists + references.** When a user supplies one or more images during `/spec-write` (pasted attachment or a local file path in the prompt), the skill, for each image: (a) classifies it durable vs. per-feature — asked one at a time via `AskUserQuestion`, **default per-feature**; (b) saves it under `specs/design/system/<slug>.<ext>` (durable) or `specs/design/frs/<id>/<slug>.<ext>` (per-feature) where `<id>` is `Provider.filenameFor(spec)`; (c) writes a repo-root-relative entry into the FR's `## Design References` section, creating the section (after `## Acceptance Criteria`) if absent. {#AC-STE-342.1}
- AC-STE-342.2: **`/brainstorm` captures + hands off, never writes specs.** When images are supplied during `/brainstorm`, the skill classifies them and threads `{path, classification, caption}` records into the `/spec-write` hand-off (Step 4) — it does **not** write any FR markdown or per-feature image file itself, preserving the no-write-during-brainstorming contract. A standalone `/brainstorm` that does not hand off lists the captured images in its closing summary so the user can re-supply them at `/spec-write` time. {#AC-STE-342.2}
- AC-STE-342.3: **FR-id ordering respected.** Per-feature images are written under `specs/design/frs/<id>/` only *after* the FR id is known (i.e., after `Provider.sync(spec)` returns in tracker mode, or after `mintId()` in `mode: none`) — never against a `<tracker-id>` placeholder. Durable images (no FR-id dependency) may be persisted as soon as they are classified. {#AC-STE-342.3}
- AC-STE-342.4: **Paths are probe-resolvable.** Every reference written into `## Design References` is a repo-root-relative path that resolves on disk to the file just saved — byte-identical to what STE-343's probe resolves. The slug is derived deterministically from the image's caption/filename (lowercased, kebab-cased, extension preserved); a collision in the target folder gets a numeric suffix. {#AC-STE-342.4}

## Technical Design

**Files touched:**
- `plugins/dev-process-toolkit/skills/spec-write/SKILL.md` — new capture step in § 0b, sequenced **after** FR-id allocation (`Provider.sync` / `mintId`) so per-feature images get the real `<id>`. Classification asked via `AskUserQuestion` (Socratic Loop Contract applies — closed-form durable/per-feature with the always-on `Other`). Writes the `## Design References` section.
- `plugins/dev-process-toolkit/skills/brainstorm/SKILL.md` — capture + classify step; thread records into the Step 4 hand-off to `/spec-write`; standalone path lists them in the summary. Explicitly no file/markdown writes (first-turn + no-write contracts intact).
- `plugins/dev-process-toolkit/adapters/_shared/src/design_asset_slug.ts` — new pure helper: `designAssetSlug(caption | originalName) → kebab-case stem`; `designAssetTargetPath(classification, frId, slug, ext) → repo-root-relative path` with collision-suffix logic. Pure + deterministic so it is unit-testable without a session.

**Capture flow in `/spec-write` (§ 0b, post-allocation):**
1. Detect supplied images (prompt attachments / local paths named in the prompt).
2. For each: `AskUserQuestion` durable vs. per-feature (default per-feature), one at a time.
3. `designAssetTargetPath(...)` → repo-root-relative dest; copy/write the bytes there (creating `specs/design/...` dirs lazily).
4. Append `- \`<repo-root-relative-path>\` — <caption>` to the FR's `## Design References` section (create the section after `## Acceptance Criteria` if absent).
5. The saved files are staged in the same `/spec-write` commit as the FR markdown (§ 7a).

**`/brainstorm` boundary:** brainstorm holds `{path, classification, caption}` in memory and passes it to `/spec-write` at hand-off; `/spec-write` performs steps 3–5 above. This keeps the persister single (no two-skill write race) and honors brainstorm's no-write rule.

## Testing

- Unit tests for `design_asset_slug.ts`: slug derivation (spaces/punctuation/case → kebab; extension preserved), target-path composition for durable vs. per-feature, and collision-suffix behavior (`login.png`, `login-2.png`).
- Prose-contract assertions under `tests/`: `/spec-write` § 0b capture step is sequenced after FR-id allocation and uses `AskUserQuestion` for classification; `/brainstorm` capture step writes no spec content and threads records into the hand-off.
- Gate: `bun test` (full gate from `plugins/dev-process-toolkit`) + `/dev-process-toolkit:gate-check` GREEN on this repo.

## Notes

- **Depends on STE-341** for the `specs/design/` tree + `## Design References` section contract.
- **Why `/spec-write` is the sole persister.** Per-feature images need the FR id, which only exists after `Provider.sync`. Centralizing the write in `/spec-write` avoids a two-skill write race and keeps `/brainstorm` within its no-write contract. Operator-approved at the 2026-06-30 draft gate.
- **Classification default.** Per-feature is the default (most mockups are built-toward-one-feature); durable is the deliberate opt-in. A mis-classification is harmless under Approach 2 — the image is in the wrong folder but no reference breaks (it never moves).
- **Out of scope:** the visual conformance check that consumes these references (Deliverable B, next milestone).
