# STE-338: Cross-branch milestone-number scan in nextFreeMilestoneNumber {#STE-338}

## Requirement

`nextFreeMilestoneNumber` (`adapters/_shared/src/next_free_milestone_number.ts`) unions four sources to pick the next safe `M<N>`: active plan files, archived plan files, CHANGELOG refs, and â€” since STE-284 â€” tracker milestones via `provider.listMilestones()`. All three file legs read the **working tree of the current checkout only** (`listMNumbers` â†’ `readdirSync(specsDir/plan)`), so an `M<N>.md` plan file committed on an unmerged local branch, a worktree, or a remote-tracking branch is invisible. A parallel branch can therefore re-allocate the same `M<N>` â€” the collision STE-284 documented (M70 over the top) and the one the maintainer hit across local worktrees.

STE-284 explicitly deferred branch-aware scanning ("STE-225-style branch-aware scanning out of scope"); this FR delivers it as a fifth, best-effort source. Approved via `/brainstorm` 2026-06-25: Approach 2 (git-scan always on) unioned into the existing scan, with the network `git fetch` opt-in only.

**Scanner shape:**
```ts
export interface BranchScanOpts { includeRemotes?: boolean; fetch?: boolean; }
export async function scanBranchMilestones(repoRoot: string, opts?: BranchScanOpts): Promise<number[]>;
```
- `fetch === true` â‡’ best-effort `git fetch --all --quiet` (swallow errors).
- refs: `git for-each-ref --format='%(refname:short)' refs/heads` (+ `refs/remotes` unless `includeRemotes === false`).
- per ref: `git ls-tree -r --name-only <ref> -- specs/plan/`; extract `M(\d+)` from `(?:^|/)M\d+\.md$` (catches both `specs/plan/M<N>.md` and `specs/plan/archive/M<N>.md`).
- dedupe â†’ sorted ascending `number[]`. Every git invocation wrapped; any throw â‡’ `[]`.

**Helper integration:** `nextFreeMilestoneNumber(specsDir, changelogPath?, provider?, branchScanner?)`. The injected `branchScanner` mirrors the existing optional `provider` (duck-typed async method). Its numbers union into `all`; `sources.branches` records them. No git access inside the pure helper â€” the scanner is injected, keeping `next_free_milestone_number.test.ts` filesystem-pure.

**Fetch-policy precedence:** `--no-fetch` > `--fetch` > `milestone_scan_fetch` config > default `false`. `/spec-write`'s plan.md guard builds `{ listBranchMilestones: () => scanBranchMilestones(repoRoot, { fetch }) }` and injects it. `/gate-check` (read-only) never fetches.

**Dogfood:** this FR was scoped after running the branch scan live during `/brainstorm` 2026-06-25 â€” only `main` + `origin/main` exist, so no parallel branch carried a plan file and M89 was confirmed free across all five sources.

## Notes

- **Relationship to STE-284.** STE-284 (M70) added the optional tracker leg + `reconcileTrackerLocal` and explicitly deferred "STE-225-style branch-aware scanning" as out of scope. STE-338 picks up that deferred work as the fifth source.
- **Lock-scan precedent (STE-28).** `LocalProvider` already does `git fetch --all` (skippable) + `for-each-ref refs/remotes/` + `git ls-tree -r <branch> -- .dpt-locks/<id>` to detect cross-clone lock collisions (`local_provider.ts:113/156/160`). STE-338 reuses the same plumbing pattern for plan files.
- **Detection, not prevention.** The pre-sync race (two clones allocate M89 before either commits/pushes) is narrowed, not eliminated. Atomic tracker reservation was considered and rejected (2026-05-26 brainstorm, recorded in STE-329's notes) as a race + network dependency for zero gain.
- **Mode-agnostic.** The git leg runs in both tracker and `mode: none`; `milestone_scan_fetch` is read regardless of whether `## Task Tracking` is present.
