# STE-197: Plan-file hygiene: trimmed template + YAML frontmatter + bootstrap allocation rules {#STE-197}

## Requirement

`/setup`-generated `specs/plan/M<N>.md` files violate three documented invariants. Trim the plan template, prepend YAML frontmatter, and define how bootstrap-milestone FRs interact with `Provider.sync`.

**Findings bundled:** F3.1, F3.2, F3.4 from `/tmp/dpt-findings-2026-05-04.md`.

**Symptoms observed in `dpt_test_lib`:**

1. `specs/plan/M1.md` contained `## M1: Foundation`, `## M2: Core arithmetic operations`, AND `## Milestone Dependency Graph` ÔÇö three milestones' worth of content in one file (F3.2). Comes from `plan.md.template` lines 35ÔÇô96 which scaffold `## M<N>`, `## M<N+1>`, plus the dependency graph; `/setup` copies the template verbatim.
2. `specs/plan/M1.md` contained literal `<tracker-id>` placeholder rows in the FR table; `Provider.sync` was never invoked for the bootstrap FRs (barrel + `add`) because the work was already produced by `dart create -t package` before the allocator could run (F3.1).
3. `/setup`-generated plan files start with HTML comment + `# Implementation Plan` ÔÇö no `---` YAML frontmatter block. `/implement` Phase 4 ┬ž Milestone Archival prose says "apply `status: active Ôćĺ archived` flip + `archived_at` to the plan frontmatter" ÔÇö targets a frontmatter that doesn't exist (F3.4).

The symptoms compound: `/spec-archive M1` refuses (zero FR match), `/implement M1` Phase 1 step 0.c has nothing to claim, `/gate-check` probe #14 fires the STE-180 advisory and refuses Ôćĺ loop.

## Acceptance Criteria

- AC-STE-197.1: `templates/spec-templates/plan.md.template` rewritten to scaffold exactly **one** `## M<N>` section + a single `## Milestone Order` heading. The `## M<N+1>: [Core Feature]` section and `## Milestone Dependency Graph` section are removed from the per-milestone template. {#AC-STE-197.1}
- AC-STE-197.2: `plan.md.template` opens with a `---` YAML frontmatter block carrying `milestone: M<N>`, `status: active`, `archived_at: null`, `kickoff_branch: null`, `frozen_at: null` ÔÇö same shape as the canonical archived `M53.md` plan. {#AC-STE-197.2}
- AC-STE-197.3: `/setup` step 8 writes `specs/plan/M1.md` from the trimmed template (one milestone, with frontmatter) AND tags the bootstrap milestone with `kind: scaffolding` in frontmatter. Bootstrap FRs (barrel + primary feature) are NOT allocated via `Provider.sync` at `/setup` time ÔÇö they get a single FR row marked `<scaffolding>` so downstream skills route them through plan-only closure (per STE-200, this milestone). {#AC-STE-197.3}
- AC-STE-197.4: `/implement` Phase 4 ┬ž Milestone Archival prose explicitly handles "no frontmatter present" by prepending `---\nstatus: archived\narchived_at: <iso>\n---\n` before the H1 in the same atomic commit. Backwards-compat for legacy `/setup`-generated plans without frontmatter. {#AC-STE-197.4}
- AC-STE-197.5: New `/gate-check` probe `plan-file-single-milestone` flags any `specs/plan/M<N>.md` (or archived) containing more than one `^## M\d+:` heading. Emits ADVISORY on legacy multi-milestone files; `/spec-write` upgrades the same condition to refusal on edits. {#AC-STE-197.5}

## Technical Design

**Template rewrite.** `plan.md.template` becomes:

- HTML comment with placeholder convention (kept).
- `---` YAML frontmatter with the 5 fields above.
- `# Implementation Plan` H1.
- `## Milestone Order` (kept; moved earlier).
- One `## M<N>: [Title] {#M<N>}` section with FR list, Tasks, Tests, Acceptance Criteria, Gate.
- No `## M<N+1>` and no `## Milestone Dependency Graph` (cross-milestone concerns live in `requirements.md` or a separate `specs/plan/dependency-graph.md` ÔÇö out of scope here, but the per-milestone template no longer carries them).

**`/setup` bootstrap-FR routing.** Step 8 detects the bootstrap-milestone case (`kind: scaffolding` in plan frontmatter) and writes the plan FR table with a single row:

```
| <scaffolding> | Bootstrap (barrel + primary feature shipped pre-toolkit) | n/a |
```

No `Provider.sync` call. No `<tracker-id>` literal. Downstream skills key off the `<scaffolding>` marker (plan-only closure path per STE-200).

**Phase 4 archival prose.** `/implement` Phase 4 prose adds:

> If the plan file lacks a `---` frontmatter block, prepend `---\nstatus: archived\narchived_at: <ISO timestamp>\n---\n\n` before the H1, in the same atomic commit. Backwards-compat for legacy `/setup`-generated plans without frontmatter.

**Probe wiring.** `/gate-check` probe iterates over `specs/plan/*.md` and `specs/plan/archive/*.md`, counts `^## M\d+:` headings; >1 Ôćĺ ADVISORY row.

## Testing

- Unit test: `plan.md.template` parses as YAML frontmatter + markdown body; exactly one `^## M\d+:` heading.
- Integration test: `/setup` against a fresh repo writes `specs/plan/M1.md` with `kind: scaffolding` frontmatter, single FR row marked `<scaffolding>`, no literal `<tracker-id>` strings.
- Integration test: `/implement M1` against a frontmatter-less legacy plan file successfully prepends frontmatter and archives in one commit.
- Probe test: `/gate-check` flags a multi-milestone plan file at ADVISORY severity; passes on a single-milestone plan.

## Notes

- F3.1 root cause is "no Provider.sync at bootstrap"; this FR doesn't try to fix bootstrap allocation by INVOKING Provider.sync (work was done pre-toolkit, no real ticket to claim). Instead it formalizes the `<scaffolding>` marker so downstream skills route around it. STE-200 closes the loop with plan-only archival.
- F3.7 (drift Pass A grep missing `<tracker-id>`) was assessed in brainstorm as subsumed by this FR ÔÇö once `/setup` no longer leaves `<tracker-id>` literals in committed files, the grep gap is moot.
- The `## Milestone Dependency Graph` section deletion is intentional. Per-milestone plans shouldn't carry cross-milestone graphs ÔÇö that drifts as new milestones land.
- Capability: a tracker-mode `/spec-write` run that creates this FR surfaces `filename_policy_override` per the unconditional rule.
