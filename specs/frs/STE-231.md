## Technical Design

**Insertion site.** All four new fixture groups are inserted into `.claude/skills/smoke-test/SKILL.md` between the existing `#### Phase 2.X summary line` section and the `### Phase 3 — Capture` section. Order preserves chronological insertion (4 → 5 → 6 → 7) — same convention as groups 1 → 2 → 3.

**Reused infrastructure.** Three of the four groups read the *existing* Phase 2 child logs (`/tmp/dpt-smoke-<tracker>-spec-write.log` for groups 5a + 6; `/tmp/dpt-smoke-<tracker>-implement.log` for group 7) and assert against substring patterns — zero new spawns, zero wall-clock cost beyond the existing chain. Only groups 4 (4 marker-bearing spawns × 2 adapters = 8 total spawns) and 5b (1 marker-absent spawn × 2 adapters = 2 total spawns) require new child invocations.

**Sub-fixture cost model.** Group 4 dominates the new wall-clock — 8 spawns at ~30–60s each adds ~5–10 min per smoke run. Groups 5a / 6 / 7 are negligible (post-hoc stdout reads). The cost is acceptable because `/smoke-test` already runs at ~10 min wall-clock; adding ~5–10 min lifts the budget to ~15–20 min per leg, still inside the operator-controlled `/conformance-loop` window.

**Anti-probe-collision (group 5b).** The marker-absent spawn snippet must NOT be a fenced reference snippet in SKILL.md, because the M59 `auto_approve_marker_in_canonical_spawns` probe scans every fenced `claude -p` reference snippet and asserts the marker is present. The same technique used by STE-226 fixture 1b applies: describe the spawn in prose; the driver constructs the heredoc body at runtime with the marker line omitted. Prose mention of `<dpt:auto-approve>v1</dpt:auto-approve>` in surrounding documentation does NOT trigger the probe (the probe scans fenced blocks).

**Group 7 fenced-block detection.** The orchestrator emits exactly three `tdd-result` fenced blocks per FR (one per phase: test-writer, implementer, refactorer). The smoke fixture asserts via `grep -c '^```tdd-result$' /tmp/dpt-smoke-<tracker>-implement.log` ≥ 3. This is robust to the orchestrator's retry-on-failure path (a retry adds blocks, never removes them) and to the multi-AC case (one per-AC block sequence — but the live smoke uses the single-AC `greet` fixture so the count is exactly 3 on success).

**Fixture group ordering rationale.** Group 4 first because its sub-fixtures are net-new spawns (heaviest); groups 5 / 6 / 7 follow as they are stdout reads or single inverse spawns. Within group 5, sub-fixture 5a (marker-present, no new spawn) precedes 5b (marker-absent, new spawn) — same precedent as STE-226 group 1.

**Diagnostic-trace excerpts.** Each new fixture's failure diagnostic appends the last 20 lines of the relevant log to the diagnostic block, identical to the M56 pattern. For group 7, also append the last 5 commits since `<run-start>` from the test project's `git log` (the orchestrator commits to the test project's working tree, so `git log` shape often surfaces failure modes that stdout content does not — same rationale as STE-222's group 3).
