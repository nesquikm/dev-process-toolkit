# STE-270: Lock /spec-write Pattern 26 first-turn contract — regression fixture + drift probe + protocol clarification (STE-251 hardening) {#STE-270}

## Requirement

`/conformance-loop` iter-1 (2026-05-10, plugin v2.19.0) Phase 8 reproduced an STE-251 contract violation in `/spec-write` under the autonomous-mode reminder + verbose pre-baked `<command-args>` + NO auto-approve marker. The captured stream-json transcript (`tests/fixtures/socratic-first-turn/spec-write-2026-05-10.json`) shows a `Write` tool call at response-stream index 19 BEFORE any `AskUserQuestion`. `assertFirstTurnShape` correctly raised `SocraticFirstTurnViolationError` post-hoc; the assertion infrastructure works. The gap is **enforcement**: STE-251's prose-only mandate at `/spec-write` SKILL.md line 10 ("FIRST ACTION (under non-interactive stdin) … the first tool call this skill emits MUST be `AskUserQuestion` … OR `RequiresInputRefusedError` raise … `Write` / `Edit` / `NotebookEdit` are forbidden before that ask/refusal") is still LLM-inferred at runtime — the same prose-only failure shape that STE-220 → STE-226 → STE-237 each hit before structural / byte-checkable hardening landed.

`/setup`, `/brainstorm`, `/report-issue` passed F1 in the same iter (`askIndex ∈ {-1, 2, 4}`). `/spec-write` is the outlier; the diluting-prose hypothesis is that `/spec-write`'s § 0a "Resolver entry" + § 0b "FR creation path" carry extensive pre-baked-args handling that the LLM read as license to skip the Socratic loop entirely under the autonomous-mode reminder.

This is hardening of STE-251 (M67), parallel to how STE-262 hardens STE-226 — both are M69 first-turn / gate enforcement work, both apply the recurring lesson "byte-checkable / structural, not prose-only."

## Technical Design

### Existing-state assessment

Infrastructure for asserting the first-turn contract already exists, landed by STE-251 (M67):

- `plugins/dev-process-toolkit/adapters/_shared/src/socratic_first_turn.ts` — `assertFirstTurnShape(transcript)` + `SocraticFirstTurnViolationError` (NFR-10-shape).
- `plugins/dev-process-toolkit/adapters/_shared/src/socratic_first_turn_stream.ts` — `parseStreamJsonTranscript(ndjson)` projects raw stream-json NDJSON into a `TranscriptEntry[]`.
- `plugins/dev-process-toolkit/adapters/_shared/src/socratic_first_turn_assert.ts` — assertion shim.
- `plugins/dev-process-toolkit/adapters/_shared/src/socratic_first_turn_replay.test.ts` — replays the 2026-05-07 regression fixture; asserts violation at `Write@23`.
- `plugins/dev-process-toolkit/adapters/_shared/src/socratic_first_turn_post_hoc_drift.test.ts` — drift-detection coverage.

The regression caught in iter-1 (2026-05-10) confirms the assertion path works: post-hoc on the captured transcript, `assertFirstTurnShape` correctly threw with `toolName === "Write"`, `index === 19`. The remediation is therefore NOT building detection — it's locking the new regression baseline + adding a structural / byte-checkable guard against SKILL.md prose drift.

### Fix architecture (mirrors STE-262 Option B — script-driven / probe-based)

Three layers; deliberately the same shape as STE-262 for review symmetry:

**Layer 1 — Regression baseline fixture.** Pin `tests/fixtures/socratic-first-turn/spec-write-2026-05-10.json` (currently untracked from the conformance-loop run) to `tests/fixtures/socratic-first-turn/regression/spec-write-2026-05-10.json` via `git mv`. The `regression/` sub-path is the load-bearing convention from STE-251 AC-STE-251.4 — it shields the fixture from per-date overwrite by future `/smoke-test` Phase 8 runs. Add a sibling replay test at `socratic_first_turn_replay_2026_05_10.test.ts` (parallel to the existing `socratic_first_turn_replay.test.ts`); structure copies STE-251's test with the new tool-name/index expectations.

**Layer 2 — SKILL.md drift probe.** New `spec_write_first_turn_drift_scan.ts` mirrors STE-262's `spec_write_alternate_trigger_scan.ts`. Same `Severity`, `*Violation`, `*Report` types; same forbidden-phrase array shape; same `/gate-check` registration pattern. Globs only `/spec-write` SKILL.md (single-file; the regression is uniquely `/spec-write`'s, not a cross-skill issue). Severity: error.

**Layer 3 — Prose hardening + protocol clarification.** Strengthen `/spec-write` SKILL.md line 10 with the explicit byte-checkable enforcement sentence (AC-STE-270.3). Mirror in `docs/auto-mode-protocol.md § Socratic Loop Contract` (AC-STE-270.5).

### Why not a runtime hook (Option C considered, deferred)

Considered: a `PreToolUse` hook scoped to `/spec-write` that intercepts `Write` / `Edit` / `NotebookEdit` calls when stdin is non-tty AND no `AskUserQuestion` has fired in the current skill execution; refuses with `RequiresInputRefusedError`. This would be true runtime enforcement, not detection.

Rejected on the same grounds STE-262 rejected its Option C hook:

1. Session-wide hooks have broad blast radius; affect every spawn including downstream users post-install.
2. Conflicts hardest with the user-supplied "check everything twice" / triple-check posture — adding a hook to enforce a contract that prose + probe can already detect is over-engineering.
3. Tracking per-session "AskUserQuestion has fired in this skill execution" requires session state the hook surface does not provide cleanly.

If post-ship smoke surfaces this layer is insufficient, a follow-up FR (M70+) can layer the hook on top.
