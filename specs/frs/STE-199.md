# STE-199: Tracker availability: first-run deferred binding + mid-session probe + --code-only fallback {#STE-199}

## Requirement

Distinguish three tracker-availability states across `/setup` and `/implement`: (1) MCP genuinely failing, (2) MCP not yet registered (first-run case), (3) MCP registered but tools not loadable in this session. Surface each with the right behavior.

**Findings bundled:** F2.2, F3.5 from `/tmp/dpt-findings-2026-05-04.md`.

**Symptoms observed in `dpt_test_lib`:**

1. **F2.2.** At `/setup` time, the Linear MCP was not registered (the operator authenticates **after** `/setup` writes `.mcp.json`). Step 7b's procedure ("reject the call with NFR-10 canonical shape if either MCP probe fails") fires on the very first `/setup` because the project hasn't authenticated Linear yet. Operator had to default-apply `team: DPT` (a guess from project name) and `project: dpt_test_lib`, both wrong — workspace was `team: STE`, no `dpt_test_lib` project. After MCP authenticated, a `/setup --migrate`-shaped follow-up commit corrected both fields.
2. **F3.5.** During `/implement`, `dpt_test_lib/.mcp.json` declared the Linear MCP and CLAUDE.md declared `mode: linear`, but `ToolSearch` returned **zero** `mcp__linear__*` tools — only Gmail/Drive/Calendar OAuth handshakes. The Linear MCP appeared configured-but-unauthenticated for that specific session. `/implement` Phase 1 steps 0.c (Claim), 0.e (Verify), 0.f (Project-milestone attach), and Phase 4d (Release) hard-require Linear writes/reads with no fallback. Operator's only documented remedy was to edit CLAUDE.md to `mode: none` mid-session.

## Acceptance Criteria

- AC-STE-199.1: `/setup` step 7b documents a third branch beyond NFR-10-refusal: **"MCP not registered yet (first-run case)"**. When the target MCP is absent from `claude mcp list` AND the run is writing `.mcp.json` for the first time, record the binding values as `<deferred>` in CLAUDE.md and emit a `workspace_binding_deferred` capability row in the closing summary. The first downstream skill to need the binding (typically `/spec-write`) runs the live probe and prompts then. {#AC-STE-199.1}
- AC-STE-199.2: Audit-log entry: `step:7b (workspace_binding) value:"<deferred>" reason:"<MCP> unregistered at /setup time; deferred to first downstream skill"`. {#AC-STE-199.2}
- AC-STE-199.3: `/setup --resume-tracker-binding` flag added — runs only step 7b's workspace-binding probe + write, designed to be invoked after the operator authenticates the MCP. Materially equivalent to `--migrate` but scoped to a single field. {#AC-STE-199.3}
- AC-STE-199.4: `/implement` Phase 1 adds a Pre-flight probe (before step 0.b) that confirms the configured MCP's tools are actually loadable in the current session: searches the available tool list for `mcp__<tracker>__*` patterns. Probe failure surfaces NFR-10 canonical refusal: "Tracker `<tracker>` configured in CLAUDE.md but `mcp__<tracker>__*` tools are not available in this session. Remedy: run `claude /mcp` to (re)authenticate, or re-invoke with `--code-only` for a degraded tracker-skipped run." {#AC-STE-199.4}
- AC-STE-199.5: `--code-only` flag documented for `/implement`: skips Phase 1 0.c/0.e/0.f tracker writes, skips Phase 4d release transition, surfaces `tracker_skipped` capability rows for each skipped step. Behavior identical to `mode: none` for the duration of the run; CLAUDE.md is **not** edited. {#AC-STE-199.5}
- AC-STE-199.6: Two new capability keys added to the static plain-language map (`spec-write` § 7 and `implement` § Closing-summary):
  - `workspace_binding_deferred` ⇒ "tracker workspace binding deferred to first downstream skill — `<MCP>` not yet registered at `/setup` time; run `/setup --resume-tracker-binding` after authenticating, or let the next skill that needs it prompt"
  - `tracker_skipped` ⇒ "tracker step skipped (`--code-only` flag or pre-flight probe failed) — re-run without flag once MCP loadable to sync"
  {#AC-STE-199.6}

## Technical Design

**`/setup` step 7b decision table (revised):**

| Probe state | Branch | CLAUDE.md write | Capability row |
|-------------|--------|-----------------|----------------|
| MCP not registered (`.mcp.json` being written this run) | First-run deferral | `team: <deferred>`, `project: <deferred>` | `workspace_binding_deferred` |
| MCP registered + probe succeeds | Normal | resolved values | none |
| MCP registered + probe fails (auth expired, network down) | NFR-10 refusal | not written | none (refusal aborts run) |

**`/implement` Phase 1 pre-flight (new step 0.a, before 0.b):**

```
if claude_md.tracker_mode != "none":
  available_tools = ToolSearch.list()
  if not any(name.startswith(f"mcp__{tracker}__") for name in available_tools):
    if "--code-only" in flags:
      log("tracker_skipped: pre-flight probe failed; --code-only flag honored")
      proceed with code-only path
    else:
      surface NFR-10 refusal naming missing tracker + remedy
      exit non-zero
```

**`--code-only` flag wiring.** Phase 1 0.c/0.e/0.f and Phase 4d each carry an early-return when `flags.code_only`:

```
if flags.code_only:
  surface capability row "tracker_skipped: <step-name>"
  return  # skip the MCP call
```

The `tracker_skipped` rows accumulate in the closing summary so the operator sees exactly which writes were dropped.

**Symmetry between F2.2 and F3.5.** Both are "tracker not actually usable right now." F2.2's first-run case becomes a `<deferred>` write at `/setup` time; F3.5's mid-session case becomes a pre-flight refusal at `/implement` time. The shared concept: detect non-availability before any state-mutating call, surface the gap clearly.

## Testing

- Integration test (F2.2): `/setup` against a fresh project with no MCP registered → CLAUDE.md gets `team: <deferred>` `project: <deferred>` AND closing summary surfaces `workspace_binding_deferred`. Subsequent `/setup --resume-tracker-binding` after MCP authentication writes the resolved values.
- Integration test (F3.5): `/implement STE-X` with CLAUDE.md `mode: linear` but no `mcp__linear__*` tools loadable → run aborts with NFR-10 naming the missing tracker. Same call with `--code-only` proceeds, skipping 0.c/0.e/0.f, surfacing `tracker_skipped` rows.
- Unit test: pre-flight probe correctly enumerates `mcp__<tracker>__*` tools from the available-tools list; correctly distinguishes "no tools" (refuse) from "tools present" (proceed).

## Notes

- F2.2 and F3.5 are bundled because both are tracker-availability-detection at skill entry — different skills, symmetric design. Splitting them would duplicate the capability-key map and the "what does available mean" prose.
- `--code-only` is a per-run flag, never persisted. CLAUDE.md `mode:` is the source of truth for the project's intent; `--code-only` is the operator's "not right now" override.
- `<deferred>` literal in CLAUDE.md is parsed by Schema L probe as "binding incomplete, prompt operator on first need." Existing Schema L parsers must tolerate it without crashing — backwards-compat covered in /gate-check probe revision.
- Capability: a tracker-mode `/spec-write` run that creates this FR surfaces `filename_policy_override` per the unconditional rule.
