# STE-291: Hook refusal exit code — switch 3 Refusing hooks from exit 1 (advisory) to exit 2 (blocking) {#STE-291}

## Requirement

STE-290 (M75, v2.22.3 "Wired", shipped 2026-05-14) successfully wired the byte-checkable toolkit-contract enforcement layer to the actual Claude Code stdin `transcript_path` contract — the hooks now fire and emit NFR-10 `Refusing:` stderr on miss. Empirical verification 2026-05-14 confirmed the hook fires correctly.

**However:** the 3 Refusing hooks (`pre-commit-gate-check`, `pre-pr-spec-review`, `pre-commit-tdd-orchestrator`) call `process.exit(found ? 0 : 1)` on miss. Claude Code's hook contract treats exit-code semantics as:

- `exit 0` → OK, allow the tool call
- `exit 2` → **blocking** (block the tool call, send stderr to the model as feedback context)
- any other non-zero (incl. **`exit 1`**) → **advisory** (show stderr to the operator only, allow the tool call)

Empirical observation (2026-05-14, post-v2.22.3 install): a `gh pr create --help` invocation with no `Skill(dev-process-toolkit:spec-review)` in the session transcript fired the hook, emitted the `Refusing:` block to the operator, but Claude Code reported "Failed with non-blocking status code" and ran the command anyway. The model never received the refusal as feedback (so it cannot react and run the missing skill); the operator sees the message but the action proceeds.

This FR switches the 3 Refusing hooks from `: 1` → `: 2` so the layer actually blocks. The advisory `pre-spec-write-brainstorm-reminder` UserPromptSubmit hook stays at `exit 0` (advisory by STE-285 design; emits `Reminder:` not `Refusing:`).

### Exit-code change pattern

```ts
// Before (STE-290): process.exit(found ? 0 : 1);  — Claude Code treats exit 1 as advisory
// After  (STE-291): process.exit(found ? 0 : 2);  — Claude Code treats exit 2 as blocking + stderr-to-model
```

### Why exit 2 specifically (Claude Code hook contract)

Per the Claude Code 2.1.x harness contract (verified empirically 2026-05-14 against the dev-process-toolkit pre-pr-spec-review hook installation):

| Exit | Harness behavior | NFR-10 stderr visible to |
|------|------------------|--------------------------|
| `0` | Tool call proceeds | (no stderr emitted) |
| `2` | **Tool call blocked**, stderr fed back to the model as context | Model + operator |
| other non-zero (`1` etc.) | Tool call proceeds; stderr shown to operator only | Operator only |

STE-285's original design assumed exit 1 would block; that assumption was falsified at the same time as STE-290 was being verified. This FR closes the gap.
