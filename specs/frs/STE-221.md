# STE-221: test(skills/gate-check) — real -p regression coverage for probe #26 ## Notes capability-gap reading {#STE-221}

## Requirement

STE-214 (M55) added a `## Notes`-section scanner to `/gate-check` probe #26 (`tracker-project-milestone-attached`): when an FR's `## Notes` body declares `milestone_attach_skipped_adapter_limit` (or the deprecated alias `milestone_attach_unavailable`, or `milestone_create_required`), probe #26 downgrades from `GATE FAILED` to `ADVISORY` instead of false-failing the chain. The 2026-05-04 v2.8.0 smoke runs (Linear Run #2 + Jira Run C) confirm the fix works empirically: Linear Run #2 reports `37/37 conformance probes PASS` with no probe #26 advisory or gate-fail, and the chain otherwise clean.

But the fix lives entirely in SKILL.md prose — no automated `-p` regression catches a future divergence between the documented contract and the LLM's actual probe behavior. STE-220 (this milestone) is the proof-of-concept that "documented but unenforced" is a real failure mode: STE-213 documented the same shape of fix and ships broken in v2.8.0 because no real `-p` invocation ever exercised the carve-out. STE-214 happens to work today, but the same surface is open.

**Findings bundled:** none new. This FR is purely defensive `-p` regression coverage for the existing STE-214 fix.

**Symptoms this FR prevents (vs. detects):**

1. SKILL.md probe #26 prose is rewritten in a way that breaks the `## Notes` scanner — silent regression, no smoke catches it.
2. The static plain-language map's milestone-attach keys are renamed without updating probe #26's recognized-set list — silent regression.
3. A future refactor consolidates probe #26 prose into a different section, the LLM stops reading the scanner contract at the right moment, false GATE FAILED returns — silent regression.

## Acceptance Criteria

- AC-STE-221.1: Positive fixture: a sandboxed Linear-mode test project with a fresh Linear project carrying zero milestones, plus an FR file under `specs/frs/<TKR>-XXX.md` whose `## Notes` body contains the canonical capability key `milestone_attach_skipped_adapter_limit` (substring match). A `claude -p /gate-check` invocation against this fixture must produce a probe #26 ADVISORY row in captured stdout (literal token: `ADVISORY: probe #26` or equivalent stable shape per `skills/gate-check/SKILL.md`), NOT `GATE FAILED`. {#AC-STE-221.1}
- AC-STE-221.2: Control fixture: identical project + FR but with the capability key removed from `## Notes`. A `claude -p /gate-check` invocation must produce a probe #26 GATE FAILED in captured stdout. The control case proves the scanner is the difference, not some unrelated probe behavior. {#AC-STE-221.2}
- AC-STE-221.3: Deprecated-alias fixture: identical to AC.1 but with `milestone_attach_unavailable` (the STE-198 deprecated alias) instead of the canonical key. Probe #26 must still produce ADVISORY (the deprecation window is honored per STE-214 AC.5). When the deprecation window closes per STE-198's rollover note, this fixture flips to expecting GATE FAILED — track via STE-198's existing rollover note, not a new FR. {#AC-STE-221.3}
- AC-STE-221.4: All three fixtures run inside the existing `/smoke-test` Phase 2 chain (or as a sibling phase 2.X) — no new test runner, no new harness. The test reuses `claude -p` invocation infrastructure already in place. Failures surface with a clear `STE-214 runtime regression: <fixture-name>` diagnostic in captured stdout. Generic "test failed" messages are insufficient — the diagnostic must name STE-214 by tracker ID and the specific fixture (positive / control / deprecated-alias) so triage is instant. {#AC-STE-221.4}

## Technical Design

**Smoke-driver extension (shared with STE-220 + STE-222).** All three M56 FRs add `-p` regression fixtures to `/smoke-test`'s Phase 2 chain. The fixtures sit alongside the existing canonical chain assertions (Phase 2 step 4 `/gate-check` invocation), each adding one or more sub-fixtures that re-invoke `claude -p /gate-check` (or `/spec-write` for STE-220, `/implement` for STE-222) against a fixture project state.

**Fixture project layout.** STE-221's fixtures share the existing `dpt-test-project-linear` smoke target. The driver creates the FR file with the appropriate `## Notes` content for each fixture (positive / control / deprecated-alias), invokes `claude -p /gate-check`, captures stdout, asserts the expected probe #26 outcome, and rolls back the FR file edit before the next fixture.

**Recognized-set sourcing.** STE-221 does NOT introduce a separate list of recognized capability keys. The recognized set is whatever STE-214's scanner already reads from the `## Notes` body. STE-221's fixtures exercise the contract; they do not duplicate it. Single source of truth: `/gate-check` probe #26 implementation.

**Diagnostic shape.** When a fixture fails, the smoke driver emits:

```
STE-214 runtime regression: <fixture-name>
  expected: <ADVISORY | GATE FAILED>
  actual:   <ADVISORY | GATE FAILED | unknown>
  stdout excerpt: <last 20 lines of captured output>
```

The `STE-214` reference is by tracker ID (the original FR being verified), not STE-221's own ID. STE-221 is the test infrastructure; STE-214 is the FR under regression test. Diagnostic clarity follows what's broken, not what discovered the breakage.

**Run cadence.** All three M56 fixtures fire on every `/smoke-test` invocation (Linear leg only for STE-221, since the Linear MCP is the primary smoke target where probe #26 fires; Jira leg N/A for probe #26 because adapter Schema M flags `project_milestone: false` for Jira and the probe is vacuous there).

## Testing

- Unit test: assert each fixture (positive, control, deprecated-alias) sets the expected `## Notes` content before invocation. Fixture-state assertion before assertion.
- Unit test: assert the smoke driver's diagnostic shape carries the literal `STE-214` tracker ID, not STE-221's.
- Integration test: real Linear smoke run with all three fixtures; assert the `/smoke-test` summary line includes `STE-214 runtime check: PASS` (or the equivalent shape established for STE-220/STE-222 at implementation time).
- Negative test: temporarily remove the `## Notes` scanner from probe #26; assert the positive fixture flips to FAILED in captured stdout; restore the scanner and re-run to confirm the diagnostic shape is what STE-221's regression-detection contract specifies.
