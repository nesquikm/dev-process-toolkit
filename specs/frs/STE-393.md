# STE-393: Declared + gated migration coverage: plan-level declaration, /ship-milestone refusal, coherence probe {#STE-393}

## Summary

Every future release now states, in its milestone plan, whether it changes the artifacts the toolkit manages inside user projects — and if so, which migration covers the change. The ship command refuses to release a milestone that has not made the declaration, or that claims a migration the registry does not carry for that release. A gate probe keeps the shipped history coherent, while releases that predate the rule are grandfathered. This closes the loop that let convention changes ship with no upgrade path for existing projects.

## Requirement

The historical inventory behind this milestone found ~25 consumer-artifact convention changes since v1.8.0, roughly half of which shipped forward-only with no migration — not because anyone decided that, but because nothing in the release process asked. M104 is the canonical example: a full relayout of the toolkit's consumer-side state directories shipped with an explicit "no migration rails" note, and the debt landed on every existing tree.

Close the loop with the **declared + gated** enforcement approved at brainstorm (2026-07-17): every milestone plan carries an explicit `migration:` declaration — the literal `none`, or the id of the STE-391 registry entry that covers the change — and `/ship-milestone` refuses to release an undeclared or incoherently-declared milestone. A gate probe keeps the shipped history coherent after the fact. The declaration is deliberately an id, not a boolean: it makes the plan↔registry join explicit and self-documenting.

**Key semantics:** `migration: <entry-id>` means this milestone's changes REQUIRE a migration on existing consumer trees; additive changes that tolerate absence (grandfathered keys, optional sections, vacuous-when-absent probes) declare `none`. The M108 plan itself declares `none` — the framework changes plugin surfaces, not consumer conventions — and is the first dogfood of the rule.

## Acceptance Criteria

- AC-STE-393.1: **Declaration key.** Schema T (`specs/technical-spec.md`) and `templates/spec-templates/plan.md.template` gain the `migration:` key (values: literal `none` or a registry entry id; the template ships `migration: none` with an explanatory comment). `/spec-write`'s plan-creation flow sets the value deliberately at drafting time. The M108 plan carries `migration: none`. {#AC-STE-393.1}
- AC-STE-393.2: **/ship-milestone pre-flight.** A new pre-flight step (helper `assertMigrationDeclared(planPath, registry, releaseVersion)` in the migrations module) runs before the Release Files bump: key absent ⇒ NFR-10 refusal naming the plan and the declare-remedy; declared id not present in the registry ⇒ refusal; declared id whose `introduced_in` ≠ the version being shipped ⇒ refusal; `migration: none` ⇒ proceed. No additional approval prompt — the step rides the existing ceremony (STE-368 shape). {#AC-STE-393.2}
- AC-STE-393.3: **Gate probe `migration_coverage`.** For every archived plan whose `shipped_in` is ≥ the epoch (the release this FR ships in): the `migration:` key must be present; a declared entry id must exist in the registry with `introduced_in` == the plan's `shipped_in`. Violations are ERROR severity with NFR-10 messages naming the plan, the offending value, and the remedy. Pre-epoch archived plans are exempt and rendered as a NOTES count (STE-369 opt-out shape). Active plans missing the key surface an advisory warning only — consumer-project-safe; the ship pre-flight is the hard gate. {#AC-STE-393.3}
- AC-STE-393.4: **Null tolerance.** `migration: null` and `migration:` (empty) are treated as absent — never as a valid declaration — per the M103 template-sentinel lesson (fix 8ed7c80). Covered in both the pre-flight helper and the probe. {#AC-STE-393.4}
- AC-STE-393.5: **Probe purity.** The probe is pure file reads plus a registry module load — no git, no network, no LLM judgment (STE-369 Technical Design bar). {#AC-STE-393.5}
- AC-STE-393.6: **Docs + pins.** `docs/ship-milestone-reference.md` and `docs/upgrade-reference.md` document the declaration contract and the pre-flight; probe-count pins bump (67 → 68) at every pinned surface. {#AC-STE-393.6}
- AC-STE-393.7: **Full gate green.** `bun test` from `plugins/dev-process-toolkit` passes; probe fixtures prove non-vacuity in both directions (a coherent fixture passes, each violation class fails); no probe regresses. {#AC-STE-393.7}

## Technical Design

**Helper:** `adapters/_shared/src/migrations/coverage.ts` — `assertMigrationDeclared(planPath, registry, releaseVersion)` (pre-flight) + the probe's shared parser. Declaration parsing tolerates the frontmatter shapes `plan_ship_stamp.ts` handles (byte-preserving reads, unclosed-frontmatter refusals per AC-STE-368.3 shape).

**Epoch:** a `MIGRATION_COVERAGE_EPOCH` constant in the module, set to this FR's release version at implementation time. The release target contends with M101 (v2.49.0 provisional) — correcting the constant if the target shifts is an explicit `/ship-milestone`-time checklist item recorded in this FR and the plan.

**Probe wiring:** new probe #68 `migration_coverage`, sibling to `plan_ship_coherence` (#63) in scope discipline: archive-scoped hard checks, active-scoped advisories only.

**Precedents:** STE-368 (ship-time frontmatter write inside the atomic ceremony, idempotent writer shape), STE-369 (ERROR probe over plan frontmatter + explicit opt-out rendered as NOTES), STE-381 (landing a new frontmatter key across template + schema + creation flow together), M103 fix 8ed7c80 (null sentinel = unstamped).

## Notes

- **Depends on STE-391** — the registry is the join target. Build order: A → C (B and C are independent of each other).
- **Epoch grandfathering** deliberately avoids retro-classifying ~95 archived plans (M99's backfill was mechanical CHANGELOG lookup; this one would need per-milestone judgment). Pre-epoch history is exempt, visible as a NOTES count.
- **M101 rebase note:** M101's active plan (on its unmerged branch) will add `migration: none` when it rebases past this release — a one-line addition; the active-plan advisory (not error) keeps its gate green until then.
- **Retro-seeded entries are unclaimed:** STE-391's four script entries + STE-392's assisted entry carry `introduced_in` values pointing at past releases (v1.16.0 … v2.46.0); no plan declares them. Legal by design — the coverage rule binds plans to entries, not entries to plans.
