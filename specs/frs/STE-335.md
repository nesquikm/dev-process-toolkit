## Requirement

The milestone-binding **write** path (`planFileHeadingToMilestoneName`,
`adapters/_shared/src/attach_project_milestone.ts:176`) and the **verify** path
(`/gate-check` probe #26 `tracker-project-milestone-attached`,
`adapters/_shared/src/tracker_project_milestone_attached.ts:79`) each carry a
private, identical regex `/^# (M\d+ — .+?)(?:\s*\{#M\d+\})?\s*$/m` that matches
only an **H1 + em-dash** heading (`# M<N> — <title>`).

The current plan template (`templates/spec-templates/plan.md.template:46`) and
`/spec-write` (`skills/spec-write/SKILL.md:155`) both emit **H2 + colon**
headings (`## M<N>: <title> {#M<N>}`). On every current-format plan:

- the **write path throws** (`planFileHeadingToMilestoneName` → "no recognizable
  H1 heading"), so milestone attachment fails during `/spec-write` / `/implement`;
- the **verify path silently skips** the FR (`readPlanHeading` returns null →
  `continue`), so probe #26 passes vacuously.

Net: M86/STE-329's milestone binding — both the Linear `object` binding and the
Jira `label` binding — is broken end-to-end for every downstream consumer whose
plans come from the current template/`/spec-write`. The toolkit does not catch
this because it has zero active plan files (probe #26 is vacuous on it by
construction); the legacy H1+em-dash format survives only in archived plans.

## Acceptance Criteria

- AC-STE-335.1: A single shared plan-heading parser (new module
  `adapters/_shared/src/plan_heading.ts`) matches a milestone heading at **either**
  `#` or `##` depth, with **either** an em-dash (`—`) or colon (`:`) separator,
  and an optional trailing `{#M<N>}` anchor; it returns the canonical milestone
  name **normalized to `M<N> — <title>`** (em-dash), or `null` when no milestone
  heading is present.
- AC-STE-335.2: `planFileHeadingToMilestoneName` delegates to the shared parser.
  On a `## M<N>: <title> {#M<N>}` plan it returns `M<N> — <title>` and no longer
  throws; it still raises on a plan file with no milestone heading at all.
- AC-STE-335.3: Probe #26's `readPlanHeading` delegates to the shared parser.
  On a current-format `## M<N>: <title>` plan it returns the canonical name
  (not null), so the probe reaches the binding check instead of `continue`-skipping.
- AC-STE-335.4: Probe #26 correctly evaluates the **Jira `label` binding** on a
  current-format plan: ticket missing `milestone-<M-token>` → violation; present → pass.
- AC-STE-335.5: Probe #26 correctly evaluates the **Linear `object` binding** on a
  current-format plan: `projectMilestone.name` byte-equals `M<N> — <title>` → pass;
  null/mismatch → violation.
- AC-STE-335.6: The legacy `# M<N> — <title>` (H1+em-dash) heading still parses to
  the identical canonical name — no regression for archived/legacy plans.
- AC-STE-335.7: A repo-wide audit leaves no other live copy of the
  `^# (M\d+ — …)` heading regex; the two known copies (`attach_project_milestone.ts`,
  `tracker_project_milestone_attached.ts`) are removed in favor of the shared parser.

## Technical Design

New module `adapters/_shared/src/plan_heading.ts`:

```ts
// Accepts H1 or H2, em-dash or colon separator, optional {#M<N>} anchor.
// Normalizes the canonical name to `M<N> — <title>` (em-dash) regardless of
// the source separator, so write and verify agree by construction and a
// heading-punctuation change never drifts the tracker-stored milestone name.
const PLAN_HEADING_RE = /^#{1,2}\s+(M\d+)\s*(?:—|:)\s*(.+?)\s*(?:\{#M\d+\})?\s*$/m;

export function parsePlanHeading(md: string): string | null {
  const m = md.match(PLAN_HEADING_RE);
  return m ? `${m[1]} — ${m[2]!.trim()}` : null;
}
```

- `attach_project_milestone.ts`: delete the private `PLAN_HEADING_REGEX`;
  `planFileHeadingToMilestoneName` reads the file, calls `parsePlanHeading`, and
  raises its existing "no recognizable heading" error **only** when the parser
  returns `null` (broaden the error text to name both accepted forms).
- `tracker_project_milestone_attached.ts`: delete the private `HEADING_RE`;
  `readPlanHeading` calls `parsePlanHeading` and returns its result (null →
  existing skip path, owned by probe #27).
- The Jira `label` branch is unaffected downstream of the parse: `milestoneLabel`
  already extracts only the leading `M\d+` token, identical under both separators.
- Probe #32 (`plan-file-single-milestone`) already keys on `^## M\d+:` — consistent
  with the live format; no change. No other consumer parses the milestone heading
  (verified by AC-STE-335.7's audit).
