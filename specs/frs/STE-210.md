# STE-210: Phase 4 archival hardening: ship-milestone archive fallback + git mv ordering {#STE-210}

## Requirement

Two adjacent bugs in the milestone-archival → release-tag chain. Both surface during the canonical `/implement M<N>` → `/ship-milestone M<N>` flow on a milestone that just shipped.

**Findings bundled:** F10, F11 from `/tmp/dpt-findings-2026-05-04-flutter-app.md` (Flutter calculator smoke, 2026-05-04).

**Symptoms observed in `dpt_test_lib2`:**

1. **F10.** `/implement M<N>` Phase 5 prompts the operator to chain into `/ship-milestone M<N>` after milestone archival completes. The archival step (Phase 4 § Milestone Archival) `git mv`s `specs/plan/M<N>.md → specs/plan/archive/M<N>.md`. `/ship-milestone` then reads `specs/plan/M<N>.md` (the **active** path, per its Step 1: `Read specs/plan/M<N>.md`). Result: `/ship-milestone` cannot find the plan file it's supposed to read because the previous step just moved it. Operator workaround in this session: read from `specs/plan/archive/M<N>.md` directly; a strict implementation following the spec would refuse with `ENOENT` or NFR-10. The two skills' contracts contradict each other on the same chain.
2. **F11.** Phase 4 § Milestone Archival prose orders steps as `git mv` first, then frontmatter flip. The actual ordering matters: if the agent (a) modifies the FR file's frontmatter in the working tree, then (b) runs `git mv`, the rename is staged with the **index content** (un-flipped), not the working-tree content. Symptom in this session: archive commit landed with `status: active` and `archived_at: null` for all 6 FRs even though the working tree had the flipped content. Caught downstream by `/ship-milestone`'s pre-flight ("Dirty working tree") and resolved by `git commit --amend` after re-staging. The toolkit's existing helper for the **untracked** case (`isFRUntrackedInPorcelain`) handles staging untracked files before `git mv`, but tracked-and-modified files have no analogous explicit `git add` step in the procedure prose.

## Acceptance Criteria

- AC-STE-210.1: `/ship-milestone` Step 1 grows an `archive/` fallback. Logic: if `specs/plan/M<N>.md` is missing, check `specs/plan/archive/M<N>.md`. If the archived plan exists AND every FR row in that plan has `status: archived` (consistent end state), proceed to read from the archive path. If the archived plan exists but FR rows are mixed (some still `status: active`), refuse with NFR-10: `"Active plan path missing but archived plan has unarchived FRs — corrupt state. Remedy: investigate; either restore plan to active/ or archive remaining FRs."` {#AC-STE-210.1}
- AC-STE-210.2: `/implement` Phase 4 § Milestone Archival prose reorders the procedure for tracked-and-modified FRs: **(1)** `git mv specs/frs/<name> specs/frs/archive/<name>` first (renames the file at the index level, preserving its current content), **(2)** edit the frontmatter at the new archive path (flip `status: active → archived`, set `archived_at: <ISO now>`), **(3)** `git add specs/frs/archive/<name>` to re-stage the post-edit content. Same reordering applied to the plan file: `git mv` plan first, then frontmatter-flip at the archive path, then `git add`. Idempotent: re-running the procedure on already-archived files is a no-op (frontmatter already shows `status: archived`; `git add` produces no diff). {#AC-STE-210.2}
- AC-STE-210.3: New helper `archiveFRWithFlip(repoRoot, frPath, archivedAt) -> archivePath` in `adapters/_shared/src/archive_fr.ts` wraps the move + flip + stage atomically. Phase 4 prose can reference the helper instead of expanding all three steps inline. Backwards-compat: skill prose may continue to call `git mv` + `git add` directly; the helper is an opt-in convenience. {#AC-STE-210.3}
- AC-STE-210.4: New `/gate-check` probe `archive-frontmatter-coherent` runs over `specs/frs/archive/*.md` and `specs/plan/archive/*.md`, asserting that every archived file has `status: archived` and `archived_at:` populated (non-null). Probe severity ERROR — a file in `archive/` with `status: active` is the F11 staging-order bug shipped to history; surfacing it is the regression signal. Existing legacy archives that pre-date this probe pass cleanly because past archival flows wrote the correct frontmatter (the bug is a recent regression — fix-forward without retroactive cleanup). {#AC-STE-210.4}
- AC-STE-210.5: Smoke regression fixture: archive a milestone via the Phase 4 procedure (using the corrected ordering or the new helper), then run `git show HEAD:specs/plan/archive/M<N>.md` and assert the frontmatter shows `status: archived` + non-null `archived_at`. Same assertion for every archived FR file. {#AC-STE-210.5}
- AC-STE-210.6: Backwards-compat / non-breaking: existing archived plans and existing release commits are untouched. `/ship-milestone`'s archive fallback only activates when the active path is genuinely missing — for milestones whose plans are still in `specs/plan/`, behavior is unchanged. The new gate-check probe runs on archived files only and does not retroactively rewrite anything; it just surfaces drift if any archived file has the F11-bug shape. {#AC-STE-210.6}

## Technical Design

**F10 — `/ship-milestone` Step 1 path resolution:**

```
plan_active = "specs/plan/M<N>.md"
plan_archive = "specs/plan/archive/M<N>.md"

if exists(plan_active):
  read plan_active
else:
  if not exists(plan_archive):
    refuse: "Plan M<N> not found in specs/plan/ or specs/plan/archive/"
  parsed = parse(plan_archive)
  if any(fr.status == "active" for fr in parsed.fr_rows):
    refuse: "Active plan path missing but archived plan has unarchived FRs — corrupt state."
  read plan_archive  # consistent end state
```

**F11 — Phase 4 archival reorder.** Current prose (paraphrased): "edit frontmatter, then `git mv`." Corrected prose:

```
for each FR in milestone:
  git mv specs/frs/<name> specs/frs/archive/<name>
  # edit frontmatter at the new path:
  edit specs/frs/archive/<name>:
    status: active → archived
    archived_at: null → <ISO timestamp>
  git add specs/frs/archive/<name>

# same for plan:
git mv specs/plan/M<N>.md specs/plan/archive/M<N>.md
edit specs/plan/archive/M<N>.md frontmatter (status, archived_at)
git add specs/plan/archive/M<N>.md
```

The reorder eliminates the staging-order bug because the index now sees the post-edit content as a normal modification of the archived path.

**`archiveFRWithFlip` helper.** Single-call wrapper that does `git mv` → frontmatter edit → `git add`. Returns the new archive path for downstream traceability rewrites. Optional — Phase 4 prose may continue to call the three primitives directly.

**`archive-frontmatter-coherent` probe.** Iterates over `specs/frs/archive/*.md` and `specs/plan/archive/*.md`. For each file, parses frontmatter and asserts `status: archived` (not `active` or missing) AND `archived_at:` is a non-null ISO timestamp. Surfaces ERROR rows for any file that fails. Runs with the rest of the gate-check suite — no manual invocation needed.

**Idempotency.** Re-running the corrected procedure on an already-archived FR is a no-op:

1. `git mv` of an already-archived file at the same path is a no-op (file already there).
2. Frontmatter edit at the archive path: if frontmatter already shows `status: archived` + `archived_at:` populated, the edit is identical content (no diff).
3. `git add` of unchanged content is a no-op.

So the corrected procedure can be re-run safely if the operator interrupts and resumes archival.

## Testing

- Integration test (F10): archive a milestone (move plan to archive/), then invoke `/ship-milestone M<N>`. Assert success — Step 1 reads from archive path. Then test the refuse path: archive plan with one FR row still `status: active` → `/ship-milestone M<N>` refuses with NFR-10.
- Integration test (F11): run Phase 4 archival via the corrected procedure on a fixture milestone. Run `git show HEAD:specs/frs/archive/<name>` and assert frontmatter shows `status: archived` + non-null `archived_at`.
- Integration test (F11 negative): create a fixture archive commit with the legacy bug shape (`status: active` in archived file). Run `/gate-check archive-frontmatter-coherent` → ERROR row surfaces.
- Unit test: `archiveFRWithFlip` helper invoked twice on the same FR is a no-op on the second call (idempotency).
- Smoke regression: full chain `/implement M<N>` → `/ship-milestone M<N>` succeeds end-to-end on a real milestone fixture (no operator intervention).

## Notes

- Both findings surfaced in the same Flutter-app smoke run on consecutive Phase 4 → Phase 5 → `/ship-milestone` steps. They're sequenced fixes: F11 (correct staging) is the upstream fix; F10 (archive fallback) is the downstream defensive read. Both ship together because either alone leaves part of the chain broken.
- F11's "edit at archive path AFTER `git mv`" pattern matches how the toolkit's existing untracked-FR helper (`isFRUntrackedInPorcelain` + `git add`) handles its case — it just wasn't extended to the tracked-modified case in the prose.
- F10's archive-fallback gate (refuse on mixed-status FR rows) prevents the archive-fallback from masking real corruption: if FRs aren't archived but the plan was moved, that's a partial-archival bug we want to surface, not paper over.
- Capability: a tracker-mode `/spec-write` run that creates this FR surfaces `filename_policy_override` per the unconditional rule.
