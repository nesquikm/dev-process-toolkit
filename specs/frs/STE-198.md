# STE-198: Milestone-attach: auto-create branch + capability key split (zero-milestone Linear projects) {#STE-198}

## Requirement

`/spec-write` § 0b step 4 conflates two distinct adapter states into one capability key. Split them so the auto-create-on-empty-list branch is taken when Linear can attach but the project has zero milestones.

**Findings bundled:** F1.1 from `/tmp/dpt-findings-2026-05-04.md`.

**Symptom observed in `dpt_test_lib`:** `/spec-write STE-193` (FR `subtract`, milestone M2) called `mcp__linear__list_milestones --project dpt_test_lib`, got `[]`, and rendered `milestone_attach_unavailable` in the closing summary instead of creating the M2 milestone and attaching the issue. The FR landed unattached. Operator: "Why did you forget to update Linear?"

**Root cause.** `skills/spec-write/SKILL.md` § 0b step 4 prescribes `attachProjectMilestone(...)` after `Provider.sync`. § 7's capability map defines:

> `milestone_attach_unavailable` ⇒ "no Linear project milestones available — milestone-attach skipped"

That key reads as if "no milestones in the project" is the same as "adapter cannot attach milestones." Linear's adapter **can** attach — it just needs the milestone to exist, which is a one-call create. The skill is silent on auto-create vs. surface-the-gap, so the implementer defaults to the easier "render the gap" branch.

## Technical Design

**Capability map split.** § 7 of `skills/spec-write/SKILL.md`:

| Capability key | Rendered prose | Triggered when |
|----------------|----------------|----------------|
| `milestone_attach_skipped_adapter_limit` | tracker adapter does not support project_milestone — milestone-attach skipped | Schema M `project_milestone: false` |
| `milestone_create_required` | tracker project had zero milestones; created `<milestone-name>` and attached the FR | `list_milestones` returns `[]` AND adapter supports milestones (auto-create branch fired) |

**Helper logic.** `adapters/_shared/src/attach_project_milestone.ts`:

```
async function attachProjectMilestone(provider, project, canonicalName, ticketId):
  if not provider.supports("project_milestone"):
    return { capability: "milestone_attach_skipped_adapter_limit" }
  milestones = await provider.list_milestones(project)
  if not any(m.name == canonicalName for m in milestones):
    await provider.save_milestone(project, name=canonicalName)
    await provider.save_issue(id=ticketId, milestone=canonicalName)
    return { capability: "milestone_create_required", created_name: canonicalName }
  await provider.save_issue(id=ticketId, milestone=canonicalName)
  return { capability: null }   // success, no row to surface
```

**Linear adapter.** Already supports `save_milestone` (`mcp__linear__save_milestone`) — no adapter change needed beyond the capability declaration.

**Jira adapter.** Schema M declares `project_milestone: false` (existing). No-capability branch path fires; no auto-create attempted.

## Testing

- Unit test: `attachProjectMilestone` against mocked provider, `list_milestones` returning `[]`, capability `project_milestone: true` → expect `save_milestone` + `save_issue` calls in order; capability key `milestone_create_required` returned.
- Unit test: same setup, capability `project_milestone: false` → expect no `save_milestone` call; capability key `milestone_attach_skipped_adapter_limit` returned.
- Smoke regression: Linear smoke fixture with zero project milestones + FR with `milestone: M2` → assert issue's `projectMilestone.name` matches `M2` post-run AND summary contains the `milestone_create_required` row.

## Notes

- This is a behavioral change: previously, `dpt_test_lib`-style projects (fresh Linear, no milestones) silently dropped milestone-attach. Now they auto-create. Authorization scope is unchanged — the operator already authorized `mcp__linear__save_*` calls for issue creation; `save_milestone` is the same scope.
- The renamed key (`milestone_attach_skipped_adapter_limit`) is a minor-version-compatible change. The deprecated `milestone_attach_unavailable` continues to render (mapped to the same prose) for one minor-version cycle.
- M53 deferred AC-STE-195.5 retroactive validation lives at the milestone level (M54 plan task), not as part of this FR.
- Capability: a tracker-mode `/spec-write` run that creates this FR surfaces `filename_policy_override` per the unconditional rule.
