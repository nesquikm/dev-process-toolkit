# STE-306: Audit-fix loop pattern doc {#STE-306}

## Requirement

The toolkit has two patterns for review→fix loops:

1. **Canonical (STE-225 M58 + STE-296 M77)**: main-context orchestrator dispatches via Skill tool into a forked child skill (`context: fork` + `agent:` + `user-invocable: false`) paired with a read-only subagent; child returns a fenced `<role>-result` block; orchestrator parses, then dispatches the fix into another fork (or halts).
2. **Legacy (HG95VF M9, marked superseded by STE-225)**: main-context orchestrator spawns a subagent via the Agent tool; subagent returns findings to main; main applies the fix in-place.

The canonical pattern is established in `/tdd` (4 forked stages). The legacy pattern lives on in `/implement` Phase 3 Stage B (`code-reviewer`), `/simplify` (`code-reviewer`), and `/spec-review` (entirely main-context, no fork at all). Skill authors have no canonical guidance choosing between them; HG95VF's supersession is buried in archive.

Codify the canonical pattern in `docs/patterns.md` as the rule, document the exception for in-process fix loops where the fixer needs the orchestrator's full context (`/implement` Stage B), and explicitly classify each existing audit-fix loop as canonical or legacy with a migration path.

## Acceptance Criteria

- AC-STE-306.1: `docs/patterns.md` contains a new section titled "Audit-fix loop" with stable anchor `{#pattern-audit-fix-loop}`, describing the canonical shape (main-context orchestrator → Skill tool → forked child → paired read-only subagent → fenced `<role>-result` block → orchestrator parses + dispatches fix into another fork or halts).
- AC-STE-306.2: Section enumerates the four current loops in the toolkit (`/tdd` four stages as one canonical group, `/spec-review`, `/implement` Phase 3 Stage B, `/simplify`) and classifies each as **canonical**, **legacy (in-process fix)**, or **legacy (no fork at all)**.
- AC-STE-306.3: Section explicitly references HG95VF (M9) as the superseded ancestor and STE-225 (M58) + STE-296 (M77) as the canonical precedents.
- AC-STE-306.4: Section documents the in-process-fix exception (`/implement` Stage B) with rationale — Claude Code forbids nested subagent spawns AND the fixer needs the orchestrator's full implementation context to fix — and the future-migration path (decompose orchestrator into main-context + forked child skills, mirroring `/tdd`'s shape; deferred to M82+).
- AC-STE-306.5: At least one existing skill file cross-references the new anchor (e.g., `plugins/dev-process-toolkit/skills/tdd/SKILL.md` links to the section as the canonical citation) so the pattern is discoverable from skill code.

## Technical Design

Edit `docs/patterns.md` to add a new pattern section following the existing pattern style. Stable anchor `{#pattern-audit-fix-loop}` per the project's anchor convention.

Section structure:

1. One-sentence summary of the pattern.
2. Text-only diagram showing the orchestrator + child skill + subagent + fenced block flow.
3. Classification table enumerating the four current loops with status (canonical / legacy in-process / legacy no-fork) and migration target milestone.
4. In-process-fix exception with rationale.
5. Cross-references: STE-225, STE-296, HG95VF, forward link to STE-307 (probe) and STE-308 (`/spec-review` migration).

No code changes outside `docs/patterns.md`. No new subagent or skill files in this FR.
