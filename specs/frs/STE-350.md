## Requirement

The `permissions.allow` allow-lists that govern the `/conformance-loop` → `/smoke-test` → skill spawn chain MUST authorize the `claude -p` child-spawn command, so nested spawns are pre-approved and never fall to the auto-mode safety classifier. Under headless auto-mode the classifier denies nested `claude -p` spawns, producing a 0-byte grandchild log and an empty findings file that the loop reports as green (the M94 false-green). This is the core M94 fix; it makes the loop complete **headless and interactive**, hands-off, on the operator's subscription.

Empirically confirmed 2026-07-02: an ad-hoc nested `claude -p` spawn is classifier-denied ("run this outside auto mode so the user can review it"), and the classifier's own remedy is "add a Bash permission rule to settings." The toolkit's `.claude/settings.json` `permissions.allow` currently enumerates 22 patterns — including `Bash(wait)` for the Phase A job-control — but not the `claude` spawn itself. Regression from STE-252 (bypassPermissions → tracked allow-list).

## Acceptance Criteria

- AC-STE-350.1: The toolkit repo's `.claude/settings.json` `permissions.allow` contains an entry authorizing the canonical child-spawn command as issued by `/conformance-loop` Phase A and `/smoke-test` Phase 2; a structural test asserts the pattern is present and matches the exact command string the skills emit (accounting for the `CLAUDE_CONFIG_DIR=` env prefix).
- AC-STE-350.2: The test-project `.claude/settings.json` scaffold that `/smoke-test` Phase 1 pre-creates (both `linear` and `jira` paths) contains the same authorizing entry; a test asserts the scaffolded allow-list equals the canonical list including the new entry.
- AC-STE-350.3: The spawn command form and the allow-list pattern are mutually consistent — either the spawning Bash block exports `CLAUDE_CONFIG_DIR` once and issues a bare `claude -p …` so `Bash(claude:*)` matches, or the allow-list entry matches the env-prefixed form; a test encodes the chosen convention so drift is caught.

## Technical Design

**Governing allow-lists (both must carry the pattern):**

1. Toolkit repo `.claude/settings.json` `permissions.allow` — governs the operator session's `/conformance-loop` Phase A spawn of `claude -p /smoke-test` (`.claude/skills/conformance-loop/SKILL.md:151,160`) and every `/smoke-test` Phase 2 child spawn issued from the toolkit-repo cwd.
2. The test-project `.claude/settings.json` scaffold that `/smoke-test` Phase 1 pre-creates from the parent Bash heredoc (`.claude/skills/smoke-test/SKILL.md:239`, verified at `:843`; "identical in both tracker paths", canonical per STE-106) — governs any nested spawn issued from a test-project cwd.

**The matching problem.** The canonical command begins with an inline env assignment: `CLAUDE_CONFIG_DIR=~/.claude-st claude -p "…" --plugin-dir …`. A `Bash(claude:*)` rule matches commands that *begin with* `claude`, so it will not match a command that begins with `CLAUDE_CONFIG_DIR=`. Two resolutions:

- **(A, recommended)** Export `CLAUDE_CONFIG_DIR` once at the top of each spawning Bash block and drop the inline prefix, so every spawn command begins with `claude` and `Bash(claude:*)` matches unambiguously. One-line change per spawn block; no dependence on how the permission matcher treats env prefixes.
- **(B)** Add an allow-list entry that matches the env-prefixed form. Deferred unless (A) proves infeasible — the matcher's env-prefix behavior is unverified and brittle.

**Touch points (update in lockstep):** `.claude/settings.json`; the scaffold heredoc's canonical allow-list in `smoke-test/SKILL.md`; `templates/settings.json` (if it carries the canonical list); the spawn blocks in `conformance-loop/SKILL.md` (Phase A) and `smoke-test/SKILL.md` (Phase 2) for resolution A; and any existing meta-test that asserts the canonical allow-list membership (`tests/` — see the DPT full-test-gate root).

- Runtime (smoke-validated; may ship `[~]` pending a live run): a headless nested spawn under the patched allow-list completes with non-empty child output and no `permission_denials` entry for a `claude` command (runtime proof overlaps STE-352.3's fixture).

## Notes

Root-cause thread: STE-252 replaced `bypassPermissions` with the tracked `permissions.allow` allow-list but omitted the one Bash command the entire skill is built on — spawning `claude -p` children. Pre-STE-252 the classifier was never consulted, so nested spawns always ran; post-STE-252 (default mode + allow-list) an unlisted `claude -p` spawn falls to the auto-mode classifier, which denies it headless. The previously-recorded "async await-notification / no event loop" diagnosis was wrong — the spawns are already synchronous (`conformance-loop` Phase A uses `{ … } & … wait`). STE-351 fences the omission; STE-352 detects the symptom.

**Risk to validate at implement (not a separate FR):** `/conformance-loop` Phase A `wait`s on both ~20-min smoke legs inside one Bash call, which may exceed a single Bash-tool timeout independent of this fix.
