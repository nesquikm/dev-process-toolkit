# STE-187: /smoke-test auto-disambiguate same-day Linear project names {#STE-187}

## Requirement

The /smoke-test driver currently prescribes the Linear project name as `DPT Smoke Test (<YYYY-MM-DD>)`. The 2026-05-01 Linear run (smoke #9) had to manually pass `2026-05-01-v2` because smoke #8 had already created and archived `DPT Smoke Test (2026-05-01)` earlier the same day.

Linear allows projects with identical names (no uniqueness constraint), so the driver would not have refused — but creating a second same-named project splits the audit trail across two indistinguishable entries, making post-hoc inspection painful. The operator-side workaround was a manual `-v2` suffix.

This FR auto-disambiguates by detecting same-day collisions and applying a deterministic suffix scheme.

Out of scope:

- Cleanup of stale Linear projects from prior runs. Phase 5 teardown already archives at `state: completed`; this FR does not change teardown.
- Jira-path equivalent. The Atlassian Rovo MCP exposes no `createJiraProject` tool — Jira Spaces are operator-pre-created and reused across runs (no per-run isolation). The collision concern applies to Linear-path only.
- Initiative / cycle / milestone collisions. Linear projects are the only same-day collision surface that affects the smoke driver.

## Technical Design

### Files touched

| File | Change |
|------|--------|

### Suffix-resolution algorithm

```
base = "DPT Smoke Test (<YYYY-MM-DD>)"
matches = list_projects(query=base).filter(p => p.name.startsWith("DPT Smoke Test (<YYYY-MM-DD>") && p.name.endsWith(")"))
if matches.empty:
    name = base
else:
    # Find smallest N >= 2 such that "DPT Smoke Test (<YYYY-MM-DD>-vN)" is not in matches.
    taken = set of suffix integers parsed from matches' names (base = 1, "-v2" = 2, etc.)
    N = min(n for n in 2..∞ if n not in taken)
    name = "DPT Smoke Test (<YYYY-MM-DD>-v" + N + ")"
save_project(name=name, team="STE")
```

The query-then-filter approach (instead of trusting `query` to be exact) handles Linear's substring-matching `query` semantics. The filter on `startsWith / endsWith` is the precise equality check.

### Why `-v<N>` not `<HHMMSS>`

Two options were considered:

- **`<HHMMSS>` timestamp suffix** (e.g., `(2026-05-01-194827)`). Pros: collision-free without a list-then-filter call. Cons: harder for operators to read at a glance (which run was first? which was the rerun?); the extra precision is unused by the audit trail.
- **`-v<N>` increment.** Pros: human-readable ordering (v2, v3, v4 — temporal order is preserved); minimal visual difference from canonical. Cons: requires the list-then-filter call (one extra MCP round-trip in Phase 1).

The `-v<N>` form is operator-friendlier for the audit-trail use case the FR exists to support. The extra MCP call is cheap (single `list_projects` query, narrow result set).
