# STE-400: smoke-child-capture: artifact-level liveness detector for clean-exit/zero-work runs {#STE-400}

## Summary

The end-of-run chain-integrity check treats a captured child session as healthy when its transcript carries a top-level result event. That is a truncation check, not a work check: a child that reaches a question it cannot answer under claude -p, states the question in prose, and ends its turn exits cleanly with a result event and zero work done — and passes. No existing detector inspects whether the skill actually finished its job.

This FR adds a detector that looks for the skill's own closing-summary marker in the child's assistant text. Every canonical skill emits a closing summary as its last act; a child that halted early never emits it. The marker's absence on an otherwise-complete capture is the clean-exit/zero-work signal, and it is scored against assistant text only so the skill body echoing its own template cannot satisfy it.

## Requirement

The 2026-07-19 conformance run produced this exact false-green and no detector caught it: a `/spec-write` child halted at an unanswerable gate, exited `subtype=success`, and created nothing — yet the capture passed `checkChildSpawnCapture` (non-empty, no denials), `checkAllowlistInert` (no warning), and `assertChainIntegrity` (a `result` event was present). The STE-355 detector family was built for the 2026-07-02 truncation shape (missing `result` event); this is a different truncation shape — clean `result`, zero work — and slips straight through.

The only signal that separated the dead run from a live one was the absence of the skill's completion output. A deterministic, byte-checkable version of that signal is the skill's closing-summary marker in the assistant-text stream. The detector must score assistant text only (not raw NDJSON), because the skill body — injected as a `user` message and echoed in tool_results — enumerates its own template tokens, so a raw-NDJSON grep would match the documentation of the marker rather than its emission (the F3-class false-positive the run also surfaced).

## Acceptance Criteria

- AC-STE-400.1: a new detector `checkSkillCompletionSignal(ndjson, child, marker)` in `smoke_child_capture.ts` returns exactly one high-severity `ChildSpawnFinding` when the capture's assistant text (via `extractAssistantText`) does NOT contain `marker`. {#AC-STE-400.1}
- AC-STE-400.2: `checkSkillCompletionSignal` returns `[]` when the assistant text contains `marker` (healthy completion). {#AC-STE-400.2}
- AC-STE-400.3: the detector scores assistant text ONLY — a `marker` string appearing solely in a `user` / tool_result event (the skill body echoing its own closing-summary template) does NOT satisfy it; such a capture still yields the finding. {#AC-STE-400.3}
- AC-STE-400.4: the finding uses the existing `ChildSpawnFinding` shape (`severity: "high"`); the diagnostic names the child and states the clean-exit/zero-work cause (a stable diagnostic prefix in the module's existing style). {#AC-STE-400.4}
- AC-STE-400.5: an empty capture / empty assistant text ⇒ marker-absent ⇒ finding (consistent with the empty-handling posture of the sibling detectors). {#AC-STE-400.5}

## Technical Design

Single-file addition to `plugins/dev-process-toolkit/adapters/_shared/src/smoke_child_capture.ts`, alongside the existing detector families. Reuse the module's own `extractAssistantText` (already exported) and the `highFinding` helper.

```ts
const COMPLETION_DIAGNOSTIC_PREFIX =
  "STE-400 regression: skill produced no completion signal — ";

/**
 * Detect a clean-exit/zero-work run: the capture parsed, exited with a
 * result event, but the skill's closing-summary marker never reached the
 * assistant-text stream — the skill halted (e.g. at an unanswerable gate)
 * without finishing its job. Scores extractAssistantText only, so the skill
 * body echoing its own template in a user/tool_result event does not satisfy
 * the check (the F3-class false-positive guard).
 */
export function checkSkillCompletionSignal(
  ndjson: string,
  child: string,
  marker: string,
): ChildSpawnFinding[] {
  const assistantText = extractAssistantText(ndjson);
  if (assistantText.includes(marker)) return [];
  return [
    highFinding(
      `${COMPLETION_DIAGNOSTIC_PREFIX}${child} (clean exit, marker "${marker}" absent from assistant text)`,
    ),
  ];
}
```

The detector is orthogonal to the existing three families (spawn-capture, allowlist-inert, chain-integrity) — same posture as the module comment already documents. Wiring it into the Phase 2.Y aggregation with a per-skill marker map (e.g. `/spec-write` ⇒ `## /spec-write summary`) is a project-local smoke-test SKILL change, outside this FR; this FR ships the helper + tests. The marker is a caller-supplied argument, so the helper is skill-agnostic and the per-skill map lives in the driver.

## Testing

Colocated `checkSkillCompletionSignal` tests (mirror the existing `smoke_child_capture` test placement):

- assistant text containing `marker` ⇒ `[]` (AC-STE-400.2).
- assistant text NOT containing `marker` ⇒ one high finding whose diagnostic names the child + carries the STE-400 prefix (AC-STE-400.1, AC-STE-400.4).
- a capture where `marker` appears ONLY in a `user` / tool_result event (built as an NDJSON fixture with the marker in a user message but absent from every assistant text block) ⇒ still one finding (AC-STE-400.3) — the assistant-text-scoping guarantee.
- empty capture (`""`) ⇒ empty assistant text ⇒ one finding (AC-STE-400.5).
- a realistic healthy `/spec-write`-shaped capture whose assistant text carries `## /spec-write summary` ⇒ `[]`.

## Notes
