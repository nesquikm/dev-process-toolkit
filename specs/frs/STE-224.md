# STE-224: feat(skills/conformance-loop): automated test+fix loop wrapping /smoke-test {#STE-224}

## Requirement

Add a project-local skill `/conformance-loop` (at `.claude/skills/conformance-loop/SKILL.md`, **not** plugin) that automates the manual two-terminal `/smoke-test` workflow:

1. Each iteration spawns parallel `claude -p /smoke-test --tracker linear --linear-team STE` and `claude -p /smoke-test --tracker jira --jira-project DST` subprocess sessions.
2. **Phase A** aggregates the two per-tracker findings files (`/tmp/dpt-smoke-findings-<date>-{linear,jira}.md`) into `/tmp/dpt-conformance-loop-<date>-iter-<N>.md` with cross-tracker dedup.
3. **Default mode (`capture-only`)** exits after iteration 1 with the aggregated report — honors `/smoke-test`'s "Capture, don't fix" rule unchanged.
4. **Opt-in `--auto-fix` mode** runs **Phase B** per finding: spawn `claude -p /dev-process-toolkit:spec-write` (allocate FR) → `claude -p /dev-process-toolkit:implement <new-tracker-id>` (build + commit), then re-iterate until termination.

The new skill is the formally-sanctioned exception to `/smoke-test`'s "Capture, don't fix" + "One run per release cycle" rules. Capture-only mode honors those rules unchanged; `--auto-fix` mode opts the operator into the automated loop with `--max-iterations` + no-progress safety rails (no budget cap — operator controls cost via iteration count).

## Technical Design

**Architecture.** The skill runs in the parent claude session (toolkit-repo cwd). Each iteration uses a single Bash heredoc that backgrounds two `claude -p /smoke-test ...` subprocess calls, captures their PIDs into shell variables, and `wait`s on both. Subprocess output is captured to per-iteration log files at `/tmp/dpt-conformance-loop-<date>-iter-<N>-{linear,jira}.log` for forensics. Phase A aggregation reads the two per-tracker findings files (existing canonical paths, **NO `/smoke-test` changes**) into a unified markdown report.

**Parallelism mechanism.** Uses Bash subprocess parallelism, **NOT the agent-team primitive** — agent teams have no `fork: true` flag and aren't recommended for serial orchestration per the Claude Code docs (`https://code.claude.com/docs/en/agent-teams`). Each subprocess is a top-level `claude -p` session, which can invoke skills via the literal-first-line pattern (sub-agents cannot, per docs).

**Fixer dispatch (`--auto-fix`).** The parent skill spawns child `claude -p /dev-process-toolkit:spec-write` and `claude -p /dev-process-toolkit:implement <new-tracker-id>` sessions sequentially per finding. Both children inherit the operator's Phase-0 batch consent via the STE-220 `-p` carve-out (auto-applies the draft + commit gates).

**Termination probes.**
- **Green:** both per-tracker findings files have zero `**Severity:** high` lines (`grep -c '\*\*Severity:\*\* high'` returns 0 in both files).
- **No-progress:** `diff` between iter-N and iter-(N-1) aggregated findings is empty, OR `git rev-parse HEAD` is unchanged after Phase B.
- **Max-iterations:** simple counter.

**Cross-tracker dedup.** First pass: exact-match on the `STE-<N> runtime regression: <fixture>` diagnostic line if present (matches the convention from `/smoke-test` Phase 2.X fixtures). Second pass: normalize body (lowercase, strip whitespace + markdown), compute substring overlap; ≥ 80% ⇒ dedup with `~probable-dup` flag for operator review.

**Override rationale.** `/smoke-test`'s "One run per release cycle" rule was justified by token cost + Linear teardown labor. With token cost dropped from this design's scope, only the teardown labor remains — the operator accepts the per-iteration teardown burden as the cost of automation. The "Capture, don't fix" rule is overridden only when `--auto-fix` is explicitly set; capture-only mode preserves it.

## Notes

**Risk — driver-side caveats trigger `/implement`.** `/smoke-test`'s findings files include both genuine plugin bugs AND driver-side caveats (claude-st -p behavior, MCP issues). `--auto-fix` would dispatch `/spec-write` + `/implement` on driver-side caveats too, producing FRs the maintainer doesn't agree with. **Mitigation:** only dispatch on `**Severity:** high`; medium/low surface in the report without auto-fix. Driver-side caveats are conventionally `medium` per `/smoke-test`'s findings template, so this filters them out by default.

**Risk — runaway loop on persistent finding.** A finding `/implement` can't actually fix would loop until `--max-iterations`. The no-progress detection (zero diff after Phase B's `/implement`) catches this on iteration 2, exiting with `status: no-progress`. Acceptable mitigation under the "operator owns iteration count" model.

**Override sanction.** This FR is the formal sanction for overriding `/smoke-test`'s "Capture, don't fix" + "One run per release cycle" rules — those rules predate the maintainer's automation request. The new skill is the authorized deviation; raw `/smoke-test` invocations continue to follow the original rules.

**Open question (deferred to M58+).** `/loop` outer wrapper integration is NOT specced here; the inner per-iteration loop lives entirely in the new skill. `/loop` is left as a separate cross-release re-run convenience the operator can choose if they want to re-run `/conformance-loop` automatically across release cycles.

**Cross-reference.** This FR is M57's sole FR; M57 ships in v2.10.0.
