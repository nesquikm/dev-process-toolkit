## Requirement

The toolkit's own machinery runs on **bun**, regardless of the consumer
project's stack. The `adapters/_shared` TypeScript helpers ŌĆö `requireOrRefuse`,
`buildFRFrontmatter`, the doctor/marker/parser modules, and
`requireCommittableBranch` (which README itself documents as called by *every
commit-producing skill*) ŌĆö plus the Linear/Jira tracker adapters ship a
`bun.lock` / `package.json` and are invoked by skill prose via `bun run ŌĆ”`.
A Flutter, Python, or (post-M88) Kotlin consumer who installs the plugin still
needs bun on PATH for those helpers to execute.

Today `README.md` has **no** Prerequisites section, and bun is documented only
narrowly ŌĆö for tracker-mode setup (`/setup` step 7b "Bun prereq" /
`docs/setup-tracker-mode.md`). That narrow framing under-states the requirement:
it reads as "bun is only for Linear/Jira users," when in fact bun is a hard,
stack-independent prerequisite for the toolkit's commit-producing skills. This
FR adds a top-level Prerequisites section stating bun as a universal
prerequisite, and reconciles the existing tracker-mode mention so the universal
statement is canonical (promote-and-widen, not a contradictory duplicate).

## Technical Design

adjacent to "Install as Plugin". Apply a one-line reconciling edit to the
`/setup` step 7b bun mention and `docs/setup-tracker-mode.md` so they point at
the new section rather than restating a tracker-scoped requirement. No code, no
