## Requirement

Strengthen `/conformance-loop`'s Phase 0 pre-flight so the M94 allow-list omission (STE-350) can never silently recur, and add a subscription-billing guard. STE-252's pre-flight (f) only asserts `permissions.allow` is non-empty — it passes even with the spawn pattern missing, which is exactly how the M94 false-green shipped undetected.

## Acceptance Criteria

- AC-STE-351.1: `/conformance-loop` Phase 0 pre-flight (f) — and `/smoke-test`'s corresponding pre-flight — asserts `permissions.allow` **contains** the child-spawn pattern (not merely `length > 0`); a missing pattern refuses with the NFR-10 canonical shape naming the remedy (add the spawn pattern to the allow-list).
- AC-STE-351.2: A `/gate-check` structural probe fails GATE when the child-spawn pattern is absent from the tracked `.claude/settings.json` allow-list or from the `/smoke-test` scaffold snippet.
- AC-STE-351.3: A Phase 0 pre-flight refuses (NFR-10 shape) when `ANTHROPIC_API_KEY` or `ANTHROPIC_AUTH_TOKEN` is set, so an on-demand run cannot silently bill an API account instead of the operator's subscription; the interactive-override path is documented.
- AC-STE-351.4: The allow-list pre-flight emits a byte-checkable capability token on the hit path (`spawn_pattern_allow_present`) consistent with the existing `permissions_allow_present` convention.

## Technical Design

- **Pre-flight (f) strengthening** (`conformance-loop/SKILL.md:82-90`): today `jq -e '.permissions.allow | length > 0'`. Replace with a contains-check for the canonical spawn pattern (e.g. `jq -e '.permissions.allow | index("Bash(claude:*)")'`, matching whatever literal STE-350 lands). On miss → NFR-10 canonical refusal (Verdict / Remedy / Context) reusing STE-350's remedy. Mirror the assertion in the `/smoke-test` pre-flight the loop delegates to (refusals (c)–(e)).
- **gate-check probe:** add a structural probe (same shape as the existing STE-252 / STE-324 allow-list probes) asserting the spawn pattern is present in `.claude/settings.json` and in the smoke-test scaffold snippet; fail GATE otherwise. Register it in the probe set + tests.
- **ANTHROPIC_API_KEY guard:** a new Phase 0 refusal (g). Read the environment; if `ANTHROPIC_API_KEY` or `ANTHROPIC_AUTH_TOKEN` is set → NFR-10 refusal (Verdict: an on-demand run would bill that API account at per-token rates rather than your subscription; Remedy: unset the key for this run, or re-run interactively acknowledging the cost; Context: the loop spawns many `claude -p` children). Document that an operator who *wants* API billing unsets the guard or passes an explicit override. Aligns with the existing STE-191 KEY-surfacing pre-flight.
- **Capability token:** log `spawn_pattern_allow_present` to `/tmp/dpt-conformance-loop-<date>-approval.txt` alongside `permissions_allow_present` on the hit path (one literal line, byte-grep-checkable).

## Testing

- Pre-flight refuses (NFR-10 shape) on a fixture `settings.json` whose allow-list lacks the spawn pattern; passes + emits `spawn_pattern_allow_present` when present.
- gate-check probe: fails on an allow-list without the pattern; passes with it. Register in the probe test suite.
- `ANTHROPIC_API_KEY` set → refuse; `ANTHROPIC_AUTH_TOKEN` set → refuse; neither set → proceed. Cover both env vars.

## Notes

This FR is the fence STE-350's absence proved was missing: a `length > 0` assertion is not the same as "the load-bearing pattern is present." The `ANTHROPIC_API_KEY` guard operationalizes the toolkit's no-budgets stance — an auth-correctness gate, not a cost cap (consistent with the project's standing no-budgets position). Depends on STE-350 landing the canonical pattern literal.
