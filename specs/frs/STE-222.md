# STE-222: test(skills/implement) — real -p regression coverage for cross-cutting spec-drift propagation {#STE-222}

## Requirement

STE-215 (M55) added a `/implement` Phase 4 hook that, on tracked file deletion, propagates the removal to `specs/technical-spec.md` and `specs/testing-spec.md` directory-tree blocks; lands a follow-up commit `chore(specs): propagate <removed-path> removal to cross-cutting specs`; and adds a safety-net `/gate-check` probe `cross-cutting-spec-stale-file-refs`. The 2026-05-04 v2.8.0 smoke runs (Linear Run #2 + Jira Run C) confirm the fix works empirically: neither run surfaces a B-F3-equivalent cross-cutting drift finding, and the dev-process-toolkit's own pre-existing drift was cleared as the fix-it-now sweep in M55.

But the fix lives entirely in SKILL.md prose (`/implement` Phase 4 hook + new `/gate-check` probe) — no automated `-p` regression catches a future divergence between the documented contract and actual runtime behavior. Same silent-breakage surface as STE-213 (which proved the failure mode is real) and STE-214 (covered by STE-221, this milestone).

**Findings bundled:** none new. Pure defensive `-p` regression coverage for the existing STE-215 fix.

**Symptoms this FR prevents (vs. detects):**

1. The Phase 4 propagation hook is silently bypassed in a future `/implement` rewrite — file deletions land but propagation commits don't. Stale refs accumulate in cross-cutting specs.
2. The `cross-cutting-spec-stale-file-refs` probe is silently dropped from `/gate-check`'s probe set during a future refactor — the safety net stops firing.
3. The propagation commit's subject convention drifts (e.g., from `chore(specs): propagate X removal …` to a different shape) — downstream tooling that greps for the convention breaks silently.

## Acceptance Criteria

- AC-STE-222.1: Positive fixture: a sandboxed test project where `specs/technical-spec.md` and `specs/testing-spec.md` directory-tree blocks reference a tracked file (e.g., `src/.placeholder.test.ts`). A `claude -p /implement <FR>` invocation against an FR whose implementation deletes that file must produce: (a) the expected feature commit, (b) a follow-up commit with subject `chore(specs): propagate <path> removal to cross-cutting specs`, AND (c) clean directory-tree blocks (the leaf entry naming the removed path is absent post-run). All three conditions assert against captured stdout + `git log` state of the fixture project. {#AC-STE-222.1}
- AC-STE-222.2: Control fixture: identical project but the FR's implementation does NOT delete any tracked file. The `claude -p /implement <FR>` invocation must produce no propagation commit (silent no-op per STE-215 AC.4). Assertion: `git log --grep "propagate.*removal to cross-cutting specs" --since "<run-start>"` returns 0 commits. The control case proves the propagation step is opt-in on actual deletions, not a blanket post-implement step. {#AC-STE-222.2}
- AC-STE-222.3: Probe fixture: a sandboxed test project where `specs/technical-spec.md` carries a directory-tree leaf for a path that doesn't exist on disk (manually-introduced drift, no `/implement` run involved). A `claude -p /gate-check` invocation must produce a `cross-cutting-spec-stale-file-refs` ADVISORY row in captured stdout (NOT GATE FAILED — STE-215 AC.5 specifies ADVISORY). The probe fixture verifies the safety-net side of STE-215; AC.1 + AC.2 verify the hook side. {#AC-STE-222.3}
- AC-STE-222.4: All three fixtures run inside the existing `/smoke-test` Phase 2 chain (or as sibling phase 2.X), reusing the smoke-driver test infrastructure landed by STE-220. Failures surface with `STE-215 runtime regression: <fixture-name>` diagnostic in captured stdout — analogous to STE-221's diagnostic shape. The `STE-215` reference is by the original FR's tracker ID, not STE-222's. {#AC-STE-222.4}

## Technical Design

**Smoke-driver extension (shared with STE-220 + STE-221).** STE-222's fixtures sit in the same `/smoke-test` Phase 2.X slot as STE-221's. The shared infrastructure is "spawn `claude -p` against a fixture project state, capture stdout, assert expected outcome." STE-220 lands the infrastructure as part of its AC.4; STE-221 + STE-222 add fixtures on top.

**Fixture project layout.** STE-222 reuses `dpt-test-project-linear` (or `-jira`, since `/implement`'s propagation hook is tracker-mode-agnostic — both legs exercise it). The driver:

1. Pre-stages a `specs/technical-spec.md` + `specs/testing-spec.md` whose directory trees reference the would-be-deleted path.
2. Pre-stages an FR whose implementation includes a deletion of that path.
3. Invokes `claude -p /implement <FR>`.
4. Captures stdout + post-run `git log`.
5. Asserts the three positive-fixture conditions.
6. Rolls back the fixture state for the next sub-fixture.

The control fixture and probe fixture follow the same shape with appropriate state setup.

**Path of deletion (positive fixture).** The deletion target is `src/.placeholder.test.ts` — the M1 placeholder shield, which has the longest history of cross-cutting drift in this repo's own smoke history (smoke #6 F4, smoke #8 F5, B-F3, plus the dev-process-toolkit's own pre-M55 drift). Reusing the canonical drift target keeps the test aligned with the historical bug surface.

**Diagnostic shape.** Same shape as STE-221:

```
STE-215 runtime regression: <fixture-name>
  expected: <propagation-commit-present | propagation-commit-absent | ADVISORY-row-present>
  actual:   <observed state>
  stdout excerpt: <last 20 lines of captured output>
  git log excerpt: <last 5 commits since run-start>
```

The `git log excerpt` line is STE-222-specific (vs. STE-221's stdout-only diagnostic) — `/implement` failures often surface in `git log` shape rather than stdout content, so the diagnostic carries both.

**Run cadence.** Both Linear and Jira legs (vs. STE-221 which is Linear-only). `/implement`'s propagation hook is adapter-agnostic.

## Testing

- Unit test: assert each fixture's pre-state setup is correct (the directory-tree leaf is present in the cross-cutting specs before the run).
- Unit test: assert the smoke driver's diagnostic shape carries `STE-215` (the FR being regression-tested), not `STE-222`.
- Integration test: real smoke run on both legs; assert all three fixtures pass; assert the diagnostic fires with the canonical shape on a deliberately-introduced regression (e.g., a temporary `git revert` of STE-215's hook).
- Negative test: temporarily disable the propagation hook in `/implement` Phase 4; assert the positive fixture's commit-present assertion flips to fail with the canonical diagnostic; restore the hook and re-run to confirm green.
