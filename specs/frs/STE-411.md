# STE-411: /upgrade Step 0: monolithic-layout hint on the never-bootstrapped path {#STE-411}

## Summary

When upgrade refuses a never-bootstrapped tree it routes the operator to setup and says there is nothing to migrate — which is misleading on a legacy project whose spec layout the toolkit itself once generated and can split automatically after bootstrap. This FR adds a read-only layout check to that refusal path: when the retired monolithic layout is present, the routing message gains lines naming it and pointing at the bootstrap-then-upgrade chain. The ownership rule itself does not move — no migration ever runs on an unmanaged tree.

## Requirement

`skills/upgrade/SKILL.md` Step 0 (lines 28–38) routes never-bootstrapped trees to `/setup` and exits before walking a single detector, printing the byte-pinned refusal block "Nothing to migrate: this project has never been bootstrapped …". On a legacy monolithic tree that message is wrong in spirit: "no legacy state to migrate" is exactly backwards — a first-class migration (`monolith-split`, 1.16.0) exists for precisely this layout, reachable after bootstrap.

The AC-STE-391.2 routing and the ownership invariant stay untouched; the refusal just stops being silent about the one layout DPT itself generated. The hint machinery is STE-410's probe helper, which already returns the row + hint on unmanaged monolithic trees — this FR wires it into the Step 0 surface and gives it a deterministic CLI entry.

## Acceptance Criteria

- AC-STE-411.1: Step 0's never-bootstrapped path runs the STE-410 probe helper read-only via a deterministic bun invocation named verbatim in the SKILL (a CLI entry on `upgrade_staleness.ts` printing the probe's notes, one per line, empty output on a clean tree); when output is non-empty, the refusal block additionally renders those lines (the `monolith-split` row + `LEGACY_MONOLITH_HINT`) verbatim; otherwise the refusal block is byte-identical to today. {#AC-STE-411.1}
- AC-STE-411.2: the ownership invariant sentence ("Never migrate a file the toolkit does not own.") survives verbatim; the path stays read-only and terminal — route to `/setup`, exit, no detector application, no writes. {#AC-STE-411.2}
- AC-STE-411.3: a meta-test pins that `skills/upgrade/SKILL.md` carries `LEGACY_MONOLITH_HINT` byte-equal to the exported constant (the shared-literal pattern from the M108/M109 docs-pin suites) plus the Step 0 invocation line. {#AC-STE-411.3}

## Technical Design

**`upgrade_staleness.ts`** gains an `import.meta.main` CLI guard: `bun run adapters/_shared/src/upgrade_staleness.ts <projectRoot>` (default `process.cwd()`) awaits `runUpgradeStalenessProbe` and prints `report.notes` joined by newlines — empty stdout on a clean or fully-managed-and-current tree. No new file; the probe function is untouched (STE-410 owns its semantics).

**`skills/upgrade/SKILL.md`** Step 0 amendment: after the never-bootstrapped verdict and before printing the refusal block, run the CLI read-only; when stdout is non-empty, append its lines to the refusal block (after the `Context:` line, preserving the NFR-10 shape of the first three lines). The exit remains terminal. On an unmanaged monolithic tree the operator therefore sees: nothing-to-migrate + route-to-setup + the monolith row + the bootstrap-then-upgrade hint, in one block.

## Testing

- `tests/m111-ste-411-upgrade-hint-pins.test.ts` (new): the SKILL carries the byte-equal `LEGACY_MONOLITH_HINT` literal, the Step 0 invocation line, and the intact ownership sentence.

## Notes

The hint slots ahead of the `/setup` routing (spec-research: AC-STE-391.2). The CLI guard lives here rather than in STE-410 because this FR is its only consumer; if a later surface wants JSON output, extend then, not now.
