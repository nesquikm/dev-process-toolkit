## M106: Milestone-keyed branch naming {#M106}

**Goal:** one canonical branch template — `{type}/m{N}-{slug}` — across both tracker
mode and `mode: none` and across all branch-creating surfaces (`/spec-write`'s § 7a
pre-commit gate, `/implement`'s 0.b″ proposal, `/setup`'s seeded defaults). The
milestone is the toolkit's unit of shipping (one branch, one PR, one release per
milestone), so the branch names the milestone, not whichever ticket happened to be
allocated first. Ticket-keyed naming survives only as a milestone-less fallback and
as legacy acceptance (existing branches never re-prompt; they are never proposed
again). Approved 2026-07-16 `/brainstorm` — approach 1 of 3 (milestone-always);
hybrid m{N}+ticket and scope-keyed status quo rejected.

**Prerequisites:** v2.46.0 shipped (main @ PR #40). Load-bearing existing machinery:
STE-64 (branch automation — `buildBranchProposal`, `isCurrentBranchAcceptable`, the
mode-split defaults this milestone retires), STE-228 (universal pre-commit branch
gate + per-skill `branchNameFor` builders + collision suffixing), STE-381
(deterministic `branchTypeFor` — the `{type}` leg stays untouched).

**Release target:** v2.48.0 (minor). Codename TBD at `/ship-milestone`. ⚠️ **Three-way
contention** — M101 (branch `feat/ste-375-jira-milestone-as-epic`) and M105 (branch
`feat/ste-385-fr-summary-section`) are both unshipped and in flight; whichever ships
first takes the next free number and the others rebase + bump. `/implement` Phase 1 +
`/ship-milestone` re-validate against the live `plugin.json`, so this is a soft
prediction, not a hard claim.
