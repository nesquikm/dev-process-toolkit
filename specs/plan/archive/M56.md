## Why this milestone exists

M55 closed six FRs from the 2026-05-04 smoke findings on the assumption that updating `skills/<X>/SKILL.md` prose was equivalent to closing each FR. The post-M55 v2.8.0 smoke runs disprove that assumption for STE-213:

1. **STE-213 runtime didn't change.** SKILL.md § 4 carries the Auto-mode / `-p` default-apply carve-out for `/spec-write`'s draft gate. AC.6 ("`claude -p` completes end-to-end") was checked off. Two v2.8.0 smoke runs (Linear F2-1 + Jira C-F1) confirm the runtime still hangs at `Approve and proceed?` — exactly the bug STE-213 was meant to close.
2. **STE-214 + STE-215 work today** (Linear Run #2 reports `37/37 conformance probes PASS` with no probe #26 advisory; no B-F3-equivalent cross-cutting drift in either run). But the same fix mechanism — SKILL.md prose with no automated `-p` regression catching divergence — leaves them exposed to the same silent-breakage surface that bit STE-213.

The systemic gap M56 closes: **AC verify lines that read "completes end-to-end under `-p`" must be satisfied by external evidence (captured stdout from a real `claude -p` invocation), not by LLM self-confirm during /implement Phase 4.** STE-213's verify line passed because the LLM ran the test once in-loop and self-confirmed; nothing external ever exercised the carve-out at runtime. M56's three FRs together: fix the broken one (STE-220 reopens STE-213's runtime path), build fences around the two that happen to work (STE-221 + STE-222 add real `-p` regression fixtures for STE-214 + STE-215). The smoke-driver test infrastructure that STE-220 lands becomes shared infrastructure for all three.

After M56 ships:

- A `claude -p /spec-write …` invocation against a fresh project on v2.9.0 completes end-to-end without prompt-body pre-authorization for the FR-draft gate. The smoke driver retires its B-F1 / F2-1 / C-F1 workaround. Captured stdout from a real `-p` run is the archival evidence — not LLM self-confirm.
- `/smoke-test` Phase 2 (or a sibling phase 2.X) carries three `-p` regression fixtures: positive + control + (where applicable) probe-side. Failures emit `STE-214 runtime regression: <fixture>` / `STE-215 runtime regression: <fixture>` diagnostics with stdout + `git log` excerpts — triage is instant.
- The "documented but unenforced" failure mode is structurally closed for the M55 cohort. Future milestones that ship SKILL.md-prose fixes are expected to include an automated `-p` regression test in their AC set; STE-220's smoke-driver extension is the canonical pattern.

## Notes

- **The systemic gap M56 closes.** M55's archival `[x]` checkmarks were granted on the basis of LLM self-confirm during /implement Phase 4 self-review, not by external evidence from real `claude -p` invocations. STE-213's AC.6 ("`claude -p` completes end-to-end") read as satisfied because the LLM running /implement believed it would work; the bug surfaced 30 minutes later when a real `claude -p` smoke ran. M56 introduces the **archival-evidence contract**: FRs that change runtime behavior under `-p` mode require captured stdout from a real `-p` invocation as archival evidence. STE-220's AC.6 is the prototype; STE-221 + STE-222 inherit the contract via their smoke-driver fixture assertions.
- **Why STE-220 is medium severity but the cohort lands as a single milestone.** STE-220 alone fixes a real bug; STE-221 + STE-222 are pure defensive coverage for fixes that work today. Splitting them across milestones would invite the exact failure mode the milestone is closing — STE-220 ships the smoke-driver test infrastructure that STE-221 + STE-222 build on, so they have a soft ordering preference but no hard dependency. Bundling under one milestone keeps the runtime-fix + fence pair atomic.
- **The smoke-driver extension is M56's reusable infrastructure.** Future milestones that ship SKILL.md-prose fixes for `/spec-write`, `/gate-check`, or `/implement` skill behavior should reuse the captured-stdout assertion harness STE-220 lands. The pattern: pre-stage fixture state → invoke `claude -p` → capture stdout → assert expected outcome → emit canonical `STE-<XXX> runtime regression: <fixture-name>` diagnostic on failure. The `STE-<XXX>` field names the FR being regression-tested, not the FR carrying the test (this lets future cohorts add fixtures for older FRs without re-naming the diagnostic).
- **`milestone_attach_unavailable` deprecation cutover.** STE-221 AC.3 honors the deprecated alias per STE-198's rollover note. When the deprecation window closes, STE-221's deprecated-alias fixture flips to expecting GATE FAILED — track via STE-198's existing rollover note, not a new FR. This is the only forward-compatibility commitment M56 carries.
- All 3 FRs surface `filename_policy_override` capability rows during `/spec-write` per the unconditional rule. This session's `/spec-write` run that produced this milestone's specs emits the row 3 times — once per FR.
- **STE-211 round-trip preservation in effect.** All 3 Linear ticket descriptions (STE-220 through STE-222) were pushed via `mcp__linear__save_issue` directly, not through the toolkit's Linear adapter. The descriptions therefore do NOT carry the AC-prefix backtick wrap that AC-STE-211.1 mandates for adapter-pushed descriptions; the local FR files in `specs/frs/STE-220.md` through `specs/frs/STE-222.md` carry plain AC prefixes (canonical per AC-STE-211.1). On the next adapter-driven push (e.g., `/implement` Phase 1 claim), the backtick wrap will land via the adapter's emit path; legacy auto-link mangling (if any) will be stripped on import via `stripLinearACFences`.
