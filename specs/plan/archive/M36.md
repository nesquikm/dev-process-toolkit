# M36 — Conventional Commits {#M36}

**Release target:** v1.37.0 (provisional minor — new public surface: commit-msg hook template + `/setup --commitlint` flag). Codename TBD; finalize at `/ship-milestone`.
**Status:** drafted 2026-04-27 from `/brainstorm` decision (Approach 3 — pure POSIX shell hook by default + opt-in commitlint for projects with Node/Bun tooling).
**Origin:** user request "ALL commits should be Conventional Commits-compliant" — switch from the bespoke `M<N>: vX.Y.Z "Codename" — title (STE-<N>)` style used for M1–M35 to CC v1.0.0, dogfooded.

## Why this milestone exists

The toolkit ships a Spec-Driven Development workflow but its own commit history (M1–M35) uses a bespoke message format that doesn't compose with downstream automation: changelog generators, semantic-release tooling, conventional-commits-aware GitHub Actions, etc. Every adopting project pays the same tax — there's no shared format to gate on. M36 closes that gap by adopting Conventional Commits v1.0.0 as the toolkit's commit convention and shipping a local `commit-msg` hook that adopting projects install via `/setup`. Dogfooding is the discipline: this repo's own M36 commits use CC and are validated by the same hook.

The hybrid hook posture (POSIX shell default, opt-in commitlint) is deliberate. The toolkit has no `package.json` and runs TS files via Bun directly — adding commitlint would introduce the toolkit's first runtime dependency. The polyglot examples (Flutter, Python) would inherit that friction. A pure shell hook keeps the zero-dep posture and works on every adopting project; commitlint is available as an explicit upgrade for projects that already have Node/Bun tooling.

## Risk scan

| Category | Risk | Severity | Mitigation |
|----------|------|----------|------------|
| Hook regex coverage | Pure shell regex misses CC edge cases (multi-line `BREAKING CHANGE:` body footer, multi-byte chars in subject) | medium | Test fixtures cover the common cases per AC-STE-133.10; commitlint opt-in is the escape hatch for projects needing full spec compliance. |
| Cross-platform shell | POSIX shell hook fails on Windows-without-WSL or odd shells | low | Document POSIX-shell requirement in CLAUDE.md template. Windows-with-WSL works; native cmd/PowerShell users can opt into `--commitlint`. |
| Skill prose drift | If `/implement`/`/ship-milestone`/`/pr` prose later reverts, agent commits drift | low | The hook is the deterministic gate per the toolkit's "deterministic gates override LLM judgment" principle. Drift fails fast at commit time. |
| `/setup` overwrite UX | Existing `commit-msg` hook in adopting projects gets overwritten or prompts surprise users | medium | Prompt-before-overwrite + diff display per AC-STE-133.2; never silently overwrite. Decline path leaves existing hook untouched and prints manual install command. |
| `--commitlint` opt-in surface | `bun add` not run automatically — user might miss the install hint and assume deps are present | low | Surface install command prominently in `/setup` output; commitlint hook itself prints a clear "missing dep" diagnostic and exits 1 if invoked without deps installed. |
| Repo first-dep introduction | Adopting commitlint in this repo would force a `package.json` and `bun install` step that doesn't exist today | medium | Avoided by design — this repo uses the shell hook. The hybrid posture is the explicit answer to this risk. |

No high-severity risks. Three medium (regex coverage, overwrite UX, first-dep introduction) — first two addressed by test fixtures + prompts; the third is closed by the design decision itself.
