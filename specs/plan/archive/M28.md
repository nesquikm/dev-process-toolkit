# M28 — Tracker-write enforcement {#M28}

**Release target:** v1.29.0. Codename TBD ("Runbook" / "Concrete" / "Operationally" candidates).
**Status:** drafted 2026-04-25 from M26 dogfood (`/implement M26` skipped all 8 claim+release tracker writes; only the user's explicit observation triggered the manual backfill).
**Origin:** M26 dogfood failure mode — skill's `Provider.claimLock` / `Provider.releaseLock` abstract references combined with the "Do NOT call raw `mcp__<tracker>__save_issue`" forbidden-rule were misinterpreted as "skip tracker writes entirely." This FR closes that interpretation gap.

## Why this milestone exists

The `/implement` skill describes tracker-mode claim/release as `Provider.claimLock` / `Provider.releaseLock` calls. Provider is a TypeScript class with no shell entry — the LLM is the runtime and must make the equivalent MCP tool calls. The forbidden-rule "Do NOT call raw `mcp__<tracker>__save_issue` ... Route through Provider.claimLock/releaseLock" reads (incorrectly) as "make zero tracker writes," because the routing target doesn't exist for the LLM-as-runtime.

M26 surfaced this concretely: 8 FRs, 0 claims, 0 releases, only caught by user observation. Saved a per-machine feedback memory but that doesn't help downstream installs — needs a plugin-level fix.
