# M47 — Skill-prose hygiene: scrub toolkit-meta tracker-ID refs {#M47}

**Release target:** v1.47.0 (minor — prose-only changes; no plugin-facing API change). Codename: TBD (finalize at `/ship-milestone`).
**Status:** drafted 2026-04-30 (single-FR placeholder; future M47 FRs append).

## Why this milestone exists

Active skill prose under `plugins/dev-process-toolkit/skills/*/SKILL.md` carries ~200+ inline `STE-<N>` and `AC-STE-<N>.<M>` tracker-ID cross-references (`STE-117 workspace binding`, `STE-118 milestone attachment`, `STE-87 active-side drift probe`, `AC-STE-110.4`, etc.). These are jargon to downstream operators — the plugin will be installed in projects with their own tracker (or none); the IDs are meaningless to anyone outside our Linear workspace. The skill itself carries the rule that rendered output must never carry toolkit-meta tracker IDs, but the source prose violates the same principle one level up. Provenance belongs in FR files + commit messages, not inline in the skill prose the LLM reads at runtime.

Origin: M46 spec-write surfaced the issue when fresh prose authoring led the user to ask whether toolkit-internal tracker-ID refs belong in plugin skill prose at all. Survey then revealed the project-wide pattern.

After M47 ships:

- `grep -rE 'STE-[0-9]+|AC-STE-[0-9]+' plugins/dev-process-toolkit/skills/*/SKILL.md` returns zero matches (excluding placeholder forms).
- `templates/spec-templates/*.template`, `docs/*.md`, and operator-facing sections of `adapters/*.md` are similarly clean.
- Skill prose reads coherently in plain language, with rule names instead of tracker-ID anchors.
- A future convention is established: never reintroduce inline tracker-ID refs in skill prose.
