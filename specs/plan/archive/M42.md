# M42 — /setup audit unconditional synthesis {#M42}

**Release target:** v1.42.0 (patch — bugfix to `/setup` audit-section post-condition, no schema change). Codename: TBD (finalize at `/ship-milestone`).
**Status:** drafted 2026-04-29 from `/smoke-test` run #4 findings against v1.41.0.

## Why this milestone exists

`/smoke-test` run #4 (2026-04-29) surfaced one high-severity contract break:

- **F1** ([STE-153](../frs/archive/STE-153.md)) — `/dev-process-toolkit:setup` ↔ `/dev-process-toolkit:gate-check` contradiction on explicit-answer runs. With all five Schema L answers pre-baked by the harness (the canonical autonomous-mode path), step 7b/7c/7d's per-step `appendAuditEntry` calls are skipped (they only fire when a default is applied), and step 8a's `synthesizeAuditSection` either no-ops on an empty resolved-defaults table or throws `AuditPostconditionUnsatisfiable`. The resulting `CLAUDE.md` lands without `## /setup audit`. `/gate-check` probe #19 (`setup-audit-section-presence`) then hard-fails on the next pass — toolkit marker present, `branch_template:` populated, `## Docs` present, audit heading absent. Verdict: **GATE FAILED** on the most common autonomous code path.

The fix flips the recording contract on `/setup`'s side: every Schema L resolution at 7b/7c/7d records into the in-scope resolved-values table with a per-entry `reason:` field (`"user-supplied"` or `"default applied"`), so step 8a's `synthesizeAuditSection` always has the full set of candidates to materialize. Probe #19 stays as-is — single source of truth for the trigger predicate.
