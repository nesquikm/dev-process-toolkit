# M69: /spec-write marker-strict gate enforcement {#M69}

Release target: v2.20.0
Codename: TBD

## Summary

Close the marker-detection inference path that lets `/spec-write`'s draft-acceptance gate (§ 0b step 4) and commit-acceptance gate (§ 7a) auto-apply on `claude -p` invocations whose prompt body lacks the literal byte-string `<dpt:auto-approve>v1</dpt:auto-approve>`. Cross-tracker `/conformance-loop` (2026-05-08, plugin v2.19.0) reproduced the regression on both Linear (commit `bbddfcd`, FR STE-261 + M3 plan written silently) and Jira (DST-23 created + commit `04fb609`, no audit row, no refusal). The runtime gate is conflating the harness's autonomous-mode reminder with the byte-checkable STE-226 marker — the exact failure shape STE-213 + STE-220 hit before STE-226 introduced the marker.

**Third attempt at this gate.** STE-213 (M55) + STE-220 (M56) were both prose-only fixes and both were falsified by subsequent smoke runs. STE-226 (M59) added the marker mechanism but kept inference paths reachable at the LLM layer. M69 enforces byte-grep at the runtime / code layer, not at the prose / instructional layer, and adds a `/gate-check` source-level probe that catches future SKILL.md drift back into the bad shape.

Triggered by: `/conformance-loop` iter-1 F1 finding (cross-tracker reproduction documented in `/tmp/dpt-conformance-loop-2026-05-08-iter-1.md`).

## Risks

- **Risk:** The byte-grep enforcement at the runtime / code layer turns out to break marker-present runs because the existing call-site doesn't pass the prompt body through `requireOrRefuse(...)` correctly. **Severity:** medium. **Mitigation:** AC-STE-262.3 is the explicit non-regression gate (existing marker-driven Phase 9-1 + 9-2a/b smoke fixtures must PASS post-fix); the user directive ("triple-check every plugin change") operationalizes as per-AC unit + integration smoke + manual diff review before /implement Phase 4 commit.
- **Risk:** AC-STE-262.4's `/gate-check` source-level probe forbidden-phrase allow-list is too aggressive and triggers false positives on legitimate SKILL.md prose. **Severity:** low. **Mitigation:** TD review pins the allow-list to phrases that are unambiguously alternate-trigger language (no narrow regex; literal substring matches only); the probe lives at error severity, so a false positive halts gate-check loudly and the operator either fixes the SKILL.md prose or tightens the allow-list — both are visible in the PR diff.
- **Risk:** The fix lands on top of M55 + M56 + M59 prior attempts, so the call-site gate logic is already complex; adding another layer of byte-grep without removing legacy inference paths could leave dead-code drift that re-introduces the regression on a future refactor. **Severity:** medium. **Mitigation:** AC-STE-262.7 sweeps the SKILL.md for legacy alternate-trigger prose; the TD review must explicitly enumerate all inference paths in `requireOrRefuse(...)` (or wherever the gate is implemented) and confirm each is removed or replaced — no "added-on-top" code.
- **Risk:** The `needs-technical-review` flag on STE-262 means TD + Testing are placeholders today; if /implement runs against STE-262 before /spec-write STE-262 (no flag) completes the TD pass, /implement will refuse per the existing `implement_refused_needs_technical_review` capability key — but the operator may not realize a second /spec-write pass is required. **Severity:** low. **Mitigation:** the closing-summary `fr_needs_technical_review` row from this /spec-write run names STE-262 explicitly; the user directive is an additional human-side reminder.
