# M68: /conformance-loop permission hardening {#M68}

Release target: v2.19.0
Codename: TBD

## Summary

Drop `--permission-mode bypassPermissions` from every spawn site reachable from `/conformance-loop` and replace the implicit-bypass posture with an explicit `permissions.allow` block in tracked `.claude/settings.json`. The toolkit's "policy artifact" goes from constraint-by-presence (STE-185 threat-model: `.claude/settings.json` exists, so the bypass is acceptable) to constraint-by-content (the file enumerates exactly what the call tree may invoke, reviewable in PR diffs).

Scope is the project-local call tree only:

- `.claude/skills/conformance-loop/SKILL.md` — Phase A `/smoke-test` pair + Phase B `/spec-write` + `/implement` spawn snippets (~5 sites).
- `.claude/skills/smoke-test/SKILL.md` — Phase 2 child spawns of `/spec-review`, `/simplify`, `/gate-check`, `/spec-write`, `/implement` (STE-185/188 sites + the L949 spawn, ~8 sites).

A Phase 0 pre-flight in `/conformance-loop` asserts `.permissions.allow` is non-empty (NFR-10 shape on miss); a new `/gate-check` probe `conformance-loop-bypass-removed` (severity: error) fails if the bypass flag returns to either SKILL.md.

Triggered by: `/brainstorm` 2026-05-08 — operator concern about audit-able policy artifact (no specific bug; the *generality* of bypassPermissions is the issue).

## In Scope

- [x] STE-252 — Replace bypassPermissions with tracked permissions.allow in /conformance-loop call-tree
  - verify: `bun test plugins/dev-process-toolkit/tests/gate-check-conformance-loop-bypass-removed.test.ts plugins/dev-process-toolkit/tests/conformance-loop-args.test.ts` exits 0; manual `/conformance-loop --max-iterations 1 --linear-team STE` run against `dpt-test-project-linear` completes with zero permission-prompt halts.

## Out of Scope (deferred)

- Toolkit-published skills (`plugins/dev-process-toolkit/skills/`) keep their existing posture — they don't carry `bypassPermissions` in spawn snippets and don't need allowlist-artifact changes.
- Linear `/spec-review` / `/simplify` / `/gate-check` themselves are not edited; only the `/smoke-test` and `/conformance-loop` *callers* drop the flag.
- The runtime PreToolUse harness hook (the M67 prose's "M68 candidate" idea for F1 enforcement) is unrelated and stays a future-milestone candidate.

## Risks

- **Risk:** Allowlist incompleteness halts a real `/conformance-loop` run mid-iteration when a child invokes a Bash command not on the list. **Severity:** medium. **Mitigation:** AC-STE-252.5 is the empirical-validation gate (one full run completes with zero halts before M68 ships); the natural feedback model post-ship is "halt → add pattern → re-run" since the loop is invoked at most once per milestone.
- **Risk:** STE-185's "empirical falsification" of `acceptEdits + per-path Write` re-applies and the new design fails the same way. **Severity:** medium. **Mitigation:** the new approach is neither prior-failed mode — it's `default-mode + content-rich permissions.allow` at command-pattern granularity (Bash + MCP, not just per-path file writes). AC-STE-252.5 catches the failure case before ship.
- **Risk:** Tracked `.claude/settings.json` allowlist drift from operator-specific overrides in `.claude/settings.local.json` causes confusion. **Severity:** low. **Mitigation:** the tracked file is the floor; local merges over it; documented in the FR's Technical Design section.
