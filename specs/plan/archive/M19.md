# M19 — Branch automation + PR sweep {#M19}

**Release target:** v1.22.0 "Branch Convention" (working title — reconsider at PR time).
**Status:** draft (not yet kicked off)
**Origin:** brainstormed 2026-04-23 after M18 ship (v1.21.0). User observed that long-lived branches accumulate milestones (e.g. `feat/m12-tracker-integration` carrying M12–M18) and that the current PR has accumulated garbage that needs a one-time sweep.

## Why this milestone exists

Two independent improvements, bundled for ship economy but split into two PRs for review clarity:

1. **Guard-rail + convention for `/implement` branches.** `/implement` has no opinion on git branches today. Users end up on `main` by accident, or on stale feature branches for months. M19 adds a prompt (not a blocker) that proposes a conventional name when the current branch is wrong for the work being started. One keystroke happy-path; explicit abort always available.

2. **Cleanup debt from M12–M18.** `feat/m12-tracker-integration` has accumulated M12 through M18 work — a lot of surface, some of which is probably cruft that slipped through review. Before M19's feature work lands, sweep the branch's diff against `main` with a three-pass audit (orphans → pattern cruft → per-file judgment) and ship the deletions as a standalone PR.

The two items were originally one brainstorm ask; the design session split them into two FRs with distinct ACs and — critically — two PRs, so the sweep's deletion noise doesn't entangle with the feature diff at review time.

## Implementation order (Approach 2 from brainstorm: sweep first, feature second)

**Phase A — Sweep (STE-63):**

- Runs on current branch `feat/m12-tracker-integration` (no branch automation yet — the feature doesn't exist).
- Three passes over `git diff main...HEAD`:
  1. **Orphans** — files with no incoming references (imports, doc links, skill/command mentions, `plugin.json`/`marketplace.json` entries, test-fixture patterns).
  2. **Pattern cruft** — filename patterns (`tmp-*`, `debug-*`, `*_old*`, `*.bak`, `scratch-*`, duplicate `*.snapshot`, `*.orig`).
  3. **Per-file judgment** — interactive walk over survivors with `[Y] keep / [d] delete / [s] skip`.
- One atomic commit: `chore(m19): garbage sweep — <N> files deleted (STE-63)`.
- Report lives in PR-1's description, not in the repo.
- **Ships as PR #1** before Phase B begins.

**Phase B — Branch automation (STE-64):**

- Fresh `feat/m19-branch-automation` branch (created manually — the feature being built is what would normally create it).
- Adds `branch_template:` to Schema L (additive, backward-compatible).
- New adapter module `adapters/_shared/src/branch_proposal.ts` with `buildBranchProposal()` + `isCurrentBranchAcceptable()` (pure functions, fully unit-tested).
- Skill edits: `/implement` Phase 1 gains the branch-proposal step; `/setup` Step 7c asks the one-time template question.
- Docs: `docs/patterns.md` (Pattern 6 update), `docs/setup-tracker-mode.md`, `templates/CLAUDE.md.template`.
- Two new test files: `branch_proposal.test.ts`, `branch_acceptable.test.ts`.
- **Ships as PR #2**.

**Phase A' — Independent guard-rails (STE-65 + STE-66 + STE-67):**

All three independent of STE-63 / STE-64 — can slot anywhere in Phase A/B/C. All three triggered by the M18 ship dogfood and the M19 spec-write session that followed.

- STE-65 — `TrackerProvider.releaseLock` adds a pre-state assertion (status must be `in_progress`). Small code surface (single method + one new error class); five negative-path tests. Bundles well with STE-64's feature PR (both touch tracker-path behavior) but standalone ship is also fine.
- STE-66 — Prose-only convention change to `/spec-write` + `/brainstorm` + two docs; one grep-gate test. Near-zero code surface; can ride with any Phase A/B PR.
- STE-67 — Cosmetic cleanup: hide the full 26-char ULID from user-facing prose (Linear descriptions, plan-file tables, active CHANGELOG / README). Edits the Linear adapter's `ticket_description_template`, `importFromTracker`'s renderer, Pattern 6, one new `/gate-check` probe, retroactive rewrite of M19 Linear descriptions + `M19.md` plan table. Zero functional behavior change. Pairs naturally with STE-66 (both prose-convention).

Recommendation: bundle STE-65 into PR-2 alongside STE-64 (both `/implement`-adjacent). Bundle STE-66 + STE-67 into PR-1 alongside STE-63 (all three subtractive / convention-level). Decision at PR time.

**Phase C — Release (no FR):**

- Bump versions per release checklist (plugin.json, marketplace.json, CHANGELOG.md, README.md).
- v1.22.0 "Branch Convention" — additive feature + cleanup + tracker-lifecycle hardening. Not a breaking change.

## Gate checks

After Phase A: `/gate-check` passes on the swept branch. No test failures from deletions. If a deletion breaks a test, revert that one deletion in a follow-up commit before PR open.

After Phase B: `/gate-check` passes. Two new test files green. `templates/CLAUDE.md.template` contains the `branch_template:` line (new hygiene assertion AC-STE-64.12). Existing projects without `branch_template:` in their CLAUDE.md continue to work (branch automation silently disabled — AC-STE-64.1).

## Deferred decisions

1. **Codename.** "Branch Convention" is a placeholder. Something catchier welcome at PR time.
2. **LLM call model.** AC-STE-64.5 requires a single LLM pass returning `{type, slug}`. Not yet decided whether to use the main `/implement` model or a cheaper haiku call. Decide during implementation; doesn't affect spec shape.
3. **Template validator.** Should `/setup` validate that the user's custom `branch_template:` contains at least `{slug}` and `{N}`-or-`{ticket-id}`? Current ACs accept any non-empty string. Could add a light syntactic check if the first invalid template causes a confusing failure. Decide during implementation.
4. **`{type}` allow-list.** ACs say `feat`/`fix`/`chore`. If the LLM returns something else (e.g., `docs`, `refactor`), do we allow it, or clamp? Default to allow-and-log-warning; revisit if noisy.

## Not in M19 scope (explicit)

- **Branch automation for skills other than `/implement`** — brainstorm explicitly scoped this to `/implement` only. `/tdd`, `/debug`, `/spec-write`, `/pr`, `/gate-check`, `/spec-archive`, `/spec-review`, `/visual-check`, `/simplify`, `/brainstorm` are untouched.
- **Recurring sweep as a skill or gate-check** — sweep is a one-time operation by design. No `/sweep` skill, no new gate-check assertion for "every file must be referenced."
- **Auto-create silent branches** — brainstorm chose (b) prompt over (c) silent auto-create. Prompt is required.
- **Block on main** — brainstorm chose (b) prompt over (a) block. User can still abort and create manually, but `/implement` never hard-refuses.
- **Main as worktree / stacked-branch workflow** — M19 assumes linear git flow. No support for stacked PRs, no worktree integration.
- **Branch deletion / cleanup of old branches** — creating branches is in scope; removing stale ones is not.
