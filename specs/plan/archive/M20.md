# M20 — Docs generation (Diátaxis) + /ship-milestone skill {#M20}

**Release target:** v1.23.0 "Diátaxis" (working title — reconsider at PR time; "Ground Truth" is an alternate reflecting STE-72's anti-hallucination thrust).
**Status:** draft (not yet kicked off; Linear tickets not yet allocated).
**Origin:** brainstormed 2026-04-23 after M19 ship (v1.22.0). User observed that (a) the toolkit has no mechanism to generate or maintain project docs, (b) the Release Checklist is manual and un-dogfooded, and (c) stale README.md has bitten releases before ("Drift Catcher" v1.14.1). Brainstorm included web research (Diátaxis framework, LLM doc drift, mermaid best practices) and duck-council critique (GPT-5.2 + Gemini-3-flash-preview). Design was materially revised from initial proposal in response to duck feedback — see `memory/project_m20_docs.md`.

## Why this milestone exists

Three converging gaps:

1. **No docs-generation mechanism.** Target projects using this toolkit have no prescribed way to produce or maintain `docs/`. Either they write markdown manually (drift quickly), or they lean on stack-specific tools (typedoc, dartdoc) with no cross-stack consistency. M20 adds `/docs` as a first-class skill with a standardized layout and LLM-assisted content.

2. **Release Checklist is un-dogfooded.** CLAUDE.md prescribes a 4-step manual ceremony for every milestone close. Partial updates have shipped before (v1.14.1 "Drift Catcher" literally fixed a README line the previous release forgot). M20 adds `/ship-milestone` to bundle the checklist + docs regeneration as one atomic, human-approved action.

3. **No hook for "milestone done".** `/implement M<N>` walks the plan but exits silently when done. Users forget to close. M20 adds an explicit opt-in prompt to chain into `/ship-milestone`.

The design lands as 8 FRs across four phases; see FR list and implementation order below.

**Phase B — Docs generation core (STE-70 + STE-71 + STE-72):**

## Deferred decisions

1. **Codename.** "Diátaxis" is on-theme but inside-baseball. "Ground Truth" emphasizes STE-72's value. Decide at PR #4 time.

2. **Non-TypeScript stacks for STE-72.** v1 explicitly regex-fallback for non-TS stacks. If Flutter dogfooding surfaces enough pain, a follow-up M21 FR can add dartdoc integration. Not M20.

3. **Cross-FR fragments (STE-74 non-scope).** If `/implement` starts supporting multi-FR batches (it currently doesn't), STE-74 would need to handle multiple fragments per run. Out of M20.

4. **`/docs --dry-run`.** The human-approval gate already acts as a dry-run (user sees diff, refuses). An explicit `--dry-run` flag that skips the approval prompt is convenient for CI but adds code paths. Defer unless dogfooding surfaces need.

5. **Whether M20 PRs merge to main vs. continue on `feat/m12-tracker-integration`.** User preference (per memory): single long-running branch, no worktrees. M20 PRs will likely all go to the same branch; whether that branch ever merges to main is a separate decision.

## Risk scan

| Category | Risk | Severity | Mitigation |
|----------|------|----------|------------|
| LLM correctness | Generated API reference hallucinates signatures (top duck-council concern) | high | STE-72 mechanical ground truth + post-generation validator + retry bound |
| Reviewer fatigue | Per-FR doc churn noises every PR | medium | Fragment-based additive model (STE-70, STE-74); canonical docs untouched until milestone close |
| Orphaned fragments | Fragment written for a never-shipped FR (FR abandoned mid-milestone) | medium | `/ship-milestone` surfaces orphan fragments in its diff; user manually deletes pre-approval |
| Stack coverage | Non-TypeScript projects get weaker STE-71 + STE-72 extraction | medium | Explicit `regex-fallback` strategy with user-facing warning banner; future FR can add dartdoc/pydoc |
| Skill size | `/docs` skill risks exceeding NFR-1's 300-line budget | medium | Overflow to `docs/docs-reference.md` is explicit AC (AC-STE-70.7); must be created with the skill, not deferred |
| Release commit size | `/ship-milestone` creates a large diff (4 release files + `docs/` regen) | low | Single human-approval gate on unified diff; reviewer sees everything before commit |
| Self-hosting risk | First `/ship-milestone` invocation is M20's own release — if buggy, could corrupt own state | medium | Dogfood on `/ship-milestone --dry-run M20` before real run; have manual rollback plan |
| Branch automation interaction | STE-64 branch template interacts with STE-74's FR resolver | low | STE-74 AC-STE-74.2 notes the interaction; existing branch-template tests cover the happy path |

No **high-severity** risks remain unmitigated. The LLM-correctness risk is the one with the tightest mitigation (mechanical ground truth) — the whole design revision from brainstorm exists to address it.
