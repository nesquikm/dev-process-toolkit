# M16 — FR Identity Stability

**Release target:** v1.19.0 "FR Identity Stability"
**Status:** draft (not yet kicked off)
**Origin:** 2026-04-22 brainstorm session — question was whether FR-N is safe under concurrent FR creation across parallel branches. Answer: no, it's a human-chosen allocation and Alice + Bob can each pick the next free number on independent branches. The ground-truth file identity (ULID) and server-allocated tracker IDs are collision-proof by construction; FR-N is the one identifier derived from "scan + increment", which is the classic race-condition shape.

## Why this milestone exists

The design goal approved in brainstorm is **one stable rule for FR identity**: ULID for file identity (already in place), and a collision-proof human-facing prefix derived from the bound tracker ID (tracker mode) or a short ULID slice (`mode: none`). FR-N survives only in immutable history after M16.

The user-stated constraint: "Task tracker issue ID should be the main ref." Tracker mode gets first-class treatment — AC prefixes are `AC-STE-47.1`, headings are `### STE-47: Title {#STE-47}`. `mode: none` falls back to a 6-char ULID slice (`AC-01KPR3.1`) to stay collision-proof without a tracker.

The user-stated scope limit: "nothing on this branch has been used by anyone yet, so we only need to migrate this repository." No reusable migration skill is packaged and no script is committed — a throwaway in-repo rewrite suffices. This is explicitly an internal one-off, not a feature for external adopters (of which there are none).
