# M80: Audit-Fix Loop Pattern {#M80}

Release target: v2.27.0 (provisional, minor)
Codename: TBD (decide at /ship-milestone)

## Summary

The toolkit has two patterns for review→fix loops. The **canonical pattern** (STE-225 M58 + STE-296 M77) uses a main-context orchestrator dispatching forked child skills (`context: fork` + paired `agent:` + `user-invocable: false`) that return findings via a fenced `<role>-result` block; the orchestrator parses and dispatches the fix into another fork, preserving context isolation at every stage. The **legacy pattern** (HG95VF M9, marked superseded by STE-225) uses the `Agent` tool to spawn a subagent that returns findings to the main session, where the fix is then applied in-place — collapsing the isolation guarantee.

The canonical pattern is established in `/tdd` (four forked stages). The legacy pattern still lives on in `/implement` Phase 3 Stage B (`code-reviewer`), `/simplify` (`code-reviewer`), and `/spec-review` (entirely main-context, no fork at all). HG95VF's supersession is documented only inside the archived FR, so skill authors have no canonical guidance for choosing between the two patterns.

M80 closes that gap with three FRs: codify the canonical pattern in `docs/patterns.md` (STE-306), add a `/gate-check` probe enforcing the frontmatter invariants on a const allowlist (STE-307), and migrate `/spec-review` as the first conformance example proving the pattern works on a real skill (STE-308). `/simplify` and `/implement` Phase 3 Stage B follow in M81+ and M82+ respectively — separated so each migration can validate the previous one before committing to the next.

Architectural twin of STE-296 in shape: per-skill audit fork + paired read-only subagent + deterministic fenced-block parser + bounded retry on parse failure. The probe shape mirrors STE-225 AC.7, STE-296 AC.8, and STE-301 AC.15 with shared helpers in `tdd_probe_helpers.ts`.

## Out of Scope (deferred to later milestones)

- `/simplify` migration to the canonical pattern — **M81+**. Smaller scope than `/implement` Stage B; serves as the second conformance example before committing to the highest-traffic loop.
- `/implement` Phase 3 Stage B migration — **M82+**. Hardest migration because the fixer needs full orchestrator state; solving the fix-fork brief-handoff problem requires two prior precedents (STE-308 + `/simplify`) to draw from. Decompose `/implement` into a main-context orchestrator + forked child skills mirroring `/tdd`'s shape.

## Dependency Order

Sequential chain: STE-306 → STE-307 → STE-308. STE-309 is parallel and independent.

- **STE-306** establishes the vocabulary and the rule. No code changes outside `docs/patterns.md`.
- **STE-307** depends on STE-306 for the canonical/legacy classification it enforces. Ships the const allowlist with the four `/tdd` entries; the allowlist module is the source of truth (not parsed from the doc).
- **STE-308** depends on STE-307's allowlist (appends the `/spec-review` entry) and on STE-306's pattern doc (the migration is the first conformance example of the documented pattern).
- **STE-309** is parallel and independent of the STE-306 → STE-307 → STE-308 chain — it touches `skills/setup/SKILL.md` + a new test file + CHANGELOG, none of which overlap with the audit-fix-loop pattern surface. May ship in any order within M80.

## Origin

Brainstorm 2026-05-18 — operator observation that audit findings flow back to main context in `/spec-review`, `/implement` Stage B, `/simplify`, breaking the isolation guarantee that `/tdd` preserves. Framing 2 ("finish the supersession") approved with steps 1–2 bundled into one milestone so the rule ships with a working conformance example, not on its own.
