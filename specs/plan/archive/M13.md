# M13: Parallel-safe Spec Layout

**Status:** planned. Target version: v1.16.0 "Parallel-safe".

Restructures `specs/` from the monolithic 4-file layout to a file-per-FR layout under `specs/frs/<ulid>.md` with repo-stable ULIDs as canonical IDs and tracker IDs as optional frontmatter attributes. Introduces a typed `Provider` interface (`LocalProvider` + `TrackerProvider`) that unifies ID lifecycle so skills never branch on tracker presence. Plan files split per-milestone; archival becomes a git rename; `/setup --migrate` handles v1→v2 migration with a backup tag, dry-run preview, and clean-tree precondition. Eliminates both ID-collision and content-collision classes that make parallel-branch merges painful under v1.

**FRs covered:** STE-26..50 (11 FRs, ~65 ACs).

**Dependencies:** M12. `TrackerProvider` composes over M12's adapter surface (`pull_acs`, `upsert_ticket_metadata`, `transition_status`, `push_ac_toggle`); `/setup --migrate` extends M12's existing migration entry point. M13 co-develops on the same branch as M12 (`feat/m12-tracker-integration`) so both milestones can be integration-tested before a combined v1.15.0 + v1.16.0 PR merges. Phase-level ordering under co-development is specified in the "Co-development on the same branch" paragraph in the Milestone Dependency Graph section below: Phases A–D start any time (all new files, zero overlap with M12); Phases E–G wait for M12 skill edits to stabilize (M12 Phase H Task 1 regression green); Phase H requires all M12 Phase H tasks green or formally deferred; Phase I runs last.

- Phase B: ULID (task 1) blocks everything else in B since provider impls call `mintId`. Within "provider + impls", the interface file (task 2) must land before `local_provider` (task 3), `tracker_provider` (task 4), and `layout` (task 5) start — those three run parallel after task 2

- Phase E skill retrofits — `/setup` (migration flag) depends on Phase D orchestrator; `/spec-write`, `/implement`, `/gate-check`, `/spec-archive`, `/spec-review` are independent retrofits and run parallel after Phase B+C land; read-only-skill verification (last bullet) runs last since it snapshots against the retrofitted outputs

- Phase G depends on Phase E (/gate-check) being landed
