# M52 — Smoke #9/run-2 Follow-ups: Heredoc + Driver Hygiene {#M52}

**Release target:** v2.5.0 (minor — additive prose for the project-local smoke driver, one /setup SKILL.md tightening, one new Jira adapter unit test, no probe-count delta, no plugin-facing API change). Codename: TBD at `/ship-milestone` time.
**Status:** drafted 2026-05-02 from the Linear smoke #9 + Jira run 2 against v2.4.0 (`/tmp/dpt-smoke-findings-2026-05-01-{linear,jira}.md`). 7 FRs: 1 high-severity driver fix (heredoc-on-stdin first-class) + 6 low-severity cosmetic / hygiene fixes spanning the smoke driver, the /setup boundary, and the Jira adapter pull_acs round-trip.

## Why this milestone exists

The 2026-05-01 smoke runs validated the v2.4.0 canonical chain (`/setup → /spec-write → /implement → /gate-check → /spec-review → /simplify`) end-to-end on both Linear and Jira paths. Run-1 Jira findings (F1–F4) all shipped in M51 / v2.4.0. The remaining net-new findings group into three families:

1. **Cross-tracker corruption from prompt-file content swap (high).** Jira run 2's `/setup` child went down the **Linear branch** because an external linter overwrote `/tmp/dpt-smoke-prompt-setup.txt` with a stale Linear-flavored stub between the parent's `Write` and the child's read. The driver caught it, wiped the test project, and recovered via heredoc-on-stdin for the remaining 5 skill invocations. This is the fourth escalation of the stale-`/tmp` family (smoke #6 F1 → #7 F2/F4 → #8 F1 → #9 F1 / run 2). Severe enough to retire the prompt-file path entirely. (STE-185)
2. **Driver hygiene cosmetics (low, parallel).** MCP-config cleanup glob misses one file class; same-day Linear project name collisions need manual `-v2` workaround; non-prompt children emit a `no stdin data` warning that's silenceable with `< /dev/null`; Jira pre-flight #8 could surface the matching KEY when an operator passes the project NAME by mistake. Each is a small, isolated SKILL.md edit. (STE-186, STE-187, STE-188, STE-191)
3. **Plugin-side cleanups (low).** /setup's specs/ scaffolding boundary is ambiguous when restrictive driver phrasing is in the operator's prompt — the LLM read "only /setup completes here" as a license to skip /setup's own scaffold. Jira's pull_acs parser tolerates ADF-escaped `\[x\]` round-trip but the tolerance is implicit; an explicit unit test prevents future regressions. (STE-189, STE-190)

After M52 ships:

- The /smoke-test driver writes zero `/tmp/dpt-smoke-prompt-*.txt` files; cross-tracker corruption via prompt-file content swap is impossible by construction.
- Phase 0.5 cleanup covers MCP config files; same-day project-name collisions auto-resolve to `-v2`/`-v3`; non-prompt child spawns emit no `no stdin data` warning; pre-flight #8 surfaces the matching KEY on single-hit name-match refusals.
- /setup's scaffold list is documented as "emitted regardless of restrictive prompt phrasing"; the LLM cannot reasonably mis-read the smoke driver's prompt.
- Jira pull_acs has explicit ADF-escape-tolerance unit tests; future regressions surface at `bun test` time, not at the next live Jira smoke.
