# M73: STE-285 hook install path portability fix {#M73}

Release target: TBD
Codename: TBD

## Summary

Forward-only fix for the STE-285 (M71) hook installer: `install_hooks.ts:204` interpolates the JS-resolved absolute `pluginRoot` path into `.claude/settings.json` `args[0]` instead of the literal string `${CLAUDE_PLUGIN_ROOT}/templates/hooks/process/<name>.sh` that the Claude Code harness expands at hook-fire time per STE-285 AC-STE-285.3. The bug only manifests on marketplace installs whose runtime plugin path differs from the install-time path; in-tree dev clones (and fixture group 8 by extension) resolve both interpretations to the same path and miss the regression. M73 aligns the implementation with STE-285's documented design intent.

Single-FR milestone. Discovery: user noticed absolute-path entries in a downstream project's `.claude/settings.json` post-M72 ship; brainstorm 2026-05-13 settled the forward-only fix with no migration (zero shipped users per `project_no_users_yet`).

## Out of Scope (deferred)

## Risks

- **Risk:** TypeScript template-literal escape (`\${CLAUDE_PLUGIN_ROOT}`) is unusual; a contributor unfamiliar with the convention may "fix" it back to `${pluginRoot}` thinking the escape is a typo. **Severity:** low. **Mitigation:** STE-288 AC.3 unit test pins the literal-string shape via byte-equality, AC.4 fixture-group-8 grep needle pins the runtime-emitted JSON shape. Both fire on every `bun test` and every `/smoke-test` run respectively.
- **Risk:** Dev clones that ran `/setup --hooks` against an earlier `install_hooks.ts` carry stale absolute-path entries; on next `/setup --hooks` re-run, those entries don't match the new prefix and the menu shows them as not-installed. **Severity:** low. **Mitigation:** operator re-checks the menu options; legacy entries remain in `settings.json` but are inert (the harness no longer expands the absolute path as `${CLAUDE_PLUGIN_ROOT}`).
