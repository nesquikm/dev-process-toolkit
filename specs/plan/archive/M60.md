# M60 — Non-technical FR drafting (--no-tech flag) {#M60}

**Release target:** v2.11.0 (minor — additive at the user-facing surface; one new flag on `/spec-write` + `/brainstorm`, one new optional FR-frontmatter field, one new `/implement` refusal site, one new `/gate-check` probe. ## Why this milestone exists

Stakeholders without technical context (PMs, product owners, operations leads) currently can't drop a draft FR into `specs/frs/` without first running through the full `/spec-write` flow, which interviews them on architecture, data model, APIs, and test strategy. Existing alternatives (filing a Linear ticket separately, sending Slack notes, ad-hoc markdown files) bypass the toolkit's source-of-truth contract.

This milestone adds a `--no-tech` flag to `/spec-write` and `/brainstorm` that lets non-technical stakeholders author the **WHAT** (requirement + acceptance criteria) and skip the **HOW** (technical design + testing). The resulting FR lands in `specs/frs/<id>.md` with `needs_technical_review: true` in frontmatter; `/implement` refuses to ship it until a technical reviewer runs `/spec-write <id>` (no flag) to fill the missing sections.

After M60 ships:

- `/spec-write --no-tech` and `/brainstorm --no-tech` are documented and callable.
- FRs with `needs_technical_review: true` carry placeholder lines in `## Technical Design` + `## Testing` and a `needs-technical-review` label on the tracker board.
- `/implement` refuses single-FR or whole-milestone runs that include flagged FRs (Phase 0.b′ refusal, no claim, no branch).
- `/spec-write <id>` (no flag) auto-detects the flag and runs the technical-completion interview.
- `/gate-check` carries a new probe `needs_technical_review_consistency` enforcing the flag ⇔ placeholder bidirectional invariant.
- A two-tier smoke covers the end-to-end pass (flagged-FR creation → /implement refusal → flag-clear → /implement success) on Linear + Jira.
