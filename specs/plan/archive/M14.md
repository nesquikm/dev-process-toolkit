# M14: Tracker-native Entry Paths

**Status:** planned. Target version: v1.17.0 "Tracker-native Entry".

Lets `/spec-write`, `/implement`, and `/spec-archive` accept tracker IDs (`LIN-1234`, `PROJ-42`, `#982`) and full tracker URLs as first-class arguments via a shared resolver utility (`resolveFRArgument` + `findFRByTrackerRef`). On first encounter with an unmapped tracker ref, `/spec-write` and `/implement` import the ticket from the tracker via `Provider.getMetadata`, mint a ULID, and create `specs/frs/<ulid>.md` — no STE-17 per-AC prompt loop on the initial import since the local side is empty. `/spec-archive` resolves existing tracker refs to ULIDs but refuses to auto-import. Ambiguity across configured trackers is resolved by an explicit `<tracker>:<id>` disambiguation form with deterministic NFR-10-shape errors.

**FRs covered:** STE-30..55 (5 FRs, ~32 ACs).

**Dependencies:** M13 (Provider interface — `mintId`, `getMetadata`, `sync`; `regenerateIndex`; file-per-FR layout; skill retrofits). M12 (adapter 4-op surface). **M14 cannot start until M13 Phases A–E are landed** — specifically Phase B (Provider + LocalProvider + TrackerProvider), Phase C (INDEX generator), and Phase E (skill retrofits) must be in place because M14 layers resolver calls on top of M13's retrofitted skill entry points. The remaining M13 phases (F lock state machine, G plan-file discipline, H dogfood, I release) are not prerequisites for M14 Phases A–C, but M14 Phase D (release) must wait until both M12 and M13 have shipped to avoid tri-release entanglement.
