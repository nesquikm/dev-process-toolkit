# M21 — De-ULID tracker mode {#M21}

**Release target:** v1.24.0 (provisional). Codename TBD.
**Status:** implemented (3 FRs, all active pending Phase 4 archive, awaiting human approval on `/implement M21`).
**Origin:** identified 2026-04-23 during M20 spec-write. User observed that `id: fr_<ULID>` frontmatter in tracker-mode FRs is ceremonial — the filename and AC prefix are the tracker ID, STE-67 (M19) made ULIDs invisible in user-facing prose, and `findFRByTrackerRef` already handles tracker-workspace renames. Scope expanded 2026-04-24 during M21 brainstorm to full scope-3 structural isolation (capability interface, Option 2) after the user asked that scripts support both modes cleanly — "eliminate ULID from everywhere in tracker mode."

## Why this milestone exists

Tracker-mode ULIDs are ceremony without corresponding value:

- Filename is the tracker ID (post-M18 STE-60).
- AC prefix is the tracker ID (post-M16 FR Identity Stability).
- User-facing prose has no ULIDs (post-M19 STE-67).
- Tracker-workspace renames are handled by `findFRByTrackerRef` via the `tracker:` map.

Removing the ceremony simplifies the mental model: "the Linear ticket is the identity in tracker mode". Mode-none keeps short-ULIDs unchanged (it has no tracker-side identity to defer to). Beyond spec-text + data migration, M21 also refactors `Provider` so the invariant "tracker-mode code never mints a ULID" is type-enforced rather than convention — any future accidental `mintId()` call on the tracker path becomes a compile error.

## Out of scope

- Mode-none ULID changes — mode-none keeps the short-ULID filename + AC prefix convention. Short-ULID IS the identity in mode-none.
- Git-history rewrite of ULID references in old commit messages, CHANGELOG entries, or archived prose — historical records remain intact. Change is forward-only from v1.24.0.
- Changes to the `tracker:` frontmatter map — that stays as the cross-ref mechanism.
- Removal of `adapters/_shared/src/ulid.ts` — still needed by `LocalProvider` for mode-none FR creation.

## Risk scan

| Category | Risk | Severity | Mitigation |
|----------|------|----------|------------|
| Invariant regression | Removing `id:` breaks a downstream consumer (skill, adapter, probe) we haven't identified | medium | Warning-first probe (STE-76 AC-STE-76.7 / STE-86 AC-STE-86.6) surfaces violations without blocking; full dogfood cycle before flipping to hard-fail |
| Migration atomicity | Partial migration leaves inconsistent archive | low | Script is all-or-nothing per-file (temp-file + rename); on any error in write-mode, script exits non-zero; rollback via `git revert` on the single atomic commit (STE-76 AC-STE-76.9) |
| Byte-identical preservation | YAML round-trip would reorder keys / strip comments / re-quote strings | low | Line-based surgery (not YAML parse); test-locked by byte-identical assertions in `strip_ulid.test.ts` |
| Type-refactor call-site miss | STE-85 misses a `mintId()` call site; typecheck passes anyway (e.g., unsafe cast) | low | Pre-refactor grep audit (STE-85 AC-STE-85.4); `tsc --noEmit` is the enforcement; any cast-bypass shows up in review diff |
| Ordering inversion | STE-76 runs before STE-86 lands → no tool to run | low | Single-PR sequencing A → B → C; phase commits are ordered; PR review catches reversal |
| Schema Q complexity | Mode-conditional `id:` confuses future spec-write readers | low | Explicit conditional clause in Schema Q with both branches + cross-ref to NFR-15; probe enforces at every gate-check |

No high-severity risks. All medium/low risks have explicit mitigations in the relevant FR ACs.
