# M40 — Surface {#M40}

**Release target:** v1.40.0 (minor — observable user-facing output changes in /spec-write and /implement). Codename: "Surface" (provisional; finalize at `/ship-milestone`).
**Status:** drafted 2026-04-28 from /smoke-test findings against v1.39.0.

## Why this milestone exists

The /smoke-test run on 2026-04-28 against plugin v1.39.0 produced three plugin findings, all clustered around user-visible output hygiene:

- **F1** (STE-147) — /spec-write leaks internal toolkit AC IDs (`AC-STE-118.4`) into user-facing summary prose. Project owners running /spec-write on their own repo see opaque jargon that has no meaning outside the toolkit's own spec set.
- **F2** (STE-148) — /implement's bounded review loop classifies Stage B Pass 2 CONCERNS as advisory, but the concerns themselves do not survive into the final assistant message under `claude -p`. Operator loses both the audit trail and the chance to override.
- **F3** (STE-149) — /setup leaves a literal `AC-<tracker-id>.1` example row in the rendered Traceability Matrix of `requirements.md`. Cosmetic, but a careful first-time reader is briefly puzzled, and a future `requirements-md-no-placeholder` gate-check rule would flag it as drift.

All three are output-formatting / template edits — no behavior change in tracker sync, no schema changes, no migrations.

## Implementation order

The three FRs are independent — different files, different skills, no shared touchpoints. Default phase ordering is by severity (B → A → C) but the operator can run them in any order, including in parallel sessions.

- **Phase A** — STE-147: edit `skills/spec-write/SKILL.md` § 7 + the in-band capability checks; replace AC-ID-keyed templating with a static plain-language map. Pure prose / template edit.
- **Phase B** — STE-148: edit `skills/implement/SKILL.md` Phase 3 Stage B + Phase 4 archive step; thread `advisoryNote[]` into the report template and the archived-FR write. Slightly larger scope (two write sites, one shared formatter).
- **Phase C** — STE-149: edit `templates/spec-templates/requirements.md.template` only; remove the example row from the rendered table, fold its convention note into the existing HTML comment. Smallest change.

## Gate

- All three FRs ship with passing tests + clean /gate-check + clean /spec-review.
- Re-run /smoke-test against v1.40.0 (post-merge); F1, F2, F3 must not re-appear in the next findings file.
- /gate-check probe `requirements-md-no-placeholder` (if added in this milestone or before) must pass against the post-/setup `requirements.md` fixture.
