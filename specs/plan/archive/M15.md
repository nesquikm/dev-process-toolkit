# M15 — Migration Hardening

**Release target:** v1.18.0 "Migration Hardening"
**Status:** draft (not yet kicked off)
**Origin:** 2026-04-22 dogfooding session — `/setup --migrate none → linear` against the plugin's own repo surfaced 8 concrete doc-code gaps; a follow-up `/implement STE-35` roundtrip probe surfaced 2 more (Findings #9 and #10); a second roundtrip probe `/implement FR-57 FR-58` surfaced 3 more (Findings #11–#13 on tracker-mode cosmetics + resolver); a brainstorm session on archive-path drift surfaced Finding #14 (v1 `specs/archive/` references across docs, skills, and live specs). Each gap is a dedicated FR; an additional Documentation + Release FR rounds out the milestone for 15 total.

## Why this milestone exists

The v1.17.0 tracker-mode surface had never been exercised end-to-end against a v2-layout project. Running it revealed that:

- The skill's migration-detection rule (`SKILL.md:33`) excludes the `mode: none → tracker` path despite the procedure doc supporting it.
- The procedure doc (`docs/setup-migrate.md`) still references v1-layout paths (`specs/requirements.md` for FR iteration, "traceability matrix" for binding writes).
- Linear's native Project Milestone field was never populated by migration — shipped projects lose their milestone taxonomy in the tracker.
- Initial ticket state defaulted to Backlog with no prompt, misrepresenting already-shipped work.
- `INDEX.md` regeneration (AC-40.4 requires rebuild after any write under `specs/frs/`) wasn't part of the procedure.
- `specs/requirements.md` overview had rotted into a de-facto changelog.
- The v1→v2 migration tool (M13 Phase D) left per-milestone blocks in `technical-spec.md` (1213 lines) and `testing-spec.md` (578 lines), violating AC-40.3.
- The v1 `specs/archive/` path (flat milestone files + rolling `index.md`) was superseded by per-unit archives at `specs/frs/archive/` + `specs/plan/archive/` (FR-45) — but README, three docs, three skills, and the live cross-cutting specs all still reference the dead v1 path.

Claude applied judgment-calls during the session to work around each gap, completing the migration successfully but leaving a trail of real bugs to close.
