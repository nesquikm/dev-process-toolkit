# M27 — Dart/Python docs parity {#M27}

**Release target:** v1.30.0. Codename TBD ("Polyglot" / "Trilingual" / "Parity" candidates — decide at `/ship-milestone`).
**Status:** drafted 2026-04-25 from M20 brainstorm follow-up. Approach 2 from the M27 brainstorm (memory: `project_m27_docs_dart_python.md`).
**Origin:** M20's STE-72 ships mechanical signature extraction for TypeScript only (`typedoc → ts-morph → regex-fallback`). Dart/Python projects fall through to `regex-fallback` with a "manual review advised" banner. Real Flutter and Python projects using this toolkit get worse packages-mode reference docs than TS projects. M27 closes that gap so the LLM-as-prose-writer-not-signature-inventor invariant (NFR-22) holds across all three stacks.

## Why this milestone exists

The `/docs` skill's packages-mode reference generation is load-bearing for downstream-project doc quality. When the signature extractor falls through to regex-fallback, downstream users get a banner-warned reference page with imprecise signatures — honest but unhelpful. Dart/Flutter and Python are explicit target stacks (per existing `examples/{flutter,python}/`), so the parity gap is felt directly.

## Risk scan

| Category | Risk | Severity | Mitigation |
|----------|------|----------|------------|
| External dependencies | `package:analyzer` ABI churn — analyzer versions every ~2-3 months; major bumps could break the bundled Dart helper | medium | Pin `analyzer: ^7.0.0` for minor-version flexibility; `pubspec.lock` committed for reproducibility. Major bumps surface at `pub get` or import time → fall through to regex with warning per AC-STE-103.2. |
| External dependencies | `griffe` JSON shape drift — griffe is actively developed; the translator (`griffeJsonToModuleSignatures()`) is the bug-prone glue | medium | Pin `griffe>=0.40.0` (current major); fixtures cover known shape; new fixtures added when downstream surfaces unhandled cases (per STE-104 Notes). Translator failures fall through to regex with stderr surfaced. |
| Bundled-helper packaging | Dart helper requires `pub get` on first use (~5-10s); Node→Dart cross-runtime spawn complexity | low | One-time cost cached after first use. `.dart_tool/` check before spawn. `pub get` failure → fall through to regex. Documented in STE-103 Notes. |
| Test infrastructure | CI environments may not have Dart SDK or `griffe` installed — tests would fail spuriously | low | `describe.skipIf(!hasDart)` and `describe.skipIf(!hasGriffe)` patterns per AC-STE-103.8 + AC-STE-104.9. CI workflow doc note recommends installing both for full coverage; absence reduces coverage but doesn't break the build. |
| Schema migration | Existing /setup records (post-M20, pre-M27) won't have `signature_extraction_preferred_strategy` field added by STE-105 | low | Probe treats missing field as "no recorded preference" → skip cleanly per AC-STE-105.5. No migration script. Field appears naturally when /setup is re-run. |
| Cross-helper asymmetry | Dart bundles a helper, Python doesn't — appears inconsistent | low | Asymmetry is intrinsic (Dart has no JSON-emitting CLI for API surface; Python does). STE-103 Notes covers this honestly. Approach 2's "no middle tier" applies per stack to chain shape, not to implementation mechanism. |
| LLM behavior on translator | `griffeJsonToModuleSignatures()` flattening may miss edge cases (overloads, generic type vars, nested classes) | medium | M20's NFR-22 post-generation validator (AC-STE-72.4) catches LLM-invented signatures regardless of extractor bugs — bug surfaces as "ground-truth missed an export" warning, not silent corruption. New fixtures added per STE-104 Notes when surfaces. |

No high-severity risks. Two medium (analyzer ABI, griffe shape) — both mitigated via version pinning + regex-fallback safety net.

- **Auto-install missing toolchains.** /setup surfaces install commands but does not run them. Auto-install crosses a security boundary (running `pip install` or modifying user's Dart SDK without explicit consent).

## Brainstorm origin

Brainstorm originally ran 2026-04-24 tentatively as M23. Re-slotted multiple times as the M2x slots filled (M23 → M25 → M27 → M27 confirmed once M28 shipped first as the Runbook hot-fix milestone). Memory file: `project_m27_docs_dart_python.md`. Approach 2 (two-tier chain per stack: external tool preferred → regex fallback; skip the bundled-AST middle tier M20 has for ts-morph) was approved. Rejected approaches:

- **Approach 1 (three-tier with bundled AST helpers):** rejected — bundled Dart/Python AST helpers in a Node plugin = three-language maintenance burden + cross-runtime spawning complexity that doesn't pay back relative to a one-command user install. Note: STE-103's bundled Dart helper is NOT a re-implementation of AST walking (which the constraint targeted) — it just delegates to `package:analyzer`. The constraint allows this per the memory's open-verification clause.
- **Approach 3 (hard-require external tool, no fallback):** rejected — diverges from M20's "works everywhere" promise. TS users get a regex fallback; denying it to Dart/Python users is inconsistent and would lose users at the install gate.

## Open verification resolved during /spec-write

The brainstorm flagged one open item: "confirm `dart doc --json` flag actually exists; if not, primary becomes `package:analyzer`." Resolved 2026-04-25 against Dart SDK 3.11.4: `dart doc --help` shows only `--output`, `--validate-links`, `--dry-run`. **No JSON output mode.** STE-103's primary strategy is therefore `package:analyzer` via the bundled Dart helper, not `dart doc --json`.
