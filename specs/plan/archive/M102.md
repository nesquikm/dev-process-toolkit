## M102: Consent — token stats opt-in flag {#M102}

**Goal:** Make the M92 per-skill token-usage stats feature **opt-in, off by default**. `/setup`
asks; disabled means *fully* off — the bundled `SessionEnd`/`Stop` capture hook no-ops (no
`.dev-process/token-ledger.jsonl` writes) and all three render sites (`/spec-write`,
`/implement` Phase 4, `/ship-milestone`) emit nothing. Existing projects stay OFF until they
re-run `/setup`, for free via the `absent-section ⇒ enabled:false` parser convention. Closes
the "no per-feature opt-out" gap: today the only way to silence token stats is to disable the
whole plugin.

**Prerequisites:** v2.43.0 shipped (main @ PR #35). Load-bearing existing machinery:
STE-344/345/346 (the M92 capture hook, ledger, render + `claimRowsForFR` — the surfaces gated
here), STE-347 (`## Verification` dedicated-section + `readVerificationConfig` safe-default
parser — the structural twin), STE-68 (`## Docs` interview + `readDocsConfig` absent⇒false +
atomic re-run splice), STE-238 (byte-checkable literal-token contract). No new external
dependencies. **Architectural constraint:** the capture hook is plugin-bundled and
auto-discovered (M74) — there is no per-consumer `settings.json` entry to omit, so "fully off"
requires the hook body itself to read the flag.

**Release target:** v2.44.0 "Consent". ⚠️ **Contends with M101** (unshipped, on branch
`feat/ste-375-jira-milestone-as-epic`, also targeting v2.44.0). Whichever milestone ships
first takes v2.44.0; the later one rebases onto main and bumps. `/implement` Phase 1 +
`/ship-milestone` re-validate the target against the live `plugin.json`, so this is a soft
prediction, not a hard claim.
