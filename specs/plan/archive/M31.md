# M31 — Tracker Workflow Hardening {#M31}

**Release target:** v1.33.0. Codename TBD (decide at `/ship-milestone`).
**Status:** drafted 2026-04-27 from a /spec-write conversation that surfaced three discretionary-check failures during M30 spec-write.
**Origin:** During M30 spec-write 2026-04-27, three failure modes appeared in succession — each because a check that should be system-enforced was left to LLM discretion:

1. **Project-binding miss.** `mcp__linear__save_issue` was called with `team` only (Linear requires it; missing → loud failure) but no `project` (Linear allows missing → silent landing in "no project"). STE-115/STE-116 landed outside the user's expected DPT board. Schema L's canonical key set is closed (per STE-114) and the workspace binding (team + project) was never given a sub-section to live in.
2. **Project-milestone miss.** Even after the project was patched, the Linear ticket had no `projectMilestone` attached. STE-103 (M27) has `projectMilestone: "M27 — Dart/Python docs parity"` because it was set manually. STE-115/116 do not. Same root cause: discretionary work the LLM forgot.
3. **Milestone-number collision.** Initial `/spec-write` pass picked **M28** for the new doc-cleanup milestone — but M28 (v1.29.0 "Runbook"), M29 (v1.30.0 "Loudly"), and M27 (v1.31.0 "Polyglot") were all already shipped. The LLM trusted a truncated `ls` instead of cross-checking against archive + CHANGELOG. User caught it; required full rename pass + Linear description rewrite.

The common shape: **memory rules and reminders only help the current session. The check has to live in skill code or `/gate-check` probe code to survive across sessions and users.**

## Why this milestone exists

The toolkit's tracker integration (M12 onward) routes every spec-touching skill through canonical Provider operations. The four operations (`pull_acs`, `push_ac_toggle`, `transition_status`, `upsert_ticket_metadata`) were designed assuming the user's *workspace identity* (which team, which project) is system-known. It isn't. M31 closes that gap and converts three discretionary checks into hard gates so they survive future sessions.

## Cross-spec additions

- **`requirements.md`:** none new — slimmed to cross-cutting per AC-STE-26.3; M31 is FR-level mechanism work.
- **`technical-spec.md`:**
  - **Schema L extension prose** — new section under §7.3 documenting the `### Linear` and `### Jira` Schema L sub-sections. Key set: `### Linear` carries `team:` (string) + `project:` (string) + optional `default_labels:` (string-array YAML). `### Jira` carries `project:` (string) + optional `default_labels:`. Parsing rule: greedy until the next `###` or `##` heading.
  - **`upsert_ticket_metadata` signature update** — adapter table in §7.4 grows `team?` + `project?` columns. Marked optional in the canonical signature; required-or-not is per-adapter (linear: project required to attach to project board; jira: project required to create at all).
  - **Probes inventory** — three new probe entries: `task-tracking-workspace-binding-present` (#25), `tracker-project-milestone-attached` (#26), `frontmatter-milestone-not-archived` (#27). Each row carries the standard probe metadata (file path, vacuous-on rules, severity).
- **`testing-spec.md`:** test-file inventory grows by ~6 files (named in Gate checks above); skip-on-mode rules unchanged.

## Out of scope (tracked separately)

- **Backfilling `projectMilestone` for archived FRs** (STE-103/M27, STE-101/M28, etc.). Archive is immutable. If ever needed, separate FR after explicit archive-edit policy change.
- **Initiative attachment.** Linear has a project → initiative hierarchy; `mcp__linear__list_projects` accepts an `initiative` filter. Out of scope — initiatives are workspace-organizational and don't have a clean toolkit-side analogue.
- **Custom-tracker adapter rollout** of the workspace binding. Only `linear` and `jira` adapters update; the `_template` adapter gains a placeholder example. Custom adapters (per `_template.md`) opt in by reading the new sub-section themselves.
- **Linear-side milestone-status sync** (e.g., flip the milestone to "completed" when `/ship-milestone` runs). Separate FR; M31 only handles attachment.
- **Mode-none milestone tracking.** Mode-none has no tracker object to attach to; FR-A/B silently no-op in mode none, FR-C still applies (the pre-allocation guard is mode-agnostic).
