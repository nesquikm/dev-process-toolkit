# M29 — /setup hardening + spec-write/spec-archive paper cuts {#M29}

**Release target:** v1.30.0. Codename TBD ("Hardening" / "Smoke" / "Loudly" candidates — decide at `/ship-milestone`).
**Status:** drafted 2026-04-25 from a `claude-st -p` smoke-test run of v1.29.0 against a fresh Bun + TypeScript project (`../dpt-test-project`) with Linear tracker integration ("DPT Smoke Test" project, STE-102). Findings file: `/tmp/dpt-smoke-findings.md`. **Updated 2026-04-25** with two additional FRs (STE-113, STE-114) from a second smoke run later the same day (findings: `/tmp/dpt-smoke-findings-2026-04-25.md`, smoke-test issue STE-112 in "DPT Smoke Test (2026-04-25)" project).
**Origin:** the v1.29.0 release was the first end-to-end exercise of `/setup` + `/spec-write` + `/implement` + `/spec-archive` under autonomous (non-interactive) operation. Eight functional gaps surfaced — five in `/setup` (silent output drops, autonomous-mode discipline, no bootstrap commit, Bun stack guidance gap, Schema L key drift) and three in the spec-write/archive pair (template `id:` leak, traceability link rot).

## Why this milestone exists

The smoke test confirmed the chain works on the happy path — Linear MCP integration is live, TDD flow produces correct code, gate-check rubric runs cleanly. But eight issues block this same chain from being agent-driven cleanly:

1. `/setup` silently drops headline outputs (`.claude/settings.json`, `.mcp.json`, `## Docs` section) under conditions that should be loud failures.
2. The skill collapses verification probes (always-run) and Q&A steps (prompt-mode-respecting) under one "do not prompt" rule.
3. `/setup` outputs are never committed — they leak into the first feature PR.
4. `/setup` lacks Bun-stack guidance: `bun test` exits 1 on zero tests, breaking the very first gate after scaffold; the only known workaround is an undocumented placeholder file. (Surfaced in the 2026-04-25 re-run, STE-113.)
5. `/setup` writes non-canonical Schema L keys under `## Task Tracking` because the skill prose doesn't constrain the key set. Drift across projects, no enforcement. (Surfaced in the 2026-04-25 re-run, STE-114.)
6. `/spec-write` emits `id:` ULID frontmatter in tracker mode despite AC-STE-86.5 saying it shouldn't (already detected by gate-check probe-13 as a warning).
7. `/spec-archive` doesn't update traceability matrix links after the file move.

M29 closes all eight. Seven of them ship a new gate-check probe so future regressions are caught at gate-check time.

## Risk scan

| Category | Risk | Severity | Mitigation |
|----------|------|----------|------------|
| Skill prose ambiguity | The "verification ≠ Q&A" split (STE-108) relies on the skill prose being unambiguous. An LLM agent might still conflate the two if the annotations are buried. | medium | Render the `default:` / `requires-input:` annotations *immediately above* each question, not in a footnote. Add an example block at the top of the skill showing the contract shape. |
| Bootstrap commit blast radius | `STE-109`'s "abort if unrelated uncommitted changes" rule could surprise users running /setup mid-feature. | low | Documented in SKILL.md "## After /setup" section. The autonomous-mode default still commits (per AC-STE-109.3); only interactive runs prompt. |
| CHANGELOG scoping (STE-111) | Released-section detection scans top-of-file until first dated `## [X.Y.Z]` header. If the CHANGELOG format ever drifts (e.g. undated headers), the scoping breaks. | low | The format is fixed by the Release Checklist (CLAUDE.md). STE-78 (M22) installed strict format conventions. Probe-level test asserts scoping behavior on the canonical and on a malformed CHANGELOG. |
| Probe-13 upgrade impact (STE-110) | Existing toolkit FRs that still carry `id:` would fail the new error-severity probe. | low | Pre-flight grep at /implement time: list all tracker-mode active FRs with `id:`, fix in the same PR. M16/M21 archived FRs are frozen and exempt. |
| Self-imposed sandbox guard origin | If F1/F2's "sandbox blocked" was actually emitted by the parent harness (not the skill), the fix in STE-106 won't help. | medium | STE-106 covers both cases (skill-self-block AND harness-block). Investigation step at the start of /implement: trace the actual write-failure path in the v1.29.0 skill prose. If harness-side, escalate to a separate FR. |
| Bootstrap commit + repo-not-git interaction | `git init` inside `/setup` means the skill becomes the canonical first-commit author. Acceptable but flag-worthy. | low | Existing skill prose already runs `git init`; STE-109 just adds the commit on top. Documented in CHANGELOG. |
| STE-113 marker convention churn | If implementation picks "marker comment" but operators prefer "magic filename" later, projects scaffolded under M29 carry the wrong convention permanently. | low | Pick at implementation time and pin in `examples/bun-typescript.md`. The migration cost is one rename; not blocking. |
| STE-114 probe false positives on legitimate adapters | A future adapter (Jira-Cloud, GitHub Issues) might want a tracker-specific Schema L key that's genuinely canonical for that mode but not in the closed set. | medium | Probe scope is `## Task Tracking` top-level only; subsection (`### Linear`, `### Jira`) contents are exempt by design. Adding a new canonical key requires a deliberate `docs/patterns.md` edit + probe code change in the same PR. |
| STE-114 migration script blast radius | If the script auto-applied, it could mangle a hand-curated CLAUDE.md. | low | Script is dry-run-only by design; outputs unified diff to stdout. Operator pipes to `patch` if they want it. Tested against malformed inputs. |

No high-severity risks. Two mediums worth a brief investigation step at /implement time (origin of the sandbox-blocked write; future-adapter Schema L extension policy).
