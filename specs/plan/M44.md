# M44 — /smoke-test Jira branch {#M44}

**Release target:** v1.44.0 (minor — driver-side `.claude/skills/smoke-test/SKILL.md` extension; no plugin-facing change). Codename: TBD (finalize at `/ship-milestone`).
**Status:** drafted 2026-04-29 as a follow-up to M43. Sequencing: M43 ships v1.43.0 first; M44 builds on the corrected adapter.

## Why this milestone exists

M43 shipped as **v1.43.0 with AC-STE-154.9 (manual smoke #5 trace) operator-deferred** — only step 1 (visibility check) was live-verified during `/implement`; steps 2–6 stayed unverified in `specs/notes/jira-smoke-5.md`. The Jira adapter is therefore live in main but not end-to-end-validated against the live MCP. Meanwhile `/smoke-test` (the dogfood driver this repo uses for Tier 5 conformance) still hard-codes `mode: linear`, so Jira regressions could re-creep in without an automated tripwire. M44 lands `/smoke-test --tracker jira` so smoke #6 closes both gaps in one run — the **retroactive validation** for M43 and the regression-tripwire for M44 onwards. After M44 ships:

- Both adapter modes (linear, jira) get the same automated conformance pass.
- Future Jira-side regressions are caught by re-running `/smoke-test` rather than by ad-hoc manual passes.
- The shape of the canonical chain (`/setup → /spec-write → /implement → /gate-check → /spec-review → /simplify`) is identical in both modes; only Phase 1 (project setup) and Phase 5 (teardown) branch.

The asymmetry vs Linear is real but bounded: Atlassian MCP exposes no `createJiraProject` / `deleteJiraIssue`. Jira runs therefore reuse a pre-existing throwaway Space (`DST` for the stellartechlab site) and clean up by transitioning labeled work items to `Done` rather than archiving a project. Documented as a limitation, not a bug.
