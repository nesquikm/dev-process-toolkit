# M23 — Self-hosted release hardening {#M23}

**Release target:** v1.24.0. Codename TBD.
**Status:** draft (kicked off 2026-04-24 from M20 dogfood findings).
**Origin:** identified 2026-04-24 during the first self-hosted `/ship-milestone` run (shipping M20 "Diátaxis"). Three friction points surfaced: (1) pre-flight #1's canned remedy text misleads when FRs are status-active-but-tracker-Done, (2) `specs/` was gitignored in this repo so archival produced no commits, (3) `TrackerProvider.releaseLock` throws on already-terminal tickets during bulk `/spec-archive`. (2) was resolved out-of-band by un-gitignoring `specs/` in a separate chore commit; M23 ships the remaining two as behavioral fixes.

## Why this milestone exists

`/ship-milestone`'s intended flow (milestone-scope `/implement M<N>` → `/ship-milestone M<N>`) was never exercised against a real milestone until M20. The dogfood surfaced two paper cuts that bite every milestone close. Both are cheap to fix individually; together they make `/ship-milestone` usable without per-call hand-holding.

## Risk scan

| Category | Risk | Severity | Mitigation |
|----------|------|----------|------------|
| Invariant regression | Loosening `releaseLock` to accept terminal-Done weakens STE-65's `Backlog → Done` silent-leap guardrail | medium | Non-Done-and-non-In-Progress pre-states still throw (the exact set STE-65 guards against); `getTicketStatus` verification in `/implement` Phase 4 step (c) is unchanged; add explicit unit test for the narrowed guard |
| Signature change ripple | `releaseLock` return type changes `Promise<void>` → `Promise<"transitioned" \| "already-released">` — 3 call sites must touch up | low | All call sites are in the same plugin; type-checker catches uncaught callers at build time |
| Misleading-remedy regression | Pre-flight #1 remedy could drift back to the misleading text in a future SKILL.md edit | low | Prose-assertion shape test locks the new wording + cross-reference to `/spec-archive` |

No high-severity risks. The `releaseLock` invariant tightening is the only behavior change with a guardrail concern — the proposed narrowing (`in_progress` → `in_progress | done`) matches the adapter's canonical terminal-state semantics and is tested.
