## M100: Attested — fork-consumer reliability {#M100}

**Goal:** A scary-looking fork output must not cascade into silent feature-death or a false
high-severity alarm. Close the `deps-research` fork enforcement asymmetry — add the
deterministic `deps_research_result_shape` gate its `spec-research` twin already has (#41),
and make every deps-research disposition byte-checkable including the previously-silent skip
paths, with no legal "compromised" disposition (STE-373) — and make `/report-issue`
evidence-based: capture the session that actually contains the incident, and cap+tag any
high-severity report whose evidence is absent (STE-374). Origin: the 2026-07-09
`/report-issue` gist that reported a deps-research prompt-injection whose own attached
transcript neither reproduced nor even invoked the fork.

**Prerequisites:** v2.42.0 shipped (main @ PR #34). Load-bearing existing machinery:
STE-301 (`parseDepsResearchBlock` parser, `deps_researcher_subagent_invariants` probe #51,
the three `deps_research_*` capability keys), STE-230 (`spec_research_result_shape` probe
#41 + persisted-log convention = the clone template), STE-238 (byte-checkable literal-token
contract), STE-229 (report-issue most-recent-mtime session capture), STE-296 (verify claims
against artifacts, not narrative). No new external dependencies. **Architectural
constraint:** no code seam exists between a `context: fork` subagent and its parent — the
fork's turn text lands directly in the parent LLM's context; only the persisted
`.dpt-locks/**` log is available to deterministic code. So enforcement is *post-hoc gate +
definition-level token contract*; the hermetic persist-and-extract seam (brainstorm
Approach #3) is explicitly **out of scope**.

**Release target:** v2.43.0 "Attested".

**Dependency order:** STE-373 → STE-374. The two FRs touch disjoint code surfaces

can land in either order; STE-373 first keeps the upstream half (what produced the scary
output) ahead of the downstream half (how the report over-reacted).
