# M58 — Multi-agent TDD orchestrator (context isolation) {#M58}

**Release target:** v2.11.0 (minor — additive; replaces existing single-context `/tdd` with a plugin-bundled skill+subagent fork orchestrator. No users → rip-and-replace per memory `project_no_users_yet`.)
**Codename:** TBD at `/ship-milestone` time.

**Status:** drafted 2026-05-05 from a brainstorm + spec-write session that locked the design across five open questions in `research/sdd-tdd-improvements-2026-05.md` Gap 1.

## Why this milestone exists

`/tdd` today runs RED → GREEN → REFACTOR in a single context. The test-writer subconsciously designs around the implementation it's about to write, so test-first becomes test-last-pretending-to-be-first. The fix (per the brainstorm) is **execution isolation via the Skills `context: fork` + custom-subagent pairing** documented at `https://code.claude.com/docs/en/skills#run-skills-in-a-subagent`: each phase runs as a subagent with locked-down tools, fresh context, and a strict text-only hand-off contract. Plugin-bundled, plugin-internal — no user-facing surface beyond the existing `/dev-process-toolkit:tdd` invocation.

After M58 ships:

- `/dev-process-toolkit:tdd` invocation works as before from the user's perspective (same name, same FR-id arg).
- Internally, each phase is a forked subagent — test-writer cannot see implementation, implementer cannot see refactorer's planned cleanup, etc.
- `/implement` Phase 3 calls the new orchestrator inline (no separate opt-in path).
- `/gate-check` carries a new structural probe `tdd_orchestrator_integrity` that asserts the four skills + three subagents have correct frontmatter.
- A two-tier 100%-automatic smoke runs on every `bun test` (Bun-mocked) + nightly via env-var-gated headless live invocation against a fixture FR.

## M58: Multi-agent TDD orchestrator {#M58-fr-block}

**Goal:** Replace the existing single-context `/dev-process-toolkit:tdd` with an orchestrator skill that runs three forked-subagent phases (test-writer / implementer / refactorer) with a strict fenced hand-off contract, bounded retry on failure, structural probe in `/gate-check`, and a 100%-automatic two-tier smoke (Bun-mocked plumbing every commit + nightly-gated headless live fixture).

## Notes

**Brainstorm provenance.** Five open questions resolved in the 2026-05-05 brainstorm session:

- Q1 (refactor batching) → once at end of FR after all ACs GREEN.
- Q2 (hand-off contract) → strict fenced ` ```tdd-result ` YAML block, deterministic parse.
- Q3 (failure recovery) → bounded retry, max 2 per role per AC for semantic failures; single targeted retry for format violations; `maxTurns` counts as a failed attempt; halt + escalate after cap; retry prompt injects only raw failing-test output.
- Q4 (gate-check probe) → full structural probe scoped to load-bearing invariants only.
- Q5 (smoke) → 100% automatic two-tier (Bun-mocked plumbing every commit + nightly-gated headless live fixture).

**Doc cross-checks (Claude Code official docs).** Before authoring, confirmed against the five docs in `research/sdd-tdd-improvements-2026-05.md` header — `context: fork` is a stable Skills feature; the experimental `Fork the current conversation` subagent feature is **not** what this milestone uses; plugin-bundled subagents drop `hooks`, `mcpServers`, `permissionMode` (constraint encoded in AC-STE-225.10).
