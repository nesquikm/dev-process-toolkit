# M79: Per-project task-status configuration {#M79}

Release target: v2.26.0
Codename: Mapped

## Summary

Today the toolkit's tracker integration speaks in two canonical role names (`in_progress`, `done`) and binds them to status names via a `status_mapping` block in each adapter's frontmatter (Linear adapter, Jira adapter). That mapping is **per-adapter** — one Linear mapping ships, one Jira mapping ships — so a Jira workspace whose workflow doesn't match the adapter's defaults breaks at the first `Provider.sync` write or trips strict gate probes (STE-87, STE-54). A live Jira project surfaced this.

M79 promotes `status_mapping` from per-adapter to per-project: a new `specs/tracker-config.yaml` carries the project's actual status vocabulary verbatim plus a role-to-status map drawn from a fixed role enum (`initial`, `in_progress`, `in_review`, `done`). `/setup` (idempotent re-entry on already-bootstrapped projects) queries the tracker MCP for the workspace's full status list + workflow, proposes a best-match role mapping by name similarity, surfaces a diff-approval gate, and writes the file. A tolerance wrapper around every Provider read/write detects non-key statuses (e.g., QA moved a task to `In QA` between plugin operations) and prompts the operator at runtime — force / skip / cancel — instead of erroring. STE-87 and STE-54 probes relax to defer to the wrapper.

Architectural twin of STE-301 in shape: per-project YAML file beside `requirements.md`, dedicated loader module, downstream consumers refactored to read from the new source.
