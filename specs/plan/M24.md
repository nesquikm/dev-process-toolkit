# M24 — Claim enforcement {#M24}

**Release target:** v1.25.0. Codename TBD ("Claim Enforcement" / "Symmetry" contenders).
**Status:** drafted 2026-04-24 from M23 "Self-Hosted" dogfood finding.
**Origin:** during M23 `/implement` (2026-04-24) Phase 1 step 0.c `Provider.claimLock` was skipped entirely; the mid-session patch used `mcp__linear__save_issue` directly, sliding around STE-65's `Backlog → Done` silent-leap guardrail. A sibling hazard surfaced during M24's own `/spec-write`: the agent narrated a specific unallocated tracker ID in conversation, violating STE-66's placeholder rule. Neither failure would be caught by any deterministic check today.

## Why this milestone exists

Three prose rules exist but have no file-level enforcement:

1. STE-54's archive-side ticket-state-drift probe (`/gate-check` probe #8) is half of a pair — no active-side counterpart exists to catch "active FR stuck at Backlog or transitioned to Done out-of-band."
2. `/implement`'s Phase 4 Close prose says "call `Provider.releaseLock`" but nothing forbids a direct MCP tool call that bypasses STE-65's precondition guard.
3. STE-66's `<tracker-id>` placeholder rule is prose-only — no probe catches an FR file whose AC prefixes don't match the file's bound tracker (e.g., drafted with a guessed ID that turned out wrong).

M24 closes all three gaps with one FR. Mirrors STE-54 symmetrically on the active side; adds a sibling probe for guessed-ID detection; adds skill-local prose rules in `/implement`, `/brainstorm`, and `/spec-write` for the hazards probes can't catch (conversational leaks).
