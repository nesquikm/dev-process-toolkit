# M81: Marker-Absent First-Turn Refusal {#M81}

Release target: v2.28.0 (provisional, minor — fix milestone, but new probe + hook touches public surface)
Codename: TBD (decide at /ship-milestone)

## Summary

`/conformance-loop` iteration 1 (2026-05-19) surfaced two high-severity regressions in the first-turn marker carve-out that STE-226 + STE-237 + STE-251 + STE-262 + STE-294 collectively designed:

- `/spec-write` draft gate (§ 0b step 4) and branch gate (§ 7a) BOTH auto-apply under `claude -p` non-tty stdin when the `<dpt:auto-approve>v1</dpt:auto-approve>` marker is absent, by rationalizing the harness `<system-reminder>` ("work without stopping for clarifying questions") as license to proceed. Cross-tracker confirmed: Linear leg created `STE-311` + `STE-312` on master; Jira leg auto-created tickets through the same code path.
- `/setup` Phase 8 Socratic first-turn writes scaffold files at `tool_use index=12` BEFORE any `AskUserQuestion` under the same conditions. `assertFirstTurnShape` raised `SocraticFirstTurnViolationError` against `setup-2026-05-19.json` (112,572 bytes).

STE-294 AC.4 explicitly fenced this regression shape in skill text, but skill text is not byte-checkable — the LLM reasons past it. M81 closes the gap with a deterministic enforcement vector: a `PreToolUse` hook that blocks `Write` / `Edit` / `NotebookEdit` in `/spec-write` and `/setup` invocations whenever the marker-check helper has not fired, plus a regression-fixture test loop replaying the three captured prompt bodies through `requireOrRefuse`.

Architectural twin of STE-262 in shape: the byte-checkable runtime helper IS the gate decision. M81 wraps the helper in a hook so the LLM literally cannot proceed past a marker-absent gate without invoking it.
