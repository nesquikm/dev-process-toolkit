# M62 — `/report-issue` skill: structured bug-report capture {#M62}

**Release target:** v2.11.0 (minor — additive; one new skill, two new helper modules, one cross-cutting edit to `/brainstorm`, one probe update on STE-228. Bundles with M58 + M59 + M60 + M61 in the same release window. No users → rip-and-replace per `project_no_users_yet`.)
**Codename:** TBD at `/ship-milestone` time.

**Status:** drafted 2026-05-05 from a `/brainstorm` session 2026-05-05 (Approach 3 selected: tiered context capture — default curated, `--full` opt-in transcript, redaction non-negotiable, hard `gh auth` prerequisite).

## Why this milestone exists

Toolkit users (and the maintainer self-debugging) currently have no structured return-channel for plugin issues. Bug reports come via Slack screenshots and partial transcripts; triage is guesswork. A 2026-04-28 smoke caught a regression where a toolkit-internal AC identifier surfaced as a rendered prose description in `/spec-write` summaries — that bug was reproduced from a screenshot, not from a structured payload, and reproducing it cost a full session of context-spelunking.

`/report-issue` closes the loop with a redaction-aware gist payload that `/brainstorm` can consume:

- A toolkit user hitting an unexpected behaviour runs `/report-issue`, answers four Socratic prompts, sees a preview, approves, and gets a gist URL back.
- The maintainer pastes the URL into `/brainstorm <gist-url>` to triage with full context.
- The same dev can also self-debug by feeding their own URL into `/brainstorm` in a fresh session.

After M62 ships:

- `/dev-process-toolkit:report-issue` is callable from any project that has the plugin installed.
- Default invocation produces a curated-only gist (no transcript); `--full` or an in-flow `[y/N]` prompt opts the transcript in.
- Every payload passes through `scrubSecrets` before upload (seven canonical patterns).
- `/dev-process-toolkit:brainstorm` accepts `https://gist.github.com/...` as the first positional argument and fetches the gist payload as its seed.
- STE-228's `commit_producing_skill_branch_gate` probe consults an explicit allowlist (`NON_COMMIT_PRODUCING_SKILLS`) rather than parsing universally; `/report-issue` is on the list.
