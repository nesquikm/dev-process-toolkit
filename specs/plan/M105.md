## M105: FR Summary section + review contract {#M105}

**Goal:** make generated FRs reviewable by humans without touching their precision. GitHub issue
#39 — the toolkit's **first external-user feedback** — reports that specs are too detailed to
review; `/brainstorm` 2026-07-16 confirmed the core problem is human review burden and approved
the additive shape: (1) `/spec-write` authors a plain-language `## Summary` as the **first FR body
section**, altitude-limited so it can never absorb implementation detail; (2) a `/gate-check`
probe enforces that altitude deterministically, vacuous when the section is absent; (3) the docs
and approval prompts state the **review contract** out loud — the human approves Summary +
Requirement + ACs; Technical Design + Testing are implementation-facing, verified by the
deterministic gates and the TDD audit. The tracker leg costs nothing: both adapters render
`{fr_body}` into the ticket description on create and every sync, so a first-in-body section
opens every ticket automatically, and the AC-sync loop provably never sees non-AC prose.

**Prerequisites:** v2.46.0 shipped (main @ PR #40). Load-bearing existing machinery: the
additive-optional-section lineage (Design References, Token Stats — each shipped with its own
convention test), the full-body ticket-description template, the WHAT/HOW review split precedent
(`--no-tech` + deferred technical review), and the probe-count drift pin (README ↔ gate-check
numbered list). Verified 2026-07-16: no deterministic gate rejects a sixth body section — the
section contract is prose, so the amendment is textual. Additive-only: no existing FR is edited,
no AC/design/testing precision changes anywhere.

**Release target:** v2.47.0 (minor). Codename TBD at `/ship-milestone`. ⚠️ **Contends with M101**
(unshipped, on branch `feat/ste-375-jira-milestone-as-epic`; its recorded target v2.46.0 is
already taken by M104, so M101 rebases + bumps regardless; whichever of M101/M105 ships first
takes v2.47.0). `/implement` Phase 1 + `/ship-milestone` re-validate against the live
`plugin.json`, so this is a soft prediction, not a hard claim.
